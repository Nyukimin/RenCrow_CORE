# ドキュメント整理 引き継ぎREADME

この引き継ぎの目的は、RenCrow / PicoClaw のドキュメントを **読めばシステムを理解できる、使える、構築判断できる** 状態に保つことです。

今回の引き継ぎでは、実装そのものではなく、ドキュメント構造・正本管理・除外/復帰判断を引き継ぎます。

## まず理解してほしいこと

RenCrow のドキュメントは、全部を読む前提ではなく、入口を固定して読む設計にしています。

- `02_正本仕様/`: 仕様・実装仕様・Runtime Config・IdleChat の正本。
- `01_理解/`: 初見の人が概要と Agent 責務を掴むための統合ガイド。
- `03_記憶検索/`: ローカルファースト検索・記憶アーキテクチャの入口。
- `04_構築指標/`: 実装順と受入基準。
- `05_運用/`: セットアップやコードベース解析など、作業前の注意点。
- `99_整理/`: 何を残し、何を外したかの根拠。

## 作業の進め方

1. まず `README_引き継ぎ版.md` とこのファイルを読む。
2. 次に `00_読む順番.md` に沿って最小セットを読む。
3. 読みながら、分からない点を `レビュー結果.md` に記録する。
4. 更新すべき文書を見つけたら、いきなり直さず `04_作業チケット雛形.md` の形でチケット化する。
5. 正本変更が必要な場合だけ `02_正本仕様/` を更新し、その後に派生文書を更新する。
6. 元 docs.zip から資料を戻す場合は、`99_整理/除外ファイル一覧.csv` の `restore_when` と照合し、戻した理由を残す。

## 判断に迷った時の原則

- 仕様の意味を変えるなら、正本を更新する。
- 読みやすくするだけなら、`01_理解/` または `04_構築指標/` を更新する。
- セットアップや実行手順なら、`05_運用/` を更新する。
- 履歴確認だけなら、元 docs.zip を参照し、最小セットには戻さない。
- 旧仕様を復活させる場合は、なぜ現行正本より優先するのかを説明できる必要がある。

## 終了条件

引き継ぎ作業の最初の区切りは、次の状態です。

- 正本・派生・運用・整理の区分を説明できる。
- 新しい担当者が、どのファイルから読むべきか説明できる。
- 更新してよいファイル、更新してはいけないファイルを判別できる。
- 未解決論点をチケットとして整理できる。
