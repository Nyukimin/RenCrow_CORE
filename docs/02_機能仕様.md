# 機能仕様

## 会話とルーティング

通常メッセージの route owner は Mio です。処理順序は、明示 command、Mio の `DecideAction`、rule dictionary、classifier、決定 route の実行です。無効な判断や利用不能な runtime は、契約で定めた failure または CHAT fallback として扱います。

RenCrow_PORTAL、Debug Viewer、CLIの通常会話は`to=mio|shiro|kuro|midori`を受け付けます。明示commandは会話相手の選択より優先されます。選択したruntimeが利用不能な場合、Mioへ黙って切り替えません。

## Worker、Coder、Advisor、Tool

- Worker は決定済み route の実行、状態、結果、side effect、承認境界を所有する。
- Coder は設計、差分、提案を返し、外部 side effect の最終責任者にならない。
- Advisor は専門的な助言を返し、採否と実行責任を持たない。
- Tool は明示された capability を実行し、人格、長期記憶、最終意思決定を持たない。

## Agent間の口頭移譲

Agentが別のAgentへ作業を移譲する場合は、内部route、task payload、logだけで完了させず、共通Chat上で次の3発言を必須とする。

1. 移譲元は移譲先のAgent名を最初に呼び、移譲内容とその時点の会話内容を伝える。
2. 移譲先は移譲元のAgent名を最初に呼び、受け取った移譲内容と会話内容を必ず復唱する。復唱がない状態で作業開始扱いにしない。
3. 作業終了後、移譲先は移譲元のAgent名を最初に呼んでから、成果、失敗、未完了点を報告する。

これらは `agent.delegate` / `agent.acknowledge` / `agent.report` の会話eventとして同一sessionに残し、移譲先にも同じ移譲内容と会話内容をtask contextとして渡す。TTSのON/OFFは音声再生の可否だけを変え、会話eventの生成と表示を省略する理由にしない。Shiro内部のWorker実行器など、独立したAgentでない内部component呼び出しはこの口頭移譲の対象外とする。

## Memory、Recall、Knowledge

- 会話履歴は発言ごとの `from` と `to` を保持する。
- RecallPack は複数の検索結果を budget、provenance、情報種別ごとの安全policyとともに組み立てる。
- Knowledge Relation は Entity、Topic、Project を制限付き 1-2 hop で横断し、採用理由を trace に残す。
- 外部検索結果、Advisor 出力、収益候補は、人間確認なしに確定 Knowledge へ昇格しない。

### Chatにおけるエピソード想起

Chatの記憶想起はKVキャッシュ制御ではなく、現在の話題に関連する過去のエピソードをCOREが検索し、Agentの非表示コンテキストとして渡す機能とする。Agentはそのエピソードを回答の参考にとどめてもよく、話題に合う場合は「そういえば以前……」などの自然な発言として会話へ出してもよい。

- 想起しただけのエピソードは会話履歴へ保存しない。
- COREは、想起結果をAgentの発言に見せかけた偽の会話messageとして履歴へ直接挿入しない。
- Agentが実際に口に出した内容は、通常のassistant発言として共通Chat履歴へ保存し、以後の会話コンテキストに自然に含める。
- 会話履歴とrolling summaryは現在の会話文脈、エピソード想起は過去の出来事の連想、Knowledge検索は仕様や事実の参照として区別する。
- 会話履歴、過去のエピソード、確定済みのユーザー記憶は全Agentの共通記憶とし、Agentのroleによってアクセスを制限しない。Knowledge、外部検索結果、運用情報は共通記憶と混同せず、能力と安全policyに応じた個別の注入制限を維持する。

## 継続作業と承認

Backlog、Scheduler、Heartbeat、Workstream が長時間作業を管理します。外部公開、送信、請求、契約、価格決定などの high-impact action は approval gate を通過するまで draft に留めます。

## Economic Objective

Opportunity、EconomicTask、RevenueEvent、EconomicReflection を管理します。Heartbeat の discovery は候補の draft 作成までに制限し、外部実行を自動化しません。

## PORTAL、Debug Viewerと観測性

Debug Viewerは会話、agent、job、log、workstream、approval、memory、advisor、economic stateをREST/SSEで表示します。RenCrow_PORTALの`view`はAI VTuber部屋の閲覧専用、`live`は部屋・操作UIを含まない配信用、`lab`はallowlist内の会話操作を許可するAI VTuber部屋です。raw traceは要約と詳細を分け、`unavailable`、`pending`、`degraded`、`error`を成功表示へ丸めません。

## IdleChatのお題とストック

IdleChatのお題は、通常カテゴリの都度生成と、未来展望（Forecast）の永続ストックを分けて扱います。通常カテゴリは各sessionの開始時に候補を生成してJudgeで1件を採用します。Forecastは、AI技術、その他技術、医療、社会保障、政治、経済の6ドメインごとに最大2件、全体で最大12件の有効なお題を保持します。

Forecastストックのキーワード抽出とお題生成はShiroが担当し、Shiroと同じWorker LLMルートを使います。Coder系のForecast providerは未来展望sessionの要約など別責務に限定し、お題ストック生成には使いません。ShiroルートではJudge応答の途中終了を避けるため、1試行の候補を最大3件にします。

Forecastストックの生成契約は次の通りです。

1. 起動時に永続ストックを読み込み、空topic、重複topic、解析不能なrecordを有効在庫として数えない。
2. 読み込み後の有効在庫が全体で0件の場合に限り、起動時bootstrap生成を行う。ストックファイル未作成、空ファイル、全件無効、全件消費済みを同じ0件条件として扱い、6ドメインを順番に各1件まで初期生成する。
3. 有効在庫が1件以上ある通常再起動では、起動時bootstrap生成を行わない。ドメインごとの不足はIdle時またはHeartbeat時に補充する。
4. Idle時およびHeartbeat時に、COREの優先処理、ユーザー会話、Chat、Worker、Coder、既存IdleChatが生成資源を使用していないことを確認し、不足しているドメインを上限2件まで段階的に補充する。
5. 1回の補充契機で生成を無制限に連鎖させない。同一ドメインの同時生成を禁止し、生成資源がbusyなら失敗扱いにせず次回のIdle／Heartbeatへ延期する。
6. お題を1件消費したら不足状態として記録し、会話開始を補充処理で待たせず、次回のIdle／Heartbeatで補充する。
7. ストックが空のまま利用要求を受けた場合はインライン生成を許可する。ただし、生成失敗、空topic、重複、seed不足を別のお題や別カテゴリの成功へ丸めない。
8. 生成・消費・破棄・補充のたびに永続ストックを更新し、再起動で有効在庫を再利用する。保存済み在庫がある限り、再起動だけを理由に同じ補充を繰り返さない。
9. CORE Debug Viewerは読み取り専用snapshotとして、合計件数、生成中状態、最終補充契機、最終エラー、各ドメインの件数、お題本文、生成日時、シードを表示する。Viewer表示や再読込を在庫生成・消費の契機にしない。

起動時bootstrapには、初回導入、ストック消失・破損、全件消費後でも最初のForecastを在庫なしで待たせない役割があります。Idle／Heartbeat補充には、通常稼働中の負荷を平準化しながら各ドメインの在庫を最大2件へ戻す役割があります。この二つを代替関係として扱わず、どちらか一方を削除してはいけません。

最低限の契約testは、ストック未作成、全件無効、部分在庫、満杯、消費後、生成資源busy、同時Heartbeat、生成失敗、保存後再起動を対象とします。部分在庫では起動時生成を行わずHeartbeatで補充されること、保存後再起動では不要な再生成が起きないことを確認します。

## 音声と外部 runtime

CORE は STT/TTS の接続、request、Viewer 操作、状態を持ちます。表示文生成、音声合成、音声取得、ブラウザ再生は別々の成功条件です。LLM も inference endpoint と management endpoint を分け、ready 確認後に選択を確定します。
