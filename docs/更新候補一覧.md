## 2026-07-15 正本仕様棚卸しの作業項目

### 進め方

- 元の課題IDを親として維持し、実作業は枝番で管理する。別名の課題IDを増やさない。
- `resolved` 3件は再実施せず、完了証跡と回帰防止だけを維持する。
- `tracked` 5件は、lifecycle明示、内容監査、正本への吸収・昇格、章分割の順に進める。
- `refs`にある文書は、ファイル名や過去の位置づけだけで正本へ昇格しない。現行code・testとの一致を確認してから採否を決める。
- 正本の章分割は、契約不足の吸収・不採用判断が終わった後に行う。古い内容をそのまま分割して固定しない。

### 領域監査で必ず記録する項目

`DOC-GAP-01`と`DOC-GAP-02`は、各候補について次を1行ずつ記録する。

| 項目 | 内容 |
| --- | --- |
| `candidate` | 候補文書のpathと対象章 |
| `claimed_role` | 正本、補助仕様、調査、作業資料、履歴のどれを名乗っているか |
| `current_owner` | COREが接続契約を持つか、外部moduleが実装本体を持つか |
| `code_evidence` | 現行実装のproducer / consumer / route / state |
| `test_evidence` | unit、contract、integration、E2Eの対応 |
| `decision` | `absorb` / `promote` / `keep_ref` / `archive` |
| `reason` | 採用・不採用理由 |
| `replaces` | 置換・廃止する文書または章。なければ`none` |
| `follow_up` | 正本、派生資料、台帳、testへ必要な次作業 |
