# 思考プロンプト自動適用システム仕様

## 0. 位置付け

この文書は、RenCrow に「思考プロンプト自動適用システム」を追加するための実装仕様である。

目的は、ユーザー入力、作業種別、リスク、出力形式に応じて、以下 5 つの思考スキルを自動選択し、必要な時だけプロンプトへ差し込むことである。

1. Semi-Formal Reasoning
2. Verbalized Confidence
3. Town Hall Debate Prompting
4. Reference Class Priming
5. Deliberate Over-Instruction

## 1. 基本方針

- 5 つの思考スキルをすべて System Prompt に常駐させてはいけない。
- 常駐させるのは「選択ルール」と「適用フロー」だけとする。
- 実際の思考スキル本文は、必要な時だけ Prompt Builder が差し込む。
- RenCrow は自律モデルであるため、ユーザーが明示しなくてもタスク状況から自動選択する。
- 通常会話では重くしすぎない。
- 既存の Chat / Worker / Coder / Verifier の責務境界を壊さない。
- Coder は破壊的操作を直接実行せず、plan / patch / proposal を生成する。
- Worker が実行主体である。
- すべての重要処理には EventId を付与する。

## 2. 実装ゴール

以下の処理フローを追加する。

```text
User Input
  ↓
Task Classifier
  ↓
Risk / Complexity Analyzer
  ↓
Reasoning Policy Engine
  ↓
Reasoning Skill Selector
  ↓
Recall Pack + Persona Prompt + Task Prompt + Reasoning Skill Prompt
  ↓
LLM Execution
  ↓
Output Gate
  ↓
必要なら再生成または Verifier へ差し戻し
  ↓
Final Response
```

## 3. 追加する主要コンポーネント

### 3.1 Task Classifier

ユーザー入力と現在コンテキストから、次の分類情報を生成する。

出力スキーマ:

```json
{
  "task_type": "chat | writing | summarization | code_review | spec_review | patch_validation | architecture_decision | product_decision | strategy | research | factual_check | implementation_plan | unknown",
  "domain": "string",
  "complexity": 1,
  "risk_flags": [],
  "needs_freshness": false,
  "needs_citation": false,
  "needs_style_consistency": false,
  "is_summarization": false,
  "is_decision": false,
  "is_code_or_spec_review": false,
  "requires_reference_example": false,
  "recommended_skills": []
}
```

`complexity` は 1 から 5 とする。

`risk_flags` には以下を含められるようにする。

- `factual_uncertainty`
- `hallucination_risk`
- `destructive_change`
- `compatibility`
- `security`
- `regression`
- `external_dependency`
- `legal_or_policy`
- `cost`
- `performance`
- `privacy`
- `data_loss`
- `user_memory`
- `architecture`
- `style_drift`
- `compression_loss`

分類は最初はルールベース中心でよい。必要なら LLM 分類に差し替え可能な interface にする。

### 3.2 Reasoning Policy Engine

Task Classifier の結果を受け取り、適用すべき思考スキルを選択する。

実装方針:

- ルールは `configs/reasoning_policy.yaml` に外出しする。
- コードに直書きしない。
- スキルは複数同時適用可能とする。
- 通常会話では原則スキルを適用しすぎない。
- スキル適用理由を metadata に残す。
- EventId とともに `skill_selection_log` を残す。

## 4. 5 つの思考スキル仕様

以下の 5 ファイルを `prompts/reasoning_skills/` 配下に追加する。

### 4.1 `semi_formal_reasoning.md`

用途:

- コードレビュー
- 仕様レビュー
- パッチ検証
- アーキテクチャ検証
- 破壊的変更の確認
- 因果関係の検証

本文:

```text
Use semi-formal reasoning before answering.

Internal steps:
1. List assumptions and confirmed facts.
2. Separate confirmed facts, inferred claims, and unknowns.
3. Trace concrete execution paths, dependencies, branches, side effects, and failure paths.
4. Derive only conclusions supported by the above.
5. Mark unsupported claims as unknown.

For code/spec review, check:
- compatibility
- destructive changes
- missing tests
- exception paths
- rollback conditions
- affected modules
- data migration risks
- public API changes

Do not expose the full internal reasoning trace.
Expose only:
- conclusion
- key evidence
- risks
- unknowns
- next checks
```

### 4.2 `verbalized_confidence.md`

用途:

- 事実確認
- 数値
- 最新情報
- API 仕様
- 外部依存
- 不確実性がある回答
- ハルシネーション抑制

本文:

```text
For each major claim, assign confidence from 0 to 100 internally.

Confidence is not proof.
Confidence is only a signal for deciding whether additional verification is needed.

Rules:
- confidence >= 85: may be stated normally if supported.
- 70 <= confidence < 85: state with caution.
- 50 <= confidence < 70: mark as needs verification.
- confidence < 50: do not use as a conclusion.

For claims below 70, identify:
- what is uncertain
- what source, test, log, file, command, or primary document should verify it
- whether the answer should be delayed, marked uncertain, or escalated

Final output should not list confidence for every sentence unless the user asks.
Instead, expose uncertainty as:
- confirmed
- likely
- uncertain
- needs verification
```

### 4.3 `town_hall_debate.md`

用途:

- 設計判断
- 方針決定
- 複数案比較
- プロダクト判断
- コスト・リスク判断
- 複雑な意思決定

本文:

```text
Evaluate the problem through multiple internal roles.

Roles:
1. Proposer:
   Argues for the plan and identifies the value.
2. Critic:
   Searches for failure modes, hidden assumptions, and weak points.
3. Operator:
   Checks implementation burden, maintenance burden, workflow impact, and operational risks.
4. Cost Owner:
   Checks cost, time, opportunity loss, and resource constraints.
5. Verifier:
   Separates confirmed facts, assumptions, and unknowns.

Procedure:
1. Each role raises up to 3 key points.
2. Each role may challenge or refine one other role's point.
3. Verifier classifies the final claims into confirmed / inferred / unknown.
4. Do not end with a simple vote.
5. Integrate the arguments into a conditional decision.

Final output:
- conclusion
- reasons to adopt
- biggest risks
- alternatives
- unverified assumptions
- required checks
- decision: adopt / conditional adopt / hold / reject
```

### 4.4 `reference_class_priming.md`

用途:

- note 記事
- X 投稿
- 仕様書
- レポート
- ニュース要約
- 同じ型の大量出力
- キャラクター口調の安定
- Qwen 系の出力揺れ対策

本文:

```text
Use the provided reference example as the quality anchor.

Match:
- tone
- structure
- granularity
- paragraph length
- reader distance
- level of explanation
- heading style
- amount of detail
- use of examples
- ending style

Do not copy the content.
Do not reuse facts from the reference unless they are relevant to the new task.
Apply the same quality standard to the new input.

Before final output, check:
- structure matches the reference
- granularity has not become shallow
- tone has not drifted
- task-specific facts are reflected
- no irrelevant phrasing was copied from the reference

If no reference example is available, fall back to the matching styleguide and template.
```

### 4.5 `deliberate_over_instruction.md`

用途:

- 要約
- 圧縮
- 議事録
- 仕様整理
- レビュー結果まとめ
- 長文ドキュメント処理
- 前提・例外・反対意見を落としたくない場面

本文:

```text
When summarizing or compressing, do not remove these fields:

- assumptions
- scope
- prerequisites
- exceptions
- boundary cases
- objections
- alternatives
- risks
- unknowns
- tests or verification gaps
- rollback conditions if applicable

Short output is allowed.
Loss of these fields is not allowed.

If the original text lacks one of these fields, write "not stated" rather than inventing it.

For technical summaries, always preserve:
- compatibility constraints
- destructive change risks
- public API changes
- data migration risks
- error paths
- untested paths

Final output should remain concise, but these fields must remain visible.
```

## 6. Output Gate

`configs/output_gates.yaml` を追加する。

```yaml
gates:
  confidence_gate:
    enabled: true
    applies_to:
      - factual_check
      - research
      - spec_review
      - code_review
      - architecture_decision
    threshold:
      caution_below: 85
      verify_below: 70
      block_below: 50
    fail_action:
      caution: mark_uncertain
      verify: add_verification_needed
      block: remove_or_isolate_claim

  compression_loss_check:
    enabled: true
    applies_to:
      - summarization
      - report
      - review_summary
      - spec_summary
    required_fields:
      - assumptions
      - exceptions
      - objections
      - unknowns
    fail_action: revise

  decision_gate:
    enabled: true
    applies_to:
      - architecture_decision
      - product_decision
      - strategy
      - implementation_plan
    required_fields:
      - conclusion
      - risks
      - alternatives
      - unknowns
      - next_action
    fail_action: revise

  review_gate:
    enabled: true
    applies_to:
      - code_review
      - spec_review
      - patch_validation
    required_fields:
      - conclusion
      - evidence
      - risks
      - unknowns
      - tests
    fail_action: revise
```

## 7. Prompt Builder 拡張

Prompt Builder は以下を順に組み立てる。

1. Base system / role prompt
2. Persona prompt
3. Recall Pack
4. Task instruction
5. Selected reasoning skill prompts
6. Output format instruction
7. Output gate hint

Selected reasoning skill prompts は常に入れるのではなく、Reasoning Policy Engine の選択結果に従う。

実装関数例:

```python
build_prompt(
  base_prompt: str,
  persona_prompt: str,
  recall_pack: str,
  task_prompt: str,
  selected_skills: list[str],
  output_format: str | None,
  gate_hints: list[str]
) -> str
```

## 10. Task Classifier の初期ルール

最初は以下のルールで十分とする。

- 「レビュー」「壊れてない」「仕様として」「実装してよい」「差分」「パッチ」
  - `spec_review` または `code_review`
  - `semi_formal_reasoning`
- 「要約」「まとめて」「圧縮」「短く」
  - `summarization`
  - `deliberate_over_instruction`
- 「どっち」「方針」「設計」「比較」「採用」「決めたい」
  - `architecture_decision` または `product_decision`
  - `town_hall_debate`
- 「最新」「調べて」「本当」「確認」「API」「価格」「法律」「いつ」
  - `factual_check` または `research`
  - `verbalized_confidence`
- 「同じ文体」「この感じ」「このトーン」「100本」「量産」
  - `writing`
  - `reference_class_priming`

複数条件に該当する場合は最大 3 つまでスキルを選択する。

危険度が高い場合は Semi-Formal Reasoning と Verbalized Confidence を優先する。

要約タスクでは Deliberate Over-Instruction を優先する。

## 11. Output Gate の動作

生成後に、Output Gate が以下を検査する。

### 11.1 `confidence_gate`

- 未確認の主張が断言されていないか
- 最新情報・数値・API 仕様・外部依存に注意書きがあるか
- confidence が低い主張が結論に混ざっていないか

### 11.2 `compression_loss_check`

- 要約時に前提条件が消えていないか
- 例外ケースが消えていないか
- 反対意見・代替案が消えていないか
- 未確認事項が消えていないか

### 11.3 `decision_gate`

- 結論があるか
- 最大リスクがあるか
- 代替案があるか
- 未確認事項があるか
- 次アクションがあるか

### 11.4 `review_gate`

- 判断根拠があるか
- 影響範囲があるか
- テスト不足があるか
- 未確認事項があるか

`fail_action` が `revise` の場合は、1 回だけ再生成または修正する。無限ループは禁止する。

## 12. ログ仕様

すべてのスキル選択とゲート結果をログに残す。

ログ項目:

```json
{
  "event_id": "evt_xxx",
  "task_type": "spec_review",
  "risk_flags": ["compatibility", "architecture"],
  "selected_skills": [
    {
      "skill_name": "semi_formal_reasoning",
      "reason": "spec_review with architecture risk"
    }
  ],
  "gates_applied": ["review_gate"],
  "gate_result": "passed | revised | failed",
  "created_at": "ISO8601"
}
```

## 13. テスト要件

最低限、以下のテストを作る。

1. 要約タスクで `deliberate_over_instruction` が選択される
2. コードレビューで `semi_formal_reasoning` が選択される
3. 最新情報確認で `verbalized_confidence` が選択される
4. 設計判断で `town_hall_debate` が選択される
5. 文体統一タスクで `reference_class_priming` が選択される
6. 複数条件に該当する場合、最大 3 つまでに制限される
7. 通常雑談ではスキルが過剰選択されない
8. `compression_loss_check` が `required_fields` 欠落を検出する
9. `decision_gate` が `conclusion` / `risks` / `unknowns` 欠落を検出する
10. Prompt Builder が選択スキルだけを差し込む

## 14. 実装時の禁止事項

- 5 つの思考スキルをすべて常時プロンプトに入れない。
- System Prompt を肥大化させない。
- Coder が破壊的操作を直接実行しない。
- reasoning trace をそのままユーザーに出さない。
- confidence の数値を真実として扱わない。
- Town Hall Debate を単純多数決にしない。
- Reference Class の内容をコピーしない。
- 要約時に前提・例外・反対意見・未確認事項を消さない。
- 外部情報が必要な場面で、未確認のまま断言しない。

## 15. 完了条件

以下を満たしたら完了とする。

- `configs/reasoning_policy.yaml` が存在する
- `configs/output_gates.yaml` が存在する
- 5 つの reasoning skill md が存在する
- Task Classifier が動作する
- Policy Engine がスキルを選択できる
- Prompt Builder が選択スキルだけを差し込める
- Output Gate が生成結果を検査できる
- EventId 付きでログが残る
- テストが通る
- README または docs に使い方が記載されている
