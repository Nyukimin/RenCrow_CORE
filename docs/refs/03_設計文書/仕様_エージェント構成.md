# RenCraw 分散マルチエージェント構成仕様
Version: 0.1
Status: Draft
Author: ルミナ
Target: RenCraw

---

## 1. 目的

本仕様は、RenCraw を複数の物理機体上で分散動作させる際の、
エージェント責務、コア機能の所在、自己拡張の起点、コード管理の正本を
誤解なく定義することを目的とする。

本仕様で最も重要な原則は以下の3点である。

1. Chat と Worker の責務を混ぜない
2. 自己拡張の意思決定権は Worker 側に置く
3. 新コードの正本管理は Coder ではなく Worker 側に置く

---

## 2. 用語定義

### 2.1 Chat
ユーザーとの対話窓口を担うエージェント。
主に以下を担当する。

- ユーザー入力の受理
- 応答生成
- 記録保存
- 音声入出力との連携
- 承認要求の提示
- ユーザーへの最終説明

Chat は「対人インターフェース」であり、
システム全体の制御正本を持たない。

### 2.2 Worker
システム全体の運営判断を担うエージェント。
主に以下を担当する。

- タスク実行可否の判断
- 既存能力で足りるかの判定
- 自己拡張の必要性判断
- Coder への実装依頼
- 結果の集約
- system up 実行判断
- 能力管理
- ジョブ管理

Worker は「システム運営主体」である。

### 2.3 Worker Core
Worker の中でも、LLM ではなく決定論的な常駐プロセスとして実装される中核制御面。
本仕様におけるコア機能の正本は、Worker Core が保持する。

### 2.4 Coder
Worker から渡された実装要求に対して、
plan と patch を生成する実装専用エージェント。

Coder は提案者であり、採用決定者ではない。

### 2.5 自己拡張
Worker が「既存能力では不十分」と判断した場合に、
Coder に新規コードまたは改修コードを生成させ、
承認・統合を経てシステム能力を増やすこと。

### 2.6 system up
新規または更新されたコードを、
RenCraw の正規能力として統合し、
次回以降の実行で利用可能な状態へ上げること。

単なる再起動を意味しない。

### 2.7 Capability Registry
RenCraw が現在どの能力を持っているか、
どの能力が未実装か、
どの能力が暫定かを管理する台帳。

この正本は Worker Core が管理する。

## 3. 基本原則

### 3.1 Chat と Worker は責務分離する
Chat と Worker は同一責務として扱ってはならない。

- Chat は対人窓口
- Worker はシステム運営

同一機体に同居してもよいが、
論理的には別コンポーネントとする。

### 3.2 自己拡張は Worker に付随する
自己拡張の起点は Chat ではなく Worker である。
Coder は自己拡張の意思決定を行わない。

### 3.3 Coder はコード作者であって管理者ではない
Coder は plan と patch を生成する。
新コードの正本管理、採用判断、統合判断は Worker Core が行う。

### 3.4 コア機能は LLM ではなく Worker Core が持つ
ジョブ管理、状態管理、承認状態、能力台帳、system up 管理は
LLM の会話状態に依存させず、
Worker Core の永続状態として保持する。

## 4. 物理構成

### 4.1 Ubuntu 機
役割:
- Worker Core 正本
- ジョブ管理
- Capability Registry
- system up 管理
- Heartbeat 集約
- Git / Workspace 正本
- 採用済みコードの統合先

この機体は、RenCraw の制御正本を保持する。

### 4.2 GTX1050 Win11 機
役割:
- Chat ノード
- Mio 用軽量 LLM
- SBV2
- 会話ログ保存
- 記録管理
- 日次ノート保存
- メモリDB / インデックス保管
- 音声キャッシュ保管

この機体は「対話と記録の拠点」であり、
Worker Core の正本を持たない。

### 4.3 RTX4060Ti Win11 機
役割:
- 高性能推論ノード
- Mio 用高性能 LLM
- Shiro 用高性能 LLM
- 必要時の推論オフロード先

この機体は計算資源ノードであり、
ジョブ正本や能力正本を持たない。

### 4.4 Coder ノード
Coder1〜3 は物理的に別機体に分散配置してよい。
ただし、いずれも「提案専用ノード」であり、
採用済みコードの正本管理は行わない。

## 6. 各エージェントの責務

## 6.1 Chat の責務

Chat は以下のみを担当する。

- ユーザー入力の受理
- 入力の対話的解釈
- Worker への問い合わせ
- Worker からの結果をユーザー向けに整形
- 承認要求の提示
- 記録保存
- 音声入出力

Chat は以下を担当しない。

- 自己拡張の必要性判断
- 新コードの採用判断
- system up 実行判断
- Capability Registry 更新

## 6.2 Worker の責務

Worker は以下を担当する。

- 現タスクが既存能力で処理可能か判定
- 不足能力の検知
- 不足が一時的か恒常的かの判定
- 自己拡張の必要性判断
- BuildRequest の生成
- Coder 選定
- 提案比較
- テスト・検証
- 統合判断
- system up 実行
- Capability Registry 更新

Worker は以下を担当しない。

- ユーザー向け自然な対話窓口
- 音声対話演出
- 最終表現の感触調整

## 6.3 Coder の責務

Coder は以下を担当する。

- BuildRequest の理解
- 実装方針の plan 生成
- patch 生成
- 必要に応じた設計説明
- 必要に応じた複数案提示

Coder は以下を担当しない。

- 自己拡張の必要性判断
- 正本ブランチ管理
- 本番統合判断
- Capability Registry 更新
- system up 実行

## 7. 自己拡張フロー

### 7.1 起点
1. Chat がユーザー要求を受ける
2. Chat が Worker に処理委譲する

### 7.2 不足判定
3. Worker が既存能力で処理可能か判定する
4. 処理不可の場合、Worker は Capability Gap を生成する

### 7.3 実装要求
5. Worker は BuildRequest を生成する
6. Worker は適切な Coder に BuildRequest を送る

### 7.4 提案生成
7. Coder は plan と patch を返す
8. 必要に応じて複数 Coder の案を比較する

### 7.5 統合判定
9. Worker は提案を評価する
10. 必要ならテストを行う
11. 必要なら Chat を経由してユーザー承認を取る

### 7.6 system up
12. Worker Core が採用を決定する
13. Worker Core がコードを統合する
14. Worker Core が Capability Registry を更新する
15. Worker Core が新能力を有効化する

### 7.7 ユーザー通知
16. Chat が結果をユーザーへ説明する

## 8. コード管理の原則

### 8.1 正本管理者
採用済みコードの正本管理者は Worker Core とする。

### 8.2 Coder の成果物
Coder の成果物は「候補コード」であり、
正本ではない。

### 8.3 Git 管理
Git の正本リポジトリは Worker Core 側に置く。
Coder は patch 提案または作業用ブランチ生成までとする。

### 8.4 反映権限
main 相当の安定系へ反映する権限は Worker Core が持つ。

### 8.5 記録
以下は必ず保存する。

- BuildRequest
- Coder plan
- Coder patch
- テスト結果
- 採用可否
- 承認履歴
- system up 実施履歴
- Capability Registry 更新履歴

## 10. Heartbeat と障害分離

### 10.1 Chat 障害
Chat ノードが停止しても、
Worker Core のジョブ正本と能力正本は失われない。

### 10.2 Coder 障害
Coder ノードが停止しても、
Worker Core は別 Coder へ再依頼できる。

### 10.3 推論ノード障害
高性能推論ノードが停止しても、
Worker Core は最低限の制御状態を維持する。

### 10.4 Worker Core 障害
Worker Core が停止した場合、
RenCraw の制御正本は停止する。
したがって最重要の復旧対象は Worker Core である。

---

## 11. Chat と Worker の関係

### 11.1 同居
Chat と Worker は同じ物理機体に置いてよい。

### 11.2 非同一化
ただし、以下は必ず分離する。

- セッション状態
- 権限
- プロンプト
- 保存対象
- 正本状態
- ジョブ責務

### 11.3 結論
Chat と Worker は「同居可、同一不可」である。

---

## 12. この仕様で固定される判断

1. 自己拡張の起点は Worker である
2. 自己拡張の実装担当は Coder である
3. ユーザー向け説明と承認提示は Chat である
4. コア機能の正本は Worker Core である
5. 採用済みコードの正本管理者は Worker Core である
6. Chat は記録庫になってよいが、制御正本にはならない
7. Coder は強くても、採用権と正本権を持たない
8. Chat と Worker は責務分離を維持する
9. 物理分散しても、論理上の正本は Worker Core に一本化する

## 14. 非目標

本仕様は以下を定義しない。

- BuildRequest の JSON 詳細
- Capability Registry のスキーマ
- Git ブランチ命名規則
- 承認 UI の具体文言
- system up のデプロイ手順詳細

これらは別仕様として定義する。

---

## 15. 最終結論

RenCraw において、
対話は Chat、
運営判断は Worker、
実装は Coder、
正本管理は Worker Core に帰属する。

この分離を崩してはならない。
