# source_preservation.md

## 目的

この文書は、記憶システムとは別に置く「原本保存」の方針を定義する。

ここで扱うのは Memory ではなく Record である。
原本保存の目的は、想起のためではなく、失われないことにある。

---

## 1. 基本方針

原本保存は、会話、報告、作業の一次情報を残すための系である。

ここでは、

- 要約しなくてよい
- すぐに使えなくてもよい
- 人格形成に直接使わなくてよい
- 将来の再解釈に耐えることが大事

と考える。

つまり、原本保存は「思い出すための記憶」ではなく、「失わないための記録」である。

---

## 2. 記憶との違い

### 記憶

- 使うために残す
- 要約される
- 昇格される
- 忘却される
- 少数断片だけが想起される

### 原本保存

- 残すために残す
- まずは原形を保つ
- 昇格の材料になる
- 通常応答では直接使わない
- 必要時に掘り直す

両者は似ているが、役割が違う。

---

## 3. 保存対象

原則として、次を保存対象にする。

### 3.1 会話記録

- ユーザー入力
- Chat の応答
- 中間報告
- 完了報告
- 委譲結果の要約

### 3.2 作業記録

- task payload
- result payload
- Event
- Hook 判定
- 実行コマンド
- 変更ファイル一覧
- 検証結果
- エラー分類

### 3.3 成果物記録

- 生成ファイルのパス
- 由来 EventId
- 生成日時
- 成果物種別
- バージョン

### 3.4 参照記録

- 参照した文書
- 参照した EventId
- 関連リンク
- 外部入力元

---

## 4. 保存しないもの

原本保存であっても、次は原則保存しない。

- モデル内部の生の思考
- 逐語的な内部推論
- 一時キャッシュそのもの
- 再生成可能な巨大中間データ
- 単なるノイズトレース

ここで残したいのは、内部思考ではなく、外に現れた行為と結果である。

---

## 5. 形式の原則

### 5.1 append-only を基本にする

原本保存は、後から上書きして整形する場所ではない。
まず append-only で積み上げる。

### 5.2 EventId で束ねる

会話、委譲、作業、検証、成果物を EventId で束ねる。
これにより、あとから時系列でも、処理単位でも追えるようにする。

### 5.3 原文と要約を混ぜない

原文は原文、要約は要約で分ける。
要約を原本の上書きとして扱わない。

### 5.4 内容と索引を分ける

本文そのものと、検索・参照用の索引を分離する。
原本保存は本文を保持し、索引は別レイヤで持つ。

---

## 6. 原本保存の使い道

使い道は今すぐ固定しなくてよい。
ただし、将来のために次の用途に耐える形で残す。

- 記憶への再抽出
- 判断過程の追跡
- 同種障害の再調査
- 成果物の由来確認
- 会話や作業の振り返り
- 別観点での再要約
- 学習データ候補の抽出

重要なのは、「今すぐ使わないから残さない」ではなく、「あとで違う見方ができるように残す」である。

---

## 7. 原本保存から記憶への受け渡し

原本保存は、直接 Memory ではない。

流れは次の通り。

1. 会話、報告、作業は原本保存に残る
2. Worker が日次・週次で候補抽出する
3. Chat が保存価値を判断する
4. 一部だけが Relationship / Identity / Project / Work Optimization Memory に昇格する

つまり、原本保存は記憶の素材置き場である。

---

## 8. 保存期間の考え方

原本保存は、記憶より長く持つ。
ただし、本当にすべて永久保存する必要はない。

考え方は次の 3 層でよい。

### 長期保持

- 会話本文
- 報告本文
- Event
- result payload
- 成果物参照

### 中期保持

- 詳細なコマンドログ
- 詳細な検証ログ
- 途中の補助記録

### 短期保持

- temp 成果物
- 一時トレース
- 再生成可能な補助データ

ここで大切なのは、思考を全部残すことではなく、あとから意味のある再構成ができる程度に原本を残すことである。

---

## 9. 記憶と原本保存の主従関係

日常応答では、原本保存を直接使わない。
普段は記憶を使う。

- 日常応答は記憶
- 深掘りと検証は原本保存

この主従関係にすることで、通常時は軽く、必要時は深く掘れる。

---

## 10. この設計の一文要約

原本保存は、Mio が普段思い出すための場所ではない。
Mio が思い出し損ねても、あとから掘り返せるように、会話、報告、作業の一次情報を失わず残すための場所である。
