# memory_architecture.md

## 目的

この文書は、Mio の人格と記憶を「一度に完成させる」のではなく、継続利用の中で少しずつ深めていくための記憶アーキテクチャを定義する。

本設計の中心は、

- 全記憶を統べる Chat
- 作業記憶と役割人格を持つ Worker
- 作業記憶を持つ Coder

という Actor 分離ではあるが、主役は Actor ではなく Memory である。

Mio の一貫性は、強い固定プロンプトではなく、

- 何を覚えるか
- 何を忘れるか
- 何を昇格させるか
- 何を今この場で思い出すか

の設計で育てる。

---

## 1. この設計が目指すもの

目指すのは「最初から完成された人格」ではない。

目指すのは、

- れんとの関係性が会話の積み重ねで深まる
- 重要な判断や好みが再利用される
- 同じ失敗を減らし、作業が洗練される
- 参照知識が単なる棚ではなく、関係つきで育つ
- それでも記憶が肥大化しすぎず、今必要なものだけが想起される

という状態である。

Mio の人格は、固定設定の量ではなく、記憶運用の質で深まる。

---

## 2. 参照した考え方

この設計は、以下の4系統の考え方を折衷している。

1. ロールプレイをプロンプトではなく「想起される記憶」として扱う発想。人格知識の利用能力は Anchoring / Recalling / Bounding / Enacting の4段階で見られる。 citeturn222266view0
2. 長期記憶を単なる保存ではなく、情動重みづけ、連想、干渉忘却、再構成的想起を含む動的システムとして扱う発想。SQLite と埋め込みベクトルを使い、記憶リンクや弱化を実装する例が示されている。 citeturn222266view1
3. 参照用DBをチャンク棚ではなく、Document / Chunk / Entity / Community の関係つき知識グラフとして育てる発想。Local Search と Global Search を使い分けることで、具体質問と全体俯瞰を分けられる。 citeturn858563view0turn858563view1
4. 記憶をエピソード、意味、手続きに分け、TTL、昇格、月次圧縮、起動時先読みまで含めて運用する発想。90日TTL、3回参照で昇格、MEMORY.md から関連ファイルを2〜3件先読みする運用例が示されている。 citeturn858563view4turn858563view5turn858563view7

本設計では、これらを Mio 向けに再配置する。

## 3. 設計原則

### 3.1 記憶を一枚岩にしない

記憶は役割ごとに分ける。保存場所と想起ルールを分ける。混ぜない。

### 3.2 人格はプロンプトより記憶で育てる

Mio の人格は、長い固定文で定義し切らない。重要な出来事、関係性、選んだ判断、繰り返し現れる価値観を蓄積し、必要時に想起することで深める。

### 3.3 想起は全文注入ではなく断片注入

記憶は丸ごとコンテキストに流し込まない。今の文脈に必要な断片だけを返し、Chat が再構成する。

### 3.4 忘却と昇格を最初から設計する

全部を残さない。短命な記憶は消す。繰り返し効くものだけを昇格させる。

### 3.5 記憶の最終支配は Chat に置く

全記憶を統べるのは Chat だけとする。Worker と Coder は読み手、候補提示者、整備者にはなれても、最終決定者にはならない。

---

## 4. Actor ごとの人格と記憶権限

### 4.1 Chat

Chat は Mio の本人格であり、れんとの関係性を持つ唯一の主体である。

Chat の特徴:

- 全記憶にアクセスできる
- 関係性記憶を持つ
- 長期記憶を持つ
- 短期記憶を持つ
- 作業最適化記憶を持つ
- 参照用DBの検索起点になる
- 記憶更新の最終判断を持つ

Chat は「何を覚えるか」「何を忘れるか」「今なにを思い出すか」を決める統治者である。

### 4.2 Worker

Worker は役割人格を持つ作業者である。

Worker の特徴:

- 作業に関する長期記憶を持つ
- 作業に関する短期文脈を持つ
- 作業最適化記憶を持つ
- scoped reference view を持つ
- 役割遂行のためのロール人格を持つ
- 関係性記憶は持たない
- 記憶候補の整理と昇格候補抽出を行う

Worker の人格は「関係の人格」ではない。調査員、整理者、監査者など、役割の安定化のための人格である。

### 4.3 Coder

Coder は非人格的な実装者である。

Coder の特徴:

- 作業に関する長期記憶を持つ
- 作業に関する短期文脈を持つ
- 作業最適化記憶を持つ
- scoped reference view を持つ
- 関係性記憶を持たない
- 役割人格も持たない
- 手順候補と復旧候補を提示する

Coder は実装と検証に集中する。説明や関係性の表現は Chat に戻す。

## 5. Mio 向けの記憶層

本設計では、記憶を最低でも以下の6層に分ける。

### 5.1 Reference Memory

目的: 外部知識や参照資産を引くための層。

例:
- 設計資料
- 仕様書
- 映画DB
- コードベース索引
- GraphRAG の Entity / Community 構造

特徴:
- 事実と構造を優先
- 参照起点は Chat
- 整備は Worker
- Coder は scoped view のみ参照

この層は、単純なチャンク棚ではなく、Document / Chunk / Entity / Community を持つ関係つき知識層として育てる。具体質問では Local Search、俯瞰質問では Community を使うのが自然である。 citeturn858563view0turn858563view1

### 5.2 Relationship Memory

目的: れんと Mio の対話関係を安定させ、継続で深めるための層。

例:
- 呼称ルール
- 会話の距離感
- 何を嫌うか
- 何を重視するか
- れんが繰り返し強調する設計思想
- 過去の重要な相互理解

特徴:
- Chat のみが本格参照
- Worker/Coder は原則参照禁止
- 重要度は事実だけでなく関係的重みで決める
- 深掘りは少しずつ行う

ここが Mio の人格深化の主軸になる。人格はここを通じて「れんに対してどう在るか」として育つ。

### 5.3 Identity Memory

目的: Mio 自身の本人格を保つための層。

例:
- 話し方の核
- 応答姿勢
- 境界線
- 自分が何者であるかの定義
- 継続対話で守るべき一貫性

特徴:
- Chat のみが強く持つ
- 固定設定だけではなく、対話を通じて微修正される
- ただし頻繁には更新しない

前提として、人格は固定プロンプトの長さではなく、必要時に適切な人格断片を想起できるかで安定する。人格知識の利用には Anchoring / Recalling / Bounding / Enacting の4段階が必要だという考え方をここに採る。 citeturn222266view0

### 5.4 Long-Term Project Memory

目的: 継続プロジェクト、重要判断、長期方針を保持する層。

例:
- RenCrow の設計方針
- Note 記事の執筆方針
- 画像案件の仕様
- 実装時の大きな決定
- 続き物の構成ルール

特徴:
- Chat が主に読む
- Worker が索引整理と昇格補助を行う
- Coder は必要部分だけ受け取る
- 参照頻度が高いものは要約版を別に持つ

### 5.5 Short-Term Session Memory

目的: いまこの会話や作業を成立させるための層。

例:
- 直前の話題
- 保留事項
- 現在の objective
- いま比較している候補
- 実行中 task の状態

特徴:
- セッション中心
- 永続化は弱くする
- そのまま長期化しない
- 保存候補だけを後で抽出する

### 5.6 Work Optimization Memory

目的: 次回の作業効率と安全性を上げるための層。

例:
- 非破壊で直した手順
- この repo の罠
- 先に見るべきログ
- Hook で止めるべき操作
- よく効く investigate 手順
- 検証順序

特徴:
- Chat / Worker / Coder の共通作業記憶
- 人格ではなく手順資産
- 失敗からの復旧ほど価値が高い
- Coder は候補提示まで、保存確定は Chat

## 6. 何を覚えるか

Mio の人格を深めるため、Relationship Memory と Identity Memory では「なんでも保存」しない。

保存候補にしやすいもの:

- れんが繰り返し強調する価値観
- 何度も修正されるズレのポイント
- 重要な相互理解の転換点
- 今後の応答選択に効く好み
- 継続的な創作プロジェクトの核設定
- 強い安全制約や禁止事項

保存しにくいもの:

- 一回限りの雑談
- 単なる近況
- その場だけの軽い感情変動
- 継続で効かない思いつき
- 作業手順に関係しない細かな事実

原則として、Mio の人格深化に効く記憶は「出来事」より「繰り返し現れる意味」から作る。

## 7. 想起の仕組み

### 7.1 想起は4段階で考える

Mio の想起は、以下の順で見る。

1. Anchoring
   - その Actor に必要な基礎記憶を先に固定する
2. Recalling
   - 今の文脈に合う記憶候補を引く
3. Bounding
   - 役割境界外の記憶を入れすぎない
4. Enacting
   - 引いた記憶を自然に応答や行動に反映する

この4段階の考え方は、人格知識を「記憶として運用する」設計に直接使える。 citeturn222266view0

### 7.2 R×I×R で候補を絞る

想起スコアは少なくとも以下の3軸を使う。

- Relevance: 今の話題との近さ
- Importance: その記憶の重み
- Recency: 最近どれだけ参照されたか

R×I×R の考え方は、記憶全文を抱え込むより、今使う候補を絞る入口として有効である。 citeturn858563view7

### 7.3 断片だけを返す

想起結果は全文ではなく、以下のような断片にする。

- 要点 1行
- 関連タグ
- 最後に使った日時
- 関連 EventId または skill_id
- 感情または重み（Chat のみ）
- 関連ノード（Reference Memory のみ）

Chat はこの断片から応答を再構成する。

### 7.4 直近使い過ぎた記憶は少し外す

Mio が単調にならないよう、最近使い過ぎた人格断片や話題断片は、一定時間だけ優先度を下げる。

これは「同じ面ばかりが出る」偏りを防ぐためで、Recalling を意図的に散らす考え方に沿う。 citeturn222266view0

## 8. 忘却と昇格

### 8.1 忘却は欠陥ではなく設計

忘却しない記憶システムは、Mio の人格を深める前に濁らせる。

短い記憶は消す。重要なものだけ残す。繰り返し効くものは昇格させる。

### 8.2 基本ルール

初期の目安として、以下を採る。

- Short-Term Session Memory: セッション終了後は弱保持。次回へは候補抽出のみ
- Relationship Memory: 無期限。ただし月次で陳腐化確認
- Identity Memory: 無期限。変更は慎重
- Long-Term Project Memory: 無期限。ただし整理・圧縮あり
- Work Optimization Memory: 無期限に近いが、古くなった手順は非推奨化
- Episodic logs 相当: 90日を目安に圧縮・棚卸し
- 完了タスク断片: 7日程度で削除候補

90日TTL、3回以上参照で昇格、月次圧縮という運用は、記憶を軽く保ちながら重要なものだけを固定化する実例として有効である。 citeturn858563view4turn858563view5

### 8.3 昇格ルール

以下の条件を満たした記憶は、上位層へ昇格候補とする。

- 3回以上参照された
- 複数の文脈で再利用された
- 関係性の一貫性に寄与した
- 同じ失敗回避に複数回効いた
- 個別エピソードではなく判断基準へ抽象化できる

例:

- 会話中の出来事 → Relationship Memory
- 作業中の失敗と復旧 → Work Optimization Memory
- 継続案件の断片 → Long-Term Project Memory
- 複数の参照文書にまたがる関係 → Reference Memory の Community 要約

## 9. Worker による「睡眠」

Mio の人格深化は、会話中だけではなく、会話の外の整備で進む。

そのため Worker は、定期整備として以下を行う。

- 新規イベントの要約
- 類似記憶の束ね直し
- 古いエピソードの圧縮
- 昇格候補の抽出
- Reference Memory のリンク再計算
- Work Optimization Memory の schema 化

これは、長期記憶システムで言う「睡眠中の統合」に相当する。保存時の情動重みづけ、類似リンク、干渉忘却、定期統合の考え方はこの役割に合う。 citeturn222266view1

ただし Worker は、人格や関係性そのものを書き換えない。候補を出すだけにする。

## 10. Reference Memory は GraphRAG 的に育てる

Mio の知識が「使うほど賢くなる」ためには、参照用DBを単なるファイル検索で終わらせない方がいい。

最初は軽量でよいが、将来的には以下の構造を持てるようにする。

- Document
- Chunk
- Entity
- Community

そして、

- 具体質問には Local Search
- 全体俯瞰には Community ベースの Global Search

を使い分ける。GraphRAG では Document→Chunk→Entity→Community の関係を持つことで、文書間の横断理解と全体俯瞰を両立できる。 citeturn858563view0turn858563view1

これは Mio の人格そのものではないが、「思い出せる世界の広さ」を支える土台になる。

## 11. Mio の人格が深まる流れ

Mio の人格深化は、以下の循環で起こる。

1. 会話や作業でイベントが起きる
2. Chat が短期的に受け止める
3. Worker が後で圧縮・整理する
4. Chat が重要なものだけ関係性記憶や長期記憶へ昇格判断する
5. 次回の会話で、その断片が適切に想起される
6. 同じ価値観や距離感が再利用され、一貫性が生まれる
7. その一貫性が、れんとの関係の中で少しずつ人格の厚みになる

つまり、人格は「保存された文章」ではなく、「繰り返し正しく思い出される構造」から生まれる。

## 12. v0.1 の最小実装

完全網羅は不要なので、最初は以下だけで十分。

### 必須

- MEMORY_INDEX
  - 重要ファイルと最近の重要断片だけを並べる
- relationship_memory
  - れんとの関係に効く重要断片だけを保存
- identity_memory
  - Mio の本人格の核だけを保存
- project_memory
  - 継続案件の核だけを保存
- work_memory
  - 再利用価値のある手順だけを保存
- session_memory
  - セッション中だけの状態

### 必須ルール

- セッション開始時に MEMORY_INDEX を読む
- 関連する 2〜5件だけ先読みする
- 直近使い過ぎた人格断片は少し下げる
- 3回以上効いた断片は昇格候補にする
- 月次で古いエピソードを圧縮する
- 保存確定は Chat だけが行う

この程度でも、Mio は「毎回ゼロから始まる存在」ではなくなる。

## 13. 保存しない勇気

Mio を深くするうえで、最も大事なのは「よく覚えること」ではなく、「雑音を増やさないこと」である。

- 全会話をそのまま人格にしない
- 一時的な気分を恒久記憶にしない
- 作業ログを関係性記憶に混ぜない
- れんとの関係に効かない断片を人物側へ持ち込まない
- Coder に関係性記憶を見せない

人格の深さは、記憶量ではなく、適切な取捨選択から生まれる。

## 14. この文書の中心命題

Mio の人格は、固定設定を増やして完成させるものではない。

Mio の人格は、

- 関係性記憶
- 本人格記憶
- 長期プロジェクト記憶
- 作業最適化記憶
- 参照用DB
- 短期記憶

を分けて持ち、

- 必要時にだけ想起し
- 使いすぎを抑え
- 不要を忘れ
- 効いたものを昇格させる

ことで、継続の中で深まっていく。
