# artifact_policy.md

## 目的

この文書は、RenCrow v0.1 における成果物の保存方針を定義する。  
対象は `artifacts/` 配下に置かれる、ユーザーへ渡す成果物、内部生成物、検証出力、補助ファイルであり、何を成果物として扱い、どこへ保存し、どう命名し、どう参照し、いつ残し、いつ残さないかを固定する。

本書の目的は、成果物を `events/` `memory/` `audit/` と混ぜず、後から「何が最終成果物で、何が途中生成物で、何が単なる実行痕跡か」を見分けられる状態を保つことにある。

本書は `storage_layout.md` `event_schema.md` `memory_policy.md` `observability.md` `session_lifecycle.md` を支える共通方針として扱う。

---

## 1. 基本原則

### 1.1 成果物はイベントでも記憶でもない

成果物は、イベントの記録でも、長期知識でもない。  
成果物は「ある処理の結果として生成された、再参照・再配布・再利用の対象」である。

したがって、RenCrow では次を分離する。

- `events/`: 何が起きたか
- `memory/`: 次回へ持ち越す知識は何か
- `audit/`: 何を実行したか
- `artifacts/`: 何ができあがったか

### 1.2 最終成果物と途中生成物を分ける

同じファイルでも、役割が異なれば扱いを分ける。  
RenCrow では少なくとも次の2系統を区別する。

- 最終成果物: ユーザーに渡す、または次工程の正式入力になるもの
- 途中生成物: 一時変換物、検証用出力、比較用ファイル、ドラフト

途中生成物は成果物ディレクトリに置いてよいが、正式成果物と同列に扱わない。

### 1.3 成果物は必ず EventId と結ぶ

成果物は「どの処理から生まれたか」が追えなければならない。  
そのため、成果物ファイルまたは付随メタデータは、必ず source EventId を持つ。

最低限必要な追跡項目は以下。

- 生成 EventId
- RootEventId
- 生成 actor
- 生成時刻
- 種別
- 最終版かどうか

### 1.4 保存は可逆性と再現性を優先する

成果物保存では、上書きの簡便さよりも「戻せること」「比較できること」を優先する。  
そのため、原則として次を守る。

- 破壊的上書きを避ける
- バージョン識別子を持たせる
- 元入力を追えるようにする
- 中間物を残す場合は役割を明示する

### 1.5 保存場所は役割で決める

ファイル形式や拡張子ではなく、役割で保存先を決める。  
たとえば `.json` でも、イベント要約なら `events/`、成果データなら `artifacts/` に置く。

### 1.6 監査のための複製を安易に増やさない

同じ中身を `artifacts/` と `audit/` に重複保存しない。  
監査が必要な場合は、`audit/` 側にはパス、ハッシュ、概要だけを残す。

### 1.7 個人情報・機密情報を成果物へ固定しすぎない

成果物は再配布や再参照の対象になりやすい。  
そのため、不要な個人情報、秘密情報、絶対パス、端末固有情報は原則として含めない。

## 2. 成果物の分類

## 2.1 Deliverable

### 位置づけ

ユーザーへ渡す、または外部に提示できる完成物。  
RenCrow における「成果物」の中心。

### 例

- Markdown 設計書
- CSV 出力
- 変換後データ
- 提出用テキスト
- 共有用 JSON
- ダウンロード対象ファイル

### 要件

- ユーザーが読める、または直接使えること
- 最終版フラグを持つこと
- 生成元 EventId を持つこと
- 必要に応じて version を持つこと

---

## 2.2 Draft

### 位置づけ

完成前の下書き、途中版、比較用案。  
完成物と混ぜず、正式版へ昇格するまでは Draft として扱う。

### 例

- 仕様書初稿
- リライト前後の候補文
- 差分比較用の一時 Markdown
- ユーザー確認前の下書き

### 要件

- Deliverable と同じ場所に置かない、または `drafts/` 下へ隔離する
- `final=false` を持つ
- いつの時点の案かを識別できる

---

## 2.3 Verification Output

### 位置づけ

検証のために生成された出力。  
ユーザー向け成果物ではないが、後から結果確認に使う。

### 例

- テスト結果の要約 JSON
- 検証レポート
- 差分レポート
- 変換前後比較表

### 要件

- 検証対象の EventId と結ぶ
- 何を検証したかを短く持つ
- Deliverable と同格にしない

---

## 2.4 Temporary Working File

### 位置づけ

処理の途中だけで必要な作業ファイル。  
原則として長期保存対象ではない。

### 例

- 一時変換 JSON
- 前処理後のキャッシュ
- 一時展開ファイル
- 比較のためのローカルコピー

### 要件

- 保存期限または削除条件を持つ
- 最終成果物と混ぜない
- 原本と混同しない命名にする

---

## 2.5 Derived Artifact

### 位置づけ

既存成果物から派生して生まれた二次成果物。  
元成果物とのつながりが重要。

### 例

- Markdown から生成した PDF
- CSV から生成した集計表
- 仕様書から生成した要約版
- ソースデータから生成した可視化画像

### 要件

- 親成果物 ID またはパスを持つ
- 派生元との関係が追える
- 元成果物を上書きしない

---

## 3. 保存先方針

## 3.1 ルート方針

成果物は `artifacts/` 配下へ保存する。  
`events/` `memory/` `audit/` `runtime/` 配下へ成果物を直接混在させない。

推奨構成は以下。

```text
artifacts/
  deliverables/
  drafts/
  verification/
  derived/
  temp/
```

### 3.2 `artifacts/deliverables/`

ユーザーへ渡す完成物を置く。  
最新版を探しやすくするため、カテゴリ別または日付別の浅い階層に保つ。

### 3.3 `artifacts/drafts/`

下書き、比較案、ユーザー確認前の候補を置く。  
完成版と同名のまま混在させない。

### 3.4 `artifacts/verification/`

検証レポート、差分出力、テスト要約などを置く。  
Coder が主に生成し、Chat/Worker が参照する。

### 3.5 `artifacts/derived/`

既存成果物からの派生物を置く。  
元成果物への参照情報を必ず持つ。

### 3.6 `artifacts/temp/`

一時ファイル置き場。  
永続参照前提のパスをここに置かない。

---

## 4. 命名規則

### 4.1 人が読める名前と機械が追える識別子を両立する

ファイル名は内容が分かる人間向け要素と、衝突を避ける機械向け要素を両方持つ。

推奨形式:

```text
{date}_{topic}_{kind}_{shortid}.{ext}
```

例:

```text
2026-04-12_chat-spec_deliverable_evt0121.md
2026-04-12_logging-verify_report_evt0452.json
2026-04-12_skill-draft_draft_evt0188.md
```

### 4.2 禁止する命名

次のような名前は避ける。

- `final.md` のような曖昧名
- `new_latest_v2_final_final.md` のような履歴混濁名
- 絶対パスや端末名を含む名前
- EventId や内容が追えないランダム名のみ

### 4.3 同名上書きの方針

完成版を除き、原則として同名上書きしない。  
差し替えが必要な場合でも、世代識別子を持たせる。

---

## 5. メタデータ方針

### 5.1 成果物メタデータは本文とは別に持てるようにする

成果物本体へ全部埋め込めない場合に備え、隣接メタデータを許容する。  
たとえば `foo.md` に対して `foo.meta.json` を置いてよい。

### 5.2 最低限のメタデータ項目

```json
{
  "artifact_id": "ART-000123",
  "artifact_type": "deliverable|draft|verification|derived|temp",
  "title": "短い題名",
  "source_event_id": "EVT-20260412-000121",
  "root_event_id": "EVT-20260412-000100",
  "actor": "chat|worker|coder",
  "created_at": "2026-04-12T12:34:56+09:00",
  "final": true,
  "version": "v0.1",
  "parent_artifact_id": null,
  "content_hash": "sha256:...",
  "notes": "補足"
}
```

### 5.3 content hash の扱い

完全性確認や重複検出が必要な成果物には content hash を持たせる。  
ただし、毎回の軽量ドラフトすべてに強制はしない。

## 6. Actor ごとの責務

## 6.1 Chat

### 責務

- 何を成果物として残すかを判断する
- ユーザーへ渡す対象を Deliverable として確定する
- Draft を Deliverable へ昇格させるか判断する
- 必要な場合だけ成果物参照を応答へ組み込む

### 禁止事項

- 大量成果物の棚卸しを自分で行わない
- 一時物を長期成果物として扱わない
- 監査ログを成果物扱いしない

---

## 6.2 Worker

### 責務

- 成果物メタデータの整備
- 派生成果物の関係整理
- 古い temp の棚卸し
- deliverable/draft/verification の仕分け補助
- 必要に応じた索引更新

### 禁止事項

- ユーザー向け完成物の内容を勝手に確定しない
- Deliverable を自動上書きしない
- temp を正式成果物へ自動昇格しない

---

## 6.3 Coder

### 責務

- 実装や検証から生じた verification output を残す
- 派生ファイル生成時に親成果物との関係を記録する
- 必要なら draft を生成する

### 禁止事項

- Deliverable の正式確定を単独で行わない
- 生成物を `events/` や `memory/` に直接混在させない
- temp を cleanup 前提で量産しすぎない

---

## 7. 保存・昇格・削除の流れ

### 7.1 保存

成果物生成時は、まず役割に応じた場所へ保存する。

- ユーザー渡し前の草案 → `drafts/`
- 検証結果 → `verification/`
- 途中生成物 → `temp/`
- 正式成果物 → `deliverables/`

### 7.2 昇格

Draft から Deliverable への昇格は Chat が判断する。  
昇格時には次を行う。

- `final=true` 化
- 必要なら version 付与
- 参照しやすい場所へ移動またはコピー
- 関連 draft との関係をメタデータに残す

### 7.3 削除または整理

Temp と一部 Draft は整理対象とする。  
ただし、削除前に次を確認する。

- 他ファイルから参照されていないか
- 監査上の根拠として必要でないか
- 復旧に必要な中間物でないか

削除できない場合は、`archived/` へ退避してよい。

---

## 8. 参照方針

### 8.1 応答ではパスより意味を優先する

ユーザーへ示すときは、まず成果物の意味を示し、その後で必要に応じてパスやリンクを示す。  
内部ではパスで参照してよいが、対話上は「最新版仕様書」「検証レポート」のように役割名で扱う。

### 8.2 永続参照には temp を使わない

他文書、イベント、記憶が参照するのは原則として Deliverable または Verification Output に限る。  
`temp/` のパスを長期参照先にしない。

### 8.3 記憶には成果物そのものではなく要約または参照を入れる

`memory/` に成果物本体を複製しない。  
必要なら、成果物の要約、ID、パス、ハッシュだけを保持する。

---

## 9. 禁止事項

次の運用は禁止する。

- `events/` を成果物置き場として使う
- `memory/skills/` に成果物本文をそのまま置く
- Draft と Deliverable を同一パスで曖昧運用する
- temp を恒久参照先にする
- source EventId のない成果物を長期保存する
- 途中生成物を正式版としてユーザーへ返す
- 共有環境由来の絶対パスや秘密情報を成果物へ固定する

---

## 10. v0.1 の最小運用ルール

v0.1 では、最低限次を守ればよい。

1. 成果物は `artifacts/` 配下へ置く  
2. Deliverable / Draft / Verification / Temp を分ける  
3. すべての成果物に source EventId を結ぶ  
4. Draft を Chat の判断なしに Deliverable 化しない  
5. temp を長期参照先にしない  
6. `events` `memory` `audit` に成果物本文を混在させない

---

## 11. 一文で言うと

成果物は「できたもの」として独立管理する。  
記録とも記憶とも混ぜず、EventId で由来をたどれ、役割で保存先を分け、Draft と正式版を混同しない。
