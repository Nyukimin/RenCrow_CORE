# 作成するドキュメント名

coding_agent_modes.md

# ドキュメントの目的

RenCrowにおけるコーディングAIを、以下の2形態に分けて運用する。

1. Safe Build Mode
   壊さずに変更を積み上げるモード。
   既存システム、既存コード、既存DB、既存運用を守りながら、小さな差分で安全に改修する。

2. Tool Build Mode
   れんが作ってほしいツールを作るモード。
   新規ツール、小物スクリプト、補助アプリ、検証用プロトタイプなどを素早く作る。

この2形態の中で、Worker と Coder が役割分担する。

Worker は、考える・調べる・設計する・検証する・判断する役割。
Coder は、読む・書く・直す・テストする・差分を出す役割。

# 前提

RenCrowは、ローカルLLM、LangGraph、記憶システム、staging / validator、Knowledge DB、News DB、Worker / Coder などを持つ構想である。

本ドキュメントでは、コーディング支援エージェントを単なる「コードを書くAI」としてではなく、以下の2種類に分ける。

- 既存システムを壊さずに育てるAI
- れんの依頼に応じて新しい道具を作るAI

## 3. Safe Build Mode

Safe Build Mode は、既存システムを壊さずに変更を積み上げるためのモードです。

対象例を含めてください。

- RenCrow本体
- Chat / Worker / Coder
- LLMサーバ
- 記憶システム
- staging / validator
- Source Registry
- 既存DB
- 既存ログ
- 既存設定ファイル
- PowerShell運用
- 既存の実行環境

Safe Build Mode の基本手順を定義してください。

1. まず読む
2. 影響範囲を確認する
3. 既存ルール・記憶・過去失敗例を確認する
4. 変更方針を出す
5. 小さく直す
6. テストする
7. ログを残す
8. 記憶候補を作る
9. validator を通す
10. 採用判断を待つ

Safe Build Mode では、勝手な大改造、環境変更、DB書き込み、Source Registry追加、記憶確定を禁止してください。

## 4. Tool Build Mode

Tool Build Mode は、れんが作ってほしい道具を作るためのモードです。

対象例を含めてください。

- ログビューア
- Markdown変換ツール
- JSONL validator
- RSS取得スクリプト
- HTML Viewer
- 小さいFastAPIサーバ
- PowerShell運用ツール
- CSV整形ツール
- 実験用CLI
- RenCrow補助ツール

Tool Build Mode の基本手順を定義してください。

1. 目的を確認する
2. 最小仕様を決める
3. 入力と出力を決める
4. 新規ディレクトリで作る
5. 動くものを作る
6. サンプル入力で確認する
7. テストを書く
8. READMEを書く
9. 必要ならRenCrow連携口を作る

Tool Build Mode では、既存本体への直接接続を急がず、まず独立した道具として完成させる方針にしてください。

## 5. Worker / Coder の役割分担

次の構造を明示してください。

Safe Build Mode
- Worker: 調査・設計・リスク確認・影響範囲確認・テスト観点作成
- Coder: 小差分実装・テスト・ログ確認・報告

Tool Build Mode
- Worker: 要望整理・仕様化・最小設計・技術選定・README構成
- Coder: 新規実装・サンプル作成・テスト作成・README作成

表形式で整理してください。

## 6. モード切り替え条件

以下のような判定ルールを作ってください。

Safe Build Mode にする条件:
- 既存コードを変更する
- 既存DBに触る
- 既存運用に影響する
- 既存環境を変更する
- 本番系設定に触る
- 過去に壊した領域に触る
- 記憶・Source Registry・validatorに関係する
- 失敗時の影響範囲が大きい

Tool Build Mode にする条件:
- 新しい小物ツールを作る
- 既存本体にまだ接続しない
- sandbox / tools / experiments 配下で作れる
- 入出力が明確
- 失敗しても既存環境に影響しない
- 単体で動作確認できる

判定に迷う場合は Safe Build Mode を選ぶ、というルールを入れてください。

## 7. ガードレール

以下を必ず含めてください。

禁止事項:
- official DB への直接write
- user:<uid> への直接upsert
- memory の直接確定
- Source Registry への無審査追加
- 検索結果の自動大量投入
- 共有環境の無断変更
- venv / CUDA / モデル配置の無断変更
- 既存ファイルの大規模削除
- 不要なリネームや移動
- Move-Item や rm -rf のような破壊的操作の無断実行
- テストなしで完了報告すること
- 3回失敗しても同じ方針で続けること

許可される範囲:
- staging/ への出力
- reports/ への出力
- logs/ への出力
- tools/ や experiments/ 配下での新規作成
- Pull Requestまたは差分案の作成
- validatorテスト
- source候補の提案
- memory candidate の作成

## 8. 記憶の使い方

コーディング用記憶を、以下の型に分けて定義してください。

- coding_rule
  常に守る開発ルール。

- repo_memory
  リポジトリ固有の構造や重要ファイル。

- failure_pattern
  過去に失敗した作業パターン。

- accepted_pattern
  成功した実装パターン。

- environment_constraint
  OS、venv、モデル配置、PowerShell運用などの制約。

- command_history
  実行したコマンドと結果。

- previous_diff
  直近の差分。

- known_risk
  触ると壊れやすい場所。

- worklog
  その日の作業記録。

Safe Build Mode では、これらを強く参照すること。
Tool Build Mode では、必要最小限として preferred_stack、tool_template、recent_tool_patterns を中心に参照すること。

## 9. 記憶の昇格ルール

以下の段階を定義してください。

1. observed
   その場で観測しただけの情報。

2. candidate
   繰り返し使えそうな記憶候補。

3. confirmed
   validator または れん の確認を通った正式記憶。

4. pinned
   常に守るべき固定ルール。

コーディングAIは、memory candidate を作ることはできるが、confirmed や pinned に直接昇格してはいけない、というルールを明記してください。

## 11. 失敗時のルール

以下を含めてください。

- 1回目の失敗: エラーを読む
- 2回目の失敗: 仮説を変える
- 3回目の失敗: 前提を見直し、Workerへ戻す
- それ以上は自動継続しない
- 失敗ログを failure_pattern 候補にする
- 破壊的な回避策を勝手に使わない

## 12. Coderへの指示テンプレート

Safe Build Mode 用と Tool Build Mode 用の2種類を作ってください。

Safe Build Mode 用テンプレートには以下を含める。

- まず関連ファイルを読む
- 影響範囲を出す
- 変更案を小さくする
- 既存挙動を壊さない
- テストを実行する
- 差分を報告する
- 記憶候補を出す

Tool Build Mode 用テンプレートには以下を含める。

- 目的を満たす最小ツールを作る
- 新規ディレクトリで作る
- READMEを付ける
- サンプル入力と実行例を付ける
- テストを付ける
- 既存RenCrow連携は後付け可能にする

## 14. 最終まとめ

最後に、以下の思想で締めてください。

- Safe Build Mode は、既存システムを守りながら育てるためのモード。
- Tool Build Mode は、れんの発想をすばやく道具にするためのモード。
- Worker / Coder は、それぞれのモード内で役割分担する。
- RenCrowのコーディングAIは、「コードを書くAI」ではなく、「安全に変更を積み上げるAI」と「道具を形にするAI」の二本立てで運用する。
