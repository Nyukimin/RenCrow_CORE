# RenCrow 物理分離前提移行仕様

## 1. 目的
本書は、`docs/10_新仕様/43_物理分離前提実装仕様.md` を前提に、RenCrow の既存コードをどの順で、どの責務ごとに、どのファイル単位で移行するかを定める移行仕様である。  
実装仕様が「何をどこへ置くか」を定めるのに対し、本書は「どの順で、どの単位で、どの確認を挟みながら動かすか」を定める。

本書の正本は次である。

- `42_物理分離前提仕様.md`
- `43_物理分離前提実装仕様.md`

## 2. 使うタイミング
この文書は次のタイミングで使う。

1. 仕様書としては、実装仕様 `43` の直後
2. 実装着手前の移行順整理
3. 既存コードを動かしながら契約を切り出すとき
4. repo 分割の前に、ローカル単一 repo のまま境界を固定するとき

### 役割分担
- `42` は物理分離の思想整理
- `43` は実装仕様
- `44` は移行実行順

`44` は設計書ではなく、実行計画である。

## 3. 移行の基本方針
### 方針
- 先に契約を切る
- 次に依存を反転する
- 最後に実装を整理する
- 一度に1責務だけ動かす
- 1 commit に複数モジュールの責務を混ぜない

### 禁止
- 一括リファクタリング
- 仕様と無関係な cleanup
- 実装と docs を同じ commit で混ぜること
- `cmd/rencrow` にルールを残したまま移行完了扱いにすること

## 4. 移行順
移行は次の順序で進める。

| 順位 | 対象 | 目的 |
| ---: | --- | --- |
| 1 | `modules/core` | 共通契約を先に固定する |
| 2 | `modules/llm` | provider / role / health 契約を切り出す |
| 3 | `modules/tts` | 合成・再生・音声ルールを切り出す |
| 4 | `modules/stt` | 認識・入力ルールを切り出す |
| 5 | `modules/chat` | ルーティング・IdleChat・記憶呼び出しを切り出す |
| 6 | `modules/worker` | 実行・Coder slot・拡張探索の契約を切り出す |
| 7 | `cmd/rencrow` | 配線だけに縮退する |
| 8 | `internal/application/*` | glue を最小化する |
| 9 | 記憶・heartbeat・感情表出 | 共通 state と policy を固める |
| 10 | App/viewer/IdleChat | 外側アプリとして整理する |

## 5. 移行フェーズ
### Phase 0: 事前固定
目的: 参照元と禁止事項を固定する。  
やること:

- `42` と `43` を再読する
- `modules/CURRENT_MAP.md` と `modules/DEPENDENCY_RULES.md` を確認する
- 変更対象を 1 責務に限定する

完了条件:

- 変更対象の責務が1つに絞られている
- 依存方向が説明できる

### Phase 1: `modules/core`
目的: すべてのモジュールが依存できる共通契約を固定する。  
対象:

- ID / response / chunk / health / manifest
- 共通 JSON 契約

移行手順:

1. 共通契約を `modules/core` に置く
2. `cmd/rencrow` の共通 helper を削る
3. `modules/*` が core を参照するようにする

確認:

- `modules/core` の unit tests
- 依存方向テスト

### Phase 2: `modules/llm`
目的: LLM の役割とロール計画を正本化する。  
対象:

- local / legacy provider plan
- Chat / Worker / Heavy / Wild
- coder provider plan
- thinking bridge

移行手順:

1. provider plan を `modules/llm` に固定する
2. `cmd/rencrow` の provider selection policy を削る
3. rawlog / health / diagnostics の契約を `modules/llm` に寄せる

確認:

- `modules/llm` unit tests
- `cmd/rencrow` の provider wiring focused tests

### Phase 3: `modules/tts`
目的: 音声合成と再生状態を分離する。  
対象:

- emotion prefix / voice mapping
- playback state / pending store / public session
- synthesis payload / event payload
- provider selection plan

移行手順:

1. 文字列整形と再生契約を `modules/tts` に寄せる
2. `cmd/rencrow/tts_*` の state policy を削る
3. `internal/infrastructure/tts` は HTTP 実体だけにする

確認:

- `modules/tts` unit tests
- playback ACK / pending / timeout focused tests

### Phase 4: `modules/stt`
目的: 音声認識と入力状態を分離する。  
対象:

- transcription result
- viewer input
- busy policy
- runtime URLs

移行手順:

1. result / payload 契約を `modules/stt` に寄せる
2. `internal/infrastructure/stt` の policy を削る
3. `cmd/rencrow` を viewer / transport wiring に限定する

確認:

- `modules/stt` unit tests
- viewer input focused tests

### Phase 5: `modules/chat`
目的: 会話と IdleChat の正本を固定する。  
対象:

- route decision
- topic generation / judge / prompt / seed
- recent similarity
- forecast provider plan
- viewer facing conversation rules

移行手順:

1. Chat の正本を `modules/chat` に寄せる
2. `internal/application/idlechat` の topic / judge / validation を削る
3. topic / speech contract を `modules/tts` と接続する

確認:

- `modules/chat` unit tests
- IdleChat focused tests

### Phase 6: `modules/worker`
目的: Worker と Coder slot の役割を固定する。  
対象:

- execution contract
- Coder1-4 slot policy
- external explicit-only policy
- failure classification

移行手順:

1. Worker の policy を `modules/worker` に寄せる
2. `cmd/rencrow` の execution policy を削る
3. external provider explicit-only を固定する

確認:

- `modules/worker` unit tests
- worker execution focused tests

### Phase 7: `cmd/rencrow`
目的: composition root だけにする。  
対象:

- runtime wiring
- provider instantiation
- route registration
- env/config injection

移行手順:

1. `cmd/rencrow` の policy を順次削る
2. module contract を受け取って配線するだけにする
3. ルートと health だけを残す

確認:

- route registration tests
- focused runtime wiring tests

### Phase 8: 記憶 / heartbeat / 感情表出 / 拡張
目的: 賢さの中核を別契約にする。  
対象:

- memory entity separation
- recall / ingest / policy
- heartbeat evaluation loop
- drive / emotion expression split
- extension discovery / install / implement / persist

移行手順:

1. memory の正本を固定する
2. heartbeat decision contract を固定する
3. Mio の emotion expression contract を固定する
4. extension discovery contract を固定する

確認:

- 契約テスト
- 依存方向テスト

## 6. ファイル単位の移行ルール
### `cmd/rencrow`
- 残してよいもの: wiring, bootstrap, HTTP registration, config injection
- 移すべきもの: policy, state ownership, role semantics, validation rules

### `internal/application/idlechat`
- 残してよいもの: orchestration glue, source fetching glue, runtime bridging
- 移すべきもの: topic contracts, judge, validation, display/speech policy

### `internal/infrastructure/tts` / `stt`
- 残してよいもの: HTTP / provider execution
- 移すべきもの: request/response semantics, policy, selection, state model

## 7. テスト順
テストは移行フェーズの順に追う。

1. `modules/core`
2. `modules/llm`
3. `modules/tts`
4. `modules/stt`
5. `modules/chat`
6. `modules/worker`
7. `cmd/rencrow` focused tests
8. `go test ./...`

### 原則
- 契約テストを先に書く
- 実装テストは後に追従する
- 1 回の移行で追うテストは最小限にする
- failed / blocked を pass にしない

## 8. commit 単位
1 commit の責務は1つだけにする。

### commit 例
- `fix: modules/core の共通契約を固定する`
- `refactor: modules/llm に provider plan を移す`
- `refactor: IdleChat topic contract を modules/chat に寄せる`

### commit 禁止
- modules と app を混在させる
- docs と実装の責務を混ぜる
- 1 commit で複数モジュールを同時に崩す

## 9. 停止条件
次の場合は止める。

- 依存方向が壊れる
- 1 commit の責務が曖昧になる
- テスト失敗が短時間で解消できない
- 外部実機 / 外部 API / 複数マシンが必要
- `cmd/rencrow` に policy が残っているのに次へ進もうとする

## 10. 受け入れ条件
次を満たしたら移行仕様の役目を果たしたとみなす。

1. 各フェーズに移行先がある
2. 各フェーズに確認テストがある
3. `cmd/rencrow` の責務が配線に収束する
4. `modules/*` が各正本を持つ
5. repo 分割前の境界が再現できる

## 要約
本書は、`43` の実装仕様を実際に動かすための移行順を定める。  
先に `core` と各機能モジュールの契約を固め、最後に `cmd/rencrow` を配線に収束させる。  
各フェーズは1責務・1 commit・1検証を原則に進める。

## 実行単位
この文書は Goal 実行や段階的移行の参照先として使う。  
実装仕様を読んだ後、移行順を決めて着手する時に利用する。
