# 旧仕様未実装候補 実装ロードマップ仕様

## 目的

この文書は、`docs/10_新仕様/` 以外の仕様文書に残る未実装、部分実装、保留、旧仕様候補を、現行 RenCrow に段階的に取り込むための実装仕様である。

`docs/10_新仕様/` を現行仕様入口として扱い、旧 docs の記述はこの文書で次のいずれかへ再分類する。

- 実装対象: 現行 RenCrow に入れる。フェーズ、受け入れ条件、検証方法、影響ファイルを必須とする。
- 部分実装: 主要実装はあるが、残差分がある。残差分のみを実装対象にする。
- 実装済み: production code と docs/10 の根拠がある。根拠ファイルと確認テストを明記する。
- 保留: 採否判断が未完了。採用基準と次の確認先を明記する。
- 旧仕様・移行対象外: 現行 RenCrow へ入れない。理由と隔離先を明記する。

この文書は Go 実装そのものではない。後続実装では、各フェーズの受け入れ条件と検証方法を満たしてから `13_実装項目インベントリ.md` と該当 docs/10 仕様へ反映する。

### Viewer 添付 PDF / text 抽出

| 観点 | 内容 |
| --- | --- |
| 状態 | 部分実装。Viewer upload / attachment 境界と text / 軽量 PDF 本文抽出は実装済み。外部チャネル添付取り込みは LINE / Slack / Discord / Telegram の主要 file payload まで実装済み。OCR と画像 PDF 高精度解析は未実装。 |
| Phase | Phase 1 |
| 現在状態 | `internal/domain/attachment`, `internal/application/attachment`, `internal/adapter/viewer/handler_send.go` に添付境界がある。`ExtractedText` / `ExtractionError` / `ExtractionTruncated` で抽出結果と失敗を分離する。 |
| 実装方針 | PDF / text から抽出した本文を attachment metadata と分け、routing classification と Viewer 表示を混同しない。サイズ、ページ数、MIME、抽出失敗時の error を定義する。 |
| 実装しない範囲 | OCR、画像 PDF 高精度解析、外部クラウド抽出サービス、添付本文の無審査 memory promote。 |
| 影響する責務境界 | Viewer handler は upload を受ける。Application は抽出 / 検証を行う。Domain は attachment contract を持つ。Routing は添付本文を分類材料として読むが、添付処理本体を持たない。 |
| 影響ファイル | `internal/domain/attachment`, `internal/application/attachment`, `internal/adapter/viewer/handler_send.go`, `internal/adapter/viewer/handler_send_attachment_e2e_test.go`。 |

| E2E または手動確認 | Viewer から PDF / text を添付して `/viewer/send` が成功し、event log / history に添付情報と抽出結果が追える。 |

### STT `queue_latest` busy policy

| 観点 | 内容 |
| --- | --- |
| 状態 | 実装済み。STT provider wrapper と config `busy_policy` で `queue_latest` / `reject` / `direct` を扱う。 |
| Phase | Phase 1 / Phase 2 |

| 必要なテスト | 実装済み: `queue_latest` が最新のみ保持し古い pending を `PROVIDER_BUSY` にする unit test。残り: browser STT での live 確認。 |

| 受け入れ条件 | busy 時の挙動が deterministic で、log / Viewer / error response に状態が出る。 |

## docs 更新ルール

- 実装対象を production code に入れたら、同じ PR / commit で `13_実装項目インベントリ.md` の状態を更新する。
- Viewer / IdleChat / STT / Memory / routing の仕様を変えたら、該当する docs/10 ファイルへ仕様を反映する。
- 実装しない判断をした項目は、この文書の `旧仕様・移行対象外` または `判断保留項目の採否基準` へ理由つきで移す。
- archive 文書は一次参照にしない。archive を直接編集しない。
- 旧 docs の「未実装」表記と現行コードが矛盾する場合は、現行コード、docs/10、検証結果を優先してこの文書に根拠を残す。
- Go 実装を伴う後続作業では、先に失敗テストまたは代替検証手順を定義する。

## 完了条件

このロードマップは、次を満たした時点で完了とする。

- 全候補が、実装対象 / 部分実装 / 実装済み / 保留 / 旧仕様・移行対象外のいずれかに分類されている。

- 実装しない項目には理由と代替または隔離先がある。
- 実装済み項目には根拠ファイルと確認テストがある。
- `docs/10_新仕様/14_旧仕様未実装候補.md` 単体で、段階的に実装を進められる。
