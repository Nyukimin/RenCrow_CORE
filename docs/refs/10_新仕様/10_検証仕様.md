# 検証仕様

## 目的

この文書は、新仕様に基づく RenCrow の検証方針を定義する。

テスト通過だけで完了扱いしない。ユーザー観測、Viewer 実表示、音声、口パク、ログ、runtime config を分けて確認する。

## 標準コマンド

通常の広範囲変更後:

```bash
GOCACHE=/tmp/rencrow-gocache go test ./...
```

`cmd/rencrow` に関係する変更:

```bash
GOCACHE=/tmp/rencrow-gocache go test ./cmd/rencrow
```

E2E:

```bash
GOCACHE=/tmp/rencrow-gocache go test -count=1 -tags=e2e ./test/e2e
```

差分確認:

```bash
git diff --check
git diff --stat
```

live health:

```bash
curl -fsS http://127.0.0.1:18790/health
```

## テスト分類

| 分類 | 目的 | 例 |
| --- | --- | --- |
| unit test | domain / application / infrastructure / adapter の局所契約確認 | `go test ./internal/domain/...` |
| integration test | 複数 package の結合確認 | `go test ./test/integration` |
| e2e test | route、Viewer、runtime config、外部境界の確認 | `go test -tags=e2e ./test/e2e` |
| live health | 実起動 service と live runtime の確認 | `/health`, `/viewer/runtime-config` |
| browser E2E | Viewer 表示、操作、DOM、SSE、audio 境界確認 | Playwright 系 E2E |

## live runtime E2E

live service の `/health` と `/viewer/runtime-config` を確認する場合:

```bash
RENCROW_LIVE_E2E=1 \
GOCACHE=/tmp/rencrow-gocache \
go test -count=1 -tags=e2e ./test/e2e \
  -run 'TestE2E_Phase25Live(RuntimeHealth|ViewerRuntimeConfigClient)' -v
```

確認対象:

- `http://127.0.0.1:18790/health`
- `http://127.0.0.1:18790/viewer/runtime-config`
- LLM endpoint
- STT stream URL
- runtime dependency readiness fields
- repo example と live runtime config の差

## browser Viewer E2E

```bash
RENCROW_BROWSER_E2E=1 \
RENCROW_LIVE_BASE_URL=http://127.0.0.1:18790 \
GOCACHE=/tmp/rencrow-gocache \
go test -count=1 -tags=e2e ./test/e2e \
  -run TestE2E_Phase25BrowserViewerSessionContract -v
```

確認対象:

- Viewer が開ける。
- 入力欄に入力できる。
- `/viewer/send` が成功する。
- 表示本文、event log、history が整合する。
- IdleChat、TTS、口パク、ログの DOM 境界が存在する。

## Viewer / IdleChat / Audio の検証

DOM 要素の存在だけで完了扱いしない。

最低 1 session で次を確認する。

- 入力
- route
- response
- SSE event
- Viewer event log
- history / session state
- error / invalid response 表示
- TTS audio trigger
- lipsync trigger
- 終了状態

IdleChat では次を分ける。

- LLM raw response
- 編集後の view data
- diagnostic / test-mode 表示
- TTS chunk
- 口パク trigger
- topic store / history / summary
- fallback / invalid response
- story validator の `valid story` / `invalid_story:<reason>`

## STT / TTS の検証

STT:

- HTTPS / browser permission
- mic capture
- trailing silence
- `/stt-ws` / `/ws` 互換 endpoint
- `busy_policy: queue_latest` では pending が最新 1 件だけ保持され、古い pending が `PROVIDER_BUSY` になること
- `busy_policy: reject` では provider busy 中の新規入力が `PROVIDER_BUSY` になること
- `openai-api` engine が `external_http` と区別できること
- final text が通常 chat に流れること
- IdleChat へ STT input を流さないこと

TTS:

- provider endpoint
- runtime factory
- bridge / audio router
- browser playback
- audio event
- lipsync trigger
- 本文表示と audio chunk を混同していないこと

## Worker execution の検証

確認対象:

- Coder は plan / patch / proposal を生成するだけである。
- Worker が実行主体である。
- `PatchCommand` の parse / validation が壊れていない。
- file / shell / git operation の dispatch が追える。
- protected file pattern が効いている。
- workspace 外操作が拒否される。
- 実行ログに job_id、session_id、route、status、結果が残る。

安全境界の変更では、成功ケースより先に拒否ケースを確認する。

## channel / external input security の検証

確認対象:

- DM 許可。
- group / room 拒否。
- sender allowlist 許可。
- unknown sender 拒否。
- paired group 許可。
- attachment 抽出文の prompt injection warning が拒否とは別の metadata として残る。
- Source Registry 由来テキストの prompt injection warning は、staging `Meta.security_warnings`、Source Registry run API の warning 件数、Viewer 表示で確認する。
- warning 付き外部入力が通常 prompt / memory と混ざらない。

## 外部依存の扱い

次は環境未準備時に skip される場合がある。

- Claude / DeepSeek / OpenAI / Gemini API key
- Google Search API key / CSE
- Ollama endpoint
- local OpenAI compatible endpoint
- STT server
- TTS server
- browser permission / HTTPS
- SSH agent
- VTube Studio / audio device

skip は成功扱いではない。外部依存が未準備である記録として扱う。

Wild distributed route は、live service の設定を変更せず一時 config で distributed mode を別ポート起動して確認できる。完了根拠には、SSH transport 接続、`routing.decision route=WILD`、`agent.response wild -> mio`、同一 `job_id` / `session_id`、local / CHAT fallback がないことを含める。

## ユーザー観測

ユーザーの実機観測を Ground Truth とする。

ユーザー観測とテスト結果、ログ解釈、AI の推測が矛盾した場合は、分析側を疑って調査をやり直す。

## 完了報告に含めるもの

- 更新した仕様ファイル。
- 実行したテストコマンド。
- 成功した検証。
- skip された外部依存。
- live service を見たかどうか。
- browser E2E を実行したかどうか。
- 確認できなかったこと。
- 残る外部依存リスク。
