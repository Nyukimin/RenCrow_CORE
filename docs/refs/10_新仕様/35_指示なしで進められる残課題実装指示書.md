# 指示なしで進められる残課題 実装指示書

## 1. 目的

この文書は、現在の RenCrow / RenCrow の残課題のうち、ユーザーの追加判断を待たずに進められる範囲を Goal 実行向けに整理した実装指示書である。

対象は、主に以下である。

- live runtime の blocked 状態の再確認
- E2E 残課題台帳の再分類
- 外部依存なしの regression 確認
- Viewer / runtime の blocked 証跡確認
- `vault/` の扱い確認

この文書は、外部 API、実 PR 作成、実外部送信、正式環境適用、依存追加、設定変更、再起動を無条件に許可するものではない。

## 3. Goal に設定するプロンプト

Goal に設定する場合は、次を使う。

```text
docs/10_新仕様/35_指示なしで進められる残課題実装指示書.md に沿って、追加判断なしで進められる残課題を順に処理してください。

各作業は、小さな検証済み commit 単位で進めてください。
各単位で、対象・変更範囲・検証コマンド・完了条件を自分で定義してから実装または確認に入ってください。
実装や docs 更新を行った場合は、該当テストと必要な runtime / Viewer 確認を行ってください。
確認済みの関連ファイルだけを選択的に stage し、日本語 commit message で commit してください。
commit 後は push してください。
push できたら、ユーザー確認を待たずに次の未完了項目へ進んでください。

ただし、次の場合だけ停止して報告してください。
- テスト失敗や runtime 確認失敗が、現在の作業範囲内で短時間に解消できない。
- 変更が複数領域に広がり、1 commit の責務が曖昧になった。
- 外部 secret / 実機 / 外部 API / 複数マシンなど、こちらで準備できない依存が必要。
- destructive operation、依存追加、CI/deploy 設定変更、ファイル削除、セーフガード変更が必要。
- RenCrow / RenCrow の再起動、runtime config 変更、一時 config 有効化が必要。
- push 対象に未確認差分、別責務の差分、`vault/`、または live E2E 生成物が混ざりそうになった。
- blocked / skipped / fail を pass に移すには、追加実装、環境準備、実機確認、または外部依存が必要だと分かった。

禁止:
- worktree 全体を一括 stage / commit しない。
- `vault/` や live E2E 生成物を明示指示なく commit しない。
- 失敗したテストを無視して push しない。
- blocked / skipped / fail を成功扱いしない。
- health ok、unit test pass、DOM 存在だけで user flow 成立扱いしない。
- Viewer / domain / persistence / runtime / docs を、責務が曖昧なまま 1 commit に混ぜない。
- 外部送信、実 PR 作成、正式環境適用、依存追加、再起動、設定変更を独断で行わない。

報告:
- 各 push 後に、commit hash、検証コマンド、次に進む対象を短く報告してください。
- 停止条件に当たらない限り、ユーザー確認は不要です。
```

## 4. 作業ルール

### 4.1 進めてよい範囲

この文書で追加判断なしに進めてよいのは、次の範囲である。

- read-only な状態確認
- local regression test の再実行
- live service が既に起動している場合の read-only HTTP 確認
- Viewer の read-only 表示確認
- docs 更新案の作成
- Goal 承認済みの場合の docs 更新 commit / push
- `vault/` の中身・生成元・commit 対象外確認

### 4.2 止める範囲

次は、追加指示なしに実行しない。

- `systemctl --user stop/start/restart rencrow.service`
- `make install`
- `~/.rencrow/config.yaml` の変更
- scheduler / sandbox / L1 / context budget などの一時 config 有効化
- 外部 API / 外部チャネルへの実送信
- GitHub / 外部 repo への実 PR 作成
- 実マイク STT / TTS 音声再生など、実機 device 前提の確認
- `vault/`、tmp screenshot、live E2E 証跡ファイルの commit

### 4.3 判定ルール

- `pass`: 対象フローの成功条件を満たし、証跡が揃っている。
- `fail`: 対象フローが失敗し、原因または失敗箇所を証跡付きで記録した。
- `blocked`: 外部依存、設定、service 状態、Human approval などが不足して進めない。
- `skipped`: 実行条件を満たさず、未確認として残す。

`blocked`、`skipped`、`fail` は成功ではない。docs 更新でも、これらを `pass` に移してはいけない。

## 6. commit / push 単位

Goal として実行し、docs 更新または実装修正が発生した場合は、以下の単位を混ぜない。

| commit 単位 | 含めてよいもの | 含めないもの |
|---|---|---|
| runtime blocked 状態の docs 更新 | `17` / `32` の blocked 証跡追記 | runtime code 変更 |
| local regression 結果の docs 更新 | test コマンド、pass / fail、未検証範囲 | E2E 成功扱いへの変更 |
| Viewer visible-state 修正 | Viewer JS / CSS / handler と対象 test | domain / persistence の横断修正 |
| client validation 修正 | `pkg/rencrowclient` と focused test | runtime route 変更 |
| domain / persistence validation 修正 | 対象 domain または store と focused test | Viewer 表示修正 |
| docs 整理 | 対応する docs のみ | `vault/` / tmp / live E2E artifacts |

commit message は日本語で書く。

例:

```text
docs: 指示なしで進められる残課題の状態を整理する
fix: Viewer の blocked 状態表示を保持する
fix: SuperAgent status の証跡検証を補強する
```
