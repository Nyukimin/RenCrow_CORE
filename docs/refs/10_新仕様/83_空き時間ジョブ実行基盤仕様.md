# 空き時間ジョブ実行基盤仕様

## 1. 目的

本仕様は、RenCrow が「時間がある時にやってほしい作業」を汎用的に登録、実行、監査、再実行できるようにするための基盤を定義する。

対象例:

```text
- ニュース / RSS / Source Registry の定期取り込み
- 映画カタログの backfill
- 映画 topic candidates 生成
- Hobby graph / Domain graph の topic candidates 生成
- Daily intake rule の sweep
- Knowledge Memory / Dream consolidation の提案生成
- Backlog / Workstream / Vault の棚卸し
- 新しく追加される低優先度・非対話型の保守作業
- ユーザーが後から登録する任意の新規作業
```

本仕様では、この仕組みを **Idle Job Scheduler** と呼ぶ。

## 2. 背景

現状の RenCrow には複数の定期実行系が存在する。

```text
Heartbeat
  定期通知・保守処理。

Daily Intake Sweeper
  Knowledge Memory の daily_intake_rule から Source Registry entry を作り、sweep する。

Movie Catalog Backfill
  映画カタログ DB がある場合に、一定間隔で backfill する。

/viewer/scheduler
  job 台帳と run log を持つ。ただし現状は executor 未接続であり、
  run は "scheduler run recorded without executor" として記録される。

SuperAgent run queue scheduler
  Agent run queue を claim して orchestrator に渡す別系統。
```

このため、ユーザー視点では「映画の情報とかニュースとか、暇な時にやってほしい」をどこで登録すべきか分かりにくい。

また、現状の `/viewer/scheduler` は汎用台帳としては使えるが、実処理 executor が未接続のため、ニュース取得や映画 backfill を直接実行できない。

本仕様は、既存の background job / Viewer API / Workstream / SuperAgent を整理し、汎用の Idle Job として登録できる形へ統合する。

## 3. 位置づけ

本仕様は以下に接続する。

```text

```

## 4. 基本方針

Idle Job Scheduler は以下を満たす。

```text
1. 新しい作業種別を plugin 的に追加できる。
2. job 台帳と実 executor を分離する。
3. 実行結果は run log と artifact に残す。
4. 外部送信、公開、課金、DB promotion は Human approval なしに行わない。
5. 低優先度作業は system load / active user session / service health を見て実行を抑制できる。
6. 失敗、skip、blocked、partial を visible terminal outcome として保存する。
7. Viewer から登録、手動実行、無効化、履歴確認できる。
8. 未知の新規作業は `custom_task` として安全な draft / proposal / workstream へ落とし、いきなり外部作用を実行しない。
```

### 4.1 LLM 利用制約

LLM を使わずに完了できる background job は、通常実行では LLM に接続しない。

対象例:

```text
- source_registry_sweep
- memory_lifecycle
- daily_intake_sweep
- parquet_export
- movie_catalog_backfill
```

失敗時だけ `background_job.failed` を `background_job -> shiro` へ出し、Shiro の調査対象にする。
同じ job_id / session_id で `job.notification` を `shiro -> mio` へ出し、Mio がユーザー影響のある失敗だけ報告できるようにする。

この接続はイベント通知であり、background job が成功系で Chat / Worker / LLM を直接起動することを意味しない。

## 5. 用語

### 5.1 Idle Job

空き時間に実行してよい作業単位。

例:

```text
daily_news_intake
movie_catalog_backfill
movie_topic_candidates_generate
hobby_topic_candidates_generate
knowledge_dream_proposal
backlog_inventory
custom_task
```

### 5.2 Job Definition

永続保存される実行定義。

```json
{
  "job_id": "idle_news_ai_daily",
  "name": "AI news daily intake",
  "target": "daily_intake_sweep",
  "schedule": "daily 07:00",
  "priority": "low",
  "enabled": true,
  "payload": {
    "rule_id": "daily_news_ai"
  }
}
```

### 5.3 Executor

`target` ごとに実処理を行う component。

Executor は request を受け取り、必ず terminal outcome を返す。

```text
completed
failed
blocked
skipped
cancelled
partial
```

### 5.4 Idle Budget

空き時間判定。

初期実装では簡易でよい。

```text
- service health が ok
- active Viewer voice / chat session がない、または job が軽量
- 同時実行数が上限未満
- job の not_before を過ぎている
- 前回失敗から cooldown 済み
```

### 5.5 Target Manifest

新しい作業種別を追加するための宣言。

Target Manifest は、executor の有無、payload schema、許可される作用、必要 approval、検証条件を定義する。

```json
{
  "target": "movie_topic_candidates_generate",
  "display_name": "Movie topic candidates generate",
  "executor": "builtin",
  "payload_schema_ref": "schemas/idle_jobs/movie_topic_candidates_generate.schema.json",
  "side_effect_level": "local_upsert",
  "approval_required": false,
  "artifact_type": "idle_job_result",
  "default_idle_policy": {
    "require_health_ok": true,
    "max_parallel": 1
  }
}
```

### 5.6 Custom Task

まだ専用 executor がない作業を、汎用の安全な形式で登録する target。

`custom_task` は自動で実作業を完了させるものではない。初期実装では Workstream Goal / Artifact / Coder draft / Command proposal を作るところまでに限定し、外部作用や正式 DB 変更は Human approval を待つ。

## 6. 対象ジョブ種別

### 6.1 daily_intake_sweep

Knowledge Memory の `daily_intake_rule` を見て、Source Registry entry 作成と sweep を行う。

既存実装:

```text
RunDailyIntakeSweep
startDailyIntakeSweeper
/viewer/knowledge-memory/daily-intake-rules
```

Idle Job executor 化:

```text
target: daily_intake_sweep
payload:
  rule_id: optional
  topic: optional
  source_hint: optional
  source_limit: optional
  minimum_trust_score: optional
```

動作:

1. rule_id が指定されていれば対象 rule を確認する。
2. rule が `active` / `reviewed` / `enabled` で http(s) source_hint を持つ場合だけ実行する。
3. Source Registry entry を作成または更新する。
4. due source sweep を行う。
5. staging / promoted count / failed count を run log に残す。

### 6.2 movie_catalog_backfill

映画 DB の不足情報を backfill する。

既存実装:

```text
startMovieCatalogBackfillJob
/viewer/movie-catalog/fetch
```

Idle Job executor 化:

```text
target: movie_catalog_backfill
payload:
  db_path: optional
  max_pages: 1
  timeout_sec: 90
  crawler_delay_sec: 2
```

動作:

1. movie catalog DB を解決する。
2. DB がなければ `skipped`。
3. backfill service を 1 回だけ実行する。
4. fetched / idle / failed を run log に残す。

### 6.3 movie_catalog_fetch

特定の映画または人物を取得する。

既存実装:

```text
POST /viewer/movie-catalog/fetch
```

Idle Job executor 化:

```text
target: movie_catalog_fetch
payload:
  kind: movie | person
  query: optional
  url: optional
  max_pages: 1..20
  follow_links: bool
  include_person_filmography: bool
```

候補が複数ある場合は `blocked` とし、候補一覧を artifact に残す。

### 6.4 movie_topic_candidates_generate

映画鑑賞履歴と movie_people edge から Mio の話題候補を生成する。

既存実装:

```text
POST /viewer/movie-catalog/topic-candidates/generate
```

Idle Job executor 化:

```text
target: movie_topic_candidates_generate
payload:
  limit: 20
```

実行結果:

```text
generated
skipped
candidate_ids
```

### 6.5 hobby_topic_candidates_generate

Hobby Graph から話題候補を生成する。

既存実装:

```text
POST /viewer/hobby-graph/topic-candidates/generate
```

Idle Job executor 化:

```text
target: hobby_topic_candidates_generate
payload:
  limit: 20
```

### 6.6 workstream_heartbeat

Workstream の pending steering / heartbeat schedule / vault update を処理する。
LLM 判断が必要な draft report は Shiro start の `OPS` route task として実行し、Mio / Chat surface の通常会話経路には入れない。

既存実装:

```text
/viewer/workstreams/heartbeats
internal/application/heartbeat
```

Idle Job executor 化:

```text
target: workstream_heartbeat
payload:
  workstream_id: optional
  mode: draft_report | safe_checkpoint | intake
```

### 6.7 command_run

登録済み command を AI Workflow 経由で実行する。

既存実装:

```text
/viewer/ai-workflow/commands/run
commands/*.md
```

Idle Job executor 化:

```text
target: command_run
payload:
  command: review-architecture
  input: optional
```

外部作用がある command は Human approval が必要。

### 6.8 custom_task

ユーザーが後から追加する任意の作業。

例:

```text
- 最近の映画ニュースから Mio の話題候補を作る
- 週1回、未整理の Source Registry staging を棚卸しする
- 月1回、古い Viewer screenshot artifact を整理候補として出す
- 新しい趣味カテゴリの候補を調べる
- 次に実装すべき小さい改善を提案する
```

Idle Job executor 化:

```text
target: custom_task
payload:
  title: required
  instructions: required
  scope: optional list
  preferred_route: draft | workstream | command | superagent
  required_skill: optional
  expected_artifact_type: optional
  approval_policy: draft_only | require_review | require_human_approval
  max_runtime_sec: optional
```

動作:

1. `instructions` をそのまま shell や外部 API に渡さない。
2. `preferred_route=draft` の場合は Workstream Artifact `pending_review` を作る。
3. `preferred_route=workstream` の場合は Workstream Goal と Artifact を作る。
4. `preferred_route=command` の場合は登録済み command の存在を確認し、未登録なら `blocked`。
5. `preferred_route=superagent` の場合は SuperAgent run queue へ `queued` として登録する。ただし scheduler / LLM 到達性がない場合は `blocked`。
6. 実ファイル変更、外部送信、DB promotion、push は自動実行しない。
7. 実行結果には「次に人間が承認すべき action」を必ず含める。

`custom_task` の初期 terminal outcome:

```text
completed: draft / goal / artifact / queue item を作成できた
blocked: route / command / store / approval boundary が不足
failed: 保存や検証に失敗
skipped: idle budget 不足
```

`custom_task` は、専用 executor が必要な作業を発見する入口でもある。繰り返し使う custom task は、Target Manifest と専用 executor へ昇格する。

## 9. Viewer

Viewer には `Scheduler / Idle Jobs` view を追加または既存 Scheduler view を拡張する。

表示するもの:

```text
- enabled jobs
- next_run_at
- last_run_at
- last status
- target
- schedule
- idle policy
- manual run button
- disable button
- recent run logs
- blocked / failed reason
```

表示ルール:

- executor 未接続 job は `blocked: executor unavailable` と表示する。
- last run が台帳のみの場合は `ledger only` と表示する。
- external side effect がある job は `approval required` を表示する。

## 10. Safety / approval

Idle Job は unattended execution であるため、通常の user request より厳しい。

Human approval なしで許可:

```text
- read-only fetch
- staging への保存
- candidate / draft / pending_review artifact 作成
- deterministic topic candidate 生成
- cache / local DB の非破壊 upsert
```

Human approval 必須:

```text
- external post / send / publish
- official DB promotion
- confirmed / pinned memory 昇格
- destructive cleanup
- package install / update
- git commit / push / PR 作成
- 課金が発生する外部 API 実行
```

禁止:

```text
- secret を log / artifact に保存する
- failed / blocked を completed に正規化する
- status 200 だけで成功扱いする
- stale current view を最新実行結果として表示する
```

## 11. 新規 job target 追加手順

新しい空き時間作業を追加する場合は、以下を満たす。

```text
1. target 名を決める。
2. payload schema を定義する。
3. executor を実装する。
4. terminal outcome を定義する。
5. artifact の内容と保存先を定義する。
6. approval boundary を定義する。
7. unit test / handler test / live smoke を追加する。
8. Viewer に status / blocked reason を表示する。
9. docs/10_新仕様/83_空き時間ジョブ実行基盤仕様.md に target を追記する。
```

### 11.1 専用 executor なしで追加する場合

新しい作業をすぐに試したい場合は、まず `custom_task` として登録する。

この場合の完了条件は「成果物 proposal を作ること」であり、正式な実行完了ではない。

```text
1. target は `custom_task` にする。
2. payload.title / payload.instructions を必須にする。
3. preferred_route を draft / workstream / command / superagent から選ぶ。
4. approval_policy を draft_only / require_review / require_human_approval から選ぶ。
5. 自動実行してよい作用を payload に書かず、approval boundary に寄せる。
6. run log に draft / artifact / queue item ID を残す。
7. 繰り返し使うことが分かったら専用 target へ昇格する。
```

### 11.2 専用 executor へ昇格する条件

以下のいずれかを満たす custom task は、専用 target / executor に昇格する。

```text
- 同じ作業が3回以上登録された
- payload schema が固定できる
- deterministic に実行できる
- Viewer で専用 status 表示した方が安全
- approval boundary が固まった
- 失敗パターンを明示的に扱う必要がある
```

昇格時は Target Manifest を追加し、`custom_task` から新 target へ job を移行する。

Executor interface の概念:

```go
type IdleJobExecutor interface {
    Target() string
    Run(ctx context.Context, job IdleJob, opts RunOptions) (IdleJobResult, error)
}
```

Result は必ず terminal status を持つ。

```go
type IdleJobResult struct {
    Status string // completed, failed, blocked, skipped, cancelled, partial
    Summary string
    ArtifactIDs []string
    Error string
}
```

## 12. 実装段階

### Phase 1: scheduler executor 接続

- `/viewer/scheduler` の `run` を executor registry に接続する。
- executor 未接続時は `blocked` run log を保存する。
- `target` / `payload` を保存できるよう job schema を拡張する。
- `custom_task` executor を追加し、draft / workstream artifact 作成だけを安全に実行できるようにする。
- Target Manifest registry を追加し、未知 target を `blocked` として扱う。

### Phase 2: deterministic executor

以下を実装する。

```text
daily_intake_sweep
movie_topic_candidates_generate
hobby_topic_candidates_generate
movie_catalog_backfill
custom_task
```

### Phase 3: Viewer UI

- Scheduler / Idle Jobs view を追加する。
- manual run / disable / recent logs を操作できるようにする。
- `ledger only`、`executor unavailable`、`approval required` を表示する。

### Phase 4: idle budget

- service health
- active voice/chat session
- max parallel
- cooldown
- not_before

を見て実行可否を判定する。

### Phase 5: Workstream / SuperAgent 統合

- heavy analysis や長時間作業は SuperAgent run queue に渡す。
- deterministic job は scheduler executor が直接処理する。
- LLM が必要な job は Workstream artifact を pending_review に保存し、Human approval を待つ。

## 13. 完了条件

初期実装の完了条件:

```text
- `/viewer/scheduler` で target / payload 付き job を作成できる。
- `run` が executor registry を通る。
- executor 未接続時は blocked として保存される。
- `custom_task` を作成でき、pending_review artifact または Workstream Goal を作れる。
- `daily_intake_sweep` を手動実行でき、Source Registry staging / run log が残る。
- `movie_topic_candidates_generate` を手動実行でき、candidate count / artifact が残る。
- Viewer で enabled job / next run / last status / blocked reason が見える。
- status 200 や ledger-only run を成功扱いしない test がある。
```

## 14. 非目標

本仕様では以下を行わない。

```text
- 外部投稿や公開を自動化する
- Human approval なしで official memory / official DB へ promote する
- すべての background job を即時統合する
- LLM 到達不可状態を成功として扱う
- OS cron を完全廃止する
```

## 15. 現状との差分

2026-06-23 時点の live 状態では次の通り。
