# 残タスク段階実装計画

この文書は、`15_残タスク実装仕様.md` を実装担当者が順に進められる作業計画へ分解したものである。対象は 4 系統の残作業だが、Source Registry warning は metadata 付与と Viewer 表示を分けて扱うため、実装 Phase は 5 段階にする。

## 1. 全体ロードマップ

実装順序:

| Phase | 作業 | 主な目的 | 完了判定 |
| --- | --- | --- | --- |
| Phase 1 | ChannelPolicy config/runtime wiring | config から policy を読み、LINE handler へ起動時注入する | 未許可入力が routing / LLM へ流れない |
| Phase 2 | Source Registry warning metadata | Source Registry staging / fetch result に prompt injection warning を残す | staging metadata に warning が保存され API で追える |
| Phase 3 | Viewer warning 表示 | Source Registry warning を Viewer で確認できるようにする | warning が本文 / memory / prompt と混ざらず表示される |
| Phase 4 | Slack / Discord / Telegram file payload | LINE 以外の file payload を共通 attachment pipeline へ接続する | 共通 `Attachment` contract に正規化される |
| Phase 5 | Wild 実機分散 E2E | distributed mode で `RouteWILD` が fallback なしに動くことを確認する | 実機分散 1 セッションで route / response / log が一致する |

横断方針:

- 先にテストまたは検証観点を定義してから実装する。
- fallback、download 失敗、policy 拒否、Viewer warning 未表示、local fallback を成功扱いしない。
- Domain / Adapter / Application / Infrastructure の責務を混ぜない。
- docs 更新は各 Phase の最後に行い、`13_実装項目インベントリ.md` と `14_旧仕様未実装候補.md` の状態表を実態に合わせる。

## 2. Phase 1: ChannelPolicy config/runtime wiring

### 2.1 目的

既存の `internal/domain/security.ChannelPolicy` と `internal/adapter/line.Handler.SetChannelPolicy` を使い、config から読み込んだ policy を local mode / distributed mode の LINE handler 起動時に注入する。

### 2.2 現状

- `internal/domain/security/channel_policy.go` に `ChannelPolicy`、`ChannelRequest`、`Evaluate` がある。
- `internal/adapter/line/handler.go` に `SetChannelPolicy` がある。
- `internal/adapter/line/handler_test.go` に ChannelPolicy reject 系テストがある。
- `internal/adapter/config/config_types.go` には Telegram / Discord / Slack / notification channel 設定はあるが、LINE channel policy 設定はない。
- `cmd/picoclaw/runtime_channels.go` と `cmd/picoclaw/runtime_distributed_mode.go` は LINE handler に attachment saver を注入しているが、policy 注入はしていない。

### 2.3 実装対象

```go
type ChannelPolicyConfig struct {
    Enabled        bool     `yaml:"enabled"`
    AllowDM        *bool    `yaml:"allow_dm"`
    AllowGroups    *bool    `yaml:"allow_groups"`
    AllowedSenders []string `yaml:"allowed_senders"`
```

YAML 例:

```yaml

  channel_access_token: "${LINE_CHANNEL_ACCESS_TOKEN}"

    allow_dm: true
    allow_groups: true

    paired_groups:
      - "Cxxxxxxxx"
```

domain `ChannelPolicy` への変換関数案:

```go
func ResolveChannelPolicyConfig(cfg ChannelPolicyConfig) (security.ChannelPolicy, bool) {
    if !cfg.Enabled {
        return security.ChannelPolicy{}, false
    }
    policy := security.ChannelPolicy{
        AllowDM:        boolValue(cfg.AllowDM, true),
        AllowGroups:    boolValue(cfg.AllowGroups, false),
        AllowedSenders: compactStrings(cfg.AllowedSenders),
        PairedGroups:   compactStrings(cfg.PairedGroups),
    }
    return policy, true
}
```

default behavior:

- `enabled: false` または未設定では `SetChannelPolicy` を呼ばない。
- 未設定時は現行互換を維持する。
- `enabled: true` で `allow_dm` / `allow_groups` が未指定の場合は、明示 default をテストで固定する。推奨は DM 許可、group 不許可。

## 3. Phase 2: Source Registry warning metadata

### 3.1 目的

Source Registry 由来の外部テキストに `DetectPromptInjectionWarnings` を適用し、staging / fetch result の metadata として warning を保存する。

### 3.2 現状

- `internal/domain/security/prompt_injection.go` に warning 検出関数がある。
- attachment では `internal/application/attachment/store.go` が `SecurityWarnings` を保存している。
- `internal/infrastructure/persistence/conversation/l1_sqlite_types.go` の `L1SourceFetchPayload` と `L1StagingItem` は `Meta map[string]interface{}` を持つ。
- `internal/application/sourcefetcher/registry_sweeper.go` は `StageSourceRegistryFetch` を呼ぶ。
- Source Registry 由来テキストの warning metadata はまだ付与されていない。

### 3.3 実装対象

変更候補:

- `internal/application/sourcefetcher/registry_sweeper.go`
- `internal/application/sourcefetcher/registry_sweeper_test.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_source_registry.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_staging.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_store_test.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_types.go`
- `internal/adapter/viewer/source_registry_handler.go`
- `internal/adapter/viewer/source_registry_handler_test.go`

metadata key 案:

```json
{
  "security_warnings": ["prompt_injection:ignore_previous_instructions"],
  "security_warning_source": "source_registry",
  "security_warning_checked_at": "2026-05-17T00:00:00Z"
}
```

schema migration 方針:

- 第一段階は `Meta` を使い、SQLite schema migration を避ける。
- 専用 column が必要になる条件は、warning の検索、集計、index、運用監査が必要になった場合。
- Meta を使うことで既存 staging / export / import との互換を保つ。

### 3.4 対象外

- warning を即 reject と同一視する挙動。
- LLM chain-of-thought の保存。
- 無審査 memory promote。

### 3.5 先に書くテスト

- sourcefetcher test: injection pattern を含む source fetch result が `security_warnings` 付きで staging される。
- persistence test: `L1StagingItem.Meta.security_warnings` が保存 / 読み出しできる。
- handler test: Source Registry run API response に warning metadata が含まれる。
- promotion test: warning 付き item を無審査で promote しない確認を追加するか、既存 validator policy に明示的な確認点を追加する。

### 3.6 実装手順

1. warning metadata helper を作る。配置は `internal/application/sourcefetcher` または security domain の薄い helper にする。
2. `registry_sweeper.go` で `payload.RawText` に `DetectPromptInjectionWarnings` を適用する。
3. warning がある場合、`payload.Meta` に `security_warnings` を追加してから `StageSourceRegistryFetch` へ渡す。
4. `StageSourceRegistryFetch` が `payload.Meta` を `L1StagingItem.Meta` へ保持していることをテストで固定する。
5. Source Registry run API response が staged item metadata を返す場合は warning を含める。返さない場合は response contract を拡張する。
6. warning と validation reject は別フィールドとして扱う。

```bash
GOCACHE=/tmp/picoclaw-gocache go test ./...
```

### 3.8 E2E / 手動確認

- Source Registry run を実行する。
- staging item の metadata に warning が保存されることを API / CLI / DB 経由で確認する。

### 3.9 完了条件

- Source Registry staging に warning metadata が残る。
- warning が reject と混同されない。
- warning 付き item が無審査で memory promote されない。
- API response で Viewer が warning を表示できる材料がある。

### 3.10 失敗扱いにする条件

- warning が staging metadata に保存されない。
- warning を通常本文や summary に混ぜる。
- warning 付き item が validator なしに confirmed memory へ入る。
- API response から warning が欠落する。

## 5. Phase 4: Slack / Discord / Telegram file payload

### 5.1 目的

Slack / Discord / Telegram の file payload を Viewer upload / LINE media と同じ `internal/application/attachment` pipeline へ接続する。

### 5.2 現状

- LINE は `SetAttachmentSaver` と media download で共通 attachment pipeline へ接続済み。
- Slack / Discord / Telegram adapter は存在するが、主に text event 正規化と outbound send が中心。
- `internal/application/attachment.IncomingFile` と `Store.SaveAll` が transport-neutral upload 境界になっている。
- `internal/application/orchestrator/message_orchestrator_task.go` は task attachments の event を出せる。

### 5.3 実装対象

変更候補:

- `internal/adapter/channels/slack/adapter.go`
- `internal/adapter/channels/slack/adapter_test.go`
- `internal/adapter/channels/discord/adapter.go`
- `internal/adapter/channels/discord/adapter_test.go`
- `internal/adapter/channels/telegram/adapter.go`
- `internal/adapter/channels/telegram/adapter_test.go`
- `internal/application/channel`
- `internal/application/attachment`
- `internal/domain/attachment`
- `internal/application/orchestrator/message_orchestrator_task.go`
- `cmd/picoclaw/runtime_channels.go`
- `cmd/picoclaw/runtime_distributed_mode.go`

channel ごとの payload 正規化:

| channel | 入力 | download | `IncomingFile` 生成 |
| --- | --- | --- | --- |
| Slack | `files[]`, `url_private_download`, filename, mimetype, size | Bot token で GET | filename / mimetype / size / response body |
| Discord | message `attachments[]`, filename, content_type, size, url | attachment URL を GET | filename / content_type / size / response body |
| Telegram | `document`, `photo`, `file_id`, mime_type, file_name | getFile -> file path -> download | file_name または推定名 / mime_type / file_size / response body |

### 5.4 対象外

- 全 channel の巨大ファイル保存。
- 未対応 MIME の自動変換。
- OCR / image PDF 高精度解析。
- channel 固有 payload を Application attachment 内へ持ち込むこと。

### 5.5 先に書くテスト

- Slack: `files[]` payload から download helper が呼ばれ `IncomingFile` が生成される。
- Slack: download 失敗時に routing / LLM へ流れない。
- Discord: attachments payload から `IncomingFile` が生成される。
- Telegram: `document.file_id` から getFile + download が呼ばれる。
- common: unsupported MIME / size 超過が通常 chat 成功として隠れない。
- runtime: local mode / distributed mode で adapter に attachment saver が注入される。

### 5.6 実装手順

1. 各 adapter に `AttachmentSaver` interface と `SetAttachmentSaver` 相当を追加する。
2. runtime setup で `attachmentapp.NewStore(cfg.WorkspaceDir)` を Slack / Discord / Telegram adapter へ注入する。
3. channel payload parser に file metadata extraction を追加する。
4. channel 固有 download helper を adapter 内に追加する。
5. download 結果から `IncomingFile` を作成する。
6. `SaveAll` の結果を channel event / orchestrator request に載せる。
7. attachment event に session_id、channel、chat_id、job_id が残ることを確認する。
8. download 失敗、MIME 不許可、size 超過は error とし、通常 chat 成功にしない。
