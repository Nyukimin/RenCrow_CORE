# 新仕様 概要

## 目的

RenCrow は、複数の LLM と実行主体を役割分担させる軽量 AI アシスタントである。

主な目的は次の通り。

- LINE / Slack / Discord / Telegram / Viewer などからの入力を受ける。
- Chat / Worker / Coder の責務分離に基づいて処理をルーティングする。
- Coder が plan / patch / proposal を生成し、Worker が実行する。
- Viewer、IdleChat、STT、TTS、Memory、Source Registry を疎結合な機能境界として扱う。
- 低スペック環境でも動作できるよう、runtime config と provider を差し替え可能にする。

## 新仕様の位置づけ

このフォルダは、`docs/01_正本仕様/` の主要内容を、Phase28 から Phase34 までのリファクタリング後の実装構造に合わせて再構成したものである。

旧リポジトリでは `docs/01_正本仕様/` が履歴上の正本仕様として残る。新規リポジトリへ移行する場合は、`docs/10_新仕様/` を仕様入口として扱う。

## 最重要原則

### モジュール化

機能単位で責務を分ける。単にファイルを分けるだけではモジュール化ではない。

各モジュールは、次を説明できる必要がある。

- 入力
- 出力
- 副作用
- 永続化
- ログ
- エラー契約
- 差し替え時の影響範囲

### 疎結合

呼び出し側は内部実装ではなく、interface、contract、event、DTO、adapter、repository、provider の境界に依存する。

ある機能を差し替えるとき、他の機能や層へ影響が広がらない構造を優先する。

### 仕様と実装箇所の対応

仕様変更時は、その仕様を実装する主担当ファイルまたは主担当パッケージを説明できる状態にする。

1 つの仕様変更が無秩序に複数箇所へ広がる場合は、仕様の粒度または実装配置が崩れている。先に `11_分割再設計候補.md` へ記録し、仕様単位またはモジュール境界を整理する。

## 主要コンポーネント

| 領域 | 位置づけ |
| --- | --- |
| Chat | ユーザー対話、ルーティング判断、結果返却 |
| Worker | 実行主体、patch / file / shell / git / test 実行 |
| Coder | plan / patch / proposal 生成、設計説明 |
| Viewer | 表示、操作、SSE event、log、history、monitoring |
| IdleChat | アイドル時の自律会話、forecast、story、TTS 連携 |
| STT | 通常 chat への音声入力経路 |
| TTS | 音声合成、audio event、browser playback、口パク trigger |
| LLM provider | Chat / Worker / Heavy / Wild / Coder への provider 実装 |
| Memory | conversation memory、RecallPack、L1SQLite、VectorDB、DuckDB |
| Source Registry | 外部ソースの observed / candidate / validated / promoted 管理 |
| Browser Actor | Headless browser 操作、profile / Cookie 管理、screenshot / trace 証跡化 |

## fallback の扱い

fallback は正常系ではない。

fallback は、分類失敗、生成失敗、provider 失敗、外部依存未準備などの失敗または回復経路として扱う。成功として隠さず、ログ、Viewer 表示、検証結果で区別する。

## Viewer / Audio / Log の分離

Viewer 表示、音声、口パク、ログは別の契約である。

- Viewer 表示は表示用 state / event / history を根拠にする。
- 音声 chunk は音声再生と口パク trigger の根拠であり、本文表示の唯一の根拠にしない。
- log は観測と追跡のための記録であり、表示契約そのものではない。

## runtime config の扱い

repo example、live runtime config、Viewer 表示値を混同しない。

- repo example はサンプルである。
- live runtime config は `~/.picoclaw/config.yaml` と実起動状態で確認する。
- Viewer runtime config は表示と操作のための投影である。

## Phase28-34 の反映

Phase28 から Phase34 により、未分化だった大型モジュールは責務別に分割済みである。

- `buildDependencies()` は runtime wiring helper へ分割済み。
- route registration は `cmd/picoclaw/routes.go` に分離済み。
- ToolRunner、config、IdleChat 補助処理、MioAgent、TTS / LLM provider、persistence は責務別ファイルへ分割済み。
- 残る大きめの production file は、型定義、単一 provider、単一 store API、単一 transport、単一 domain contract として説明できる。
