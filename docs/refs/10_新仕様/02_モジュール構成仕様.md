# モジュール構成仕様

## 全体構成

RenCrow は Clean Architecture 風の構成を維持する。

```text
cmd/
  picoclaw/
  picoclaw-agent/
  kb-admin/
  glossary/

internal/
  domain/
  application/
  infrastructure/
  adapter/
```

## 層の責務

| 層 | 置くもの | 置かないもの |
| --- | --- | --- |
| `cmd/` | 起動、CLI、composition root、runtime wiring、HTTP server | usecase 本体、provider 詳細、Viewer 状態変換、DB state 遷移 |
| `internal/domain/` | 役割、値、契約、route、task、proposal、patch、memory、session | HTTP handler、DB 実装、LLM API 呼び出し、Viewer DOM |
| `internal/application/` | orchestration、Worker execution、IdleChat、sourcefetcher、TTS coordination | provider 固有 HTTP request、SQLite schema、静的 asset |
| `internal/infrastructure/` | LLM provider、persistence、security、tools、transport、STT/TTS、VTuber、MCP | route 決定 usecase、Viewer handler、domain contract 本体 |
| `internal/adapter/` | Viewer、channel、health、config、chrome、entry など外部入出力 | Worker 実行本体、LLM provider 本体、memory state 遷移本体 |

## 1 対 1 対応が崩れた場合

次の状態は分割・再設計候補として扱う。

- 1 つの仕様変更で 3 つ以上の主担当パッケージを同時に変更する。
- 主担当が `cmd/picoclaw` だけになり、Application / Adapter / Infrastructure へ分離できていない。
- `service` / `manager` / `helper` / `util` が責務を吸収している。
- テストだけが仕様の置き場所になっている。
- Viewer 表示、音声、口パク、ログが同じデータ契約として扱われている。
- repo example、live runtime config、Viewer runtime config が混同されている。

この場合は `11_分割再設計候補.md` に記録し、実装前に仕様単位を分ける。
