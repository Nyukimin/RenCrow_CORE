## 目的

以下を達成してください。

1. `docs/10_新仕様/` に書かれているが、現行 production code に未実装の項目を洗い出す
2. 実装済み / 部分実装 / 未実装 / 要確認 / 対象外 を分類する
3. 重複仕様を整理する
4. 対立仕様を検出し、どちらを優先すべきか整理する
5. 実装単位へ分解する
6. 実装担当者がそのまま着手できる粒度の実装仕様を作成する

## 重要ルール

- `docs/10_新仕様/` を現行仕様の入口とする
- 実装判断で迷った場合の一次参照は `docs/01_正本仕様/実装仕様.md` とする
- archive 文書は一次参照にしない
- 旧仕様を根拠にする場合は、必ず `docs/10_新仕様/` へ反映すべき差分として扱う
- テストだけを根拠に実装済み扱いしない
- code path の存在だけで完了扱いしない
- fallback を成功扱いしない
- 実装そのものは行わない
- production code を変更しない
- 今回は仕様整理と実装仕様作成のみ行う
- 重複仕様や対立仕様があった場合は、黙って片方を採用せず、整理方針を明記する
- 実装仕様の通し番号は、現状仕様の番号と一致させる必要はない
- ただし、実装仕様内に「参考仕様番号 / 参考文書」として元仕様を必ず明記する
- Chat / Worker / Coder / Adapter / Application / Domain / Infrastructure の責務境界を崩さない
- Coder が破壊的操作を直接実行する前提の仕様にしない

## 作業手順

1. `git status --short` を確認する
2. `docs/10_新仕様/00_README.md` を読み、仕様一覧を把握する
3. `docs/10_新仕様/` の各仕様を読む
4. 未実装・部分実装・要確認と思われる項目を抽出する
5. 関連 production code を `rg` で確認する
6. 必要に応じて `go test` や既存 test 名を確認する。ただし、今回は実装・修正はしない
7. 各項目を以下に分類する

```text
実装済み:
  production code / runtime wiring / config / tests / 表示/API確認まで揃っている

部分実装:
  domainや一部adapterにはあるが、config / runtime wiring / Viewer / E2Eなどが欠ける

未実装:
  仕様はあるが production code の実装が見当たらない

要確認:
  code path はあるが、runtime / E2E / 実機確認が不足している

対象外:
  仕様上、将来構想または明示的に実装対象外
```

8. 重複仕様を検出する
9. 対立仕様を検出する
10. 整理方針を作る
11. 実装仕様を作成する

## production code 確認の観点

未実装判定では、以下のどれが欠けているかを必ず明記してください。

```text
domain:
  型、値オブジェクト、ポリシー、contract

application:
  usecase、service、orchestrator、pipeline

adapter:
  LINE / Slack / Discord / Telegram / Viewer / API handler

infrastructure:
  persistence、transport、tool runner、security gate

cmd runtime:
  起動時の依存注入、configからのwiring、distributed mode wiring

config:
  schema、defaults、validation、example

viewer/api:
  表示、JSON response、警告表示、artifact review

tests:
  unit、handler、integration、runtime、E2E、手動確認手順
```

## 重複仕様の整理方針

重複仕様がある場合、以下の形式で整理してください。

```markdown
### 重複仕様: {テーマ名}

#### 関連仕様
- `docs/10_新仕様/XX_...md`
- `docs/10_新仕様/YY_...md`

#### 重複内容
何が重複しているか。

#### 整理方針
- 正本にする仕様
- 補助仕様にする仕様
- 将来分離する仕様
- 削除または統合候補

#### 実装仕様での扱い
今回の実装仕様ではどちらを参照するか。
```

重複の可能性が高いテーマ例:

- Tool Harness / Sandbox / SuperAgent Harness の safety gate
- Workstream / AI Native Engineering Workflow / SuperAgent Harness の subagent・context管理
- DCI / Browser Trace to API Discovery の evidence / Source Registry 接続
- Revenue / Workstream の日次 routine・heartbeat
- Persona Observation / Workstream Vault の observation log / memory update

## 対立仕様の整理方針

対立仕様がある場合、以下の形式で整理してください。

```markdown
### 対立仕様: {テーマ名}

#### 仕様A
- 文書:
- 内容:

#### 仕様B
- 文書:
- 内容:

#### 対立点
何が矛盾しているか。

#### 優先判断
どちらを優先するか。

#### 判断理由
- 現行 architecture
- production code
- 安全性
- Chat / Worker / Coder 責務境界
- runtime実現性
- ユーザー方針

#### 実装仕様での扱い
採用する仕様と、保留する仕様を明記する。
```

対立の可能性が高いテーマ例:

- 自動実行範囲と Human approval 境界
- Sandbox 内 write 許可と Coder 破壊的操作禁止
- Source Registry への staging / validator 必須と fetcher 自動化
- Heartbeat の自動処理と外部送信・公開禁止
- Skill auto-trigger とユーザー明示指示の優先順位

## 実装仕様に必ず含める構成

作成する実装仕様には、以下を含めてください。

```markdown
# 未実装項目実装仕様

## 1. 目的
この実装仕様の目的。

## 2. 調査範囲
読んだ仕様書、確認した production code。

## 3. 分類結果サマリ
実装済み / 部分実装 / 未実装 / 要確認 / 対象外 の件数と要点。

## 4. 重複仕様の整理
重複している仕様と整理方針。

## 5. 対立仕様の整理
対立している仕様と優先判断。

## 6. 未実装項目一覧
表で整理する。

| ID | 項目 | 状態 | 参考仕様 | 関連コード | 不足内容 | 優先度 | 整理メモ |
|---|---|---|---|---|---|---|---|

## 7. 実装Phase
Phaseごとに実装順序を示す。

各Phaseには以下を含める。

- 目的
- 参考仕様
- 現状
- 実装対象
- 対象外
- 先に書くテスト
- 実装手順
- runtime wiring
- config
- Viewer / API 表示
- E2E / 手動確認
- 完了条件
- 失敗扱いにする条件
- 更新する docs

Phase ID は `P1`, `P2` のように実装順で振ってください。元仕様番号と一致させる必要はありません。
ただし、各 Phase の `参考仕様` には必ず元文書を列挙してください。

## 8. 検証マトリクス

| 項目 | unit test | integration test | runtime確認 | Viewer/API確認 | E2E確認 | 完了条件 | 残リスク |
|---|---|---|---|---|---|---|---|

## 9. docs更新方針
実装後に更新すべき docs。

## 10. 保留項目
今回実装仕様化しないものと理由。

## 11. 次アクション
実装担当者が最初に着手すべき順序。
```

## 優先度の付け方

優先度は、以下の観点で決めてください。

```text
P0:
  既存動作の安全性、runtime wiring、config、fallback失敗扱い、責務境界に関わるもの

P1:
  仕様上の中核で、Viewer/API/永続化/Source Registryなど実運用に必要なもの

P2:
  便利機能、観測性、レポート、運用改善

P3:
  将来構想に近く、現時点では保留でもよいもの
```

優先度は実装順の目安であり、低優先度を「不要」とは扱わないでください。

## 注意

- 旧 OpenClaw の広範 provider / OAuth / ClawHub / Docker sandbox など、現行 RenCrow の方針外のものを混ぜない
- LangGraph、FAISS、旧キャラ、旧6キャラ会話など、ユーザーが対象外または将来扱いにしたものは未実装課題として扱わない

- 実装仕様内で「fallbackで代替できるため完了」と書かない
