# RenCrow 物理分離前提仕様

## 1. 全体目的
RenCrow は、単一の実装として動作しながらも、将来は `repo` 単位でも切り出せるように設計する。  
目的は、会話、実行、推論、音声、認識、記憶、拡張探索を混ぜずに分離し、各機能を独立して保守・移植・検証できる状態にすること。  
この仕様の正本は「責務」「境界」「状態所有」「依存方向」であり、実装ファイル配置は後から変わってもよい。

## 2. モジュール構成
| モジュール | 正本の責務 | 正本の状態 | 依存先 |
| --- | --- | --- | --- |
| Core | 共通契約、ID、状態遷移、健康状態 | session/request/response/chunk の共通規則 | 他に依存しない |
| Chat | 対話、ルーティング、表示、IdleChat | 会話の正本、Viewer表示 | Core, LLM, TTS, STT |
| Worker | 実行、ファイル操作、テスト、拡張作業 | 実行ジョブの正本 | Core, LLM |
| LLM | 推論、プロバイダ、ロール割当 | モデル選択と応答契約 | Core |
| TTS | 音声合成、再生契約、音声表現 | 合成結果、音声ルール | Core |
| STT | 音声認識、入力契約 | 認識結果、入力ルール | Core |
| App | Viewer、IdleChat、API、起動配線 | 起動構成、I/O配線 | 各モジュールを利用 |

### モジュールの原則
- 各モジュールは「何を提供するか」を持ち、「どこで使うか」は持たない。
- `App` は composition root であり、ドメインルールの正本ではない。
- `modules/*` は repo 分割時の切り出し単位として成立する粒度を目標にする。

## 3. エージェント構成
| エージェント | 役割 | 正本の責務 |
| --- | --- | --- |
| Chat | 対話、判断、返答 | ユーザーとの会話の進行 |
| Worker | 実行 | コマンド、編集、検証、運用作業 |
| ChatWorker | IdleChat 会話専用 Worker ロール | IdleChat の Shiro など会話出力 |
| Heavy | 重い推論の Worker ロール | 長文・高コスト推論 |
| Wild | 例外的・柔軟な Worker ロール | 通常ロールで難しい処理 |
| Coder1-4 | コード生成のスロット | 設計案、patch、plan の生成 |

### エージェントの原則
- `ChatWorker` は Worker の別人格ではなく、**会話専用ロール**。
- `Heavy` と `Wild` は Worker の運用ロールであり、独立アプリではない。
- `Coder1-4` は別エージェントではなく、Coder ロールのスロットである。
- エージェントは権限ではなく役割で切る。

## 4. 機能モジュール構成
### LLM
- 正本: プロバイダ契約、ロール割当、応答整形、局所的な health/diagnostics
- 禁止: 音声生成、STT 処理、Viewer の状態所有

### TTS
- 正本: 合成、音声ルール、感情 prefix、再生イベント契約、再生状態の観測契約
- 禁止: 会話の正本化、会話内容の再構成、ルーティングの主導

### STT
- 正本: 音声入力、認識、認識結果正規化、入力状態
- 禁止: 会話の返答生成、TTS への逆流

### Core
- 正本: 共通 ID、状態所有、汎用 health、JSON 契約、境界ルール
- 禁止: 具体 provider 実装

### App
- 正本: 配線、起動順、HTTP ルート、Viewer 連携、runtime config の注入
- 禁止: 永続的なドメインルールの所有

## 5. 記憶システム構成
記憶は「保存先」ではなく「性質」で分ける。

| 領域 | 正本 | 例 |
| --- | --- | --- |
| Self memory | 自分自身の定義 | 私の役割、好み、禁則 |
| Relation memory | 私ではない重要人物 | 娘、家族、重要人物 |
| Episodic memory | 経験 | いつ何が起きたか |
| Semantic memory | 抽象化された知識 | 経験から一般化したルール |
| Knowledge memory | 永続知識 | 映画、作品、定義、固定情報 |
| Volatile memory | 短命・不確実情報 | 株、相場、時事、未確定情報 |
| Recall | 引き出し・検索 | 関連順で再構成 |
| Ingest | 入手経路 | 会話、RSS、Web、手動入力 |
| Policy | 保存/破棄/候補留め | 昇格条件、TTL、信頼度 |

### 記憶の原則
- `self` と `non-self` を分ける。
- 娘などの重要人物は `relation entity` として扱い、`self` に混ぜない。
- 経験は `episodic`、抽象化された知識は `semantic` にする。
- 株のような不確実情報は `volatile` に置き、永続化の可否を明示する。
- 保存・破棄・候補留めは policy が決める。
- 記憶の正本は「誰の、何の性質の情報か」で決まる。

## 6. heartbeat と常時評価
heartbeat は、常時何かをし続けるための評価ループである。  
目的は「今、ユーザーのために何をするのが最良か」を定期的に再評価すること。

### heartbeat の責務
- 状態収集
- 優先度評価
- 未完了タスクの再評価
- 記憶更新要否の判定
- 機能不足の探索要否の判定
- 実行/保留/調査/提案/何もしない の分岐

### heartbeat の禁止事項
- 暴走して無限実行しない
- ユーザー意図を無視して勝手に実行しない
- 危険操作を policy gate なしで通さない

### 状態遷移
`idle -> evaluate -> decide -> {noop | hold | investigate | propose | execute | update_memory | explore_extension} -> idle`

## 7. モチベーションと感情表出の分離
### 内部モチベーション
- drive
- priority
- continuation
- retry

これは内部制御信号である。  
「やる気」を人間の感情で表す必要はない。

### 感情表出
- Mio に見せる表現層
- 口調、絵文字、強度、間、語尾
- 内部 drive をそのまま出さない

### 原則
- 内部制御と人格表現を混ぜない。
- Mio 用の感情表現モジュールを別に持つ。
- Shiro など他人格へ漏れない。
- 同じ内部状態でも、人格ごとに表出が異なってよい。

## 8. 拡張可能性
RenCrow の賢さには、現在機能で足りないときに能力を増やせることを含める。

### 拡張の流れ
1. 現状機能で解けるか評価する
2. 足りない場合は新機能を探す
3. それでも足りなければインストールする
4. まだ足りなければ作る
5. 次回以降使える形で定着させる

### 拡張の正本
- 探索は `find`
- 追加判断は policy
- 実行は Worker
- 定着は memory / module registry

### 禁止事項
- 探索と実行を混ぜる
- 危険な導入を無審査で行う
- 一回限りの裏技で済ませる

## 9. 依存方向と禁止事項
### 許可方向
- `Chat -> Core, LLM, TTS, STT`
- `Worker -> Core, LLM`
- `LLM/TTS/STT -> Core`
- `App -> 各モジュール`

### 禁止方向
- `LLM -> TTS/STT`
- `TTS -> LLM/STT`
- `STT -> LLM/TTS`
- `Worker -> TTS/STT`
- `Core -> 他モジュール`

### 禁止事項
- 表示、音声、認識、記憶、実行を同じ state に混ぜる
- provider 実装が module 契約を上書きする
- `App` がドメインルールの正本になる
- 1つの概念に複数の所有者を置く

## 10. 状態所有
| 状態 | 所有者 | 正本の意味 |
| --- | --- | --- |
| 会話本文 | Chat | ユーザーに見せる言葉 |
| 音声合成結果 | TTS | 再生される音声 |
| 音声認識結果 | STT | 入力された音声の文字化 |
| 実行ジョブ | Worker | コマンドや編集の結果 |
| LLM プロバイダ健康 | LLM | モデル稼働状態 |
| 共通ID/遷移 | Core | session/response/chunk の契約 |
| Self/Relation/Episodic/Knowledge | Memory | 各記憶の本体 |
| heartbeat 判断 | App + policy | 次の行動の決定 |

### 正本ルール
- 正本は1つだけ。
- 読み取り用の派生 state は複数あってよいが、書き込み正本は1つ。
- 正本が曖昧な状態は設計不備として扱う。

## 11. 将来の物理分離方針
物理分離は、以下の順で進めるのがよい。

1. `core` の契約固定
2. `llm` / `tts` / `stt` の機能モジュール固定
3. `chat` / `worker` のエージェント境界固定
4. `memory` 系の正規化
5. `heartbeat` と拡張探索の独立化
6. `viewer` / `idlechat` など app 層の切り出し

### repo 分割候補
- `rencrow-core`
- `rencrow-chat`
- `rencrow-worker`
- `rencrow-llm`
- `rencrow-tts`
- `rencrow-stt`
- `rencrow-app-*`

### 分割条件
- 公開 API が安定している
- 依存方向が逆流しない
- 状態所有が一意
- 代替実装に差し替え可能
- テストがモジュール単位で成立する

- `Mio` の感情表出語彙の正本
- `ChatWorker / Heavy / Wild` の将来的な独立度
- `Viewer` を app としてどこまで分離するか
- `拡張探索` を外部アプリ連携とどう切るか

## 要約
RenCrow は、Chat/Worker/LLM/TTS/STT/Core に加えて、記憶・heartbeat・拡張探索・感情表出を分離し、将来 repo 単位で切り出せるようにする。  
正本は「責務」「状態所有」「依存方向」であり、実装配置はその次である。

## 物理分離の優先順
1. `core`
2. `llm` / `tts` / `stt`
3. `chat` / `worker`
4. `memory`
5. `heartbeat` / `extension`
6. `app/viewer/idlechat`

## 次に実装へ落とすべき最小単位
- `memory` の state schema と owner を固定する
- `heartbeat` の decision contract を定義する
- `Mio` の感情表出 module を独立契約にする
- `ChatWorker` / `Heavy` / `Wild` の役割を `Worker` 内のロール契約として固定する
- `拡張探索` の判定結果を `proposal` と `execute` に分ける
