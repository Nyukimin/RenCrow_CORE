# 知識・記憶システム構想

## 1. 目的

この文書は、RenCrow の知識・記憶システムについて、実装詳細に入る前の入口方針をまとめる。

既存の `09_Memory_SourceRegistry仕様.md` は、現在の永続化境界、Source Registry、L1 SQLite、RecallPack、Viewer API を扱う実装仕様である。この文書はその上位にある思想、分類、責務境界を扱う。

## 2. 基本方針

RenCrow の記憶は、単なる会話要約ではない。

中心に置くべきものは、次の分離である。

- 原本と派生物を混ぜない。
- 個人記憶、作品知識、ニュース、日次取得ソース、RenCrow 自身の運用記憶を分ける。
- 圧縮、要約、検索用 index、prompt 注入用 context は原本の代替ではない。
- `skip`、`fallback`、未検証情報、推測を正式記憶として扱わない。
- 記憶は常に参照可能であることが望ましいが、prompt へ常時全投入しない。

## 3. Personal Archive

Personal Archive は、ユーザー本人に関する一次情報の保護領域である。

対象:

- 過去の text。
- 生い立ち。
- BIO。
- 人格、背景、嗜好、価値観、判断基準の元になる原文。
- ユーザーが明示的に残すべきと考えた自己情報。

方針:

- 圧縮しない。
- 要約で置き換えない。
- 古いものを勝手に消さない。
- LLM が勝手に再解釈して上書きしない。
- 追記型にする。
- オリジナルを保護する。

圧縮版や検索用 index は作ってよい。ただし、それらは派生物であり、Personal Archive の原本ではない。

## 4. Creative Knowledge

Creative Knowledge は、作品、作者、関係者、制作物を扱う知識層である。

対象:

- 映画。
- 音楽。
- 小説。
- 漫画。
- その他、タイトル、制作物、関係者が存在する文化的・創作的対象。

目的:

- RenCrow の会話の元にする。
- 雑談、推薦、連想、IdleChat、創作、比較、深掘りに使う。
- 作者名、監督名、出演者名、作曲者名、シリーズ、ジャンル、内容、テーマから関連作品を引けるようにする。

方針:

- 自動で取得し続ける。
- 形式は固定しない。
- 単なる全文ログではなく、作品、人物、シリーズ、テーマ、内容特徴、関係性をたどれる構造を持つ。
- 作品知識とニュース系情報を混ぜない。

## 5. News / Current Affairs

政治、経済、芸能、Tech などのニュース系情報は、Creative Knowledge と似ているが、時点依存が強いため分けて扱う。

ニュース系は、次の 3 層に分ける。

| 層 | 役割 | 例 |
| --- | --- | --- |
| Canonical Knowledge | 比較的長く使える基礎知識 | 政治制度、経済指標の意味、企業、技術、製品カテゴリ |
| News Event Log | 時点付きの一次ニュース | 発表、決算、事件、政策、リリース、脆弱性、受賞 |
| Derived Current View | 会話用の現在ビュー | 最新状況、この数日の流れ、現在確度高く言えること |

方針:

- ニュース記事を永続的真実として保存しない。
- ニュースは必ず `published_at`、`source`、`source_url`、`retrieved_at`、`topic`、`entities`、`event_type` を持つ。
- 古いニュースを現在の事実として使わない。
- Current View は派生物であり、原本ではない。
- ニュースから抽出された安定事実だけを Canonical Knowledge に昇格する。

## 6. User-Directed Daily Intake

User-Directed Daily Intake は、ユーザーが指示した情報を毎日取得する領域である。

これは通常のニュース取得よりも優先度が高い。ユーザーが指定した情報源やテーマは、ユーザーのアイデア元であると同時に、RenCrow 自身のアイデア元でもある。

対象:

- ユーザーが指定した情報源。
- ユーザーが指定した人物、作品、分野、テーマ。
- 毎日追跡すべき topic。
- RenCrow の改善に使える観測対象。

役割:

- ユーザーの発想源になる。
- RenCrow の発想源になる。
- RenCrow の改善候補を生む。
- 日次の観測、学習、発想の入口になる。

保存時の分離:

| 種別 | 役割 |
| --- | --- |
| raw intake | 毎日取得した原本 |
| daily digest | その日の要約 |
| idea seeds | 発想に使える断片 |
| improvement candidates | RenCrow 自身の改善候補 |
| confirmed knowledge / memory | 検証・採用されたもの |

RenCrow はこの情報をもとに改善案を出す義務を持つ。ただし、改善実装は勝手に行わず、提案、仕様化、承認を経る。

## 7. Temporal Memory Gradient

記憶は短期と長期の二分ではなく、時間方向のグラデーションを持つ。

想定する層:

| 層 | 役割 | prompt との関係 |
| --- | --- | --- |
| 超短期記憶 | 今まさに必要な会話文脈 | SystemPrompt の次に置く候補 |
| スレッド内記憶 | 現在 thread の文脈 | thread continuation に使う |
| 当日記憶 | 今日の会話、観測、取得情報 | 日中の連続性に使う |
| 3日記憶 | 直近数日の関心、未完了、連続話題 | 忘れやすいが重要な流れ |
| 1週間記憶 | 週単位の関心、反復、作業テーマ | topic continuity に使う |
| 1か月記憶 | 月単位の傾向、プロジェクト、関心 | 中期 memory |
| 年単位記憶 | 長期傾向、価値観、重要事実 | long-term memory |
| 原本 archive | 保護された一次情報 | prompt へ直接全投入しない |

方針:

- どの層からでもアクセス可能であることが望ましい。
- ただし prompt には必要なものだけを選んで入れる。
- 長期記憶は利用頻度、重要度、ユーザー明示性、再利用価値を考慮して残す。
- 忘却は原本削除ではなく、基本的にはアクセス優先度の低下として扱う。

## 8. Biological Memory Model

長期記憶は、生物的な記憶に近い挙動を目指す。

必要な性質:

- 利用頻度が高いものを残す。
- 重要度が高いものを残す。
- ユーザーが明示したものを強く残す。
- 繰り返し出てくる話題を強化する。
- 古いが重要なものは消さず、検索優先度を調整する。
- 一時的な気分、推測、未検証情報を長期事実にしない。

記憶の種類は分ける。

| 種類 | 例 | 扱い |
| --- | --- | --- |
| ユーザーの事実 | 生い立ち、BIO、明示設定 | 原本保護、強い保持 |
| ユーザーの好み | 好きな作品、文体、関心 | 更新可能、履歴つき |
| 一時的状態 | 今日の気分、短期作業 | 低い永続性 |
| RenCrow の推測 | 推定嗜好、仮説 | confirmed にしない |
| RenCrow の自己記憶 | 過去の判断、失敗、保留 | OperationMemory として分離 |

## 9. Dream Consolidation

夢は、単なる夜間バッチ要約ではない。

RenCrow における dream / consolidation は、次を担う。

- その日の会話、ニュース、取得情報、作業ログを再編成する。
- Personal Archive に関係する情報を勝手に上書きせず、追記候補として扱う。
- Creative Knowledge と News Event Log から関連を見つける。
- User-Directed Daily Intake から idea seeds を抽出する。
- RenCrow 自身の改善候補を生成する。
- 重複、矛盾、古い Current View を検出する。
- 長期記憶へ昇格する候補を作る。

夢の出力は、直接 confirmed memory ではない。

想定する出力:

- daily reflection。
- memory promotion candidates。
- idea seeds。
- improvement candidates。
- contradiction notes。
- stale current view warnings。
- next actions proposal。

## 10. 会話注入方針

記憶と知識は、常に全部使うのではなく、必要なときに引く。

ただし、超短期記憶は prompt layout 上で強い位置を持つ。SystemPrompt の直後に置く候補として扱う。

注入時に分けるもの:

- raw original。
- compressed summary。
- current view。
- recall candidate。
- prompt context。
- Viewer 表示用 view。

Personal Archive の原本を、会話 prompt に無制限に直接注入してはいけない。

## 11. 所有者と権限

Personal Archive の所有者はユーザーである。

RenCrow ができること:

- 読む。
- 参照候補を出す。
- 追記候補を提案する。
- 圧縮版や検索 index を作る。
- 矛盾や不足を報告する。

RenCrow が勝手にしてはいけないこと:

- 原本を上書きする。
- 原本を削除する。
- 推測をユーザーの事実として確定する。
- 未検証外部情報を confirmed memory にする。

## 12. 未確定の入口質問

今後、詳細仕様へ進む前に決めるべきこと。

1. Personal Archive に対して、RenCrow は追記提案までできるのか、読むだけなのか。
2. 忘却はアクセス優先度低下だけか、明示削除可能な領域も持つのか。
3. Dream は要約だけでなく、連想、仮説生成、改善案生成まで含めるのか。
4. User-Directed Daily Intake は一般ニュースより常に優先するのか。
5. Personal Archive はどの条件で prompt context に出すのか。
6. ユーザーの事実、好み、一時的状態、RenCrow の推測をどの schema で分けるのか。
7. 低信頼ソースを発想源として使うことを許容するのか。
8. RenCrow 自身の自己記憶を OperationMemory としてどこまで持つのか。

## 14. 実装状況

MVP として、知識・記憶拡張の台帳入口は実装済み。

実装済み:

- `internal/domain/knowledgememory` に `PersonalArchiveEntry` / `CreativeKnowledgeItem` / `NewsKnowledgeItem` / `DailyIntakeRule` / `TemporalMemoryMarker` / `DreamConsolidationRun` と validation を追加。
- Personal Archive 原本は `protected=true` を必須化。
- Dream Consolidation は `review_status=approved` での自動登録を拒否。
- Temporal Memory layer は `thread` / `today` / `3days` / `week` / `month` / `year` / `long_term` に限定。
- `internal/infrastructure/persistence/knowledgememory` に JSONL store と SQLite store を追加。
- `knowledge_memory.*` config を追加し、Personal Archive 保護と Dream Human Review を必須にした。
- `knowledge_memory.storage` / `sqlite_path` により、runtime で JSONL / SQLite store を切り替えられる。
- `/viewer/knowledge-memory`、`/viewer/knowledge-memory/personal-archive`、`/viewer/knowledge-memory/creative-knowledge`、`/viewer/knowledge-memory/news-knowledge`、`/viewer/knowledge-memory/daily-intake-rules`、`/viewer/knowledge-memory/temporal-markers`、`/viewer/knowledge-memory/dream-runs` を追加。
- Viewer Ops に `Knowledge Memory` summary を追加。
- Daily Intake Rule の `active` / `reviewed` / `enabled` かつ URL `source_hint` を、enabled Source Registry entry へ同期し、既存 Source Registry sweeper へ渡す自動 job を追加。
- Dream Consolidation proposal API を追加し、既存の Personal Archive / Creative / News / Temporal marker から `IdeaSeeds` を生成して `review_status=pending` の Dream Run として保存する。
- Dream Consolidation review API を追加し、pending proposal を `approved` / `rejected` としてレビューし、必要な場合は Dream Run の状態だけを `promoted` にできる。
- Dream Run の `promoted` は正式 memory / Source Registry への自動昇格ではない。`auto_promote=false` を維持し、正式化は既存の staging / validator / promote 経路へ分ける。

未実装 / 残作業:

- Knowledge Memory 保存時の L1 staging pending candidate 登録と、News URL / Daily Intake Rule の URL source_hint の disabled Source Registry candidate 登録は追加済み。
- Memory タブ内に Knowledge Memory Ledger を追加し、Personal Archive / Creative Knowledge / News Knowledge / Daily Intake Rule / Temporal Marker / Dream Run を同じ台帳で確認し、`detail_type` + `id` の detail API を直接開けるようにした。
- Memory タブ内の Knowledge Memory Ledger に type / review state / flag の横断フィルタを追加し、original / protected / compressed / warning を行フラグとして確認できるようにした。
- Detail 表示では Original / Protected、Compressed / Summary、Warning / Review を分けて表示し、原本保護・圧縮要約・warning metadata・review status を JSON 原文と並べて比較できる。
- Source Registry staging と Knowledge Memory Ledger の相互関連表示を追加し、Knowledge 側から関連 staging、staging 側から関連 Knowledge item を確認できるようにした。
- Detail 表示に Review / Promote Comparison を追加し、warning 数、protected original、関連 staging の validated 状態、promote blocker を本文とは分けて確認できるようにした。
- `/viewer/knowledge-memory/review` と Memoryタブの Review / Promote / Reject 操作により、Creative / News / Daily Intake の review / promote 状態遷移と正式 promote 先との差分確認用 comparison を確認できる。promote は Knowledge Memory 内の状態遷移に限定し、正式 memory / Source Registry へは自動昇格しない。
- Creative Knowledge の外部取得 / 関連作品探索。
- Dream Consolidation の review / promote 状態記録は実装済み。正式 memory / Source Registry への自動昇格は行わない。
- Memory Viewer での原本 / 圧縮版 / current view / prompt context の詳細分離表示。

この文書は、知識・記憶システムを再設計するための入口構想である。実装状況は MVP の台帳入口であり、正式な自動取得・昇格・Dream 実行の完了宣言ではない。
