3. 主要メッセージフロー

ユーザー入力は Adapter から MessageOrchestrator へ入り、Mio が route を決めます。CODE 系では Coder が候補を作り、Worker が実行します。

fallback は正常系ではありません。分類失敗や provider 失敗などの回復経路として、ログと検証結果で区別します。

4. Chat / Worker / Coder 詳細図

Chat は対話と判断、Worker は実行、Coder は提案生成です。Coder が破壊的操作を直接行う構造は仕様違反です。

Chat: Mioユーザー対話、route 判断、結果返却。Coder へ直接指示しない。
Worker: Shiro / Worker Core実行可否判断、Coder 選定、patch / command 実行、安全境界。
Coder: AO / Aka / Kinplan / patch / proposal を生成。採用・実行はしない。

Worker と Coder ができること

Worker ができること 実行可否判断、Coder 選定、proposal 検証、protected file / workspace / policy 確認、file / shell / git / test 実行、patch 適用、report / event / log 記録。
Coder ができること plan / patch / proposal / risk / cost hint の生成、設計説明、複数案提示。成果物は実行命令ではなく候補として扱う。
Coder ができないこと 採用決定、正本ブランチ管理、patch 適用、file / shell / git 実行、破壊的操作の直接実行。
境界守ること主な実装

6. IdleChat / STT / TTS / Audio 詳細図

IdleChat は自律会話、STT は通常 chat 入力、TTS は音声出力です。STT 入力を IdleChat に直接流さないこと、TTS chunk を本文表示の根拠にしないことが重要です。

10. 仕様変更時の見方

仕様を変えるときは、まず「どのモジュールが主担当か」を決めます。1 対 1 で説明できない場合は、実装前に分割・再設計候補として記録します。

禁止fallback を正常系として隠す。
禁止Viewer 表示、音声、口パク、ログを同じ契約にする。
禁止Coder が破壊的操作を直接実行する。
