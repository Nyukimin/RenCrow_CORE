# 80 音声入力 同期 / 非同期境界仕様

**作成日**: 2026-06-15
**ステータス**: 設計仕様
**関連**: `74_Viewer音声直結LLM_WS契約.md` / `77_STT音声_LLM音声_命名と経路仕様.md` / `79_LLM音声_発話区間判定仕様.md`

---

## 1. 目的

Viewer Chat の音声入力は、マイク、VAD、RenCrow_STT、RenCrow_LLM、Chat 表示、EventHub、IdleChat、TTS、永続ログが連鎖する。

これらを 1 本の同期処理として扱うと、ログ保存、Monitor 更新、TTS、IdleChat などの副作用が詰まっただけで、ユーザーが待っている音声応答まで止まる。

本仕様は、音声入力まわりだけを対象に、同期パスと非同期パスの境界を明確にする。

---

## 2. 対象範囲

対象:

- Viewer Chat mic から始まる STT音声
- Viewer Chat mic から始まる LLM音声
- `/stt`
- `/voice-chat`
- `ProcessVoiceDirect`
- Viewer Chat へのユーザー発話 / Mio 応答表示
- 音声入力に付随する EventHub / session log / Monitor / metrics / IdleChat / TTS 連携

対象外:

- IdleChat 本体の会話生成仕様
- TTS の音声合成品質仕様
- RenCrow_LLM のモデル選定
- RenCrow_STT の認識品質チューニング

---

## 3. 用語

| 用語 | 意味 |
| --- | --- |
| 音声入力正本 | 1 発話に対して生成される、後続処理の基準となる結果 |
| 同期パス | ユーザーが応答を待っているため、完了・失敗を即時に返す必要がある処理 |
| 非同期パス | 遅延・失敗しても、音声入力の完了を阻害してはいけない副作用 |
| 副作用 | ログ保存、Monitor 更新、metrics 保存、TTS 再生、IdleChat 後処理など |
| VoiceInputResult | 音声入力正本を表す抽象結果 |
| VoiceInputSideEffects | VoiceInputResult 後に非同期で走る副作用群 |

---

## 4. 音声入力正本

音声入力の成功判定は、EventHub 配送やログ保存ではなく、`VoiceInputResult` の生成で判断する。

### 4.1 STT音声の正本

STT音声では、正本は RenCrow_STT の final text である。

```text
VoiceInputResult {
  mode: "stt",
  utterance_id,
  session_id,
  user_text,
  reply: optional,
  timings,
  source: "RenCrow_STT final"
}
```

STT音声では、STT final text を通常 Chat 入力として RenCrow_LLM に渡す。Mio 応答は通常 Chat 処理の結果として別途生成される。

### 4.2 LLM音声の正本

LLM音声では、正本は RenCrow_LLM audio input の `llm.final` である。

```text
VoiceInputResult {
  mode: "llm",
  utterance_id,
  session_id,
  user_text,
  reply,
  timings,
  source: "RenCrow_LLM llm.final"
}
```

LLM音声で構造化 JSON が返る場合:

```json
{
  "user_text": "ユーザー発話の復元文",
  "reply": "Mioとしての返事"
}
```

- `user_text` はユーザー発話として Chat に表示する。
- `reply` は Mio 応答として Chat に表示する。
- JSON 全体をそのまま Chat 表示正本にしてはいけない。
- `user_text` が空でも `reply` がある場合は、Mio の聞き返しとして扱う。

## 5. 同期パス

同期パスは、ユーザーの音声入力に対する最小の応答成立までに必要な処理だけで構成する。

### 5.1 STT音声

```text
Viewer mic
  -> VAD / 発話区間確定
  -> /stt
  -> RenCrow_STT
  -> STT final
  -> Chat text input
  -> Mio reply
  -> Viewer Chat display event
```

同期パスに含める処理:

- マイク入力開始
- 発話区間確定
- STT final text の取得
- Chat 入力への投入
- Mio 応答取得
- Viewer Chat に必要な最小 event 発行

### 5.2 LLM音声

```text
Viewer mic
  -> VAD / 発話区間確定
  -> /voice-chat session.start
  -> PCM16 streaming
  -> session.commit
  -> RenCrow_LLM input_audio / audio session
  -> llm.final
  -> VoiceInputResult
  -> Viewer Chat display event
```

同期パスに含める処理:

- マイク入力開始
- 発話区間確定
- `/voice-chat` session 管理
- RenCrow_LLM audio input 呼び出し
- `llm.final` の取得
- `VoiceInputResult` 生成
- Viewer Chat に必要な最小 event 発行

---

## 6. 非同期パス

以下は非同期副作用であり、同期パスを待たせてはいけない。

| 処理 | 非同期である理由 |
| --- | --- |
| session jsonl 保存 | ファイル I/O 遅延で音声応答を止めないため |
| EventLogStore 追記 | 永続ログは証跡であり、応答成立の正本ではないため |
| Viewer Monitor 更新 | Ops 表示更新であり、ユーザー応答に必須ではないため |
| latency metrics 保存 | 観測用であり、応答成立に必須ではないため |
| debug trace 保存 | 調査用であり、応答成立に必須ではないため |
| TTS 再生 | 表示後の出力副作用であり、Chat 応答確定とは別であるため |
| IdleChat 停止 / 再開後処理 | 競合制御であり、音声結果生成の正本ではないため |
| Ops / Timeline 補助表示 | 可視化であり、音声入力正本ではないため |

非同期副作用は失敗しても、同期パスの成功を失敗扱いにしてはいけない。

ただし、失敗はログまたは debug trace に残す。

---

## 7. 禁止する結合

次の結合は禁止する。

| 禁止事項 | 理由 |
| --- | --- |
| EventHub 配送完了を音声入力成功条件にする | 遅い Viewer / SSE / client に音声応答が巻き込まれる |
| EventLogStore Append 完了を `ProcessVoiceDirect` の次イベント条件にする | ファイル I/O が Chat 応答を止める |
| Monitor 更新完了を Chat 応答表示条件にする | Ops 表示とユーザー応答の責務が違う |
| TTS 再生 ACK を Mio 応答確定条件にする | 音声再生とテキスト応答は別ライフサイクル |
| IdleChat 停止完了を LLM音声 `session.start` の必須前提にする | IdleChat 制御の遅延でユーザー入力を落とす |
| STT音声 final と LLM音声 `llm.final` を同じ event 名で扱う | 正本が異なる |
| LLM音声の JSON 応答全体を Chat 本文に表示する | `user_text` と `reply` の役割が混ざる |

## 8. 必須モジュール境界

音声入力まわりは、次の境界に分ける。

### 8.1 VoiceInputSession

責務:

- `utterance_id`
- `session_id`
- 発話状態
- sample rate / channels / format
- 発話開始 / commit / cancel
- 発話区間タイミング

禁止:

- EventHub へ直接 emit すること
- session jsonl へ直接書くこと
- IdleChat / TTS を直接操作すること

### 8.2 VoiceInputGateway

責務:

- STT音声: RenCrow_STT へ PCM を送る
- LLM音声: RenCrow_LLM へ PCM / WAV / audio input を送る
- partial / final / error を正規化する

禁止:

- Viewer DOM 表示を知ること
- IdleChat / TTS を操作すること
- 永続ログ保存を成功条件にすること

### 8.3 VoiceInputResultBuilder

責務:

- STT final または LLM `llm.final` から `VoiceInputResult` を生成する
- LLM音声 JSON の `user_text` / `reply` を分離する
- no-audio / meta 応答を検出し、通常応答と区別する

禁止:

- JSON 全体をそのまま Chat 本文へ流すこと
- user_text と reply を同一 role に混ぜること

### 8.4 VoiceInputPublisher

責務:

- `VoiceInputResult` を Viewer Chat 表示用 event に変換する
- 最小限の `message.received` / `routing.decision` / `agent.response` を発行する
- 同期パスに必要な event のみ扱う

禁止:

- 永続ログ保存を待つこと
- Monitor 更新を待つこと
- TTS 再生完了を待つこと

### 8.5 VoiceInputSideEffects

責務:

- session log
- EventLogStore
- Monitor
- metrics
- debug trace
- TTS
- IdleChat 後処理

禁止:

- 同期パスをブロックすること
- 副作用失敗を `VoiceInputResult` 失敗として扱うこと

## 9. EventHub の扱い

EventHub は Viewer への live event 配送機構であり、音声入力正本ではない。

EventHub への emit は同期パスで許可する。ただし、次を守る。

- client 送信は non-blocking であること。
- 遅い client は drop してよい。
- EventHub history 追加は軽量であること。
- 永続ログ保存は EventHub live 配送後の非同期副作用とする。
- EventHub が詰まった場合でも、VoiceInputResult は生成済みとして扱える設計にする。

---

## 10. IdleChat との境界

音声入力開始時、IdleChat は停止対象である。

ただし、IdleChat 停止は「ユーザー入力の競合制御」であり、音声入力正本の生成条件ではない。

### 10.1 必須動作

- Chat テキスト入力開始時、IdleChat 停止要求を出す。
- マイク音声入力開始時、IdleChat 停止要求を出す。
- LLM音声 `session.start` 時、`chat_busy=true` を通知する。
- `llm.final` / error / cancel / close 後、`chat_busy=false` を通知する。

### 10.2 禁止動作

- IdleChat 停止 ACK を待って `/voice-chat` 開始を遅らせる。
- IdleChat TTS drain を待って user speech を捨てる。
- IdleChat の TTS audio chunk を LLM音声入力の一部として扱う。

---

## 11. TTS との境界

TTS は Mio 応答後の副作用である。

### 11.1 必須動作

- Mio 応答テキストが確定した後に TTS 対象にする。
- TTS 再生中でも、barge-in 条件を満たすユーザー発話は LLM音声として受け付ける。
- barge-in 時は TTS 停止要求を出す。

### 11.2 禁止動作

- TTS 再生 ACK を Chat 応答確定条件にする。
- TTS drain 完了を次の音声入力受付条件にする。
- TTS failure を `llm.final` failure として扱う。

---

## 12. ログと観測

ログは原因切り分けのため、同期パスと非同期パスを分けて出す。

### 12.1 同期パスログ

必須:

- `session.start`
- first audio received
- `session.commit`
- upstream request start
- upstream final received
- `VoiceInputResult` generated
- Chat publish start
- Chat publish completed

LLM音声では最低限、次の対応が 1 utterance で追えること。

```text
voice-chat session.start utterance_id=...
voice-chat session.commit utterance_id=...
voice-chat input_audio finalized utterance_id=... text_len=...
voice-chat ProcessVoiceDirect completed utterance_id=... elapsed_ms=...
```

### 12.2 非同期パスログ

必須:

- side effect name
- utterance_id
- success / failure
- elapsed_ms

非同期副作用失敗時は、同期パスを失敗にせず、警告として残す。

---

## 13. エラー扱い

同期パスのエラー:

- マイク取得不可
- WebSocket 接続不可
- upstream RenCrow_STT / RenCrow_LLM 不可
- audio empty
- final empty
- malformed `llm.final`
- no-audio / meta 応答
- timeout

非同期パスのエラー:

- session log 書き込み失敗
- EventLogStore 書き込み失敗
- Monitor 更新失敗
- metrics 保存失敗
- TTS 失敗
- IdleChat 停止 / 再開通知失敗

同期パスのエラーは Viewer に表示する。

非同期パスのエラーは Viewer debug / Ops / log に表示してよいが、音声入力そのものを成功から失敗へ変更してはいけない。

## 14. テスト要件

### 14.1 ユニット / 契約テスト

| テスト | 期待 |
| --- | --- |
| VoiceInputResultBuilder parses JSON | `user_text` と `reply` が分離される |
| no-audio/meta response is rejected | upload 要求などを Chat 正本にしない |
| EventLogStore slow append does not block publish | Chat publish は完了する |
| Monitor slow update does not block publish | `agent.response` は遅延しない |
| IdleChat busy starts/ends | `session.start` で true、final/error/cancel/close で false |
| TTS failure does not fail VoiceInputResult | Chat 応答は成立する |

### 14.2 E2E テスト

| テスト | 期待 |
| --- | --- |
| LLM音声 E2E | Browser mic -> `/voice-chat` -> `llm.final` -> Chat |
| STT音声 E2E | Browser mic -> `/stt` -> STT final -> Chat |
| fake mic LLM音声 E2E | fake audio で `llm.final` と Chat event を確認 |
| slow side effect E2E | ログ保存を遅延させても Chat 応答が返る |
| IdleChat active 中の音声入力 | IdleChat が停止要求され、音声入力が優先される |
| TTS 再生中の barge-in | TTS が止まり、新規音声入力が処理される |

### 14.3 報告時の必須項目

E2E 報告では次を明記する。

- 実マイクか fake mic か
- `voice_input_mode`
- STT音声 / LLM音声のどちらか
- `utterance_id`
- `session_id`
- `session.start` 時刻
- `session.commit` 時刻
- final 到達時刻
- Chat publish 完了時刻
- side effect failure の有無

## 15. 実装移行方針

### Phase 1: 境界固定

- 現行 `/voice-chat` / `/stt` の動作を維持する。
- 同期パスログを追加する。
- EventLogStore / Monitor / metrics / TTS / IdleChat 後処理を同期パスから外す。
- `viewer_session_id` を LLM音声で `viewer` に揃える。

### Phase 2: VoiceInputResult 導入

- STT音声 / LLM音声の result model を共通化する。
- `user_text` / `reply` の分離を result model 側へ移す。
- Viewer JS に散らばった表示判断を Publisher へ寄せる。

### Phase 3: SideEffects 分離

- `VoiceInputSideEffects` queue を作る。
- session log / EventLogStore / Monitor / metrics / TTS / IdleChat を個別 worker に分ける。
- side effect ごとに timeout と failure log を持つ。

### Phase 4: 回帰防止

- slow side effect test を追加する。
- LLM音声 E2E / STT音声 E2E を CI または手元 runbook 化する。
- 仕様変更時は `77` / `79` / 本仕様を同時に更新する。

## 16. 判定基準

本仕様に沿っている状態:

- 音声入力の成功判定が `VoiceInputResult` で説明できる。
- Chat 表示に必要な event が副作用 I/O 待ちで止まらない。
- STT音声と LLM音声の正本が混ざらない。
- IdleChat / TTS / Monitor / EventLogStore の失敗が音声入力応答を壊さない。
- ログで同期パスと非同期パスのどちらが詰まったか判定できる。

本仕様に反している状態:

- `input_audio finalized` 後に、ログ保存や Monitor 更新待ちで `agent.response` が遅れる。
- EventHub client の遅延で `ProcessVoiceDirect` が進まない。
- LLM音声の JSON 全体が Mio 応答として表示される。
- session log に書けないだけで Chat 応答が失敗する。
- IdleChat/TTS の drain 待ちでユーザー音声入力を捨てる。
