# 新規リポジトリ移行方針

## 目的

この文書は、RenCrow を新規リポジトリへ移すときに持っていく仕様群と、持っていかない旧文書を整理するための方針である。

## 持っていく仕様入口

新規リポジトリでは `docs/10_新仕様/` を仕様入口にする。

持っていく対象:

- `docs/10_新仕様/00_README.md`
- `docs/10_新仕様/01_新仕様_概要.md`
- `docs/10_新仕様/02_モジュール構成仕様.md`
- `docs/10_新仕様/04_Chat_Worker_Coder仕様.md`
- `docs/10_新仕様/05_Viewer仕様.md`
- `docs/10_新仕様/06_IdleChat仕様.md`
- `docs/10_新仕様/07_STT_TTS仕様.md`
- `docs/10_新仕様/08_LLM_provider仕様.md`
- `docs/10_新仕様/09_Memory_SourceRegistry仕様.md`
- `docs/10_新仕様/10_検証仕様.md`
- `docs/10_新仕様/11_分割再設計候補.md`
- `docs/10_新仕様/12_新規リポジトリ移行方針.md`
- `docs/10_新仕様/13_実装項目インベントリ.md`

## 参考として持っていく候補

新規リポジトリでも履歴確認が必要な場合に限り、次を reference として持っていく。

- `AGENTS.md`
- `CLAUDE.md`
- `rules/`
- `docs/refactor/リファクタリング指針.md`
- `docs/refactor/フォルダ構成方針.md`
- `docs/refactor/検証方針.md`

## 持っていかない旧 docs

原則として、新規リポジトリの仕様入口には次を持ち込まない。

- 旧 `docs/archive/`
- 古い Phase ごとの作業プロンプト
- 実装済みリファクタリングの中間計画
- 現在実装とズレた古い仕様断片
- codebase-map の古い静的解析結果

これらは削除済みであり、必要な内容は正本仕様または `docs/10_新仕様/` へ統合してから参照する。

## docs/01_正本仕様 との関係

旧リポジトリでは `docs/01_正本仕様/` が履歴上の正本仕様である。

新規リポジトリでは、`docs/10_新仕様/` を整理済み正本候補として扱う。`docs/01_正本仕様/` をそのまま持っていく場合は、重複と古い実装前提が混ざるため、一次参照から外す。

旧 docs から現行実装として必要な情報は、`13_実装項目インベントリ.md` と該当する `docs/10_新仕様/` の仕様文書へ反映してから移行する。旧 docs をそのまま現行仕様の代替として使わない。

## 移行時の注意

- repo example と live runtime config を混同しない。
- 新規リポジトリの config example は、移行先の runtime に合わせて別途確認する。
- Viewer / IdleChat / STT / TTS は実ブラウザまたは同等 E2E で確認する。
- fallback を正常系として扱わない。
- Source Registry を無審査で正式 memory へ昇格しない。

## 移行後の仕様更新ルール

仕様変更時は、まず `docs/10_新仕様/` の該当文書を更新する。

仕様と実装箇所が 1 対 1 に説明できない場合は、`11_分割再設計候補.md` に記録し、実装前に仕様単位またはモジュール境界を再設計する。
