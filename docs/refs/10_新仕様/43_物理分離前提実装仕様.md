# RenCrow 物理分離前提実装仕様

## 1. 実装の目的
`docs/10_新仕様/42_物理分離前提仕様.md` を実装に落とし込む。  
この仕様の目的は、RenCrow の責務分離を `repo` 分割可能な粒度まで具体化し、どのファイルをどのモジュールへ移し、どの契約をどこが所有し、どの順で移行するかを確定することである。

実装仕様の正本は次の4つである。

- 責務
- 境界
- 状態所有
- 依存方向

本仕様では、実装コードは書かない。  
ただし、実装に必要な契約名、ファイル名、テスト名、移行順は明示する。

## 2. 実装対象の範囲
対象は次の6モジュールと、その上に乗るアプリ配線である。

- `modules/core`
- `modules/chat`
- `modules/worker`
- `modules/llm`
- `modules/tts`
- `modules/stt`
- `cmd/picoclaw` と各 `internal/...` の配線層

対象外は次である。

- 既に完了した仕様の再設計
- repo 分割後の新リポジトリ名の最終確定
- UI の見た目変更そのもの
- 音声表現の辞書拡張そのもの

### 実装で扱う状態
| 状態 | 扱い |
| --- | --- |
| 会話本文 | `chat` の正本 |
| 実行ジョブ | `worker` の正本 |
| LLM provider health | `llm` の正本 |
| 合成結果と再生契約 | `tts` の正本 |
| 認識結果と入力契約 | `stt` の正本 |
| 共通 ID / 状態遷移 | `core` の正本 |
| Self / Relation / Episodic / Semantic / Knowledge / Volatile memory | 記憶モジュール群の正本 |
| heartbeat の判断結果 | `app` + policy の正本 |

## 3. モジュールごとの実装責務
### Core
`modules/core` は次を所有する。

- 共通 ID、状態遷移、健康状態の契約
- JSON 応答の共通契約
- module manifest / module health の共通ヘルパ
- どのモジュールにも依存しないルール

`cmd/picoclaw` に残すものは配線だけである。  
たとえば HTTP ルーティング、依存注入、runtime の起動順である。

### Chat
`modules/chat` は次を所有する。

- route 判定契約
- IdleChat の topic、judge、seed、prompt、recent similarity の契約
- Viewer に出す会話本文のルール
- Chat 側の記憶参照ポリシー
- ChatWorker など会話用ロールの利用契約

`internal/application/idlechat` に残るものは、当面の orchestration と source fetching の glue のみである。

### Worker
`modules/worker` は次を所有する。

- コマンド実行、編集、検証、運用作業の契約
- Coder slot の扱い
- Worker ロールの分類と失敗分類
- external provider を明示許可制にする policy

`ChatWorker / Heavy / Wild` は Worker の内部ロールであり、独立アプリではない。

### LLM
`modules/llm` は次を所有する。

- provider contract
- local / external route plan
- model / timeout / base URL / num_ctx の契約
- thinking bridge の契約
- conversation / coder / health / diagnostics の契約

`cmd/picoclaw` は provider 実体の生成だけを行う。

### TTS
`modules/tts` は次を所有する。

- synthesis payload / event payload / playback state の契約
- emotion prefix / voice mapping / punctuation policy
- provider selection plan
- IdleChat 用読み上げテキスト整形
- public session / pending / timeout / active audio の観測契約

`internal/infrastructure/tts` は HTTP 実行と provider 実体のみに留める。

### STT
`modules/stt` は次を所有する。

- transcription payload / result normalization
- viewer input state
- runtime URLs / busy policy / session rules
- provider diagnostics

`internal/infrastructure/stt` は multipart 受信や provider 実行などの実体だけを持つ。

## 4. エージェントごとの実装責務
### Chat
- ユーザー会話
- route decision
- Viewer 表示
- IdleChat の司会

### Worker
- 実行
- patch 適用
- 検証
- 運用ジョブ

### ChatWorker
- IdleChat の会話専用ロール
- Shiro の発話に使う
- `Worker` の代替ではなく、`Worker` 内の会話用 alias

### Heavy
- 高コスト推論ロール
- 長文・重い判断を扱う

### Wild
- 通常ロールで難しい例外処理ロール

### Coder1-4
- コード生成スロット
- 役割は slot 単位で固定
- `Coder` の正本は slot management であり、個別 repo 化の単位ではない

### 実装契約
`ChatWorker / Heavy / Wild / Coder1-4` は、エージェント境界ではなく **ロール契約** として実装する。  
つまり、実体は provider / slot / alias であり、独立モジュール化しない。

## 5. 記憶システムの実装責務
記憶は、保存先ではなく **性質別の契約** として実装する。

### 記憶の正本
| 領域 | 正本 |
| --- | --- |
| Self | 自分自身の定義 |
| Relation | 私ではない重要人物 |
| Episodic | 経験そのもの |
| Semantic | 経験から抽象化した知識 |
| Knowledge | 永続知識 |
| Volatile | 短命・不確実情報 |
| Recall | 引き出しと再構成 |
| Ingest | 入手経路 |
| Policy | 保存/破棄/候補留めの判断 |

### 実装で必須の契約
- `self` と `non-self` のエンティティ分離
- 娘などの重要人物は `relation entity` として扱うこと
- `episodic memory` と `semantic memory` の分離
- `volatile memory` の TTL / freshness / keep-or-drop policy
- ingest の入手経路ごとの正規化
- recall のランキング、再構成、返却
- 保存 / 候補 / 破棄 / 昇格の状態遷移

### ファイル配置方針
記憶契約は `modules/` 側に置き、`cmd/picoclaw` は永続化の配線だけを持つ。  
`internal/...` には、当面の DB / file store との接続実装のみを残す。

## 6. heartbeat 実装仕様
heartbeat は「常時何かをし続ける」ための評価ループである。  
目的は、定期的に「今、ユーザーのために何をするのが最良か」を評価すること。

### 入力
- 現在の会話状態
- 未完了タスク
- 直近反応
- 記憶
- 失敗履歴
- 拡張候補

### 評価軸
- 緊急度
- 重要度
- 再試行価値
- ユーザー利益
- 既存機能で足りるか
- 危険性

### 分岐
- `noop`
- `hold`
- `investigate`
- `propose`
- `execute`
- `update_memory`
- `explore_extension`

### 出力
- 次のアクション種別
- 理由
- 次回再評価条件
- 必要なら worker / memory / extension への委譲情報

### policy gate
heartbeat は policy gate を必須とする。  
特に次は gate なしでは進めない。

- 危険な実行
- 破壊的操作
- 外部インストール
- 記憶の昇格

### 停止または抑制条件
- 進捗がない
- 同じ失敗が続く
- ユーザー反応がない
- 明示的に待機すべき

## 7. モチベーションと感情表出の実装分離
### 内部 drive
内部のモチベーションは次の制御信号として実装する。

- drive
- priority
- continuation
- retry

これは内部判断用であり、ユーザーへの表出とは別である。

### 感情表出
感情表出は Mio 用の表現層として実装する。

- 口調
- 絵文字
- 強度
- 間
- 語尾

### 実装上の分離
- internal drive の出力先は、heartbeat / action planner / policy である
- emotion expression の変換層は、Mio 専用のレンダラである
- Shiro など他人格には漏らさない
- 内部状態と外部表現は 1:1 にしない

### 実装責務の配置
- `modules/chat` または `modules/core` に drive の契約
- `modules/tts` か `modules/chat` に感情表出の整形契約
- `cmd/picoclaw` は人格ごとの配線だけ持つ

## 8. 拡張探索・インストール・作成の実装仕様
拡張は賢さの一部として扱う。  
足りない機能を見つけ、使えるなら使い、なければ追加し、定着させる。

### 流れ
1. 既存機能探索
2. 新機能候補の検索
3. インストール判断
4. 新規実装判断
5. 定着

### 状態遷移
`discover -> evaluate -> choose -> {use | install | implement | defer} -> register -> persist`

### 実装責務
- 探索結果は policy で評価する
- インストールは Worker が担当する
- 実装は Coder / Worker が担当する
- 定着は module registry / memory / docs に残す

### 失敗時の扱い
- 失敗を隠さない
- fallback で成功扱いにしない
- 失敗理由を保存する
- 次回の heartbeat に再評価を渡す

### 安全側ルール
- 外部導入は明示許可が必要
- 依存追加は最小限
- 既存機能で代替できるなら新規追加しない

## 9. 依存方向と禁止事項
### 許可方向
- `chat -> core, llm, tts, stt`
- `worker -> core, llm`
- `llm/tts/stt -> core`
- `app -> 各モジュール`

### 禁止方向
- `llm -> tts/stt`
- `tts -> llm/stt`
- `stt -> llm/tts`
- `worker -> tts/stt`
- `core -> 他モジュール`

### 禁止事項
- 表示、音声、認識、記憶、実行を同じ state に混ぜる
- provider 実体が module 契約を上書きする
- `cmd/picoclaw` がドメインルールの正本になる
- 1つの状態に複数の所有者を持たせる

## 10. 状態所有とデータフロー
### 状態所有
| 状態 | 所有者 |
| --- | --- |
| 会話本文 | Chat |
| 実行ジョブ | Worker |
| LLM health | LLM |
| TTS synthesis / playback contract | TTS |
| STT recognition / input contract | STT |
| 共通 ID / state transition | Core |
| memory entities | Memory contracts |
| heartbeat decision | App + policy |

### データフロー
1. `Chat` が会話を進める
2. 必要に応じて `LLM` へ推論依頼する
3. 音声化が必要なら `TTS` へ渡す
4. 入力が音声なら `STT` が文字化する
5. 実行が必要なら `Worker` が担当する
6. 記憶更新は memory policy を通す
7. heartbeat は次の行動候補を更新する

### 正本ルール
- 正本は1つだけ
- 派生 state は複数可
- 正本と派生を同じ契約にしない

## 11. 物理分離に向けた移行順
### 優先順位
1. `core` の契約固定
2. `llm / tts / stt` の契約固定
3. `chat / worker` のロール境界固定
4. 記憶の正本分離
5. heartbeat の契約分離
6. 感情表出の契約分離
7. 拡張探索の契約分離
8. `viewer / idlechat / app` の切り出し

### ファイルの扱い
- `modules/*` に契約を置く
- `cmd/picoclaw` には起動・配線だけ残す
- `internal/...` には当面の実体実装だけ残す

### 典型的な移行先
| 現在の責務 | 移行先 |
| --- | --- |
| module health / diagnostics | `modules/*` |
| provider factory | `cmd/picoclaw` の配線 + `modules/*` の契約 |
| TTS/STT 実体 HTTP | `internal/infrastructure/*` |
| IdleChat topic / judge / seed | `modules/chat` |
| Worker policy / role plan | `modules/worker` |

## 12. テスト仕様
### まず書くべきテスト
- `modules/chat` の topic / judge / seed / similarity 契約テスト
- `modules/llm` の local / legacy provider plan テスト
- `modules/tts` の playback / public session / emotion prefix テスト
- `modules/stt` の result / busy policy / viewer input テスト
- `modules/worker` の role / policy / external explicit-only テスト
- `modules/core` の manifest / health / state contract テスト

### 移行時のテスト
- `cmd/picoclaw` の route registration テスト
- `internal/application/idlechat` の alias / speaker provider / topic flow テスト
- `internal/infrastructure/*` の adapter compatibility テスト

### テストの原則
- 契約テストを先に書く
- 実装テストは後に追従する
- 1つのモジュールに対して責務単位でテストする
- `fallback` を成功扱いにしない

## 13. 受け入れ条件
次を満たしたら本仕様の実装段階に入れる。

1. `modules/*` に各モジュールの公開契約がある
2. `cmd/picoclaw` が配線以外のルールを持っていない
3. `ChatWorker / Heavy / Wild / Coder1-4` のロール契約が固定されている
4. 記憶の self / relation / episodic / semantic / knowledge / volatile が分離されている
5. heartbeat の入力・評価・分岐・出力が定義されている
6. drive と emotion expression が別契約になっている
7. 拡張探索・インストール・作成・定着の流れが定義されている
8. 依存方向違反が検査可能になっている
9. 状態所有が一意になっている
10. repo 分割しても契約が崩れない
