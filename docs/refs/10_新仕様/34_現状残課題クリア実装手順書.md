# RenCrow 現状残課題クリア実装手順書

## 1. 目的

この文書は、`docs/10_新仕様/33_現状残課題整理作業手順書.md` の残課題を、長時間 Goal 実行で潰し合いを起こさずに消化するための実装手順書である。

`33_現状残課題整理作業手順書.md` は残課題の棚卸しと判定手順を定義する。本書は、それを実際に commit / push 可能な作業単位へ分解し、次の Goal に設定できる粒度へ落とす。

## 2. Goal に設定するプロンプト

Goal に設定する場合は、次を使う。

```text
docs/10_新仕様/34_現状残課題クリア実装手順書.md に沿って、33_現状残課題整理作業手順書.md の残課題を順に処理してください。

各作業は、小さな検証済み commit 単位で進めてください。
各単位で、対象・変更範囲・検証コマンド・完了条件を自分で定義してから実装してください。
実装後、該当テストと必要な runtime / Viewer 確認を行ってください。
確認済みの関連ファイルだけを選択的に stage し、日本語 commit message で commit してください。
commit 後は push してください。
push できたら、ユーザー確認を待たずに次の未完了項目へ進んでください。

ただし、次の場合だけ停止して報告してください。
- テスト失敗や runtime 確認失敗が、現在の作業範囲内で短時間に解消できない。
- 変更が複数領域に広がり、1 commit の責務が曖昧になった。
- 外部 secret / 実機 / 外部 API / 複数マシンなど、こちらで準備できない依存が必要。
- destructive operation、依存追加、CI/deploy 設定変更、ファイル削除、セーフガード変更が必要。
- push 対象に未確認差分や live E2E 生成物が混ざりそうになった。

禁止:
- worktree 全体を一括 stage / commit しない。
- `vault/` や live E2E 生成物を明示指示なく commit しない。
- 失敗したテストを無視して push しない。
- blocked 項目を成功扱いしない。
- Viewer / domain / persistence / runtime / docs を、責務が曖昧なまま 1 commit に混ぜない。

報告:
- 各 push 後に、commit hash、検証コマンド、次に進む対象を短く報告してください。
- ユーザー確認は不要です。止まる条件に当たった場合だけ質問または報告して停止してください。
```

## 5. 処理順

残課題は、外部依存が少ないものから処理する。

| 順序 | 作業単位 | 目的 | push 条件 | 停止条件 |
|---:|---|---|---|---|
| 1 | Viewer Go handler / fixture 整合 | domain validation 強化で落ちた Viewer Go test を直す | `go test ./internal/adapter/viewer -count=1` pass | 修正が domain / persistence へ広がり責務が曖昧になる |
| 2 | Viewer visible-state JS 境界 | API failure / copy failure / STT / TTS / runtime failure を画面に残す | `node internal/adapter/viewer/viewer_memory_panel.test.mjs` pass、必要なら `/viewer` live 表示確認 | Go handler test が落ちたまま |
| 3 | domain validation 境界 | timestamp 欠落、重複 ID、terminal 証跡不足を保存前に拒否 | 対象 domain package test pass | handler fixture や store fixture と整合しない |
| 4 | persistence current view 境界 | JSONL / SQLite が最新 state per ID と validation を満たす | 対象 persistence package test pass | domain validation と schema の責務が混ざる |
| 5 | rencrowclient status validation | 2xx response でも malformed current view を成功扱いしない | `go test ./pkg/rencrowclient -count=1` pass | live API 実装側の未整合が発覚 |
| 6 | runtime readiness / blocked route 表示 | `/health`、LLM Ops、Source Registry、Memory Layers、Sandbox などを blocked 証跡として表示 | focused runtime / viewer tests pass、必要なら `make install` と live `/viewer` 確認 | live dependency 不足を成功扱いしそうな場合 |
| 7 | docs 整理 | 実装済み / blocked / 未確認を 13 / 17 / 32 / 33 に反映 | 対応する実装 commit が push 済み、docs 差分が対応範囲だけ | docs が未 push 実装を完了扱いする |

## 6. 作業単位別の実装手順

### 6.1 Viewer Go handler / fixture 整合

対象:

- `internal/adapter/viewer/*_handler.go`
- `internal/adapter/viewer/*_handler_test.go`
- 必要最小限の domain / test fixture

手順:

1. `GOCACHE=/tmp/rencrow-gocache go test ./internal/adapter/viewer -count=1` を実行する。
2. 失敗 test を 1 件ずつ読む。
3. 失敗理由が test fixture の不足か、production handler の契約違反かを分ける。
4. fixture 不足なら、domain validation が要求する `created_at` / `updated_at` / terminal 証跡を test data に追加する。
5. production handler の契約違反なら、handler が保存前に正しい証跡を持つ item を作るよう最小修正する。
6. `go test ./internal/adapter/viewer -count=1` が pass したら commit / push する。

commit message 例:

```text
fix: Viewer handler の検証用証跡を整合させる
```

### 6.2 Viewer visible-state JS 境界

対象:

- `internal/adapter/viewer/assets/js/**`
- `internal/adapter/viewer/assets/css/**`
- `internal/adapter/viewer/viewer.html`
- `internal/adapter/viewer/viewer_memory_panel.test.mjs`

手順:

1. `node internal/adapter/viewer/viewer_memory_panel.test.mjs` を実行する。
2. 失敗があれば、表示上の failure state が stale / empty / console-only になっていないか確認する。
3. Go handler test が落ちていないことを確認する。
4. Viewer production 変更を含む場合は、必要に応じて `make install`、`rencrow.service` restart、`/viewer` live 確認を行う。
5. Node test と必要な live 表示確認が揃ったら commit / push する。

commit message 例:

```text
fix: Viewer の失敗状態を画面上に保持する
```

### 6.3 domain validation 境界

対象:

- `internal/domain/**/validation.go`
- `internal/domain/**/*_test.go`

手順:

1. 対象 domain を 1 つだけ選ぶ。
2. 必須 ID、timestamp、terminal evidence、status enum、重複 ID のどれを扱うか明示する。
3. 先に失敗 test または validation test を追加する。
4. validation を最小実装する。
5. 対象 domain package test を実行する。
6. 関連 handler / persistence fixture が壊れていないか、最小の関連 package test を実行する。
7. pass したら commit / push する。

commit message 例:

```text
fix: SuperAgent 台帳の保存前検証を強化する
```

### 6.4 persistence current view 境界

対象:

- `internal/infrastructure/persistence/**/jsonl_store.go`
- `internal/infrastructure/persistence/**/sqlite_store.go`
- 対象 store test

手順:

1. 対象 store を 1 つだけ選ぶ。
2. JSONL と SQLite の current view 差分を確認する。
3. 同一 ID の履歴が最新 state per ID として返ること、保存前 validation が効くことを test する。
4. 対象 persistence package test を実行する。
5. 関連 client / viewer status test がある場合は focused test を実行する。
6. pass したら commit / push する。

commit message 例:

```text
fix: Workstream JSONL store の current view を整合させる
```

### 6.5 rencrowclient status validation

対象:

- `pkg/rencrowclient/client.go`
- `pkg/rencrowclient/client_test.go`

手順:

1. 1 API family だけ選ぶ。
2. malformed response の種類を 1 commit 内で増やしすぎない。
3. 2xx でも必須 ID / timestamp / evidence / status enum / top-level readiness が不整合なら error にする test を追加する。
4. `GOCACHE=/tmp/rencrow-gocache go test ./pkg/rencrowclient -count=1` を実行する。
5. 可能なら関連 live E2E の current view read test を実行する。
6. pass したら commit / push する。

commit message 例:

```text
fix: Revenue status の malformed response を拒否する
```

### 6.6 runtime readiness / blocked route 表示

対象:

- `cmd/rencrow/runtime_*.go`
- `cmd/rencrow/routes.go`
- `internal/adapter/health/**`
- `internal/adapter/viewer/debug_system_handler.go`
- `test/e2e/phase25_*`

手順:

1. live dependency が足りない項目を 1 つ選ぶ。
2. readiness response で secret 値を出さず、present / missing / configured / blocked を表現する。
3. blocked route は 404 ではなく、可能なら body 付き 503 を返す。
4. client / Viewer が body を保持することを test する。
5. production runtime を変更した場合は、クリーン停止、`make install`、restart、`/viewer` 確認を行う。
6. `/health` が 503 の場合は、down checks の理由を記録し、成功扱いしない。
7. pass したら commit / push する。

commit message 例:

```text
fix: runtime 依存不足を blocked 証跡として返す
```

### 6.7 docs 整理

対象:

- `docs/10_新仕様/13_実装項目インベントリ.md`
- `docs/10_新仕様/17_E2E残課題.md`
- `docs/10_新仕様/31_未実装項目実装仕様.md`
- `docs/10_新仕様/32_E2E_runtime確認チェックリスト.md`
- `docs/10_新仕様/33_現状残課題整理作業手順書.md`
- 本書

手順:

1. 対応する実装 commit が push 済みか確認する。
2. docs だけ先行して pass 扱いにしない。
3. 実装済み、local 境界確認、live E2E、blocked、skipped を分けて記録する。
4. docs 更新だけなら `git diff --check` を実行する。
5. 対応する実装範囲と docs 範囲が一致したら commit / push する。

commit message 例:

```text
docs: 現状残課題の確認結果を整理する
```

## 7. live service 再起動手順

production Go / Viewer asset を変更した場合だけ実行する。

```bash
systemctl --user stop rencrow.service
pgrep -a rencrow || true
ss -ltnp | rg ':18790' || true
curl -sS --max-time 3 -i http://127.0.0.1:18790/health || true
GOCACHE=/tmp/rencrow-gocache make install
systemctl --user start rencrow.service
pgrep -a rencrow || true
ss -ltnp | rg ':18790' || true
curl -sS --max-time 5 -i http://127.0.0.1:18790/viewer | sed -n '1,12p'
curl -sS --max-time 5 -i http://127.0.0.1:18790/health | sed -n '1,40p'
```

`/health` が 503 の場合でも、原因が既知の `local_llm_chat` / `local_llm_worker` endpoint timeout なら runtime blocked として記録する。health success として扱わない。

## 8. commit / push 手順

各作業単位で以下を行う。

```bash
git status --short
git diff --stat -- <対象path>
git add <確認済み対象pathのみ>
git diff --cached --stat
git commit -m "<日本語のcommit message>"
git push origin feature/RenCrow_Start
```

禁止:

- `git add .`
- `git add -A`
- `vault/` の stage
- 未確認 docs の stage
- test failure がある状態での push

## 10. 完了条件

この手順書の完了は、すべての外部 E2E が pass することではない。完了条件は以下である。

- `33_現状残課題整理作業手順書.md` の各残課題が、pass / fail / skipped / blocked / 対象外のいずれかに分類されている。
- pass は該当範囲の authoritative evidence で裏付けられている。
- blocked / skipped は成功扱いされていない。
- 実装済み範囲は検証済み commit として push 済み。
- 未確認差分や live E2E 生成物が commit に混ざっていない。
- 残る blocked 項目には、解除に必要な外部依存が明記されている。
