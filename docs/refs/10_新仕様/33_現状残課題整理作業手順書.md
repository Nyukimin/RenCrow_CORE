# RenCrow 現状残課題整理 作業手順書

## 1. 目的

この手順書は、`docs/10_新仕様` と現行実装を照合して確認した残課題を、次の作業者が 1 件ずつ処理するための実務手順に落としたものである。

現状は「主要機能の入口が未実装」という段階ではない。残課題の中心は、以下である。

- 接続残
- Human approval 後の適用残
- 実運用 E2E 未確認
- 外部環境待ちの E2E
- 保留 / 対象外の明確化

## 2. この文書の使い方

この文書は、残課題の一覧表ではなく、作業者が次に何を確認し、どの条件で pass / fail / skipped / blocked と判定し、どの docs に結果を戻すかを決めるための手順書である。

作業者は、次の流れで進める。

1. `3. 作業前に必ず読むもの` を確認する。
2. `4. 作業ルール` で、今回の作業が調査だけか、実装変更を含むかを分ける。
3. `5. 現在の残課題と処理順` から 1 件を選ぶ。
4. `7. 課題別作業手順` の該当節に従って確認する。
5. `10. 結果記録テンプレート` で証跡を残す。
6. `11. docs 更新先` に従って台帳へ反映する。

1 回の作業で複数項目をまとめて完了扱いしない。複数項目を触る場合でも、判定と証跡は項目ごとに分ける。

## 4. 作業ルール

### 4.1 編集可否

- この手順書を使う段階では、まず調査と確認を行う。
- 実装変更に入る場合は、ユーザーの明示的な OK を得る。
- 実装変更では、先に再現テスト、失敗テスト、または代替検証手順を明文化する。
- 無関係な cleanup は行わない。

### 4.2 開始条件

作業を始める前に、次を明確にする。

| 確認項目 | 内容 |
|---|---|
| 対象 | どの残課題を扱うか |
| 作業種別 | 調査 / E2E確認 / 設計 / 実装変更 / docs反映 |
| 外部依存 | API key、実ブラウザ、実マイク、複数マシン、外部チャネルの有無 |
| Human approval | 外部送信、PR作成、正式環境適用、sandbox promotion が必要か |
| 記録先 | `17_E2E残課題.md`、`31_未実装項目実装仕様.md`、または別の該当仕様 |

外部依存が不足している場合は、実行前に `skipped` または `blocked` の候補として扱い、成功扱いにしない。

### 4.3 終了条件

作業を終える前に、次を確認する。

| 確認項目 | 内容 |
|---|---|
| 判定 | pass / fail / skipped / blocked のいずれかに分類したか |
| 証跡 | config、route、session、job、event log、response を残したか |
| fallback | fallback を成功扱いしていないか |
| docs反映 | 結果を該当 docs に反映したか、または反映不要の理由を記録したか |
| 次アクション | 未完了の場合、次に何を準備すれば進むかを書いたか |

終了条件を満たせない場合は、完了扱いにせず `blocked` または `fail` として理由を残す。

### 4.4 判定ルール

- skip は成功ではなく、環境未準備または未確認として扱う。
- fallback は成功ではなく、期待 route 不成立として扱う。
- health ok だけで user flow 成立扱いにしない。
- handler / stub / unit test の成功だけで実 API E2E 完了扱いにしない。
- 実ブラウザ UI は DOM 存在ではなく、visible state と実操作の成立で判定する。
- 外部送信、PR 作成、正式環境適用は Human approval 必須として扱う。
- 削除済みの旧 docs / archive / old / codebase-map は現行判断の一次参照にしない。

### 4.5 証跡として残すもの

各作業では、最低限以下を記録する。

| 証跡 | 内容 |
|---|---|
| 日時 | 実行日と時刻 |
| 実行環境 | local / live service / browser / distributed / external API |
| config source | `~/.rencrow/config.yaml`、runtime-config、または一時 config |
| 実行コマンド / 操作 | `go test`、curl、Viewer 操作、外部イベント送信など |
| route / session / job | route、job_id、session_id、channel id |
| event log | routing、agent response、attachment、warning、error |
| response | ユーザーに返った本文または失敗表示 |
| 判定 | pass / fail / skipped / blocked |
| skip / fail 理由 | secret 不足、実機不足、device 不足、route 不成立など |

### 4.6 Goal 実行時の自律進行ルール

この手順書を Goal に設定して長時間実行する場合は、不要な確認待ちで作業を止めず、検証済み commit 単位で自律的に進める。ただし、未確認差分を混ぜたまま進めてはいけない。

Goal 用プロンプトには、以下を必ず含める。

```text
この作業手順書に沿って、未完了項目を小さな検証済み commit 単位で順に処理してください。

各単位で、対象・変更範囲・検証コマンド・完了条件を自分で定義してから実装してください。
実装後、該当テストと必要な runtime / Viewer 確認を行ってください。
確認済みの関連ファイルだけを選択的に stage し、日本語 commit message で commit してください。
commit 後は push してください。
push できたら、ユーザー確認を待たずに次の未完了項目へ進んでください。

ただし、次の場合だけ停止して報告してください。
- テスト失敗や runtime 確認失敗が、現在の作業範囲内で短時間に解消できない。
- 変更が複数領域に広がり、1 commit の責務が曖昧になった。
- 外部 secret / 実機 / 外部 API / 複数マシンなど、こちらで準備できない依存が必要。
- destructive operation、依存追加、CI/deploy 設定変更、ファイル削除、セーフガード変更が必要。
- push 対象に未確認差分や live E2E 生成物が混ざりそうになった。

禁止:
- worktree 全体を一括 stage / commit しない。
- `vault/` や live E2E 生成物を明示指示なく commit しない。
- 失敗したテストを無視して push しない。
- blocked 項目を成功扱いしない。
- Viewer / domain / persistence / runtime / docs を、責務が曖昧なまま 1 commit に混ぜない。

報告:
- 各 push 後に、commit hash、検証コマンド、次に進む対象を短く報告してください。
- ユーザー確認は不要です。止まる条件に当たった場合だけ質問または報告して停止してください。
```

この Goal 用プロンプトを手順書へ組み込む、または既存手順書を Goal 実行向けに変換する場合は、repo skill `.agents/skills/goal-workflow-hardening/SKILL.md` を使用する。

## 7. 課題別作業手順

### 7.1 SuperAgent Harness

#### 目的

run queue scheduler の実運用 E2E と、長時間再開 E2E の確認を行う。

#### 主な関連実装

- `internal/application/superagent`
- `internal/domain/superagent`
- `internal/infrastructure/persistence/superagent`
- `internal/adapter/viewer/superagent_handler.go`
- `internal/application/orchestrator/superagent_runtime.go`

#### 手順

1. `docs/10_新仕様/31_未実装項目実装仕様.md` の P11 を読む。
2. `superagent_harness.enabled`、`run_queue_scheduler_enabled`、`run_queue_scheduler_interval_sec`、`run_queue_scheduler_claim_limit` の実効 config を確認する。
3. `/viewer/superagent/run-queue` で小さい queued item を作る。
4. scheduler が有効な場合、due item が claim されるか確認する。
5. local / distributed orchestrator に `superagent` channel の work item として渡るか確認する。
6. `trace_event` と `run_queue` の status が `completed` / `failed` / `cancelled` のいずれかに収束するか確認する。
7. fallback route に落ちていないか確認する。
8. 結果を `docs/10_新仕様/17_E2E残課題.md` に追記する。

#### 完了条件

- queued item が scheduler によって claim される。
- route / run_id / trace_event / queue status が同一フローとして追える。
- 完了または失敗が明示状態で記録される。
- fallback を成功扱いしていない。

#### 失敗扱い

- queued のまま止まる。
- claimed のまま終了状態にならない。
- local fallback / CHAT fallback で処理されたものを成功扱いする。
- event log が run と紐づかない。

### 7.5 Complexity Hotspot Skill

#### 目的

report-only / proposal / Sandbox Gate の範囲で安全に確認する。patch 適用や外部 PR 作成は対象外とする。

#### 主な関連実装

- `internal/domain/complexity`
- `internal/application/complexity`
- `internal/infrastructure/persistence/complexity`
- `internal/adapter/viewer/complexity_hotspot_handler.go`

#### 手順

1. `docs/10_新仕様/31_未実装項目実装仕様.md` の P10 を読む。
2. 小さい fixture または限定ディレクトリで scan を行う。
3. hotspot report artifact が保存されることを確認する。

- 証跡不足の場合は `needs_review` / `needs_more_tests` になる。
