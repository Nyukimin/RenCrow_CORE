## 1. 実装目的

音声入力の同期パスを、ユーザーが待っている応答成立までに限定する。

ログ保存、Monitor 更新、metrics、TTS、IdleChat 後処理などの副作用は、音声入力の正本生成と Chat 表示をブロックしない構成へ分離する。

この実装仕様では、既存 `/stt` / `/voice-chat` の外部契約を維持したまま、内部境界を段階的に整理する。

## 3. 実装後の処理フロー

### 3.1 LLM音声

```text
Viewer mic
  -> viewer.js VDS state machine
  -> /voice-chat
  -> RenCrow_LLM audio input
  -> llm.final
  -> voiceinput.ResultBuilder
  -> voiceinput.Publisher
  -> EventHub live Chat event
  -> voiceinput.SideEffects enqueue
```

同期で待つのは `Publisher` の live Chat event 発行まで。

非同期化するもの:

- session jsonl
- EventLogStore
- MonitorStore
- metrics 永続化
- TTS
- IdleChat 後処理
- debug trace 保存

### 3.2 STT音声

```text
Viewer mic
  -> /stt
  -> RenCrow_STT final
  -> normal Chat input
  -> Mio reply
  -> voiceinput.ResultBuilder
  -> voiceinput.Publisher
  -> EventHub live Chat event
  -> voiceinput.SideEffects enqueue
```

STT音声では、`user_text` は STT final text。`reply` は通常 Chat の Mio 応答。

## 4. `VoiceInputResult` model

### 4.1 型

```go
package voiceinput

type Mode string

const (
    ModeSTT Mode = "stt"
    ModeLLM Mode = "llm"
)

type Result struct {
    Mode        Mode
    UtteranceID string
    SessionID   string
    Channel     string
    ChatID      string
    UserText    string
    Reply       string
    RawFinal    string
    Source      string
    Timings     Timings
}

type Timings struct {
    StartedAt      time.Time
    CommitAt       time.Time
    FirstTokenAt   time.Time
    FinalAt        time.Time
    PublishedAt    time.Time
}
```

### 4.2 validation

`Result` は publish 前に validation する。

必須:

- `Mode`
- `UtteranceID`
- `SessionID`
- `Channel`
- `ChatID`
- `Source`

LLM音声:

- `Reply` は必須。
- `UserText` は空を許可する。
- `RawFinal` は必須。

STT音声:

- `UserText` は必須。
- `Reply` は Chat 応答後に必須。

禁止:

- `RawFinal` の JSON 全体を `Reply` としてそのまま publish すること。
- no-audio / meta 応答を通常の `Reply` として publish すること。

## 5. `VoiceInputResultBuilder`

### 5.1 LLM音声 JSON 分離

入力:

```go
BuildFromLLMFinal(req BuildLLMRequest) (Result, error)
```

`req.FinalText` が JSON の場合:

- `user_text`
- `transcript`
- `user_utterance`
- `recognized_text`

上記の最初の非空値を `UserText` にする。

- `reply`
- `assistant_text`
- `mio_response`
- `response`
- `answer`

上記の最初の非空値を `Reply` にする。

JSON ではない場合:

- `Reply = finalText`
- `UserText = ""`

ただし、no-audio / meta 応答は error とする。

### 5.2 no-audio / meta 応答判定

次を含む final は通常応答にしない。

- `音声内容を入力してください`
- `音声内容を提示してください`
- `音声ファイルをアップロード`
- `音声が提供されていない`
- `音声が入力されていない`
- `入力をお待ちしております`

既存 `isVoiceDirectMetaNoAudioFinal` のロジックを `voiceinput` package へ移し、WebSocket relay と input_audio 経路で共用する。

### 5.3 STT音声

STT音声では、STT final text から `UserText` を作る。

通常 Chat 処理後の Mio 応答を `Reply` に入れる。

## 6. `VoiceInputPublisher`

### 6.1 役割

`VoiceInputResult` から Viewer Chat に必要な最小 event を発行する。

LLM音声:

1. `message.received` user -> mio
   - content: `UserText`
   - `UserText` が空の場合は、空文字をそのまま出すか、Viewer 表示では省略する。JSON 全体は出さない。
2. `routing.decision` mio -> ""
   - route: `CHAT`
   - content: `confidence 100% evidence=voice_direct:matched:CHAT utterance_id=...`
3. `agent.response` mio -> user
   - content: `Reply`

STT音声:

- 通常 Chat の `ProcessMessage` 経路を優先する。
- Publisher は STT final と Mio reply の対応が取れる場合のみ `VoiceInputResult` を生成する。

### 6.2 同期保証

Publisher は次を待ってはいけない。

- EventLogStore append
- Monitor update
- session jsonl write
- TTS request
- IdleChat drain

Publisher の戻り値は、live Chat event の publish 結果のみを表す。

## 7. `VoiceInputSideEffects`

### 7.1 worker 構成

最初は単一 queue + worker でよい。

```go
type SideEffectItem struct {
    Result voiceinput.Result
    Events []orchestrator.OrchestratorEvent
}
```

queue は bounded とする。

推奨初期値:

- queue size: 256
- worker count: 1
- per item timeout: 3 sec

queue full の場合:

- 同期パスは成功のまま返す。
- `WARN: voiceinput side effect queue full` を出す。

### 7.2 副作用一覧

worker は段階的に次を担当する。

Phase 1:

- EventLogStore append
- Monitor update

Phase 2:

- session jsonl
- metrics 永続化

Phase 3:

- TTS request
- IdleChat 後処理
- debug trace 保存

### 7.3 failure policy

副作用失敗時:

- warning log に残す。
- `VoiceInputResult` を失敗へ変更しない。
- Viewer Chat に表示済みの応答を取り消さない。

## 8. EventHub / runtime relay 実装

### 8.1 現行維持すること

`EventHub.OnEvent` の live client broadcast は維持する。

必要条件:

- client channel 送信は non-blocking。
- slow client は drop。
- history append は軽量。

### 8.2 変更すること

`idleAwareEventListener.OnEvent` では、live EventHub delivery を先に行う。

非同期へ逃がすもの:

- `EventLogStore.Append`
- `MonitorStore.OnEvent`

実装例:

```go
func (l *idleAwareEventListener) OnEvent(ev orchestrator.OrchestratorEvent) {
    l.hub.OnEvent(ev)
    l.notifyIdleIfNeeded(ev)
    l.sideEffects.Enqueue(ev)
}
```

暫定実装では `go l.recordEventAsync(ev)` でもよいが、最終形は bounded queue にする。

## 9. IdleChat 実装境界

### 9.1 voice-chat runtime

`session.start`:

- `IdleNotifier.NotifyActivity()`
- `IdleNotifier.SetChatBusy(true)`

終了条件:

- `llm.final`
- error
- `session.cancel`
- WebSocket close

上記で必ず `SetChatBusy(false)` を呼ぶ。

### 9.2 テスト

`TestVoiceChatInputAudioBridge_InterruptsIdleChatDuringVoiceSession`

期待:

- `session.start` 後に `chatBusy=true`
- final 後に `chatBusy=false`
- final 受信直後の race を避けるため、短い wait loop で確認する

追加するテスト:

- error frame 後に `chatBusy=false`
- client close 後に `chatBusy=false`
- empty audio error 後に `chatBusy=false`

## 10. Viewer JS 実装

### 10.1 session.start

LLM音声の `sendVDSSessionStart()` は次を必ず送る。

```js
{
  type: 'session.start',
  utterance_id,
  sample_rate,
  channels: 1,
  format: 'pcm16le',
  voice_input_mode: 'vds_sub',
  viewer_session_id: 'viewer',
  channel: 'viewer',
  chat_id: 'default',
  prompt: VDS_DEFAULT_PROMPT
}
```

`viewer_session_id` を省略しない。省略すると session jsonl が `vds-sess-...` に分散し、Viewer Chat の会話履歴と相関しにくくなる。

### 10.2 Chat 表示

LLM音声:

- `transcript.final` がある場合は `user_text` として表示する。
- `llm.final` は Mio 応答として表示する。
- `llm.final.text` が JSON の場合、Viewer 側でも `reply` を抽出できるようにする。
- JSON 全体を Chat bubble に表示しない。

ただし、最終的な JSON 分離の正本は server side `VoiceInputResultBuilder` とする。

Viewer 側 JSON 分離は、server 移行完了までの防御実装とする。

## 12. 実装フェーズ

### Phase 1: 現行経路の安全固定

目的:

- 既存 `/voice-chat` / `/stt` を壊さず、同期パスを副作用から守る。

作業:

1. `runtime_event_relay.go` で live EventHub delivery を先行する。
2. EventLogStore / Monitor 更新を同期パスから外す。
3. input_audio 経路に `ProcessVoiceDirect completed` ログを追加する。
4. Viewer `sendVDSSessionStart()` に `viewer_session_id: 'viewer'` を追加する。
5. voice-chat busy 終了テストを race-free にする。

完了条件:

- LLM音声 live probe で `input_audio finalized -> agent.response -> ProcessVoiceDirect completed` が即時に出る。
- `session_YYYY-MM-DD_viewer.jsonl` に voice_direct turn が記録される。

### Phase 2: ResultBuilder 導入

目的:

- `llm.final` の JSON 分離と no-audio 判定を 1 箇所に集約する。

作業:

1. `internal/application/voiceinput/result.go` 追加。
2. `result_builder.go` 追加。
3. 既存 `splitVoiceChatStructuredText` / `isVoiceDirectMetaNoAudioFinal` を移植または委譲する。
4. input_audio 経路と WS relay 経路の両方で ResultBuilder を使う。

完了条件:

- JSON final から `user_text` / `reply` が分離される。
- no-audio / upload request は Chat 正本にならない。
- plain text final は `Reply` として扱われる。

### Phase 3: Publisher 導入

目的:

- `ProcessVoiceDirect` の event 発行責務を `VoiceInputPublisher` に移す。

作業:

1. `publisher.go` 追加。
2. `ProcessVoiceDirect` は互換 API として残し、内部で Publisher を呼ぶ。
3. `message.received` の content を `UserText` にする。
4. `agent.response` の content を `Reply` にする。
5. JSON 全体が Chat に出ないことをテストで固定する。

完了条件:

- LLM音声でユーザー発話は「れん」の発言として入る。
- Mio 応答は `reply` だけが表示される。
- `RawFinal` は debug / log に残るが Chat 本文には出ない。

### Phase 4: SideEffects queue 導入

目的:

- archive / monitor / session log / metrics を bounded queue へ分離する。

作業:

1. `side_effects.go` 追加。
2. runtime wiring で `VoiceInputSideEffects` を起動する。
3. `EventLogStore.Append` と `MonitorStore.OnEvent` を queue worker へ移す。
4. queue full / timeout / failure log を追加する。
5. slow side effect test を追加する。

完了条件:

- slow EventLogStore でも `agent.response` が遅れない。
- Monitor 更新 failure でも Chat 応答が成功する。
- queue full は warning になるが音声入力は失敗しない。

### Phase 5: STT音声統合

目的:

- STT音声も `VoiceInputResult` で観測できるようにする。

作業:

1. STT final text から Result を生成する。
2. 通常 Chat 応答と Result を相関する。
3. STT音声 E2E のログ形式を LLM音声と揃える。

完了条件:

- STT音声 / LLM音声を同じ観測 schema で比較できる。
- ただし正本は混ぜない。

## 14. 実装時の禁止事項

- `/voice-chat` の外部 WS 契約を同時に変えない。
- Viewer UI の大規模リファクタを同時に行わない。
- IdleChat / TTS の仕様変更を同時に行わない。
- EventHub を音声入力専用にしない。
- `VoiceInputResult` 導入前に JSON 分離ロジックを複数箇所へ増やさない。
- 副作用 queue 導入前に archive / monitor の同期呼び出しを増やさない。
- fallback 成功として扱わない。fallback は error path として表示・ログ化する。

## 16. 完了判定

実装完了とみなす条件:

- LLM音声で `input_audio finalized` から `agent.response` までが副作用 I/O に依存しない。
- LLM音声 JSON final の `user_text` がユーザー発話、`reply` が Mio 応答として表示される。
- STT音声と LLM音声の正本が型・ログ上で区別される。
- EventLogStore / Monitor / session log の遅延や失敗で Chat 応答が止まらない。
- IdleChat / TTS の停止・再開が音声入力正本を壊さない。
- 失敗時に同期パスか非同期副作用かをログだけで判定できる。

## 17. 推奨実装順

1. Phase 1 を現在の差分として固める。
2. `voiceinput.Result` と ResultBuilder を追加する。
3. input_audio 経路だけ ResultBuilder に接続する。
4. WS relay 経路も ResultBuilder に接続する。
5. Publisher を導入し、`ProcessVoiceDirect` を薄い互換層にする。
6. SideEffects queue を導入する。
7. STT音声を Result 観測に統合する。
8. live E2E runbook を追加する。

各 Phase ごとに commit し、Phase をまたぐ UI 変更や IdleChat/TTS 変更を混ぜない。
