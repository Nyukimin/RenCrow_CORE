# 残タスク実装仕様

この文書は、`docs/10_新仕様/13_実装項目インベントリ.md` と `docs/10_新仕様/14_旧仕様未実装候補.md` に残る部分実装 / 要確認項目を、段階的に実装できる粒度へ落とした実装仕様である。

## 1. 全体方針

対象範囲:

- ChannelPolicy の config / runtime wiring。
- Slack / Discord / Telegram の file payload を共通 attachment pipeline へ接続する。
- Source Registry 由来テキストへ prompt injection warning metadata を付与し、Viewer で確認できるようにする。
- Wild の実機分散 E2E を定義し、local fallback や代替 route を成功扱いしない。

対象外範囲:

- 旧 OpenClaw の広範 provider / OAuth / ClawHub / Docker sandbox。
- channel account binding の全面 UI 化。
- OCR、画像 PDF 高精度解析、外部クラウド抽出。
- Wild の新規人格設計や LangGraph への移行。

既存実装との差分:

| 項目 | 既存実装 | 残差分 |
| --- | --- | --- |
| ChannelPolicy | `internal/domain/security/channel_policy.go` と LINE handler の任意注入 | config schema と local / distributed runtime 起動時注入は実装済み。Slack / Discord / Telegram への policy 展開判断は別作業 |
| 外部チャネル添付 | LINE / Slack / Discord / Telegram の主要 file payload は `internal/application/attachment` へ接続済み | 実 API payload 差分、署名不正 + media、巨大ファイル運用の E2E 確認 |
| Source Registry warning | attachment 抽出文と Source Registry fetch 由来テキストの warning metadata | Viewer run status で warning 件数表示済み。検索性が必要なら schema 拡張を再検討 |
| Wild distributed | `DistributedOrchestrator.SetWildAgent` と `RouteWILD` 経路。未接続時の local fallback 成功は禁止済み | 一時 distributed server `:18791` で SSH transport 接続、`/wild` route evidence / logs / response を確認済み |

責務境界:

| 層 | 責務 |
| --- | --- |
| Domain | `ChannelPolicy`、`Attachment`、prompt injection warning の contract を持つ |
| Adapter | channel 固有 payload、署名検証、download API 呼び出し、Viewer 表示を扱う |
| Application | attachment 保存 / 抽出、Source Registry fetch / staging / validate、Orchestrator route 実行を扱う |
| Infrastructure | config 読み込み、SQLite / transport / LLM provider 実装を扱う |
| Chat / Worker / Coder | 未許可入力や外部入力 warning を通常 prompt と混ぜず、責務外の実行を抱え込まない |

fallback 方針:

- fallback は失敗時の証跡であり、成功判定ではない。
- download 失敗、policy 拒否、Source Registry warning 未表示、Wild が local route に落ちた状態を正常完了にしない。
- E2E では route、session_id、job_id、event log、response の一致を確認する。

## 2. ChannelPolicy config/runtime wiring

### 2.1 追加する config schema

既存の `domain/security.ChannelPolicy` を重複実装せず、config から domain 型へ変換する。

変更候補:

- `internal/adapter/config/config_types.go`
- `internal/adapter/config/config_defaults.go`
- `cmd/picoclaw/runtime_channels.go`
- `cmd/picoclaw/runtime_distributed_mode.go`
- `internal/adapter/line/handler.go`

config 型案:

```go
type ChannelPolicyConfig struct {
    Enabled        bool     `yaml:"enabled"`
    AllowDM        *bool    `yaml:"allow_dm"`
    AllowGroups    *bool    `yaml:"allow_groups"`
    AllowedSenders []string `yaml:"allowed_senders"`
    PairedGroups   []string `yaml:"paired_groups"`
}

type LINEConfig struct {
    // existing fields
    ChannelPolicy ChannelPolicyConfig `yaml:"channel_policy"`
}
```

`AllowDM` / `AllowGroups` は、未設定と false を区別するため pointer を推奨する。変換時は policy 未設定なら現行互換として `SetChannelPolicy` を呼ばない。

YAML 例:

```yaml
line:
  channel_secret: "${LINE_CHANNEL_SECRET}"
  channel_access_token: "${LINE_CHANNEL_ACCESS_TOKEN}"
  channel_policy:
    enabled: true
    allow_dm: true
    allow_groups: true
    allowed_senders:
      - "Uxxxxxxxx"
    paired_groups:
      - "Cxxxxxxxx"
```

default behavior:

- `line.channel_policy.enabled` が false または未設定なら、現行互換として policy 未設定。
- policy 未設定時は既存の LINE 挙動を変えない。
- policy 設定時は、拒否された入力を routing / LLM へ流さない。

### 2.2 runtime wiring

実装方針:

1. `internal/adapter/config` に config 型と validation を追加する。
2. `cmd/picoclaw/runtime_channels.go` で LINE handler 生成後に config を domain `ChannelPolicy` へ変換する。
3. `lineHandler.SetChannelPolicy(policy)` を呼ぶ。
4. distributed mode の LINE handler 生成経路にも同じ helper を使う。

helper 案:

```go
func applyLineChannelPolicy(handler *line.Handler, cfg config.LineConfig) {
    policy, ok := config.ResolveChannelPolicy(cfg.ChannelPolicy)
    if !ok {
        return
    }
    handler.SetChannelPolicy(policy)
}
```

拒否時 log / response:

- handler は拒否理由を `channel_policy.reject` 相当で log に残す。
- response は channel ごとの制約に合わせ、LINE では短い拒否メッセージを返すか、既存仕様に合わせて 200 + no-op とする。
- どちらの場合も routing / LLM へ渡さない。

Slack / Discord / Telegram への展開:

- Phase 1 では LINE の config wiring を完了条件にする。
- Slack / Discord / Telegram は adapter の event payload に `SenderID` / `ChatID` / `SourceType` が揃うことを確認してから展開する。
- 展開時も domain `ChannelPolicy` を再利用し、adapter 固有の別 policy 型を作らない。

### 2.3 テストと完了条件

先に書くテスト:

- `internal/adapter/config`: YAML から `ChannelPolicyConfig` を読み、未設定 / enabled false / enabled true を区別できる。
- `cmd/picoclaw`: runtime channel setup が policy 設定時に LINE handler へ注入する。
- `internal/adapter/line`: config 由来 policy で未許可 group が routing に到達しない。

確認コマンド:

```bash
GOCACHE=/tmp/picoclaw-gocache go test ./internal/domain/security ./internal/adapter/config ./internal/adapter/line ./cmd/picoclaw
GOCACHE=/tmp/picoclaw-gocache go test ./...
```

完了条件:

- policy 未設定時に既存 LINE 挙動が変わらない。
- policy 設定時に未許可入力が routing / LLM へ流れない。
- 拒否理由が log で追える。
- local mode と distributed mode の両方で LINE handler へ注入される。

## 3. Slack / Discord / Telegram file payload attachment pipeline

### 3.1 正規化方針

全 channel の file payload は、最終的に `internal/application/attachment.IncomingFile` へ変換する。

```go
type IncomingFile struct {
    Filename    string
    ContentType string
    SizeBytes   int64
    Reader      io.Reader
}
```

Adapter は channel 固有 API と署名検証だけを扱い、保存、本文抽出、SecurityWarnings、サイズ制限は `internal/application/attachment.Store` に任せる。

channel 別方針:

| channel | payload 例 | download 方針 | 注意 |
| --- | --- | --- | --- |
| Slack | Events API の `files[]`、`url_private_download` | Bot token で download | Slack 署名検証後に処理する |
| Discord | message attachments、interaction attachments | attachment URL を HTTP GET | Discord signature / relay payload の検証境界を崩さない |
| Telegram | `document` / `photo` / `voice` の file_id | getFile で path 解決後 download | Bot API の file size / mime を metadata として使う |

LINE 実装との差分:

- LINE は `MediaDownloader.DownloadContent` と `SetAttachmentSaver` が既にある。
- Slack / Discord / Telegram は現状 text event 正規化が中心で、file payload が `ProcessMessage` の attachments へ入らない。
- 新規実装では各 adapter に `AttachmentSaver` 注入口を追加し、LINE と同じ Application boundary へ寄せる。

### 3.2 実装候補

変更候補:

- `internal/adapter/channels/slack/adapter.go`
- `internal/adapter/channels/discord/adapter.go`
- `internal/adapter/channels/telegram/adapter.go`
- `cmd/picoclaw/runtime_channels.go`
- `cmd/picoclaw/runtime_distributed_mode.go`
- `internal/application/channel` または channel event contract
- `internal/application/attachment`
- `internal/domain/attachment`

実装手順:

1. channel event contract に attachments を載せるか、adapter が直接 processor へ attachments 付き request を渡せるようにする。
2. 各 adapter に `SetAttachmentSaver` 相当を追加する。
3. channel ごとの download helper を adapter 内に閉じ込める。
4. download 成功後、`IncomingFile` を作って `SaveAll` へ渡す。
5. 保存済み `domain/attachment.Attachment` を orchestrator request に載せる。
6. download / MIME / size / unsupported file の失敗は error log と channel response に残し、通常 chat 成功として隠さない。

MIME / size / unsupported file:

- 判定は `internal/application/attachment.Store` と `internal/domain/attachment` を優先する。
- Adapter は channel metadata の `ContentType` / `Filename` / `SizeBytes` を渡すだけにする。
- `SaveAll` が拒否した場合は adapter 側で channel error として扱い、LLM へ通常 message として流さない。

追跡:

- session_id は既存 `BuildSessionID` / channel event の session_id を使う。
- event log には `viewer.attachment.received` と同等の attachment count、channel、chat_id、job_id を残す。
- download error は channel 名、file id、chat_id、session_id を含める。

### 3.3 テストと完了条件

先に書くテスト:

- Slack: files payload から `IncomingFile` が作られ、private URL download が認証 header 付きで呼ばれる。
- Discord: attachment URL payload から `IncomingFile` が作られる。
- Telegram: document file_id から getFile + download を経由して `IncomingFile` が作られる。
- common: unsupported MIME / oversize / download 失敗が routing / LLM に流れない。
- Orchestrator: channel 由来 attachment が task attachments と event log に入る。

確認コマンド:

```bash
GOCACHE=/tmp/picoclaw-gocache go test ./internal/adapter/channels/... ./internal/application/attachment ./internal/application/orchestrator ./cmd/picoclaw
GOCACHE=/tmp/picoclaw-gocache go test ./...
```

channel stub E2E:

- Slack / Discord / Telegram の webhook stub から file payload を送る。
- 保存 path、attachment metadata、event log、history の session_id を照合する。
- download 失敗 payload では LLM response が生成されないことを確認する。

完了条件:

- Slack / Discord / Telegram file payload が Viewer upload / LINE media と同じ `Attachment` contract に正規化される。
- channel 固有 download と Application attachment 境界が混ざらない。
- 失敗が通常 chat 成功として隠れない。

## 4. Source Registry prompt injection warning + Viewer 表示

### 4.1 metadata 方針

Source Registry 由来テキストは、staging 時に `DetectPromptInjectionWarnings` を適用し、warning を `L1StagingItem.Meta` に保存する。既存 `L1SourceFetchPayload.Meta` と `L1StagingItem.Meta` があるため、第一段階では専用 column ではなく metadata key で始める。

metadata key 案:

```json
{
  "security_warnings": ["prompt_injection:ignore_previous_instructions"],
  "security_warning_source": "source_registry",
  "security_warning_checked_at": "2026-05-17T00:00:00Z"
}
```

attachment との違い:

| 入力 | 保存先 | 目的 |
| --- | --- | --- |
| Viewer / LINE attachment | `domain/attachment.Attachment.SecurityWarnings` | 添付本文に対する risk metadata |
| Source Registry fetch result | `L1StagingItem.Meta.security_warnings` | source fetch / staging に対する risk metadata |

warning と拒否の違い:

- warning は拒否ではない。
- validation policy が warning を理由に reject するかは別判断にする。
- ただし warning 付き staging item の無審査 promote は禁止する。

### 4.2 実装候補

変更候補:

- `internal/application/sourcefetcher/registry_sweeper.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_source_registry.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_staging.go`
- `internal/infrastructure/persistence/conversation/l1_sqlite_types.go`
- `internal/adapter/viewer/source_registry_handler.go`
- `internal/adapter/viewer/assets/js/tabs/memory.js`
- `internal/adapter/viewer/viewer_memory_panel.test.mjs`

実装手順:

1. `StageSourceRegistryFetch` またはその直前で `payload.RawText` に `DetectPromptInjectionWarnings` を適用する。
2. warning がある場合、`payload.Meta` または `L1StagingItem.Meta` に `security_warnings` を保存する。
3. `RecentStagingItems` / Source Registry run API response に warning metadata が含まれることを確認する。
4. Viewer Memory / Source Registry UI で warning badge または warning row を表示する。
5. promote 操作時に warning metadata がある item は、validator 結果または手動確認の状態を明示する。

schema / migration:

- 第一段階は既存 `Meta map[string]interface{}` を使うため SQLite schema migration は不要にする。
- 将来 warning 検索や集計が必要になった場合のみ、`security_warnings_json` などの専用 column を検討する。

Viewer 表示:

- Source Registry table または staging list に warning badge を出す。
- badge は本文と混ぜず、metadata として表示する。
- warning detail は hover / details 表示で warning code を確認できるようにする。

### 4.3 テストと完了条件

先に書くテスト:

- sourcefetcher: injection pattern を含む fetch result が `security_warnings` metadata 付きで staging される。
- persistence: `L1StagingItem.Meta` の warning が保存 / 読み出しできる。
- viewer handler: Source Registry API response に warning metadata が含まれる。
- viewer JS: warning badge が表示される。

確認コマンド:

```bash

GOCACHE=/tmp/picoclaw-gocache go test ./...
```

E2E:

- prompt injection pattern を含む test source を Source Registry に登録する。
- run 後、Viewer で warning が metadata として見えることを確認する。
- warning 付き item が通常 memory / prompt 本文と混ざらないことを確認する。

完了条件:

- Source Registry staging に warning metadata が残る。
- Viewer で warning を確認できる。
- warning 付き外部入力が無審査で memory promote されない。

## 5. Wild 実機分散 E2E

### 5.1 現行実装済み範囲

実装済み経路:

- `internal/application/orchestrator/distributed_orchestrator_ports.go` の `DistributedOrchestrator.SetWildAgent`。
- `internal/application/orchestrator/distributed_orchestrator_routes.go` の `RouteWILD` 分岐。
- `cmd/picoclaw/runtime_distributed_mode.go` の `distOrch.SetWildAgent(wildAgent)`。
- `cmd/picoclaw/runtime_orchestrator.go` の通常 Orchestrator `SetWildAgent`。

local-only test との差:

- local test は route 分岐と mock Wild agent の呼び出し確認に留まる。
- 実機分散 E2E は distributed mode、transport、remote/local agent attribution、event log、response の一致まで確認する。

### 5.2 E2E 確認仕様

確認観点:

- 明示 `/wild` または Wild 判定 message が `RouteWILD` になる。
- distributed mode で `distOrch.SetWildAgent(wildAgent)` が有効。
- response が Wild provider / Wild agent 経由で返る。
- route evidence、event log、session_id、job_id、channel、chat_id が一致する。
- TTS stream hooks が必要な場合、RouteWILD として扱われる。
- 代替 route、CHAT fallback、local fallback に落ちた場合は失敗。

SSH / local transport 手順:

1. runtime config を確認し、distributed mode の agent / transport 設定を記録する。
2. local transport で再現できる最小 E2E を先に実行する。
3. SSH transport を使う場合、host key、接続先、remote agent 名、route mapping を記録する。
4. `/wild test` を送信する。
5. event log と response から route `WILD`、agent `wild`、job_id、session_id を照合する。

確認コマンド例:

```bash
GOCACHE=/tmp/picoclaw-gocache go test ./internal/application/orchestrator ./internal/infrastructure/transport ./cmd/picoclaw
GOCACHE=/tmp/picoclaw-gocache go test ./...
```

実機確認は環境依存のため、テストコマンドだけで完了扱いにしない。実機ログを最低 1 セッション分保存する。

失敗時の切り分け:

| 症状 | 見る場所 | 判定 |
| --- | --- | --- |
| route が CHAT | routing evidence / Mio decision | Wild 判定失敗 |
| route は WILD だが local fallback | distributed route log / transport log | E2E 失敗 |
| response が空 | raw LLM log / Wild provider log | provider 失敗 |
| job_id / session_id が欠落 | event log / execution report | trace 不備 |
| remote 到達不可 | transport log / SSH error | fallback 成功扱い禁止 |

変更が必要になった場合の候補ファイル:

- `internal/application/orchestrator/distributed_orchestrator_routes.go`
- `internal/application/orchestrator/distributed_orchestrator_ports.go`
- `internal/infrastructure/transport`
- `cmd/picoclaw/runtime_distributed_mode.go`
- `cmd/picoclaw/runtime_orchestrator.go`
- `internal/domain/routing`

完了条件:

- 実機分散接続で `RouteWILD` が Wild agent 経由で応答する。
- route evidence、event log、response が一致する。
- local fallback / CHAT fallback / 代替 route が発生していない。
- 失敗時のログと切り分け結果が残る。
