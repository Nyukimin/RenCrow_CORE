# RenCrow_STT 実装仕様（AUDIO_Client向け）

本書は、Chat から RenCrow_STT を組み込む際の実装手順と実装要件を定義します。  
運用前提は「STT サーバの IP は TTS と同一」です。  
本実装は **TDD（Red -> Green -> Refactor）必須** とします。

---

## 1. 開発プロセス前提（TDD必須）

- 全実装は `Red -> Green -> Refactor` の順序で実施する
- 失敗テスト（Red）を確認するまで、実装コードを追加しない
- 1サイクルは 1論点（1テストID）に限定する
- 受け入れはテストID単位で判定し、証跡を残す
- 分岐判定は `error.code` のみを使用する（文言一致禁止）

実装ゲート:

1. Red: 対象テストが失敗していること
2. Green: 対象テストのみ成功する最小実装であること
3. Refactor: 重複整理後も対象＋回帰テストが成功すること

---

## 2. 実装方針

1. 起動時に `GET /health` と `GET /ready` を確認する
2. `RENCROW_STT_URL`（`/stt`）へ WS 接続する
3. 接続直後に `config` を送る
4. 音声バイナリを送信する
5. `final` 受信時に Chat の既存入力フローへ橋渡しする
6. `error.code` ごとに復旧動作を実施する

---

## 3. コンポーネント責務

- STTClient
  - WS 接続/再接続
  - 送受信イベントの正規化
- STTAudioSender
  - `config` 送信
  - 音声バイナリ送信
  - 停止時 `final_pending` 送信
- STTEventBridge
  - `final` を user 発話として Router/Worker へ引き渡し
  - `error` を通知層へ引き渡し

---

## 4. 接続仕様（実装値）

### 3.1 URL 生成

- 優先: `RENCROW_STT_URL`
- 未指定時の推奨組み立て:
  - TTS のホスト/IP を流用
  - ポートを `8090`
  - パスを `/stt`

例:

- TTS: `https://192.168.1.36:8770`
- STT: `ws://192.168.1.36:8090/stt`（LAN/本番は `wss://...`）

### 3.2 接続タイムアウト/再接続

- 接続タイムアウト: 5000ms
- バックオフ: 2s -> 5s -> 10s
- 上限到達後は 10s 間隔で継続再試行

---

## 5. 送信仕様（Client -> STT）

### 4.1 `config`

接続直後に 1 回送信:

```json
{ "type": "config", "mimeType": "audio/wav" }
```

### 4.2 音声バイナリ

- 推奨: WAV PCM16 16kHz mono
- チャンク送信間隔の目安: 300ms〜700ms
- マイク停止時に `final_pending` を送信して最終確定を促す

```json
{ "type": "final_pending" }
```

---

## 6. 受信仕様（STT -> Client）

### 5.1 `session_info`

- 接続直後受信
- セッション追跡情報として保存（`session_id`）

### 5.2 `final`

- `text` を user 発話として Router/Worker に渡す
- 空文字は破棄するか、UI には非表示でログのみ残す

### 5.3 `draft` / `speech_start` / `status`

- UI 状態更新には利用可
- 会話処理の確定トリガには使わない（`final` のみ）

### 5.4 `error`

- `code` 単位で復旧分岐する
- `text` は表示用途のみ

---

## 7. エラーハンドリング実装規約

| `error.code` | 代表要因 | 推奨対応 |
|---|---|---|
| `PROVIDER_UNAVAILABLE` | バックエンド未起動 | バックオフ再接続 |
| `PROVIDER_TIMEOUT` | 推論遅延 | 再接続 + タイムアウト監視 |
| `PROVIDER_HTTP_ERROR` | バックエンドHTTP失敗 | 再接続 + 警告通知 |
| `PROVIDER_EXCEPTION` | 実行時例外 | 再接続 + 警告通知 |
| `AUDIO_TOO_SHORT` | 音声長不足 | ユーザー再入力促進 |
| `INVALID_PAYLOAD` | 不正JSON/音声形式不正 | 送信フォーマット修正 |

実装ルール:

- 分岐キーは必ず `error.code`
- 接続断で即座にマイク停止しない（通信層で復旧）
- 復旧不能時のみユーザーに明示通知

---

## 8. 既知制約（実装時の注意）

- `RENCROW_STT_BUSY_POLICY=queue_latest` は未実装（`drop` フォールバック）
- `RENCROW_STT_ENGINE=openai-api` は未実装（指定時起動エラー）
- 非 RIFF 音声を受けるには:
  - `RENCROW_STT_ALLOW_NON_RIFF=true`
  - `RENCROW_STT_DRAFT_ENABLED=true`

---

## 9. 推奨実装シーケンス（TDD順）

1. `CT-STT-001` Red: `/stt` 接続と `session_info` の失敗テスト作成
2. `CT-STT-001` Green: 接続管理（URL/timeout/backoff）を最小実装
3. `CT-STT-002` Red/Green: `config` + 音声送信 + `final` 受信を実装
4. `CT-STT-003` / `CT-STT-004` Red/Green: `error.code` 別復旧実装
5. Refactor: `session_info` / `status` のログ整理と重複除去

---

## 10. 最小テスト項目（Client）

- `CT-STT-001`: `/stt` 接続成功 + `session_info` 受信
- `CT-STT-002`: 音声送信で `final` 受信
- `CT-STT-003`: `PROVIDER_UNAVAILABLE` で再接続
- `CT-STT-004`: `INVALID_PAYLOAD` でフォーマットエラー通知
- `CT-STT-005`: `final` 受信後に既存 LLM/TTS フロー継続

詳細なテスト運用規約は `CLIENT_TDD仕様.md` を正本とする。

---

## 11. 検証コマンド例

```bash
curl -sS http://192.168.1.36:8090/health
curl -sS http://192.168.1.36:8090/ready
wscat -c ws://192.168.1.36:8090/stt
```

接続後に `{"type":"config","mimeType":"audio/wav"}` を送信し、`session_info` と `final` を確認する。
