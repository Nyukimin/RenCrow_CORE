# TTS 仕様（正本エントリ）

本書は `TTS` クライアント仕様の入口であり、実装準拠の契約を短く参照するための正本エントリです。  
詳細契約は `実装仕様.md` を正本とし、本書は要点の索引として扱います。

## 1. 対象
- `GET /health/live`
- `GET /health/ready`
- `POST /synthesis`（主経路）
- `POST /synthesize`（fallback）
- `GET /audio/{file_name}`
- `WS /sessions`
- Chat 内部イベント `tts.audio_chunk`

## 2. 最小契約
- エラー分岐は必ず `error.code` 基準で行う。
- `voice_id` は常に明示する。
- `audio_path` または `audio_url` のいずれかを返す。
- `audio_chunk_ready* -> session_completed` を満たす。
- `chunk_index` は同一 `(session_id, track)` 内で単調増加。

## 2.1 フィールド契約（必須/任意）
### `POST /synthesis` request
- 必須:
  - `text` (`string`, 1..1000 目安)
  - `voice_id` (`string`)
- 任意:
  - `speed` (`number`, `>0`)
  - `pitch` (`number`)
  - `track` (`string`)
  - `provider_params` (`object`, 許可キーのみ)

### `POST /synthesis` response
- 必須:
  - `voice_id` (`string`)
  - `audio_path` または `audio_url` のどちらか一方以上
- 任意:
  - `duration_ms` (`number|null`)
  - `request_id` (`string`)

### Chat 内部イベント `tts.audio_chunk` payload
- 必須:
  - `session_id` (`string`)
  - `response_id` (`string`)
  - `utterance_id` (`string`)
  - `chunk_index` (`number >= 0`)
  - `character_id` (`string`)
  - `text` (`string`)
  - `audio_path` または `audio_url` のどちらか一方以上
- 任意:
  - `track` (`string`, 未指定時は `default`)

## 2.2 `provider_params` 許可キー
- `model_name`, `model_file`, `speaker_id`, `speaker_name`
- `style`, `style_weight`, `language`
- `sdp_ratio`, `noise`, `noise_w`, `split_interval`
- `line_split`, `length`

## 3. 実装準拠（現行）
- Chat 側主経路は `POST /synthesis` の逐次呼び出し。
- 送信単位は句点優先分割（6..72文字目安）。
- 生成先行で実行し、再生側はキューで順序制御する。
- `tts.audio_chunk` は `response_id` / `utterance_id` を持つ。

## 4. 参照先（正本）
- 契約本文: `実装仕様.md`
- 実装・運用: `実装.md`
- 構成説明: `README.md`

```json
{
  "session_id": "chat-001-7f3a",
  "response_id": "7f3a",
  "utterance_id": "chat-001-7f3a:0002",
  "chunk_index": 2,
  "character_id": "mio",
  "text": "今日はここまでを先に話します。",
  "audio_path": "cache-a/oneshot-xxxx.wav",
  "audio_url": "http://192.168.1.36:8765/audio/oneshot-xxxx.wav"
}
```

## 6. エラーコード別の入力例と期待結果
### 6.1 `invalid_request`（未知 `provider_params`）
入力例:
```json
{
  "text": "テストです。",
  "voice_id": "mio",
  "provider_params": {
    "unknown_key": 1
  }
}
```
期待結果:
- HTTP: `400`
- `error.code`: `invalid_request`
- クライアント動作: 再試行せず入力修正へ分岐

### 6.2 `invalid_request`（`speed <= 0`）
入力例:
```json
{
  "text": "テストです。",
  "voice_id": "mio",
  "speed": 0
}
```
期待結果:
- HTTP: `400`
- `error.code`: `invalid_request`
- クライアント動作: 再試行しない

### 6.3 `voice_not_found`（未知 `voice_id`）
入力例:
```json
{
  "text": "テストです。",
  "voice_id": "unknown_voice"
}
```
期待結果:
- HTTP: `404`
- `error.code`: `voice_not_found`
- クライアント動作: 既定 `voice_id` へ 1 回フォールバック再試行（実装方針）

### 6.4 `engine_unavailable`（推論エンジン未準備）
入力例:
- `health/ready` が `ready=false` の状態で `/synthesis` を実行
期待結果:
- HTTP: `503`
- `error.code`: `engine_unavailable`
- クライアント動作: 指数バックオフで最大 2 回再試行

### 6.5 `synthesis_failed`（推論失敗）
入力例:
- エンジン内部例外やモデルロード失敗時の `/synthesis`
期待結果:
- HTTP: `500`
- `error.code`: `synthesis_failed`
- クライアント動作: 1 回のみ再試行し、失敗時は次発話へ継続

### 6.6 transport error（接続断/タイムアウト）
入力例:
- `connection reset by peer`
- `connection refused`
- `client timeout exceeded`
期待結果:
- HTTP レスポンスなし（ネットワーク失敗）
- クライアント動作: 最大 2 回再試行後に失敗記録

## 7. エラー判定の正規化ルール（クライアント）
- 比較前に `trim -> uppercase -> '-' を '_'` へ正規化する。
- 例:
  - `voice_not_found` -> `VOICE_NOT_FOUND`
  - `engine-unavailable` -> `ENGINE_UNAVAILABLE`
