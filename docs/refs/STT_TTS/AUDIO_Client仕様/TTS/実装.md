# TTS 実装（正本）

## 1. 実装方針
- 実装は `POST /synthesis` を主経路にする。
- `POST /synthesize` は互換確認のみで利用する。
- エラー処理は `error.code` で統一する。
- 1PR 1論点で小さく進める。

## 2. 環境・起動確認
1. TTS サーバ起動
2. `GET /health/ready` で `status=ready` を確認
3. Chat 側を起動

推奨設定例（Chat 側）:
- `RENCROW_TTS_URL`
- `RENCROW_TTS_WS_URL`
- `RENCROW_TTS_DEFAULT_VOICE_ID`
- `RENCROW_TTS_MODE`（`oneshot` / `streaming`）
- `RENCROW_TTS_AUDIO_ROOT`

## 3. TDD 実施ルール
- Red: 失敗テストを先に追加
- Green: 最小実装で通す
- Refactor: 重複を整理し回帰を通す
- 1サイクルは1論点（境界値1つ、エラー1つなど）に限定

## 4. テストID運用
- Health: `CT-HL-*`, `CT-HR-*`
- Synthesis: `CT-SY-*`
- Synthesize fallback: `CT-SZ-*`
- Audio: `CT-AU-*`
- WS: `CT-WS-*`
- 推奨追補（未契約）: `REC-*`

## 5. 優先実装バックログ

### P1（必須）
- `CT-HL-001`: `/health/live` 正常
- `CT-HR-001`: `/health/ready` 正常
- `CT-SY-001`: `/synthesis` 最小正常
- `CT-SY-002B`: `text` 超過で `400 invalid_request`
- `CT-SY-002C`: `speed <= 0` で `400 invalid_request`
- `CT-SY-002D`: `length <= 0` で `400 invalid_request`
- `CT-SY-004`: 未知 `provider_params` キー拒否
- `CT-SY-005`: 未知 `voice_id` で 404 系

### P2（重要）
- `CT-AU-001`: `/audio/{file}` 正常取得
- `CT-AU-003`: path traversal 拒否
- `CT-WS-001`: `audio_chunk_ready* -> session_completed`
- `CT-WS-003`: `INVALID_SEQ`
- `CT-WS-004`: `UNKNOWN_MESSAGE_TYPE`

### P3（補強）
- `CT-SZ-001`: `/synthesize` 互換確認
- `REC-ER-001`: 数値変換失敗時の契約エラー化
- `REC-ER-002`: `pitch` の扱い明文化

## 6. SBV2 実装時の注意
- `provider_params` は許可キーのみ送る。
- `line_split` は bool 正規化して送る。
- `language` は列挙内のみ送る。
- `X-RenCrow-TTS-Request-Id` を付与し、`request_id` と相関ログを取る。

## 6.1 文章同期制御（現行実装）
- 送信（生成）と再生を分離し、生成側は先行して `POST /synthesis` を進める。
- 再生側は `tts.audio_chunk` をキュー保持し、`(session_id, track, chunk_index)` で順序制御する。
- 同期キー:
  - `response_id`: 応答単位（`job_id`）
  - `utterance_id`: 発話断片単位（`{session_id}:{chunk_index4桁}`）
- UI同期:
  - 再生開始時に `character_id` と `text` を「Now Playing」表示へ反映
  - 再生終了/エラー時に表示をクリア
- 完了制御:
  - TTS stream forwarder は内部キューを drain してから `session_end` 相当を完了する（途中打ち切り禁止）

## 7. 通信運用ルール（必須）
- timeout 既定値:
  - HTTP `/synthesis`: 30 秒
  - WS 接続確立: 10 秒
  - WS チャンク待ち: 15 秒
  - WS 完了待ち: 20 秒
- リトライ方針:
  - `engine_unavailable` / `ENGINE_UNAVAILABLE`: 指数バックオフで 2 回再試行
  - `voice_not_found` / `VOICE_NOT_FOUND`: 既定 `voice_id` で 1 回再試行
  - `invalid_request`: 再試行せず入力修正
  - `synthesis_failed` / `SYNTHESIS_FAILED`: 1 回のみ再試行し、失敗時は次発話へ継続
- transport error リトライ:
  - `connection reset by peer`, `connection refused`, timeout 系は最大2回再試行
- コード比較前に `error.code` を正規化（trim -> uppercase -> `-` を `_` へ変換）する。
- timeout または接続断は無限待ちせず失敗として記録する。

## 11. RenCrow_TTS の使い方（詳細）
### 11.1 目的
- Chat から RenCrow_TTS を呼び出して、文章を音声へ変換し再生する。
- 現行実装は「生成先行 + 再生キュー制御 + テキスト同期表示」を採用する。

### 11.2 最小設定（`config.yaml`）
```yaml
tts:
  enabled: true
  output_dir: "./workspace/tts"
  http_base_url: "http://192.168.1.31:7870"
  timeout_ms: 15000
  voice_id: "mio"
  provider_priority: ["irodori", "sbv2", "azure", "eleven"]
  playback_commands: []
  irodori:
    enabled: true
    base_url: "http://192.168.1.31:7870"
    voice_id: "Female_01"
    voice_name: "Mio"
    reference_audio: "/Users/yukimi/GenerativeAI/TTS/irodori/voices/Female_01_Mio/reference.wav"
    reference_audio_url: "http://192.168.1.31:8090/voice_profile_tests/Female_01_Mio_test.wav"
    model_device: "mps"
    model_precision: "fp32"
    codec_device: "mps"
    codec_precision: "fp32"
    enable_watermark: false
    num_steps: 40
  sbv2:
    enabled: false
```
- `playback_commands: []` は browser-only 再生モード（推奨）。
- Irodori はMac上のGradio APIをRenCrowサーバ側から呼び、生成WAVを `output_dir` へ保存する。
- Viewer は `tts.audio_chunk.audio_path` を受け取り、同一originの `/viewer/tts/audio?path=...` からWAVを再生する。

### 11.3 起動前チェック
1. Irodori-TTS サーバ疎通確認:
   - `curl -sS http://192.168.1.31:7870/gradio_api/info`
2. 期待:
   - `/_run_generation`
   - `/_load_model`
3. Chat サーバ起動:
   - `go build -o rencrow ./cmd/rencrow`
   - `bash ./start_rencrow.sh`

### 11.4 単体動作確認（HTTP）
```bash
curl -sS -X POST "http://192.168.1.31:7870/gradio_api/run/_run_generation" \
  -H "Content-Type: application/json" \
  -d '{"data":["Aratako/Irodori-TTS-500M-v2","mps","fp32","mps","fp32",false,"テストです。",null,16,1,"","independent",3.0,5.0,"",0.5,1.0,true,"","","","","0.9",""]}'
```
- 成功条件:
  - `audio_path` または `audio_url` が返る
  - `voice_id` が返る

### 11.5 Chat 統合時のデータフロー
1. Chat が `StartSession` を開始（`session_id`, `response_id` を確定）
2. LLM 出力を句点優先で分割（6..72文字）
3. 各チャンクを `/synthesis` に送信
4. 応答ごとに `tts.audio_chunk` を発行
5. Viewer がキューに格納し、`chunk_index` 順に再生
6. 再生中は `Now Playing` に `character_id + text` を表示
7. 送信キュー drain 後に `EndSession`

### 11.6 同期制御で見るべきキー
- `session_id`: TTS セッション単位
- `response_id`: 応答単位（`job_id`）
- `utterance_id`: 発話断片単位（`{session_id}:{chunk_index4桁}`）
- `chunk_index`: 再生順制御キー（同一 `session_id/track` で単調増加）

### 11.7 よくある失敗と対処
- `voice_not_found`:
  - `voice_id` と TTS 側 `voices` の一致を確認
- `synthesis_failed` / 接続リセット:
  - TTS 側プロセス負荷、GPU/モデル状態、ネットワーク断を確認
  - transport error はクライアント側で最大2回再試行される
- 音が出ない:
  - viewer のスピーカー解除（autoplay 許可）
  - `audio_url` が `/audio/...` へ正規化されているか確認
- 口パクだけ動く:
  - `tts.audio_chunk` に `audio_path/audio_url` が入っているか確認

### 11.8 運用ログでの確認ポイント
- Chat ログ:
  - `TTS RenCrow bridge enabled (/synthesis base=...)`
  - `tts.audio_chunk` 発行数と `chunk_index`
- TTS ログ:
  - `X-RenCrow-TTS-Request-Id` と応答の対応
- Viewer:
  - 再生キュー消化、`Now Playing` 表示、口パク遷移

### 11.9 安定運用の推奨値
- `style`: `Neutral`
- `style_weight`: `2.0`（違和感があれば 1.6..2.0 を調整）
- `line_split`: `false`
- `timeout_ms`: 15000..30000（ネットワーク品質に応じて）
