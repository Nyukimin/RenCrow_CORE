# TTS 実装仕様（正本）

## 0. 先に見る最小仕様（実装準拠）
### 0.1 送受信フロー（Chatクライアント）
1. `StartSession(session_id, response_id, character_id, voice_id)`
2. LLMトークンを句点優先でチャンク化（6..72文字目安）
3. チャンクごとに `POST /synthesis`
4. 応答ごとに `tts.audio_chunk` を発行（`response_id`, `utterance_id`, `chunk_index` を含む）
5. 全チャンク送信キューを drain 後に `EndSession`

### 0.2 `tts.audio_chunk` 例（正本）
```json
{
  "session_id": "chat-001-7f3a",
  "response_id": "7f3a",
  "utterance_id": "chat-001-7f3a:0002",
  "chunk_index": 2,
  "character_id": "mio",
  "text": "今日はここまでを先に話します。",
  "audio_path": "cache-a/oneshot-xxxx.wav",
  "audio_url": "http://192.168.1.36:8765/audio/oneshot-xxxx.wav"
}
```

### 0.3 再生順序契約（viewer）
- キューは `(session_id, track, chunk_index)` を主キーに昇順再生する。
- `track` 欠落時は `default` を採用する。
- 再生開始で `Now Playing` に `character_id + text` を表示し、終了/エラーでクリアする。

## 1. 目的
本仕様は、Chat から利用する TTS サーバの契約を定義する。  
現行 Provider は SBV2 だが、契約は可能な限り Provider 非依存で維持する。

## 2. 対象API
- `GET /health/live`
- `GET /health/ready`
- `POST /synthesis`（主経路）
- `POST /synthesize`（Bridge fallback）
- `GET /audio/{file_name}`
- `WS /sessions`

## 3. 基本契約
- Client のエラー分岐は必ず `error.code` で行う（`message` 文言では分岐しない）。
- `voice_id` は常に明示する。
- `audio_path` または `audio_url` のいずれかを返す（推奨: 両方）。
- ストリーミングは `session_start -> text_delta* -> session_end` を受理する。
- `audio_chunk_ready` は 0 回以上返却可能で、最後に `session_completed` を返す。
- 順序契約として `chunk_index` の単調増加を満たす。

## 3.1 現行クライアント実装準拠（2026-04-26）
- Chat 側の主経路は `POST /synthesis` の逐次呼び出し（WS `/sessions` は契約定義を維持）。
- TTS セッションIDは `"{chat_session_id}-{job_id}"` を生成して利用する。
- `response_id` は `job_id` を採用する。
- `utterance_id` は `"{tts_session_id}:{chunk_index4桁}"` を採用する（例: `sess-abc:0003`）。
- `OnSessionCompleted` は `session_id` に加えて `character_id` を返す契約とする。

## 4. API 詳細

### 4.1 `GET /health/live`
- 期待: `200` + `{ "status": "live" }`

### 4.2 `GET /health/ready`
- 期待: `200` + `status=ready`
- 代表項目:
  - `voices`（`string[]` を正本とする）
  - `device`
  - `cuda_available`
  - `capabilities.streaming`
  - `capabilities.multi_chunk`
  - `capabilities.multi_track`
- 補足:
  - Provider 実装によっては `voices` がオブジェクト配列になる場合があるが、Client 契約上は `voice_id` の解決に必要な `string[]` として扱う。

### 4.3 `POST /synthesis`（主経路）
- 必須リクエスト:
  - `text`
  - `voice_id`
- 推奨ヘッダ:
  - `X-RenCrow-TTS-Request-Id`
- 任意パラメータ:
  - `emotion`, `speed`, `pitch`, `track`
  - `provider_params`（SBV2 制御用）
- 成功時:
  - `audio_path` または `audio_url`
  - `duration_ms`（数値または `null`）
  - `voice_id`
- `provider_params` 許可キー（正本）:
  - `model_name`
  - `model_file`
  - `speaker_id`
  - `speaker_name`
  - `style`
  - `style_weight`
  - `language`
  - `sdp_ratio`
  - `noise`
  - `noise_w`
  - `split_interval`
  - `line_split`
  - `length`
- Client 実装上の補足:
  - リクエストヘッダ `X-RenCrow-TTS-Request-Id` は `"{sanitized_session_id}-{chunk_index4桁}"` を送る。
  - transport error は最大2回まで再試行する（`connection reset/refused`, timeout 系）。

### 4.4 `POST /synthesize`（fallback）
- 互換確認用途として維持する。
- 成功時は `text`, `audio_path`/`audio_url`, `chunk_index`, `track` を返す。

### 4.5 `GET /audio/{file_name}`
- `/synthesis` 等で取得した音声ファイルを返す。
- path traversal（`../` 等）は拒否する。

### 4.6 `WS /sessions`
- Client -> Server:
  - `session_start`（必須: `type`, `session_id`, `voice_id`）
  - `text_delta`（必須: `type`, `session_id`, `seq`, `text`。`seq` は単調増加）
  - `session_end`（必須: `type`, `session_id`）
- Server -> Client:
  - `audio_chunk_ready`（必須: `type`, `session_id`, `chunk_index`。推奨: `track`, `audio_path` or `audio_url`）
  - `session_completed`（必須: `type`, `session_id`）
  - `error`（必須: `type`, `code`）

### 4.7 Chat 内部イベント `tts.audio_chunk`（実装契約）
- 発行元: Chat クライアントブリッジ（TTS応答受領時）
- 必須ペイロード:
  - `session_id`
  - `response_id`
  - `utterance_id`
  - `chunk_index`
  - `character_id`
  - `text`
  - `audio_path` または `audio_url`
- 順序契約:
  - 同一 `(session_id, track)` 内では `chunk_index` 昇順で再生する。
  - `track` が欠落した場合は `default` として扱う。

## 5. SBV2 実装準拠（参考を統合した契約化）
- `text`: `1..max_text_length`（既定 1000、環境変数で変更可）
- `speed`: `> 0`
- `length`: `> 0`（指定時は `speed` 換算より優先）
- `pitch`: 現行は閾値チェックなし
- `provider_params`:
  - 許可キーのみ受理（未知キーは `400 invalid_request`）
  - `style_weight`, `sdp_ratio`, `noise`, `noise_w`, `split_interval` は現行チェックなし
  - `line_split` は bool 解釈（`"1"`, `"true"`, `"yes"`, `"on"` を true）
  - `language` は SBV2 列挙外で `400 invalid_request`
- チャンク送信単位（Chat 側）:
  - 句点系（`。！？.!?\n`）を優先境界に分割する。
  - 補助境界として読点系（`、，,;；:：`）と空白を利用する。
  - 1チャンク最小文字数は 6、最大文字数は 72 を基準とする。
  - 生成は先行実行し、再生側キューで順序制御する。

## 6. エラー契約
- HTTP 代表:
  - `invalid_request`（400）
  - `text_too_long`（400、実装により `invalid_request` へ包含されうる）
  - `voice_not_found`（404）
  - `audio_not_found`（404）
  - `conflict`（409）
  - `engine_unavailable`（503）
  - `synthesis_failed`（500）
- WS 代表:
  - `INVALID_SEQ`
  - `VOICE_NOT_FOUND`
  - `SESSION_NOT_FOUND`
  - `UNKNOWN_MESSAGE_TYPE`
  - `SYNTHESIS_FAILED`
  - `ENGINE_UNAVAILABLE`

## 7. エラーコード正規化ルール（Client）
- 大文字小文字差を吸収するため、Client は比較前に正規化する。
- 正規化ルール:
  1. 前後空白を除去
  2. 英字を大文字化
  3. `-` を `_` に統一
- 例:
  - `voice_not_found` -> `VOICE_NOT_FOUND`
  - `engine-unavailable` -> `ENGINE_UNAVAILABLE`

## 8. タイムアウト契約（実装既定）
- HTTP `/synthesis`: 30 秒（超過時は fail-fast 扱い）
- WS:
  - 接続確立待ち: 10 秒
  - `text_delta` 送信後のチャンク待ち: 15 秒
  - `session_end` 後の完了待ち: 20 秒
- いずれも timeout 時は `error` として扱い、無限待ちを禁止する。

## 9. 受け入れ基準
1. `GET /health/live` が 200 + `status=live`
2. `GET /health/ready` が 200 + `status=ready`
3. `POST /synthesis` が 2xx + `audio_path` または `audio_url`
4. 不正入力が `error.code` 付きで契約どおり返る
5. `WS /sessions` が順序契約と完了契約を満たす

## 10. TDD 適用範囲（正本）
- 本仕様の変更・実装は TDD（Red -> Green -> Refactor）を必須とする。
- 1 サイクル 1 論点（1 つの境界値、1 つのエラー契約、1 つのイベント契約）で進める。
- 実装コードの追加・変更より先に、対応する契約テスト ID を追加する。

## 11. TDD テストID契約
- Health:
  - `CT-HL-001`: `GET /health/live` 正常応答
  - `CT-HR-001`: `GET /health/ready` 正常応答

  - `CT-SY-001`: 最小正常系（`text`, `voice_id`）

  - `CT-SY-002D`: `length <= 0` で `400 invalid_request`
  - `CT-SY-003`: `X-RenCrow-TTS-Request-Id` 伝播
  - `CT-SY-004`: 未知 `provider_params` キー拒否

  - `CT-SY-006`: `engine_unavailable` 系で 503
- Synthesize fallback:
  - `CT-SZ-001`: 互換レスポンス確認
- Audio:

  - `CT-AU-002`: 不在ファイルで 404

  - `CT-WS-001`: `audio_chunk_ready* -> session_completed`

  - `CT-WS-003`: `INVALID_SEQ`
  - `CT-WS-004`: `UNKNOWN_MESSAGE_TYPE`

## 12. Red / Green / Refactor 完了条件
- Red 完了条件:
  - 対象 ID のテストが失敗し、失敗理由が契約差分に一致する。
  - 失敗が既存不具合ではなく、今回追加したテストの期待によることを確認する。
- Green 完了条件:
  - 対象 ID のテストが成功する。
  - 同カテゴリ既存テスト（同じ API 群）が回帰しない。
- Refactor 完了条件:
  - 重複コードを削減しても対象 ID と既存回帰が維持される。
  - `error.code` 分岐と timeout 契約が退行していない。

## 13. トレーサビリティ規約
- すべてのテスト名に契約 ID（`CT-*` / `REC-*`）を含める。
- 実装 PR には少なくとも以下を記録する:
  - 対象契約 ID
  - 追加/変更したエンドポイント
  - Red 時の失敗内容
  - Green 後の検証結果
- 契約変更時は、本仕様とテスト ID を同一 PR で更新する。
