# STT 既知良好WAV 分離テスト結果（2026-04-10）

## 1. テスト目的

既知良好WAVを基準信号として、以下の4分類を1回で切り分ける。

- A: 入力WAV品質/形式
- B: whisper-server変換/推論経路
- C: voice-bridge連携層
- D: 環境依存（ffmpeg/パス/権限/起動引数）

## 2. 実施条件

- 実行環境: Windows側
- STT: `whisper-server` (`http://127.0.0.1:8080/inference`)
- Gateway: `voice-bridge` (`ws://127.0.0.1:8090/ws`)
- 既知良好WAV: `D:\RenCrow\tmp\known-good.wav`

## 3. 5ステップ結果

### Step1: 既知良好WAV固定

- 実施:
  - TTSで音声生成し `known-good.wav` を固定
  - `ffmpeg` で `mono/16kHz/PCM16/RIFF` へ変換
  - `ffprobe` でメタ確認
- 結果: PASS
  - `codec_name=pcm_s16le`
  - `sample_rate=16000`
  - `channels=1`
  - `bits_per_sample=16`
  - `format_name=wav`
  - `duration=4.574313`
- 判定: A（入力品質/形式）は除外

### Step2: Whisper単体HTTP

- 実施:
  - `curl.exe -X POST http://127.0.0.1:8080/inference -F "file=@D:\RenCrow\tmp\known-good.wav" -F "response_format=json"`
- 結果: FAIL
  - 応答: `{"error":"FFmpeg conversion failed."}`
- 判定: B または D へ進む

### Step3: Whisperログ突合

- stdout 観測:
  - `Received request: audio.wav`
  - `Successfully loaded audio.wav`
- stderr 観測（直近失敗シグネチャ）:
  - `whisper_full_with_state: failed to decode`
  - `whisper_full_with_state: failed to encode`
  - `client disconnected, aborted processing`
- 判定:
  - 入力読み込みは通過しているため、単純な input-open 系ではない
  - B（推論/内部デコード経路）寄り

### Step4: voice-bridge経由

- 実施:
  - `ws://127.0.0.1:8090/ws` に `config` + 同一WAVを送信
- 結果:
  - 受信: `speech_start`（確認）
  - `draft/final` は未取得
  - 追加で `invalid JSON` が1回観測（WS受信処理中のテストスクリプト側副作用）
- 判定:
  - 直叩きでも失敗しているため C単独より B/D 優先

### Step5: 設定1項目ずつ差分試験

- 実施キー（1回ずつ再起動）:
  - `STT_MIN_AUDIO_BYTES=1`
  - `STT_TIMEOUT_MS=60000`
  - `STT_ALLOW_NON_RIFF=1`
  - `STT_BUSY_POLICY=drop`
- 観測（全ケース共通）:
  - `speech_start=1 draft=0 final=0 error=0`
- 判定:
  - 連携層パラメータ変更で改善せず
  - Cの可能性は低く、B/D を優先

## 4. 最終切り分け判定

- **主因候補: B（whisper-server変換/推論経路）**
- 補助候補: D（環境依存の一部）
- 除外寄り:
  - A（入力WAV品質/形式）
  - C（voice-bridgeパラメータ差）

## 5. 次アクション（推奨）

1. whisper-server を `--convert` なし設定で単体比較（同一WAV）
2. whisper.cpp同梱のサンプルWAVで `/inference` を再試験
3. Whisper再起動直後のログ先頭から、該当リクエストID相当で突合（decode/encode発生点の前後20行）
4. 必要なら whisper.cpp バイナリ/モデル組み合わせ（ビルド版とモデル版）を固定化
