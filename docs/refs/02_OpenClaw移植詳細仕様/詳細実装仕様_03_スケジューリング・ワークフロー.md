# 詳細実装仕様 03: スケジューリング・ワークフロー

**更新日**: 2026-03-26
**ステータス**: 差分整理（全項目未実装）
**親文書**: `OpenClaw機能差分比較表_20260326.md`

---

## 1. 概要

RenCrow にはアプリケーションレベルのスケジューリング機能がほぼ存在しない。
OpenClaw は内蔵 Cron スケジューラーと Lobster ワークフローエンジンを備え、時間駆動・イベント駆動の完全な自動化が可能。

**現状の差**:

| 項目 | OpenClaw | RenCrow |
|------|:--------:|:-------:|
| 内蔵 Cron スケジューラー（one-shot / 繰り返し / cron 式） | ✓ | ✗ |
| Cron CLI（add/list/run/edit） | ✓ | ✗ |
| Lobster ワークフロー（多ステップ決定論的パイプライン） | ✓ | ✗ |
| 承認チェックポイント付きパイプライン | ✓ | ✗ |
| 再開トークン（resumeToken） | ✓ | ✗ |
| 汎用 Inbound Webhook 自動化 | ✓ | △（LINE Webhook のみ） |
| Gmail Pub/Sub トリガー | ✓ | ✗ |
| イベントドリブン Hooks（auth-monitoring / polling / standing orders） | ✓ | ✗ |
| Ollama ヘルスチェック / 自動再起動 | △ | ✓（RenCrow 独自） |

## 2. OpenClaw のスケジューリング設計

### 2.1 Cron スケジューラー

OpenClaw の Gateway に内蔵。ジョブは `~/.openclaw/cron/jobs.json` に永続化される。

**ジョブタイプ**:

```
one-shot   : at <datetime>       — 一度だけ実行
recurring  : every <interval>    — 繰り返し（例: every 1h, every 30m）
cron-expr  : cron <expression>   — 標準 cron 式（例: 0 9 * * 1-5）
```

**CLI**:
```
openclaw cron add "毎朝9時にメール確認" --cron "0 9 * * *" --prompt "メールを確認してサマリーを送って"
openclaw cron list
openclaw cron run <job-id>        # 即時トリガー
openclaw cron edit <job-id>
openclaw cron delete <job-id>
```

**ジョブ定義（JSON）**:
```json
{
  "id": "job_abc123",
  "name": "朝のメール確認",
  "schedule": "0 9 * * *",
  "prompt": "メールを確認してサマリーを送って",
  "agent": "mio",
  "channel": "line",
  "enabled": true,
  "last_run": "2026-03-26T09:00:00Z",
  "next_run": "2026-03-27T09:00:00Z"
}
```

### 2.2 Lobster ワークフローエンジン

複数ステップを決定論的に実行するパイプライン。YAML または JSON 形式の `.lobster` ファイルで定義。

```yaml
# example.lobster
name: daily-report
steps:
  - id: fetch-data
    prompt: "昨日のアクセスログをまとめて"
    agent: shiro

  - id: analyze
    prompt: "fetch-data の結果を分析してトレンドを抽出して"
    agent: gin
    depends_on: [fetch-data]

  - id: approve
    type: checkpoint        # ユーザー承認ゲート
    message: "このレポートを送信してよいですか？"

  - id: send
    prompt: "分析結果を LINE に送って"
    agent: mio
    depends_on: [analyze, approve]
```

**再開トークン**:
- チェックポイントで停止するとき `resumeToken` を発行
- ユーザーが承認した後 `openclaw resume <resumeToken>` で続行
- トークンは `~/.openclaw/lobster/pending/` に保存

## 3. RenCrow の現状

### 3.1 現行スケジューリング機能

| 機能 | 実装 | 詳細 |
|------|------|------|
| 日次カットオーバー | ✓ | 毎日 00:00 にセッションリセット（hardcoded） |
| Ollama ヘルスチェック | ✓ | 60 秒間隔で Ollama 状態確認・自動再起動 |
| IdleChat monitorLoop | ✓ | BreakInterval（200/500/1000ms）のタイマー |
| ログローテーション | ✓ | cron（OS crontab）で毎日 04:00 実行 |

日次カットオーバーとログローテーションは OS cron に依存しており、アプリケーション内にスケジューリングエンジンは存在しない。

### 3.2 なぜ必要か

現在 RenCrow でスケジュール実行をするには、ユーザーが手動でコマンドを送るか、OS cron でシェルコマンドを実行するしかない。
定型タスクの自動化（朝のサマリー送信、週次レポート、定期データ取得）ができない。

## 4. 実装方針

### Priority 1: 内蔵 Cron スケジューラー

**実装場所**: `internal/application/scheduler/`

```go
// CronJob はスケジュールジョブを表す
type CronJob struct {
    ID          string    `json:"id"`
    Name        string    `json:"name"`
    Schedule    string    `json:"schedule"`  // cron 式 or "every 1h" or "at 2026-03-27T09:00:00Z"
    Prompt      string    `json:"prompt"`
    AgentRoute  string    `json:"agent_route"`  // "CHAT" / "CODE3" 等
    Channel     string    `json:"channel"`       // "line" / "slack" / "cli"
    Enabled     bool      `json:"enabled"`
    LastRun     time.Time `json:"last_run"`
    NextRun     time.Time `json:"next_run"`
}

type CronScheduler interface {
    Add(job CronJob) error
    Remove(id string) error
    List() []CronJob
    RunNow(id string) error
    Start(ctx context.Context) error
}
```

永続化: `~/.rencrow/cron/jobs.json`

既存の `robfig/cron` パッケージ（Go 標準的ライブラリ）を使用し、ジョブ実行時は `MessageOrchestrator.HandleMessage()` を呼ぶ。

**コマンドインターフェース**（チャットコマンド経由）:
```
/cron add "名前" --every 1h --prompt "xxxxxx"
/cron add "名前" --cron "0 9 * * *" --prompt "xxxxxx"
/cron list
/cron delete <id>
/cron run <id>
```

### Priority 2: Lobster ワークフロー（簡易版）

RenCrow に必要な最小実装:

```go
// LobsterWorkflow は多ステップワークフローを定義する
type LobsterWorkflow struct {
    Name  string         `yaml:"name"`
    Steps []LobsterStep  `yaml:"steps"`
}

type LobsterStep struct {
    ID         string   `yaml:"id"`
    Type       string   `yaml:"type"`        // "prompt" or "checkpoint"
    Prompt     string   `yaml:"prompt"`
    AgentRoute string   `yaml:"agent_route"`
    DependsOn  []string `yaml:"depends_on"`
}
```

実装: `internal/application/workflow/lobster_runner.go`

チェックポイントは Slack/LINE でユーザーに承認を求め、`resumeToken` は DuckDB に保存。

### Priority 3: イベントドリブン Hooks

```yaml
hooks:
  - event: "message.received"
    filter:
      channel: "line"
      contains: "レポート"
    action:
      type: "run_workflow"
      workflow: "daily-report.lobster"
```

実装: `internal/application/scheduler/hook_engine.go`

### Priority 4: Gmail Pub/Sub トリガー（保留）

Google Cloud Pub/Sub への依存が発生するため、現時点では優先度低。
必要になった時点で `internal/adapter/inbound/gmail/` として実装する。

## 5. 既存機能との整合

- 日次カットオーバー（00:00 リセット）は Cron スケジューラーに移行できる（`0 0 * * *` に相当）
- Ollama ヘルスチェックは RenCrow 独自機能として維持。Cron スケジューラーとは独立したループとする
- IdleChat の BreakInterval タイマーとは別系統のスケジューラー
- ジョブ実行は既存の `MessageOrchestrator` の `HandleMessage()` を経由することで、ルーティング・セーフガード・ログが自動適用される

---

## 6. 既知の制約

- LINE の 5 秒 Webhook タイムアウトはスケジューラーには関係しない（push API を使うため）
- ただし Slack 等の応答型 Webhook とは区別して、スケジューラー起点のメッセージは push モードで送信する
- cron 式パース: `github.com/robfig/cron/v3` （Go のデファクト標準）を採用
