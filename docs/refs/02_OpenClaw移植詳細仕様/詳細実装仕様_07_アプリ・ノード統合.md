## 2. OpenClaw のノード設計

### 2.1 ノードアーキテクチャ

OpenClaw は「Gateway（サーバー）」と「Node（クライアントデバイス）」の 2 層構成。

```
Gateway (Linux/VPS)
  ├── チャネル受信（LINE / Discord / Telegram）
  ├── LLM ルーティング
  └── Node への指令

Node
  ├── macOS ノード  : メニューバーアプリ、Finder 統合、Spotlight
  ├── iOS ノード    : Shortcuts 連携、音声入力、カメラ
  └── Android ノード: SMS/通知/位置情報、音声ウェイクワード
```

Gateway は WebSocket で各 Node に接続し、ツール実行を Node 側で行わせる。

### 2.2 macOS ノード

- メニューバーアプリとして常駐（Swift / SwiftUI）
- ショートカットキーでチャット入力
- スクリーンショット取得
- Finder ファイル選択
- TTS 再生（macOS の AVSpeechSynthesizer）

```yaml
nodes:
  macos:
    enabled: true
    tts:
      engine: "system"  # macOS AVSpeechSynthesizer
      voice: "Kyoko"
    hotkey: "Cmd+Shift+Space"
```

### 2.3 iOS ノード

- Shortcuts アプリ経由でコマンド送信
- カメラで撮影した画像を LLM に渡す
- iOS の音声認識（SFSpeechRecognizer）による音声入力

### 2.4 Android ノード

```
Termux + OpenClaw Android ノード
```

- 通知読み上げ・送信
- GPS 位置情報取得
- SMS 送受信（Accessibility Service）
- Voice Wake（Porcupine / Vosk）

### 2.5 Voice Wake + Talk Mode

```yaml
nodes:
  android:
    voice:
      wake_word: "ねえクロー"        # ウェイクワード
      engine: "porcupine"
    talk_mode:
      enabled: true
      silence_timeout_ms: 2000     # 2秒無音で発話終了と判定
      tts_engine: "sbv2"
```

Talk Mode: ウェイクワード → 音声認識 → LLM → TTS という連続ループ。

### 2.6 Canvas + A2UI

エージェントがメッセージだけでなく UI コンポーネント（ボタン・フォーム・カード）を生成できる仕組み。

```json
{
  "type": "canvas",
  "components": [
    { "type": "text", "content": "どのタスクを実行しますか？" },
    { "type": "button", "label": "コードレビュー", "action": "/code review" },
    { "type": "button", "label": "テスト実行", "action": "/ops test" }
  ]
}
```

## 3. RenCrow の現状

### 3.1 実装済み音声・ノード機能

| 機能 | 実装 | 詳細 |
|------|------|------|
| TTS（SBV2） | ✓ | Style-Bert-VITS2、日本語高品質音声 |
| VTuber ブリッジ | ✓ | VTuberアバター連携（RenCrow 独自） |
| Windows エージェントノード | ✓ | rencrow-agent.exe（SSH Transport） |

### 3.2 RenCrow の音声スタック

```
ユーザー発話
  → 音声認識（外部 STT）
  → LINE / Slack テキストとして送信
  → RenCrow 処理
  → テキスト応答
  → SBV2 TTS → 音声再生
  → VTuber ブリッジ → アバター口パク同期
```

現状は STT（Speech-to-Text）が未統合。音声入力はユーザーが外部ツールでテキストに変換してから送信する必要がある。

## 4. 実装方針

### Priority 1: STT 統合（音声入力）

**理由**: VTuber ブリッジと TTS は既にあり、STT があれば音声入出力のループが完成する。

実装概要:
- `internal/adapter/inbound/voice/stt_handler.go`
- Whisper（ローカル）または Google Cloud STT（クラウド）を選択可能
- `ChannelAdapter` インターフェースに `RecordAndTranscribe()` を追加

```yaml
voice:
  stt:
    engine: "whisper"          # whisper / google_cloud_stt
    model: "small"             # whisper モデルサイズ
    language: "ja"
  wake_word:
    enabled: false             # 将来実装
```

### Priority 2: Voice Wake（ウェイクワード）

ウェイクワード検出で常時録音の電力・CPU 消費を最小化:

```go
type WakeWordDetector interface {
    Start(ctx context.Context) (<-chan WakeEvent, error)
    Stop() error
}
```

実装候補:
- Vosk（OSS、ローカル動作、日本語モデルあり）
- Porcupine（Picovoice、軽量、ただし商用ライセンス）

Vosk を優先（OSS・日本語対応）。

### Priority 3: Talk Mode

```go
type TalkMode struct {
    stt           STTProvider
    wakeDetector  WakeWordDetector
    orchestrator  MessageOrchestrator
    tts           TTSProvider
    silenceTimeout time.Duration  // デフォルト 2秒
}

func (tm *TalkMode) Run(ctx context.Context) error {
    for {
        // 1. ウェイクワード待機
        <-tm.wakeDetector.Events()
        // 2. 音声録音 → STT
        text := tm.stt.RecordUntilSilence(ctx, tm.silenceTimeout)
        // 3. LLM 処理
        reply := tm.orchestrator.HandleMessage(ctx, text)
        // 4. TTS 再生
        tm.tts.Speak(ctx, reply)
    }
}
```

### Priority 4: macOS ノード（将来）

Go でクロスプラットフォームアプリ（systray パッケージ）を作成するか、Swift でネイティブ macOS アプリを作成するかを選択する。
現時点では RenCrow の主要デプロイ先が Linux（$10 デバイス）であるため、macOS ノードは低優先度。

### Priority 5: Android ノード（将来）

Termux + Go バイナリとして動作する最小実装を想定。
デバイスコマンド（SMS/通知）は Android Accessibility Service が必要なため、APK ビルドが必要。
現時点では優先度低。

### スコープ外

| 機能 | 理由 |
|------|------|
| Canvas + A2UI | LINE/Slack の UI コンポーネント制限が強い。将来的に Slack Block Kit で部分実現可能 |
| iOS ノード | Apple の Shortcuts 制限・AltStore 配布の複雑さ |
| カメラ / スクリーン録画 | マルチモーダル LLM への画像送信は別仕様で対応（要件が出た時点で実装） |

## 5. 既存機能との整合

- SBV2 TTS は `internal/adapter/outbound/tts/sbv2/` に実装済み。STT からのテキストも同じ TTS パイプラインを経由する
- VTuber ブリッジは TTS 出力と連動しているため、Talk Mode でも自動的に口パク同期が動作する
- Windows エージェントノード（rencrow-agent.exe）は SSH Transport で Gateway と接続する既存設計を維持

---

## 6. 既知の制約

- Whisper の `small` モデルは約 240MB。メモリ制約 <10MB はデーモン本体の目標であり、Whisper はオンデマンドロードで対応する
- Talk Mode はローカルデバイス前提のため、LINE/Slack 等のリモートチャネルには不適用（ローカル CLI チャネルのみ）
- Vosk 日本語モデルは約 1.8GB。Raspberry Pi 等の $10 デバイスではストレージ制約になる可能性がある。クラウド STT（Google Cloud Speech-to-Text）をフォールバックとして用意する
