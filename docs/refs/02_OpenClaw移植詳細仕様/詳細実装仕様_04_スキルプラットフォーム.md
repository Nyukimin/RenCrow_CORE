**更新日**: 2026-03-26

## 1. 概要

RenCrow はスキルのテンプレートを持つが、動的ロードの仕組みがない。
OpenClaw は SKILL.md フォーマット、優先順位付きロード、ClawHub 公開レジストリを備えた完全なスキルプラットフォームを持つ。

**現状の差**:

| 項目 | OpenClaw | RenCrow |
|------|:--------:|:-------:|
| SKILL.md フォーマット（YAML frontmatter） | ✓ | △（テンプレートあり、動的ロードなし） |
| スキルロード優先順位（workspace > user > bundled） | ✓ | ✗ |
| ロード時ゲーティング（requires.bins / env / config） | ✓ | ✗ |
| スラッシュコマンドディスパッチ | ✓ | △（固定コマンドのみ） |
| ClawHub レジストリ（公開スキルストア） | ✓ | ✗ |
| macOS スキル UI（インストールフロー） | ✓ | ✗ |
| OS フィルタリング | ✓ | ✗ |

## 2. OpenClaw のスキル設計

### 2.1 SKILL.md フォーマット

```markdown
---
name: daily-report
description: 毎日のサマリーレポートを生成して送信する
version: 1.0.0
author: nyukimin
requires:
  bins: [curl, jq]
  env: [REPORT_API_KEY]
  config: [slack.enabled]
os: [linux, macos]
commands:
  - /report
  - /daily
---

## 使い方
`/report` と送ると、昨日の活動をまとめたレポートを送信します。

## プロンプトテンプレート
昨日（${DATE}）の活動をまとめてください。...
```

### 2.2 ロード優先順位

```
workspace スキル  : ./skills/*.md              （最優先）
user スキル       : ~/.picoclaw/skills/*.md    （次優先）
bundled スキル    : 組み込みスキル             （フォールバック）
```

同名スキルがある場合、workspace > user > bundled の順で上書きされる。

### 2.3 ゲーティング（requires チェック）

スキルロード時に依存関係を検証し、満たされない場合はそのスキルを無効化（エラーにはしない）:

```yaml
requires:
  bins: [ffmpeg]      # which ffmpeg が通らなければ無効化
  env: [OPENAI_KEY]   # 環境変数が未設定なら無効化
  config: [discord.enabled]  # config が false なら無効化
  os: [linux]         # OS が一致しなければ無効化
```

### 2.4 ClawHub（公開スキルストア）

```
openclaw hub search <keyword>
openclaw hub install <skill-id>
openclaw hub publish <skill-path>
```

スキルは SKILL.md 形式で ClawHub に公開・配布される。

### 2.5 スラッシュコマンドディスパッチ

ロードされたスキルの `commands:` に定義されたコマンドがチャットコマンドとして有効になる:

```
/report    → daily-report スキルの実行
/daily     → 同上（エイリアス）
```

## 3. RenCrow の現状

### 3.1 スキルテンプレート（実装済み）

`templates/tool/` にスキル雛形がある:
- `SKILL.md.tmpl` — フロントマター付き Markdown テンプレート
- `main.go.tmpl` — Go 実装テンプレート
- `test_sample.json` — テスト入力サンプル

これらは**手動で参照して作る**ためのテンプレートであり、動的ロードはしていない。

### 3.2 チャットコマンド（実装済み）

```go
// 固定コマンドとして実装済み
/code, /code1, /code2, /code3
/ops
/local, /cloud
/status, /new, /reset, /compact, /context, /stop
```

ユーザー定義スキルからコマンドを動的追加する仕組みはない。

## 4. 実装方針

### Priority 1: SKILL.md 動的ロード + ゲーティング

**実装場所**: `internal/application/skill/`

```go
// SkillManifest は SKILL.md の YAML frontmatter を表す
type SkillManifest struct {
    Name        string   `yaml:"name"`
    Description string   `yaml:"description"`
    Version     string   `yaml:"version"`
    Author      string   `yaml:"author"`
    Requires    struct {
        Bins   []string `yaml:"bins"`
        Env    []string `yaml:"env"`
        Config []string `yaml:"config"`
        OS     []string `yaml:"os"`
    } `yaml:"requires"`
    Commands []string `yaml:"commands"`
}

type SkillLoader interface {
    Load(searchPaths []string) ([]Skill, error)
    Validate(manifest SkillManifest) error  // ゲーティング
}
```

検索パス（優先順位順）:
```go
searchPaths := []string{
    "./skills/",                      // workspace
    os.ExpandEnv("$HOME/.picoclaw/skills/"),  // user
    // bundled は Go embed で組み込み
}
```

### Priority 2: スラッシュコマンドディスパッチ拡張

現行の固定コマンドディスパッチをスキルロードと連動させる:

```go
type CommandDispatcher struct {
    builtinCommands map[string]CommandHandler  // 既存固定コマンド
    skillCommands   map[string]*Skill          // 動的ロードされたスキル
}

func (d *CommandDispatcher) Dispatch(cmd string, args string, ctx MessageContext) (string, error) {
    if handler, ok := d.builtinCommands[cmd]; ok {
        return handler(args, ctx)
    }
    if skill, ok := d.skillCommands[cmd]; ok {
        return d.runSkill(skill, args, ctx)
    }
    return "", ErrUnknownCommand
}
```

### Priority 3: ToolRegistry との統合（CapabilityAdaptation v1.0 連携）

スキルが提供するツールは ToolRegistry（DuckDB）に登録する:

```go
// スキルロード時にツールを ToolRegistry へ自動登録
func (sl *SkillLoader) registerTools(skill Skill, registry ToolRegistry) error {
    for _, tool := range skill.Tools {
        tool.Source = "skill"
        tool.TrustLevel = "user"  // shiro-generated と同等の信頼レベル
        return registry.Register(tool)
    }
}
```

`CapabilityAdaptation v1.0`（`docs/01_正本仕様/07_機能拡張.md` §3）の ToolRegistry インターフェースを使う。

### Priority 4: ClawHub 連携（保留）

外部ネットワーク依存・認証フローが必要なため、現時点では優先度低。
`internal/adapter/outbound/clawhub/` として将来実装する。

## 5. 既存機能との整合

- SKILL.md の `commands:` で定義されたコマンドは、既存の固定コマンド（`/code`, `/ops` 等）と共存する
- 固定コマンドが優先（スキルによる上書き不可）
- スキル実行は `MessageOrchestrator.HandleMessage()` を経由し、既存ルーティング・セーフガードが適用される
- `requires.bins` チェックは起動時のヘルスチェックと同時に実行する

---

## 6. 既知の制約

- RenCrow はメモリ <10MB 制約があるため、スキルロードは必要最小限のスキルのみとし、全スキルを常駐させない
- `.lobster` ワークフローとスキルは別概念（スキル = コマンド定義、ワークフロー = 実行シーケンス）
- ClawHub は RenCrow 専用の独自スキルストアとして将来構築するか、OpenClaw の ClawHub との互換性を持たせるか、優先度が上がった時点で決定する
