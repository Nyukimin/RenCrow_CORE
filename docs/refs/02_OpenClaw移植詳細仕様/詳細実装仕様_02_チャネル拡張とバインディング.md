## 1. 概要

RenCrow は現在 LINE / Slack / CLI の 3 チャネルを持つ。
OpenClaw は 20+ チャネルを統一モデルで管理し、さらに**バインディングエンジン**によってチャネル/ユーザー/グループ単位で異なるエージェントへのルーティングを決定論的に制御する。

**現状の差**:

| 項目 | OpenClaw | RenCrow |
|------|---------|---------|
| チャネル数 | 20+ | 3（LINE/Slack/CLI）|
| Telegram | ✓ | ✗ |
| Discord | ✓ | ✗ |
| WhatsApp | ✓ | ✗ |
| Google Chat / Signal / iMessage | ✓ | ✗ |
| WebChat（内蔵） | ✓ | △（Viewer UI のみ）|
| バインディングエンジン | ✓ | ✗ |
| チャネル別モデル上書き | ✓ | ✗ |
| DM pairing（未知送信者制御） | ✓ | ✗（→ 06_セキュリティ）|

## 2. OpenClaw のバインディングエンジン

### 2.1 仕組み

バインディングは「どのメッセージをどのエージェントに渡すか」の決定論的ルール。

```yaml
bindings:
  - match:
      channel: "discord"
      guildId: "123456789"
      channelId: "987654321"
    agent: "coder-agent"

  - match:
      channel: "slack"
      teamId: "T012345"
    agent: "ops-agent"

  - match:
      channel: "line"
    agent: "default-agent"
```

### 2.2 優先順位（最も詳細なルールが優先）

```
peer（DM相手） > parentPeer > guildId+roles > guildId > teamId > accountId > channel-wide > fallback
```

### 2.3 チャネル別モデル上書き

```yaml
bindings:
  - match:
      channel: "discord"
      channelId: "dev-channel"
    model: "anthropic/claude-opus-4-6"
```

特定チャネルだけ高品質モデルを使う、など。

## 3. RenCrow の現状

### 3.1 現行チャネル実装

| チャネル | 実装 | 備考 |
|---------|------|------|
| LINE | ✓ | HMAC-SHA256 署名検証、CHAT ルート固定 |
| Slack | ✓ | Slack Events API |
| CLI | ✓ | ローカル実行用 |

チャネル拡張仕様（`04_実装仕様_機能拡張/実装仕様_チャネル拡張_v1.md`）に Discord/音声アダプターの設計がある。

### 3.2 ルーティング決定

現行は `MessageOrchestrator` が全チャネルの入力を受け、`MioAgent.DecideAction()` でルートを決定する。
チャネルごとにルーティングを変える仕組みはなく、LINE は常に CHAT ルート固定。

## 4. 実装方針

### Priority 1: Telegram

**理由**: 国内外で広く使われ、Bot API が安定している。

実装概要:
- `internal/adapter/inbound/telegram/handler.go`
- POST `/webhook/telegram`
- `X-Telegram-Bot-Api-Secret-Token` 検証
- `sendMessage` API で outbound

config.yaml:
```yaml
telegram:
  enabled: true
  bot_token: "${TELEGRAM_BOT_TOKEN}"
  webhook_secret: "${TELEGRAM_WEBHOOK_SECRET}"
```

### Priority 2: Discord

**理由**: 開発者コミュニティでの利用が多い。スレッドバウンドセッションの将来拡張にも対応しやすい。

実装概要:
- `internal/adapter/inbound/discord/handler.go`
- Interactions Endpoint URL 方式 + Ed25519 署名検証
- Slash command `/rencrow <message>` をエントリポイントに

### Priority 3: バインディングエンジン（簡易版）

OpenClaw の完全実装は複雑だが、RenCrow に必要な最小版:

```yaml
bindings:
  - match:
      channel: "slack"
      team_id: "T012345"
    route: "CODE3"      # このチャネルは Gin(Claude) 直指定

  - match:
      channel: "line"
    agent: "mio"        # デフォルト
```

実装:
- `BindingRule` 値オブジェクト（channel / team_id / channel_id / route / agent）
- `BindingEngine.Match(event)` → `RoutingOverride`
- `MessageOrchestrator` で DecideAction の前に適用

### Priority 4: WhatsApp / iMessage

**保留理由**: WhatsApp は Business API の取得が必要（費用・審査あり）。iMessage は Apple 制限が強い。
現時点では優先度低。

## 5. 共通チャネル抽象化

新チャネル追加を容易にするため、共通インターフェースを維持する:

```go
// 既存パターンの踏襲
type ChannelAdapter interface {
    Name() string
    Send(ctx context.Context, chatID, text string) error
    Probe(ctx context.Context) error
}
```

各チャネルは独自の `http.Handler` で Webhook を受け、`ChannelEvent` に正規化して `MessageOrchestrator` に渡す。

---

## 6. 既知の制約

- LINE は Webhook 応答時間制限（5秒）があるため CHAT ルート固定を維持する。この制約は Telegram / Discord にはない。
- Discord の Interactions は 3 秒以内に初期レスポンスが必要。重い処理は defer 応答パターンが必要。
- バインディングエンジンを導入すると LINE の `ForcedRoute: RouteCHAT` との整合性調整が必要になる。
