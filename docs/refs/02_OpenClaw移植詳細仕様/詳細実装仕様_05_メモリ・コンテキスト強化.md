## 2. OpenClaw のメモリ・コンテキスト設計

### 2.1 ハイブリッド BM25 + ベクトル検索

RenCrow が採用するベクトル検索はセマンティック類似度に強いが、固有名詞・コード・コマンド名の完全一致に弱い。
OpenClaw はキーワード検索（BM25）とベクトル検索を組み合わせ、両方のスコアを RRF（Reciprocal Rank Fusion）で統合する。

```
final_score(d) = α × bm25_score(d) + (1-α) × vector_score(d)
```

デフォルト: α = 0.4（BM25 40%, ベクトル 60%）

### 2.2 MMR ダイバーシティ再ランキング

Maximal Marginal Relevance（MMR）でトップ K 件の多様性を確保:

```
MMR(d) = λ × sim(d, query) - (1-λ) × max_{d_i ∈ S} sim(d, d_i)
```

S = すでに選択済みのドキュメント集合。同じ内容のメモリが重複して注入されるのを防ぐ。

### 2.3 時間減衰（Temporal decay）

古いメモリのスコアを低減:

```
decayed_score = score × exp(-λ × age_days)
```

デフォルト: λ = 0.01（100日で約37%に減衰）

### 2.4 コンテキスト圧縮前の自動メモリ書き込み

コンテキストがトークン上限に近づいたとき、会話の重要部分を自動的に長期メモリに書き込んでから圧縮する。

### 2.5 Markdown ファイルベースメモリ（MEMORY.md）

```markdown
---
name: user-preference-dark-mode
type: preference
created: 2026-03-26
---
ユーザーはダークモードを好む。UI の提案時には常にダークテーマを前提にする。
```

人間が直接読み書きできる形式。`.rencrow/memory/*.md` に保存。

### 2.6 ワークスペースブートストラップ

```
AGENTS.md   : エージェントへの恒常的な指示・記憶注入
TOOLS.md    : ツールの使い方ノート（ツール選択時に注入）
IDENTITY.md : エージェントの名前・アバター絵文字・口調
USER.md     : ユーザープロファイル（職業・好み・禁止事項）
BOOTSTRAP.md: 初回起動時に一度だけ実行するリチュアル（DBセットアップ等）
```

これらはワークスペースルートに置くと自動的に検出・注入される。

## 3. RenCrow の現状

### 3.1 実装済みメモリ基盤

| コンポーネント | 実装 | 詳細 |
|-------------|------|------|
| MemoryStore（Redis） | ✓ | 短期会話バッファ |
| DuckDB サマリ | ✓ | 中期圧縮サマリ |
| Qdrant gRPC:6334 | ✓ | 長期ベクトル検索 |
| RecallPack | ✓ | 検索結果をプロンプト注入 |
| ContextBuilder | ✓ | メモリ + KB を組み合わせた注入 |

### 3.2 不足している機能

- Qdrant の純粋ベクトル検索のみで、BM25 によるキーワード検索がない
- 検索結果に多様性制御がなく、同質なメモリが重複注入される可能性がある
- ユーザープロファイルはセッション内インメモリのみで、再起動後に消える
- コンテキスト圧縮（`/compact`）時に重要情報がロストするリスクがある

## 4. 実装方針

### Priority 1: ハイブリッド BM25 + ベクトル検索

**実装場所**: `internal/infrastructure/memory/hybrid_search.go`

Qdrant は v1.7 以降でスパース＋デンスのハイブリッド検索をネイティブサポートしている（SPLADE + ベクトルの RRF）。
Qdrant の `sparse_vectors` フィールドに BM25 スパースベクトルを保存し、検索時に RRF で統合する。

```go
type HybridSearcher struct {
    qdrantClient *qdrant.Client
    bm25Encoder  BM25Encoder
    alpha        float32  // BM25 の重み（デフォルト 0.4）
}

func (hs *HybridSearcher) Search(ctx context.Context, query string, topK int) ([]MemoryResult, error) {
    // 1. BM25 スパースベクトル生成
    sparseVec := hs.bm25Encoder.Encode(query)
    // 2. Qdrant ハイブリッド検索（RRF）
    return hs.qdrantClient.SearchHybrid(ctx, sparseVec, denseVec, topK)
}
```

### Priority 2: Markdown ファイルベースメモリ（MEMORY.md）

人間が直接編集できるメモリ形式。`~/.rencrow/memory/*.md` を起動時に読み込み、ベクトルDBへ upsert する。

```go
type MarkdownMemoryLoader struct {
    memDir   string  // ~/.rencrow/memory/
    qdrant   QdrantClient
}

func (ml *MarkdownMemoryLoader) Load(ctx context.Context) error {
    // *.md を walk して frontmatter パース → Qdrant upsert
}
```

起動時と `~/.rencrow/memory/` の変更検知（fsnotify）でリロード。

### Priority 3: コンテキスト圧縮前の自動メモリ書き込み

`/compact` コマンド実行時（または自動圧縮時）のフック:

```go
func (o *MessageOrchestrator) HandleCompact(ctx context.Context, session Session) error {
    // 1. 現在の会話から重要発言を抽出（LLM）
    summary := extractKeyPoints(session.Messages)
    // 2. 長期メモリに書き込み
    o.memoryStore.SaveLongTerm(ctx, summary)
    // 3. 圧縮実行
    return o.compressSession(session)
}
```

### Priority 4: ワークスペースブートストラップ（AGENTS.md 等）

**実装場所**: `internal/application/workspace/bootstrap.go`

```go
type WorkspaceBootstrap struct {
    workspaceDir string
}

// BootstrapFiles は起動時に検出・ロードするファイル名
var BootstrapFiles = []string{
    "AGENTS.md",   // エージェント指示
    "TOOLS.md",    // ツールノート
    "IDENTITY.md", // ペルソナ定義
    "USER.md",     // ユーザープロファイル
}

func (wb *WorkspaceBootstrap) Load() (*WorkspaceContext, error) {
    // 各ファイルを読み込み、プロンプト注入用データとして返す
}
```

`ContextBuilder` がこのデータを参照してシステムプロンプトに注入する。

### Priority 5: 時間減衰 + MMR（低優先）

Qdrant のスコアリング段階で適用可能。実装コストが高いため、Priority 1-4 完了後に検討する。

## 5. 既存機能との整合

- RecallPack は `ContextBuilder` → `MessageOrchestrator` の既存フローで動作しており、BM25 強化後もインターフェースは変わらない
- MEMORY.md の内容は Qdrant に upsert されるため、既存の RecallPack 検索でそのまま取得できる
- ユーザープロファイルの DB 永続化は USER.md または DuckDB `user_profiles` テーブルに保存する（どちらかで統一）

---

## 6. 既知の制約

- Qdrant ハイブリッド検索は Qdrant v1.7+ 必須。現行 Qdrant バージョンを確認してから実装する
- BM25 スパースベクトル生成は SPLADE モデル（Qdrant 推奨）または TF-IDF ベースの軽量実装から選択する。軽量 RenCrow の制約上、TF-IDF ベースを優先する
- BOOTSTRAP.md の「初回起動リチュアル」は冪等性を保証する必要がある（再起動のたびに実行されないよう、実行済みフラグを DuckDB に保存）
