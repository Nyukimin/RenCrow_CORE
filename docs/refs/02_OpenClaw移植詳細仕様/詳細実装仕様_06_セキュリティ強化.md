## 1. 概要

RenCrow は API キー管理・ログマスキング・保護ファイルパターン・ワークスペース制限を実装済みだが、OpenClaw が持つ送信者制御・サンドボックス・コマンド承認ゲートが欠けている。

**現状の差**:

| 項目 | OpenClaw | RenCrow |
|------|:--------:|:-------:|
| API キー管理（環境変数） | ✓ | ✓ |
| ログマスキング | ✓ | ✓ |
| 保護ファイルパターン | ✓ | ✓ |
| ワークスペース制限（パストラバーサル防止） | ✓ | ✓ |
| DM ペアリング（未知送信者にコード要求） | ✓ | ✗ |
| dmPolicy / groupPolicy（allowlist / open / disabled） | ✓ | ✗ |
| Docker ベースサンドボックス | ✓ | ✗ |
| `doctor` コマンド（設定検証・セキュリティ監査） | ✓ | ✗ |
| Exec allowlist / safe-bin ポリシー | ✓ | ✗ |
| プロンプトインジェクション警告 | ✓ | ✗ |
| CODE 以外クラウド禁止（/local フラグ） | ✗ | ✓（RenCrow 独自） |

## 2. OpenClaw のセキュリティ設計

### 2.1 DM ペアリング

LINE DM のように「誰でもメッセージを送れる」チャネルに対して、未知の送信者にはペアリングコードを要求する仕組み。

```yaml
security:
  dm_policy: "allowlist"    # allowlist / open / disabled
  allowlist:
    - "U1234567890"          # LINE userId
    - "U0987654321"
  pairing:
    enabled: true
    code_length: 6          # ランダム6桁
    expires_minutes: 10
```

**フロー**:
1. 未知のユーザーから DM を受信
2. ランダムなペアリングコードを生成して返答
3. ユーザーがコードを送り返すと allowlist に追加
4. 以降のメッセージを通常処理

### 2.2 グループポリシー

```yaml
security:
  group_policy: "mention"   # mention（メンション時のみ反応）/ always / disabled
```

### 2.3 Docker ベースサンドボックス

Worker のシェルコマンド実行を Docker コンテナ内に閉じ込める:

```yaml
sandbox:
  enabled: true
  image: "openclaw/sandbox:latest"
  network: "none"
  memory: "256m"
  cpu: "0.5"
  workspace_mount: "/workspace"
```

ホスト OS へのアクセスをブロックしつつ、プロジェクトディレクトリのみをマウント。

### 2.4 Exec allowlist / safe-bin ポリシー

```yaml
security:
  exec:
    policy: "allowlist"     # allowlist / ask / any
    allowed_bins:
      - git
      - npm
      - python3
      - go
    blocked_bins:
      - rm
      - curl
      - wget
```

### 2.5 プロンプトインジェクション警告

Web 検索結果・ファイル読み込み結果に「Ignore all previous instructions」等のパターンを検出した場合、LLM に渡す前にユーザーへ警告する。

### 2.6 `doctor` コマンド

```
picoclaw doctor
```

設定ファイル・環境変数・セキュリティ設定を検証し、問題があれば修正案を提示:

```
✓ API keys: all set
✓ Config file: valid YAML
✗ dm_policy: not set (default: open — anyone can message the bot)
  → Recommend setting security.dm_policy: allowlist
✗ exec.policy: not set (default: any — all commands allowed)
  → Recommend setting security.exec.policy: ask
```

## 3. RenCrow の現状

### 3.1 実装済みセキュリティ機能

| 機能 | 実装場所 | 詳細 |
|------|---------|------|
| API キー（環境変数） | `~/.picoclaw/.env` | chmod 600、EnvironmentFile= |
| ログマスキング | `internal/infrastructure/logger/` | API キー・トークンを `[MASKED]` に置換 |
| 保護ファイルパターン | `TOOL_CONTRACT.md` | `.env`, `.ssh/`, `*.key` 等 |
| パストラバーサル防止 | `internal/infrastructure/tool/` | `../` チェック、制御文字除去 |
| `${ENV_VAR}` 展開 | `internal/infrastructure/config/` | 環境変数注入、平文保存禁止 |
| /local, /cloud コマンド | `internal/application/idlechat/` | ルートごとのクラウド LLM 制限 |

### 3.2 不足している機能

- LINE は `allowlist` なしで誰でも送れる状態（HMAC-SHA256 で Bot からの正規送信は確認するが、送信者は制限しない）
- Worker のシェル実行は Goプロセスのパーミッション内で完全に動作し、コンテナ分離がない
- 実行コマンドの事前承認ゲートがない（セーフガードはあるが、実行可否の都度確認ではなくルール固定）

## 4. 実装方針

### Priority 1: dmPolicy / allowlist

**実装場所**: `internal/application/security/dm_policy.go`

```go
type DMPolicy string

const (
    DMPolicyAllowlist DMPolicy = "allowlist"
    DMPolicyOpen      DMPolicy = "open"
    DMPolicyDisabled  DMPolicy = "disabled"
)

type SecurityConfig struct {
    DMPolicy  DMPolicy `yaml:"dm_policy"`
    Allowlist []string `yaml:"allowlist"`  // userId リスト
}

type DMPolicyChecker interface {
    IsAllowed(ctx context.Context, userID string, channel string) (bool, error)
    Pair(ctx context.Context, userID string, code string) error
}
```

`MessageOrchestrator.HandleMessage()` の冒頭で `DMPolicyChecker.IsAllowed()` を呼ぶ。
拒否された場合、ペアリングコードを送信して処理を中断する。

**config.yaml 追加**:
```yaml
security:
  dm_policy: "allowlist"
  allowlist:
    - "${LINE_ALLOWED_USER_1}"
    - "${LINE_ALLOWED_USER_2}"
  pairing:
    enabled: true
    expires_minutes: 10
```

### Priority 2: Exec allowlist

```go
type ExecPolicy string

const (
    ExecPolicyAny       ExecPolicy = "any"
    ExecPolicyAllowlist ExecPolicy = "allowlist"
    ExecPolicyAsk       ExecPolicy = "ask"
)

type ExecPolicyChecker interface {
    Check(cmd string) (ExecDecision, error)
}

type ExecDecision struct {
    Allowed bool
    Reason  string
}
```

`internal/infrastructure/tool/tool_runner.go` の `ExecuteV2()` で実行前に確認。
`ask` モードはユーザーに承認を求め、返答まで実行を保留する（Lobster チェックポイントと同じ仕組みを使用）。

### Priority 3: `doctor` コマンド

```go
// picoclaw doctor として実行可能な検証ツール
type DoctorChecker struct {
    config     *Config
    secChecker SecurityChecker
}

func (d *DoctorChecker) Run() []DoctorResult {
    checks := []Check{
        d.checkAPIKeys(),
        d.checkDMPolicy(),
        d.checkExecPolicy(),
        d.checkProtectedFiles(),
        d.checkOllamaHealth(),
    }
    // 問題点と推奨設定を返す
}
```

CLI エントリポイント: `picoclaw doctor` または `/doctor` チャットコマンド。

### Priority 4: プロンプトインジェクション警告

```go
type InjectionDetector struct {
    patterns []*regexp.Regexp
}

var injectionPatterns = []string{
    `(?i)ignore\s+(all\s+)?previous\s+instructions?`,
    `(?i)disregard\s+.{0,20}instructions?`,
    `(?i)you\s+are\s+now\s+`,
    `(?i)system\s*:\s*`,
}

func (d *InjectionDetector) Detect(content string) bool
```

Web 検索結果・ファイル内容を LLM に渡す前にチェック。検出時はユーザーに警告して確認を取る。

### Priority 5: Docker サンドボックス（保留）

Docker 依存が発生し、軽量 RenCrow の設計思想に反する。
必要性が高まった時点（マルチユーザー対応など）で実装を検討する。

## 5. 既存機能との整合

- HMAC-SHA256 署名検証（LINE）はチャネル認証であり、`dmPolicy` のユーザー認証とは独立して動作する
- `/local, /cloud` コマンドは RenCrow 独自のクラウド LLM 制限であり、OpenClaw にはない。維持する
- 保護ファイルパターンは `TOOL_CONTRACT.md` で定義済み。Exec allowlist 実装後もここを参照する

---

## 6. 既知の制約

- `ask` モードの承認ゲートは非同期通信（LINE は push API、Slack は chat.postMessage）で実現するため、応答待ちの間はジョブをキューに保留する必要がある
- DM ペアリングのコードは DuckDB に保存し、TTL（10分）を自前で管理する
- Docker サンドボックスがない場合、Worker の実行はホスト OS の `$HOME/.picoclaw/workspace/` に制限することで代替する（パストラバーサル防止は既実装）
