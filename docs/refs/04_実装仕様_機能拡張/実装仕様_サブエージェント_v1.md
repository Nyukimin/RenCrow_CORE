# 実装仕様: サブエージェントシステム v1.0

**更新日**: 2026-03-19
**ステータス**: 実装済み
**ベース**: Worker の tool calling 実行ループ

---

## 1. 概要

サブエージェントシステムは、Worker 役の `ShiroAgent` が 1 回の LLM 応答で終わらず、tool calling を使って複数回の推論とツール実行を回すための ReAct ループ実装である。

現行実装の責務は次の 3 つである。

1. `ToolCallingProvider.Chat()` を使った反復推論
2. `ToolRunner.ExecuteV2()` を使った tool call 実行
3. `ShiroAgent.Execute()` からの同期呼び出し

名称は「サブエージェント」だが、現行実装は複数 agent の並列管理よりも、Worker が内部でツールを自律選択する loop 実装が中心である。

---

## 2. 公開インターフェース

### 2.1 Domain I/F

**ファイル**: `internal/domain/agent/subagent.go`

```go
type SubagentTask struct {
    AgentName    string
    Instruction  string
    SystemPrompt string
}

type SubagentResult struct {
    AgentName  string // 実行したサブエージェント名
    Output     string // 最終出力テキスト
    Iterations int    // ReActループの反復回数
}

type SubagentManager interface {
    RunSync(ctx context.Context, task SubagentTask) (SubagentResult, error)
}
```

### 2.2 Tool calling I/F

**ファイル**: `internal/domain/llm/provider.go`

現行実装では既存 `LLMProvider` に加えて、以下を持つ `ToolCallingProvider` を使う。

```go
type ToolCallingProvider interface {
    LLMProvider
    Chat(ctx context.Context, req ChatRequest) (ChatResponse, error)
}
```

関連型:

- `ChatMessage`
- `ToolCall`
- `ToolDefinition`
- `ChatRequest`
- `ChatResponse`

## 3. 現行実装

### 3.1 Manager

**ファイル**: `internal/application/subagent/manager.go`

`Manager` は以下を保持する。

- `provider llm.ToolCallingProvider`
- `toolRunner tool.RunnerV2`
- `toolDefs []llm.ToolDefinition`
- `loopConfig toolloop.Config`

`RunSync()` の処理:

1. `Instruction` 空なら error
2. `SystemPrompt` 未指定なら既定 prompt を採用
3. `system` + `user` の 2 メッセージを組む
4. `toolloop.Run(...)` を同期実行
5. 最終テキストを `SubagentResult.Output` に入れて返す

### 3.2 ToolLoop

**ファイル**: `internal/application/toolloop/loop.go`

`toolloop.Run()` は ReAct ループを実装する。

反復ごとの流れ:

1. `provider.Chat(messages, tools)` を呼ぶ
2. assistant メッセージを履歴へ追加
3. `tool_calls` がなければその `Content` を返して終了
4. `tool_calls` があれば各 call を `toolRunner.ExecuteV2()` で実行
5. 実行結果を `role="tool"` メッセージとして履歴へ追加
6. 次 iteration へ進む

終了条件:

- final assistant response を得たとき
- `ctx.Done()`
- `MaxIterations` 到達

`MaxIterations` 到達時は、最後の assistant message を返す。

### 3.3 Shiro 統合

**ファイル**: `internal/domain/agent/shiro.go`

`ShiroAgent.Execute()` は `subagentManager != nil` のとき、通常の単発 `Generate()` ではなく `RunSync()` を呼ぶ。

```go
result, err := s.subagentManager.RunSync(ctx, SubagentTask{
    AgentName:    "shiro",
    Instruction:  t.UserMessage(),
    SystemPrompt: s.systemPrompt,
})
```

`subagentManager == nil` の場合のみ、従来の単発 `llmProvider.Generate()` にフォールバックする。

## 4. 起動時配線

**ファイル**: `cmd/picoclaw/main.go`

`cfg.Subagent.Enabled` が true のとき:

1. `resolveSubagentProvider(cfg, workerProvider)` で provider を選ぶ
2. `workerToolRunnerV2.ToolDefinitions()` で tool 定義一覧を取得
3. `subagentapp.NewManager(...)` を構築
4. `workerToolRunnerV2.RegisterSubagent("worker", ...)` で subagent tool を登録
5. `ShiroAgent` へ `subagentMgr` を注入

設定:

```go
type SubagentConfig struct {
    Enabled       bool
    MaxIterations int
    Provider      string
    Model         string
}
```

デフォルト:

- `MaxIterations = 10`
- `Provider` 空時は worker provider を基準に解決

利用可能 provider:

- `ollama`
- `claude`
- `openai`
- `deepseek`

## 5. Tool 定義と再帰防止

**ファイル**: `internal/infrastructure/tools/runner.go`

`ToolRunner.ToolDefinitions()` は LLM に渡す tool list を返すが、`subagent` 自体は除外する。

理由:

- subagent が subagent を再帰的に呼ぶループを避けるため

一方で実行系としては `RegisterSubagent()` により `subagent` tool 自体は登録されるため、外部から manager を経由して同期実行することはできる。

## 6. 実装到達点

2026-03-19 時点で確認できる到達点:

- `ToolCallingProvider` は domain に導入済み
- `ollama`, `openai`, `claude`, `deepseek` の各 provider に `Chat()` 実装がある
- `Manager.RunSync()` と `toolloop.Run()` は実装済み
- `ShiroAgent.Execute()` は subagent 有効時に ReAct loop を使う
- `ToolRunner` は tool definitions と `subagent` 登録に対応済み

---

## 7. 既知の制約

1. 現行の中心は Worker (`shiro`) の loop 実装であり、複数サブエージェントの高度なスケジューリングはない。
2. `RunSync()` は同期実行のみで、queue や long-running lifecycle 管理は持たない。
3. system prompt の既定値は簡素で、agent ごとの専用最適化は限定的である。
4. `MaxIterations` 超過時は error にせず最後の assistant message を返す。
5. Viewer 側に subagent 専用の可視化タブは現状ない。

---

## 8. 確認観点

- `cfg.Subagent.Enabled=true` で起動ログに `Subagent enabled` が出る
- `ShiroAgent.Execute()` が `RunSync()` 経由で動く
- tool call を返す provider で loop が複数 iteration 回る
- `MaxIterations` 到達時も panic せず最後の assistant response を返す
- `subagent` tool が tool definition 一覧には混ざらない

以上をもって、サブエージェント仕様は「実装開始」ではなく、Worker の ReAct ループとして実装済みの現行機能である。
