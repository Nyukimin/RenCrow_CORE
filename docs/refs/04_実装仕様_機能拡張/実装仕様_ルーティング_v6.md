# 実装仕様: ルーティングシステム v6.2

**作成日**: 2026-03-19
**更新日**: 2026-03-27 (CODE4 ルート追加)
**ステータス**: 現行実装の明文化 + 将来改善方針

## 1. 概要

RenCrow のルーティングは、ユーザーメッセージを `CHAT / PLAN / ANALYZE / OPS / RESEARCH / CODE / CODE1 / CODE2 / CODE3 / CODE4` のいずれかに振り分ける。

現行実装の基本方針:

- レイテンシ優先で `RuleDictionary` を使う
- 明示コマンドを最優先する
- 判定できなければ安全側で `CHAT` に倒す
- ただし、ファイル編集依頼は `CHAT` に誤分類しないよう `CODE` を優先する

## 2. ルーティングカテゴリ

| Route | 用途 | 現行の主処理先 |
|---|---|---|
| `CHAT` | 会話・通常応答 | Mio |
| `PLAN` | 計画整理 | Mio |
| `ANALYZE` | 分析・診断 | Mio |
| `OPS` | 実行・起動・再起動などの運用操作 | Shiro |
| `RESEARCH` | 深い調査 | Mio |
| `CODE` | 汎用コーディング | Mio -> Shiro -> Coder (fallback: coder1→2→3→4) |
| `CODE1` | Coder1 明示指定 (仕様設計) | Mio -> Shiro -> Coder1 |
| `CODE2` | Coder2 明示指定 (実装) | Mio -> Shiro -> Coder2 |
| `CODE3` | Coder3 明示指定 (レビュー・推論) / Chrome・ブラウザ操作 | Mio -> Shiro -> Coder3 |
| `CODE4` | Coder4 明示指定 (高速プロトタイピング・実験) | Mio -> Shiro -> Coder4 |

定義は [route.go](/home/nyukimi/picoclaw_multiLLM/internal/domain/routing/route.go) にある。

## 3. ルーティング決定フロー

現行の決定順序は以下。

1. `HandleChatCommand()`
2. `parseExplicitCommand()`
3. `RuleDictionary.Match()`
4. `CHAT` fallback

`HandleChatCommand()` は [message_orchestrator.go](/home/nyukimi/picoclaw_multiLLM/internal/application/orchestrator/message_orchestrator.go) 側で先に処理される。  
`parseExplicitCommand()` と `RuleDictionary.Match()` は [mio.go](/home/nyukimi/picoclaw_multiLLM/internal/domain/agent/mio.go) の `DecideAction()` で使われる。

### 3.1 チャットコマンド

ルーティング前に処理するチャットコマンド:

- `/status`
- `/stop`
- `/compact`
- `/context`
- `/new`

これらは `Handled=true` なら route 決定に進まない。

### 3.2 明示ルーティングコマンド

明示コマンドは `confidence=1.0` で固定する。

- `/code` - 汎用コーディング（coder1→2→3→4 の順でフォールバック）
- `/code1` - Coder1 明示指定（仕様設計特化）
- `/code2` - Coder2 明示指定（実装特化）
- `/code3` - Coder3 明示指定（レビュー・推論特化）
- `/code4` - Coder4 明示指定（高速プロトタイピング・実験）
- `/plan` - 計画整理
- `/chat` - 会話・通常応答
- `/ops` - 運用操作
- `/analyze` - 分析・診断
- `/research` - 深い調査

### 3.3 RuleDictionary

`RuleDictionary.Match()` は `confidence=0.85` を基本にした静的判定である。現行では少なくとも以下を扱う。

| Route | 主な判定語 |
|---|---|
| `CODE` | `実装して`, `修正して`, `変更を入れて`, `追記して`, `編集して`, `更新してください`, `ファイルを更新`, `ファイルを変更`, `json ファイル`, `text フィールドを` |
| `PLAN` | `計画`, `プラン`, `設計して`, `アーキテクチャ` |
| `ANALYZE` | `分析して`, `調査して`, `解析して`, `診断して`, `レビューして` |
| `OPS` | `実行して`, `起動して`, `デプロイ`, `ビルドして`, `停止して`, `再起動` |
| `RESEARCH` | `リサーチ`, `情報を集`, `ドキュメントを探` |
| `CODE3` | `chrome`, `ブラウザ`, `画面操作`, `スクレイピング`, `ページを開`, `webを操作` |

注意:

- `調べて`, `検索して` は `RESEARCH` に含めない
- これらは現行では `CHAT` 側の即答や Web 検索に寄せる前提

### 3.4 ファイル編集依頼の `CODE` 優先

現行実装では `isCodeEditRequest()` が追加されており、単なるキーワード辞書に加えて以下を見ている。

- ファイル参照がある
  - 例: `README.md`, `viewer.html`, `foo.json`, `bar.go`
- かつ編集動詞がある
  - 例: `変更`, `修正`, `追記`, `編集`, `更新`, `実装`, `追加`, `削除`, `書き換`, `直して`

この条件を満たす場合は `confidence=0.9` で `CODE` に寄せる。

対象拡張子:

- `.go`
- `.py`
- `.js`
- `.ts`
- `.tsx`
- `.jsx`
- `.json`
- `.yaml`
- `.yml`
- `.md`
- `.html`
- `.css`
- `.sh`
- `.txt`

この仕様により、`README.md に1行だけ変更を入れて。具体的には末尾に確認用コメントを1行追記してください。` のような依頼は `CHAT` ではなく `CODE` と判定する。

### 3.5 フォールバック

明示コマンドにも辞書にも当たらない場合は `CHAT` に倒す。

- route: `CHAT`
- confidence: `0.7`
- reason: `No rule match, default to CHAT`

## 4. 実行ディスパッチ

ルーティング後の実行は、旧来の「CODE は coder に直送」ではなく、現行の分散実行に従う。

### 4.1 DistributedOrchestrator の現行動作

[distributed_orchestrator.go](/home/nyukimi/picoclaw_multiLLM/internal/application/orchestrator/distributed_orchestrator.go) の現行動作:

- `CHAT`
  - `mio` が直接処理
- `OPS`
  - `mio -> shiro`
- `CODE / CODE1 / CODE2 / CODE3`
  - `mio -> shiro -> coder`
  - 完了後は `shiro -> mio -> user`

`routeNoticeText()` のユーザー向け文面もこの前提で作られている。

### 4.2 Coder 選択

現行の `CODE` は単純固定ではなく、`routeToCoderForMessage()` を通る。

- `CODE1` は `coder1`
- `CODE2` は `coder2`
- `CODE3` は `coder3`
- `CODE` は接続状態や capability に応じて `coder1-3` から選択

ただしルーティング仕様の主責務は route 決定であり、coder 選択の詳細アルゴリズム自体は分散実行仕様側の責務とする。

## 5. 現行の判定例

### 5.1 `CHAT`

- `今日は寒いね`
- `この技術について調べて`

後者は現行仕様では `RESEARCH` ではなく `CHAT` fallback に寄る。

### 5.2 `CODE`

- `このコードを修正して`
- `JSON ファイルの text フィールドを更新してください`
- `README.md に1行だけ変更を入れて。具体的には末尾に確認用コメントを1行追記してください。`

### 5.3 `CODE3`

- `Chrome を操作してログインして`
- `ブラウザでページを開いて確認して`

## 6. 理由文字列と confidence

現行の `routing.Decision` は少なくとも以下を持つ。

- `route`
- `confidence`
- `reason`

理由文字列は現行では以下に固定される。

- `Explicit command`
- `Rule dictionary match`
- `No rule match, default to CHAT`

confidence の現行値:

- 明示コマンド: `1.0`
- `isCodeEditRequest()`: `0.9`
- RuleDictionary 通常マッチ: `0.85`
- `CHAT` fallback: `0.7`

## 7. 既知の制約

- 静的キーワード方式なので、意図が強くても語彙がずれると `CHAT` に落ちうる
- `RESEARCH` は intentionally 狭く、`調べて` 程度では乗らない
- ファイル参照 + 編集動詞は `CODE` に寄るが、「読むだけ」「説明して」は `CODE` にしない
- route 判定理由は文字列のみで、構造化された evidence はまだ返していない

## 8. 将来改善方針

将来は以下の stronger routing を入れる余地がある。

- file path, patch, command, stacktrace, JSON bulk text などの evidence ベース判定
- `reason` だけでなく `evidence[]` を返す構造化 route 判定
- `CODE` と `OPS` の境界を「編集要求」と「実行要求」でより厳密に分ける
- Viewer で route 判定根拠を直接見えるようにする

ただし現時点の正本は、あくまで現在の `parseExplicitCommand + RuleDictionary + isCodeEditRequest + CHAT fallback` の実装である。

## 9. 検証観点

現行仕様として最低限通るべき観点:

- `/code3` は `CODE3`
- `このコードを修正して` は `CODE`
- `README.md に1行だけ変更を入れて...` は `CODE`
- `Chrome を操作して` は `CODE3`
- `この技術について調べて` は `CHAT`
- 通常会話は `CHAT`
