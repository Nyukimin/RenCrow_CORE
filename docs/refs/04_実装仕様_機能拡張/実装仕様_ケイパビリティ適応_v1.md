## 0. 概要と目的

### 0.1 解決する問題

現行実装（v4.0）では、エージェントの LLM・ツール構成が **config.yaml に静的に記述**されている。

```yaml
# 現行: 固定配線
providers:
  claude:
    model: claude-sonnet-4-6
  ollama:
    model: gemma3:4b
```

これにより以下の問題が生じる：

| 問題 | 影響 |
|---|---|
| 低スペック環境で大モデルを指定するとOOM | デプロイ失敗 |
| API キーのないノードに Coder3 を割り当てる | 実行エラー |
| Shiro が作ったツールを Coder は知らない | ツールの孤立 |
| Windows と Linux でツールパスが違う | 手動設定が必要 |

### 0.2 解決方針

**単一ソース・ケイパビリティ適応型**:

- ソースコードは1つ。ビルドターゲット（OS/アーキテクチャ）だけを変える
- 各ノードが**起動時に自分のケイパビリティを検出**し、それに合わせて動作を決定する
- ツールは **ToolRegistry** に永続化し、対応プラットフォームのノードが共有する

### 0.3 v4.0 との関係

```
v4.0: 複数ノード間でメッセージを送れるようになった
           ↓ この仕様
v4.1: 各ノードが「自分に何ができるか」を宣言し、タスクが適切なノードに流れる
```

v4.0 の Transport 抽象化（LocalTransport / SSHTransport）を前提とする。

## 1. NodeCapabilities — 起動時ケイパビリティ検出

### 1.1 データ構造

```go
// NodeCapabilities は起動時に検出・構築される、このノードが提供できる能力の全体像
type NodeCapabilities struct {
    NodeID   string           // ノード識別子（hostname + プロセスID）
    Platform PlatformInfo     // OS・アーキテクチャ
    Memory   MemoryInfo       // 利用可能メモリ
    LLMs     []LLMCapability  // 利用可能な LLM 一覧
    Tools    []ToolCapability // 利用可能なツール一覧（ToolRegistry から取得）
}

type PlatformInfo struct {
    OS   string // "linux" / "windows" / "darwin"
    Arch string // "amd64" / "arm64"
}

type MemoryInfo struct {
    TotalMB     uint64
    AvailableMB uint64
}

type LLMCapability struct {
    ProviderName string  // "ollama" / "claude" / "openai" / "deepseek"
    ModelName    string  // 例: "gemma3:4b"
    MaxContext   int     // コンテキスト長（トークン数）
    MaxMemoryMB  uint64  // このモデルの推定メモリ使用量
    Available    bool    // 実際に疎通確認できたか
    Quality      int     // 品質ランク（1=低 〜 5=高）。ルーティング優先順位に使用
}

type ToolCapability struct {
    Name      string   // ツール名
    Platforms []string // 対応 OS（["linux", "windows"] 等）
    Source    string   // "builtin" / "shiro-generated"
}
```

### 1.2 検出フロー

起動時に `CapabilityDetector.Detect()` を実行する:

```
起動
├── 1. PlatformInfo: runtime.GOOS / runtime.GOARCH
├── 2. MemoryInfo: /proc/meminfo (Linux) / GlobalMemoryStatusEx (Windows)
├── 3. LLMs: config.yaml に記載された各プロバイダーに疎通確認
│   ├── Ollama: GET /api/tags → モデル一覧・サイズ取得
│   ├── Claude: APIキー環境変数の有無 + テスト呼び出し（省略可）
│   ├── OpenAI: 同上
│   └── DeepSeek: 同上
├── 4. Tools: ToolRegistry から自ノードのプラットフォームに対応するツールを取得
└── → NodeCapabilities を構築してメモリ保持
```

### 1.3 動作プロファイルの自動決定

検出結果に基づき、ノードの動作プロファイルを決定する:

| 条件 | プロファイル | 挙動 |
|---|---|---|
| Claude API 疎通OK | `coder-high` | Coder3 として高品質推論 |
| OpenAI / DeepSeek 疎通OK | `coder-mid` | Coder2 / Coder1 として実装 |
| Ollama のみ（メモリ 8GB+） | `coder-local` | ローカル Coder として動作 |
| Ollama のみ（メモリ 4GB未満） | `worker-only` | Shiro（実行）のみ。Coder 不可 |
| API なし・Ollama なし | `unavailable` | 起動拒否 |

## 2. ケイパビリティベース LLM ルーティング

### 2.1 現行と新方式の比較

**現行（静的）**:
```
CODE3 → 必ず Claude API を使う（ノードに API キーがなければエラー）
```

**新方式（動的）**:
```
CODE3 リクエスト
  → 接続済みノードの LLMCapability 一覧を確認
  → Quality ランクが最も高いノードを選択
  → そのノードで利用可能な LLM を使用
```

### 2.2 LLM 選択アルゴリズム

```go
// SelectLLM は要求品質（minQuality）を満たす最適な LLM を選択する
func SelectLLM(nodes []NodeCapabilities, route Route) (NodeID string, provider LLMCapability, err error) {
    minQuality := qualityRequirement(route) // route ごとの最低品質要件

    candidates := []candidate{}
    for _, node := range nodes {
        for _, llm := range node.LLMs {
            if llm.Available && llm.Quality >= minQuality {
                candidates = append(candidates, candidate{node.NodeID, llm})
            }
        }
    }

    if len(candidates) == 0 {
        // 縮退: minQuality を下げて再試行
        return SelectLLMDegraded(nodes, route)
    }

    // 最高品質のものを選択（同品質なら利用可能メモリが多いノードを優先）
    sort.Slice(candidates, func(i, j int) bool {
        return candidates[i].llm.Quality > candidates[j].llm.Quality
    })
    return candidates[0].nodeID, candidates[0].llm, nil
}
```

**route ごとの最低品質要件**:

| Route | minQuality | 理由 |
|---|---|---|
| `CODE3` | 5 | 高品質推論が必要 |
| `CODE2` | 4 | 実装品質が必要 |
| `CODE1` | 3 | 仕様策定。ある程度の品質が必要 |
| `OPS` | 2 | ツール実行主体。LLM 品質より実行能力が重要 |
| `CHAT` | 2 | 会話。軽量でよい |

### 2.3 縮退動作

要求品質を満たすノードがない場合の縮退ルール:

```
CODE3 要求 → CODE3 不可 → CODE2 相当で代替 → ユーザーに通知
CODE2 要求 → CODE2 不可 → CODE1 相当で代替 → ユーザーに通知
全 Coder 不可 → Mio が「Coder が利用できません」と返す
```

## 3. ToolRegistry — ツール永続化と共有

### 3.1 概要

ToolRegistry は、組み込みツールと Shiro が生成したツールの両方を永続管理する。各ノードは起動時に ToolRegistry を参照し、自分のプラットフォームで使えるツールだけをロードする。

### 3.2 データ構造

```go
// ToolEntry はレジストリに保存されるツール定義
type ToolEntry struct {
    Name        string              // ツール名（ユニーク）
    Description string              // LLM に渡す説明文
    Schema      llm.ToolDefinition  // パラメータスキーマ
    Platforms   []string            // 対応 OS: ["linux"], ["windows"], ["linux", "windows"]
    Source      ToolSource          // builtin / shiro-generated
    CreatedAt   time.Time
    CreatedBy   string              // "shiro" / "builtin"
}

type ToolSource string

const (
    ToolSourceBuiltin         ToolSource = "builtin"
    ToolSourceShiroGenerated  ToolSource = "shiro-generated"
)
```

### 3.3 永続化

既存の DuckDB インフラ（`internal/infrastructure/persistence/`）を拡張して使用する。

```sql
CREATE TABLE tool_registry (
    name         TEXT PRIMARY KEY,
    description  TEXT NOT NULL,
    schema_json  TEXT NOT NULL,         -- llm.ToolDefinition の JSON
    platforms    TEXT NOT NULL,         -- JSON配列: ["linux", "windows"]
    source       TEXT NOT NULL,         -- "builtin" / "shiro-generated"
    created_at   TIMESTAMP NOT NULL,
    created_by   TEXT NOT NULL
);
```

### 3.4 インターフェース

```go
type ToolRegistry interface {
    // Register はツールを登録する（Shiro が新ツールを作成したときに呼ぶ）
    Register(ctx context.Context, entry ToolEntry) error

    // ListForPlatform は指定 OS で使用可能なツールを返す
    ListForPlatform(ctx context.Context, platform string) ([]ToolEntry, error)

    // Get は名前でツールを取得する
    Get(ctx context.Context, name string) (ToolEntry, error)

    // Close はデータベース接続を閉じる
    Close() error
}
```

## 4. Shiro → Coder ツール共有フロー

### 4.1 全体フロー

```
[Shiro ノード]
  タスク実行中に workspace/tools/<name>.sh を作成
    ↓
  register_tool メタツールで ToolRegistry.Register(entry) を呼ぶ
    ↓
[Coder ノード]
  次回の SubagentManager 初期化時に ToolRegistry.ListForPlatform("linux") を呼ぶ
  → 登録済みツールが ToolDefinition に追加される
  → ReActLoop でそのツールを呼び出せるようになる
```

### 4.2 プラットフォームフィルタリング

```go
// ノード起動時の ToolDefinition 構築
func buildToolDefs(registry ToolRegistry, platform string) ([]llm.ToolDefinition, error) {
    entries, err := registry.ListForPlatform(platform)
    if err != nil {
        return nil, err
    }

    defs := make([]llm.ToolDefinition, 0, len(entries))
    for _, entry := range entries {
        defs = append(defs, entry.Schema)
    }
    return defs, nil
}
```

### 4.3 ツール共有の制約

| 制約 | 理由 |
|---|---|
| 未承認ツールは ToolDefinition に含めない | セキュリティ（Shiro 生成コードの無制限実行を防ぐ） |
| `BinaryPaths` に対応 OS のパスがなければ除外 | 存在しないバイナリを LLM に提示しない |
| `subagent` ツール自体はレジストリに登録しない | 再帰呼び出し防止（既存ルールと同じ） |

## 5. クロスプラットフォームバイナリ戦略

### 5.1 ビルドターゲット

```makefile
# 全ターゲットをまとめてビルド
build-all:
    GOOS=linux   GOARCH=amd64  go build -o dist/picoclaw-linux-amd64    ./cmd/picoclaw
    GOOS=linux   GOARCH=arm64  go build -o dist/picoclaw-linux-arm64    ./cmd/picoclaw
    GOOS=windows GOARCH=amd64  go build -o dist/picoclaw-windows-amd64.exe ./cmd/picoclaw
    GOOS=darwin  GOARCH=arm64  go build -o dist/picoclaw-darwin-arm64   ./cmd/picoclaw

# agent（Coder/Worker 専用スタンドアロン）も同様
build-agent-all:
    GOOS=linux   GOARCH=amd64  go build -o dist/picoclaw-agent-linux-amd64    ./cmd/picoclaw-agent
    GOOS=windows GOARCH=amd64  go build -o dist/picoclaw-agent-windows-amd64.exe ./cmd/picoclaw-agent
```

### 5.2 OS 固有処理の分離方針

プラットフォーム固有コードはビルドタグで分離する:

```go
// capability_linux.go
//go:build linux

func readAvailableMemory() uint64 {
    // /proc/meminfo を読む
}

// capability_windows.go
//go:build windows

func readAvailableMemory() uint64 {
    // GlobalMemoryStatusEx を呼ぶ
}
```

ビジネスロジック（ルーティング・ToolRegistry・LLM 選択）はプラットフォーム非依存のまま維持する。

### 5.3 設定ファイルのパス解決

```go
// ExpandPlatformPath は OS に応じてパスを解決する
func ExpandPlatformPath(path string) string {
    path = os.Expand(path, os.Getenv) // ${ENV_VAR} 展開
    if runtime.GOOS == "windows" {
        path = filepath.FromSlash(path) // / → \
    }
    return path
}
```
