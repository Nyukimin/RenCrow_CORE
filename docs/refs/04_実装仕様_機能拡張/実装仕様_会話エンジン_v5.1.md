# RenCrow v5.1 実装仕様書: ConversationEngine

**バージョン**: 5.1.1
**更新日**: 2026-03-19
**ステータス**: 実装済み
**前提バージョン**: v5.0（3層記憶インフラ実装済み）
**上位仕様**: `docs/01_正本仕様/06_会話エンジン.md`

---

## 1. 概要

ConversationEngine は Mio の会話時に 3 層記憶を取りまとめる薄い統合層である。  
`ConversationManager` の recall/store/thread 操作をラップし、`RecallPack` を LLM プロンプトへ注入できる形に変換する。

現行実装の責務は次の 4 点に限定される。

1. ターン開始時の recall と `RecallPack` 構築
2. ターン終了時の user / mio 発話保存
3. スレッド境界検出による thread flush / recreate
4. in-memory `UserProfile` の best-effort 更新

ConversationEngine は `cfg.Conversation.Enabled` が true のときのみ有効化される。無効時は `MioAgent` が従来どおり会話し、会話エンジン機能は利用しない。

## 2. 公開インターフェース

### 2.1 ConversationEngine

**ファイル**: `internal/domain/conversation/engine.go`

```go
type ConversationEngine interface {
    BeginTurn(ctx context.Context, sessionID string, userMessage string) (*RecallPack, error)
    EndTurn(ctx context.Context, sessionID string, userMessage string, response string) error
    GetPersona() PersonaState
    FlushCurrentThread(ctx context.Context, sessionID string) error
    GetStatus(ctx context.Context, sessionID string) (*ConversationStatus, error)
    ResetSession(ctx context.Context, sessionID string) error
}
```

補足:

- `GetUserProfile` は現行 I/F に存在しない
- `BeginTurn` は recall 失敗時も空に近い `RecallPack` を返して会話継続を優先する
- `EndTurn` は保存や profile extract の失敗を warning 扱いに留める

### 2.2 RecallPack

**ファイル**: `internal/domain/conversation/recall_pack.go`

```go
type RecallPack struct {
    ShortContext []Message
    MidSummaries []ThreadSummary
    LongFacts    []string
    KBSnippets   []string
    Persona      PersonaState
    UserProfile  UserProfile
    Constraints  PromptConstraints
}
```

`PromptConstraints` の現行デフォルト値:

- `MaxTotalTokens = 8192`
- `MaxPromptTokens = 4000`
- `MaxResponseTokens = 512`

`RecallPack.ToPromptMessages()` は user message を含めず、次の順で `[]llm.Message` を構築する。

1. persona + user profile の system prompt
2. 過去文脈の system prompt
3. `ShortContext` の履歴

`MioAgent.Chat()` 側で最後に現在の user message を追加する。

補足:

- `KBSnippets` フィールドは存在するが、現行の `BeginTurn()` は KB 検索結果を `LongFacts` に `[KB] ...` 形式で追加している
- したがって KB 断片の主な実利用先は現状 `LongFacts` である

### 2.3 PersonaState / UserProfile

**ファイル**: `internal/domain/conversation/persona.go`  
**ファイル**: `internal/domain/conversation/user_profile.go`

- `PersonaState` は `Name / SystemPrompt / Tone / Mood` を持つ
- 起動時は `conversation.NewMioPersona(cfg.Prompts.MioPersona)` を使用する
- `UserProfile` は `UserID / Preferences / Facts / UpdatedAt` を持つ
- `UserProfile.Merge()` は preference 上書きと fact の重複排除を行う
- `UserProfile` は現行実装では `RealConversationEngine.profiles[sessionID]` の in-memory キャッシュであり永続化されない

## 3. RealConversationEngine の現行動作

### 3.1 初期化

**ファイル**: `internal/infrastructure/persistence/conversation/engine_impl.go`

`NewRealConversationEngine(manager, persona)` は次を持つ実装を返す。

- `manager domconv.ConversationManager`
- `persona domconv.PersonaState`
- `detector domconv.ThreadBoundaryDetector`（任意）
- `profileExtractor domconv.ProfileExtractor`（任意）
- `profiles map[string]domconv.UserProfile`

オプション設定:

- `WithDetector(d)` でスレッド境界検出を有効化
- `WithProfileExtractor(pe)` で profile extract を有効化

### 3.2 BeginTurn()

`BeginTurn(ctx, sessionID, userMessage)` の処理順:

1. `RecallPack{Persona, Constraints}` を初期化
2. `profiles[sessionID]` があれば `UserProfile` を注入
3. `ConversationManager.Recall(ctx, sessionID, userMessage, 3)` を実行
4. recall 結果を次の規則で分類
   - `SpeakerUser` / `SpeakerMio` -> `ShortContext`
   - `SpeakerSystem` かつ `[Summary]` 接頭辞 -> `MidSummaries`
   - `SpeakerSystem` かつ `[LongTermMemory]` 接頭辞 -> `LongFacts`
   - その他の非空 system message -> `LongFacts`
5. `manager` が `*RealConversationManager` であれば `SearchKB(ctx, domain, userMessage, 3)` を実行
6. KB 結果を `"[KB] " + doc.Content` として `LongFacts` に追加

補足:

- 現在の domain は `GetActiveThread()` から取得し、失敗時は `general`
- recall 失敗時は warning を出して、空に近い `RecallPack` を返す
- KB 検索失敗も warning のみで、会話自体は継続する

### 3.3 EndTurn()

`EndTurn(ctx, sessionID, userMessage, response)` の処理順:

1. detector があれば `GetActiveThread()` で現在 thread を取得
2. `detector.Detect(thread, userMessage, "")` を実行
3. `ShouldCreateNew` の場合:
   - `FlushThread(ctx, thread.ID)`
   - `CreateThread(ctx, sessionID, thread.Domain)`
4. user message を `Store(ctx, sessionID, NewMessage(SpeakerUser, ...))`
5. mio response を `Store(ctx, sessionID, NewMessage(SpeakerMio, ...))`
6. profile extractor があれば `Extract(ctx, thread, existingProfile)` を実行
7. `result.HasData()` の場合は `Merge(newPrefs, newFacts)` して `profiles[sessionID]` を更新

補足:

- store / flush / create / extract はすべて best-effort で、失敗時も warning のみ
- profile は thread 取得に失敗した場合は更新されない
- detector による thread 切替は store より先に走る

### 3.4 補助 API

- `GetPersona()` は現在の `PersonaState` を返す
- `FlushCurrentThread()` は active thread を flush し、同 domain で新 thread を作る
- `GetStatus()` は `ConversationStatus` を返す
  - `SessionID`
  - `ThreadID`
  - `ThreadDomain`
  - `TurnCount`
  - `ThreadStart`
  - `ThreadStatus`
- `ResetSession()` は active thread を flush 後、`general` domain で新 thread を作る

## 4. Mio 連携

### 4.1 通常会話フロー

**ファイル**: `internal/domain/agent/mio.go`

`MioAgent.Chat()` における現行フロー:

1. `conversationEngine.BeginTurn(chatID, userMessage)`
2. `recallPack.ToPromptMessages()`
3. 必要なら web search 結果を system message として追加
4. attribution guard / recent glossary context を追加
5. 現在の user message を最後に追加
6. `llmProvider.Generate()`
7. 発言帰属違反が疑われる場合は 1 回だけ retry
8. `conversationEngine.EndTurn(chatID, userMessage, response)`

補足:

- `conversationEngine == nil` のときは recall/store を省略して通常の LLM 会話だけを実行する
- `BeginTurn()` 失敗時も会話自体は継続する
- `EndTurn()` 失敗時も応答は返す

### 4.2 Persona edit 連携

`detectPersonaEditIntent(userMessage)` が true で persona edit に成功した場合も、戻り値の文面を `EndTurn()` で保存する。

### 4.3 チャットコマンド連携

**ファイル**: `internal/domain/agent/chat_commands.go`

ConversationEngine を使うチャットコマンド:

- `/status`
  - `GetStatus()` を呼び、session / thread / domain / turn count を返す
- `/compact`
  - `FlushCurrentThread()` を呼び、現在 thread を即時切替する
- `/context`
  - `BeginTurn(sessionID, "")` を呼び、現在の `RecallPack` 内容を表示する
- `/new`
  - `ResetSession()` を呼び、新しい会話を開始する

エンジン無効時は一律 `会話エンジンが無効です。` を返す。

## 5. 起動時 DI と依存構成

**ファイル**: `cmd/rencrow/main.go`

`cfg.Conversation.Enabled` が true のとき、起動時に次を構築する。

1. `NewRealConversationManager(cfg.Conversation.RedisURL, cfg.Conversation.DuckDBPath, cfg.Conversation.VectorDBURL)`
2. `cfg.Conversation.EmbedModel` があれば embedder を manager に注入
3. `cfg.Conversation.SummaryModel` もしくは chat model を summarizer に使用
4. 同じ model で `ProfileExtractor` を構築
5. embedder 付き `ThreadBoundaryDetector` を構築
6. `NewRealConversationEngine(realMgr, NewMioPersona(cfg.Prompts.MioPersona))`
7. `WithDetector(detector)`
8. `profileExtractor != nil` なら `WithProfileExtractor(profileExtractor)`

主な設定値:

- `cfg.Conversation.Enabled`
- `cfg.Conversation.RedisURL`
- `cfg.Conversation.DuckDBPath`
- `cfg.Conversation.VectorDBURL`
- `cfg.Conversation.EmbedModel`
- `cfg.Conversation.SummaryModel`
- `cfg.Prompts.MioPersona`

`cfg.Conversation.Enabled == false` の場合は `convEngine=nil` で起動し、ログには `Conversation LLM disabled (v3/v4 mode)` が出る。

## 6. 既知の制約

1. `UserProfile` は in-memory のみで、再起動後は失われる。
2. `KBSnippets` は構造体にあるが、現行 `BeginTurn()` では populated されず、KB 情報は `LongFacts` に寄せている。
3. recall / KB / detector / profile extract の各失敗は fatal ではない。会話継続が優先される。
4. thread 切替は `EndTurn()` 内で実施されるため、厳密な topic segmentation を保証するものではない。
5. `ConversationEngine` は Mio 会話用であり、Worker / Coder の実行経路には直接使われない。

---

## 7. 確認観点

実装確認時の観点は次のとおり。

- `cfg.Conversation.Enabled=true` で `ConversationEngine v5.1 enabled` が起動ログに出る
- `MioAgent.Chat()` で `BeginTurn()` と `EndTurn()` が両方呼ばれる
- `/status`, `/compact`, `/context`, `/new` が会話エンジン経由で動く
- recall 失敗時でも会話応答自体は返る
- profile extractor 有効時に `UserProfile updated` ログが出る
- KB 検索結果が `LongFacts` に `[KB] ...` として入る

以上をもって、v5.1 の ConversationEngine は「設計草案」ではなく、Mio 会話経路に接続済みの現行実装として扱う。
