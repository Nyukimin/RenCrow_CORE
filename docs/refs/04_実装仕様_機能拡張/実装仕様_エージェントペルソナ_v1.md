## 1. 概要

### 1.1 目的

Shiro（Worker）および Coder エージェント（Aka/Ao/Claude）に**軽量なペルソナ（AgentPersona）**と**インメモリ短期会話履歴（LightMemory）**を付与する。

これにより、タスク実行の応答（「修正完了」等）にも各キャラクターの個性が反映され、連続する会話ターンで短い文脈を保持した応答ができる。

### 1.2 設計思想

- **AgentPersona は ConversationEngine ではない**: Mio が持つ v5.1 ConversationEngine（Redis/DuckDB/Qdrant 3 層）とは別の仕組み。外部インフラ不要、純粋な Go struct。
- **LightMemory はインメモリ限定**: プロセス再起動でリセットされる（意図的）。Mio が文脈を管理するため、Shiro/Coder に長期記憶は不要。
- **Persona は system prompt に前置**: 別メッセージではなく、system prompt 本文の先頭にペルソナブロックを追加する。
- **技術指示とペルソナは分離**: `proposalPrompt`（技術指示）と `AgentPersona.Personality`（人格記述）は別のソースから来る。合成順序は `persona → task prompt` 固定。
- **後方互換**: `agents:` セクションが config.yaml に存在しない場合、全機能無効（従来どおりの動作）。

### 1.3 非目的（スコープ外）

- Mio の ConversationEngine（v5.1）は変更しない
- Persona の LLM 自己編集（Mio の PersonaEditor 相当）は v1.0 では実装しない
- Coder のコード生成品質チューニングはこの仕様の範囲外
- LightMemory の永続化（DB 保存）は v1.0 では実装しない

### 1.4 既存型との対比

| 項目 | `conversation.PersonaState` | `agent.AgentPersona` |
|------|----------------------------|---------------------|
| 対象 | Mio 専用 | Shiro / Coder 共通 |
| 外部依存 | ConversationEngine（Redis 等） | なし |
| 自己編集 | PersonaEditor で LLM が書き換え | YAML 手動変更のみ（v1.0） |
| Mood | あり | なし |
| パッケージ | `internal/domain/conversation` | `internal/domain/agent` |

## 2. 公開インターフェース

### 2.1 AgentPersona 型

**ファイル**: `internal/domain/agent/persona.go`（新規）

```go
// AgentPersona は Shiro/Coder 向けの軽量ペルソナ定義。
// conversation.PersonaState（Mio専用）とは独立した型。
type AgentPersona struct {
    Name        string // Agent表示名（例: "Shiro", "Aka"）
    Personality string // ペルソナ記述。system prompt 先頭に前置される。
    Tone        string // 口調ヒント（例: "calm", "analytical"）将来のTTS連携用
}

// BuildSystemPrompt はペルソナブロックとタスクプロンプトを合成する。
// Personality が空文字の場合は taskPrompt をそのまま返す（後方互換）。
// 合成順序: Personality + "\n\n" + taskPrompt
func (p AgentPersona) BuildSystemPrompt(taskPrompt string) string
```

### 2.2 LightMemory 型

**ファイル**: `internal/domain/agent/light_memory.go`（新規）

```go
// LightMemory はセッション単位のインメモリ短期会話履歴。
// プロセス再起動でリセット（意図的）。goroutine-safe。
type LightMemory struct {
    mu       sync.Mutex
    sessions map[string]*sessionBuffer // key: chatID（LINE UserID 等）
    maxTurns int                       // セッションあたりの最大保持ターン数
}

// NewLightMemory は新しい LightMemory を作成する。
// maxTurns はセッションあたりの保持ターン数上限（推奨: 3〜5）。
func NewLightMemory(maxTurns int) *LightMemory

// Record は会話ターン（userMessage + response ペア）を記録する。
// maxTurns を超えた古いターンは FIFO で破棄される。
func (m *LightMemory) Record(sessionID, userMessage, response string)

// RecentMessages は指定セッションの直近ターンを llm.Message スライスとして返す。
// user/assistant の交互メッセージを返す。system prompt は含まない。
// セッションが存在しない場合は nil を返す。
func (m *LightMemory) RecentMessages(sessionID string) []llm.Message

// Clear は指定セッションの履歴を削除する。
func (m *LightMemory) Clear(sessionID string)

// ClearAll は全セッションの履歴を削除する（日次カットオーバー用）。
func (m *LightMemory) ClearAll()
```

内部型:

```go
type sessionBuffer struct {
    turns []turn
}

type turn struct {
    userMessage string
    response    string
    timestamp   time.Time
}
```

## 3. system prompt 構築ルール

### 3.1 合成順序

```
┌──────────────────────────────────────────────┐
│  [1] AgentPersona.Personality                │ ← 人格・口調の記述
│      （config.yaml の agents.*.persona 等）  │
├──────────────────────────────────────────────┤
│  [2] Task Prompt                             │ ← 技術指示
│      （worker.md / coder.md の内容）         │
└──────────────────────────────────────────────┘
```

### 3.2 Shiro の場合

```
[AgentPersona.Personality]     ← "あなたはShiro。落ち着いていて..."
\n\n
[cfg.Prompts.Worker]           ← "You are a worker agent. Execute tasks..."
```

### 3.3 Coder の場合

```
[AgentPersona.Personality]     ← "あなたはAka。設計思考が得意で..."
\n\n
[cfg.Prompts.CoderProposal]    ← "You are a professional coder agent..."
```

### 3.4 LightMemory を含む完全メッセージ列

```
messages = [
    {role: "system",    content: persona + "\n\n" + taskPrompt},
    {role: "user",      content: "前回の依頼の続きで..."},    // LightMemory[0]
    {role: "assistant", content: "了解。前回は..."},          // LightMemory[0]
    {role: "user",      content: "テストも追加して"},         // LightMemory[1]
    {role: "assistant", content: "テスト追加完了"},           // LightMemory[1]
    {role: "user",      content: "今回のリクエスト"},         // 現在のリクエスト
]
```

### 6.3 Execute() の変更点

```go
func (s *ShiroAgent) Execute(ctx context.Context, t task.Task) (string, error) {
    // ペルソナ付き system prompt を構築
    fullSystemPrompt := s.persona.BuildSystemPrompt(s.systemPrompt)

    if s.subagentManager != nil {
        result, err := s.subagentManager.RunSync(ctx, SubagentTask{
            AgentName:    "shiro",
            Instruction:  t.UserMessage(),
            SystemPrompt: fullSystemPrompt, // persona を含む
        })
        if err != nil {
            return "", err
        }
        // LightMemory に記録
        if s.lightMemory != nil {
            s.lightMemory.Record(t.ChatID(), t.UserMessage(), result.Output)
        }
        return result.Output, nil
    }

    // フォールバック: 単純 LLM 呼び出し（LightMemory の過去ターンを注入）
    messages := []llm.Message{
        {Role: "system", Content: fullSystemPrompt},
    }
    if s.lightMemory != nil {
        if recent := s.lightMemory.RecentMessages(t.ChatID()); len(recent) > 0 {
            messages = append(messages, recent...)
        }
    }
    messages = append(messages, llm.Message{Role: "user", Content: t.UserMessage()})

    req := llm.GenerateRequest{
        Messages:    messages,
        MaxTokens:   4096,
        Temperature: 0.3,
    }
    resp, err := s.llmProvider.Generate(ctx, req)
    if err != nil {
        return "", err
    }
    if s.lightMemory != nil {
        s.lightMemory.Record(t.ChatID(), t.UserMessage(), resp.Content)
    }
    return resp.Content, nil
}
```

**注意**: SubagentManager（ReActLoop）経由の場合、`SubagentTask` は `SystemPrompt` と `Instruction` しか受け取らないため、LightMemory の過去ターンは messages には注入されない（v1.0 の制約、§9.2 参照）。

## 8. Mio との関係

### 8.1 変更なし

Mio の ConversationEngine（v5.1）には一切変更を加えない。

| | Mio | Shiro | Coder |
|--|-----|-------|-------|
| 記憶 | Redis/DuckDB/Qdrant 3層 | LightMemory（インメモリ） | LightMemory（インメモリ） |
| ペルソナ | PersonaState（LLM自己編集可） | AgentPersona（YAML固定） | AgentPersona（YAML固定・各Coder別） |
| 再起動後 | Redis から復元 | リセット | リセット |

### 8.2 ルーティング経路

```
User → LINE → Mio.Chat()
  → CHAT:  Mio 自身が応答 [ConversationEngine v5.1]
  → OPS:   Shiro.Execute() [AgentPersona + LightMemory]
  → CODE1: Aka（CoderAgent + DeepSeek） [AgentPersona + LightMemory]
  → CODE2: Ao（CoderAgent + OpenAI） [AgentPersona + LightMemory]
  → CODE3: Claude（CoderAgent + Claude API） [AgentPersona + LightMemory]
```

## 9. IdleChat との統合

### 9.1 現状

IdleChat は `personalities map[string]string`（`LoadedPrompts.IdleChatAgents`）を独自に保持。`agents.*.persona` と `prompts/idle_chat/*.md` が 2 系統に分かれる可能性がある。

### 9.2 Phase 1 方針（v1.0）: 共存

`agents.*.persona` が設定されていれば IdleChat の `personalities` をその値で上書き、設定がなければ既存の `idle_chat/*.md` がそのまま使われる。

`cmd/picoclaw/main.go` の IdleChat 構築時（既存コードの近く）:

```go
// agents.*.persona が設定されていれば IdleChat の personalities を上書き
idleChatPersonalities := cfg.Prompts.IdleChatAgents
for agentName, agentCfg := range map[string]config.AgentConfig{
    "shiro": cfg.Agents.Shiro,
    "aka":   cfg.Agents.Aka,
    "ao":    cfg.Agents.Ao,
} {
    if agentCfg.Persona != "" {
        key := strings.ToUpper(agentName[:1]) + agentName[1:] // "Shiro", "Aka", "Ao"
        idleChatPersonalities[key] = agentCfg.Persona
    }
}
idleChatOrch := idlechat.NewIdleChatOrchestrator(
    ..., idleChatPersonalities, ...
)
```

`idleSpeakerContract()` のハードコードされたスタイルルールは本仕様の範囲では変更しない。

### 9.3 Phase 2（将来）: 完全統一

`LoadedPrompts.IdleChatAgents` を廃止し、`AgentsConfig` に一本化。`idleSpeakerContract()` も YAML 化。本仕様の範囲外。

## 13. 既知の制約

1. **SubagentManager 経由は LightMemory の過去ターンが注入されない**: `SubagentTask` は `SystemPrompt` と `Instruction` のみ受け取る。Shiro が SubagentManager（ReActLoop）を使う場合、過去ターンは `messages` 列に注入できない。LightMemory への**記録**は行うが、**注入**は単純 LLM フォールバック経路のみ。将来的に `SubagentTask` に `HistoryMessages []llm.Message` を追加することで解消できるが、v1.0 ではスコープ外。

2. **LightMemory のセッション数に上限がない**: 多ユーザー環境では sessions map が増え続ける。v1.0 では `max_turns` が小さい（3）ため深刻にならないが、将来的に LRU eviction を検討する。

3. **Coder persona が Proposal フォーマットに干渉するリスク**: ペルソナが日本語で書かれ、proposalPrompt が英語の場合、LLM が Plan/Patch セクション名を日本語にする可能性がある。対策: ペルソナの末尾に「ただし、技術的な出力は指示のフォーマットに従うこと」を推奨として追記する。

4. **Persona の自己編集は未実装**: Mio のような「キャラを変えて」への対応は v1.0 ではない。YAML を手動編集してプロセスを再起動する必要がある。

5. **日次カットオーバーとの連携**: `LightMemory.ClearAll()` をカットオーバー処理から呼ぶことで古い文脈の持ち越しを防げるが、カットオーバー処理への組み込みは Phase 4 の配線作業に含める。

## 14. 確認観点

- `agents.shiro.persona` 設定時、Shiro の応答文体が変化する
- `agents.aka.persona` 設定時、Aka（DeepSeek）の Proposal 応答文体が変化する
- Ao と Claude の persona がそれぞれ独立して反映される
- `light_memory.enabled: true` + 連続 2 メッセージで、2 回目の応答が 1 回目の文脈を参照する
- `agents:` セクションなし時（ゼロ値）でも既存動作と完全に同一
- Shiro の `WithPersona()` を呼ばない場合、`BuildSystemPrompt()` が元の `systemPrompt` をそのまま返す（後方互換）
- 起動ログに persona 設定状態が出力される
