# CLI運用型 投資研究基盤 仕様化プロンプト

## 目的

このプロンプトは、RenCrow / picoclaw_multiLLM の株式・ETF・暗号資産データ基盤を、CLI主体の投資研究・紙運用基盤として仕様化するためのものである。

ユーザーの目的は、余剰資金100万円をできるだけ早く200万円へ近づけることである。ただし、仕様上は「最短倍増だけ」を直接最大化しない。集中、レバレッジ、高回転、イベント前の過剰リスク、LLM裁量売買を抑制し、最大ドローダウン制約下での税後複利成長を目的関数にする。

## 仕様化プロンプト

```text
あなたは RenCrow / picoclaw_multiLLM の投資研究基盤を仕様化するシニアエンジニア兼クオンツ設計者です。

次の source of truth を最初に読み、矛盾がある場合は下記の優先順位で判断してください。

1. docs/株式/株式_アルゴリズム評価.md
2. docs/株式/学習基盤_実装仕様書.md
3. docs/株式/株式_学習基盤.md
4. docs/株式/01_クオンツ戦略アーキテクト.md から 15_規制コンプライアンス枠組み.md までの関連文書
5. rencrow-data/ 配下の現在の実装、DBスキーマ、CLI、テスト

## 前提

- ユーザーは余剰資金100万円をできるだけ早く200万円へ近づけたい。
- NISAの投資信託は長期非課税コア資産として完全に分離する。
- RenCrowは課税口座の戦術オーバーレイとして扱う。
- 実運用の初期対象は高流動性ETF中心とする。
- 個別株、指数、暗号資産、商品ETF、マクロ系列は学習・比較・特徴量・レジーム判定には使ってよい。
- ライブ売買は初期スコープ外とし、データ基盤、バックテスト、リスク管理、イベントveto、紙運用を先に完成させる。
- 売買判断の直接主体はLLMではない。LLMは候補整理、異常要約、イベント解釈、仕様書生成、週次レポート生成など、説明補助が必要な箇所だけに限定する。
- 運用は基本的にCLIで行う。取得、検証、特徴量生成、バックテスト、紙運用、レポート生成、スナップショット作成はすべて再現可能なコマンドとして定義する。

## 守るべき設計原則

- LLMの出力だけで売買しない。
- 先読みバイアスを許さない。
- in-sample成績だけで採用しない。
- イベント検知は攻めの材料ではなく、新規建て停止、サイズ縮小、現金退避のvetoとして扱う。
- 税、手数料、スプレッド、スリッページを評価から外さない。
- NISA口座、課税口座、紙運用、本番運用を混ぜない。
- rawデータ、取得元、取得時刻、fetch status、補正履歴、feature版、snapshot hash、decision logを保存する。
- マーケットメイキング、高頻度売買、板データ前提の戦略、空売り必須の統計的アービトラージは初期実運用から除外する。
- 100万円を200万円へ近づける目的は重視するが、破綻確率を上げるレバレッジ、集中、過剰回転をシステム側で弾く。

## 仕様化する対象

CLI中心に、次の仕様を作成してください。

1. データ取得CLI
   - 銘柄ユニバース同期
   - 過去データ取得
   - 増分取得
   - 取得失敗、partial、stale、欠損率の記録
   - asset_type別の扱い: ETF, STOCK, INDEX, FX, RATE, MACRO, CRYPTO, CASH_PROXY

2. データ品質検証CLI
   - 価格欠損
   - 異常リターン
   - adjusted factor anomaly
   - corporate action差分
   - 銘柄ごとの開始日、終了日、行数、欠損率
   - 取得元ごとのfail/partial集計

3. 特徴量生成CLI
   - 週次リターン
   - 4週、12週、26週モメンタム
   - 直近1週skipのモメンタム
   - realized volatility
   - drawdown
   - moving average gap
   - volume change
   - USD建て資産の円換算
   - マクロ特徴量
   - イベントフラグ

4. バックテストCLI
   - 週次ETF回転戦略を最小ベース戦略とする
   - 1週ラグ執行
   - イベントveto
   - cash-like ETF退避
   - 税前、税後、手数料後の評価
   - OOS、walk-forward、期間分割
   - CAGR, Sharpe, Sortino, MaxDD, turnover, hit rate, exposure, tax drag

5. リスク管理CLI
   - 最大DD制限
   - 週次損失制限
   - 銘柄集中制限
   - asset class集中制限
   - ボラ目標
   - イベント週のサイズ縮小
   - 新規建て停止条件
   - kill switch条件

6. 紙運用CLI
   - 週次decision candidate生成
   - human approval前提
   - paper_trade_log記録
   - 約定仮定
   - TCA
   - 税ロット近似
   - 8〜12週間の紙運用検証ゲート

7. LLM利用CLI
   - LLMを使う箇所と使わない箇所を明確に分ける
   - LLM入力、出力、モデル名、プロンプト版、参照データsnapshot_idを保存する
   - LLM出力は売買命令ではなく、説明、要約、仮説、警告、レポートとして扱う
   - LLMが不確実、矛盾、外部確認必要と判断した場合は、人間確認またはno tradeへ倒す

8. スナップショットと監査
   - snapshot_registry
   - db_hash
   - features_hash
   - source_summary_json
   - event_state_json
   - decision_log
   - paper_trade_log
   - order_logは将来用
   - tax_lot_logは課税口座のみ

## 実装順序

優先順位は次の通りにしてください。

1. データ取得と品質検証
2. feature_weeklyの再現性固定
3. snapshot作成
4. 週次ETFベース戦略バックテスト
5. イベントveto
6. 税・手数料・スリッページ反映
7. リスク制約
8. decision candidate生成
9. 紙運用
10. LLMレポート補助

## 禁止事項

- LLMに直接売買判断を委任する仕様
- ライブ発注を初期MVPに含める仕様
- NISAと課税口座を混ぜる仕様
- マーケットメイキングや高頻度売買を初期実運用に含める仕様
- 成績が良いin-sampleだけを根拠に採用する仕様
- 取得元や補正履歴を残さない仕様
- 失敗をsuccessに丸める仕様
- 「100万円を最短で200万円にする」ためにDD制限を外す仕様

## 重要な判断基準

仕様の良し悪しは、次で判断してください。

- CLIだけで同じ結果を再現できるか
- その週に見えていたデータだけで判断できるか
- 取得失敗、欠損、イベント、税、手数料、DDが売買候補に反映されるか
- LLMの出力を監査できるか
- 人間が後から「なぜその候補になったか」を追えるか
- 100万円から200万円を狙う意図を保ちながら、破綻しやすい賭け方を弾けるか

以上を満たす、実装可能でCLI中心の仕様書を作成してください。
```

## 使用方法

1. 上記プロンプトを新しい仕様化セッションの冒頭に貼る。
2. 仕様化前に `docs/株式/株式_アルゴリズム評価.md` と `docs/株式/学習基盤_実装仕様書.md` を必ず読ませる。
3. 仕様化結果は `docs/株式/` 配下に追加し、CLI、DB、テスト、運用手順へ分解する。
4. 仕様化後の実装では、LLMを売買決定者として扱わないことを再確認する。
