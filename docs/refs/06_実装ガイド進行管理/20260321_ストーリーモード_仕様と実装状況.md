## 1. 実装したい仕様

### 1.1 品質目標（最上位）

「生成が成功する」ではなく「ユーザーに見せる価値のある物語」を安定して出力する。

**合格基準（全条件を満たすこと）**:

| # | 条件 |
|---|---|
| Q1 | タイトルを読まずに元話が認識できる |
| Q2 | 出来事として描写される（テーマの説明・説教でなく） |
| Q3 | 機能する twist が見える（元話とどこが違うかが分かる） |
| Q4 | 余韻で終わる（教訓の言い直し・総括でなく） |
| Q5 | `StorySource.Text` の単純な再話より明らかに良い |
| Q6 | 人が「面白かった」と言いそう |

**判定基準**: 3回連続ライブプローブが Q1〜Q6 をすべて通過するまで成功とみなさない。

---

### 1.2 パイプライン仕様

```
RunStorySession()
├── Step 1: スタイル選択          [決定論的] chooseStoryRewriteStyle()
└── loop (最大3ソース試行)
    ├── Step 2-6: prepareStory()  [決定論的・LLMなし]
    │   ├── 2. ソース選択          selectStorySourceExcluding()
    │   ├── 3. ソース分析          analyzeStorySource()
    │   ├── 4. 改変プラン生成      buildStoryRewritePlan()
    │   ├── 5. ビートプラン生成    groundedStoryBeatPlan()
    │   └── 6. アダプテーション    buildStoryAdaptationPlan()
    │
    │   [Step 6.5: ヒューマンレビューゲート — preview-run モードのみ]
    │
    ├── Step 7: 第1稿生成          [LLM: shiro × 4ビート, MaxTokens=300, Temp=0.6]
    │   └── 失敗時: deterministicStoryDraft → safeStoryRetelling（安全装置）
    └── Step 8: 改稿               [LLM: shiro, MaxTokens=1800, Temp=0.5]
        └── 失敗時: repairStoryDraft → deterministicStoryDraft（安全装置）
```

**設計原則**:
- Step 2〜6 は決定論的。LLM は Step 7・8 のみ
- フォールバック（deterministicStoryDraft / safeStoryRetelling）は**安全装置**であり**目標ではない**
- フォールバックで生成された再話は品質合格とみなさない

### 1.3 データ仕様

**コーパス**: 17作品（アクティブ）+ 10作品（アーカイブ）

| 区分 | 件数 | 場所 |
|------|------|------|
| アクティブ | 17 | `data/story/*.json` |
| アーカイブ（データ欠損） | 10 | `data/story/archive/*.json` |

**アーカイブ対象作品**（`data/story/archive/`）:
`aladdin`, `ali40`, `beauty`, `bremen`, `bunbuku`, `frogprince`, `hansel`, `kintaro`, `puss`, `sleepingbeauty`

アーカイブ理由: `text` フィールドの scene 密度不足など、必須フィールドのデータが基準を満たさない。
アーカイブ作品は生成対象外。データを補完次第アクティブに復帰可能。

**各 JSON の必須フィールド**:
- `text`: 場面順序が分かる長めの scene-aware synopsis（梗概ではなく、主要イベントの流れを含む）
- `spec.skeleton.required_beats`: 4ビート（導入・逸脱・反転・着地相当）
- `spec.skeleton.recognition_cues`: 元話認識に必要な固有語（2個以上）
- `spec.twists`: 5スタイル全て（1文の抽象的構造観察）

**各 JSON のオプションフィールド**（空の場合はデフォルト値を使用）:
- `opening_seed`: 第1ビートの起点文。空の場合 `"{Title}は{Setting}で足を止め…"` を自動生成
- `setting`: 舞台設定（例: `"大晦日の路地と商店街"`）。空の場合 `"現代の地方都市とその周辺"` を使用

> **設計原則**: 物語固有のデータ（起点文・舞台）は JSON に持つ。Go コードに switch/case で埋め込まない。

**5スタイル**: `role_shift` / `view_shift` / `value_shift` / `inversion` / `scale_shift`

**4ジャンル**: ノワール / ホラー / コメディ / ノーマル

---

### 1.4 Step 7 プロンプト仕様（第1稿）

- 各ビートを**2〜4文の散文**として個別生成し結合
- 第1ビートは `openingSeed`（固定文）で始める
- 禁止事項をプロンプトで明示:
  - 「もし〜だったら」等で前置きせず、直接場面を書く
  - 情景描写だけで始めない（行動か対話で始める）
  - 元話テキストを一文そのまま使わない
  - 現代ハイテク・企業名・プロジェクト名を新作しない
  - メタ的な設計説明を本文に書かない

---

### 1.5 テスト仕様

**必須カバレッジ**:

| テスト対象 | 仕様 |
|---|---|
| Step 1 | 全5スタイルから直前と異なるスタイルが選ばれること |
| Step 2 | excluded / blocked セットによるソース除外が正しく機能すること |
| Step 3 | アクティブ17作品で `analyzeStorySource()` が非空の Motifs / TabooOrRule を返すこと |
| Step 4 | 5スタイル × 4ジャンルの全20通りで `StoryRewritePlan` の必須フィールドが非空であること |
| Step 5 | RequiredBeats が 0〜4件の各ケースで `StoryBeatPlan` が非空文字列を返すこと |
| Step 2-6 | アクティブ17作品 × 5スタイルで `prepareStory()` がパニックせず非空の `StoryPrep` を返すこと |

### 1.6 Step 6.5 仕様（ヒューマンレビューゲート）

Step 6（決定論的プラン生成）と Step 7（LLM 第1稿生成）の間に入るヒューマンレビューゲート。
`preview-run` モード専用。通常の `RunStorySession()` では省略される。

**フロー**:
```
preview-run SOURCE [STYLE]
├── Step 6:   BuildStoryPrepForSource()  → StoryPrep 生成
├── Step 6.5: 情報収集フェーズ（必ずこの順で全出力する）
│   ├── [必須] DumpStoryPrep() — ANALYSIS / PLAN / BEAT PLAN / ADAPTATION を全表示
│   ├── [必須] ScanStoryPlanProblems() — 既知アンチパターンを警告
│   ├── [必須] GenerateRawBeats() — BEAT 0〜3 の生文章（バリデーションなし）を表示
│   └── 確認プロンプト（Enter: 続行 / q: 中止）← ユーザーの OK が出るまで先に進まない
└── Step 7+8: GenerateFromPrep()  ← OK が出た場合のみ実行
    ├── DRAFT (Step 7 出力) 表示
    └── STORY (Step 8 出力) 表示
```

**目的**: プランが意図通りか人が確認してから LLM 生成コストをかける。品質問題の切り分けにも使用する。

## 4. ライブプローブ手順

```bash
# テスト
env GOCACHE=/home/nyukimi/RenCrow_CORE/.gocache \
    GOMODCACHE=/home/nyukimi/RenCrow_CORE/.serena/gomodcache \
    go test ./internal/application/idlechat

# ビルド
env GOCACHE=/home/nyukimi/RenCrow_CORE/.gocache \
    GOMODCACHE=/home/nyukimi/RenCrow_CORE/.serena/gomodcache \
    go build -o test-story ./cmd/test-story

# 全ソース一覧
./test-story list

# プランのみ確認（LLM不使用）
./test-story dump-plan SOURCE [STYLE]
./test-story dump-all SOURCE

# プランを確認してから LLM 生成（ヒューマンレビューあり）
./test-story preview-run SOURCE [STYLE]

# アンチパターンスキャン
./test-story scan SOURCE STYLE
./test-story scan-all

# フル実行（特定ソース強制）
STORY_SOURCE=momotaro timeout 180s ./test-story
```

**合否の確認ポイント**:
- `story_text` が `story_draft_text` より良いか
- タイトルを隠して読んで元話が分かるか
- 「もし〜だったら」「（情景）」で始まっていないか
- 出来事として書かれているか（説明・説教でないか）
- ひねり（twist）が文章に現れているか（倉庫に変えただけ、ではないか）
