# RenCrow リファクタリング指針

## この文書の役割

このディレクトリは、RenCrow のフォルダ構成整理とモジュール化リファクタリングの作業指針を集約する場所である。

今後、リファクタリングに関する判断、移行計画、検証観点、禁止事項は `docs/refactor/` 配下にまとめる。実装仕様そのものは引き続き `docs/01_正本仕様/実装仕様.md` を一次参照とし、このディレクトリは「正本仕様に合わせて既存コードを段階的に整えるための作業指針」として扱う。

## 目的

- 一般的で読みやすい Go プロジェクト構成へ寄せる。
- 継ぎ足しで肥大化した処理を、仕様上の責務に合わせてモジュール化する。
- モジュール化と疎結合を最重要原則とし、機能ごとに差し替え可能な境界を作る。
- Chat / Worker / Coder の責務境界をコード構造でも読み取れるようにする。
- Viewer、IdleChat、LLM routing、TTS/STT、runtime config、memory/source registry の結合点を明示する。
- 一度に大きく動かさず、テストと実機確認を伴う小さな移行に分ける。

## 最重要原則: モジュール化と疎結合

RenCrow のリファクタリングでは、モジュール化と疎結合を最も重視する。

目的は、機能ごとに責務を切り分け、必要に応じて差し替えできる構造を保つことである。ある機能を差し替えるとき、他の機能や別の層へ影響が広がらない構造を目指す。

特に次の領域は、差し替え可能な境界を意識して整理する。

- Chat / Worker / Coder
- Viewer
- IdleChat
- LLM provider
- STT / TTS
- Memory / Source Registry
- runtime config
- tool execution / security policy

単にファイルを分けるだけではモジュール化ではない。入力、出力、副作用、永続化、ログ、エラー契約が明確で、内部実装に依存せず利用できる単位をモジュールとして扱う。

境界には、必要に応じて interface、contract、event、DTO、adapter を置く。呼び出し側は内部実装ではなく、明示された契約に依存する。

共通処理の抽出は、意味のある責務境界がある場合だけ行う。「便利だから共有する」「似ているからまとめる」だけの共通化は禁止する。巨大な `service` / `manager` / `helper` / `util` を新設して、依存を集約するだけの整理は行わない。

## 不変の責務境界

### Chat

- ユーザー対話を受ける。
- ルーティング判断を行う。
- 結果を返す。
- 実行の詳細、ファイル操作、破壊的操作を抱え込まない。

### Worker

- 実行主体である。
- Coder が生成した plan / patch を実行する。
- ファイル編集、コマンド実行、テスト実行、実行ログ記録を担当する。
- protected file、workspace 制約、policy 判定を迂回しない。

### Coder

- 設計、plan、patch 生成を担当する。
- 破壊的操作を直接実行しない。
- 実行責務を Application / Worker 側へ戻す。

## 層の責務

### `cmd/`

起動、CLI、composition root、依存注入、runtime wiring を置く。業務ロジックや usecase の本体を増やさない。

### `internal/domain/`

ビジネス上の契約、値、状態、責務境界を置く。外部 API、DB、HTTP、ファイルシステム、Viewer 実装へ依存しない。

### `internal/application/`

usecase と orchestration を置く。Chat / Worker / Coder の流れ、IdleChat、Worker execution、source fetch、audio coordination など、複数の domain / infrastructure を組み合わせる処理を担当する。

### `internal/infrastructure/`

LLM provider、persistence、security、tool execution、transport、STT/TTS、VTuber、MCP など技術詳細を置く。

### `internal/adapter/`

LINE、Slack、Viewer、health、config API、channel handler など外部入出力を置く。Application 層を呼び出す入口であり、業務判断を溜め込まない。

## 旧解析資料の扱い

旧 `docs/codebase-map/` は削除済みであり、現行リファクタリング判断では参照しない。構造確認は現在コード、正本仕様、`docs/10_新仕様/` を使う。

ただし、`codebase-map` は静的解析の地図であり、最終判断ではない。実装前には対象 symbol body、周辺コード、テスト、必要に応じて live runtime / Viewer 実挙動を確認する。

## 進め方

1. 対象のユースケースを決める。
2. `codebase-map` で関係する結合点を確認する。
3. 正本仕様と現在実装の差分を短く書く。
4. モジュール境界の入力、出力、副作用、永続化、ログ、エラー契約を書く。
5. 移動対象、残す対象、触らない対象を分ける。
6. 先にテストまたは検証観点を決める。
7. 最小単位で移動または分割する。
8. import、テスト、実機確認を行う。
9. 次の段階へ進む前に差分を整理する。

## 禁止事項

- 正本仕様を読まずに移動する。
- `cmd/picoclaw/main.go` の肥大化を別の巨大ファイルへ移すだけで終える。
- Chat / Worker / Coder の責務を曖昧にする。
- ファイルを分けただけでモジュール化できたと判断する。
- 内部実装へ直接依存する呼び出しを増やす。
- 「便利だから共有する」「似ているからまとめる」だけの共通化を行う。
- 巨大な `service` / `manager` / `helper` / `util` を新設する。
- Worker 実行安全境界を「整理」の名目で薄くする。
- fallback を正常系として隠す。
- Viewer 表示、音声、口パク、ログを同じデータ契約として扱う。
- runtime config、repo example、Viewer 表示値を同一視する。
- archive 文書を一次参照にする。
- 大規模移動と機能変更を同じ差分に混ぜる。
- テストや実機確認なしで「構成だけの変更だから安全」と判断する。

## 成果物ルール

- リファクタリング方針は `docs/refactor/` 配下へ置く。
- ファイル名は日本語にする。
- 実装仕様の正本を置き換えない。
- 変更計画は段階単位で書く。
- 完了条件にはテストまたは代替検証を必ず書く。
- 後から読んだ人が「なぜこの順で動かすのか」を追えるようにする。
