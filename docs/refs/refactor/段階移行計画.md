# 段階移行計画

## 基本方針

リファクタリングは「移動」「分割」「仕様変更」を混ぜない。

最初は挙動を変えず、構造だけを小さく整える。挙動変更が必要な場合は、別段階としてテストまたは実機確認を先に定義する。

すべての段階で、モジュール化と疎結合を最重要判断基準にする。機能を分けるときは、単にファイルを分割するのではなく、差し替え可能な境界を作る。

各段階の開始前に、対象モジュールの入力、出力、副作用、永続化、ログ、エラー契約を明記する。境界が説明できない場合は、移動より先に責務分析を行う。

## Phase 0: 現状固定

目的:

- 現在の動作と構造を固定する。
- どの差分がリファクタで、どの差分が仕様変更かを分ける。
- 機能ごとの差し替え境界を仮定ではなく現在コードから確認する。

作業:

- 現在コード、正本仕様、`docs/10_新仕様/13_実装項目インベントリ.md` を読む。
- 変更対象の周辺コードを読む。
- 対象 package のテストを確認する。
- 既存の failing test があれば記録する。
- live runtime が関係する場合は `~/.rencrow/config.yaml` と `/health` を確認する。
- 対象機能の interface / contract / event / DTO / adapter の有無を確認する。

完了条件:

- 対象ユースケース、対象ファイル、確認するテストが明記されている。
- 実装前の検証観点が書かれている。
- 差し替えたい単位と、差し替え時に影響させない範囲が書かれている。

## Phase 1: `cmd/rencrow` の composition root 整理

目的:

- `cmd/rencrow/main.go` に集まった wiring と実処理を分ける。
- 起動処理は残し、usecase の本体を Application / Infrastructure へ寄せる。

候補:

- runtime factory の責務確認。
- handler 登録処理のまとまり整理。
- config loading と runtime display の境界確認。
- IdleChat / STT / TTS / VTuber bridge の wiring と本体の分離。
- wiring と実装詳細の間に必要な contract を確認する。

注意:

- service restart や runtime config を伴う変更は別扱いにする。
- `cmd/` から抜いた処理を新しい巨大 package に集めない。
- composition root を薄くするために、依存を別の巨大 manager へ移さない。

完了条件:

- 起動テストまたは関連 `cmd/rencrow/*_test.go` が通る。
- 必要なら `http://127.0.0.1:18790/health` で live 起動を確認する。
- 起動側が各機能の内部実装へ直接依存していないことを説明できる。

## Phase 2: Chat / Worker / Coder route chain の明確化

目的:

- route 判断、route dispatch、response assembly を読みやすくする。
- Chat / Worker / Coder の責務境界をコード上で追いやすくする。

候補:

- `MessageOrchestrator.ProcessMessage` の分岐を小さな内部関数へ分ける。
- route ごとの処理責務を明示する。
- Coder proposal 生成と Worker 実行の境界を再確認する。
- route 追加時の確認点をテスト化する。
- route ごとの入力、出力、event、error contract を整理する。

注意:

- Coder に実行責務を戻さない。
- route fallback は正常成功として扱わない。
- Viewer event、session、report、TTS hook を落とさない。
- route 分岐を共通化しすぎて、Chat / Worker / Coder の差を隠さない。

完了条件:

- route 別の unit test または integration test が通る。
- Viewer から最低 1 セッションの表示・イベントを確認する。
- agent 実装を差し替える場合の影響範囲が説明できる。

## Phase 3: Worker execution 安全境界の分離

目的:

- Worker 実行の file edit / shell / git / protected check / logging を読みやすくする。
- 安全境界を薄めず、むしろ追跡しやすくする。

候補:

- `WorkerExecutionService` の command dispatch を責務別に整理する。
- protected pattern 判定と workspace 制約のテストを増やす。
- `ToolRunner` と `PolicyEngine` の責務差を文書化する。
- 実行ログの必須項目を固定する。
- tool request、policy result、execution record の契約を固定する。

注意:

- セーフガードを無効化しない。
- file/shell/git の実行経路を横断的に変更しない。
- protected file の扱いは最初にテストで固定する。
- 実行方式を差し替えるために、安全判定を個別実装へ埋め込まない。

完了条件:

- Worker execution の既存テストが通る。
- protected file / workspace 外操作 / blocked command の検証がある。
- ToolRunner または PolicyEngine を差し替えた場合に守る契約が説明できる。

## Phase 4: Viewer / IdleChat 表示契約の分離

目的:

- LLM raw response、表示本文、TTS chunk、口パク trigger、ログを混同しない構造にする。
- IdleChat と Viewer の契約を実装上も追いやすくする。

候補:

- IdleChat の response model を表示用、raw diagnostic、audio trigger に分ける。
- Viewer SSE event と JS rendering の契約を整理する。
- fallback / invalid response をエラー経路として表示する。
- raw と view data の境界を明示する。
- Viewer component や IdleChat generator を差し替えられる event / DTO 境界を確認する。

注意:

- 音声 chunk を本文表示の唯一根拠にしない。
- fallback を成功として隠さない。
- DOM 要素の存在だけで完了扱いしない。
- JS/CSS の都合を Application の内部契約へ持ち込まない。

完了条件:

- Viewer で最低 1 セッションを追う。
- 表示本文、event log、終了状態、必要なら TTS/口パク trigger を照合する。
- Viewer 表示実装と IdleChat 生成実装を別々に差し替えられる境界が説明できる。

## Phase 5: Memory / Source Registry / persistence 境界の整理

目的:

- memory、staging、Source Registry、news、knowledge、search cache の状態遷移を読みやすくする。
- `L1SQLiteStore` に集中した責務を、状態境界に沿って整理する。

候補:

- state transition の関数を明確化する。
- observed / candidate / validated / rejected / promoted の流れをテストで固定する。
- Source Registry handler と persistence の責務を分ける。
- repository contract と state transition contract を分ける。

注意:

- observed/candidate から confirmed/pinned へ直接昇格しない。
- cache を整合性設計の代替にしない。
- raw data、Viewer 表示、prompt injection data を混ぜない。
- persistence 差し替えのために状態遷移ルールを DB 実装へ閉じ込めない。

完了条件:

- state transition のテストがある。
- Viewer memory/source panels の表示確認がある。
- persistence、validator、sweeper を個別に差し替える場合の契約が説明できる。

## Phase 6: LLM provider / runtime config の整理

目的:

- provider factory、middleware、runtime config、Viewer LLM ops の関係を読みやすくする。
- repo example、live config、Viewer 表示の三重化による誤診断を減らす。

候補:

- provider alias と endpoint 解決の責務確認。
- raw log middleware と health/status 表示の境界整理。
- `docs/LLM運用/` と実装の対応確認。
- provider interface、request/response DTO、raw log 契約を確認する。

注意:

- repo example だけを見て runtime 判断しない。
- `/ready` など未実装 endpoint を前提にしない。
- token / model / endpoint の古いログを現在状態として扱わない。
- provider 固有仕様を Application 層へ漏らさない。

完了条件:

- live `~/.rencrow/config.yaml` と実際の health / inference が一致している。
- Viewer LLM ops 表示と backend status の対応が説明できる。
- provider を差し替えても route / Viewer / logging の契約が保たれることを説明できる。

## Phase 7: Audio / STT / TTS 境界の整理

目的:

- Viewer mic input、STT provider、TTS provider、browser playback、audio router を追いやすくする。
- Go-first の runtime 実装と Viewer/browser 側の責務を分ける。

候補:

- STT/TTS runtime factory の責務確認。
- provider 実装と Viewer handler の分離確認。
- browser permission / HTTPS / SSE / playback の検証手順を固定する。
- STT/TTS provider interface と audio event 契約を整理する。

注意:

- IdleChat に voice input を入れない。
- Go テストだけでブラウザ録音・再生の完了扱いにしない。
- provider timeout と UI 表示不具合を混同しない。
- provider 差し替えで Viewer 表示契約や IdleChat 契約を変えない。

完了条件:

- STT/TTS の対象テストまたは実ブラウザ確認がある。
- ログと Viewer 表示の対応が説明できる。
- provider、transport、browser playback を個別に差し替えられる境界が説明できる。

## 各 Phase の共通完了条件

- 差分が小さい。
- 責務が正しい層にある。
- 仕様変更と構造変更が分かれている。
- モジュール境界の入力、出力、副作用、永続化、ログ、エラー契約が書かれている。
- 差し替え時の影響範囲が限定されている。
- テストまたは代替検証がある。
- Viewer が関係する場合は実表示を確認している。
- runtime config が関係する場合は live config を確認している。
- 残リスクが文書化されている。
