# フォルダ構成方針

## 目的

RenCrow のフォルダ構成を、Go の一般的な構成と Clean Architecture の責務境界に合わせて読みやすくする。

この文書は理想構成だけでなく、現実の既存コードを段階的に近づけるための判断基準を示す。

## 最優先の構成判断

フォルダ構成は、モジュール化と疎結合を実現するために決める。

機能単位で境界を作り、差し替えたい実装だけを入れ替えられる配置を優先する。配置を変えても、呼び出し側が実装詳細に依存しているままなら、構成整理としては不十分である。

各モジュールは次を説明できる状態にする。

- 入力。
- 出力。
- 副作用。
- 永続化の有無。
- ログ。
- エラー契約。
- 差し替え時に守る interface / contract / event / DTO / adapter。

共通 package は慎重に作る。複数機能から使われるという理由だけでまとめず、意味のある責務境界がある場合だけ抽出する。

## 配置判断

### `cmd/` に置くもの

- process 起動。
- CLI 引数処理。
- HTTP server 起動。
- runtime factory の組み立て。
- adapter / application / infrastructure の wiring。
- 開発用コマンドの entrypoint。

`cmd/` に置かないもの:

- route 判断の本体。
- Worker 実行の本体。
- LLM provider の詳細処理。
- Viewer の状態変換。
- DB state 遷移。

### `internal/domain/` に置くもの

- route、task、proposal、patch、session、memory などの値と契約。
- Chat / Worker / Coder の責務を表す interface / value object。
- 外部技術に依存しない validation。

`domain` に置かないもの:

- HTTP handler。
- DB 実装。
- LLM API 呼び出し。
- ファイルシステム実行。
- Viewer DOM や SSE の詳細。

### `internal/application/` に置くもの

- MessageOrchestrator。
- WorkerExecutionService。
- IdleChat orchestration。
- Source Registry の usecase。
- TTS/STT と Viewer をつなぐ coordination。
- policy、tool、report、event を組み合わせた usecase。

`application` に置かないもの:

- provider 固有の HTTP request。
- SQLite の schema 操作。
- Viewer の静的 asset。
- 純粋な domain value の定義。

### `internal/infrastructure/` に置くもの

- LLM provider 実装。
- SQLite / file persistence。
- security policy evaluator の実装。
- tool runner 実装。
- STT/TTS provider 実装。
- MCP / transport / VTuber など外部接続。

`infrastructure` に置かないもの:

- ユーザー入力から route を決める usecase。
- Viewer handler。
- domain contract の本体。

### `internal/adapter/` に置くもの

- Viewer HTTP handler / SSE / assets。
- LINE / Slack / Discord / Telegram handler。
- health endpoint。
- config 表示 / API。
- chrome bridge。

`adapter` に置かないもの:

- Worker execution の本体。
- LLM provider の本体。
- memory state 遷移の本体。
- route dispatch の中核。

## 現在の集中点

現行コード上、次は影響が集中している。

- `cmd/rencrow/main.go`
- `internal/application/orchestrator/message_orchestrator.go`
- `internal/application/service/worker_execution_service.go`
- `internal/infrastructure/tools/` と `internal/infrastructure/security/`
- `internal/infrastructure/persistence/conversation/l1_sqlite_store.go`
- `internal/adapter/viewer/` handlers / assets
- LLM provider factory / middleware

この集中点は、一度に動かさない。まず周辺の小さい責務を切り出し、テストで固定してから移動する。

## 命名方針

- package 名は短く、責務を表す。
- `manager`、`helper`、`util` は原則避ける。
- `service` は Application 層の usecase coordination に限定する。
- `common`、`shared`、`base` は、責務境界と契約を説明できる場合だけ使う。
- provider 実装は `infrastructure/<domain>/providers/` または既存の局所規約に合わせる。
- Viewer の JS/CSS は機能単位の tab / component に分ける。
- test helper は本体 package の責務を隠さない名前にする。

## 差し替え境界

差し替え可能性を意識して、次の境界を明示する。

| 領域 | 境界として固定するもの | 差し替え対象 |
|---|---|---|
| Chat / Worker / Coder | route、task、proposal、patch、response contract | agent 実装、provider、実行方式 |
| Viewer | HTTP API、SSE event、表示 state、DTO | JS tab、CSS、表示 component |
| IdleChat | timeline event、raw/view/audio trigger の契約 | 生成器、話題選択、要約、品質レビュー |
| LLM provider | provider interface、request/response DTO、raw log 契約 | Ollama、OpenAI互換、Claude など |
| STT / TTS | runtime factory、provider interface、audio event | provider、transport、browser playback |
| Memory / Source Registry | state transition、repository contract、promote 契約 | persistence、validator、sweeper |
| Tool / Security | tool request、policy result、execution record | tool runner、policy evaluator |

差し替え境界の外側は、内部実装の型や provider 固有の挙動へ依存しない。

## 移動単位

良い移動単位:

- import 変更が少ない。
- public API を増やさずに済む。
- 既存テストで確認できる。
- 1つの責務だけを移す。
- runtime wiring の変更が小さい。
- 入力、出力、副作用、エラー契約を説明できる。
- 差し替え時に影響する caller が限定されている。

悪い移動単位:

- `main.go` から大量に抜くが、移動先が新しい巨大ファイルになる。
- package 名だけ変えて責務が変わらない。
- 移動と仕様変更を同時に行う。
- Viewer 表示契約や Worker 安全境界を同時に変える。
- interface なしに別 package の内部型へ直接依存する。
- 似た処理をまとめただけで、機能境界が曖昧になる。

## archive と draft の扱い

- 旧 archive / old / codebase-map は削除済みであり、正本判断に使わない。
- draft や古い仕様を現在仕様に昇格する場合は、正本仕様または `docs/refactor/` に根拠を書く。
