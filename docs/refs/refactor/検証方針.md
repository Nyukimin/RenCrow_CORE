# 検証方針

## 基本原則

リファクタリングでは、テストが通ることだけで完了扱いしない。

構造変更でも、対象ユースケースが実際に成立することを確認する。特に Viewer、IdleChat、音声、LLM routing、Worker execution、runtime config は、コード上の整合だけでは判断しない。

## 実装前に決めること

各変更の前に、次を短く書く。

- 対象ユースケース。
- 変更する責務の層。
- モジュール境界。
- 入力。
- 出力。
- 副作用。
- 永続化の有無。
- ログ契約。
- エラー契約。
- 差し替え時に守る interface / contract / event / DTO / adapter。
- 触るファイル。
- 触らないファイル。
- 期待する挙動。
- 先に固定するテストまたは検証観点。
- 失敗時に見るログ。

## 差し替え可能性の検証

モジュール化と疎結合に関係する変更では、次を確認する。

- 呼び出し側が内部実装ではなく、interface / contract / event / DTO / adapter に依存している。
- 対象機能を差し替えるとき、変更が必要な package が限定されている。
- provider、persistence、Viewer component、tool runner などの実装詳細が Application / Domain へ漏れていない。
- 共通処理の抽出が、責務境界に基づいている。
- `service` / `manager` / `helper` / `util` が新しい依存集中点になっていない。
- 入力、出力、副作用、永続化、ログ、エラー契約がテストまたは文書で確認できる。

差し替え可能性の確認は、実際に全実装を差し替えることを必須とはしない。ただし、差し替え時にどこを変更すればよいか、どこへ影響させてはいけないかを説明できる状態にする。

## Go テスト

原則:

- 対象 package に最も近いテストを優先する。
- 既存テスト構造に合わせる。
- package 移動時は import 修正だけでなく、責務が変わっていないか確認する。
- sandbox では `GOCACHE=/tmp/...` を使う。

例:

```bash
GOCACHE=/tmp/rencrow-gocache go test ./internal/application/...
GOCACHE=/tmp/rencrow-gocache go test ./internal/domain/...
GOCACHE=/tmp/rencrow-gocache go test ./cmd/rencrow/...
```

広範囲に触った場合:

```bash
GOCACHE=/tmp/rencrow-gocache go test ./...
```

## Viewer 検証

Viewer に関係する変更では、最低 1 セッションを追う。

確認するもの:

- 表示本文。
- SSE event。
- event log。
- history。
- error / invalid response の表示。
- session の開始と終了状態。
- 音声や口パクが関係する場合は trigger と本文表示の分離。

DOM 要素が存在するだけでは完了にしない。表示内容と状態遷移を確認する。

## IdleChat 検証

IdleChat では次を分けて確認する。

- LLM raw response。
- 編集後の表示本文。
- diagnostic / test-mode 表示。
- TTS chunk。
- 口パク trigger。
- topic store / history / summary。
- fallback / invalid response。

fallback は成功扱いしない。空応答、invalid response、generation error は原因を隠さず記録する。

## Worker execution 検証

Worker execution に関係する変更では、次を確認する。

- Coder は plan / patch を生成するだけである。
- Worker が実行主体である。
- `PatchCommand` の parse / validation が壊れていない。
- file edit / shell / git operation の dispatch が追える。
- protected file pattern が効いている。
- workspace 外操作が拒否される。
- 実行ログに job_id、session_id、route、status、結果が残る。

安全境界の変更では、成功ケースより先に拒否ケースを確認する。

## Runtime config 検証

runtime config に関係する変更では、repo 内の example だけで判断しない。

確認するもの:

- `~/.rencrow/config.yaml`
- `http://127.0.0.1:18790/health`
- Viewer runtime config / LLM ops 表示
- 実際の inference endpoint
- raw log

古いログと現在の binary / config を混ぜない。ログを根拠にするときは、必要ならログをリセットして fresh run を取る。

## Audio / STT / TTS 検証

音声関連では、Go 側とブラウザ側を分けて確認する。

確認するもの:

- provider endpoint。
- runtime factory。
- Viewer HTTPS / browser permission。
- mic capture。
- trailing silence。
- SSE / audio route。
- browser playback。
- TTS と口パク trigger。

STT/TTS の失敗を Viewer 表示不具合として扱わない。逆に Viewer 表示不具合を provider failure と決めつけない。

## 再起動が必要な場合

RenCrow / RenCrow の再起動前には、既存作業を停止する。

確認順:

1. `systemctl --user stop rencrow.service`
2. 残存する `rencrow` process がないことを確認する。
3. `:18790` が listen されていないことを確認する。
4. `http://127.0.0.1:18790/health` が応答しないことを確認する。
5. build / install / restart を行う。
6. `/health` と対象フローを確認する。

## 完了報告に含めること

- 変更したファイル。
- 変更した責務。
- 実行したテスト。
- 実機確認の有無。
- 確認できなかったこと。
- 残るリスク。

「テストは通ったが Viewer は未確認」「live config は未確認」などは、そのまま明記する。
