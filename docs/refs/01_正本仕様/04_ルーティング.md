## 1. ルーティングカテゴリ

| Route | 用途 | 現行の処理先 |
|-------|------|------------|
| `CHAT` | 会話・通常応答・説明・雑談・レビュー | Mio |
| `PLAN` | 計画策定・仕様化・タスク分解 | Mio |
| `ANALYZE` | 分析・診断・集計・タグ付け | Mio |
| `OPS` | 運用操作・障害対応・起動/停止/再起動 | Shiro |
| `RESEARCH` | 深い調査（意図的に狭く設定） | Mio |
| `CODE` | 汎用コーディング（Coder 自動選択） | Mio → Shiro → Coder |
| `CODE1` | Coder1(AO/DeepSeek) 明示指定 | Mio → Shiro → Coder1 |
| `CODE2` | Coder2(Aka/OpenAI) 明示指定 | Mio → Shiro → Coder2 |
| `CODE3` | Coder3(Kin/Claude) 明示指定 / Chrome・ブラウザ操作 | Mio → Shiro → Coder3 |
| `CODE4` | Coder4(Gin/Gemini) 明示指定 | Mio → Shiro → Coder4 |

**Go 実装**（`internal/domain/routing/route.go`）:

```go
type Route string

const (
    RouteCHAT     Route = "CHAT"
    RoutePLAN     Route = "PLAN"
    RouteANALYZE  Route = "ANALYZE"
    RouteOPS      Route = "OPS"
    RouteRESEARCH Route = "RESEARCH"
    RouteCODE     Route = "CODE"
    RouteCODE1    Route = "CODE1"
    RouteCODE2    Route = "CODE2"
    RouteCODE3    Route = "CODE3"
    RouteCODE4    Route = "CODE4"
)

func (r Route) IsCoderRoute() bool {
    return r == RouteCODE || r == RouteCODE1 || r == RouteCODE2 || r == RouteCODE3 || r == RouteCODE4
}
```

## 2. ルーティング決定フロー（v6.1 現行実装）

`message_orchestrator.go` と `mio.go` の `DecideAction()` で以下の順序で決定される。

```
1. ChatCommand 処理（/status, /stop, /compact, /context, /new）
   └─ Handled=true → route 決定に進まない

2. parseExplicitCommand()  ← confidence=1.0 固定
   明示コマンドがメッセージ先頭に存在する場合

3. RuleDictionary.Match()  ← confidence=0.85 基本
   静的パターンマッチング

4. isCodeEditRequest()  ← confidence=0.9
   ファイル参照 + 編集動詞のある依頼は CODE に寄せる

5. CHAT フォールバック  ← confidence=0.7
   いずれにも当たらない場合
```

**注意**: 現行実装（v6.1）は LLM 分類器を使わず RuleDictionary のみ。LLM 分類器は将来改善候補として設計済みだが現時点では未実装。

## 3. 明示コマンド（優先度 1）

メッセージ先頭に以下があれば confidence=1.0 で固定:

| コマンド | Route |
|---------|-------|
| `/chat` | CHAT |
| `/plan` | PLAN |
| `/analyze` | ANALYZE |
| `/ops` | OPS |
| `/research` | RESEARCH |
| `/code` | CODE |
| `/code1` | CODE1 |
| `/code2` | CODE2 |
| `/code3` | CODE3 |

**重要**: 入力本文内の「/code と書いて」は無効。先頭コマンドのみ有効。

`/local`, `/cloud` は特別コマンド:
- `/local` — クラウド利用禁止フラグを立てる（`local_only=true`）
- `/cloud` — `/local` を解除する
- `local_only=true` 中に `/code` が来た場合は CODE 実行を拒否し、`/cloud` で解除を案内

## 4. RuleDictionary 主要パターン（優先度 2）

| Route | 判定語（代表） |
|-------|-------------|
| `CODE` | `実装して`, `修正して`, `変更を入れて`, `追記して`, `編集して`, `更新してください`, `ファイルを更新/変更`, `json ファイル`, `text フィールドを` |
| `PLAN` | `計画`, `プラン`, `設計して`, `アーキテクチャ` |
| `ANALYZE` | `分析して`, `調査して`, `解析して`, `診断して`, `レビューして` |
| `OPS` | `実行して`, `起動して`, `デプロイ`, `ビルドして`, `停止して`, `再起動` |
| `RESEARCH` | `リサーチ`, `情報を集`, `ドキュメントを探` |
| `CODE3` | `chrome`, `ブラウザ`, `画面操作`, `スクレイピング`, `ページを開`, `webを操作` |

**注意**:
- `調べて`, `検索して` は RESEARCH に含めない（現行では CHAT 即答または Web 検索に寄せる）
- RESEARCH は intentionally 狭く設定されている

## 5. ファイル編集依頼の CODE 優先（isCodeEditRequest）

以下を両方満たす場合は confidence=0.9 で CODE に寄せる:

- **ファイル参照がある**（例: `README.md`, `viewer.html`, `foo.json`, `bar.go`）
- **編集動詞がある**（例: `変更`, `修正`, `追記`, `編集`, `更新`, `実装`, `追加`, `削除`, `書き換`, `直して`）

対応拡張子: `.go`, `.py`, `.js`, `.ts`, `.tsx`, `.jsx`, `.json`, `.yaml`, `.yml`, `.md`, `.html`, `.css`, `.sh`, `.txt`

**例**: `README.md に1行だけ変更を入れて。具体的には末尾に確認用コメントを1行追記してください。` → CODE と判定

## 6. ルーティング Decision 型

```go
// Decision はルーティング決定の結果（不変値オブジェクト）
type Decision struct {
    Route      Route   // 決定されたルート
    Confidence float64 // 確信度（0.0〜1.0）
    Reason     string  // 決定理由（文字列のみ）
}
```

現行の Reason 文字列:
- `"Explicit command"` — 明示コマンド
- `"Rule dictionary match"` — RuleDictionary マッチ
- `"No rule match, default to CHAT"` — フォールバック

Confidence の現行値:
- 明示コマンド: `1.0`
- `isCodeEditRequest()`: `0.9`
- RuleDictionary 通常マッチ: `0.85`
- CHAT fallback: `0.7`

## 7. ループ制御と再ルート

### 7.1 ループ制限

- **max_loops**: 最大ループ回数（デフォルト: 5）
- **max_millis**: 最大実行時間（デフォルト: 120,000ms = 2分）

終了条件:
1. Worker 出力の `needs_next_loop: false`
2. max_loops 到達
3. max_millis 到達
4. エラー発生

### 7.2 自動再ルート（最大 1 回）

Worker 出力の `suggested_route` に基づき、最大 1 回のみ自動再ルート可能。

発火条件（全て満たす場合）:
- `suggested_route` が `initial_route` と異なる
- `confidence >= 0.8`
- `fit < 0.3`（現在のルートに不適合）
- まだ再ルートしていない

**実装**: `internal/application/service/llm_iterator_service.go`

## 8. ワーカー出力契約

Worker（Shiro）がルーティング後に返す結果のメタ情報:

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `result` | string | 作業成果 |
| `needs_next_loop` | bool | 追加ループの必要性 |
| `why` | string | 次が必要な理由 |
| `next_actions` | []string | 次にやるべき候補（最大3） |
| `confidence` | float | 成果の確からしさ (0.0〜1.0) |
| `risk` | enum | `low` / `medium` / `high` |
| `fit` | bool | （任意）このルートが適切だったか |
| `suggested_route` | string | （任意）再ルート提案 |

ユーザー確認が発生する条件: `risk==high` / 信頼度低+重大分岐 / 追加情報が必須

| 入力 | Route | 根拠 |

## 10. 既知の制約と将来改善

**現行の制約**:
- 静的キーワード方式なので、意図が強くても語彙がずれると CHAT に落ちうる
- route 判定理由は文字列のみ（構造化 evidence は未実装）

**将来改善候補**（未実装）:
- file path / patch / command / stacktrace などの evidence ベース判定
- `reason` だけでなく `evidence[]` を返す構造化 route 判定
- LLM 分類器（confidence スコア付き）の追加
- CODE と OPS の境界をより厳密に分ける
