## 1. アーキテクチャ設計

### 1.1 Clean Architecture 4層構造

```
┌──────────────────────────────────────┐
│  Adapter Layer (外部インターフェース)  │ ← LINE, Slack, CLI Webhook
│  - LINE Handler (CHAT固定)           │
│  - Slack/CLI Handler                │
│  - Notification Sender              │
└──────────────────────────────────────┘
           ↓ 依存（Use Case呼び出し）
┌──────────────────────────────────────┐
│ Application Layer (Use Cases)       │
│  - ProcessUserMessage               │
│  - WorkerExecutionService           │
│  - MessageOrchestrator (分解済み)    │
│  - Services (Routing, Worker実行等) │
└──────────────────────────────────────┘
           ↓ 依存（Domain呼び出し）
┌──────────────────────────────────────┐
│ Domain Layer (ビジネスロジック)       │
│  - MioAgent/ShiroAgent/CoderAgent   │
│  - PatchCommand (Value Object)     │
│  - Task, Proposal (Value Object)   │
└──────────────────────────────────────┘
           ↑ 依存性逆転（interface）
┌──────────────────────────────────────┐
│ Infrastructure Layer (技術詳細)      │
│  - LLM Providers (Ollama/Claude等)  │
│  - ToolRunner (ファイル編集/コマンド)│
│  - Config Loader                   │
│  - Logger                          │
└──────────────────────────────────────┘
```

**依存方向**: 外層 → 内層（Domain層は何にも依存しない）

### 1.2 セーフガード設計

即時実行でも、以下のセーフガードで安全性を確保:

**1. 詳細ログ記録**（必須）:
- 実行前: plan/patch 全文、実行サマリ（コマンド数、種別）
- 実行中: 各コマンドの stdout/stderr
- 実行後: 実行結果、変更ファイルリスト

**2. Git auto-commit**（推奨、有効化時は必須）:
- 実行前後に自動コミット
- `git revert` で即座に巻き戻し可能
- コミットメッセージ: `[Worker Auto-Commit] Before/After patch: <summary>`
- **重要**: `AutoCommit=true` の場合、コミット失敗 = 実行中断（安全性担保）

**3. 保護ファイルパターン**（推奨）:
- 機密ファイルの誤削除・誤編集を防止
- デフォルト保護パターン: `.env*`, `*credentials*`, `*.key`, `*.pem`
- 保護ファイルへの操作は即座にエラー

**4. ドライモード**（オプション）:
- `RENCROW_DRY_RUN=true` で有効化
- 実行せず、実行内容のみログ出力
- 破壊的操作の事前確認に有用

**5. ワークスペース制限**（必須）:
- ファイル操作は workspace 内のみ許可
- workspace 外への書き込みはエラー

**6. 実行前サマリ表示**（推奨）:
- 実行前に変更内容の概要をログ出力
- コマンド総数、種別（file_edit/shell_command/git_operation）
- トラブルシューティングの効率化

### 1.3 DDD（Domain-Driven Design）

| パターン | 実装例 | 役割 |
|---------|--------|------|
| **エンティティ** | MioAgent, ShiroAgent, CoderAgent | 一意性を持つドメインオブジェクト |
| **値オブジェクト** | Task, JobID, Proposal, RoutingDecision, PatchCommand | 不変、値の等価性で比較 |
| **サービス** | ToolRunner, WorkerExecutionService | ドメインロジックの調整 |

### 1.4 Wire依存性注入

```go
// cmd/rencrow/wire.go
//go:build wireinject
// +build wireinject

func InitializeApp(cfg *config.Config) (*App, error) {
    wire.Build(
        // Infrastructure
        provideLLMProviders,
        provideToolRunner,
        provideLogger,

        // Domain Entities

        domain.NewShiroAgent,
        domain.NewCoderAgent,

        // Application Services

        service.NewWorkerExecutionService,

        // Use Cases
        usecase.NewProcessMessage,

        // Adapters
        adapter.NewLINEHandler,

        // App
        wire.Struct(new(App), "*"),
    )
    return nil, nil
}
```

## 3. スコープ・責務境界

### 3.1 対象プラットフォーム

- **LINE**: メインチャネル（CHAT固定）
- **Slack**: サブチャネル（全ルート対応）
- **CLI**: 開発・デバッグ用（全ルート対応）

### 3.2 役割固定（Chat/Worker/Coder）

| 役割 | 愛称 | 実体LLM（デフォルト） | 責務 |
|------|------|---------------------|------|
| **Chat** | Mio（澪） | Ollama Chat | 意思決定・委譲判断・ユーザー対話 |
| **Worker** | Shiro（白） | Ollama Worker | 実行・道具係・ツール呼び出し |
| **Coder1** | AO（青） | DeepSeek API | 仕様設計向けコーディング |
| **Coder2** | Aka（赤） | OpenAI API | 実装向けコーディング |
| **Coder3** | Kin（金） | Anthropic Claude API | 高品質コーディング・推論 |
| **Coder4** | Gin（銀） | Gemini API | 高速プロトタイピング・実験 |

**重要**: 役割名は固定、実体LLMは設定で差し替え可能

### 3.3 責務の明確な分離

#### Chat（Mio）の責務

1. **ルーティング決定**（Mio委譲判断）
   - 明示コマンド検出（/code, /analyze等）
   - ルール辞書マッチング
   - 分類器呼び出し（Worker経由）
   - 安全側フォールバック（CHAT）

2. **ユーザー対話**
   - 最終回答生成
   - エラー通知
   - 進捗報告

3. **実行結果の統合**
   - Worker実行結果の受信
   - Coder生成結果の整形
   - 複数ルートの結果統合

#### Worker（Shiro）の責務

1. **ツール実行**
   - ファイル編集
   - コマンド実行
   - テスト実行

2. **差分適用**
   - Coder生成のpatch適用
   - 実行結果の報告

3. **分類補助**
   - ルーティング分類器としての判定

#### Coder（Aka/Ao/Gin）の責務

1. **設計・実装案作成**
   - plan生成
   - patch生成（unified diff形式）
   - risk評価
   - cost_hint提供

2. **コード生成**
   - 新規ファイル作成案
   - 既存ファイル変更案

**重要**: Coderは原則として破壊的操作を**直接実行せず**、plan と patch を生成。実行はWorkerが**即座に**担当。

### 3.4 LINE入口の特殊仕様

**仕様**: LINE入口は必ずCHAT（Mio）に固定（`line_forced_chat`）

**理由**:
- LINE APIの制約（Webhook応答時間制限）
- ユーザー体験の一貫性
- Worker即時実行の確実な動作

**実装**: `internal/adapter/inbound/line/handler.go`

```go
    }
    // ...
}
```

### 3.5 エージェントペルソナ仕様（v4.2）

> ペルソナ設計の思想・6層構造については `docs/03_設計文書/仕様_Charペルソナ.md` を参照。

#### ペルソナファイル配置

各エージェントのペルソナは `{workspace_dir}/persona/` 以下の個別ファイルで管理する。

| ファイル | エージェント | 用途 |
|----------|------------|------|
| `persona/mio.md` | Mio（Chat） | 6層ペルソナ。`FilePersonaEditor` で LLM 自己編集可 |
| `persona/shiro.md` | Shiro（Worker） | 6層ペルソナ |
| `persona/aka.md` | Coder1（赤） | 役割・専門性の記述（Coder は軽量でよい） |
| `persona/ao.md` | Coder2（青） | 同上 |
| `persona/gin.md` | Coder3（銀） | 同上 |
| `persona/kin.md` | Coder4（金） | 同上 |

**Chat からすべてのファイルを読み書き可能**（`AllowedWritePaths` に全パスを登録）。

#### config.yaml スキーマ

```yaml
# Mio ペルソナファイル（workspace_dir からの相対パス）
mio_persona_file: "persona/mio.md"

worker:
  persona_file: "persona/shiro.md"   # workspace_dir からの相対パス
  tone: "丁寧で正確"                  # TTS 連携用

coder1:
  persona_file: "persona/ao.md"
  tone: "丁寧で論理的"
  # personality: は廃止方向（persona_file がない場合のフォールバック）
```

#### ペルソナ読み込み優先順位

起動時 `setupCoders()` / Mio・Shiro 配線でそれぞれ以下の順で解決する:

1. `persona_file` が設定されている → ファイルを読み込み `AgentPersona.Personality` に設定
2. `persona_file` が空で `personality` インラインが設定されている → インライン文字列を使用
3. いずれもなし → ペルソナ無効（`AgentPersona` を注入しない、後方互換）

#### Chat 経由のペルソナ編集フロー

```
ユーザー: 「Shiro のペルソナを〜に変えて」
  ↓
Mio: detectPersonaEditIntent() でペルソナ編集意図を検出
  ↓
editPersona() が FilePersonaEditor.ReadPersona() で現在のペルソナを読み取り
  ↓
LLM がユーザー指示に従ってペルソナを書き換え
  ↓
FilePersonaEditor.WritePersona() で対象ファイルに書き込み
  ↓ (次回起動 / 再読み込み時に反映)
```

**注意**: ペルソナ変更はファイルに永続化されるが、実行中エージェントへの反映は次回起動時。

```go
import (
    "context"
    "github.com/sipeed/rencrow/internal/domain/routing"
    "github.com/sipeed/rencrow/internal/domain/task"

// MioAgent は Chat 役割（Mio/澪）のエンティティ
type MioAgent struct {
    role            Role
    ruleDictionary  *RuleDictionary
    classifier      Classifier // Worker呼び出し経由
}

func NewMioAgent(ruleDictionary *RuleDictionary, classifier Classifier) *MioAgent {
    return &MioAgent{
        role:           RoleChat,
        ruleDictionary: ruleDictionary,
        classifier:     classifier,
    }
}

// DecideAction は Mio委譲判断ロジック（仕様準拠: 前段Routerではなく会話中に判断）
func (m *MioAgent) DecideAction(ctx context.Context, t task.Task) (routing.Decision, error) {
    // 1. 明示コマンド優先（/code, /analyze, /plan等）
    if cmd := parseExplicitCommand(t.UserMessage()); cmd != "" {
        return routing.Decision{
            Route:      routing.Route(cmd),
            Source:     "explicit",
            Confidence: 1.0,
        }, nil
    }

    // 2. ルール辞書マッチング（強証拠）
    if route := m.ruleDictionary.Match(t); route != routing.RouteUnknown {
        return routing.Decision{
            Route:      route,
            Source:     "rule",
            Confidence: 0.95,
        }, nil
    }

    // 3. 分類器に問い合わせ（Worker呼び出し）
    decision, err := m.classifier.Classify(ctx, t)
    if err != nil || decision.Confidence < 0.7 {
        // 4. 安全側フォールバック（CHAT）
        return routing.Decision{
            Route:      routing.RouteCHAT,
            Source:     "fallback",
            Confidence: 0.5,
        }, nil
    }

    return decision, nil
}

// Role returns the agent's role
func (m *MioAgent) Role() Role {
    return m.role
}

// parseExplicitCommand は明示コマンドを解析（/code, /analyze等）
func parseExplicitCommand(message string) string {
    // 明示コマンドマッピング
    commands := map[string]string{
        "/chat":    "CHAT",
        "/plan":    "PLAN",
        "/analyze": "ANALYZE",
        "/ops":     "OPS",
        "/research": "RESEARCH",
        "/code":    "CODE",
        "/code1":   "CODE1",
        "/code2":   "CODE2",
        "/code3":   "CODE3",
    }

    // メッセージの先頭から明示コマンドを検出
    for cmd, route := range commands {
        if strings.HasPrefix(strings.TrimSpace(message), cmd) {
            return route
        }
    }

    return "" // 明示コマンドなし
}
```

```go
    // レスポンスから「## Plan」セクションを抽出
    content := response.Content
    planStart := strings.Index(content, "## Plan")
    if planStart == -1 {
        return ""
    }

    // 次のセクション（## Patch等）までを抽出
    planEnd := strings.Index(content[planStart+7:], "##")
    if planEnd == -1 {
        return strings.TrimSpace(content[planStart+7:])

    return strings.TrimSpace(content[planStart+7 : planStart+7+planEnd])

    // レスポンスから「## Risk」セクションを抽出
    content := response.Content
    riskStart := strings.Index(content, "## Risk")
    if riskStart == -1 {
        return proposal.RiskLevelMedium

    riskSection := content[riskStart : riskStart+50] // 最初の50文字を検査

    // Risk levelを判定
    if strings.Contains(strings.ToLower(riskSection), "high") {
        return proposal.RiskLevelHigh
    } else if strings.Contains(strings.ToLower(riskSection), "low") {
        return proposal.RiskLevelLow
    }

    return proposal.RiskLevelMedium
}
```

```go
// RuleDictionary はルーティングルール辞書
type RuleDictionary struct {

type RoutingRule struct {

    Route      routing.Route  // ルーティング先
    Confidence float64        // 確信度（0.0-1.0）
}

func NewRuleDictionary(rules []RoutingRule) *RuleDictionary {

}

// Match はタスクメッセージとルールをマッチング
func (rd *RuleDictionary) Match(t task.Task) routing.Route {
    message := t.UserMessage()

    for _, rule := range rd.rules {
        matched, _ := regexp.MatchString(rule.Pattern, message)
        if matched {
            return rule.Route
        }
    }

    return routing.RouteUnknown
}
```

### 4.11 Patch解析関数（Domain層）

**ファイル**: `internal/domain/patch/parser.go`

**重要**: parsePatch関連関数はDomain層の純粋なビジネスロジック。Application層のWorkerExecutionServiceから呼び出される。

```go
package patch

import (
    "encoding/json"
    "fmt"
    "regexp"
    "sort"
    "strings"
)

// ParsePatch は JSON または Markdown 形式の patch を解析
func ParsePatch(patchStr string) ([]PatchCommand, error) {
    // JSON 形式判定（先頭が '['）
    trimmed := strings.TrimSpace(patchStr)
    if len(trimmed) > 0 && trimmed[0] == '[' {
        return parseJSONPatch(patchStr)
    }

    // Markdown 形式判定（コードブロック検出）
    if strings.Contains(patchStr, "```go:") || strings.Contains(patchStr, "```bash") {
        return parseMarkdownPatch(patchStr)
    }

    return nil, fmt.Errorf("unknown patch format")
}

// parseJSONPatch は JSON 配列形式の patch を解析
func parseJSONPatch(patchStr string) ([]PatchCommand, error) {
    var commands []PatchCommand

    if err := json.Unmarshal([]byte(patchStr), &commands); err != nil {
        return nil, fmt.Errorf("JSON parse error: %w", err)
    }

    // バリデーション
    for i, cmd := range commands {
        if cmd.Type == "" {
            return nil, fmt.Errorf("command[%d]: type is required", i)
        }
        if cmd.Action == "" {
            return nil, fmt.Errorf("command[%d]: action is required", i)
        }
        if cmd.Target == "" {
            return nil, fmt.Errorf("command[%d]: target is required", i)
        }
    }

    return commands, nil
}

// parseMarkdownPatch は Markdown コードブロック形式の patch を解析
func parseMarkdownPatch(patchStr string) ([]PatchCommand, error) {
    // 位置情報付きコマンド
    type positionedCommand struct {
        pos int
        cmd PatchCommand
    }
    var positioned []positionedCommand

    // 正規表現: ```言語:ファイルパス\n内容\n```
    // (?s) フラグ: DOTALL モード（.が改行にもマッチ）
    reCodeBlock := regexp.MustCompile("(?s)```([a-z]+):([^\n]+)\n(.*?)```")
    matches := reCodeBlock.FindAllStringSubmatchIndex(patchStr, -1)

    for _, match := range matches {
        lang := patchStr[match[2]:match[3]]
        filePath := strings.TrimSpace(patchStr[match[4]:match[5]])
        content := patchStr[match[6]:match[7]]

        cmd := PatchCommand{
            Type:    TypeFileEdit,
            Action:  ActionUpdate,
            Target:  filePath,
            Content: content,
            Metadata: map[string]string{
                "language": lang,
            },
        }
        positioned = append(positioned, positionedCommand{pos: match[0], cmd: cmd})
    }

    // 正規表現: ```bash\nコマンド\n```
    // (?s) フラグ: DOTALL モード（複数行コマンド対応）
    reShellBlock := regexp.MustCompile("(?s)```bash\n(.*?)```")
    shellMatches := reShellBlock.FindAllStringSubmatchIndex(patchStr, -1)

    for _, match := range shellMatches {
        command := strings.TrimSpace(patchStr[match[2]:match[3]])

        cmd := PatchCommand{
            Type:   TypeShellCommand,
            Action: ActionRun,
            Target: command,
        }
        positioned = append(positioned, positionedCommand{pos: match[0], cmd: cmd})
    }

    if len(positioned) == 0 {
        return nil, fmt.Errorf("no valid code blocks found in Markdown patch")
    }

    // 出現順にソート
    sort.Slice(positioned, func(i, j int) bool {
        return positioned[i].pos < positioned[j].pos
    })

    // コマンドリストに変換
    commands := make([]PatchCommand, len(positioned))
    for i, p := range positioned {
        commands[i] = p.cmd
    }

    return commands, nil
}
```

## 5. ルーティング決定仕様

### 5.1 ルーティングカテゴリ

| カテゴリ | 説明 | 担当Agent | 用途例 |
|---------|------|----------|--------|
| `CHAT` | 会話・意思決定 | Mio（Chat） | 雑談、質問応答、対話管理 |
| `PLAN` | 計画策定 | Mio（Chat） | プロジェクト計画、タスク分解 |
| `ANALYZE` | 分析 | Shiro（Worker） | ログ分析、性能分析 |
| `OPS` | 運用操作 | Shiro（Worker） | デプロイ、再起動、設定変更 |
| `RESEARCH` | 調査 | Shiro（Worker） | 文献調査、API仕様調査 |
| `CODE` | コーディング（汎用） | Aka/Ao/Gin（Coder） | 一般的なコーディング |
| `CODE1` | 仕様設計向け | Aka（Coder1） | アーキテクチャ設計、仕様策定 |
| `CODE2` | 実装向け | Ao（Coder2） | 実装、テストコード作成 |
| `CODE3` | 高品質コーディング/推論 | Gin（Coder3） | 難解な実装、最適化、高度な推論 |

**会話系検索ポリシー**:
- 通常の `CHAT` では、会話記憶と UserMemory を優先する。
- 外部検索ツール（`web_search` / Google Custom Search API）は quota を消費するため、`検索して`、`調べて`、`調査して`、`Webで調べて` などユーザーが明示的に外部検索を求めた場合だけ利用する。
- `最新`、`今日`、`ニュース`、`Xについて教えて`、`Xとは` だけでは Google API を叩かない。必要なら Mio は「リアルタイム確認が必要なら検索指示をください」と返す。
- `idleChat` では外部検索（Web/API検索）を禁止し、既存の内部文脈のみで会話する。
- `CHAT` / `idleChat` ともに内部データベース（ConversationEngine の Recall/KB 検索）は利用可能。ただし、UserMemory 確認や通常会話を外部Web検索結果と混同してはいけない。

**検索システムDB確定方針**:
RenCrow の検索・外部情報収集・記憶昇格に必要なDBは、検索候補を扱う hot store、外部世界の関係事実を扱う Domain Graph DB、意味検索用 VectorDB、任意の archive 境界に分ける。

通常の検索システムとしては L1 SQLite と Qdrant を基本必須境界とする。Movie、漫画、音楽、小説、ゲーム、人物、組織、キャラクター、シリーズ、受賞歴など、外部世界の関連性そのものを扱うドメインでは Domain Graph DB を追加の正本DBとして必須にする。

| DB | 必須度 | 役割 |
| --- | --- | --- |
| L1 SQLite | 必須 | search cache、fetch cache、Source Registry、staging、validation 状態、promotion 履歴、event log、軽い knowledge の正本 |
| Domain Graph DB | ドメイン採用時必須 | Movie / 漫画 / 音楽 / 小説 / ゲームなどの作品、人物、組織、キャラクター、シリーズ、関係 edge、出典 assertion の正本 |
| Qdrant | 必須寄り | validated / promoted 済み KB のベクトル検索 |
| DuckDB | 任意 | 長期 archive、集計、履歴分析、Parquet export |

L1 SQLite は検索システムの hot store / 正本DBである。検索候補、外部取得結果、RSS/Atom/sitemap 由来情報、Google Custom Search 結果、Web Gather 結果は、まず L1 SQLite の cache または staging に保存する。無審査で UserMemory、正式 Knowledge、Qdrant へ直書きしてはいけない。

Domain Graph DB は、外部世界に存在する作品・人物・組織・キャラクター・シリーズ・ジャンル・関連作品などの関係を保持する正本DBである。これは search cache、staging、汎用 KB、Qdrant、DuckDB の代替ではない。外部取得結果は、L1 SQLite の staging で validation した後、関係事実として採用するものだけを Domain Graph DB へ promote する。

Domain Graph DB は、最初から専用 graph DB 製品を必須にしない。現行の Movie ドメインのように、SQLite の relational table で `movies`、`people`、`movie_people`、`movie_watch_events`、`movie_title_observations`、`movie_preference_signals`、`movie_topic_candidates` を持つ形で開始してよい。漫画など新しいドメインでも、`works`、`people`、`characters`、`organizations`、`series`、`genres`、`relationships`、`sources`、`assertions` 相当の境界を保つ。

Domain Graph DB では、外部カタログ事実とユーザー固有状態を混ぜない。例として、映画作品の `movies` に `seen` を追加したり、人物カタログの `people` に `favorite` を追加したりしてはいけない。「見た」「読んだ」「好き」「苦手」「話題にしたい」は user event / preference signal として別テーブルへ保存する。

Domain Graph DB の assertion には、source URL、source ID、取得/観測時刻、confidence、validation status、evidence を残す。矛盾する事実は無言で overwrite せず、assertion と validation 状態で扱う。

Qdrant へ入れるのは、validated / promoted 済みで、意味検索する価値がある KB のみとする。Google検索結果、RSS取得結果、Webページ本文、Source Registry staging を Qdrant へ直接 upsert してはいけない。

Domain Graph DB から Qdrant へ同期する場合は、作品・人物・関係の要約や説明など、意味検索に使う文書表現だけを同期する。関係 edge の正本は Domain Graph DB に残す。

DuckDB は運用必須ではない。古い staging、古い event、古い search cache、日次 digest、分析用 archive を扱う cold store として後段で使う。

SearXNG、YaCy、Google Custom Search、RSS/Atom、sitemap は DB ではなく discovery provider / source である。Source Registry は独立DBではなく、L1 SQLite 内の管理テーブルである。

Movie Graph 仕様（`docs/10_新仕様/49_Movie_Graph_Mio_Topic仕様.md`）と Movie Watch Event 仕様（`docs/10_新仕様/51_Movie_Watch_Event実装仕様.md`）は、この Domain Graph DB 方針に基づく Movie ドメインの詳細仕様として扱う。漫画など別ドメインを追加する場合も、この境界を守る。

Domain Graph DB の経路実装は `docs/10_新仕様/52_Domain_Graph_DB経路実装仕様.md` を正とする。

**IdleChat会話生成ポリシー**:
- 話題出しは必ず `chat (mio)` が行う（会話開始者も `chat`）。
- 話題カテゴリ比率:
  - ユーザー関係 30%
  - 時事 20%
  - テック 20%
  - worker作成DB/履歴 20%
  - 完全ランダム 10%
- `chat` と `worker` は話題に積極的な関心を示し、深く考察する。
- Mio の通常発話では、必要に応じて別候補を1つ生成し、「楽しさ」を0〜100%で評価して高い候補を会話本文として採用する。
- Shiro の通常発話では、追加の楽しさ候補生成を行わず、短めの max token と Shiro 向け検証で本文を確定する。
- 会話がループし始めた場合は一度停止し、`worker (shiro)` が要約保存後、新しい話題で再開する。
- セッション終端時は `worker` が最終要約を作成し、`何月何日の何の話題まとめ` 形式で保存する。

**Viewer IdleChatモード制御**:
- Timeline に `IdleChat開始` / `IdleChat停止` を設置する。
- `IdleChatモード` は停止ボタン、ユーザー問いかけ、他エージェント通知で停止する。
- LLM Ops が有効な場合、IdleChat 開始前に `Heavy`（Kuro）/ `Wild`（Midori）が稼働中でないことを確認する。稼働中なら開始を 409 で拒否する。停止中なら `Heavy` / `Wild` を停止状態へ揃えたうえで `Worker`（Shiro）を起動し、IdleChat を開始する。Shiro は IdleChat 起動時の準備用途に限定し、Kuro / Midori の作業中には IdleChat を割り込ませない。
- Local OpenAI-compatible LLM の通常 `/health` は多段 health とする。`/v1/models` または管理 daemon の短い liveness/readiness で API と alias/model が生きているかを判定し、重い chat completion は通常 `/health` の `down` 判定に使わない。推論 probe は diagnostics / 詳細検査として扱い、timeout 時は `down` ではなく `degraded` または `probing_timeout` として表現する。API 自体が unreachable、HTTP 5xx、alias/model 未設定は `down`、API は生きているが対象 alias が一覧にない場合は `degraded` とする。
- Viewer に `IdleChat` タブを設置し、現在トピックと要約履歴を参照可能にする。
- `idleChat` 会話イベント（`idlechat.message`, `idlechat.summary`）は通常 Timeline にも表示する。
- `IdleChat` タブでは要約タイトル（topic/まとめ）クリックで transcript（会話履歴）を表示する。
- API:
  - `POST /viewer/idlechat/start`
  - `POST /viewer/idlechat/stop`
  - `GET /viewer/idlechat/status`
  - `GET /viewer/idlechat/logs`

### 5.2 Mio委譲判断の4段階優先順位

**仕様準拠**: 前段Routerではなく、Mioが会話中に委譲判断を行う

#### 優先度1: 明示コマンド（最優先）

ユーザーが明示的にコマンドを指定した場合、それを優先。

**明示コマンド一覧**:
- `/chat` → CHAT
- `/plan` → PLAN
- `/analyze` → ANALYZE
- `/ops` → OPS
- `/research` → RESEARCH
- `/code` → CODE（デフォルトCoder）
- `/code1` → CODE1（Aka）
- `/code2` → CODE2（Ao）
- `/code3` → CODE3（Gin）

**実装**: `MioAgent.DecideAction()`の最初でチェック

#### 優先度2: ルール辞書（強証拠）

明示コマンドがない場合、ルール辞書でマッチング。

**ルール辞書例**:
```yaml
rules:
  - pattern: "ファイルを作成"
    route: CODE
    confidence: 0.95

  - pattern: "ログを確認"
    route: ANALYZE
    confidence: 0.9

  - pattern: "状態|status|確認"
    route: CHAT
    confidence: 1.0
```

#### 優先度3: 分類器（LLM判定）

ルール辞書でマッチしない場合、Worker（Shiro）を分類器として呼び出し。

**分類器プロンプト**（Worker）:
```
以下のメッセージを分類してください:
"{user_message}"

以下のカテゴリから最も適切なものを選んでください:
- CHAT: 会話・質問応答
- CODE: コーディング
- ANALYZE: 分析
- OPS: 運用操作
- RESEARCH: 調査

JSON形式で回答してください:
{
  "route": "CHAT",
  "confidence": 0.85,
  "reason": "..."
}
```

#### 優先度4: 安全側フォールバック

分類器が失敗またはConfidence < 0.7の場合、安全側にCHATへフォールバック。

**実装**: `MioAgent.DecideAction()`の最後

## 6. ループ制御と再ルート

### 8.3 ReActループ統合（SubagentManager）

`ShiroAgent.Execute()` は OPS ルート処理で ReActLoop を使い、ツールを自律的に選択・実行する。

**動作モード**:

| 条件 | 動作 |
|---|---|
| `cfg.Subagent.Enabled = true` | `SubagentManager.RunSync()` 経由で ReActLoop 実行 |
| `cfg.Subagent.Enabled = false` | `llmProvider.Generate()` による単発 LLM 呼び出し（フォールバック） |

**ReActループのフロー**:
```
ShiroAgent.Execute(ctx, Task)
└── SubagentManager.RunSync(ctx, SubagentTask)
    └── toolloop.Run(ctx, provider, toolRunner, toolDefs, messages, cfg)
        ├── provider.Chat(messages + tools) → assistant response
        ├── tool_calls あり → toolRunner.ExecuteV2() 実行 → role="tool" メッセージ追加 → 繰り返し
        └── tool_calls なし → Content を返して終了
```

**終了条件**: tool_calls なし / `ctx.Done()` / `MaxIterations` 到達（デフォルト 10）

**設定**（`config.yaml`の `subagent` セクション）:
```yaml
subagent:
  enabled: true
  max_iterations: 10
  provider: ""   # 空の場合 worker provider を使用
  model: ""
```

詳細仕様: `docs/実装仕様_サブエージェント_v1.md`

## 9. Worker即時実行仕様

### 9.1 基本フロー

```
Coder3 が plan/patch を生成
  ↓
Worker が即座に patch を解析
  ↓
Worker が即座に patch を適用（即時）
  ↓

```

**特徴**:
- ✅ 完全自動（ユーザー操作不要）
- ✅ 高速（待機なし）
- ⚠️ リスク高（破壊的操作も即座に実行）→ セーフガードで対応

### 9.3 Worker実行ロジック

#### executeWorkerPatch() フロー

1. **patch解析**（parsePatch）
   - JSON または Markdown 形式を自動判定
   - PatchCommand リストに変換

2. **実行前サマリ表示**（ShowExecutionSummary=trueの場合）
   - コマンド総数、種別（file_edit/shell_command/git_operation）をログ出力
   - トラブルシューティングの効率化

3. **実行前 Git コミット**（AutoCommit=trueの場合）
   - `git add -A`
   - `git commit -m "[Worker Auto-Commit] Before patch"`
   - **重要**: コミット失敗時は即座にエラーを返し、実行中断（安全性担保）

4. **patch適用**（executeCommand × N）
   - **保護ファイルチェック**: 保護パターンに一致するファイルはエラー
   - **ワークスペースチェック**: workspace外のファイルはエラー
   - ファイル編集: create, update, delete, append
   - シェルコマンド: bash実行（タイムアウト5分）
   - Git操作: add, commit, reset, checkout

5. **実行後 Git コミット**（AutoCommit=trueの場合）
   - `git add -A`
   - `git commit -m "[Worker Auto-Commit] After patch: <summary>"`
   - コミット失敗時はログ警告（実行後のため中断しない）

6. **ログ記録**
   - 成功: `logger.InfoCF("worker", "Worker execution succeeded", ...)`
   - 失敗: `logger.ErrorCF("worker", "Worker execution failed", ...)`

7. **結果返却**
   - 成功: "実行完了。変更内容: [summary]"
   - 失敗: "実行失敗。エラー: [error]"

### 9.4 エラーハンドリング戦略

#### 継続モード（デフォルト）
- 1つのコマンドが失敗しても残りを実行
- 全体の成功/失敗は最後に判定

#### 中断モード（オプション）
- `StopOnError: true` で有効化
- 最初の失敗で即座に中断

#### Worker失敗時のロールバック処理

**実行前Git auto-commitがONの場合**:
1. 実行前コミットが成功 → パッチ適用開始
2. パッチ適用中にエラー発生 → `StopOnError=true` の場合は即座に中断
3. **ロールバック**: `git reset --hard HEAD~1` で実行前の状態に戻す
4. エラーログ記録:
   ```go
   logger.ErrorCF("worker", "Worker execution failed, rolled back", map[string]interface{}{
       "job_id": jobID.String(),
       "failed_command": cmd.Type + ":" + cmd.Action,
       "error": err.Error(),
       "rollback_commit": commitHash,
   })
   ```

**実行前Git auto-commitがOFFの場合**:
- ロールバック不可
- エラーログのみ記録
- ユーザーに手動リカバリを促すメッセージを返す:
  ```
  "実行失敗: {error}. Git auto-commit がOFFのため、ロールバックできません。手動で復旧してください。"
  ```

#### 分類器エラー時の詳細ログ

**MioAgent.DecideAction() 内で分類器呼び出し失敗時**:
```go
// internal/domain/agent/mio.go
func (m *MioAgent) DecideAction(ctx context.Context, t task.Task) (routing.Decision, error) {
    // ... 明示コマンド、ルール辞書チェック ...

    // 分類器呼び出し
    decision, err := m.classifier.Classify(ctx, t)
    if err != nil {
        logger.ErrorCF("routing", "Classifier failed, fallback to CHAT", map[string]interface{}{
            "job_id": t.JobID().String(),
            "message": t.UserMessage(),
            "error": err.Error(),
            "error_type": fmt.Sprintf("%T", err), // エラー型
            "fallback_route": "CHAT",
        })
        // 安全側フォールバック: CHAT
        return routing.Decision{
            Route:      routing.RouteCHAT,
            Confidence: 0.5,
            Reason:     fmt.Sprintf("Classifier error (fallback): %v", err),
        }, nil
    }

    return decision, nil
}
```

**詳細ログ項目**:
- `job_id`: ジョブID
- `message`: ユーザーメッセージ（先頭100文字）
- `error`: エラーメッセージ
- `error_type`: エラー型（`*url.Error`, `context.DeadlineExceeded` 等）
- `fallback_route`: フォールバック先ルート

#### MCP部分失敗時のロールバック

**MCPツール実行が部分的に失敗した場合**:

**シナリオ**: Coder3が複数のMCPツールを提案 → Worker実行 → 1つ目成功、2つ目失敗

**対応**:
1. **継続モード（デフォルト）**: 失敗したツールをスキップ、残りを実行
   - ログ: 成功数/失敗数を記録
   - 返信: "実行完了 (2/3 コマンド成功). 失敗: {error}"

2. **中断モード（`StopOnError=true`）**: 失敗時点で即座に中断
   - Git auto-commit ONの場合: `git reset --hard HEAD~1` でロールバック
   - Git auto-commit OFFの場合: ユーザーに手動リカバリを促す

**MCPツール実行ログ**:
```go
// executeCommand() でMCPツール実行時
logger.InfoCF("worker", "MCP tool execution", map[string]interface{}{
    "job_id": jobID.String(),
    "tool_name": mcpToolName,
    "tool_args": toolArgs,
})

// 失敗時
logger.ErrorCF("worker", "MCP tool execution failed", map[string]interface{}{
    "job_id": jobID.String(),
    "tool_name": mcpToolName,
    "error": err.Error(),
    "partial_success": successCount,
    "total": totalCount,
})
```

```go
package service

import (
    "context"
    "github.com/sipeed/rencrow/internal/domain/task"
)

// MessageOrchestrator はAgent Loop分解の中核
type MessageOrchestrator struct {
    routingService   RoutingService
    llmIterator      LLMIteratorService
    workerExecution  WorkerExecutionService
    memoryService    MemoryService
}

func NewMessageOrchestrator(
    routingService RoutingService,
    llmIterator LLMIteratorService,
    workerExecution WorkerExecutionService,
    memoryService MemoryService,
) *MessageOrchestrator {
    return &MessageOrchestrator{
        routingService:  routingService,
        llmIterator:     llmIterator,
        workerExecution: workerExecution,
        memoryService:   memoryService,
    }
}

// ProcessMessage はメインフロー調整
func (o *MessageOrchestrator) ProcessMessage(ctx context.Context, input InputMessage) (*Response, error) {
    // 1. Task作成
    t := task.NewTask(task.NewJobID(), input.Message, input.Channel, input.ChatID)
    if input.ForcedRoute != "" {
        t = t.WithForcedRoute(input.ForcedRoute)
    }

    // 2. ルーティング決定（Mio委譲判断）
    decision, err := o.routingService.Decide(ctx, t)
    if err != nil {
        return nil, err
    }
    t = t.WithRoute(decision.Route)

    // 3. LLM呼び出しループ
    result, err := o.llmIterator.Execute(ctx, t, decision)
    if err != nil {
        return nil, err
    }

    // 4. Worker即時実行（Coder3の場合）
    if decision.Route == routing.RouteCODE3 && result.Proposal != nil {
        execResult, err := o.workerExecution.ExecuteProposal(ctx, t.JobID(), result.Proposal)
        if err != nil {
            return &Response{
                Content: fmt.Sprintf("Worker実行失敗: %v", err),
                Error:   err,
            }, err
        }

        // 実行結果を整形して返却
        return &Response{
            Content: fmt.Sprintf("実行完了 (%d/%d コマンド成功)\n\n【Plan】\n%s\n\n【実行結果】\n%s",
                execResult.ExecutedCmds,
                execResult.ExecutedCmds + execResult.FailedCmds,
                result.Proposal.Plan(),
                execResult.Summary),
        }, nil
    }

    // 5. その他のルート（CHAT, WORKER等）
    return &Response{Content: result.Content}, nil
}
```

### 17.3 LLMIteratorService

**ファイル**: `internal/application/service/llm_iterator_service.go`

```go
package service

import (
    "context"
    "fmt"
    "time"
    "github.com/sipeed/rencrow/internal/domain/llm"
    "github.com/sipeed/rencrow/internal/domain/proposal"
    "github.com/sipeed/rencrow/internal/domain/routing"
    "github.com/sipeed/rencrow/internal/domain/task"
)

// LLMIteratorService はLLM呼び出しループを管理（リトライ、タイムアウト等）
type LLMIteratorService interface {
    Execute(ctx context.Context, t task.Task, decision routing.Decision) (*LLMIteratorResult, error)
}

type LLMIteratorResult struct {
    Content  string
    Proposal *proposal.Proposal // Coderルートの場合のみ設定
}

type llmIteratorService struct {
    providers  map[string]llm.LLMProvider
    maxRetries int
    timeout    time.Duration
}

func NewLLMIteratorService(providers map[string]llm.LLMProvider, maxRetries int, timeout time.Duration) LLMIteratorService {
    return &llmIteratorService{
        providers:  providers,
        maxRetries: maxRetries,
        timeout:    timeout,
    }
}

func (l *llmIteratorService) Execute(ctx context.Context, t task.Task, decision routing.Decision) (*LLMIteratorResult, error) {
    // ルートに応じたプロバイダー選択
    provider := l.selectProvider(decision.Route)

    // LLM呼び出し（リトライ付き）
    for i := 0; i < l.maxRetries; i++ {
        response, err := provider.Generate(ctx, buildRequest(t))
        if err == nil {
            return &LLMIteratorResult{
                Content:  response.Content,
                Proposal: extractProposalFromResponse(response), // Coderルートの場合のみ
            }, nil
        }
        // リトライ
        time.Sleep(time.Second * time.Duration(i+1))
    }

    return nil, fmt.Errorf("LLM call failed after %d retries", l.maxRetries)
}

// selectProvider はルートに応じてプロバイダーを選択
func (l *llmIteratorService) selectProvider(route routing.Route) llm.LLMProvider {
    // ルーティングに応じたプロバイダーマッピング
    providerMap := map[routing.Route]string{
        routing.RouteCHAT:     "ollama-chat",
        routing.RouteANALYZE:  "ollama-worker",
        routing.RouteOPS:      "ollama-worker",
        routing.RouteRESEARCH: "ollama-worker",
        routing.RouteCODE1:    "deepseek",
        routing.RouteCODE2:    "openai",
        routing.RouteCODE3:    "claude",
    }

    providerName, ok := providerMap[route]
    if !ok {
        providerName = "ollama-chat" // デフォルト
    }

    if provider, ok := l.providers[providerName]; ok {
        return provider
    }

    return l.providers["ollama-chat"] // フォールバック
}

// buildRequest はタスクからLLMリクエストを構築
func buildRequest(t task.Task) llm.GenerateRequest {
    return llm.GenerateRequest{
        Messages: []llm.Message{
            {
                Role:    "user",
                Content: t.UserMessage(),
            },
        },
        MaxTokens:    4096,
        Temperature:  0.7,
        SystemPrompt: "", // ルートに応じてシステムプロンプトを設定可能
    }
}

// extractProposalFromResponse はLLM応答からProposalを抽出（Coderルートの場合のみ）
func extractProposalFromResponse(response llm.GenerateResponse) *proposal.Proposal {
    // Coderルートの場合のみProposal抽出
    // Plan/Patch/Risk/CostHintセクションを抽出
    plan := extractPlanFromContent(response.Content)
    patch := extractPatchFromContent(response.Content)
    risk := extractRiskFromContent(response.Content)
    costHint := extractCostHintFromContent(response.Content)

    if plan == "" || patch == "" {
        return nil // CoderルートでないまたはProposal形式でない
    }

    return proposal.NewProposal(plan, patch, risk, costHint)
}

// extractPlanFromContent は応答コンテンツからPlanセクションを抽出
func extractPlanFromContent(content string) string {
    planStart := strings.Index(content, "## Plan")
    if planStart == -1 {
        return ""
    }
    planEnd := strings.Index(content[planStart+7:], "##")
    if planEnd == -1 {
        return strings.TrimSpace(content[planStart+7:])
    }
    return strings.TrimSpace(content[planStart+7 : planStart+7+planEnd])
}

// extractPatchFromContent は応答コンテンツからPatchセクションを抽出
func extractPatchFromContent(content string) string {
    patchStart := strings.Index(content, "## Patch")
    if patchStart == -1 {
        return ""
    }
    patchEnd := strings.Index(content[patchStart+8:], "##")
    if patchEnd == -1 {
        return strings.TrimSpace(content[patchStart+8:])
    }
    return strings.TrimSpace(content[patchStart+8 : patchStart+8+patchEnd])
}

// extractRiskFromContent は応答コンテンツからRiskレベルを抽出
func extractRiskFromContent(content string) proposal.RiskLevel {
    riskStart := strings.Index(content, "## Risk")
    if riskStart == -1 {
        return proposal.RiskLevelMedium
    }
    riskSection := content[riskStart : min(riskStart+50, len(content))]
    if strings.Contains(strings.ToLower(riskSection), "high") {
        return proposal.RiskLevelHigh
    } else if strings.Contains(strings.ToLower(riskSection), "low") {
        return proposal.RiskLevelLow
    }
    return proposal.RiskLevelMedium
}

// extractCostHintFromContent は応答コンテンツからCostヒントを抽出
func extractCostHintFromContent(content string) string {
    costStart := strings.Index(content, "## Cost")
    if costStart == -1 {
        return ""
    }
    costEnd := strings.Index(content[costStart+7:], "##")
    if costEnd == -1 {
        return strings.TrimSpace(content[costStart+7:])
    }
    return strings.TrimSpace(content[costStart+7 : costStart+7+costEnd])
}
```

### 17.4 MemoryService

**ファイル**: `internal/application/service/memory_service.go`

```go
package service

import (
    "context"
    "fmt"
    "sync"
    "github.com/sipeed/rencrow/internal/domain/task"
)

// MemoryService はセッションメモリ管理
type MemoryService interface {
    Save(ctx context.Context, jobID task.JobID, key string, value interface{}) error
    Load(ctx context.Context, jobID task.JobID, key string) (interface{}, error)
    Clear(ctx context.Context, jobID task.JobID) error
}

type memoryService struct {
    store map[string]map[string]interface{} // jobID -> key -> value
    mu    sync.RWMutex
}

func NewMemoryService() MemoryService {
    return &memoryService{
        store: make(map[string]map[string]interface{}),
    }
}

func (m *memoryService) Save(ctx context.Context, jobID task.JobID, key string, value interface{}) error {
    m.mu.Lock()
    defer m.mu.Unlock()

    id := jobID.String()
    if _, ok := m.store[id]; !ok {
        m.store[id] = make(map[string]interface{})
    }
    m.store[id][key] = value
    return nil
}

func (m *memoryService) Load(ctx context.Context, jobID task.JobID, key string) (interface{}, error) {
    m.mu.RLock()
    defer m.mu.RUnlock()

    id := jobID.String()
    if session, ok := m.store[id]; ok {
        if value, ok := session[key]; ok {
            return value, nil
        }
    }
    return nil, fmt.Errorf("key not found: %s", key)
}

func (m *memoryService) Clear(ctx context.Context, jobID task.JobID) error {
    m.mu.Lock()
    defer m.mu.Unlock()

    delete(m.store, jobID.String())
    return nil
}
```

### 17.6 WorkerExecutionService

**ファイル**: `internal/application/service/worker_execution_service.go`

詳細は **9.3 Worker実行ロジック** を参照。ここでは主要な実装ポイントのみ記載。

```go
package service

import (
    "context"
    "fmt"
    "os"
    "os/exec"
    "path/filepath"
    "strings"
    "time"
    "github.com/sipeed/rencrow/internal/domain/agent"
    "github.com/sipeed/rencrow/internal/domain/patch"
    "github.com/sipeed/rencrow/internal/domain/proposal"
    "github.com/sipeed/rencrow/internal/domain/task"
    "github.com/sipeed/rencrow/pkg/logger"
)

type WorkerConfig struct {
    AutoCommit         bool
    CommitMessagePrefix string
    CommandTimeout     int // 秒
    GitTimeout         int // 秒
    StopOnError        bool
    Workspace          string
    ProtectedPatterns  []string // 保護ファイルパターン
    ActionOnProtected  string   // "error", "skip", "log"
    ShowExecutionSummary bool   // 実行前サマリ表示
}

type WorkerExecutionService interface {
    ExecuteProposal(ctx context.Context, jobID task.JobID, p *proposal.Proposal) (*patch.PatchExecutionResult, error)
}

type workerExecutionService struct {
    shiroAgent *agent.ShiroAgent
    config     WorkerConfig
}

func NewWorkerExecutionService(shiroAgent *agent.ShiroAgent, config WorkerConfig) WorkerExecutionService {
    return &workerExecutionService{
        shiroAgent: shiroAgent,
        config:     config,
    }
}

func (w *workerExecutionService) ExecuteProposal(ctx context.Context, jobID task.JobID, p *proposal.Proposal) (*patch.PatchExecutionResult, error) {
    // 1. Patch解析（Domain層のParsePatch関数を使用）
    commands, err := patch.ParsePatch(p.Patch())
    if err != nil {
        return nil, fmt.Errorf("patch parse error: %w", err)
    }

    // 2. 実行前サマリ表示（推奨）
    if w.config.ShowExecutionSummary {
        summary := w.generateExecutionSummary(commands)
        logger.InfoCF("worker", "Execution summary",
            map[string]interface{}{
                "job_id":         jobID.String(),
                "total_commands": summary.TotalCommands,
                "file_edits":     summary.FileEdits,
                "shell_commands": summary.ShellCommands,
                "git_operations": summary.GitOperations,
            })
    }

    // 3. 実行前 Git コミット（AutoCommit=trueの場合は必須）
    var beforeCommit string
    if w.config.AutoCommit {
        beforeCommit, err = w.autoCommitChanges(ctx, "Before patch")
        if err != nil {
            // 重要: AutoCommit有効時、コミット失敗 = 実行中断（安全性担保）
            return nil, fmt.Errorf("auto-commit failed (safety requirement), execution aborted: %w", err)
        }
        logger.InfoCF("worker", "Auto-commit before patch succeeded",
            map[string]interface{}{"commit": beforeCommit})
    }

    // 3. コマンド実行ループ
    result := &patch.PatchExecutionResult{
        Success:      true,
        ExecutedCmds: 0,
        FailedCmds:   0,
        Results:      make([]patch.CommandResult, 0),
    }

    for _, cmd := range commands {
        output, err := w.executeCommand(ctx, cmd)
        cmdResult := patch.CommandResult{
            Command: cmd,
            Success: err == nil,
            Output:  output,
        }

        if err != nil {
            cmdResult.Error = err.Error()
            result.FailedCmds++
            result.Success = false

            // 中断モード
            if w.config.StopOnError {
                result.Results = append(result.Results, cmdResult)
                return result, fmt.Errorf("command failed (stop on error): %w", err)
            }
        } else {
            result.ExecutedCmds++
        }

        result.Results = append(result.Results, cmdResult)
    }

    // 4. 実行後 Git コミット（オプション）
    if w.config.AutoCommit {
        afterCommit, err := w.autoCommitChanges(ctx, "After patch: "+p.Plan()[:min(50, len(p.Plan()))])
        if err != nil {
            logger.WarnCF("worker", "Auto-commit after patch failed", map[string]interface{}{"error": err})
        } else {
            result.GitCommit = afterCommit
        }
    }

    // 5. ログ記録
    if result.Success {
        logger.InfoCF("worker", "Worker execution succeeded", map[string]interface{}{
            "job_id": jobID.String(),
            "executed": result.ExecutedCmds,
        })
    } else {
        logger.ErrorCF("worker", "Worker execution failed", map[string]interface{}{
            "job_id": jobID.String(),
            "failed": result.FailedCmds,
            "total": len(commands),
        })
    }

    return result, nil
}

// executeCommand は単一のコマンドを実行
func (w *workerExecutionService) executeCommand(ctx context.Context, cmd patch.PatchCommand) (string, error) {
    switch cmd.Type {
    case patch.TypeFileEdit:
        return w.executeFileEdit(ctx, cmd)
    case patch.TypeShellCommand:
        return w.executeShellCommand(ctx, cmd)
    case patch.TypeGitOperation:
        return w.executeGitOperation(ctx, cmd)
    default:
        return "", fmt.Errorf("unknown command type: %s", cmd.Type)
    }
}

// executeFileEdit はファイル編集を実行
func (w *workerExecutionService) executeFileEdit(ctx context.Context, cmd patch.PatchCommand) (string, error) {
    // 1. workspace 外への書き込みを禁止
    if !strings.HasPrefix(cmd.Target, w.config.Workspace) {
        return "", fmt.Errorf("file path outside workspace: %s", cmd.Target)
    }

    // 2. 保護ファイルチェック
    if w.isProtectedFile(cmd.Target) {
        switch w.config.ActionOnProtected {
        case "error":
            return "", fmt.Errorf("protected file cannot be modified: %s", cmd.Target)
        case "skip":
            logger.WarnCF("worker", "Skipped protected file", map[string]interface{}{"file": cmd.Target})
            return "skipped (protected file)", nil
        case "log":
            logger.WarnCF("worker", "Modifying protected file", map[string]interface{}{"file": cmd.Target})
            // 続行
        }
    }

    switch cmd.Action {
    case patch.ActionCreate, patch.ActionUpdate:
        // ファイル作成/上書き（親ディレクトリが存在しない場合は自動作成）
        dir := filepath.Dir(cmd.Target)
        if err := os.MkdirAll(dir, 0755); err != nil {
            return "", fmt.Errorf("failed to create parent directory: %w", err)
        }
        return "", os.WriteFile(cmd.Target, []byte(cmd.Content), 0644)

    case patch.ActionDelete:
        return "", os.Remove(cmd.Target)

    case patch.ActionAppend:
        f, err := os.OpenFile(cmd.Target, os.O_APPEND|os.O_WRONLY, 0644)
        if err != nil {
            return "", err
        }
        defer f.Close()
        _, err = f.WriteString(cmd.Content)
        return "", err

    case patch.ActionRename:
        // ファイル/ディレクトリ移動・リネーム（Target → Content）
        if cmd.Content == "" {
            return "", fmt.Errorf("rename requires Content field (destination path)")
        }
        // デスティネーションも保護ファイルチェック
        if w.isProtectedFile(cmd.Content) {
            return "", fmt.Errorf("cannot rename to protected file: %s", cmd.Content)
        }
        // 移動先の親ディレクトリが存在しない場合は自動作成
        destDir := filepath.Dir(cmd.Content)
        if err := os.MkdirAll(destDir, 0755); err != nil {
            return "", fmt.Errorf("failed to create destination directory: %w", err)
        }
        return "", os.Rename(cmd.Target, cmd.Content)

    case patch.ActionMkdir:
        // ディレクトリ作成（親ディレクトリも自動作成）
        return "", os.MkdirAll(cmd.Target, 0755)

    case patch.ActionCopy:
        // ファイルコピー（Target → Content）
        if cmd.Content == "" {
            return "", fmt.Errorf("copy requires Content field (destination path)")
        }
        // デスティネーションも保護ファイルチェック
        if w.isProtectedFile(cmd.Content) {
            return "", fmt.Errorf("cannot copy to protected file: %s", cmd.Content)
        }
        // コピー先の親ディレクトリが存在しない場合は自動作成
        destDir := filepath.Dir(cmd.Content)
        if err := os.MkdirAll(destDir, 0755); err != nil {
            return "", fmt.Errorf("failed to create destination directory: %w", err)
        }
        // ファイルコピー
        srcData, err := os.ReadFile(cmd.Target)
        if err != nil {
            return "", fmt.Errorf("failed to read source file: %w", err)
        }
        if err := os.WriteFile(cmd.Content, srcData, 0644); err != nil {
            return "", fmt.Errorf("failed to write destination file: %w", err)
        }
        return fmt.Sprintf("copied %s -> %s", cmd.Target, cmd.Content), nil

    default:
        return "", fmt.Errorf("unknown action: %s", cmd.Action)
    }
}

// executeShellCommand はシェルコマンドを実行
func (w *workerExecutionService) executeShellCommand(ctx context.Context, cmd patch.PatchCommand) (string, error) {
    execCtx, cancel := context.WithTimeout(ctx, time.Duration(w.config.CommandTimeout)*time.Second)
    defer cancel()

    // シェル種別の決定（デフォルト: bash）
    shell := cmd.Shell
    if shell == "" {
        shell = "bash"
    }

    // コマンド構築
    execCmd := exec.CommandContext(execCtx, shell, "-c", cmd.Target)

    // 作業ディレクトリの設定
    if cmd.WorkDir != "" {
        // workspace内かチェック
        if !strings.HasPrefix(cmd.WorkDir, w.config.Workspace) {
            return "", fmt.Errorf("workdir outside workspace: %s", cmd.WorkDir)
        }
        execCmd.Dir = cmd.WorkDir
    }

    // 環境変数の設定
    if len(cmd.Env) > 0 {
        // 既存の環境変数を継承し、指定された環境変数を追加/上書き
        execCmd.Env = os.Environ()
        for key, value := range cmd.Env {
            execCmd.Env = append(execCmd.Env, fmt.Sprintf("%s=%s", key, value))
        }
    }

    output, err := execCmd.CombinedOutput()
    return string(output), err
}

// executeGitOperation は Git 操作を実行
func (w *workerExecutionService) executeGitOperation(ctx context.Context, cmd patch.PatchCommand) (string, error) {
    execCtx, cancel := context.WithTimeout(ctx, time.Duration(w.config.GitTimeout)*time.Second)
    defer cancel()

    var gitCmd *exec.Cmd
    switch cmd.Action {
    case patch.ActionAdd:
        gitCmd = exec.CommandContext(execCtx, "git", "add", cmd.Target)
    case patch.ActionCommit:
        gitCmd = exec.CommandContext(execCtx, "git", "commit", "-m", cmd.Content)
    case patch.ActionReset:
        gitCmd = exec.CommandContext(execCtx, "git", "reset", cmd.Target)
    case patch.ActionCheckout:
        gitCmd = exec.CommandContext(execCtx, "git", "checkout", cmd.Target)
    default:
        return "", fmt.Errorf("unknown git action: %s", cmd.Action)
    }

    output, err := gitCmd.CombinedOutput()
    return string(output), err
}

// autoCommitChanges は Git 自動コミットを実行
func (w *workerExecutionService) autoCommitChanges(ctx context.Context, message string) (string, error) {
    // git add -A
    addCmd := exec.CommandContext(ctx, "git", "add", "-A")
    if err := addCmd.Run(); err != nil {
        return "", err
    }

    // git commit
    commitMsg := fmt.Sprintf("%s %s", w.config.CommitMessagePrefix, message)
    commitCmd := exec.CommandContext(ctx, "git", "commit", "-m", commitMsg)
    if err := commitCmd.Run(); err != nil {
        return "", err
    }

    // コミットハッシュ取得
    hashCmd := exec.CommandContext(ctx, "git", "rev-parse", "HEAD")
    hash, err := hashCmd.Output()
    if err != nil {
        return "", err
    }

    return strings.TrimSpace(string(hash)), nil
}

func min(a, b int) int {
    if a < b {
        return a
    }
    return b
}

// isProtectedFile は保護ファイルパターンに一致するか判定
func (w *workerExecutionService) isProtectedFile(filePath string) bool {
    for _, pattern := range w.config.ProtectedPatterns {
        matched, _ := filepath.Match(pattern, filepath.Base(filePath))
        if matched {
            return true
        }
    }
    return false
}

// generateExecutionSummary は実行前サマリを生成
func (w *workerExecutionService) generateExecutionSummary(commands []patch.PatchCommand) ExecutionSummary {
    summary := ExecutionSummary{}
    for _, cmd := range commands {
        summary.TotalCommands++
        switch cmd.Type {
        case patch.TypeFileEdit:
            summary.FileEdits++
        case patch.TypeShellCommand:
            summary.ShellCommands++
        case patch.TypeGitOperation:
            summary.GitOperations++
        }
    }
    return summary
}

type ExecutionSummary struct {
    TotalCommands int
    FileEdits     int
    ShellCommands int
    GitOperations int
}
```

```go
package line

import (
    "context"
    "github.com/line/line-bot-sdk-go/v7/linebot"
    "github.com/sipeed/rencrow/internal/application/usecase"
    "github.com/sipeed/rencrow/internal/domain/routing"
)

type LINEHandler struct {
    processMessageUC *usecase.ProcessMessage
    bot              *linebot.Client
}

func NewLINEHandler(processMessageUC *usecase.ProcessMessage, bot *linebot.Client) *LINEHandler {
    return &LINEHandler{
        processMessageUC: processMessageUC,
        bot:              bot,
    }
}

func (h *LINEHandler) HandleMessage(event *linebot.Event) error {
    // LINE入口は必ずCHAT（Mio）に固定
    input := usecase.InputMessage{
        Channel:     "line",
        ChatID:      event.Source.UserID,
        Message:     event.Message.(*linebot.TextMessage).Text,
        ForcedRoute: routing.RouteCHAT, // 仕様準拠: LINE固定
    }

    response, err := h.processMessageUC.Execute(context.Background(), input)
    if err != nil {
        return err
    }

    // LINE API経由で返信
    if _, err := h.bot.ReplyMessage(event.ReplyToken, linebot.NewTextMessage(response.Content)).Do(); err != nil {
        return err
    }

    return nil
}
```

## 19. OpenClaw移植仕様（実装能力）

### 19.1 目的

本章は、OpenClawにおける「指示を受けて実装を完了させる能力」を、既存のGo基盤（RenCrow）へ移植するための正本仕様である。  
ここでの対象は **TTS機能そのもの** ではなく、`TTSを導入して` のような依頼を **提案止まりにせず完了まで遂行する能力** とする。

### 19.2 移植対象モジュール

移植は次の4モジュール単位で行う。

1. `Intent/Contract`: ユーザー依頼を機械実行契約へ正規化
2. `Executor`: 実装・依存導入・起動・検証・修復の実行ループ
3. `Capability Pack`: ドメイン特化の手順テンプレート（例: TTS導入）
4. `Evidence`: 完了証跡の保存と可視化

### 19.3 実行契約（Execution Contract）

実装能力は、次の必須フィールドを持つ実行契約を起点に実行する。

- `goal`: 達成目標（例: TTS導入と再生可能化）
- `acceptance`: 受け入れ条件（E2E実再生必須）
- `constraints`: 禁止操作/境界条件
- `artifacts`: 生成物（コード、設定、音声ファイル、ログ）
- `verification`: 検証手順
- `rollback`: 失敗時の復旧方針

Coder出力は契約バリデーション必須とし、実行不能な提案（設計文、疑似JSON、設定オブジェクトのみ等）は失敗として再生成へ回す。

### 19.4 自律実行ループ

実行ループは次の順で固定する。

1. `Plan`: 実行契約に基づく実装計画生成
2. `Apply`: patch/commandの適用
3. `Verify`: 技術検証 + 体験検証（E2E再生）
4. `Repair`: 失敗分類に応じた修復
5. `Verify`: 再検証

完了条件は、`ProcessMessage COMPLETE` のみではなく、`E2E playback success` を含む証跡の成立とする。

### 19.5 Provider戦略

TTS導入タスクのProvider選択は次の優先順で実施する。

1. OpenAI
2. ElevenLabs
3. Local TTS

上位が利用不能な場合は自動で次候補へフォールバックする。  
全候補失敗時は、修復不可として失敗証跡を確定する。

### 19.6 安全境界（完全自動運用）

完全自動実行を前提としつつ、次の操作は内部ガードで禁止する。

- 破壊的削除（例: `rm -rf`）
- 強制リセット系Git操作（`git reset --hard` 等）
- 秘密情報の露出・外部送信

### 19.7 証跡仕様（Execution Evidence）

ジョブ単位で `execution_report` を保存し、最低限次を記録する。

- 入力契約
- 実行ステップと結果
- 生成音声ファイル情報
- 再生確認結果
- 最終判定（passed/failed）

Viewerでは提案文ではなく、この証跡を一次情報として表示する。

## 20. Viewer / Evidence / Job 実装仕様

### 20.1 対象 API

Viewer は execution evidence と live job 状態を参照する。

| エンドポイント | メソッド | 用途 |
|---|---|---|
| `/viewer/evidence/recent` | `GET` | 直近 evidence 一覧 |
| `/viewer/evidence/detail` | `GET` | job 単位 evidence 詳細 |
| `/viewer/evidence/summary` | `GET` | evidence 集計 |
| `/viewer/jobs` | `GET` | job 一覧 |
| `/viewer/job/detail` | `GET` | job 詳細 |

### 20.2 live jobs と execution evidence の責務

`live jobs` は `MonitorStore` の event reducer から導出した現在状態である。進行中の job を見せるための観測用 snapshot であり、完了結果の正本ではない。

`execution evidence` は `workspace/execution_report.jsonl` に保存される完了結果の長期証跡である。完了済み job の成功・失敗判定では、`execution evidence` を正本とする。

`/viewer/job/detail` は live job detail に加え、対応する `ExecutionReport` があれば `evidence` として返す。実装上は archived events も参照し、必要に応じて job events を補完する。

### 20.3 完了判定の優先順位

job の実行結果を判定する場合、対応する `ExecutionReport` が存在するなら `execution evidence` を `live jobs` より優先する。

優先順位は以下とする。

1. 対応する `ExecutionReport` があり、`evidence.status=passed` の場合は実行成功完了
2. 対応する `ExecutionReport` があり、`evidence.status=failed` の場合は実行失敗完了
3. 対応する `ExecutionReport` がまだない場合は、`live jobs` の `status` / `phase` / event を進行中表示として使う

`live jobs` の `JobSnapshot.status` や `phase` は event reducer の観測結果である。そのため、`evidence.status=passed` が存在する job について、`JobSnapshot.status=running`、`phase=reporting` が残っていても、Viewer の実効表示は完了成功として扱う。

### 20.4 `mio_reported`

`mio_reported=false` は、Mio による最終ユーザー報告 event が観測されていない状態を表す補助状態である。これは Worker 実行失敗や job 未完了を意味しない。

`evidence.status=passed` と `mio_reported=false` が同時に存在する場合、Viewer は「実行は成功完了、最終報告 event は未観測」として表示する。最終報告未観測は warning または補助表示にとどめ、実行完了状態を running に戻してはならない。

### 20.5 運用上の見方

- 進行中か、どこで止まっているかを見る場合は `live jobs` を使う
- 完了結果や証跡を見る場合は `execution evidence` を使う
- `job は見えるのに evidence がない` 場合は、まず進行中かどうかを確認し、次に `System` または persisted logs を見る
- `evidence.status=passed` だが `job detail` が `phase=reporting` / `status=running` / `mio_reported=false` の場合、実行自体は成功完了であり、再実行は不要
- 最終ユーザー報告の表示有無だけを確認したい場合は、job event、`/viewer/evidence/detail`、persisted log を照合する

- 実行した build / test / restart command

- Worker: 実行ログ、変更ファイル、コマンド出力、成果物

## 22. Mio 会話中の並列ジョブ管理・割り込み通知仕様

### 22.1 目的

Mio はユーザーとの会話を維持しながら、Shiro / Heavy / Wild / Coder 群へ複数の作業を委譲できるものとする。

この仕様の目的は以下である。

- Mio の通常会話をブロックせず、裏で複数 job を進める
- job ごとに module root、担当 role、共有文脈、証跡を分離する
- Worker / Heavy / Wild を等価差し替え可能な実行・思考 role として扱う
- 完了、失敗、判断待ち、停止だけを割り込み通知として Mio 会話へ上げる
- 進捗ログの垂れ流しを避け、ユーザーが必要な時に `jobs` view / CLI で詳細確認できるようにする

### 22.2 全体構成

並列 job 管理は、Mio と各 role 実行系の間に `JobManager` を置く。

```text
User
  <-> Mio
      -> JobManager
         -> JobQueue
         -> JobStateStore
         -> SharedRoleContextStore
         -> NotificationDispatcher
         -> Shiro / Worker
         -> Heavy
         -> Wild
         -> Coder1 / Coder2 / Coder3 / Coder4
```

`JobManager` は次を担う。

- job 作成、状態遷移、キャンセル、再試行
- module registry による `module_id` / `module_root` 解決
- 並列実行制限の判定
- role への context 注入
- completion / failed / blocked / waiting_user の割り込み通知生成
- evidence / execution report / shared context へのリンク保存

### 22.3 Job モデル

job は最低限以下の属性を持つ。

```json
{
  "job_id": "job_20260618_abc123",
  "title": "RenCrow_STT stop後のfinal/error競合修正",
  "module_id": "stt",
  "module_root": "/home/nyukimi/RenCrow/RenCrow_STT",
  "route": "CODE",
  "assignee": "Shiro",
  "coder_roles": ["Coder1", "Coder2"],
  "status": "running",
  "priority": "normal",
  "created_by": "Mio",
  "parent_conversation_id": "conv_...",
  "parent_message_id": "msg_...",
  "interrupt_policy": "notify_on_done_or_blocked",
  "created_at": "2026-06-18T00:00:00Z",
  "updated_at": "2026-06-18T00:00:00Z"
}
```

必須属性:

| field | 説明 |
| --- | --- |
| `job_id` | job の一意 ID |
| `title` | Mio と Viewer に表示する短い要約 |
| `module_id` | module registry で解決された module |
| `module_root` | Worker / role 実行 workspace |
| `route` | `CODE` / `OPS` / `RESEARCH` などの routing |
| `assignee` | 主担当 role。例: `Shiro`, `Heavy`, `Wild` |
| `status` | job 状態 |
| `parent_conversation_id` | Mio 会話と job を紐付ける ID |
| `interrupt_policy` | どの状態で割り込み通知するか |

### 22.4 Job 状態

job status は以下を正本とする。

| status | 意味 |
| --- | --- |
| `queued` | 実行待ち |
| `running` | 実行中 |
| `waiting_user` | ユーザー判断待ち。Mio へ割り込み通知する |
| `blocked` | role が自律的に進めない。Mio へ割り込み通知する |
| `failed` | 失敗終了。Mio へ割り込み通知する |
| `succeeded` | 成功終了。Mio へ割り込み通知する |
| `cancelled` | ユーザーまたは policy により中止 |
| `superseded` | 新しい指示・新 job に置き換えられた |

状態遷移の原則:

- `queued -> running` は JobManager の parallel limit 判定後に行う
- `running -> waiting_user` は human approval / 仕様確認 / conflict 判断が必要な場合に使う
- `running -> blocked` は同じ blocker が解消不能な場合に使う
- `running -> failed` はコマンド失敗、検証失敗、push 失敗、policy violation などを含む
- `succeeded` / `failed` / `cancelled` / `superseded` は終端状態とする
- 終端状態の job を再開する場合は、新しい `job_id` を作る。ただし `supersedes_job_id` で関連付ける

### 22.5 並列実行制限

既定の並列制限は以下とする。

```yaml
parallel_limits:
  global: 3
  per_module: 1
  coding_jobs: 2
  long_research_jobs: 1
  destructive_ops: 1
```

制限の意味:

- `global`: 全 job の同時実行上限
- `per_module`: 同じ module root を編集する job の同時実行上限
- `coding_jobs`: CODE / CODE1 / CODE2 / CODE3 系 job の同時実行上限
- `long_research_jobs`: 長時間学習・調査 job の同時実行上限
- `destructive_ops`: build / install / restart / push / migration など副作用が大きい操作の同時実行上限

同じ module に対する編集 job は原則 `per_module: 1` とし、後続 job は `queued` にする。
同じ module に対する read-only 調査 job は、実装時に `read_only=true` を明示できる場合のみ例外として並列許可してよい。

### 22.6 SharedRoleContext

Worker / Heavy / Wild は等価差し替え可能な role として扱うため、job ごとに共有文脈を持つ。

`SharedRoleContext` は最低限以下を含む。

```json
{
  "job_id": "job_20260618_abc123",
  "user_intent": "STT stop後のfinal/error競合を直す",
  "module_id": "stt",
  "module_root": "/home/nyukimi/RenCrow/RenCrow_STT",
  "relevant_files": [
    "src/rencrow/stt/server.py",
    "tests/test_stt_streaming_contract.py"
  ],
  "decisions": [
    "partial provider定型文では即errorにしない"
  ],
  "constraints": [
    "終端イベントはfinalまたはerrorの一方だけ"
  ],
  "current_plan": "再現テスト、実装修正、対象テスト、push",
  "latest_status": "tests passed; push pending",
  "artifacts": [],
  "handoff_notes": "Heavyへ渡す場合は終端状態ガードを重点確認"
}
```

共有文脈に入れるもの:

- ユーザー意図
- module_id / module_root
- 関連ファイル
- 採用済み判断
- 制約
- 現在 plan
- 最新状態
- evidence / artifact link
- handoff notes

共有文脈に入れないもの:

- raw secret
- API key
- 不要な全文ログ
- 未検証の推測を確定事実として書いたもの

Heavy / Wild / Worker のいずれかへ job を差し替える場合、JobManager は最新 `SharedRoleContext` を role prompt に注入する。

### 22.7 Mio 会話と job の関係

Mio は通常会話を続けながら job を作成できる。

例:

```text
User: STTを直して。あとToolsにも検証CLIを作って。
Mio:
  - job A: RenCrow_STT 修正
  - job B: RenCrow_Tools 検証CLI作成
  - 会話は継続
```

Mio は進行中 job の詳細ログを通常会話へ流さない。
通常会話で表示するのは以下に限る。

- job を作成したこと
- job が queued / running になったことの短い要約
- completion / failed / blocked / waiting_user の割り込み通知
- ユーザーが明示的に `進捗見せて` と聞いた時の要約

### 22.8 割り込み通知

完了通知は `NotificationDispatcher` が生成し、Mio の通常会話へ interrupt event として差し込む。

通知対象の状態:

| status | 通知 |
| --- | --- |
| `succeeded` | 通知する |
| `failed` | 通知する |
| `blocked` | 通知する |
| `waiting_user` | 通知する |
| `queued` | 原則通知しない |
| `running` | 原則通知しない |
| `cancelled` | ユーザー操作起点なら短く通知する |
| `superseded` | 置換理由が必要な場合だけ通知する |

通知 payload:

```json
{
  "type": "job_notification",
  "level": "done",
  "job_id": "job_20260618_abc123",
  "title": "RenCrow_STT stop後のfinal/error競合修正",
  "status": "succeeded",
  "summary": "修正、対象テスト、pushまで完了",
  "next_actions": [],
  "interrupt": true,
  "created_at": "2026-06-18T00:00:00Z"
}
```

Mio 表示例:

```text
割り込み: STT修正が完了しました。
対象テストは通過し、main に push 済みです。
```

失敗時:

```text
割り込み: Tools検証CLIの作業が停止しました。
理由: GitHub push が reject され、rebase conflict が発生しています。
判断が必要です。
```

### 22.9 割り込み優先度

通知優先度は以下とする。

| priority | 表示タイミング | 例 |
| --- | --- | --- |
| `critical` | 直ちに表示 | データ破損リスク、認証失敗、破壊的操作失敗 |
| `blocked` | 直ちに表示 | ユーザー判断がないと進めない |
| `done` | 会話の区切りで表示 | 成功完了 |
| `failed` | 会話の区切りまたは直ちに表示 | test / push / build failed |
| `progress` | 原則表示しない | 中間進捗 |

`critical` / `blocked` は現在の Mio 応答生成を中断してでも表示してよい。
`done` / `failed` は Mio の現在発話終了後に差し込む。
`progress` はユーザーが明示的に進捗を要求した場合だけ表示する。

### 22.10 CLI / Viewer 要件

最小 CLI:

```bash
rencrow jobs list
rencrow jobs show <job_id>
rencrow jobs cancel <job_id>
rencrow jobs notifications
```

`rencrow` / backend CLI 要件:

- `rencrow jobs list` で job 一覧を確認できる
- `rencrow jobs show <job_id>` で job と `SharedRoleContext` を確認できる
- `rencrow jobs create` / `start` / `status` / `cancel` で job 状態を操作できる
- `rencrow jobs notifications` で割り込み通知を確認できる

`RenCrow_CLI` 要件:

- `RenCrow_CLI` は Mio との chat loop 中も job notification を購読する
- 購読元は当面 `/viewer/job-notifications?limit=N` の polling とする
- polling interval は既定 3 秒とし、CLI 起動オプションまたは設定で変更可能にする
- CLI は `job_id + status + level + created_at` を dedupe key として、同じ通知を二重表示しない
- 表示対象は `interrupt=true` の通知に限定する
- 表示 role は `assignee` に従う。`Worker` / `Heavy` は等価差し替え role として `Shiro` 表示に寄せてよい
- 通知は現在入力中のユーザー入力を破壊しない。TTY では次の安全な改行位置に割り込み行として表示する
- 通知表示後も Mio chat loop は継続する
- CLI が non-TTY / script mode の場合は、割り込み表示を標準エラーまたは structured event stream に分離する

`RenCrow_CLI` の表示例:

```text
[Shiro interrupt] CODE / succeeded / job_20260618_abc123
RenCrow_STT stop後のfinal/error競合修正
修正、対象テスト、pushまで完了
```

`RenCrow_CLI` の起動オプション案:

```bash
rencrow chat --job-interrupts=on
rencrow chat --job-interrupts=off
rencrow chat --job-interrupt-interval=3s
rencrow jobs notifications --watch
```

将来要件:

- backend が `job.notification` を `/viewer/events` SSE へ直接送れる場合、`RenCrow_CLI` は polling の代わりに SSE を購読してよい
- SSE と polling を併用する場合も dedupe key は同一にし、二重表示しない
- `blocked` / `waiting_user` は即時表示、`succeeded` / `failed` は現在のMio応答区切りで表示する

Viewer 要件:

- Jobs tab で queued / running / waiting_user / blocked / failed / succeeded を一覧表示する
- job detail で SharedRoleContext、最新 evidence、実行 command、test result、artifact link を表示する
- `waiting_user` / `blocked` は上部 summary に出す
- 完了通知は toast または conversation interrupt として表示する
- progress log は初期表示せず details に閉じる

### 22.11 永続化

最小実装では JSONL でよい。

推奨ファイル:

```text
workspace/jobs/job_state.jsonl
workspace/jobs/job_context.jsonl
workspace/jobs/job_notifications.jsonl
```

将来 SQLite / DuckDB backend へ置き換える場合も、domain model と event payload は互換にする。

保存イベント:

| event | 用途 |
| --- | --- |
| `job.created` | job 作成 |
| `job.queued` | queue 入り |
| `job.started` | 実行開始 |
| `job.context_updated` | SharedRoleContext 更新 |
| `job.waiting_user` | 判断待ち |
| `job.blocked` | 停止 |
| `job.failed` | 失敗 |
| `job.succeeded` | 成功 |
| `job.cancelled` | 中止 |
| `job.notification_emitted` | Mio へ割り込み通知生成 |

### 22.12 安全制約

- destructive ops は `destructive_ops: 1` を超えて同時実行しない
- 同じ module の編集 job は `per_module: 1` を守る
- build / restart / push は job evidence に必ず保存する
- `waiting_user` の job は勝手に destructive step へ進めない
- interrupt notification は成功を過大報告しない
- `succeeded` にできるのは、必要な test / build / push / restart / health check の完了状態が明示されている場合だけ
- `failed` / `blocked` 通知には、失敗した command、原因、次に必要な判断を含める

## 付録F: Chat経路障害時のRepair Plane

Mio / Chat 経路がユーザー発話の反復、無応答、誤ルーティングなどで通常会話から修復指示を受け付けられない場合に備え、Chat LLM を通さない Out-of-band Repair Plane を持つ。

Repair Plane は通常会話ではなく Ops 経路として扱い、ユーザー修復指示、対象ルート、対象エージェント、ログ範囲を構造化して記録する。

| Endpoint | Method | 用途 |
|---|---:|---|
| `/viewer/repair/run` | POST | ユーザー修復指示を受け取り、修復ジョブ受付イベントと割り込み通知を発行する |

初期 payload は次の通り。

```json
{
  "reason": "user-directed-repair",
  "instruction": "直近ログを見て、Chat経路の異常を診断し、修復してください。",
  "recent": 100,
  "target_route": "CHAT",
  "target_agent": "mio"
}
```

Viewer Chat には修復ボタンを置く。ボタン押下時、入力欄に文字列があればそれを `instruction` として `/viewer/repair/run` に送る。これは通常の `/viewer/send` ではなく、Mio への発話として扱わない。

RenCrow CLI の対話モードでは次の slash command を提供する。

- `/repair`
- `/repair ログを見てMioの反復を直して`
- `/repair help`

Repair Plane は `repair.requested` と `job.notification` を発行し、Viewer / CLI の割り込み通知経路でユーザーへ受付を知らせる。実修復の自動実行、ログ収集、差分適用、再ビルド、再起動、検証は後続の Repair Job 実行器が担当する。

## 付録G: Idea / Unimplemented Backlog

RenCrow は、ユーザー、Mio、Shiro、Coder が「面白そう」「後で実装したい」「未実装」と判断した項目を保存する Backlog 台帳を持つ。

Backlog は通常の一時指示キューではなく、実装・テスト・修正・OK 確認まで進めるための永続表である。

Backlog は RenCrow の実装予定項目の正規入口である。調査、仕様照合、レビュー、会話、運用中の観測で「未実装」「部分実装の残作業」「環境待ちの E2E」「後で実装する価値があるアイデア」を見つけた場合、回答や一時メモだけで終わらせず Backlog item として登録する。登録時点で実装許可がない項目は `status=open` または `blocked` に留め、実装作業とは分離する。

| Endpoint | Method | 用途 |
|---|---:|---|
| `/viewer/backlog` | GET | Backlog 項目一覧を返す。`limit` / `kind` / `status` で絞り込める |
| `/viewer/backlog` | POST | Backlog 項目の作成または更新を行う |

保存先は `workspace_dir/logs/backlog.jsonl` とする。標準の live workspace では `~/.rencrow/workspace/logs/backlog.jsonl` になる。JSONL は更新履歴を追えるよう追記型で保存し、一覧表示時は `item_id` ごとの最新項目を表示する。同一 `item_id` の更新は過去行を削除せず、新しい行として追記する。

Backlog は通常の Chat 入力、Repair Plane、Workstream、Sandbox、SuperAgent run queue の代替ではない。Backlog は「何を後で扱うか」の台帳であり、実際の実装、修復、検証、承認は対応する作業ジョブ、Workstream、Sandbox promotion、または手動作業へ引き渡す。

Backlog item の主要フィールドは次の通り。

- `item_id`: 項目 ID。未指定時はサーバが `backlog-<nanoseconds>` 形式で採番する
- `kind`: `idea` / `unimplemented`
- `title`, `body`
- `source`: `ren` / `mio` / `shiro` / `coder` など
- `owner`, `implementer`
- `status`: `open` / `implementing` / `testing` / `fixing` / `blocked` / `ok` / `rejected`
- `priority`: `low` / `normal` / `high` / `urgent`
- `tags`
- `implementation`: 実装方針、対象 module、関連 PR / job / commit などの実装メモ
- `test_result`
- `check_ok`, `checked_by`
- `created_at`, `updated_at`

`kind=unimplemented` は、現行 production code として未実装、または仕様上の残作業が明確な項目に使う。実装済みだが E2E 未確認、外部 credential 待ち、実機環境待ちなどは、本文または `tags` / `implementation` に「確認待ち」「環境待ち」を明記する。

Viewer は Backlog タブで表を表示し、追加、実装中、テスト中、修正中、OK チェックを操作できる。

Mio / Shiro が実装を進める場合、Backlog item を作業ジョブや実装報告に紐づけ、テスト結果を `test_result` に残す。OK 判断は `check_ok=true` と `checked_by` で明示する。

未実装調査を行ったエージェントは、見つけた項目を Backlog に登録するか、登録しない理由を報告する。登録する場合は、最低限 `kind`、`title`、`body`、`source`、`status`、`priority` を設定し、対象 module と根拠ファイルを `body` または `implementation` に残す。これにより「未実装を見つけたが会話から流れて消える」状態を禁止する。

Backlog item は、放置するためのメモではなく、順次 Workstream へ昇格して実装・検証する候補である。自動実装の起点は hook ではなく、Heartbeat / scheduler が担当する。hook は新規登録通知、登録漏れ検出、危険操作 block などの短い決定的処理に限定する。

Backlog Intake Heartbeat は、定期的に `status=open` または再開可能な `status=blocked` の item を読み、`priority`、`updated_at`、依存関係、環境 readiness、人間承認の有無を見て次に扱う item を選ぶ。選ばれた item は `status=implementing` に更新し、対応する Workstream Goal / Artifact / Sandbox Promotion Request / SuperAgent run queue のいずれかに接続する。実装作業を開始できない場合は、理由を `implementation` または `test_result` に残して `status=blocked` のままにする。

Backlog Runner Heartbeat は、`implementing` / `testing` / `fixing` の active item がある間は新しい item を昇格しない。active item がまだ Runner に投入されていない場合、`/code2 Backlog item <item_id>` の実装依頼を RenCrow 内部へ投げ、`implementation` に `Backlog Runner Heartbeat started` を記録する。同じ item は marker により重複実行しない。Runner 起動に失敗した場合は、その item を `blocked` にし、`test_result` に起動失敗理由を残す。

全 Backlog を順次実装する仕組みの本体は Heartbeat である。Cron は外部から Heartbeat を起こす補助、hook は登録時通知や短い検査、Kill は停止・緊急遮断だけに使う。Backlog item の実装を進める目的で Kill を使ってはならない。

標準の状態遷移は次の通り。

```text
open
  -> implementing   Backlog Intake Heartbeat が作業化し、Workstream へ昇格
  -> testing        実装差分または提案ができ、検証中
  -> fixing         検証失敗またはレビュー指摘により修正中
  -> blocked        依存、環境、承認、外部 credential、仕様未確定で停止
  -> ok             検証結果と OK 判断が揃った完了状態
  -> rejected       実装しないと判断した終了状態
```

`ok` にするには、少なくとも `test_result`、`check_ok=true`、`checked_by` が必要である。実コード変更を伴う item は、必要に応じて Workstream Artifact、Sandbox Gate Log、commit、PR、live verification のいずれかを `implementation` に記録する。ユーザー明示の承認が必要な変更は、Backlog から直接 apply せず Sandbox / AI Workflow の承認経路を通す。

## 付録H: Instruction Steering 配置方針

RenCrow は、AI エージェントへの指示、運用手順、自動化、実行 API を同じ場所へ詰め込まない。指示は性質ごとに、次の層へ分けて配置する。

| 層 | RenCrow での置き場所 | 用途 | 使ってはいけない用途 |
|---|---|---|---|
| 常時前提 | `AGENTS.md`, module `AGENTS.md` | module 境界、repo 構造、正本仕様の入口、必須の作業前提 | 長い手順、特定作業だけの詳細 runbook |
| 条件付き制約 | `rules/`, path scoped rules | Viewer / STT / TTS / LLM / security / testing など、対象領域にだけ効く制約 | 全作業に無関係な制約の常時注入 |
| 手順 | `skills/`, `workspace/skills`, `prompts/skills` | 未実装監査、Backlog 登録、再ビルド再起動、live Viewer 検証、release check などの定型手順 | 絶対に自動発火すべき処理 |
| 隔離調査 | Subagent / SuperAgent / Heavy Worker | 広域検索、ログ深掘り、依存監査、仕様差分調査など、途中経過が多い調査 | 親会話で逐次判断したい作業 |
| 決定的自動化 | Hook / scheduler / event handler | 編集後の検証、危険操作 block、完了通知、Backlog 候補漏れ検出など、モデル判断に任せず発火させる処理 | 意味判断が必要な設計判断そのもの |
| 表現調整 | output style / response style | 返答の形、簡潔さ、説明量など | 安全制約、破壊的操作禁止、検証義務 |
| 一時追加指示 | append system prompt 相当の起動時追加 | その実行だけに必要な補足方針 | repo 恒久ルール、後続セッションへ残すべき仕様 |
| 永続台帳 / 実行面 | Backlog API, Workstream, Sandbox, Repair Plane, AI Workflow | ルールや手順の結果を保存し、実行・承認・検証へ引き渡す | ルール本文や長い手順の置き場 |

配置判断の原則は次の通り。

1. 「常に知っていないと危険」な短い前提は `AGENTS.md` に置く。
2. 「この path / module に触る時だけ必要」な制約は `rules/` に置く。
3. 「毎回同じ順序で実行する」作業は skill にする。
4. 「中間結果が多く、親会話に戻すのは要約でよい」作業は subagent / Heavy Worker に渡す。
5. 「必ず起きなければならない」処理は hook / scheduler / event handler / permission guard として実装する。
6. 「あとで扱う項目」は Backlog に保存し、「実行する作業」は Workstream / Repair / Sandbox / SuperAgent / AI Workflow へ渡す。
7. 仕様や運用判断は、最終的に正本仕様または module docs へ戻す。

Backlog 運用はこの配置方針に従う。Backlog 登録義務は `AGENTS.md` または正本仕様に置くルールであり、登録手順は skill、登録漏れ検出や定期棚卸しは hook / scheduler、広域未実装探索は subagent、永続保存は `/viewer/backlog` API が担当する。

この方針により、長い手順を常時 context に載せ続けること、重要な自動処理をモデルの任意判断だけに任せること、実装予定項目を会話ログだけに残すことを禁止する。
