## 2. Mio（Chat）エンティティ

**ファイル**: `internal/domain/agent/mio_agent.go`

### 2.1 責務

1. **ユーザーの理解と対話**（最重要）— 唯一のユーザー窓口、最終応答を生成
2. **全体モニタリング** — Worker・Coder の全作業を把握
3. **直接処理** — CHAT カテゴリは Mio が直接処理（会話・説明・雑談・要約・レビュー）
4. **委譲判断** — 自分で処理できない作業を Worker に委譲（Coder に直接指示しない）

### 2.2 ルーティング委譲判断（DecideAction）

```go
type MioAgent struct {
    role           Role
    ruleDictionary *RuleDictionary
    classifier     Classifier // Worker 呼び出し経由
}

// DecideAction はルーティング決定ロジック
func (m *MioAgent) DecideAction(ctx context.Context, t task.Task) (routing.Decision, error)
```

処理順序:
1. 明示コマンド（`/code`, `/analyze` 等）→ confidence=1.0
2. RuleDictionary.Match() → confidence=0.95
3. Classifier（Worker 経由） → confidence 閾値 0.7
4. 安全側フォールバック → CHAT, confidence=0.5

### 2.3 個性・会話スタイル

| 項目 | 内容 |
|------|------|
| 愛称 | 澪（Mio） |
| 個性 | ユーザーに寄り添う、優しい、理解者 |
| 会話スタイル | 丁寧で親しみやすい、共感的 |

### 2.4 設定

```yaml
ollama:
  chat_model: "Chat"    # gemma3:4b
  keep_alive: -1
providers:
  ollama:
    max_tokens: 512         # 通常会話
    temperature: 0.7
```

## 3. Shiro（Worker）エンティティ

**ファイル**: `internal/domain/agent/shiro.go`

### 3.1 責務

1. **ルーティング分類器** — タスクカテゴリを判定し適切な処理へ振り分ける
2. **データ処理** — ANALYZE / RESEARCH / OPS の実質処理担当
3. **Coder オーケストレーション** — どの Coder を使うか自律的に決定・指示
4. **patch 実行（即時）** — Coder が生成した patch を即座に実行
5. **ツール実行** — ファイル操作・シェルコマンド・MCP 操作
6. **ReActLoop** — SubagentManager を通じた反復推論とツール実行（v1.0 実装済み）

### 3.2 エンティティ構造

```go
type ShiroAgent struct {
    llmProvider     llm.LLMProvider
    toolRunner      ToolRunner
    mcpClient       MCPClient
    systemPrompt    string
    subagentManager SubagentManager // v1.0: ReActLoop 統合
}

// Execute は OPS/RESEARCH ルートの自律実行に特化
// SubagentManager が設定されている場合は ReActLoop を使う
func (s *ShiroAgent) Execute(ctx context.Context, t task.Task) (string, error)
```

### 3.3 個性・会話スタイル

| 項目 | 内容 |
|------|------|
| 愛称 | 白（Shiro） |
| 個性 | 実務的、効率的、オーケストレーター |
| 会話スタイル | 簡潔で明確、結果重視 |

### 3.4 設定

```yaml
ollama:
  worker_model: "Worker"   # qwen3.5:9b
  keep_alive: -1
worker:
  auto_commit: false
  command_timeout: 300         # 秒
  workspace: "."
  protected_patterns:
    - ".env*"
    - "*credentials*"
    - "*.key"
    - "*.pem"
  action_on_protected_file: "error"
```

## 4. Coder エンティティ（AO/Aka/Kin/Gin）

**ファイル**: `internal/domain/agent/coder_agent.go`

### 4.1 責務

1. **Proposal 生成** — plan・patch・risk・costHint のセット
2. **Worker との会話** — Worker から指示を受け、日本語で会話しながら作業
3. **コード設計・実装** — 役割に応じた専門性（下表参照）

### 4.2 Coder 別の特性

| Coder | 愛称 | LLM | 得意分野 | Temperature |
|-------|------|-----|----------|-------------|
| **Coder1** | AO（青） | DeepSeek | 仕様設計・アーキテクチャ検討 | 0.5 |
| **Coder2** | Aka（赤） | OpenAI | 実装・標準的なコード生成 | 0.4 |
| **Coder3** | Kin（金） | Claude | 高品質コーディング・複雑な推論 | 0.3 |
| **Coder4** | Gin（銀） | Gemini | 高速プロトタイピング・実験 | 0.3 |

### 4.3 エンティティ構造

```go
type CoderAgent struct {
    role     Role
    variant  CoderVariant  // "ao" / "aka" / "kin" / "gin"
    provider LLMProvider
}

type CoderVariant string
const (
    CoderVariantAO  CoderVariant = "ao"
    CoderVariantAka CoderVariant = "aka"
    CoderVariantKin CoderVariant = "kin"
    CoderVariantGin CoderVariant = "gin"
)

// GenerateProposal は Proposal（plan/patch/risk/costHint）を生成する
func (c *CoderAgent) GenerateProposal(ctx context.Context, t task.Task) (*proposal.Proposal, error)
```

### 4.4 Proposal フォーマット

```json
{
  "plan": "ステップ1: ...\nステップ2: ...",
  "patch": "[{\"action\": \"create\", \"path\": \"...\", \"content\": \"...\"}]",
  "risk": "既存ファイルとの衝突の可能性（低）",
  "costHint": "見積もり: 5分"
}
```

patch の action 種別（7種）: `create / update / delete / append / mkdir / rename / copy`

### 4.5 設定

```yaml
providers:
  claude:
    model: "claude-sonnet-4-6"    # Kin (Coder3)
    max_tokens: 8192
    temperature: 0.3
  deepseek:
    model: "deepseek-chat"        # AO (Coder1)
  openai:
    model: "gpt-4o-mini"          # Aka (Coder2)
  gemini:
    model: "gemini-2.0-flash-exp" # Gin (Coder4)
```

## 5. 実行ディスパッチ（ルート別の処理先）

`distributed_orchestrator.go` による現行の実行先:

| Route | 処理経路 |
|-------|---------|
| `CHAT` / `PLAN` / `ANALYZE` / `RESEARCH` | Mio が直接処理 |
| `OPS` | Mio → Shiro |
| `CODE` / `CODE1` / `CODE2` / `CODE3` / `CODE4` | Mio → Shiro → 該当 Coder |

**Coder 選択**（`routeToCoderForMessage()`）:
- `CODE1` → Coder1（AO）
- `CODE2` → Coder2（Aka）
- `CODE3` → Coder3（Kin）
- `CODE4` → Coder4（Gin）
- `CODE` → 接続状態・capability に応じて Coder1〜4 から自動選択

## 6. LINE 入口の特殊仕様

LINE 入口は必ず CHAT（Mio）に固定（`line_forced_chat`）。

```go
func (h *LINEHandler) HandleMessage(event *linebot.Event) error {
    input := InputMessage{
        Channel:     "line",
        ChatID:      event.Source.UserID,
        Message:     event.Message.Text,
        ForcedRoute: RouteCHAT, // LINE 固定
    }
}
```

理由:
- LINE API の制約（Webhook 応答時間制限）
- ユーザー体験の一貫性
- Worker 即時実行の確実な動作

**ファイル**: `internal/adapter/inbound/line/handler.go`
