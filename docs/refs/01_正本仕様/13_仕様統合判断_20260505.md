# RenCrow 仕様統合判断メモ

**作成日**: 2026-05-05
**対象**: 現行正本仕様と 2026-05-01 最新追加仕様のアンマッチ
**ステータス**: 採用判断メモ

この文書は、現行正本仕様と最新追加仕様の衝突点を洗い出し、どちらを採用するかを決めるためのメモである。
原則として、ユーザー体験・既存実装資産・ローカルファースト方針を優先する。

## 1. 結論

現時点では、最新追加仕様をそのまま全面採用するのではなく、以下の方針で統合する。

| 領域 | 採用判断 | 理由 |
|---|---|---|
| プロダクト目的 | 最新仕様へ寄せる | 単なる軽量アシスタントより、記憶OSつき会話基盤のほうが現在の構想に合う |
| 実装ランタイム | 現行Go実装を維持 | Viewer、IdleChat、TTS、ConversationEngine など既存資産が大きい |
| LangGraph | 概念だけ採用、直導入は保留 | Go実装を捨てて Python/LangGraph 中心へ移すコストが高い |
| キャラクター体系 | 現行6エージェントを採用 | ユーザーの直近方針は mio / shiro / aka / ao / gin / kin |
| ルミナ人格 | Chat人格として取り込む | ルミナは別名キャラではなく、Chat の会話品質・口調設計として扱う |
| LLM接続 | 最新仕様を採用 | Ollama ではなく MLX OpenAI互換ローカルサーバへ切り替える方針が明確 |
| LLM alias | 最新仕様を採用 | model名は `Chat` / `Worker` / `Wild` で切り替える |
| 記憶方針 | 最新仕様へ寄せる | staging / validator / namespace 分離は記憶汚染を防ぐため必要 |
| hot store | SQLiteを優先、Redisは任意化 | ローカルPC中心MVPでは SQLite のほうが運用が単純 |
| cold store | DuckDB継続、Parquetは段階導入 | DuckDB実装は既にあるため壊さず、アーカイブ用途でParquetを追加する |
| VectorDB | Qdrant継続 | 既存実装に寄せる。FAISSは追加しない |
| 外部取得 | 最新仕様を採用 | Source Registry -> staging -> validator -> promoter は安全上必要 |
| UI | 現行Viewerを拡張 | 3D UI先行ではなく、Memory Inspector と参照表示を先に作る |

## 2. アンマッチと採用判断

### 2.1 基本目的

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| 目的 | 超軽量AIアシスタント | 記憶OSつきマルチキャラクター会話基盤 | 最新仕様 |
| ユーザー窓口 | Mio 1つ | 複数キャラが自然に会話 | 折衷 |
| クラウド依存 | Coder等でクラウド利用 | ローカルファースト | 最新仕様 |

判断:

- RenCrow の方向性は、最新仕様の「ローカル記憶OS」へ寄せる。
- ただし、ユーザーへの主窓口は散らさない。基本は `Chat` が最終応答をまとめる。
- 複数エージェントは裏側で分担し、必要なときだけ UI に見せる。

### 2.2 エージェント名・人格

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| 会話役 | Mio | ルミナ | `Chat = mio`、人格設計にルミナ要素を取り込む |
| 実務役 | Shiro | クラリス / ノクス相当 | `Worker = shiro` |
| 創作役 | 未整理 | Wild 相当 | `Wild` を追加 |
| Coder | AO / Aka / Kin / Gin | 仕様外 | 現行維持 |
| ユーザー利用エージェント | mio / shiro / aka / ao / gin / kin | ルミナ / クラリス / ノクス | 現行6エージェント |

判断:

- 正式な運用名は `mio / shiro / aka / ao / gin / kin` を維持する。
- `Chat` は `mio` の役割 alias とする。
- `Worker` は `shiro` の役割 alias とする。
- `Wild` は創作系ローカルモデルの役割 alias として新設する。
- ルミナ / クラリス / ノクスは、現時点では独立キャラ名として採用しない。人格・会話設計の参照名として扱う。

推奨対応表:

| 役割 alias | 運用エージェント | モデル | 用途 |
|---|---|---|---|
| `Chat` | `mio` | Gemma E4B | 会話、音声UI、最終応答、ルミナ的な自然対話 |
| `Worker` | `shiro` | Qwen3.6 35B 通常版 | 実務処理、要約、整理、RAG、通常画像解析 |
| `Wild` | `shiro` から呼び分け、または専用slot | Qwen3.6 35B Heretic | 物語、画像プロンプト、創作画像解析 |
| `Coder1` | `aka` | 既存設定 | 実装案・patch |
| `Coder2` | `ao` | 既存設定 | 実装案・patch |
| `Coder3` | `gin` | 既存設定 | 実装案・patch |
| `Coder4` | `kin` | 既存設定 | 実装案・patch |

### 2.3 会話ランタイム

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| 中核 | Go `MessageOrchestrator` | LangGraph Turn Controller | 現行Go |
| 状態制御 | Go service / EventHub | LangGraph checkpoint | Go実装で概念を取り込む |
| キャラ分離 | Agent構造 | subgraph per character | Agent単位で維持 |

判断:

- LangGraph を本番ランタイムとして急に導入しない。
- 最新仕様の `Turn Controller`、`Recall Orchestrator`、`Persona Prompt Builder`、`Memory Candidate Extractor` は、Go 側の責務名として取り込む。
- 将来 LangGraph を使う場合も、まずは外部実験ランタイムとして扱い、Go の正本ランタイムを壊さない。

理由:

- 現行実装には Viewer / IdleChat / TTS / STT / Worker / Coder 連携がすでにある。
- LangGraph 導入は言語境界と運用プロセスを増やし、今すぐの価値より移行リスクが大きい。

### 2.4 LLM接続

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| ローカル推論 | Ollama | MLX OpenAI互換 | 最新仕様 |
| provider | Ollama provider 直生成 | OpenAI互換ローカル provider | 最新仕様 |
| model alias | `Chat`, `Worker` | `Chat`, `Worker`, `Wild` | 最新仕様 |
| 同時実行 | 未整理 | 低並列・長timeout・warmup | 最新仕様を調整採用 |

判断:

- Chat / Worker 主経路は MLX OpenAI互換ローカル provider に切り替える。
- RenCrow 側に MLX 専用 provider は作らない。
- `Chat` / `Worker` / `Wild` を model alias として扱う。
- timeout は 120秒以上を標準にする。
- 起動時に `Chat` / `Worker` / `Wild` へ `max_tokens: 1` の warmup を送る。

同時リクエスト方針:

- 6エージェントを使っても、同一MLXサーバへ無制限に同時送信しない。
- 最初の推奨値は、global concurrency `2`、model別 concurrency `1`。
- Chat の応答体感を優先し、Worker / Wild は queue で待たせる。
- MLXサーバをモデル別に別プロセス化できる場合だけ、model別 concurrency を増やす。

### 2.5 記憶ストア

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| short/hot | Redis | SQLite WAL | SQLite優先 |
| mid/cold | DuckDB | DuckDB + Parquet | DuckDB継続、Parquet追加 |
| vector | Qdrant想定 | FAISS または Qdrant | Qdrant |
| レイヤー名 | 短期/中期/長期 | L0/L1/L2/L3 | 最新仕様 |

判断:

- 新規の hot store は SQLite WAL を標準にする。
- Redis は削除前提ではなく、複数プロセス・常時Web UI・高速pubsubが必要な場合の optional backend とする。
- DuckDB は継続し、Parquet は月次/長期アーカイブから段階導入する。
- Vector sidecar は既存実装に合わせて Qdrant を優先する。FAISSは当面採用しない。
- 呼称は L0/L1/L2/L3 に寄せる。

推奨マッピング:

| 最新仕様 | 採用ストア | 備考 |
|---|---|---|
| L0 現在会話 | Go runtime + rolling summary | LangGraph checkpoint は使わない |
| L1 今日の記憶 | SQLite WAL | 今日の会話、検索キャッシュ、イベントログ |
| L2 今月の記憶 | DuckDB | 月次要約、topic cluster |
| L3 長期記憶 | DuckDB + Qdrant + Parquet | canonical memory、長期archive |

### 2.6 記憶確定・namespace

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| namespace | 未統一 | `conv:` / `user:` / `char:` / `kb:` | 最新仕様 |
| 記憶昇格 | 未統一 | `observed -> candidate -> confirmed` | 最新仕様 |
| 本番DB write | Workerが扱う余地あり | validator通過後のみ | 最新仕様 |

判断:

- namespace分離は採用する。
- 推測記憶の直接確定は禁止する。
- まず `observed` / `candidate` / `confirmed` の状態を schema に入れる。
- `user:<uid>` への確定保存は明示操作または validator 通過後に限定する。

### 2.7 外部取得・ニュース

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| RSS | Glossary / IdleChat 用途 | Source Registry 管理 | 最新仕様 |
| 検索API | research用途 | registry-first / search-last | 最新仕様 |
| raw/summary | 未分離 | raw_text と summary_draft を分離 | 最新仕様 |
| validator | 未実装 | 必須 | 最新仕様 |

判断:

- Source Registry -> Scheduler -> Fetcher -> Staging -> Validator -> Promoter を採用する。
- 検索APIは背骨にしない。
- RSS / 公式API / GitHub / Hugging Face / PyPI を優先する。
- 外部取得結果は即 official DB に入れない。

### 2.8 UI

| 観点 | 現行正本 | 最新仕様 | 採用 |
|---|---|---|---|
| Viewer | 実装済み | Memory Inspector等を追加 | 現行拡張 |
| 3Dキャラ | 未実装 | 候補あり | 後回し |
| Summary Review | IdleChat内に実装済み | 仕様外 | 現行維持 |
| 参照表示 | 未実装 | 必要 | 最新仕様 |

判断:

- UIは現行 Viewer を正として拡張する。
- 次に作るべきは 3Dキャラではなく、Memory Inspector、Recall参照表示、News Pack表示。
- キャラ選択UIは `mio/shiro/aka/ao/gin/kin` と `Chat/Worker/Wild` の関係が固定されてから入れる。

## 3. 採用しない、または保留する項目

| 項目 | 判断 | 理由 |
|---|---|---|
| LangGraph全面移行 | 保留 | Go実装資産が大きく、移行効果がまだ不確定 |
| ルミナ / クラリス / ノクスを正式名にする | 採用しない | ユーザー運用は6エージェント名で固まっている |
| FAISS追加 | 採用しない | Qdrant想定実装があり、二重管理になる |
| 3DキャラUI先行 | 保留 | 記憶・参照・運用UIのほうが優先度が高い |
| クラウドLLM完全排除 | 採用しない | Coder用途では既存クラウド連携を残す価値がある |
| MLX専用provider | 採用しない | OpenAI互換providerで足りる |

## 4. 仕様正本への反映方針

次に正本仕様へ反映する場合、以下の順で行う。

1. `Chat` / `Worker` / `Wild` の alias を §3 エージェント定義へ追加する。
2. `Ollama` 固定記述を、`local OpenAI-compatible provider` 中心へ改める。
3. Mio / Shiro / AO / Aka / Kin / Gin を正式運用名として維持し、ルミナ等は人格参照名に下げる。
4. 会話エンジン §6 を L0/L1/L2/L3 名称へ寄せる。
5. `Source Registry`、`Staging`、`Validator`、`Promoter` を新しい正本セクションとして追加する。
6. Viewer仕様へ Memory Inspector / Recall参照表示 / News Pack を追加する。

## 5. 実装優先順位

### Phase 1: LLM切り替え

- OpenAI互換ローカル provider を Chat / Worker 主経路で使えるようにする。
- model alias `Chat` / `Worker` / `Wild` を設定化する。
- timeout 120秒以上、warmup、concurrency queue を入れる。

### Phase 2: 記憶の汚染防止

- SQLite hot store schema を追加する。
- namespace `conv:` / `user:` / `char:` / `kb:` を導入する。
- `observed` / `candidate` / `confirmed` を導入する。
- staging schema と validator を先に作る。

### Phase 3: Recall強化

- RecallPack を L0/L1/L2/L3 表記へ寄せる。
- token予算をモデルcontext比率で制御する。
- Search Cache を追加する。

### Phase 4: UI拡張

- Viewer に Memory Inspector を追加する。
- Recall参照表示を追加する。
- News Pack 表示を追加する。

## 6. 最終判断

RenCrow の上位方針は、最新仕様の **ローカルLLM主役 + 記憶OS + staging/validator** を採用する。

ただし、実装の土台は現行 Go 実装を維持する。
LangGraph、3キャラ正式名、FAISS、3D UI は、今すぐの正本にはしない。

短く言うと、採用すべき中心は次である。

```text
現行Go実装
  + MLX OpenAI互換接続
  + Chat / Worker / Wild alias
  + mio / shiro / aka / ao / gin / kin 運用名
  + L0/L1/L2/L3 記憶
  + SQLite hot store
  + DuckDB + Qdrant + Parquet cold/long store
  + Source Registry / staging / validator / promoter
  + Viewer Memory Inspector
```
