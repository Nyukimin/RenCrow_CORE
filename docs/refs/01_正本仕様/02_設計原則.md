## 1. 責務の三分割

```
対話は Chat、運営判断は Worker、実装は Coder、正本管理は Worker Core に帰属する。
この分離を崩してはならない。
```

### 1.1 各役割の担当/非担当

| 役割 | 担当すること | 担当しないこと |
|------|------------|--------------|
| **Chat** | ユーザー対話・応答生成・委譲判断・記録保存・音声連携・承認要求提示 | 自己拡張判断・Capability Registry 更新・system up |
| **Worker** | タスク分類・実行可否判定・自己拡張判断・Coder 選定・patch 実行・能力管理 | ユーザー向け自然対話・音声演出・最終表現 |
| **Coder** | BuildRequest 解釈・plan 生成・patch 生成・設計説明・複数案提示 | 自己拡張の意思決定・正本ブランチ管理・system up・Capability Registry 更新 |

### 1.2 Worker Core の役割

Worker のうち LLM ではなく決定論的な常駐プロセスとして実装される中核制御面。

- ジョブ管理・状態管理・承認状態・能力台帳（Capability Registry）
- system up 管理・Git 正本管理
- これらを LLM の会話状態に依存させず、永続状態として保持する

## 2. 指揮命令系統

```
ユーザー
  ↓ 会話
Mio（全体オーケストレーター）— Worker にのみ指示する
  ↓ 委譲
Shiro（Worker・Coder オーケストレーター）— Coder 選択を自律的に判断
  ↓ 指示
Coder1(AO) / Coder2(Aka) / Coder3(Kin) / Coder4(Gin)
```

**重要な原則**:
- Mio は Coder の存在と作業内容を**知っている**（モニタリング）が、**直接指示は出さない**
- Worker は Mio に Coder 選択を確認せず、**自律的に判断**する
- すべての会話は日本語（JSON フォーマットの `message` フィールドに自然言語を含める）
- すべての会話はモニタリング可能（Mio が全体把握）

---

## 3. Coder の立場

Coder は**提案者**であり、採用決定者ではない。

- Coder の成果物（plan/patch）は「候補コード」であり、正本ではない
- 採用判断・Git 正本管理・system up の実行は **Worker Core** が行う
- Coder は原則として破壊的操作を**直接実行しない**（plan/patch を生成し、実行は Worker が担当）

---

## 4. 自己拡張フロー

```
1. Chat がユーザー要求を受け、Worker に処理委譲
2. Worker が既存能力で処理可能か判定
3. 処理不可 → Worker が Capability Gap を検出
4. Worker が BuildRequest を生成し、適切な Coder に送る
5. Coder が plan/patch を返す（複数 Coder の比較も可能）
6. Worker が提案を評価（テスト・検証を行う場合もある）
7. 必要なら Chat 経由でユーザー承認を取る
8. Worker Core が採用を決定しコードを統合
9. Worker Core が Capability Registry を更新し新能力を有効化
10. Chat が結果をユーザーに説明する
```

## 5. Clean Architecture 4 層構造

```
┌──────────────────────────────────────┐
│  Adapter Layer（外部インターフェース）  │ ← LINE, Slack, CLI
└──────────────────────────────────────┘
           ↓ 依存（Use Case 呼び出し）
┌──────────────────────────────────────┐
│ Application Layer（Use Cases）       │
│  - MessageOrchestrator              │
│  - WorkerExecutionService           │
│  - RoutingService                   │
└──────────────────────────────────────┘
           ↓ 依存（Domain 呼び出し）
┌──────────────────────────────────────┐
│ Domain Layer（ビジネスロジック）       │
│  - MioAgent / ShiroAgent / CoderAgent│
│  - PatchCommand（Value Object）      │
│  - Task, Proposal（Value Object）    │
└──────────────────────────────────────┘
           ↑ 依存性逆転（interface）
┌──────────────────────────────────────┐
│ Infrastructure Layer（技術詳細）      │
│  - LLM Providers（Ollama/Claude 等） │
│  - ToolRunner（ファイル編集/コマンド）│
│  - Config Loader / Logger           │
└──────────────────────────────────────┘
```

**依存方向**: 外層 → 内層（Domain 層は何にも依存しない）

---

## 6. Chat と Worker の同居ルール

Chat と Worker は同一物理機体に置いてよい（**同居可、同一不可**）。

以下は必ず分離する:
- セッション状態 / 権限 / プロンプト
- 保存対象 / 正本状態 / ジョブ責務

## 7. DDD パターン

| パターン | 実装例 | 役割 |
|---------|--------|------|
| **エンティティ** | MioAgent, ShiroAgent, CoderAgent | 一意性を持つドメインオブジェクト |
| **値オブジェクト** | Task, JobID, Proposal, RoutingDecision, PatchCommand | 不変、値の等価性で比較 |
| **サービス** | ToolRunner, WorkerExecutionService | ドメインロジックの調整 |

## 8. セーフガード設計

Worker による即時実行でも、以下のセーフガードで安全性を確保する。

| セーフガード | 説明 | 必須/推奨 |
|------------|------|---------|
| **詳細ログ記録** | 実行前後に plan/patch・結果・変更ファイルをログ | 必須 |
| **Git auto-commit** | 実行前後に自動コミット（`git revert` で即座に巻き戻し可能） | 推奨 |
| **保護ファイルパターン** | `.env*`, `*credentials*`, `*.key`, `*.pem` への操作は即エラー | 推奨 |
| **ワークスペース制限** | ファイル操作は workspace 内のみ許可 | 必須 |
| **ドライモード** | `PICOCLAW_DRY_RUN=true` で実行内容のみログ出力 | オプション |
| **実行前サマリ表示** | 変更内容の概要を実行前にログ出力 | 推奨 |
