## 1. 概要

RenCrow v5.0/v5.1 の会話エンジンは Mio（Chat）に統合された 3 層記憶システム。
`cfg.Conversation.Enabled = true` のときのみ有効化される。

**キャラクター別の記憶対象**:

| キャラクター | Agent | 記憶の扱い |
|------------|-------|-----------|
| **Mio** | Chat | 常に記憶（主会話） |
| **Shiro** | Worker | タスク空き時のみ参加。Mio が知らない情報なら記憶 |
| **Aka/Ao/Gin** | Coder | 将来的に参加可能。Mio が知らない情報なら記憶 |

---

## 2. インフラ構成

| サービス | ポート | 役割 | 備考 |
|---------|--------|------|------|
| Redis | 6379 | 短期記憶（TTL: 24h） | |
| DuckDB | ファイル | 中期記憶 | `~/.rencrow/memory.duckdb` |
| Qdrant | **6334 (gRPC)** / 6333 (REST) | 長期記憶・KB（VectorDB） | gRPC を使うこと |
| Ollama | 11434 | Embedding | `nomic-embed-code:latest` |

**重要**: VectorDBStore は Qdrant の **gRPC ポート 6334** に接続する。REST ポート 6333 ではない。

```yaml
conversation:
  enabled: true
  redis_url: "redis://localhost:6379"
  duckdb_path: "/home/nyukimi/.rencrow/memory.duckdb"
  vectordb_url: "localhost:6334"       # gRPC ポート
  embed_model: "nomic-embed-code:latest"
  summary_model: "Chat"
```

## 3. 記憶ストア構成と lifecycle L番号

L0〜L4 は保存媒体ではなく lifecycle 上の位置で定義する。正本定義と物理ストア / コード命名の対応は `18_Memory_Lifecycle_Recall_Context.md#311-実装命名物理ストアとの対応` を参照する。本節の表は v5.0/v5.1 実装で利用するストア構成を示す。

| 領域 | 実装 | TTL | 用途 |
|----|------|-----|------|
| 短期（Short-term） | RAM / Redis | セッション内 / 24h | 直近の会話履歴 |
| 中期（Mid-term） | DuckDB | 数週間〜 | Thread サマリ・構造化データ |
| 長期（Long-term） | Qdrant (VectorDB) | 永続 | 意味検索・エピソード記憶 |
| KB | Qdrant (VectorDB) | 永続 | Web 検索結果・ドメイン別知識 |

## 4. ConversationEngine インターフェース（v5.1）

**ファイル**: `internal/domain/conversation/engine.go`

```go
type ConversationEngine interface {
    // ターン開始: recall を実行し RecallPack を返す
    BeginTurn(ctx context.Context, sessionID string, userMessage string) (*RecallPack, error)

    // ターン終了: user/mio の発話を保存
    EndTurn(ctx context.Context, sessionID string, userMessage string, response string) error

    // 現在のペルソナ状態を返す
    GetPersona() PersonaState

    // スレッド境界検出時に現在のスレッドをフラッシュ
    FlushCurrentThread(ctx context.Context, sessionID string) error

    // セッション状態取得
    GetStatus(ctx context.Context, sessionID string) (*ConversationStatus, error)

    // セッションリセット（日次カットオーバー）
    ResetSession(ctx context.Context, sessionID string) error
}
```

## 5. RecallPack（プロンプト注入用）

**ファイル**: `internal/domain/conversation/recall_pack.go`

```go
type RecallPack struct {
    ShortContext []Message         // 直近の会話履歴（RAM / Redis）
    MidSummaries []ThreadSummary   // Thread サマリ（DuckDB）
    LongFacts    []string          // 長期記憶 + KB スニペット（[KB] プレフィックス付き）
    KBSnippets   []string          // KB 断片（現行実装では LongFacts に統合）
    Persona      PersonaState      // Mio のペルソナ状態
    UserProfile  UserProfile       // ユーザープロファイル
    Constraints  PromptConstraints // トークン制約
}
```

`PromptConstraints` の現行デフォルト値:
- `MaxTotalTokens = 8192`
- `MaxPromptTokens = 4000`
- `MaxResponseTokens = 512`

**プロンプト構築順**（`RecallPack.ToPromptMessages()`）:
1. persona + user profile の system prompt
2. 過去文脈の system prompt（MidSummaries）
3. ShortContext の履歴
4. （MioAgent.Chat() 側で）現在の user message を追加

## 6. PersonaState / UserProfile

**ファイル**: `internal/domain/conversation/persona.go`, `user_profile.go`

```go
// PersonaState: Mio のペルソナ
type PersonaState struct {
    Name         string
    SystemPrompt string
    Tone         string
    Mood         string  // LLM による自己更新が可能（PersonaEditor）
}

// UserProfile: ユーザー情報
type UserProfile struct {
    UserID      string
    Preferences map[string]string
    Facts       []string
    UpdatedAt   time.Time
}
```

起動時: `conversation.NewMioPersona(cfg.Prompts.MioPersona)` で初期化
`UserProfile` は現行では in-memory キャッシュのみ（永続化なし）

## 7. Thread / Session の概念

| 単位 | 説明 | 対応する lifecycle |
|------|------|-------------|
| **Thread** | 会話の話題のまとまり（6〜8ターン） | L1 から L2 へ promotion |
| **Session** | プロセス再起動や割り込みを跨ぐまとまり（24h〜7d） | L1 / L2 |
| **Conversation** | 長期にわたるユーザーとの関係全体 | L3 / L4 |

スレッド境界検出: `ThreadBoundaryDetector`（オプション）
検出時: `FlushCurrentThread()` → DuckDB に Thread サマリを保存 → 新 Thread 作成

## 8. KB（ナレッジベース）機能

### 8.1 KB の構造

ドメイン別コレクション（Qdrant VectorDB）に保存:
- `SaveKB(domain, content)` — KB にデータを保存
- `SearchKB(query, domain)` — KB を検索

### 8.2 Web 検索結果の自動 KB 保存

Mio の `executeWebSearch()` が RESEARCH ルートで呼ばれると、検索結果を自動的に KB に保存する（`SaveWebSearchToKB()`）。

### 8.3 RAG 統合

`BeginTurn()` で自動的に KB 検索を実行し、結果を `LongFacts` に `[KB] ...` 形式で追加する。

## 10. Shiro / Coder の軽量メモリ（v1.0 設計済み・未実装）

Mio の ConversationEngine（Redis/DuckDB/Qdrant 3 層）とは**完全に独立**した仕組み。

- **LightMemory**: セッション単位のインメモリ短期会話履歴（goroutine-safe）
- プロセス再起動でリセット（意図的 — Mio が長期文脈を管理するため不要）
- 外部インフラ不要（純粋な Go struct）
- `maxTurns` で保持ターン数を制限（推奨: 3〜5）
