## 1. エージェントペルソナ v1.0

**ステータス**: 設計済み・未実装（`feature/RenCrow_Start` ブランチ）

### 1.1 目的

Shiro（Worker）および Coder エージェント（Aka/Ao/Gin）に**軽量なペルソナ（AgentPersona）**と**インメモリ短期会話履歴（LightMemory）**を付与する。

### 1.2 設計思想

- **AgentPersona は ConversationEngine ではない**: Mio の v5.1 ConversationEngine（Redis/DuckDB/Qdrant）とは完全に独立。外部インフラ不要、純粋な Go struct
- **LightMemory はインメモリ限定**: プロセス再起動でリセット（意図的）。Mio が長期文脈を管理するため不要
- **Persona は system prompt に前置**: 別メッセージではなく、system prompt 先頭にペルソナブロックを追加
- **後方互換**: `agents:` セクションが config.yaml になければ全機能無効

| 項目 | Mio | Shiro | Coder |
|------|-----|-------|-------|
| 記憶 | Redis/DuckDB/Qdrant 3層 | LightMemory（インメモリ） | LightMemory（インメモリ） |
| ペルソナ | PersonaState（LLM自己編集可） | AgentPersona（YAML固定） | AgentPersona（YAML固定） |
| 再起動後 | Redis から復元 | リセット | リセット |

### 1.3 AgentPersona 型

**ファイル**: `internal/domain/agent/persona.go`（新規）

```go
type AgentPersona struct {
    Name        string // 表示名（例: "Shiro", "Aka"）
    Personality string // ペルソナ記述。system prompt 先頭に前置
    Tone        string // 口調ヒント（"calm", "analytical"等）将来のTTS連携用

// Personality が空なら taskPrompt をそのまま返す（後方互換）
// 合成順序: Personality + "\n\n" + taskPrompt
func (p AgentPersona) BuildSystemPrompt(taskPrompt string) string
```

### 1.5 system prompt 構築ルール

```
[1] AgentPersona.Personality  ← 人格・口調の記述
\n\n
[2] Task Prompt               ← 技術指示（worker.md / coder.md）
```

LightMemory を含む完全メッセージ列:

```
messages = [
    {role: "system",    content: persona + "\n\n" + taskPrompt},
    {role: "user",      content: "前回の続きで..."},    // LightMemory[0]
    {role: "assistant", content: "了解。前回は..."},    // LightMemory[0]
    {role: "user",      content: "今回のリクエスト"},   // 現在
]
```

### 1.8 既知の制約（v1.0）

- SubagentManager（ReActLoop）経由の場合、LightMemory の過去ターンは注入不可（`SubagentTask` は SystemPrompt + Instruction のみ受け取るため）
- Persona の自己編集は未実装（YAML 手動変更 + プロセス再起動が必要）
- LightMemory のセッション数に上限なし（max_turns が小さいため深刻にならない）

## 2. SubagentManager v1.0

**ステータス**: 実装済み（2026-03-19）

### 2.1 概要

Worker 役の `ShiroAgent` が 1 回の LLM 応答で終わらず、tool calling を使って複数回の推論とツール実行を回す ReAct ループ実装。

**責務**:
1. `ToolCallingProvider.Chat()` を使った反復推論
2. `ToolRunner.ExecuteV2()` を使った tool call 実行
3. `ShiroAgent.Execute()` からの同期呼び出し

### 2.2 Domain インターフェース

**ファイル**: `internal/domain/agent/subagent.go`

```go
type SubagentTask struct {
    AgentName    string
    Instruction  string
    SystemPrompt string
}

type SubagentResult struct {
    AgentName  string // 実行したサブエージェント名
    Output     string // 最終出力テキスト
    Iterations int    // ReActループの反復回数
}

type SubagentManager interface {
    RunSync(ctx context.Context, task SubagentTask) (SubagentResult, error)
}
```

### 2.3 ToolCallingProvider

**ファイル**: `internal/domain/llm/provider.go`

```go
type ToolCallingProvider interface {
    LLMProvider
    Chat(ctx context.Context, req ChatRequest) (ChatResponse, error)
}
```

### 2.4 ToolLoop（ReAct ループ）

**ファイル**: `internal/application/toolloop/loop.go`

反復ごとの流れ:

1. `provider.Chat(messages, tools)` を呼ぶ
2. assistant メッセージを履歴へ追加
3. `tool_calls` がなければ `Content` を返して終了
4. `tool_calls` があれば各 call を `toolRunner.ExecuteV2()` で実行
5. 実行結果を `role="tool"` メッセージとして履歴へ追加
6. 次 iteration へ進む

終了条件: final assistant response / `ctx.Done()` / `MaxIterations` 到達

`MaxIterations` 到達時は最後の assistant message を返す（error にしない）。

### 2.5 Shiro 統合

**ファイル**: `internal/domain/agent/shiro.go`

```go
// subagentManager != nil のとき、通常の単発 Generate() の代わりに RunSync() を呼ぶ
result, err := s.subagentManager.RunSync(ctx, SubagentTask{
    AgentName:    "shiro",
    Instruction:  t.UserMessage(),
    SystemPrompt: s.systemPrompt,
})
```

### 2.6 設定

```go
type SubagentConfig struct {
    Enabled       bool
    MaxIterations int    // デフォルト: 10
    Provider      string // "ollama" / "claude" / "openai" / "deepseek"
    Model         string
}
```

### 2.7 Tool 定義と再帰防止

`ToolRunner.ToolDefinitions()` は `subagent` ツール自体を除外してLLMに渡す（再帰呼び出し防止）。

## 3. ケイパビリティ適応 v1.0

**ステータス**: 設計済み・未実装

### 3.1 概要

現行（v4.0）は config.yaml に LLM・ツール構成が静的記述されている。v4.1 相当の本機能では、各ノードが**起動時に自分のケイパビリティを検出**し、それに合わせて動作を決定する。

### 3.2 NodeCapabilities 構造体

**ファイル**: `internal/domain/capability/capabilities.go`（新規）

```go
type NodeCapabilities struct {
    NodeID   string
    Platform PlatformInfo
    Memory   MemoryInfo
    LLMs     []LLMCapability
    Tools    []ToolCapability
}

type LLMCapability struct {
    ProviderName string
    ModelName    string
    MaxContext   int
    MaxMemoryMB  uint64
    Available    bool
    Quality      int // 1〜5（ルーティング優先順位）
}
```

### 3.3 動作プロファイル

| 条件 | プロファイル | 挙動 |
|------|------------|------|
| Claude API 疎通OK | `coder-high` | Coder3 として高品質推論 |
| OpenAI / DeepSeek 疎通OK | `coder-mid` | Coder2 / Coder1 として実装 |
| Ollama のみ（8GB+） | `coder-local` | ローカル Coder として動作 |
| Ollama のみ（4GB未満） | `worker-only` | Shiro のみ（Coder 不可） |
| API なし・Ollama なし | `unavailable` | 起動拒否 |

### 3.4 LLM 選択アルゴリズム

```go
// route ごとの最低品質要件で最適 LLM を選択。品質不足時は縮退動作
func SelectLLM(nodes []NodeCapabilities, route Route) (NodeID string, provider LLMCapability, err error)
```

| Route | minQuality |
|-------|-----------|
| `CODE3` | 5 |
| `CODE2` | 4 |
| `CODE1` | 3 |
| `OPS` / `CHAT` | 2 |

縮退: `CODE3 → CODE2相当 → ユーザー通知`

### 3.5 ToolRegistry

```go
type ToolRegistry interface {
    Register(ctx context.Context, entry ToolEntry) error
    ListForPlatform(ctx context.Context, platform string) ([]ToolEntry, error)
    Get(ctx context.Context, name string) (ToolEntry, error)
    Close() error
}
```

永続化: 既存 DuckDB インフラを拡張（`tool_registry` テーブル）

登録されたツールは `workspace/tools/<name>.sh` が存在すれば即座に利用可能になる。

### 3.6 実装プラン

| Phase | 内容 |
|-------|------|
| Phase 1 | NodeCapabilities 構造体 + CapabilityDetector（OS/メモリ/LLM 疎通） |
| Phase 2 | ToolRegistry インターフェース + DuckDB 実装 |
| Phase 3 | SelectLLM() + MessageOrchestrator ルーティング拡張 |
| Phase 4 | Shiro → Coder ツール共有フロー |
