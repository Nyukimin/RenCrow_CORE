## 2. P0: LLM接続

### 2.1 MLX OpenAI互換 provider 主経路

実装状況:

- `local_llm.enabled=true` の場合、Chat / Worker / Wild は local OpenAI互換 provider で生成される。
- `local_llm.enabled=true` の場合、Ollama設定なしでも config validation を通過できる。
- 既存Ollama運用は `local_llm.enabled=false` のまま維持する。

実装方針:

- MLX専用 provider は作らない。実装済み。
- 既存の OpenAI互換 provider を local base URL で使う。実装済み。
- provider種別は `local_openai`。実装済み。
- model名は `Chat` / `Worker` / `Wild` を送る。実装済み。

最小受け入れ条件:

- configだけで Chat / Worker を MLXサーバへ向けられる。
- Ollamaなしでも Chat / Worker が起動できる。
- Wild用途の呼び出し先を生成できる。

### 2.2 `Chat` / `Worker` / `Wild` alias routing

実装状況:

- `Chat` / `Worker` / `Wild` は `local_llm` config の model alias として実装済み。
- `mio` は Chat provider、`shiro` は Worker provider に接続済み。
- Wild provider は生成済み。`/wild` 明示指定では `RouteWILD` を通じてlocal modeのWild agentを呼ぶ。
- 物語、画像プロンプト、創作用画像解析のキーワードでは用途判定により `RouteWILD` を選ぶ。
- 分散実行側のWild接続は未実装。

実装方針:

| alias | 実装上の役割 | 用途 |
|---|---|---|
| `Chat` | `mio` の最終応答 provider | 会話、音声UI、自然なやり取り |
| `Worker` | `shiro` の実務 provider | 要約、整理、RAG、通常画像解析 |
| `Wild` | 創作用 provider | 物語、画像プロンプト、創作画像解析 |

最小受け入れ条件:

- ルーティングが `Wild` を選べる。明示指定と用途判定は実装済み。
- 明示指定または用途判定で Wild provider を呼べる。
- Chat / Worker のエイリアスはユーザー向け名として安定する。

### 2.3 timeout / warmup / concurrency

実装状況:

- MLX初回ロードを想定した 120秒以上の timeoutを追加済み。
- 起動後 warmupを追加済み。
- global concurrency と model別 concurrency queue を追加済み。

実装方針:

| 項目 | 初期値 |
|---|---|
| request timeout | 120秒以上 |
| global concurrency | 2 |
| per model concurrency | 1 |
| warmup | `Chat`, `Worker`, `Wild` へ `max_tokens: 1` |
| 優先度 | Chat > Worker > Wild |

最小受け入れ条件:

- 6エージェント利用時も同一MLXサーバへ無制限並列送信しない。
- Chat応答がWorker/Wildの長い処理で詰まりにくい。

## 3. P1: 記憶OSの土台

### 3.1 SQLite hot store

実装状況:

- 会話L1/hot storeとして `L1SQLiteStore` を追加済み。
- `conversation.l1_sqlite_path` が設定されている場合、`Store()` 時に同じメッセージを SQLite にミラー保存する。
- 保存時は `conv:<thread_id>` namespace、`observed` state、`L1` layer を付与する。
- Redis active thread が取れない場合、L1 SQLite の直近会話イベントをRecall候補として返す。
- Search Cache はL1 SQLiteに部分実装済み。
- Event Log は `l1_event_log` として保存/取得API/Viewer表示まで部分実装済み。
- 会話保存、Search Cache保存、記憶状態更新はEvent Logへ自動記録する。
- Event LogはMemoryタブでnamespace別に確認できる。

実装方針:

- Redisを即削除せず、SQLiteを新規標準hot storeにする。部分実装済み。
- Redisは optional backend として残す。継続中。
- L1用途を SQLite WAL で扱う。実装済み。

最小受け入れ条件:

- 今日の会話イベントをSQLiteへ保存できる。
- `thread_id`, `namespace`, `memory_state`, `created_at`, `updated_at` を持つ。
- アプリ再起動後も今日の記憶をRecall候補にできる。

### 4.3 validator / promoter

部分実装:

- `ValidateStagingItem()` で L1 staging item を検証できる。
- 通過時は `validated`、失敗時は `rejected` に更新する。
- 検証結果は `staging.item_validated` として Event Log に残す。
- 現時点の検証項目は、URL妥当性、raw_hash重複、日付妥当性、source trust、license note、memory type、sensitive marker。
- `PromoteValidatedStagingItemToMemory()` で、validated済みstagingだけを `user:` / `char:` / `kb:` 等の `confirmed` memoryへ昇格できる。
- `PromoteValidatedStagingItemToNews()` で、validated済み外部取得stagingだけを News DBへ昇格できる。
- 昇格結果は `memory.promoted_from_staging` として Event Log に残す。

未実装:

- Source Registryのtrust scoreと連動した自動検証は実装済み。
- DuckDB archive tableへのpromoter同期とQdrant Knowledge DB同期は実装済み。
- 検索結果や外部取得結果を各official storeへ入れるための型別promoter。

実装方針:

validatorで見る項目:

- schema妥当性
- URL妥当性
- hash重複
- 日付妥当性
- source trust
- sensitivity
- license note
- memory type

最小受け入れ条件:

- validator失敗データは official store に入らない。L1 staging itemのrejected化は実装済み。
- promoterは validator通過済みだけを昇格する。`confirmed` memory / News DBへの昇格では実装済み。
- Codex/Coworkは staging まで。本番DB直接writeは禁止。

## 5. P2: Recall / RAG / News

### 5.1 Search Cache

部分実装:

- L1 SQLiteにSearch Cacheを持つ。
- query正規化、hash、retrieved_at、expires_at、source_urlを管理する。
- TTL内の同一queryはfresh hitとして取得できる。
- fresh hitはRecall Packの `SearchCacheSnippets` に出典付きで入る。
- MioのWeb検索はcache-firstで動き、miss時の検索結果をL1 Search Cacheへ保存する。
- ToolRunnerの `web_search` はV1/V2ともcache hookを参照でき、main起動時にConversation L1へ配線する。

実装済み:

- 類似query抑制は `GetSimilarFreshSearchCache()` で実装済み。
- Search Cacheの類似hitと手動invalidateは実装済み。

実装方針:

- 検索前に `GetFreshSearchCache` を確認し、hitした場合は外部検索を省略する。
- miss時だけ外部検索し、結果を `SaveSearchCache` に保存する。

最小受け入れ条件:

- 同一queryの短時間再検索を避ける。
- cache hitもRecall Packに出典付きで入れられる。

### 5.2 News DB / Daily Digest

- category別に `RecentNewsItems()` で取得できる。

- `BuildDailyDigest()` で日付・category別にNews DBから決定的digestを作成できる。
- `BuildDailyDigestForSlot()` で `morning` / `noon` / `evening` 別digestを作成できる。
- `RecentDailyDigests()` でdigestを取得できる。

- `RENCROW_PARQUET_EXPORT_DIR` を指定すると起動時と `RENCROW_PARQUET_EXPORT_INTERVAL_SEC` 間隔でParquet export jobが実行される。
- L1 staging保存時とpromoterからL1 memory / News DB / Knowledge DBのDuckDB archive table自動同期は実装済み。

最小受け入れ条件:

- 共通コアschemaのDuckDB/Parquet archive化。MVPのtable/export、Source Registry / promoter からの自動投入、JSONL初期投入CLIは実装済み。
- 映画・小説・音楽などの初期投入pipelineは `picoclaw knowledge import-core-jsonl <path>` でstaging投入できる。

実装方針:

- 共通コア JSONL importer からstagingへ投入し、既存validator/promoterでKnowledge DBへ昇格する。
- DuckDB FTS / Qdrant を組み合わせる。
- 最初は小さいdomainから始める。

最小受け入れ条件:

- `kb:<domain>` で保存・検索できる。L1保存と一覧取得は実装済み。
- 固有名詞検索と意味検索を併用できる。

### 5.4 RecallPack token予算制御

部分実装:

- `PromptConstraints.RecallBudgetRatio` を追加済み。
- `DefaultConstraints()` は `RecallBudgetRatio=0.10` を返す。
- `RecallPack.ApplyRecallBudget()` で中期要約、長期事実、KB、検索キャッシュ候補をcontext長比率でtrimできる。
- ConversationEngineはRecallPack返却前に予算制御を適用する。

未実装:

- 精密tokenizerを注入するための `TokenEstimator` 入口は実装済み。
- roleごとの詳細budget / 優先度調整。

実装方針:

- context長の約1割をRecall Pack目安にする。
- Chat / Worker / Wild のモデルごとに上限を分ける。

最小受け入れ条件:

- 長い記憶が増えても prompt が肥大化しない。概算tokenでは実装済み。
- どの候補を落としたかログに残る。
