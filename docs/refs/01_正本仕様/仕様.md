## 1. プロジェクト概要

### 1.1 目的

LINE / Slack からの指示を受け、複数の LLM を適切に使い分けて応答・実行する **超軽量 AI アシスタント**。

- メモリ使用量 **<10MB** を目標とし、$10 デバイスでも動作する
- ユーザーが直接会話する相手は **Mio（Chat）1 つ**。複数エージェントの存在はユーザーに意識させない
- コード生成・ファイル操作が必要な場合のみクラウド API を使い、それ以外はローカル LLM で完結する

### 1.2 実装制約

| 制約 | 値 | 備考 |
|------|-----|------|
| メモリ目標 | <10MB | Ollama モデル除く |
| Ollama MaxContext | 8192 トークン以下 | これを超えるモデルは NG |
| Ollama 常駐 | `keep_alive: -1` | Chat/Worker モデルは永続化必須 |
| API キー保管 | 環境変数のみ | 設定ファイル平文保存禁止 |
| 言語 | Go 1.23 | |
| アーキテクチャ | Clean Architecture 4 層 | domain / application / infrastructure / adapter |

### 1.3 用語

| 用語 | 定義 |
|------|------|
| **Chat** | ユーザーへの最終応答を生成する唯一の窓口役割 |
| **Worker** | 分類・実行・オーケストレーションを担う非対話役割 |
| **Worker Core** | Worker のうち LLM ではなく決定論的な常駐プロセスとして実装される中核制御面 |
| **Coder** | コード生成・実装案作成に特化した役割（plan + patch を返す） |
| **Router** | 入力をカテゴリに分岐させ、ワーカー呼び出しを制御するシステム部 |
| **LoopController** | ループ回数・停止条件・安全制約を管理し次処理を回す最終決定者（Router に含む） |
| **Proposal** | Coder が返す `plan + patch + risk + costHint` のセット |
| **Capability Registry** | RenCrow が現在どの能力を持つかを管理する台帳（Worker Core が保持） |
| **system up** | 新規・更新コードを正規能力として統合し、次回実行から利用可能な状態へ上げること |

## 2. 設計原則

### 2.1 責務の三分割

```
対話は Chat、運営判断は Worker、実装は Coder、正本管理は Worker Core に帰属する。
この分離を崩してはならない。
```

| 役割 | 担当すること | 担当しないこと |
|------|------------|--------------|
| **Chat** | ユーザー対話、応答生成、委譲判断、記録保存、音声連携 | 自己拡張判断、Capability Registry 更新、system up |
| **Worker** | タスク分類、実行可否判定、自己拡張判断、Coder 選定・統括、patch 実行 | ユーザー向け自然対話、音声演出、最終表現の感触調整 |
| **Coder** | BuildRequest の解釈、plan 生成、patch 生成、設計説明、複数案提示 | 自己拡張の意思決定、正本ブランチ管理、Capability Registry 更新、system up |

### 2.2 指揮命令系統

```
ユーザー
  ↓ 会話
Mio（全体オーケストレーター）
  ↓ 委譲（Worker にのみ指示する）
Shiro（Worker・Coder オーケストレーター）
  ↓ 指示
Coder1(AO) / Coder2(Aka) / Coder3(Kin) / Coder4(Gin)
```

- Mio は Coder の存在と作業内容を**知っている**（モニタリング）が、**直接指示は出さない**
- Worker は Mio に Coder 選択を確認せず、**自律的に判断**する
- すべての会話は日本語。JSON フォーマットの `message` フィールドに自然言語を含める

### 2.3 Coder の立場

Coder は **提案者** であり、採用決定者ではない。

- Coder の成果物（plan/patch）は「候補コード」であり、正本ではない
- 採用判断・Git 正本管理・system up の実行は **Worker Core** が行う
- Coder は原則として破壊的操作を**直接実行しない**

### 2.4 Chat と Worker の同居ルール

Chat と Worker は同一物理機体に置いてよい（**同居可、同一不可**）。
以下は必ず分離する：セッション状態 / 権限 / プロンプト / 保存対象 / 正本状態 / ジョブ責務

## 3. エージェント定義

### 3.1 エージェント一覧

| 役割 | 愛称 | LLM | 実体モデル | 温度 | 状態 |
|------|------|-----|----------|------|------|
| **Chat** | Mio（澪） | Ollama | gemma3:4b (Chat) | 0.7 | 常駐 |
| **Worker** | Shiro（白） | Ollama | qwen3.5:9b (Worker) | 0.3 | 常駐 |
| **Coder1** | AO（青） | DeepSeek | deepseek-chat | 0.5 | オンデマンド |
| **Coder2** | Aka（赤） | OpenAI | gpt-4o-mini | 0.4 | オンデマンド |
| **Coder3** | Kin（金） | Anthropic | claude-sonnet-4-6 | 0.3 | オンデマンド |
| **Coder4** | Gin（銀） | Gemini | gemini-2.0-flash-exp | 0.3 | オンデマンド |

### 3.2 Mio（Chat）の責務

1. **ユーザー理解と対話**: 唯一のユーザー窓口。最終応答を生成する
2. **全体モニタリング**: Worker・Coder の全作業を把握する
3. **直接処理**: CHAT カテゴリ（会話・説明・雑談・要約・レビュー）は Mio が直接処理
4. **委譲判断**: 自分で処理できない作業を Worker に委譲する

設定:
```yaml
ollama:
  chat_model: "Chat"   # gemma3:4b
  chat_keep_alive: -1
```

### 3.3 Shiro（Worker）の責務

1. **ルーティング分類**: タスクをカテゴリに分類し適切な処理へ振り分ける（§4 参照）
2. **データ処理**: ANALYZE / RESEARCH / OPS の実質処理担当
3. **Coder オーケストレーション**: どの Coder を使うか自律的に決定し指示する
4. **Patch 実行（即時）**: Coder が生成した patch を即座に実行する
5. **ツール実行**: ファイル操作・シェルコマンド・MCP 操作
6. **ReActLoop（v1.0実装済み）**: tool calling を使った複数回推論とツール実行ループ

設定:
```yaml
ollama:
  worker_model: "Worker"   # qwen3.5:9b
  worker_keep_alive: -1
worker:
  auto_commit: false
  command_timeout: 300
  workspace: "."
  protected_patterns: [".env*", "*credentials*"]
```

### 3.4 Coder の責務と Proposal フォーマット

各 Coder は Worker からの BuildRequest を受け取り、以下を返す：

```json
{
  "plan": "ステップ1: ...\nステップ2: ...",
  "patch": "[{\"action\": \"create\", \"path\": \"...\", \"content\": \"...\"}]",
  "risk": "影響範囲・リスク評価",
  "costHint": "見積もり工数"
}
```

patch の action 種別: `create / update / delete / append / mkdir / rename / copy`（7種）

## 4. ルーティング

### 4.1 ルーティングカテゴリ

| Route | 用途 | 現行の処理先 |
|-------|------|------------|
| `CHAT` | 会話・説明・雑談・レビュー | Mio |
| `PLAN` | 計画策定・仕様化・タスク分解 | Mio |
| `ANALYZE` | 抽出・集計・傾向分析・タグ付け | Mio |
| `OPS` | 運用操作・障害対応・コマンド実行 | Shiro |
| `RESEARCH` | 調査・出典確認・比較 | Mio |
| `CODE` | 汎用コーディング（ローカル Coder1 のみ） | Mio → Shiro → Coder1 |
| `CODE1` | Coder1(AO/DeepSeek) 明示指定 | Mio → Shiro → Coder1 |
| `CODE2` | Coder2(Aka/OpenAI) 明示指定 | Mio → Shiro → Coder2 |
| `CODE3` | Coder3(Kin/Claude) 明示指定 / Chrome操作 | Mio → Shiro → Coder3 |
| `CODE4` | Coder4(Gin/Gemini) 明示指定 | Mio → Shiro → Coder4 |

Coder1 / Coder2 / Coder3 / Coder4 の外部 API 利用は、`CODE1` / `CODE2` / `CODE3` / `CODE4` などの明示指定時だけ許可する。汎用 `CODE` は外部 Coder を使わず、外部 Coder へ自動切替しない。

### 4.2 ルーティング決定フロー（v6.1 現行実装）

```
1. ChatCommand 処理（/status, /stop, /compact, /context, /new）
   → Handled=true なら route 決定に進まない

2. parseExplicitCommand()  ← confidence=1.0 固定
   /code, /code1, /code2, /code3, /plan, /chat, /ops, /analyze, /research

3. RuleDictionary.Match()  ← confidence=0.85
   静的パターンマッチによる高速分類

4. CHAT フォールバック（安全側）
```

**注意**: 現行実装（v6.1）は LLM 分類器を使わず RuleDictionary のみ。
将来改善として LLM 分類器（confidence スコア付き）の追加を検討しているが、現時点では未実装。

### 4.3 RuleDictionary 主要パターン

| Route | 判定語（代表）|
|-------|-------------|
| `CODE` | `実装して`, `修正して`, `変更を入れて`, `追記して`, `編集して`, `ファイルを更新/変更` |
| `PLAN` | `計画`, `プラン`, `設計して`, `アーキテクチャ` |
| `ANALYZE` | `分析して`, `解析して`, `診断して`, `レビューして` |
| `OPS` | `実行して`, `起動して`, `デプロイ`, `ビルドして`, `停止して`, `再起動` |
| `RESEARCH` | `調べて`, `調査して`, `最新`, `比較` |

### 4.4 明示コマンドの優先原則

- 明示コマンドはプロンプト注入より強い（入力本文内の「/code と書いて」は無効）
- `local_only=true` 中に `/code` が来た場合は CODE 実行を拒否し、`/cloud` で解除を案内

### 4.5 ワーカー出力契約

Worker（Shiro）は作業結果に加えてメタ情報を返す：

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `result` | string | 作業成果 |
| `needs_next_loop` | bool | 追加ループの必要性 |
| `why` | string | 次が必要な理由 |
| `next_actions` | []string | 次にやるべき候補（最大3） |
| `confidence` | float | 成果の確からしさ (0.0〜1.0) |
| `risk` | enum | low / medium / high |
| `fit` | bool | （任意）このルートが適切だったか |
| `suggested_route` | string | （任意）再ルート提案 |

### 4.6 ループ制御

- 次を回す/止める最終決定は **LoopController**（Mio/Coder は提案のみ）
- 自動再ルーティングは最大 **1 回**（無限ループ防止）
- 最大ループ回数・最大時間は実装仕様（§12-a）で定義
- ユーザー確認が発生する条件: `risk==high`, 信頼度低 + 重大分岐, 追加情報が必須

4. LLM 失敗時のみリトライ（成功時の重複応答を防ぐ）

## 6. 会話エンジンとメモリ

### 6.1 Mio の会話エンジン（v5.0/v5.1 実装済み）

Mio は 3 層のメモリインフラを持つ：

| 層 | 実装 | 用途 |
|----|------|------|
| Short Memory | ConversationEngine（Redis） | 直近会話履歴 |
| Long Memory | DuckDB | 構造化トピック・事実 |
| Semantic Memory | Qdrant (gRPC:6334) | 意味検索・RecallPack |

PersonaState: LLM による自己編集が可能（PersonaEditor）。Mood 管理あり。

### 6.2 Shiro / Coder のメモリ（v1.0 設計済み・未実装）

Shiro と各 Coder には **LightMemory**（インメモリ短期履歴）を付与する予定。

- プロセス再起動でリセット（意図的）
- 最大保持ターン数: `maxTurns`（設定可能）
- Mio の ConversationEngine とは独立した純 Go struct
- 外部インフラ不要

詳細: `docs/04_実装仕様_機能拡張/実装仕様_エージェントペルソナ_v1.md`

## 7. 機能拡張（設計済み・未実装）

### 7.1 エージェントペルソナ v1.0

**目的**: Shiro / Coder の応答に個性を持たせる。

- `AgentPersona` 型: `Name / Personality / Tone` フィールド
- `BuildSystemPrompt(taskPrompt)`: ペルソナブロックを system prompt 先頭に前置
- `LightMemory`: セッション単位のインメモリ会話履歴（goroutine-safe）
- config.yaml の `agents:` セクションが存在しない場合は全機能無効（後方互換）

対象ファイル: `internal/domain/agent/persona.go`（新規）, `light_memory.go`（新規）

### 7.2 ケイパビリティ適応 v1.0

**目的**: 各ノードが起動時に自分の能力を自己検出し、適切な LLM・ツールを動的に割り当てる。

解決する問題:
- 低スペック環境での OOM（静的 config で大モデルを指定するリスク）
- API キーのないノードへの誤割り当て
- Windows と Linux でのツールパス差異

コンポーネント:
- `NodeCapabilities`: 起動時検出（Platform / Memory / LLMs / Tools）
- `ToolRegistry`: ツールを永続化し対応プラットフォームのノードが共有
- ベース: v4.0 Transport 抽象化を前提とする

詳細: `docs/04_実装仕様_機能拡張/実装仕様_ケイパビリティ適応_v1.md`

### 7.3 サブエージェント（ReActLoop）v1.0

**ステータス**: 実装済み（SubagentManager）

Shiro（Worker）が 1 回の LLM 応答で終わらず、tool calling を使って複数回の推論とツール実行を回すループ。

- `ToolCallingProvider.Chat()` による反復推論
- `ToolRunner.ExecuteV2()` によるツール実行
- `SubagentManager.RunSync()` で同期呼び出し

詳細: `docs/04_実装仕様_機能拡張/実装仕様_サブエージェント_v1.md`

## 8. IdleChat

ユーザーがアイドル中にエージェント同士（Mio / Shiro 等）が自律的に会話するシステム。
本番タスクが来たら**即座に中断**する（最優先）。

### 8.1 セッションモード

| モード | 開始方法 | 特徴 |
|-------|---------|------|
| `idle` | 自動（アイドル検知）/ 手動 | 通常カテゴリ雑談（Single / Double / External / Movie / News） |
| `forecast` | 手動のみ「未来展望」ボタン | 6ドメイン固定順・番組形式（100ターン/ドメイン） |
| `story` | 手動 `/viewer/idlechat/story` | 昔話・童話の改変朗読 |

IdleChat のお題カテゴリは、ユーザー観測・ログ・E2E 評価では `Single / Double / External / Movie / Forecast / Story / News` の 7 種とする。
通常 IdleChat の自動ローテーションおよび通常評価では、`single → double → external → movie → news → forecast → story-simple` の順で最低 1 巡できること。
`forecast` と `story` はモード別カテゴリとして扱うが、自動ローテーションにも含め、手動起動または専用 E2E でも個別に検証できること。

News は純粋なニュース深掘りカテゴリであり、ランダムジャンルを混ぜない。
News を External へ黙ってすり替えること、または Movie を通常カテゴリの隠しフラグとして扱うことは禁止する。

### 8.2 ストーリーモード

昔話・童話を 5 スタイル（role_shift / view_shift / value_shift / inversion / scale_shift）で改変し短編朗読物語を生成。

**パイプライン**:

```
Step 1: スタイル選択（決定論的）
↓
Step 2〜6: prepareStory()（決定論的・LLM なし）
  ├─ Step 2: ソース選択
  ├─ Step 3: ソース分析
  ├─ Step 4: 改変プラン
  ├─ Step 5: ビートプラン
  └─ Step 6: アダプテーションプラン
↓
Step 7: 第1稿生成（LLM: Shiro / MaxTokens=300 / Temperature=0.6）
↓
Step 8: 改稿（LLM: Shiro / MaxTokens=1800 / Temperature=0.5）
```

フォールバック: LLM 失敗 → 決定論的ドラフト → 原話再話の順で必ず出力

対象データ: `data/story/*.json`（17ファイル・アクティブ）

### 8.3 LLM 役割分担

| 処理 | 担当 |
|------|------|
| 通常トピック生成 | Mio (gemma3:4b) |
| 未来展望キーワード抽出・トピック生成 | ローカル Forecast provider（local Coder / Worker） |
| 発話生成 | 各話者の LLM |
| まとめ・既出テーマ抽出 | Shiro (qwen3.5:9b) |

## 9. セキュリティとデータ取り扱い

- CODE 以外の目的でクラウドを使わない（原則）
- `/local` フラグ中はクラウド呼び出しをシステムで強制禁止
- 秘密情報・個人情報・トークンはクラウド送信対象から除外（実装でサニタイズ）、ログにも残さない
- プロンプト注入対策: 明示コマンドとルーティングはシステムルール優先
- 保護ファイルパターン（`.env*`, `*credentials*`）へのアクセスは Worker が `error` で拒否
- Git auto-commit（オプション）でセーフガード

API キー管理:
```
ANTHROPIC_API_KEY  → 環境変数または ~/.rencrow/.env (chmod 600)
DEEPSEEK_API_KEY   → 同上
OPENAI_API_KEY     → 同上
```

## 10. ログとトレーサビリティ

### 10.1 必須ログイベント

| イベント種別 | タイミング |
|------------|----------|
| `router.decision` | ルーティング決定時 |
| `worker.executed` | Worker による patch 実行時 |
| `worker.success` / `worker.fail` | Worker 実行結果 |
| `coder.plan_generated` | Coder による plan/patch 生成時 |
| `classifier.error` | 分類エラー発生時 |

### 10.2 必須保存フィールド

| フィールド | 説明 |
|-----------|------|
| `job_id` | タスク識別子（`job_{timestamp}_{random}` 形式） |
| `session_id` | セッション識別子 |
| `initial_route` / `final_route` | ルーティング情報 |
| `execution_status` | `success` / `partial_success` / `failed` |
| `coder_output` | plan/patch/risk の要約 |
| `worker_result` | 実行結果（成功数/失敗数、Git commit hash） |

ログ保存先: `~/.rencrow/logs/rencrow.log`
ログローテーション: `~/.rencrow/bin/log-rotate.sh`（cron 毎日 04:00）

### 10.3 画像監査ログ

画像付きリクエストには `provider.http` ログへ以下を記録：
`source_path`, `url_type`, `image_url_length`, `included`, `drop_reason`（リクエスト前後）
