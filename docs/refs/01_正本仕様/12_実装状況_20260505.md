## 5. 最新仕様との差分詳細

### 5.1 会話制御

最新仕様:

```text
User Input
  -> LangGraph Turn Controller
  -> Recall Orchestrator
  -> Persona Prompt Builder
  -> Local LLM
  -> Memory Candidate Extractor
  -> Staging / Validator
```

現行実装:

```text
User Input
  -> MessageOrchestrator
  -> Mio.DecideAction
  -> Mio.Chat / Shiro.Execute / Coder
  -> EventHub / Viewer / TTS
```

差分:

- LangGraphは未導入。
- Turn Controller / character subgraph は未実装。
- 既存のGoオーケストレーターに統合するか、LangGraphを別ランタイムとして接続するか未決定。

### 5.2 LLM接続

最新仕様:

- MLXローカルサーバ
- OpenAI互換API
- model名は `Chat` / `Worker` / `Wild`
- RenCrow側はMLX専用providerを新規実装しない

現行実装:

- Chat provider: `local_llm.enabled=true` なら OpenAI互換local provider、falseなら Ollama
- Worker provider: `local_llm.enabled=true` なら OpenAI互換local provider、falseなら Ollama
- Wild provider: `local_llm.enabled=true` なら OpenAI互換local providerとして生成
- Embedding / summarizer / profile extractor は `local_llm.enabled=true` でOpenAI互換local provider、falseならOllama fallback
- Coder provider は `deepseek/openai/claude/gemini/ollama`

差分:

- `cmd/picoclaw/main.go` の Chat/Worker主経路は `buildPrimaryLLMProviders()` 経由に変更済み。
- OpenAI互換ローカルproviderを Chat/Worker/Wild に差し込む `local_llm` 設定は実装済み。
- `local_llm.enabled=true` の場合、Ollama設定なしでも validation を通過する。
- Wild provider は生成され、`/wild` 明示指定と用途判定による自動Wild選択で利用できる。
- Embedding / summarizer / profile extractor は `local_llm.enabled=true` でOpenAI互換local providerを利用する。Ollama fallbackは維持する。

### 5.3 記憶

最新仕様:

L0〜L4 は保存媒体ではなく lifecycle 上の位置で定義する。正本定義と物理ストア / コード命名の対応は `18_Memory_Lifecycle_Recall_Context.md#311-実装命名物理ストアとの対応` を参照する。

現行実装:

| 領域 | store |
|---|---|
| 短期 | Redis |
| hot store | SQLiteに会話イベントをミラー保存可能 |
| 中期 | DuckDB |
| 長期 | Qdrant想定VectorDB |

差分:

- SQLite hot store への保存とRedis miss時のRecall fallbackは実装済み。
- L1 memory/news/knowledge/staging のParquet archive tableとexportはあり、L1 staging保存時とpromoter昇格時のDuckDB archive自動同期は実装済み。
- 旧文書では物理ストア基準の L0/L1/L2/L3 名が混在していた。正本では lifecycle 基準に統一済み。
- `conv:<thread_id>` namespace はL1保存で導入済み。`user:` / `char:` / `kb:` のbuilder/validator/promoterも導入済み。

### 5.4 外部取得

最新仕様:

```text
Source Registry
  -> Scheduler
  -> Fetcher
  -> Staging
  -> Validator
  -> Promoter
  -> Official DB
```

現行実装:

- RSS parser はある。

- L1 Source Registry / staging / validator / promoter / News DB / Daily Digest はある。
- Source Registry sweeperで due source のRSS/AtomとHTTP系sourceを自動取得し、stagingからNews DB/Knowledge DB/Digestへ流せる。

差分:
