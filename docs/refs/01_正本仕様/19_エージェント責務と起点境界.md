# エージェント責務と起点境界

**ステータス**: 正本（active）
**作成日**: 2026-07-01
**対象**: RenCrow runtime / Viewer / Chat / IdleChat / Heartbeat / background job / provider alias

この文書は、RenCrow における以下の境界を定義する正本である。

- 入力 surface
- Agent identity
- runtime provider alias
- 作業起点
- 通常時と異常時の agent 接続

既存の `03_エージェント定義.md`、`04_ルーティング.md`、`08_IdleChat.md`、`docs/10_新仕様/04_Chat_Worker_Coder仕様.md`、`docs/10_新仕様/83_空き時間ジョブ実行基盤仕様.md` と矛盾する場合、この文書の用語境界を優先する。
個別実装の詳細は各仕様へ委譲する。

---

## 1. 基本原則

RenCrow では、次の4層を混同しない。

| 層 | 意味 | 例 |
| --- | --- | --- |
| 入力 surface | ユーザーや外部 system が RenCrow に入る入口 | `viewer_chat`, `voice_chat`, `external_channel`, `idlechat`, `heartbeat` |
| Agent identity | 誰として考え、話し、責任を持つか | `Mio`, `Shiro`, `Midori`, `Kuro`, `AO`, `Aka`, `Kin`, `Gin` |
| runtime provider alias | 実行時にどの LLM / capability slot を使うか | `Chat`, `Worker`, `ChatWorker`, `Wild`, `Heavy`, `Coder1`-`Coder4` |
| 作業起点 | なぜ処理が始まったか | user input, periodic work, queued work, explicit repair, maintenance |

`Chat` は人格名でも単一 model 名でもない。
`Chat` はユーザー入力を受ける frontend / conversation surface であり、会話対象 agent は別に決まる。

`Mio` は Chat surface の代表 target agent であるが、RenCrow の全 agent が Mio だけという意味ではない。
Shiro / Midori / Kuro なども conversation target agent になり得る。

LLM を使わずに完了できる作業では、通常時に LLM を使わない。
LLM 接続は、必要な判断・会話・調査・生成・失敗時の報告に限定する。

---

## 2. Agent Identity

RenCrow の agent identity は、ユーザーに見える人格・責任主体である。

| Agent | 主責務 | 通常の会話可否 | 備考 |
| --- | --- | --- | --- |
| Mio | ユーザー対話、関係性、最終応答、作業結果の受け渡し | 可 | Chat surface の代表 target agent |
| Shiro | 実務、調査、修復、ops、作業統括、失敗調査 | 可 | 作業起点の主担当 |
| Midori | Wild / 発散的探索 / creative path | 可 | `Wild` alias の人格側 |
| Kuro | Heavy / 深い分析 / 重い検討 | 可 | `Heavy` alias の人格側 |
| AO | Coder slot | 原則 work conversation | `Coder1` |
| Aka | Coder slot | 原則 work conversation | `Coder2` |
| Kin | Coder slot | 原則 work conversation | `Coder3` |
| Gin | Coder slot | 原則 work conversation | `Coder4` |

Coder 系 agent は、原則として plan / patch / proposal を生成する提案者である。
採用判断、実行、検証、ログ記録、安全境界は Shiro / Worker 側が担当する。

---

## 3. Provider Alias

provider alias は runtime capability の名前であり、人格そのものではない。

| Provider alias | 標準 identity | 用途 | 注意 |
| --- | --- | --- | --- |
| `Chat` | Mio / other conversation target | Chat surface の会話生成 | `Chat` は frontend / conversation provider であり人格名ではない |
| `Worker` | Shiro | ops / repair / investigation / work execution | 作業実行権限を持つ |
| `ChatWorker` | Shiro | Shiro の短い会話・IdleChat 発話 | Worker 実行権限の別人格ではない |
| `Wild` | Midori | 発散探索、Wild route | 通常 Chat の fallback にしない |
| `Heavy` | Kuro | 重い分析、深い検討 | 通常 Chat の fallback にしない |
| `Coder1` | AO | Coder proposal | 直接実行しない |
| `Coder2` | Aka | Coder proposal | 直接実行しない |
| `Coder3` | Kin | Coder proposal | 直接実行しない |
| `Coder4` | Gin | Coder proposal | 直接実行しない |

### 3.1 Chat と ChatWorker の境界

Mio の通常会話 provider は `Chat` を優先する。
`ChatWorker` は Shiro の会話用 alias であり、Mio の通常会話先ではない。

実装上の互換 fallback として `ChatWorker` が使われることがあっても、正本上の標準経路は次である。

```text
user input
  -> Chat surface
  -> target agent = Mio
  -> provider alias = Chat
```

Shiro が会話する場合は、用途により `ChatWorker` または `Worker` を使い分ける。

```text
IdleChat Shiro speech
  -> target agent = Shiro
  -> provider alias = ChatWorker

repair / investigation / ops
  -> working agent = Shiro
  -> provider alias = Worker
```

## 4. Chat Surface

Chat は RenCrow の会話 frontend surface である。
入口の違いは Chat の種類ではなく、同じ Chat surface への adapter 差分として扱う。

| Surface | 入力 | target agent | 区分 |
| --- | --- | --- | --- |
| `viewer_chat` | Viewer のテキスト入力 | Mio など target agent | Chat 入口違い |
| `voice_chat` / STT | ユーザー音声入力の確定文字列 | Mio など target agent | Chat 入口違い |
| `external_channel` | LINE / Discord / CLI など | Mio など target agent | Chat 入口違い |

`voice_chat` は音声処理方式の違いを内包する。
STT primary では `/stt` の final を `/viewer/send` へ渡し、Voice Direct / input_audio では `/voice-chat` の `llm.final` を `ProcessVoiceDirect` で Chat event へ正規化する。
CLI の `--audio` は `/stt/chat-input` で文字起こし後に `/viewer/send` へ渡す external channel 入口だが、音声を通常 Chat に流す adapter family として `voice_chat` と同じ境界を守る。

標準経路:

```text
viewer_chat / voice_chat / external_channel
  -> Chat surface
  -> target agent conversation
  -> response to user-facing channel
```

Chat surface は「ユーザーに返す会話の入口」であり、作業起点すべてを意味しない。
IdleChat、Heartbeat、repair、superagent run queue、background job は別起点である。

## 5. 作業起点

作業起点は「なぜ処理が始まったか」を表す。
agent identity や provider alias と混同しない。

| 起点 | きっかけ | 標準 start agent | 区分 | 備考 |
| --- | --- | --- | --- | --- |
| `viewer_chat` / `voice_chat` / `external_channel` | ユーザー入力 | Mio など target agent | Chat 起点 | 入口違いであり同じ Chat surface |
| `idlechat` | system が会話を進める | 参加 agent | 会話起点 | ユーザー Chat の入口ではない |
| `heartbeat` | 定期監視 / 定期作業 | Shiro | 作業起点 | 作業なので Shiro start を標準にする |
| `repair` | 明示修復 | Shiro | 作業起点 | Mio は報告側に回る |
| `superagent_run_queue` | queued work scheduler | Shiro / Worker / Coder 系 | 作業起点 | daily data refresh そのものではない |
| `background_jobs` | maintenance / deterministic periodic work | agent なし | system 起点 | 失敗時だけ Shiro / Mio に接続 |
| `backlog` | 作業候補・アイデア・未実装台帳 | なし | ledger | Heartbeat 等が拾う候補であり、それ自体は起点ではない |

### 5.1 IdleChat

IdleChat は Chat surface ではない。
IdleChat は system 起点の agent conversation surface であり、参加 agent が会話を進める。

```text
surface = idlechat
mode = normal / forecast / story / other
participants = configured agents
```

IdleChat 内の Shiro 発話は `ChatWorker` を使ってよい。
ただし、IdleChat の発話 alias と Worker 実行権限を混同しない。

### 5.2 Heartbeat

Heartbeat は定期監視・定期作業の起点である。
作業の起点なので、標準 start agent は Shiro とする。

```text
system.heartbeat
  -> Shiro start
  -> deterministic checks / work dispatch
  -> Mio report only when user-facing report is needed
```

Heartbeat がユーザー向け会話を生成する必要がある場合、Mio は最終報告者として使う。
ただし、作業開始主体は Mio ではなく Shiro である。

### 5.3 Backlog

Backlog は作業起点ではなく ledger である。

```text
Backlog
  = idea / unimplemented item / work candidate ledger
```

Backlog item は Heartbeat、manual action、scheduler、SuperAgent などから処理対象として拾われ得る。
Backlog 自体が LLM や Worker を自動起動するとは扱わない。

### 5.4 SuperAgent Run Queue

`superagent_run_queue` は queued work scheduler である。
毎日のデータ取得そのものではない。

```text
superagent_run_queue
  -> queued work item
  -> claim
  -> Shiro / Worker / Coder route
  -> completion / failure ledger
```

daily data refresh は別の deterministic job / command / scheduler target として扱う。
必要ならその結果や失敗だけが SuperAgent run queue に接続される。

## 6. Background Jobs

background job は maintenance / deterministic periodic work を扱う system 起点である。

対象例:

```text
source_registry_sweep
memory_lifecycle
daily_intake_sweep
parquet_export
movie_catalog_backfill
artifact_cleanup
scheduled data refresh
```

通常時は agent なしで動く。
LLM を使わずに完了できる作業は LLM を使わない。

```text
background job success
  -> deterministic execution
  -> log / metric / ledger
  -> no Chat / Worker / LLM call
```

失敗時だけ Shiro と Mio に接続する。

```text
background job failure
  -> background_job.failed
     from = background_job
     to = shiro
     purpose = investigation

  -> job.notification
     from = shiro
     to = mio
     purpose = user-facing report if needed
```

この接続はイベント通知であり、background job が通常時に Chat / Worker / LLM を直接起動することを意味しない。

Mio は「ユーザーに見せる必要がある失敗」を報告する。
Shiro は「原因調査・修復・再実行判断」を担当する。

## 7. 正常と異常の境界

### 7.1 正常

次は正常である。

- Chat surface の入口が Viewer / voice / external channel に分かれる。
- Ren の通常問いかけが Mio target agent に入り、`Chat` provider で応答する。
- IdleChat の Shiro 発話に `ChatWorker` が使われる。
- Heartbeat の作業開始が Shiro になる。
- background job が成功時に LLM を使わず完了する。
- background job 失敗時に Shiro 調査イベントと Mio 通知イベントが出る。
- Backlog item が ledger として保存され、別起点から処理される。

### 7.2 異常

次は異常または要調査である。

- Chat surface / target agent / provider alias が混同され、Mio の通常会話が `ChatWorker` 主経路になる。
- background job が成功系で不要に LLM を呼ぶ。
- deterministic job の失敗が log のみで終わり、Shiro / Mio へ通知されない。
- Heartbeat の作業起点が Mio 会話として扱われる。
- Backlog が ledger ではなく自動実行主体として扱われる。
- `Chat` が人格名または単一 model 名として設定・UI・ログに固定される。

## 8. 命名ルール

### 8.1 禁止する混同

`Chat` という語を次の意味で同時に使わない。

- frontend surface
- Mio identity
- model name
- provider alias
- route name

正本上は次のように分ける。

| 呼びたいもの | 推奨名 |
| --- | --- |
| ユーザー入力面 | Chat surface |
| Mio の人格 | Mio |
| provider slot | Chat provider / `Chat` alias |
| route | `CHAT` route |
| model 実体 | model id / backend model name |

### 8.2 表示名と内部名

UI では人間に分かる agent name を優先する。

```text
Mio
Shiro
Midori
Kuro
AO
Aka
Kin
Gin
```

ログ・設定・runtime では、agent identity と provider alias を両方残す。

```json
{
  "target_agent": "mio",
  "provider_alias": "Chat",
  "surface": "viewer_chat",
  "origin": "user"
}
```

### 8.3 外部向け指示サンプル

RenCrow 内部の開発者は `Chat` の文脈を理解できる場合がある。
しかし、外部 agent / 外部チーム / 汎用 coding agent に RenCrow を構築・修正させる場合、指示語は agent を主語にする。

原則:

```text
Agent-first, Surface-qualified
```

つまり、誰が責任主体かを先に書き、入口 surface は補足として添える。

推奨する構造:

```text
target_agent = Mio
surface = viewer_chat
provider_alias = Chat
origin = user
```

#### 推奨表現

| 目的 | 推奨指示 |
| --- | --- |
| Viewer の通常会話を直す | Mio の `viewer_chat` 応答経路を直す |
| 音声入力後の会話を直す | `voice_chat` surface の STT final を target_agent=Mio に渡す経路を直す |
| 外部 channel の通常会話を直す | `external_channel` surface から target_agent=Mio へ届く会話経路を直す |
| Heartbeat 作業を整理する | Heartbeat 起点の作業は start_agent=Shiro として扱う |
| background job の失敗通知を直す | background job は成功時 agent なし、失敗時だけ Shiro 調査・Mio 報告通知へ接続する |
| IdleChat の Shiro 発話を直す | IdleChat surface の Shiro 発話は provider_alias=ChatWorker を使う |

#### 避ける表現

| 避ける表現 | 理由 |
| --- | --- |
| Chat を直す | surface / route / provider / agent のどれか不明 |
| ChatWorker を Mio に使う | Shiro 会話 alias と Mio 通常会話 provider が混ざる |
| Heartbeat を Mio に処理させる | 作業起点は Shiro start が標準 |
| Backlog に作業させる | Backlog は ledger であり実行主体ではない |
| background job で Shiro を常時呼ぶ | deterministic job は通常時 LLM 不使用 |

外部向けの短い指示では、次の形を標準とする。

```text
Mio の viewer_chat 応答経路を直す。
```

詳細が必要な場合は次のように展開する。

```text
viewer_chat surface から target_agent=Mio へ届く通常会話経路を直す。
provider_alias は Chat を優先し、ChatWorker は Mio 通常会話の主経路にしない。
```

作業起点を指す場合:

```text
Heartbeat 起点の作業は start_agent=Shiro とする。
Mio は必要な場合だけユーザー向け報告を行う。
```

system job を指す場合:

```text
background_jobs は成功時 agent なしで deterministic に完了する。
失敗時だけ background_job.failed を Shiro へ、job.notification を Mio へ出す。
```

移行途中の互換 fallback は認めるが、標準経路として扱わない。

## 10. 検証観点

この正本の回帰を防ぐため、次を検証対象にする。

- Mio 通常会話は `Chat` provider を優先する。
- Shiro の IdleChat 発話は `ChatWorker` を使えるが、Worker 実行権限とは別である。
- Heartbeat の作業は Shiro start として扱う。
- background job 成功時に LLM を呼ばない。
- background job 失敗時に `background_job.failed` と `job.notification` が同じ `job_id` / `session_id` で出る。
- `viewer_chat` / `voice_chat` / external channel は Chat 入口違いとして扱う。
- Backlog は ledger であり、自動実行主体ではない。
