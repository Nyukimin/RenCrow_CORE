# RenCrow 追加仕様: 記憶OSつきマルチキャラクター会話基盤

**基準日**: 2026-05-01
**作成日**: 2026-05-05
**ステータス**: 追加仕様メモ

この文書は、2026-05-01 時点の最新メモを優先して統合した RenCrow の追加仕様である。
現行実装・既存正本との差分を含むため、実装反映時は既存仕様との整合を別途確認する。

一言でいうと、RenCrow は **ローカルLLMを主役にした、記憶OSつきマルチキャラクター会話基盤** である。
単なる複数LLMチャットではなく、会話・記憶・ニュース・知識DB・想起・外部取得を分けて運用する。

## 1. 基本目的

RenCrow は、ユーザーが最初に話しかけると、複数の AI キャラクターがそれぞれの個性・記憶・得意分野を持って会話し、必要に応じて検索や知識DBを参照しながら自然に応答するシステムである。

初期構想では、ローカルLLMを3体並べ、ユーザー発話後に `会話LLM1 -> 会話LLM2 -> 会話+検索LLM` のように順番に会話させる設計だった。
現在方針では、エージェント周辺の状態管理を明示するため、LangGraph 中心の会話制御へ寄せる。

基本整理:

- 検索APIは外部センサー。
- DBは記憶の土台。
- RAGは知識参照の仕組み。
- ローカルLLMは推論エンジン。
- LangGraphは会話と状態制御の中枢。

RenCrow は検索APIやクラウドLLMに依存せず、**ローカル推論 + ローカル記憶 + 必要時だけ外部確認** を基本とする。

## 2. 中核アーキテクチャ

RenCrow は、1つの巨大なRAGや1つのエージェントに全責務を押し込まず、役割ごとに分割する。

| 層 | 役割 |
|---|---|
| モデル層 | ローカルLLM。生成と埋め込みを担当 |
| 会話ランタイム | LangGraph。ターン制御、割り込み、キャラノード、状態管理 |
| hot store | SQLite。今日の記憶、検索キャッシュ、イベントログ |
| cold store | DuckDB + Parquet。ニュース、知識DB、月次記憶、長期記憶 |
| vector sidecar | FAISS または Qdrant。意味検索・想起 |
| 外部取得 | RSS、公式API、GitHub、Hugging Face、PyPI、MediaWiki など |
| staging / validator | 外部データや記憶候補を正式DBへ入れる前の検証層 |

モデルサーバは推論だけを担当し、会話状態や記憶は RenCrow 側で保持する。
LangGraph は親 graph でターン全体を制御し、各キャラクターを subgraph として分離する。

全体フロー:

```text
User Input
  ↓
LangGraph Turn Controller
  ↓
Recall Orchestrator
  ├─ L0: 今の会話
  ├─ L1: 今日の記憶
  ├─ L2: 今月の記憶
  ├─ L3: 長期記憶
  ├─ User Memory
  ├─ Character Memory
  ├─ Knowledge DB
  ├─ News DB
  ├─ Search Cache
  └─ External Search only if needed
  ↓
Recall Pack Builder
  ↓
Persona Prompt Builder
  ├─ personas/*.yaml
  ├─ styleguides/*.md
  └─ templates/*.md
  ↓
Local LLM
  ↓
Response
  ↓
Memory Candidate Extractor
  ↓
Staging / Validator
  ↓
Official Memory DB
```

## 3. キャラクター仕様

構想上の中心キャラクターは、ルミナ / クラリス / ノクスの3体である。

| キャラ | 役割 | 個性・口調 | 検索 |
|---|---|---|---|
| ルミナ | 司会・要約・雑談・次の論点提示 | フレンドリー、洞察型、会話を回す | 可 |
| クラリス | 深掘り解説・理論整理 | 穏やか、理論派、背景説明が得意 | 基本なし |
| ノクス | 情報ハンター・検索・要約 | クール、端的、出典重視 | 高速検索担当 |

キャラクター人格はファインチューンではなく YAML + Markdown で管理する。

```text
personas/
  lumina.yaml
  claris.yaml
  nox.yaml

styleguides/
  lumina.md
  claris.md
  nox.md

templates/
  answer_core.md
  cite_block.md
  disagree.md
```

YAMLには、役割、目標、禁止事項、文体、出力ルール、記憶参照方針を置く。
Markdownには、口調ガイド、例文、テンプレートを置く。
知識は RAG / VectorDB / Knowledge DB で扱い、口調や人格は運用データとして変更可能にする。

## 4. 会話ルーティング

会話制御は LangGraph の Router Node で行う。

優先順位:

1. ユーザー指名: `@ルミナ`, `@クラリス`, `@ノクス`, `@all`
2. ドメイン適性: 映画、歴史、技術、ニュースなど
3. ラウンドロビン + クールタイム: 同じキャラの連投を避ける
4. スムーズさ補正: 直前発話者や文脈に応じて自然さを調整する

指名インターフェース:

| 入力 | 動作 |
|---|---|
| `@ルミナ こんにちは` | ルミナだけ応答 |
| `@ルミナ,@ノクス 調べて` | ルミナ -> ノクスの順で応答 |
| `@all` または指名なし | 全員、または Router 判断 |

UIでは、キャラバッジやチェックボックスで同じ操作を可能にする構想。

## 5. 記憶システム

記憶は L0〜L3 の時間レイヤーと、会話 / ユーザー / キャラ / 知識DB の namespace 分離で整理する。

### 5.1 時間レイヤー

| 層 | 内容 | 推奨ストア |
|---|---|---|
| L0 現在会話 | 今のスレッド、直近メッセージ、割り込み状態 | LangGraph checkpoint + rolling summary |
| L1 今日の記憶 | 当日エピソード、検索キャッシュ、イベントログ | SQLite WAL |
| L2 今月の記憶 | 月間ハイライト、会話要約、topic cluster | DuckDB + Parquet |
| L3 長期記憶 | ユーザー嗜好、長期エピソード、canonical memory | DuckDB + Parquet + vector sidecar |
| Knowledge DB | 映画・小説・ドラマ・音楽など | DuckDB + Parquet + FTS |
| News DB | ニュース本文、出典メタデータ、日次digest | DuckDB + Parquet + vector sidecar |
| Staging DB | raw fetch、仮要約、validator結果 | JSONL / Parquet |
| Vector sidecar | 意味検索 | MVPはFAISS、将来Qdrant |

ローカルPC中心のMVPでは、Redisではなく SQLite を hot store に寄せる。
Redisは複数プロセス化やWeb UI常時稼働が必要になった段階の候補とする。

### 5.2 namespace分離

| 種類 | namespace | 内容 | 共有範囲 |
|---|---|---|---|
| 会話記憶 | `conv:<thread_id>` | 今の会話、今日、今月、長期エピソード | 参加キャラで共有 |
| ユーザー記憶 | `user:<uid>` | 嗜好、プロフィール、プロジェクト、制約 | 全キャラが参照可 |
| キャラ記憶 | `char:<persona>` | 口調成長、KPI、得意不得意、関係性 | 基本は本人 |
| 知識DB | `kb:<domain>` | 映画、音楽、ニュース、技術知識など | 全キャラが参照可 |

ユーザー記憶 type:

| type | 内容 |
|---|---|
| `profile` | 明示されたプロフィール |
| `preference` | 出力や話題の好み |
| `project` | 進行中プロジェクト |
| `constraint` | 常時守るべき制約 |
| `relationship` | キャラとの距離感 |
| `episode` | 会話や判断の記録 |
| `skill` | ユーザーの得意分野 |
| `sensitive` | 健康・家庭など慎重に扱う情報 |

推測した記憶は、いきなり確定しない。

```text
observed
  ↓
candidate
  ↓
confirmed
```

記憶候補は staging に置き、明示確認や反復観測を経て確定する。

## 6. スレッド仕様

短期記憶は「今まさに話しているひとまとまりの話題 = 1スレッド」とする。

開始条件:

- 初回入力
- 新トピック
- ドメイン変化
- 話題切替語

終了条件:

- トピック変化
- 類似度低下
- 10分アイドル
- ターン上限

短期メモリには最大12メッセージ程度を保持する。
終了時には、要約、キーワード、embeddingを生成し、中期・長期側へ流す。
最新版では L0 仕様に統合し、LangGraph checkpoint + rolling summary で扱う。

## 7. 想起フロー

毎ターン想起を行う。ただし毎ターン外部検索はしない。

想起順:

```text
L0 現在会話
  ↓
L1 今日の記憶
  ↓
role-filtered L2/L3
  ↓
lexical retrieval
  ↓
vector retrieval
  ↓
Search Cache
  ↓
External Search only if needed
```

古い会話全文をそのままLLMへ再送しない。
L0は rolling summary + 直近数往復に抑える。
過去記憶は、要約・キーワード・embedding・出典メタデータ付きで取り出す。

Recall Pack には次を入れる。

- 現在の会話: 直近話題、最新意図、未解決論点
- ユーザー記憶: 好み、制約、プロジェクト情報
- 今日の記憶
- 今月の記憶
- 関連知識: Knowledge DB / News DB / Search Cache
- 出典・根拠: `source_id`, `source_url`, `published_at`, `retrieved_at`, `event_id`

想起パックのサイズは固定件数ではなく、モデル文脈長に対する比率で制御する。
目安として全体の1割前後に抑える。

## 8. 知識DB・RAG・連想ネットワーク

対象ドメインは、映画・小説・ドラマ・漫画・音楽・演劇・ボドゲなど。
知識DBは共通コア + ドメイン別拡張で設計する。

共通コア例:

```json
{
  "id": "global_uid",
  "domain": "movie|novel|tv|manga|music|stage|boardgame",
  "title": "作品名",
  "alt_titles": ["別名", "原題"],
  "creators": [{"role": "director", "name": "..."}],
  "year": 2014,
  "genres": ["SF"],
  "keywords": ["宇宙", "時間", "父と娘"],
  "summary": "要約",
  "themes": ["孤独", "希望"],
  "embedding": "...",
  "meta": {"language": "ja"}
}
```

検索技術:

| 技術 | 役割 |
|---|---|
| DuckDB FTS / BM25 | 固有名詞、作品名、引用句に強い lexical retrieval |
| VectorDB | 意味検索、類似エピソード検索 |
| 共起ネットワーク | 関連語、テーマ、作者、概念の連想 |
| RAG | 取り出した知識をLLMに注入 |
| optional graph DB | 創作者・作品・テーマの関係可視化 |

目標は、作品名から意味近傍・共起語・個人エピソードを自然に想起できる状態である。

## 9. ニュース雑談機能

日々のニュースを雑談の入口にする。
取得元は検索APIではなく、RSS / Atom / 公式APIを優先する。

ニュース処理:

```text
RSS / API
  ↓
staging
  ↓
validator
  ↓
News DB
  ↓
要約・キーワード抽出
  ↓
embedding / FTS index
  ↓
Daily Digest
  ↓
会話中の想起
```

朝・昼・夜にニュースパックを作り、ユーザーが雑談を振ったときに News DB、今日の記憶、今月の記憶、長期記憶を組み合わせる。
外部から取った `raw_text` と、LLMが作った `summary_draft` は混ぜない。

## 10. 検索APIの位置づけ

検索APIは RenCrow の背骨ではない。
方針は **registry-first / search-last**。

| 場面 | 方針 |
|---|---|
| 初期DB育成 | 原則使わない。公式API、RSS、Wikipedia系、CSVを使う |
| 日々のニュース取得 | 使わない。RSSや公式feedを使う |
| 知らない固有名詞 | 使う |
| 最新情報確認 | 使う |
| 公式情報確認 | 公式URL発見のために使うことはある |
| 雑談中の補足 | まずローカル。足りないときだけ使う |
| RAG用ソース探索 | 限定的に使う |
| 出典の裏取り | 使う |
| 定期DB更新 | 避ける |

検索APIで見つけた情報は即正式DBに入れない。
まず Search Cache や staging に入り、validatorを通過してから Source Registry や Knowledge DB へ昇格する。

## 11. 外部取得・staging・validator

定期取得は自由検索ではなく、管理された取得パイプラインにする。

```text
Source Registry
  ↓
Scheduler
  ↓
Fetcher
  ↓
Staging JSONL / Parquet
  ↓
Validator
  ↓
Promoter
  ↓
SQLite / DuckDB / VectorDB
  ↓
FTS rebuild / Embedding update / Daily Digest
```

取得元:

| 対象 | 取得方法 |
|---|---|
| 毎朝ニュース | RSS / Atom / 公式API |
| AI関連ニュース | 公式blog / RSS |
| GitHub Releases | GitHub API |
| Hugging Face | Hub API / Webhook |
| PyPI | RSS / JSON API |
| Wikipedia系 | MediaWiki recentchanges |
| 映画・小説・音楽DB | 公式API / Wikipedia系 / CSV / 手元DB |
| 不明語 | 検索API fallback |

staging JSONL には `event_id`, `source_id`, `source_url`, `fetched_at`, `published_at`, `raw_text`, `raw_hash`, `summary_draft`, `keywords`, `license_note`, `validation_status` などを入れる。

validator では、schema妥当性、URL妥当性、hash重複、日付妥当性、source trust、sensitivity、license note、memory type を確認する。

## 12. Claude Cowork / Codex の役割

Claude Cowork や Codex は使えるが、RenCrow の本番DBには直接触らせない。

| ツール | 役割 |
|---|---|
| Claude Cowork | 調査、整理、複数ソース比較、source候補作成 |
| Claude Skills | ニュース分類、出典確認、要約テンプレ適用 |
| Codex Skills | fetcher、validator、DDL、テスト、ETLスクリプト |
| Codex Automations | 毎日/毎週の取得ジョブ、差分検出、レポート生成 |
| GitHub Actions | repo中心の定期実行、CI、公開ソース取得 |
| local cron | DuckDB更新、embedding、ローカルLLM要約 |
| RenCrow Worker | validator通過後の正式DB投入 |

許可する範囲:

- staging
- reports
- logs
- PR作成
- validatorテスト
- source候補提案

許可しない範囲:

- official DB への直接write
- `user:<uid>` への直接upsert
- memory確定
- Source Registry への無審査追加
- 検索結果の自動大量投入

## 13. データ肥大化対策

長期運用時のDB肥大化には、寿命・圧縮・分層・間引きで対応する。

| 対策 | 内容 |
|---|---|
| 寿命 | L0/L1は短命。要約後に圧縮・削除 |
| 圧縮 | Parquet + Zstd、embedding量子化 |
| 分層 | Hot / Warm / Cold に分ける |
| 間引き | 参照されない低スコア記憶を削除 |
| 重複排除 | content_hash、正規化キー、MinHash |
| 共起制御 | PMIの低いedge削除、出次数上限 |
| Search Cache | 同一・類似検索の抑制 |

運用頻度案:

| 頻度 | 対象 |
|---|---|
| 毎時 | 重要ニュース、障害情報、必要時のみ |
| 毎日 | ニュース、AI公式blog、RSS |
| 毎週 | GitHub releases、Hugging Face注目モデル、映画/音楽DB差分 |
| 毎月 | Wikipedia系まとめ、知識DB再クラスタリング、共起再計算 |
| 必要時 | 外部検索、未知語、裏取り |

## 15. 成長システム

キャラはファインチューンではなく、KPIと設定更新で成長する。

| KPI | 内容 |
|---|---|
| `user_thumbs_up` | ユーザーが良い反応をした |
| `answer_hits` | 推薦や回答が採用された |
| `search_success` | 検索・引用が役立った |
| `recall_success` | 想起が会話に役立った |
| `conversation_flow` | 会話の流れが自然だった |

例:

```text
level = floor(sqrt(total_kpi / 10))
```

レベルは衣装、バッジ、口調の微調整に反映する構想。

## 16. 実装ロードマップ

最初に作るべきは「賢いキャラ」ではなく、**汚れない記憶と想起の土台**である。

### Phase 0: 設計固定

```text
config/system.yaml
personas/registry.yaml
memory/schema.sql
source_registry.yaml
staging_schema.json
```

決めるもの:

- EventId形式
- user_id形式
- namespace形式
- memory type
- source trust score
- 検索API fallback条件

### Phase 1: 1キャラ + L0/L1/L2想起

```text
User -> L0/L1/L2 recall -> Persona prompt -> Local LLM -> response
```

まず毎ターン想起を実装する。
L3は最初は no-op でもよい。

### Phase 2: ユーザー記憶

操作:

```text
覚えて
忘れて
ピン留め
これは違う
要約して保存
```

最初は明示保存だけにする。推測保存は後回し。

### Phase 3: staging / validator

外部取得データと記憶候補を正式DBに入れる前に検証する。

### Phase 4: Source Registry + RSS取得

最初はRSS 5〜10本で十分。

候補:

- AI公式blog
- 技術ニュース
- 一般ニュース
- 映画/文化系
- GitHub release

### Phase 5: Knowledge DB最小版

映画・小説・音楽などを少量から入れる。
最初は各ドメイン10〜100件程度で、正規化・検索・想起の動作確認をする。

### Phase 6: 3キャラ化

ルミナ / クラリス / ノクスを LangGraph subgraph として分離し、Routerで発話順を制御する。
