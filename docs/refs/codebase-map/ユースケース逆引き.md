## ユースケース: Viewerから会話を送信してMio/Worker/Coderへ流す

### トリガー

- Browser Viewerのchat inputまたはHTTP clientが`/viewer/send`へPOSTする。

### 処理フロー

1. **[route登録]** `cmd/picoclaw/routes.go` -> `registerViewerDynamicRoutes()`
   - `dependencies.viewerSend`を`/viewer/send`へ登録する。
2. **[HTTP adapter]** `internal/adapter/viewer/handler_send.go` -> `HandleSendWithAttachments()`
   - request validation、attachment保存、message handler呼び出し、JSON応答を行う。
3. **[会話処理]** `internal/application/orchestrator/message_orchestrator*.go`
   - routing、session、agent/provider/tool/memory連携を進める。
4. **[live更新]** `internal/adapter/viewer/hub.go` / `monitor_reducers.go`
   - orchestrator eventをSSEとmonitor stateへ反映する。

### 判断ポイント

| 箇所 | 条件 | Trueの場合 | Falseの場合 |
|---|---|---|---|
| `handler_send.go` | request/attachmentが妥当 | orchestratorへ渡す | HTTP error/onError |
| orchestrator routing | CODE/OPS/CHATなどの分類 | 該当agent/tool経路へ委譲 | Chat/Mio側で応答 |
| EventHub | Viewer接続あり | SSEで即時表示 | historyには残るが画面更新は次回取得 |

### よくある誤解

- **誤解**: `/viewer/send`だけ見れば会話全体が分かる。  
  **実際**: HTTP adapterは薄く、routing/session/tool/memoryはorchestratorとdomain/infrastructureを追う必要がある。

## ユースケース: Viewer音声入力をSTTから会話経路へ渡す

### トリガー

- Viewerの音声入力、STT capture/admin endpoint、またはVoiceChat websocket/runtime。

### 処理フロー

1. **[route登録]** `cmd/picoclaw/routes.go` -> `registerSTTAndAudioRoutes()`
2. **[STT capture]** `internal/adapter/viewer/stt_capture_handler.go`
3. **[STT rules]** `modules/stt/session_rules.go` -> `NormalizeTranscriptText()`
4. **[voice direct]** `internal/application/orchestrator/voice_direct.go`
5. **[TTS/audio]** `internal/infrastructure/tts`, `internal/adapter/viewer` TTS routes

### 判断ポイント

| 箇所 | 条件 | Trueの場合 | Falseの場合 |
|---|---|---|---|
| STT health/admin | STT runtimeが応答 | 音声入力を処理 | admin restart/diagnostics対象 |
| transcript正規化 | 空/重複/ノイズ | draft/finalの扱いを調整 | 会話経路へ投入 |
| voice direct | orchestratorが利用可能 | Mio応答/TTSへ連携 | error event/log |

### よくある誤解

- **誤解**: STT不具合は常に`RenCrow_STT`だけの問題。  
  **実際**: Viewer capture、modules/stt正規化、orchestrator voice direct、外部STT runtimeの境界を分ける。

## ユースケース: IdleChatを開始・停止・監視する

### トリガー

- `/viewer/idlechat/start`、`/viewer/idlechat/stop`、`/viewer/idlechat/status`など。

### 処理フロー

1. **[route登録]** `cmd/picoclaw/routes.go` -> `registerIdleChatRoutes()`
2. **[handler]** `cmd/picoclaw`のIdleChat handler群
3. **[application]** `internal/application/idlechat/orchestrator*.go`
4. **[topic/quality/watchdog]** `internal/application/idlechat/topic_*`, `quality_*`, `watchdog_*`
5. **[Viewer event]** EventHub/monitorへ状態を通知

### 判断ポイント

| 箇所 | 条件 | Trueの場合 | Falseの場合 |
|---|---|---|---|
| start gate | LLM opsが利用可能 | IdleChat開始 | LLM ops status確認 |

| manual stop | stop route実行 | status/logで停止確認 | 残タスク確認 |
