## 1. 概要

ストーリーモードは IdleChat の特殊セッション形式。昔話・童話を変換軸（5スタイル）で改変し、短編朗読物語として Mio が語る。

### 1.1 目的

- アイドル時間に「作り話」としての聞き応えのあるコンテンツを生成
- 毎回異なる改変スタイルとジャンルの組み合わせで多様性を確保
- TTS 朗読と Viewer タイムライン表示を前提とした短文出力

### 1.2 設計思想

- **LLM は物語本文だけ**: 準備（Step 2〜6）はすべて決定論的。LLM を使うのは草稿生成（Step 7）と改稿（Step 8）のみ
- **フォールバック多重化**: LLM が失敗しても決定論的ドラフト → 原話再話の順で必ず何かを出力できる
- **骨格検証**: 生成結果が認識手がかりを含むか・散文らしいかを機械的にチェックし、不合格なら棄却
- **元話との距離管理**: 元話の文章をそのままコピーした出力を verbatim コピーとして検出・棄却

## 2. パイプライン全体図

```
RunStorySession()
│
├── Step 1: スタイル選択 [決定論的]
│       chooseStoryRewriteStyle(history)
│       → role_shift / view_shift / value_shift / inversion / scale_shift
│
└── loop (最大 3 ソース試行)
    │
    ├── prepareStory(style, excluded)  ← Step 2〜6 / LLMなし
    │   ├── Step 2: ソース選択  selectStorySourceExcluding()
    │   ├── Step 3: ソース分析  analyzeStorySource()
    │   ├── Step 4: 改変プラン  buildStoryRewritePlan()
    │   ├── Step 5: ビートプラン groundedStoryBeatPlan()
    │   └── Step 6: アダプテーション buildStoryAdaptationPlan()
    │       → StoryPrep を返す
    │
    ├── Step 7: 第1稿生成 [LLM: shiro]
    │       retryStoryDraft() → generateStoryDraftByBeats()
    │       ・4ビートを個別にLLM生成して結合
    │       ・MaxTokens=300, Temperature=0.6
    │       ・失敗時: deterministicStoryDraft → safeStoryRetelling
    │
    └── Step 8: 改稿 [LLM: shiro]
            retryStoryRevision() → reviseStoryNarrative()
            ・第1稿を受け取り一括改稿
            ・MaxTokens=1800, Temperature=0.5
            ・出力形式: REVISION_NOTE:\nSTORY:
            ・失敗時: repairStoryDraft → deterministicStoryDraft

成功後: 物語をチャンク分割して emitStoryChunk() 経由で送出
失敗時（3ソース全失敗）: idle モードで通常チャットに切り替え
```

**LLM 境界**: `prepareStory()` の戻り値 `StoryPrep` が LLM-free フェーズの終点。

## 3. データ構造

### 3.1 StorySource（物語ソース）

```go
type StorySource struct {
    ID           string  // 例: "momotaro"
    Title        string  // 例: "桃太郎"
    SourceLabel  string  // 例: "日本昔話"
    Kind         string  // "folktale"
    Language     string  // "ja"
    PublicDomain bool
    Text         string  // 物語の要約テキスト
}
```

### 3.2 StorySkeleton（物語の骨格）

```go
type StorySkeleton struct {
    ID                  string      // StorySource.ID と同じ
    SourceTitle         string
    CanonicalMotifs     []string    // 例: ["桃から生まれる", "きびだんご", ...]
    RequiredBeats       []StoryBeat // 例: [{id:"birth", label:"異様な出発"}, ...]
    RoleConstraints     []string    // 例: ["育てられた主役", "同行する三人の仲間", ...]
    TabooOrRule         string      // 例: "独りの手柄にせず、与えたものの縁を裏切らないこと"
    RewardPunishment    string      // 例: "分け与えた信頼が勝利を呼び、奪う側は拠点ごと失う"
    EmotionalAftertaste string      // 例: "連帯の熱と戦いの荒さが同時に残る"
    RecognitionCues     []string    // 例: ["桃", "きびだんご", "犬", "猿", "雉", "鬼ヶ島"]
}

type StoryBeat struct {
    ID    string  // 例: "birth"
    Label string  // 例: "異様な出発"
}
```

### 3.3 StorySpec（物語スペック）

```go
type StorySpec struct {
    Skeleton StorySkeleton
    Twists   map[string]string  // スタイル名 → 1文の変換軸
}
```

### 3.4 StoryRewritePlan（改変プラン）

```go
type StoryRewritePlan struct {
    SourceTitle  string
    RewriteStyle string    // "role_shift" 等
    StoryTitle   string    // 例: "ノワール版・桃太郎"
    Premise      string    // 例: "討伐される者から見た正義の構造を、ノワールの文脈で描く。"
    Setting      string    // 例: "深夜の港の倉庫街"
    Viewpoint    string    // 例: "対立役の近接一人称"
    Tone         string    // 例: "乾いた緊張"
    Hook         string    // 変換軸の文（storyTransformationAxis の返り値）
    EndingShape  string    // 例: "力の構造が最後に反転する"
    EndingFlavor string    // 例: "構造の露呈"
    CoreMotifs   []string  // Skeleton.CanonicalMotifs のコピー
    MotifMap     []string  // 例: ["桃から生まれる=>出自を持ち出す武器", ...]
}
```

### 3.5 StoryBeatPlan（ビートプラン）

4つのビートを日本語の指示文として保持する。

```go
type StoryBeatPlan struct {
    Opening   string  // 導入ビート
    Deviation string  // 逸脱ビート
    Reversal  string  // 反転ビート
    Landing   string  // 着地ビート
}
```

### 3.6 StoryAdaptationPlan（アダプテーションプラン）

SkeletonビートとBeatPlanを対応付けたメタ情報。LLMプロンプトには渡さず、品質検証に使用。

```go
type StoryAdaptationPlan struct {
    SkeletonID      string
    RewriteStyle    string
    BeatMappings    []string  // "元ビート=>改変ビート" 形式
    MotifMappings   []string  // MotifMap のコピー
    RoleRemap       []string
    EndingFlavor    string
    RecognitionCues []string
}
```

### 3.7 StoryPrep（準備フェーズ出力）

Step 2〜6 の出力をまとめた構造体。これがLLMフェーズへの入力になる。

```go
type StoryPrep struct {
    Source     StorySource
    Analysis   StorySourceAnalysis
    Plan       StoryRewritePlan
    BeatPlan   StoryBeatPlan
    Adaptation StoryAdaptationPlan
}
```

## 4. 各ステップの詳細

### 4.1 Step 1: スタイル選択（`chooseStoryRewriteStyle`）

直近の履歴（最大12件）から直前のスタイルを読み取り、同じスタイルを候補から除外してランダム選択。

**5スタイル**:

| スタイルID | 日本語ラベル | 概念 |
|---|---|---|
| `role_shift` | 役割転換 | 対立役・被支配側の視点で語る |
| `view_shift` | 視点変更 | 傍観者・脇役の目線で語る |
| `value_shift` | 価値反転 | 善悪・損得の評価を逆転させる |
| `inversion` | 因果反転 | 結末や因果を逆にする |
| `scale_shift` | 規模変換 | 事件の及ぶ範囲・規模を変える |

### 4.2 Step 2: ソース選択（`selectStorySourceExcluding`）

- 環境変数 `STORY_SOURCE` が設定されている場合はそのIDを強制使用（テスト用）
- 履歴（最大12件）から直近4件で使用したソースをブロックリストに追加
- ブロックリストと除外セット（`excluded`）を除いたプールからランダム選択
- プールが空になった場合は段階的にブロックを解除して選択

### 4.3 Step 3: ソース分析（`analyzeStorySource`）

`storySkeleton(source)` で data/story/*.json の `spec.skeleton` を読み込み、`StorySourceAnalysis` を構築する。JSONにスペックがない場合は空のSkeletonを返す（フォールバック）。

### 4.4 Step 4: 改変プラン（`buildStoryRewritePlan`）

決定論的にパラメータを決定する:

1. **ジャンル**: `storyGenres` からランダム選択（ノワール/ホラー/コメディ/ノーマル）
2. **変換軸（Hook）**: `storyTransformationAxis()` — JSONの `twists[style]` を使用、なければデフォルト文を生成
3. **スケール**: `scale_shift` の場合のみ「極小」「極大」からランダム選択
4. **舞台**: ジャンルごとの固定候補（`storySettingsByGenre`）からランダム選択
5. **タイトル・視点・トーン・結末形・結末余韻**: スタイルごとの固定テンプレートで決定
6. **MotifMap**: Skeleton.CanonicalMotifs を変換（`transformMotif(style, motif)`）

**ジャンル別舞台候補**:

| ジャンル | 舞台候補 |
|---|---|
| ノワール | 深夜の港の倉庫街 / 錆びた橋の下の路地 / 雨の降る古い商店街 |
| ホラー | 霧の深い山道 / 廃屋の離れ / 人気のない神社の境内 |
| コメディ | にぎやかな市場の片隅 / 田舎の郵便局前 / 裏通りの八百屋 |
| ノーマル | 川沿いの小さな町 / 山あいの集落 / 海辺の漁村 |

### 4.5 Step 5: ビートプラン（`groundedStoryBeatPlan`）

`StorySkeleton.RequiredBeats` のラベルを使い、4ビートを日本語の指示文として構築:

```
Opening:   "<RequiredBeat[0]>を、<setting>で始める。"
Deviation: "<RequiredBeat[1]>が起こり、<rewriteStyleLabel>というひねりが見える。"
Reversal:  "<RequiredBeat[2]>によって、<CoreMotifs[0]>と<CoreMotifs[1]>という元話の骨格をはっきり見せる。"
Landing:   "<RequiredBeat[3]>で終え、最後に<EndingFlavor>が残る。"
```

RequiredBeats が足りない場合は「導入」「逸脱」「反転」「着地」のデフォルトラベルを使用。

### 4.6 Step 6: アダプテーションプラン（`buildStoryAdaptationPlan`）

Skeleton の各ビートと BeatPlan の文を `"元ビート=>改変ビート"` 形式で対応付けた `BeatMappings` を生成。品質検証（`storySatisfiesSkeleton`）で使用する。

### 4.7 Step 7: 第1稿生成（`generateStoryDraftByBeats`）

ビートごとに LLM を呼び出し、段落を順次生成して結合する。

**LLM設定**:
- モデル: `shiro`（Ollama `Worker` = qwen3.5:9b）
- MaxTokens: 300（1ビートあたり）
- Temperature: 0.6

**プロンプト構造**（各ビートに送信、`buildBeatMessages` 実装準拠）:
```
system: "あなたは朗読短編作家です。指定された場面だけを、聞き取りやすい日本語の短い段落で書いてください。"
user:
  元作品: <source.Title>
  改題: <plan.StoryTitle>
  改変の核（この一点を物語に反映させる）: <storyTransformationAxis(source, style)>
  文体・トーン: <plan.Tone>
  舞台: <plan.Setting>
  視点: <plan.Viewpoint>
  必須モチーフ: <CoreMotifs>
  認識手がかり: <RecognitionCues>
  前の場面（完結済み・この出来事は書かない）: <prevBeatContent>（Beat 2 以降のみ）
  今回書く場面: <section.label>（前の場面とは異なる出来事）
  この場面の役割: <section.content>
  後の場面に残す（この場面では書かない）: <forbiddenBeats>（後続ビートが存在する場合）
  要件: [禁止事項リスト]
  <Beat 3 のみ: endingFlavorHint() による EndingFlavor の具体指示>
  <beatInstruction(i, openingSeed)>
  出力は本文だけ
```

> **注意**: `元話の要約本文`（source.Text）はビートプロンプトに含めない。LLM が元話に引きずられる原因となるため除外（Q9対策）。

**第1ビートの特別処理**: `beatInstruction(0, openingSeed)` により「第1文は必ずこの文をそのまま使う: <openingSeed>」が追加される。

**openingSeed**: 作品IDごとに個別の固定開始文（例: 桃太郎 → `"その夜、桃太郎は川沿いの倉庫で小さな包みを仲間に配った。"`）。デフォルトは汎用テンプレート。

**ビート別棄却条件**:
- 空文字列
- `storyHasOutlineLanguage()`（メタ言語を含む）
- `storyHasOverblownSetting()`（現代ハイテク語句を含む）
- `storyParagraphIsVerbatimCopy()`（元話の文を15文字以上丸コピー）

**全ビート結合後の棄却条件**:
- `storyNarrativeLooksLikeProse()` が false
- `storySatisfiesSkeleton()` が false

**フォールバック（retryStoryDraft 内）**:
1. LLM 3回失敗 → `deterministicStoryDraft()`（テンプレートベース）
2. deterministicStoryDraft が空 → `safeStoryRetelling()`（openingSeed + 元話本文）

### 4.8 Step 8: 改稿（`reviseStoryNarrative`）

第1稿を受け取り、LLM が一括で第2稿に改稿する。

**LLM設定**:
- モデル: `shiro`（同上）
- MaxTokens: 1800
- Temperature: 0.5

**プロンプト構造**:
```
system: "あなたは朗読短編の編集者です。..."
user:
  元作品: <source.Title>
  元話の要約本文: <source.Text>
  改題: <plan.StoryTitle>
  改変方式: <plan.RewriteStyle>
  余韻: <plan.EndingFlavor>
  必須モチーフ: <CoreMotifs>
  必須イベント順: <BeatLabels>
  認識手がかり: <RecognitionCues>
  禁忌/約束: <analysis.TabooOrRule>
  報酬と罰: <analysis.RewardAndPunish>
  ビート: 導入/逸脱/反転/着地の各文
  第1稿: <draftText>
  改稿方針: [詳細ガイドライン]
  出力は次の形式だけ
  REVISION_NOTE:
  STORY:
```

**改稿ガイドライン（主要禁止事項）**:
- 元話の要約本文を文章ごとコピーしない
- 第1文は `openingSeed` を一字一句変えずに保持
- 巨大企業・AI支配・世界規模の話に膨らませない
- SNS・スマホ・会員制施設・プロジェクト名を新作しない
- 幼少期の回想・象徴の説明・教訓の言い直しを足さない
- 二人称で説教・勧誘しない

**出力パース**: `parseStoryRevision()` で `REVISION_NOTE:` と `STORY:` を分離。

**棄却条件**:
- `storyNarrativeLooksSettled()` が false（メタリーク、文末が文末記号でない等）
- `storyNarrativeLooksLikeProse()` が false
- `storySatisfiesSkeleton()` が false

## 5. 品質検証

### 5.1 storyNarrativeLooksLikeProse（散文判定）

以下をすべて満たすこと:

| 条件 | 判定 |
|---|---|
| 160文字（rune）以上 | ≥160 |
| メタ言語なし | `!storyHasOutlineLanguage()` |
| 過剰なハイテク語なし | `!storyHasOverblownSetting()` |
| 脱線なし | `!storyHasDistractingDigression()` |
| 文末記号（。！？）が3つ以上 | sentenceCount ≥ 3 |
| 文末記号（。！？）が32以下 | sentenceCount ≤ 32 |

### 5.2 storyNarrativeLooksSettled（完成度判定）

改稿後のみ適用:

| 条件 | 判定 |
|---|---|
| メタリーク文なし | `!storyHasMetaLeak()` |
| 最終文字が文末記号 | `。！？」』` のいずれか |

### 5.3 storySatisfiesSkeleton（骨格充足判定）

`RecognitionCues` のうち2つ以上が物語本文中に含まれること。

### 5.4 storyHasOutlineLanguage（メタ言語検出）

以下の文字列が1つでも含まれれば棄却:

```
"どうひねったか", "よく分からないけど", "物語の始まりを予感",
"最初の違和感として立ち上がる", "意外な意味に変わる",
"因果として結び直される", "最後に残るのは",
"という感触だった", "導入:", "逸脱:", "反転:", "着地:",
"要件:", "改稿方針:", "REVISION_NOTE:", "STORY:"
```

### 5.5 storyHasOverblownSetting（過剰設定検出）

以下の文字列が含まれれば棄却:

```
"AI開発部", "量子コンピューター", "未来テック", "2040"〜"2045",
"巨大企業", "世界最大手", "大規模言語モデル", "量子",
"SNS", "いいね", "観光客", "スマホ", "会員制",
"不動産会社", "株式会社", "開発計画", "プロジェクト",
"ランキング", "評価システム", "アプリ", ...
```

また「まるで」が9回以上、冒頭60文字内に「あなたは」がある場合も棄却。

### 5.6 storyHasDistractingDigression（脱線検出）

```
"幼い頃", "子どもの頃", "思い出した", "思い出す",
"記憶のよう", "象徴している", "物語の一部だった",
"結局のところ", "最も恐ろしい", "悪だった", "唯一無二の宝"
```

### 5.7 storyHasMetaLeak（メタ情報混入検出）

改稿専用:

```
"元の『", "元作品", "禁じられていたのは",
"ここではそれが別の形", "読後感だった",
"改変方式", "必須モチーフ", "報酬と罰"
```

## 6. フォールバック階層

LLM 失敗時は以下の順で代替出力を試みる:

```
1. retryStoryDraft: LLM を最大3回リトライ
   ↓ 全失敗
2. deterministicStoryDraft: テンプレートベース生成
   ・openingSeed + 元話文章から段落を組み立て
   ・skeletonチェックを通過しなければ空を返す
   ↓ 空
3. safeStoryRetelling: 最小限の再話
   ・openingSeed + source.Text + EndingFlavor で結合

改稿フェーズのフォールバック:
1. retryStoryRevision: 最大3回リトライ
   ↓ 全失敗
2. repairStoryDraft: 第1稿を機械的に整形
   ・stripStoryMetaLeak で設計説明文を除去
   ・EndingFlavor がなければ BeatPlan.Landing を末尾追加
   ・skeletonチェック失敗時は deterministicStoryDraft へ
```

3ソース試行後も成功しない場合は `idle` モードに戻り `runChatSession(StrategySingleGenre)` を実行。

## 7. 正規化処理

### 7.1 normalizeStoryNarrative

LLM 出力のクリーニング:

1. 改行コード統一（`\r\n` → `\n`）
2. `REVISION_NOTE:` / `STORY:` の除去
3. `---`、`*   **`、`- ` で始まる行以降を除去（Markdown リストが出た時点で切る）
4. LLM 口癖除去（「わかりました。」「以下に、」「いかがでしょうか」）
5. `（余韻）`「余韻:」行の除去
6. `#` で始まる行の除去
7. 後処理: `stripStoryMetaSentences`、`dedupeStoryParagraphs`、`dedupeStorySentences`
8. 連続3改行を2改行に圧縮（`\n\n\n` → `\n\n`）

### 7.2 重複除去

- **dedupeStoryParagraphs**: `\n\n` で分割し、`storySignature` が重複する段落を除去
- **dedupeStorySentences**: 句読点で分割し、同一文を除去
- **storySignature**: 記号・空白・句読点を除いた文字列で比較

## 8. 出力フォーマット

### 8.1 チャンク分割（splitStoryNarration）

| パラメータ | 値 |
|---|---|
| `storyChunkMaxRunes` | 90 rune |
| `storyChunkMinRunes` | 28 rune |

物語テキストを最大90文字のチャンクに分割して送出。TTS 朗読と Viewer タイムライン表示の両方を想定。

### 8.2 発話フォーマット

```
"今夜の物語です。元になったのは『<SourceTitle>』。<Hook>。"
"改題は『<StoryTitle>』。"
<storyText（チャンク分割）>
"元の『<SourceTitle>』を下敷きにした、<rewriteStyleLabel>の物語でした。"
```

```go
    StoryEndingFlavor: <string>,
    TopicProvider:     "shiro",
    SummaryProvider:   "shiro",
    Transcript:        [発話チャンク配列],
}
```

### 9.2 履歴の活用

- `chooseStoryRewriteStyle`: 最大12件の履歴で直前スタイルを除外
- `selectStorySourceExcluding`: 最大12件の履歴で直近4件のソースをブロック
- ソース重複判定は `SourceTitle` 文字列の一致で行う（ID ではない）

### 10.3 twists の設計方針

`twists` は各改変スタイルに対して「この物語でそのスタイルを適用したとき、どこを変えるか」を **1文の抽象的構造観察** として記述する。

- LLM が物語を生成する前に `storyTransformationAxis()` で参照される
- JSON になければ `source.Title` を使ったデフォルト文が生成される
- 具体的なプロット指示ではなく、構造・関係性の観察として書く

**例（桃太郎）**:
```json
"twists": {
  "role_shift": "討伐される者から見た正義の構造",
  "view_shift": "英雄の使命を与える側と担う側の立場の差",
  "value_shift": "征服によって得た宝の正当性",
  "inversion": "この物語の鬼が善であったら何が残るか",
  "scale_shift": "使命が及ぶ範囲を変えたとき、何が見えるか"
}
```

## 13. 設計上の制約・注意事項

### 13.1 元話テキストの扱い

- `source.Text` は LLM への文脈提供のために渡すが、**本文の丸コピーを禁止**
- `storyParagraphIsVerbatimCopy()` は15文字（rune）以上の一致文を検出
- プロンプト中に「元話の要約本文の文章をそのままコピーしない」を明示

### 13.2 ジャンルと舞台の分離

- ジャンル（ノワール/ホラー等）はトーンと舞台候補を決める
- 舞台はジャンル別候補からランダム選択されるが、各作品IDごとの `defaultGroundedStorySetting` が個別に定義されている（未使用の場合あり）

### 13.3 openingSeedの固定

アクティブ17作品それぞれに固定の第1文（openingSeed）を持つ。これにより:
- 生成された物語が必ず一定の文から始まる（再現性）
- 「情景描写で始まるな」という要件を確実に満たす
- 改稿フェーズでも「第1文は変えるな」として保持させる

### 13.4 LLM パラメータの意図

| パラメータ | 第1稿 | 改稿 |
|---|---|---|
| Temperature | 0.6（創造的多様性を確保） | 0.5（安定しつつ整文の自由度を残す） |
| MaxTokens | 300（1ビート=短段落） | 1800（全文・settling失敗対策で1400から引き上げ） |

```
  - 本文のみ出力（解説・メタ発言不要）
```

### 15.7 設計意図と位置づけ

- **エンタメ優先**: バリデーションなし・Temperature 0.9 で、失敗リスクより多様性を優先
- **Coder2 使用**: 高品質な文章生成が期待できる OpenAI 系モデルを使用（通常モードの Ollama shiro とは異なる）
- **独立フロー**: `data/story/*.json` のコーパス整備状況に依存せず動作可能
- **補完的役割**: 通常ストーリーモードのフォールバックではなく、別モードとして使い分け
