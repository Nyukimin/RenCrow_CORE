# RenCrow Viewer 新 UI タブ生成プロンプト

このプロンプトは、RenCrow Viewer に Home / Chat / Develop / Instructions / Reports などの常用 UI タブを追加または再設計するときに使う。

作業前に必ず `rules/rules_viewer_ui.md` を読むこと。このプロンプトは同ルールを LLM に渡しやすい形へ整理したものであり、ルールの代替ではない。

````md
# TASK: RenCrow Viewer 新 UI タブの HTML / CSS / JS を作成する

## 目的

RenCrow Viewer に、新しい常用 UI タブを追加または再設計する。

今回作りたい画面は、添付画像のような雰囲気の「夜の作業机」風 UI である。

画像をそのまま模写するのではなく、実際に使える HTML / CSS / JavaScript として、読みやすく、迷わず使える画面にする。

## 最重要方針

新 UI は、従来の監視ダッシュボードではない。

目的は、れんが日常的に使うための「大きく、読みやすく、迷わない、AI キャラクターが横にいる作業机」を作ることである。

重要: このタスクでは「便利そうな情報を追加する」ことを禁止する。まず少ない情報量で読みやすい画面を完成させること。

## 必ず参照するもの

- `AGENTS.md`
- `CLAUDE.md`
- `rules/rules_viewer_ui.md`
- `rules/common/rules_frontend.md`
- `docs/09_Viewer/Viewer仕様.md`
- 対象実装ファイル
  - `internal/adapter/viewer/viewer.html`
  - `internal/adapter/viewer/assets/css/`
  - `internal/adapter/viewer/assets/js/`

## 絶対に守る UI ルール

- 大きなキャラクター表示を左側、または画面の主導線上に置く。
- 大きな会話カード、または大きな情報カードを中央に置く。
- 入力画面では入力欄を下部に置く。
- Chat の最下行入力欄は常時画面内に表示する。
- Chat の最下行入力欄は 1 から 2 行程度の高さに抑え、無駄に縦へ大きくしない。
- 閲覧画面では明確な次アクション領域を下部または主導線上に置く。
- 情報量を少なくする。
- 画面タイトルは 20px から 22px を基準にする。
- 本文文字は 12px から 14px を基準にする。
- 入力欄は 13px から 14px を基準にする。
- タブ、ピル、短いメタ表示は 10px 未満にしない。
- 夜の作業机の雰囲気にする。
- 必要な情報だけを表示する。
- 詳細ログや内部状態は初期表示しない。
- 監視ダッシュボード風にしない。
- 小さいログ一覧にしない。
- 右側に細かいカードや監視パネルを大量に並べない。
- 専用タブがある情報を、常用画面の右サイドへ重複表示しない。

## デザイン参照

添付画像 `image.png` がある場合はデザイン参照である。

再現すべき要素:

- ダークネイビー系の背景
- 透明感のあるガラス風パネル
- 青から紫の控えめなグロー
- 大きな角丸カード
- 左側の大きなキャラクター領域
- 中央の大きな会話カードまたは情報カード
- 下部の大きすぎない入力欄、または閲覧画面では明確な次アクション領域
- 余白の多い構成
- 読みやすい文字

再現しないこと:

- 画像そのものの完全コピー
- キャラクター画像の完全再現
- 細かい装飾の追加
- ログ、DB、ステータス、監視パネルの常時表示
- 白背景や業務ダッシュボード風の見た目

## 実装対象

次のどれか 1 画面を対象にする。

```text
Home / Daily Desk
Chat
Develop
Instructions
Reports
```

ユーザー指定がある場合はそれに従う。指定がない場合は `Chat` を作る。

## 画面別の表示内容

### Home / Daily Desk

出してよいもの:

- 今日の状態
- 最後の会話
- 進行中の作業
- 未読レポート
- 未完了指示
- 呼び出せるキャラクター
- 入力欄

出しすぎてはいけないもの:

- 詳細ログ全文
- 開発者向けイベント一覧
- 技術監視の細かな内部状態

### Chat

出してよいもの:

- 会話カード
- 現在の対話相手
- 送信先切替
- 入力欄
- 音声入力、添付、送信
- 必要最小限の状態表示

出しすぎてはいけないもの:

- Memory / Source Registry / Recall Trace の常時サイド表示
- 生ログ全文
- Debug 情報
- Jobs / Reports / Develop 情報の常時混在

### Develop

出してよいもの:

- 現在の開発タスク
- 指示内容
- 担当 Agent
- 進捗フェーズ
- 作業ログの要約
- 差分・成果物
- 確認待ち
- エラーと再試行
- 次に押すボタン

出しすぎてはいけないもの:

- 生のイベントログ全文
- 関係ないセッション一覧
- 監視用の細かすぎるメトリクス

### Instructions

出してよいもの:

- 未完了指示
- 進行中指示
- 保留中指示
- 完了済み指示
- 優先度
- 対象 Agent
- 期限またはタイミング
- 再指示ボタン
- キャンセル
- 指示から生成された Job

出しすぎてはいけないもの:

- 指示と無関係な長文ログ
- 技術詳細の羅列

### Reports

出してよいもの:

- 作業完了報告
- 要約
- 何を変更したか
- 何を確認したか
- 検証結果
- 失敗したこと
- 未確認のこと
- Evidence
- 成果物リンク
- 次の判断
- Copy Summary
- Markdown Export

出しすぎてはいけないもの:

- 生ログ全文
- 人が読みにくい JSON の羅列を初期表示すること

## レイアウト制約

- 1920x1080 を基準にする。
- 1366x768 でも破綻しないこと。
- 画面全体をスクロール前提にしすぎない。
- 左キャラクターパネルは固定幅を基本にする。
- メイン領域を広く取る。
- 右サイドへ常時表示する補助パネルは原則追加しない。
- 専用タブがある情報は、その専用タブへ逃がす。
- 詳細は折りたたみ、drawer、別画面、別タブに逃がす。
- 固定ヘッダー、固定フッター、固定入力バーに隠れない余白を確保する。
- スクロール領域を作る場合は、末尾要素まで実クリックできることを確認する。

## 推奨 DOM 構造

UI 検討用モックの場合:

```html
<body>
  <main class="rc-shell">
    <aside class="rc-character-pane">
      <header class="rc-brand"></header>
      <section class="rc-character-portrait"></section>
      <section class="rc-status-card"></section>
      <section class="rc-character-grid"></section>
    </aside>

    <section class="rc-main-pane">
      <nav class="rc-tabs"></nav>
      <section class="rc-content-area"></section>
      <footer class="rc-input-dock"></footer>
    </section>
  </main>
</body>
```

RenCrow Viewer 本体へ組み込む場合:

- `viewer.html` の既存タブ構造に合わせる。
- CSS は `internal/adapter/viewer/assets/css/` の既存構成に合わせる。
- JS は `internal/adapter/viewer/assets/js/tabs/` の画面別責務に合わせる。
- 1 ファイル HTML モックをそのまま本体へ貼り付けない。
- 既存 API、SSE event、Viewer 表示契約、STT/TTS/IdleChat 契約を壊さない。

## 操作可能性と重なり防止

見えているだけでは合格にしない。主要操作対象が実際にクリック、入力、選択、コピー、スクロールで到達できることを確認する。

必須:

- すべてのタブ、主要ボタン、入力欄、select、リンク、カード操作を実クリックで確認する。
- 最上部と最下部の操作要素が、固定ヘッダー、固定フッター、固定入力バー、toast、drawer に隠れないことを確認する。
- スクロール後の最後の要素が、固定 UI の下に潜らずクリックできることを確認する。
- desktop と narrow viewport の両方で到達性を確認する。
- PC の縦タブと mobile panel select の両方で、追加した画面へ移動できることを確認する。
- `elementFromPoint`、Playwright の実クリック、または同等の確認で、クリック位置の最前面要素が意図した対象であることを確認する。

禁止:

- DOM に存在するだけで操作可能と判断すること。
- 目視で見えているだけでクリック可能と判断すること。
- スクロールバーがあるだけで到達可能と判断すること。
- 主要導線を `force click` 前提にすること。
- fixed / sticky / z-index の影響を確認せずに完了扱いすること。

## 成果物

UI 検討用モックの場合:

- 1 ファイル完結の HTML を作成する。
- HTML / CSS / JavaScript を 1 ファイルにまとめる。
- 外部ライブラリは使わない。
- 画像がなくても崩れない。
- 1920x1080 で見やすいこと。
- 1366x768 でも破綻しないこと。

RenCrow Viewer 本体へ組み込む場合:

- 既存の責務分割に従って、HTML / CSS / JS を適切なファイルへ分ける。
- 既存タブを壊さない。
- 既存の送信、STT、TTS、IdleChat、Memory、Reports、Jobs の契約を変えない。
- 専用タブがある情報を右サイドへ重複表示しない。

## 検証

最低限、次を実行する。

```bash
node --check internal/adapter/viewer/assets/js/viewer.js
GOCACHE=/tmp/rencrow-gocache go test ./internal/adapter/viewer
git diff --check
```

Viewer 本体へ組み込んだ場合は、Playwright または同等のブラウザ確認で次を確認する。

- 1920x1080 で主要操作が隠れない。
- 1366x768 で主要操作が隠れない。
- 追加タブへ移動できる。
- 既存タブへ移動できる。
- Chat の入力欄が常時画面内に表示される。
- Chat の入力欄が縦に大きすぎない。
- 固定 UI がクリックを奪っていない。
- スクロール末尾の要素がクリックできる。
- 専用タブがある情報を右サイドへ重複表示していない。

## 受け入れ条件

- 左に大きなキャラクター領域、または対話相手が分かる強い視覚要素がある。
- 中央のカードが大きく読みやすい。
- 入力画面では入力欄が常時見える。
- Chat の最下行入力欄が縦に大きすぎず、会話表示領域を圧迫していない。
- 閲覧画面では次アクションが分かりやすい。
- 情報量が少ない。
- 文字が読みやすい。
- 夜の作業机の雰囲気がある。
- 右側に監視ダッシュボードや専用タブ情報の重複表示が増えていない。
- 小さいログ一覧が主役になっていない。
- 初見で「何をする画面か」が分かる。
- 1920x1080 と 1366x768 で主要操作が隠れない。
- 固定 UI によってタブ、ボタン、入力欄、カード操作が塞がれていない。
````
