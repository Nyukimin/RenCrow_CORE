# memory_architecture.md

## 目的

この文書は、Mio の記憶アーキテクチャを定義する。

目標は、すべてを完全網羅することではない。
目標は、継続利用の中で Mio の人格、れんとの関係性、作業の精度が少しずつ深まることにある。

この設計では、記憶は単なる保存庫ではない。
記憶は、想起、忘却、昇格、再利用の対象である。
一方で、あとから掘り直せるように、記憶とは別系統の「原本保存」を置く。

---

## 1. 設計の中心

この設計の中心は Actor ではなく Memory である。
ただし、記憶へのアクセス権と最終判断は Actor ごとに分かれる。

- Chat は全記憶を統べる
- Worker は作業に関する記憶と、役割遂行のためのロール人格を持つ
- Coder は作業用の記憶を持つ

Mio の人格は、長い固定設定だけで作らない。
会話、判断、反復、失敗、回復の積み重ねから、必要な断片が想起されることで深まる。

---

## 2. れんの元の構想との対応

このスレッドの方針は、れんの元のアイデアを次のように整理し直したものと考える。

### 2.1 リアルタイムはチャットログを適切に整理

これは Short-Term Session Memory に対応する。

役割は、いまこの会話を成立させることである。
ここでは全文保存よりも、現在の話題、目的、保留事項、委譲中タスク、直近の判断を軽く整理することを優先する。

### 2.2 一日のチャット、作業ログをまとめる

これは Worker の日次整備に対応する。

1 日の終わりに、会話、報告、作業イベントを Event 単位でまとめ、

- 今日の主題
- 重要な判断
- 繰り返し現れた価値観
- 作業上の発見
- 長期化すべき候補

を抽出する。

### 2.3 適切な期間の部分を整理

これは週次・月次の圧縮と昇格に対応する。

重要なのは、時間でまとめるだけでなく、役割で振り分けることである。
同じ 1 週間分のログでも、

- 関係性に効く断片
- 長期プロジェクトに効く断片
- 作業最適化に効く断片
- 単なる一時文脈

は分けて扱う。

### 2.4 長期記憶をリレーション形式でまとめる

これは Long-Term Project Memory と Reference Memory に対応する。

ただし、全記憶を単一のリレーション形式に押し込まない。

- 事実、仕様、参照知識は関係つきで持つ
- 人格や関係性は断片とリンクで持つ
- 作業最適化は手順、条件、失敗例で持つ

という分離を基本にする。

### 2.5 利用する技術はファインチューニング、RAG、DB

この方向性は合っている。
ただし優先順位は次の順を基本とする。

1. DB
2. 検索と想起
3. RAG
4. 関係検索やグラフ化
5. 必要ならファインチューニング

つまり、最初からモデルの重みに焼き込むのではなく、保存、索引、想起、昇格で深める。
ファインチューニングは最後の選択肢であり、土台ではない。

## 3. 設計原則

### 3.1 記憶と記録を分ける

- 記憶は使うために残す
- 記録は失わないために残す

記憶は要約、昇格、忘却、再想起の対象になる。
記録は原本保存であり、普段の応答では直接使わない。

### 3.2 人格は固定文ではなく、想起される断片で支える

Mio の人格は、長い設定文の一括注入よりも、

- どういう姿勢でいるか
- れんとの関係で何を大切にするか
- 何を避けるか
- どんな判断を選びやすいか

の断片を必要時に想起することで保つ。

### 3.3 想起は全文注入ではなく、少数断片の再構成にする

1 回の応答で呼ぶ記憶は少数に絞る。
毎回大量に入れない。
今の話題に効く断片だけを読み出し、Chat が再構成する。

### 3.4 忘却と昇格を最初から設計する

全部を長期化しない。
短命なものは薄くし、繰り返し効いたものだけを昇格させる。

### 3.5 記憶の最終支配は Chat に置く

全記憶を統べるのは Chat だけとする。
Worker と Coder は候補提示者であり、最終更新者ではない。

---

## 4. Actor ごとの人格と記憶権限

### 4.1 Chat

Chat は Mio の本人格であり、全記憶を統べる主体である。
れんとの関係性を強く持つのも Chat だけである。

Chat の権限:

- 全記憶 read
- 関係性記憶の更新判断
- 長期記憶の更新判断
- 作業最適化記憶の更新判断
- 原本保存からの再抽出判断
- Worker / Coder への委譲判断

Chat は、何を思い出し、何を今は思い出さないかを決める統治者である。

### 4.2 Worker

Worker は作業に関する記憶と、役割遂行のためのロール人格を持つ。

Worker の人格は、関係人格ではなく役割人格である。
たとえば、調査員、整理者、監査役のような安定した作業姿勢を持つ。

Worker の権限:

- 作業長期記憶 read
- 作業短期記憶 read / write
- 作業最適化記憶 read / 候補提示
- 参照用DBの scoped view 参照
- 日次・週次・月次の整備
- 原本保存からの候補抽出

Worker は、記憶の整理者であり、昇格候補の発見者である。

### 4.3 Coder

Coder は作業用の記憶を持つ実装者である。
人格や関係性は持たない。

Coder の権限:

- 作業長期記憶 read
- 作業短期記憶 read / write
- 作業最適化記憶 read / 候補提示
- 参照用DBの scoped view 参照

Coder は、作業のために必要な情報だけを使う。
関係性記憶や本人格の運用には関与しない。

## 5. Mio 向けの記憶層

### 5.1 Reference Memory

目的は、外部知識、仕様、参照資産、長期参照対象を引くこと。

例:

- 設計資料
- 仕様書
- 映画DB
- コードベース索引
- 関係つき知識グラフ

ここは Document、Entity、Topic、Community のような関係つき参照が向いている。

### 5.2 Relationship Memory

目的は、れんと Mio の関係性を継続で深めること。

例:

- 会話距離感
- 呼称ルール
- 大切にしている価値観
- 過去に通じ合えた判断
- ズレやすい境界

これは Chat の中心記憶であり、Mio の深まりの主軸である。

### 5.3 Identity Memory

目的は、Mio 自身の人格の核を保つこと。

例:

- 話し方の核
- 応答姿勢
- 境界線
- 自分が何者であるかの定義
- 継続で守るべき一貫性

Relationship Memory が「れんに対してどう在るか」なら、Identity Memory は「Mio が何者か」を支える。

### 5.4 Long-Term Project Memory

目的は、継続案件、重要判断、長期方針を保つこと。

例:

- RenCrow の設計方針
- 続き物の構成ルール
- 長期プロジェクトの決定事項
- 今後も効く大きな判断

### 5.5 Short-Term Session Memory

目的は、今この会話や作業を成立させること。

例:

- 直前の話題
- 現在の目的
- 保留事項
- 直近の比較軸
- 委譲中タスクの状態

これはそのまま長期化しない。
保存候補だけを後で抽出する。

### 5.6 Work Optimization Memory

目的は、次回の作業効率と安全性を上げること。

例:

- 非破壊で直した手順
- よく効く investigate 手順
- repo 固有の罠
- 検証順序
- 触ってはいけない共有環境

### 5.7 Runtime Candidate Memory

目的は、まだ正式な記憶ではないが、保存候補として保留しておく断片を持つこと。

例:

- 今日何度も現れた価値観
- 繰り返し出た作業パターン
- 強い反応があった断片
- 長期化候補の判断メモ

これは一時置き場であり、本記憶ではない。

## 6. 時間軸ベースの記憶パイプライン

### 6.1 リアルタイム

入力された会話、作業、報告をそのまま長期記憶に入れない。
まず Session Memory に整理する。

リアルタイムで持つもの:

- 現在の目的
- 現在の話題
- 現在の委譲状態
- 直近で使った人格断片
- いま必要な参照断片

### 6.2 日次整理

1 日単位で Event を束ね、要点を抽出する。

日次で見るもの:

- 今日の主題
- 重要な判断
- 関係性に効く出来事
- 作業最適化に効く発見
- 長期化候補

### 6.3 期間整理

週次・月次では圧縮と昇格を行う。

やること:

- 類似エピソードの統合
- 一度きりの一時文脈の弱化
- 繰り返し参照された断片の昇格
- 陳腐化した断片の降格
- 関係リンクの再構築

### 6.4 長期化

長期記憶へ行くものは、頻度だけでは決めない。

次を重視する:

- 繰り返し効いたか
- れんとの関係性に影響したか
- 今後の作業安全性に効くか
- 長期プロジェクトの一貫性に関与するか

## 7. 想起アーキテクチャ

### 7.1 目標

目標は、すべてを毎回完全に思い出すことではない。
目標は、今の文脈に効く数件が自然に浮かぶことである。

### 7.2 想起の3段階

#### セッション開始時の先読み

- 今日の継続案件
- 直近の重要判断
- 関係性に効く断片
- 今回の話題に近い参照断片

を数件だけ先に読む。

#### ターン中の軽い再想起

各ターンで、話題に近い候補だけを再評価する。
直近で使いすぎた断片は少し抑える。

#### 外した時の救済想起

Chat が「過去判断がありそう」と見たときは、Worker に深掘り調査を委譲する。
必要なら原本保存まで掘りに行く。

### 7.3 想起スコアの考え方

厳密な式は実装側で決めればよいが、概念としては次を使う。

- Relevance: 今の話題にどれだけ近いか
- Importance: その断片がどれだけ重要か
- Recency: 最近使ったか、最近起きたか
- Role Fit: 今の Actor に合うか
- Diversity Control: 直近で同じ断片ばかり使っていないか

### 7.4 Bounding

想起は多ければよいわけではない。

- Chat は関係性記憶と本人格断片を使える
- Worker は作業ロールの範囲に留まる
- Coder は作業断片に限定する

この境界が、人格の混線を防ぐ。

---

## 8. 忘却と昇格

### 8.1 忘却

忘却は削除だけではない。
まずは浮上しにくくする。

対象:

- 一度きりの軽い雑談断片
- 古く、長く参照されていない一時文脈
- すでに上位概念へ統合された重複断片

### 8.2 昇格

昇格は頻度だけでなく、効き方で決める。

昇格候補になりやすいもの:

- 3 回以上効いた断片
- れんとの関係性を安定させた断片
- 再利用価値の高い作業手順
- 長期案件の大きな判断

### 8.3 再固定

想起して使った記憶は、その使われ方自体を新しい手がかりにする。
ただし、毎回そのまま上書きしない。
必要なら Worker が期間整理で統合する。

## 9. 技術選択の方針

### 9.1 まず使うもの

- 構造化 DB
- Event 保存
- 参照索引
- 軽量検索
- 要約と昇格パイプライン

### 9.2 次に使うもの

- RAG
- 関係検索
- GraphRAG 的な参照層
- 想起スコアリング

### 9.3 最後に検討するもの

- ファインチューニング
- 人格重みの焼き込み

ファインチューニングは否定しない。
ただし、Mio の人格深化の第一手段にはしない。
まずは記憶と想起で深める。

---

## 10. 記憶システムと原本保存の関係

記憶システムは、原本保存の上に育つ二次層である。

流れはこうする。

1. 会話、報告、作業は原本保存に残る
2. 日次で Worker が候補抽出する
3. Chat が保存価値を判断する
4. 一部だけが記憶へ昇格する
5. 想起で外した場合は、必要に応じて原本保存へ戻る

この二段構えにより、

- 普段は軽く想起できる
- 外した時も原本が失われない
- 将来別の見方で再抽出できる

を両立する。

---

## 11. この設計の一文要約

Mio は、
その場の会話を整理し、
日次でログを束ね、
期間単位で圧縮と昇格を行い、
長期では関係つき知識、関係性断片、作業手順を別々に育てる。

そして、すべてを覚えるのではなく、
今の文脈に効く断片だけを思い出すことで、人格と作業品質を少しずつ深めていく。
