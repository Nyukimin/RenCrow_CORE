# RenCrow ドキュメント引き継ぎパック

このパックは、RenCrow / PicoClaw のドキュメント整理を他の担当者へ渡すための入口です。
目的は、次の担当者が元の大量 docs に埋もれず、まず **仕様を理解し、使えるようになり、構築判断ができる最小セット** から作業を始められるようにすることです。

## 渡すもの

この ZIP 全体を渡してください。中身は、前回作成した最小ドキュメントセットに、引き継ぎ用の説明・チェックリスト・編集ルール・作業チケット雛形を追加したものです。

まず読ませる順番は次です。

1. `00_引き継ぎREADME.md`
2. `00_引き継ぎ/01_次の担当者への依頼文.md`
3. `00_読む順番.md`
4. `01_理解/01_システム概要.md`
5. `01_理解/02_キャラクター・エージェント仕様.md`
6. `02_正本仕様/03_Runtime_Config.md`
7. `04_構築指標/01_構築ロードマップ.md`
8. `04_構築指標/02_受入基準・テスト観点.md`

## 引き継ぎ時の最重要ルール

1. **元 docs.zip から直接作業を始めない。** まずこの最小セットから理解する。
2. **正本は `02_正本仕様/`。** `01_理解/` と `04_構築指標/` は正本を読みやすくした派生文書として扱う。
3. **除外済み資料を戻すときは、必ず理由を書く。** 判断は `99_整理/除外ファイル一覧.csv` の `restore_when` を見て行う。
4. **旧キャラ体系を復活させない。** 現行は Mio / Shiro / Aka / Ao / Gin / Kin / Kuro / Midori を前提にする。
5. **ドキュメント整理と実装作業を混ぜない。** 実装が必要な論点はチケット化し、仕様変更が必要なら先に正本へ反映する。

## 担当者にお願いする成果物

最初の担当者には、いきなり全面改修させるより、次の3点だけを成果物にすると引き継ぎが安定します。

- `レビュー結果.md`: 読んで分かったこと、分からなかったこと、矛盾候補。
- `更新候補一覧.md`: どのファイルを更新すべきか、なぜ更新するのか。
- `復帰候補一覧.md`: 元 docs.zip から戻す必要がある詳細資料と、その理由。
