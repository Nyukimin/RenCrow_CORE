# 受入基準・テスト観点

## MVP の Definition of Done

MVP は「1キャラで動くが、人数上限なし・長期記憶・多人数協議へ拡張できる土台がある」状態です。

必須条件。

| 領域 | 受入基準 |
|---|---|
| Config | `.picoclaw/config.yaml` を正本として読み、壊れている場合は暗黙fallbackせず起動失敗する |
| Runtime | Viewer / picoclaw は RenCrow_LLM / RenCrow_TTS / RenCrow_STT 経由でのみ backend を呼ぶ |
| Chat | Mio または初期Chat Agent が最終返答を生成する |
| Recall | 毎ターン L0/L1/L2 から Recall Pack が生成される |
| Memory | L1 今日の記憶が SQLite に保存される |
| Staging | 外部取得と記憶候補は staging を通る |
| Validator | official DB への昇格前に schema/hash/date/source/sensitivity を確認する |
| Logging | event_id / source_id / retrieved_at / raw_or_summary が追跡できる |
| Safety | Coder は直接実行せず、Worker が採用・適用・ログ記録する |
| Search | 検索APIは fallback で、毎ターン常用されない |

## Runtime テスト

- Config が存在する場合、正しく LLM/TTS/STT endpoint を読む。
- Config が存在しない場合、Repo 内 snapshot へ暗黙 fallback しない。
- `PICOCLAW_CONFIG` 指定時だけ、その path を明示的に使う。
- Chat 相手切替中は入力を停止し、ready 後に再開する。
- `local_llm` 推論経路と `llm_ops` 起動/切替管理経路が混ざらない。

## Agent 責務テスト

- Mio はユーザー向け最終文を返す。
- Shiro は実行前に実行可否を判断し、失敗時に原因を報告する。
- Coder は patch/proposal/risk/cost hint を返すが、直接適用しない。
- Kuro は高リスク変更時に前提・根本原因・安全面をレビューする。
- Midori は創作/視覚/横方向探索に限定する。

## 未確定論点の受入保留

次の項目は、現時点ではこの受入基準だけで完了判定しない。`00_引き継ぎ/DOCチケット_初回未解決論点.md` と個別チケットで正本・実装・履歴を確認してから扱う。

- 承認フロー存廃。
- `risk=high` 時の停止ルール。
- サニタイズ責務の場所。

## Recall テスト

- ユーザー発話ごとに Recall Pack が空でも生成される。
- L0 は raw 履歴を無制限に戻さず、rolling summary + 直近数往復にする。
- L1 は今日の記憶から関連候補を返す。
- L2 は今月ハイライトから関連候補を返す。
- L3 が disabled の場合も呼び口は壊れず no-op で返る。
- Recall Pack は token budget を超えたら dedupe / rank / trim する。

## Staging / Validator テスト

- raw_text と summary_draft が別フィールドで保存される。
- 同一 URL / 同一 hash の重複が検出される。
- source trust が低い情報は official DB へ自動昇格しない。
- user memory は `observed -> candidate -> confirmed` を通る。
- `忘れて` 操作で該当記憶が検索対象から外れる。

## IdleChat テスト

- ユーザー操作で即時中断する。
- chatBusy / workerBusy 中は自動開始しない。
- TTS 完了イベントで次アクションへ進む。
- ループ検出とリトライが働く。
- IdleChat 中は外部検索を禁止し、内部文脈で会話する。
- Heavy / Wild が稼働中の場合は IdleChat を割り込ませない。

## 構築時に見てはいけない成功条件

以下だけで成功扱いにしないでください。

- LLM が返答しただけ。
- 1回だけ動いた。
- 検索APIで答えが取れた。
- raw履歴を全部プロンプトに入れて文脈が続いたように見えた。
- Coder が patch を出しただけ。

RenCrow の成功条件は、**汚れない記憶、毎ターン想起、安全な実行境界、再現可能なログ** が揃うことです。
