# 構築ロードマップ

## 原則

- 最初は1キャラで実験してよい。
- 設計は人数上限なし。`personas/registry` と委員会制で拡張できる形にする。
- L0/L1/L2 と毎ターン想起は初期から必須。
- L3 長期記憶、Knowledge DB、News DB、VectorDB、多人数協議は段階導入。
- 検索APIは最後に fallback として入れる。

## Phase 0: 設計固定

作るもの。

```text
config/system.yaml
personas/registry.yaml
memory/schema.sql
source_registry.yaml
staging_schema.json
```

決めるもの。

```text
EventId形式
user_id形式
namespace形式
memory type
source trust score
検索API fallback条件
runtime config の選択順
```

完了条件。

- 正本 Config の場所と fallback 禁止が明確。
- Agent責務と安全境界が明確。
- staging schema がある。

## Phase 1: 1キャラ + L0/L1/L2 想起

最小の会話ループ。

```text
User -> L0/L1/L2 recall -> Persona prompt -> Local LLM -> response -> memory update
```

完了条件。

- 直近会話要約が毎ターン更新される。
- 今日の記憶が SQLite に残る。
- 今月ハイライトの器がある。
- Recall Pack が必ず生成される。
- L3 関数は no-op でも呼び口がある。

## Phase 2: ユーザー記憶

先に明示操作だけ実装します。

```text
覚えて
忘れて
ピン留め
これは違う
要約して保存
```

完了条件。

- `user:<uid>` の schema がある。
- 記憶候補は `observed -> candidate -> confirmed` で昇格する。
- sensitive な情報は原則保存しないか、明示確認を挟む。

## Phase 3: staging / validator

外部取得データと記憶候補を正式DBに入れる前に検証します。

validator 項目。

```text
schema妥当性
URL妥当性
hash重複
日付妥当性
source trust
sensitivity
license note
memory type
raw/summary 分離
```

完了条件。

- Cowork/Codex/検索API/手動投入の全経路が staging に落ちる。
- official DB への直接 write を禁止できる。

## Phase 4: Source Registry + RSS取得

最初は 5〜10 source で十分です。

候補。

```text
AI公式blog
技術ニュース
一般ニュース
映画/文化系
GitHub release
PyPI RSS
MediaWiki recentchanges
```

完了条件。

- Source Registry に trust score がある。
- fetch 結果が staging に保存される。
- validator 通過後だけ News/Knowledge DB に昇格する。

## Phase 5: Knowledge DB 最小版

映画・小説・音楽など、各ドメイン 10〜50件で始めます。

完了条件。

- 共通 schema がある。
- title / creators / themes / summary / source がある。
- lexical retrieval ができる。

## Phase 6: Vector layer

最初は FAISS か Qdrant のどちらか一つです。

完了条件。

- L3 / Knowledge / News の代表テキストを embedding 化できる。
- Recall Pack に source_id と event_id が残る。
- vector を万能視せず、lexical retrieval と併用する。

## Phase 7: Search API fallback

最後に入れます。

完了条件。

- 未知語、freshness、公式情報、裏取りのときだけ起動する。
- 結果は Search Cache + staging に保存される。
- 直接 official DB に入らない。

## Phase 8: 3キャラ化

Mio / Shiro / Coder を明確に分けます。

完了条件。

- Mio が最終返答を統合する。
- Shiro が実行主体になる。
- Coder は候補だけ返す。
- Worker 実行ログが残る。

## Phase 9: 多人数協議

10人以上は全員発話ではなく委員会制にします。

```text
Core Committee
Domain Committee
Fact Check Committee
Style Committee
Judge Committee
```

完了条件。

- 設計上の人数は無制限。
- 実行時の発話人数は制限する。
- 重複意見はクラスタリングして統合する。
- 合議結果はMioまたは司会Agentが最終化する。
