# ローカルファースト検索・記憶アーキテクチャ（最小正本）

## 方針

RenCrow は、検索APIを背骨にしません。検索APIは未知語、最新確認、裏取りの外部センサーとして使います。日常の想起、会話継続、知識参照はローカルDBとローカル推論を主役にします。

中核は次の分離です。

```text
Conversation Runtime: LangGraph / turn control / interrupt / multi-agent
Hot Store: SQLite / L1 / control / cache / ledger / event log
Cold Store: DuckDB + Parquet / L2-L3 / Knowledge / News / staging
Vector Layer: FAISS or Qdrant / semantic retrieval
Acquisition: RSS / official API / feeds / webhooks first
Validation: staging -> validator -> promoter -> official DB
```

## 記憶レイヤー

| 層 | 推奨ストア | 主な内容 | 初期MVP |
|---|---|---|---|
| L0 現在会話 | LangGraph checkpoint + rolling summary | 現在スレッド、直近発話、割り込み状態 | 必須 |
| L1 今日 | SQLite WAL | 当日記憶、task note、Search Cache、Citation ledger、Event Log | 必須 |
| L2 今月 | DuckDB + Parquet | 月間ハイライト、会話要約、topic cluster、共起集計 | 必須 |
| L3 長期 | DuckDB + Parquet + vector sidecar | ユーザー嗜好、長期エピソード、canonical memory | 設計だけ先に用意。段階導入 |
| Knowledge DB | DuckDB + Parquet + FTS + vector | 映画・小説・ドラマ・音楽などの正規化知識 | 少量から |
| News DB | DuckDB + Parquet + vector | 記事本文、出典 metadata、日次 digest | RSS少量から |
| Staging DB | JSONL / Parquet | raw fetch、summary_draft、validator result | 必須 |

## namespace 分離

| namespace | 内容 | 共有範囲 |
|---|---|---|
| `conv:<thread_id>` | 今の会話、今日、今月、長期エピソード | 参加キャラで共有 |
| `user:<uid>` | 嗜好、プロフィール、プロジェクト、制約 | 全キャラ参照可 |
| `char:<persona>` | 口調成長、KPI、得意不得意、関係性 | 原則本人、必要時共有 |
| `kb:<domain>` | 映画、音楽、ニュース、技術知識 | 全キャラ参照可 |

## 毎ターン想起フロー

外部検索ではなく、ローカル想起を毎ターン実行します。

```text
User Message
  ↓
Intent / Domain / Entity extraction
  ↓
L0 current thread
  ↓
L1 today memory
  ↓
L2 monthly memory
  ↓
User Profile Memory
  ↓
Character Memory
  ↓
L3 long-term user memory（初期はno-op可）
  ↓
Knowledge DB / News DB / Search Cache
  ↓
External Search only if needed
  ↓
Recall Pack -> Persona Prompt -> LLM
```

## Search API の使いどころ

使う場面は限定します。

- ローカルDBにない未知語の同定。
- 最新情報の確認。
- 公式情報の発見。
- 出典の裏取り。
- 新規 source 候補の発見。

使わない場面は明確にします。

- 初期DB育成の大量収集。
- 毎日のニュース取得の主経路。
- 毎ターンの雑談補足。
- 検索結果の正式DBへの直接投入。

## staging / validator の原則

外部取得、Cowork/Codex、検索API、記憶候補は、正式DBへ直接書かせません。

```text
Source Registry
  ↓
Fetcher / Cowork / Codex / Worker
  ↓
staging JSONL or Parquet
  ↓
validator
  ↓
promoter
  ↓
SQLite / DuckDB / Vector layer
```

raw text と summary draft は分離します。要約はLLM生成物であり、原文とは別の二次表現として管理します。

## ユーザー記憶の昇格ルール

会話から推測した記憶を、いきなり確定しません。

```text
observed -> candidate -> confirmed
```

明示された「覚えて」「忘れて」「ピン留め」「これは違う」は、MVPで先に実装します。推測保存は後回しにします。

## MVPで固定すること

- L0/L1/L2 想起は必須。
- L3 は関数と schema だけ先に置き、初期は no-op でもよい。
- VectorDB は MVP では FAISS か Qdrant のどちらか一つ。両方は入れない。
- Knowledge / News は少量データでよい。最初から完全DB化しない。
- 正式DBに入れる前に staging と validator を必ず通す。
