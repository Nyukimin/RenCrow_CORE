# システム概要

## RenCrow / PicoClaw の目的

RenCrow は、ローカルLLMを主役にしながら、Chat / Worker / Coder / Heavy / Wild の役割を分けて、会話、実行、コード提案、深い分析、創作探索を扱うための会話AI基盤です。PicoClaw は入口、セッション、Viewer、各 RenCrow module server への橋渡しを担います。

## 最小構成の流れ

```text
User / Viewer / LINE / Slack
  ↓
PicoClaw 入口・セッション管理
  ↓
Mio / Chat 窓口
  ↓ 必要時のみ委譲
Shiro / Worker または Aka/Ao/Gin/Kin / Coder
  ↓
Mio が結果を統合して返答
```

## Runtime 経路の原則

Viewer や picoclaw server は、LLM / TTS / STT backend を直接呼びません。必ず RenCrow module server を通します。

```text
LLM: picoclaw -> RenCrow_LLM -> LLM backend
TTS: picoclaw -> RenCrow_TTS -> TTS backend
STT: picoclaw -> RenCrow_STT -> STT backend
```

STT が LLM の音声入力機能を使う場合も、Viewer / picoclaw から見た入口は RenCrow_STT のままにします。

## 責務境界

- Chat/Mio は、ユーザー対話、委譲判断、結果統合、最終返答を担う。
- Worker/Shiro は、実行、ツール呼び出し、コマンド/patch適用、ログ記録を担う。
- Coder/Aka/Ao/Gin/Kin は、実装案、patch案、risk、cost hint を返す。直接実行しない。
- Heavy/Kuro は、深い分析、根本原因調査、最終技術レビュー、安全ゲートを担う。
- Wild/Midori は、創作、視覚解釈、横方向の探索を担う。

## 現行仕様で固定すること

- LINE入口は `CHAT(Mio)` 固定。
- Viewer の通常チャットでは Mio / Shiro / Kuro / Midori を明示選択できるが、Viewer は route/model alias へ変換しない。
- local LLM 推論経路と、モデル起動/切替管理経路は分ける。
- Config の正本はユーザー環境側の `.picoclaw/config.yaml`。
- Repo 内 runtime yaml はスナップショットであり、暗黙 fallback 先にしない。

## 初期実装の考え方

最初から全員を常時会話させるのではなく、**1キャラでも動く設計**から始めます。ただし、registry / route / committee の設計は人数上限なしを前提に残します。実行時にはアクティブ話者数を制御し、必要時だけ複数Agentを召集します。
