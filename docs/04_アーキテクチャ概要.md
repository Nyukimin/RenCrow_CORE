# アーキテクチャ概要

```text
User / Channel / RenCrow_ASSISTANT / RenCrow_PORTAL / Debug Viewer
          |
          v
 MessageOrchestrator
          |
          v
 Mio route decision
    |       |       |
   CHAT    OPS    HEAVY/WILD
    |       |       |
 Chat     Shiro   Kuro/Midori
 runtime  Worker   runtime
            |
      Coder / Advisor / Approval
          |
          v
 RenCrow_LLM Gateway (logical Agent ID)
          |
          v
 physical LLM / Codex command target

Conversation / Recall / Knowledge / Events / Viewer projection
          |
External RenCrow modules and providers
```

TTS経路は`CORE -> RenCrow_TTS Go Gateway -> RenCrow_Irodori_TTS`とする。
COREはGatewayの公開APIだけを参照し、Irodori固有のmodel、reference音声、SEED、
推論parameterを設定または所有しない。

LLM経路は`CMD/PORTAL -> CORE -> RenCrow_LLM Gateway -> physical target`とする。COREはCHAT/OPS/CODE等の意味routeとAgent選択を行い、RenCrow_LLMは受け取った論理Agent IDを実行targetへ解決する。Shiroの通常CHATは`shiro`としてChatWorker targetへ、Kuroの通常CHATとANALYZEは`kuro`としてCodex targetへ送る。COREの`codex.run` Toolは独立したTool contractであり、Kuroの通常LLM経路と兼用しない。Gateway障害時にCOREやCMDが物理targetへ直接fallbackしてはならない。

## UI境界

| surface | 所有repo | 責務 |
| --- | --- | --- |
| `RenCrow_PORTAL` view | `RenCrow_PORTAL` | 外部利用者向けの読み取り専用表示 |
| `RenCrow_PORTAL` live | `RenCrow_PORTAL` | 部屋・操作UIを含まない配信用表示 |
| `RenCrow_PORTAL` lab | `RenCrow_PORTAL` | allowlist内の会話操作 |
| Debug Viewer | `RenCrow_CORE` | logs、Ops、Repair、設定、内部状態の確認 |
| CLI facade | `RenCrow_CMD` | CORE、PORTAL、ASSISTANTの起動、Public APIのCLI操作 |

PORTALとCLIはCOREのclientであり、Persona、Memory、Job、会話状態を所有しません。外部向け画面の変更をDebug ViewerとCORE runtimeの再テストから隔離し、debug/admin APIをPORTALから中継しないことを境界条件とします。

ASSISTANTもCORE Public APIを利用しますが、単なるclientではなく別domainの常駐serviceです。personal／family scope、生活Routine、PUSH、端末deliveryはASSISTANTが所有し、Agent、Memory、Knowledge、複雑なTaskはCOREが所有します。PORTALはASSISTANTの状態を表示・操作できますが、Routineやdelivery状態の正本にはなりません。

## Interaction client profile

COREはPORTAL、CMD、ASSISTANTごとに別の会話systemを作りません。次の意味をPublic API
contractとして共通化し、surface差はclient profileへ閉じ込めます。

| 共通意味論 | COREの責務 |
| --- | --- |
| Chat | 明示recipientと利用者scopeでmessageを処理し、responseを相関する |
| IdleChat | event、status、開始・停止の意味と権限境界を定義する |
| recipient選択 | client-localな表示選択と実messageの`to`を分ける |
| event／session | 再接続、重複、順序、request／response／task相関を扱う |
| STT／TTS | input ownership、合成、audio取得、端末再生を別の成功条件にする |
| Task | 通常会話と複雑な継続作業を分け、状態とprovenanceを返す |
| error | unavailable、degraded、denied、expired、failedを成功へ丸めない |

```text
PORTAL    = CORE Interaction Client + Web Renderer
CMD       = CORE Interaction Client + Terminal Renderer + Operations
ASSISTANT = CORE Interaction Client + Proactive Trigger + Device Delivery
            + Assistant-owned personal / family / routine state
```

これは同一UI、同一transport、共有database、または共通SDKの先行作成を意味しません。
実caller間でDTO、再接続、error処理の重複が確認された時だけ、安定したcontractを
`modules/*`へ昇格します。公開可否はclient profile、認証scope、PORTAL mode、Device
capabilityで制御します。

## コード上の境界

| path | 責務 |
| --- | --- |
| `modules/*` | 外部利用可能な DTO、event、純粋 policy、安定 contract |
| `internal/features/*` | feature ownership、route registrar、dependency handoff |
| `internal/domain/*` | domain value、state、validation |
| `internal/application/*` | use case、orchestration |
| `internal/adapter/*` | Viewer、channel、provider、compatibility adapter |
| `internal/infrastructure/*` | persistence、transport、外部技術実装 |
| `cmd/rencrow` | config load、依存組立、server 起動を行う composition root |

## 依存方針

- domain と public contract は特定 UI、DB、外部 provider に依存しない。
- adapter と infrastructure は contract を実装し、composition root が接続する。
- 新しい public package は複数 adapter で安定した契約だけを昇格する。
- 移行中の legacy-body は、挙動を維持しながら小さい route/job 単位で feature ownership を移す。
- route、provider、store、CLI、background job の二重登録・二重起動を防ぐ。

## 状態と観測

操作 API は REST、継続状態は SSE と MonitorStore、永続履歴は各 store が担います。Job ID、from/to、route、provenance、approval、結果を追跡可能にし、外部 runtime の health と CORE 自身の health を分けて表示します。

## 記憶ストレージ

COREの常時稼働する記憶ストレージはGoから利用できる次の構成を正本とします。

| 用途 | 技術 | 主な内容 |
| --- | --- | --- |
| 短期会話 | Redis | active thread、直近会話 |
| L1 hot store | SQLite | 会話event、UserMemory、Knowledge候補、Recall trace |
| L2 archive | SQLite | thread summary、中期会話、L1 archive |
| 意味検索 | Qdrant | 会話要約・Knowledgeのvector検索 |
| cold export | Parquet | Goライブラリによるバックアップ・オフライン分析用出力 |

DuckDBはCORE runtimeの必須依存としません。L2 archiveは純Go SQLite driverを使い、Parquet出力もGo実装で行います。
