## 3. Memory lifecycleの現行正本化範囲

正本へ吸収するのは、現行code・testで確認できる次の範囲に限定する。

- `RecallPack`はDB全体ではなく、選別済みcontextをLLMへ渡すprompt入力契約である。
- role filterとtoken budgetで採用・不採用を分け、不採用理由をtraceへ残す。
- UserMemoryは`observed`、`candidate`、`confirmed`、`pinned`を区別する。
- promptへ注入できるのはactiveかつsupersede / decayされていない`confirmed`または`pinned`で、sensitiveを除外する。
- 外部入力はstagingとvalidationを経てからKnowledge / Memoryへpromotionする。
- Recall Traceはsession、response、role、採用項目、不採用理由を追跡する。
- Persona、Conversation、User、Knowledge、Operation、Runtime logを同一の正式記憶として混ぜない。

旧仕様のL0〜L4全自動運用、全role最適化、全store統合、完全なdecay / forgettingは、code・testで確認できる範囲を超えて正本化しない。

## 4. TTS / Viewer同期の現行正本化範囲

- 表示本文の真実は表示event / 表示stateであり、TTS chunkだけから本文を再構成しない。
- `tts.audio_chunk`は`session_id`、`message_id`または`turn_index`、`chunk_index`で発話と順序を対応付ける。
- `speech_text`は読み上げ、`display_text`は表示補助であり、用途を入れ替えない。
- pending / queueはbrowserまたはruntimeの一時状態で、表示済み・再生済みの監査事実ではない。
- backendの完了判定はplayback ACKまたはbounded timeoutを使う。deprecated fallback ACKは成功に扱わない。
- TTS engine、voice asset、synthesis本体は`RenCrow_TTS`が所有し、COREはprovider接続、event、Viewer再生同期を所有する。

## 5. Viewerの現行正本化範囲

- route登録は`cmd/rencrow/routes.go`からfeature registrarへ委譲し、handlerとURLを別々に変更しない。
- `/viewer/send`はJSONまたはmultipartを受け、text / attachmentを通常のmessage経路へ渡す。
- `/viewer/events`はSSEを提供し、`Last-Event-ID`以降の再送を扱う。音声chunk等のtransient eventはhistory replayしない。
- MonitorStoreは表示用projectionであり、監査log、evidence、永続storeの代替ではない。
- Viewerは表示本文、SSE、history、event log、audio / lipsync trigger、debug stateを混同しない。
- route / tab / APIの完全一覧は変化が速いため、正本本文へ固定せずregistrar、contract test、browser E2Eから確認する。

## 6. ルール辞書監査

### 現行契約

| 項目 | 現行実装 |
| --- | --- |
| producer | `internal/domain/agent.MioAgent.DecideAction` |
| dictionary | `internal/infrastructure/routing.RuleDictionary` |
| input | `internal/domain/task.Task.UserMessage()` |
| order | 明示command → RuleDictionary → classifier → CHAT safe fallback |

| output | `routing.Route`、confidence、matched |

旧資料にある「priority降順」は現行`rule`にpriority fieldがなく実装されていない。現行契約は配列の定義順であり、正本も現行codeに合わせる。将来priorityを導入する場合は、先にdomain / infrastructure testを追加してから仕様変更する。

監査時に、`test/e2e/routing_test.go`のgeneric `CODE`からCoder2へ暗黙fallbackする期待と、自然言語実装依頼を`CODE`とする期待が現行codeからずれていることを検出した。前者はCoder1不在時に明示routeを要求する契約、後者はRuleDictionaryの`CODE2`契約へ同期した。外部coder応答に依存せずroute contractを検証し、`-tags=e2e`でrouting E2Eを再実行した。

## 7. WorkerInput監査

repo全体を検索した結果、`WorkerInput`という現行型、producer、consumer、validator、contract testは存在しない。旧分割仕様の名称だけを正本契約として復活させない。

現在のWorker入力境界は次に分かれている。

| 境界 | 現行型 |
| --- | --- |
| user / channelからのtask | `internal/domain/task.Task` |
| Viewer入力 | `internal/adapter/viewer.SendRequest` → `task.Task` |
| Chat module公開契約 | `modules/chat.Input` / `modules/chat.Output` |
| Worker module結果 | `modules/worker.Result` |
| orchestration | `internal/application/orchestrator.ProcessMessageRequest`等のuse case固有request |

したがって`WorkerInput`は`archive`判断とし、`02_実装仕様.md`のplaceholderを削除する。単一の統合型が必要になった時は、producer / consumerが2箇所以上存在すること、互換性、validation、testを先に示して新規仕様として提案する。
