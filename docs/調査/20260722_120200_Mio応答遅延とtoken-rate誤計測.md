# Mio応答遅延とtoken rate誤計測の調査

調査日: 2026-07-22 JST

## 概要

`rencrowctl chat --to mio おはようございます`の低速表示を、CMD、CORE、RenCrow_LLM Gateway、物理Mio targetの順に分解した。物理生成は40 token/secを超えていたが、COREの逐次転送、CMDの誤計測、会話終了時の旧Ollama依存、CMDが消費しないTTSの同一GPU並走が利用者観測を悪化させていた。

## 仮説と反証

1. 物理Mio target自体が遅い
   - 反証: 物理targetのllama.cpp `timings.predicted_per_second`は49.92、Gateway経由は48.50だった。
2. RenCrow_LLM Gatewayがstreamをまとめている
   - 反証: 制御したSSEではGatewayの`[DONE]`到着は1.141秒、EOFは1.143秒で、targetとの差は小さかった。
3. COREとCMDが逐次streamを失っている
   - 確認: COREのOpenAI互換providerは`[DONE]`まで本文callbackを保留し、CMDは`agent.thinking`を表示していなかった。
4. LLM生成後に同期処理が詰まっている
   - 確認: job `20260722-025054-913abbf9`は02:50:59に本文生成を終えた後、到達不能な旧Ollamaの要約timeoutを待ち、`agent.response`が02:51:29になった。さらに旧構成ではembedding timeoutも同じ終了経路に存在した。
5. 表示されたtoken/secが実token throughputである
   - 反証: CMDは日本語の本文delta数をtoken数として数えていた。Gatewayの実SSE最終chunkは`completion_tokens=7`、`predicted_per_second=51.1808`を保持していた。
6. 入力tokenまたはcontext上限を減らさなければ40 token/secを超えない
   - 反証: Persona全文と8192 context上限を維持し、物理`/slots`で`n_prompt_tokens=5624`を確認したE2Eが46.1 token/secだった。prompt短縮案は採用せず撤回した。
7. CLI text chatにTTSが必要である
   - 反証: `operation_source="RenCrow_CMD"`のrequestでも、旧実装は音声を再生しないCMDのために`.205`のTTSをLLM stream中から起動していた。CMD由来だけTTSを止めると、同じ63 completion tokenで45.8〜46.1 token/secになった。
8. thread要約は応答完了前に同期実行すべきである
   - 反証: 12メッセージ境界でWorker要約が`Store()`を塞ぎ、本文生成済みのCLIを待たせた。新threadとraw messageを先に同期保存すれば、旧threadのL2要約はbackgroundで実行できる。
9. 通常文のLLM classifierは小さい処理なので待ち時間へ影響しない
   - 反証: Gemma 4は分類promptでも内部推論を100 token生成し、最終カテゴリを出す前に上限へ到達していた。実backendでは、旧requestが約2.47秒かけて空の`content`を返した。`chat_template_kwargs.enable_thinking=false`と8 token上限にすると、`CHAT`を2 tokenで返し、cacheなし約0.47秒、cacheあり約0.16秒だった。

## 根本原因

- CORE OpenAI互換stream readerが通常本文まで終端までbufferしていた。
- CMDが逐次`agent.thinking`を無視し、総所要時間と文字ベース推定でtoken/secを出していた。
- 明白な挨拶もLLM classifierへ送り、不要な推論を1回追加していた。
- 到達不能な旧Ollama embeddingを会話終了の同期経路で呼んでいた。
- RenCrow_LLM Gatewayが有効でも、会話要約providerの選択条件が`local_llm.enabled`に依存し、旧Ollamaへ逸脱していた。
- Viewer共通入口が`operation_source`をorchestratorへ渡さず、音声を使わないCMDでもTTSを開始してMioと同一hostのGPU処理を重ねていた。
- `RealConversationManager.Store()`が12メッセージ境界のLLM要約を同期実行していた。
- profile候補でない単純挨拶でもProfileExtractorを実行できる構造だった。
- Mioと同じLLM slotを使うclassifierが、分類に不要な100 tokenの内部推論を生成してから空回答になっていた。

## 修正

- 通常本文deltaを受信時にcallbackへ渡し、思考らしいprefixだけを安全確認のため短くbufferする。
- CMDはdeltaを逐次表示し、backend throughput eventがある場合だけ`token/sec`として表示する。値がなければ`delta/sec`または推定値と明示する。
- 単純な挨拶をrule dictionaryでCHATへ確定し、複合操作指示は既存action ruleを優先する。
- 停止中のembedding modelをlive configから外し、Redis/L1/L2の非vector記憶は維持する。
- primary Workerが存在する場合、会話要約とProfileExtractorをRenCrow_LLM GatewayのWorkerへ固定する。
- `X-RenCrow-Client: RenCrow_CMD`をorchestratorまで伝搬し、CMD text chatだけTTS sessionを作らない。PORTAL／Debug ViewerのTTSは維持する。
- thread rolloverは新threadと最新raw messageを同期保存し、旧threadのLLM要約だけを45秒上限のbackground処理へ移す。timeout時は新しい5秒contextで簡易要約をL2へ保存し、期限切れcontextの再利用によるarchive失敗を防ぐ。
- ProfileExtractorは明示的な永続profile候補を含む発話だけで起動する。
- classifierは思考生成を無効にし、カテゴリ名に十分な8 tokenで上限を固定する。
- 作業意図を含まない高確度の質問・直接回答要求はrule dictionaryでCHATへ確定し、分類器自体を呼ばない。稼働確認、ログ、エラー、実装、運用、調査、設計の可能性が残る入力はclassifierへ残す。
- `Chat`は通常text CHAT、input_audio CHAT等の全経路で`think=false`と`chat_template_kwargs.enable_thinking=false`をprovider境界で強制する。GPT-OSS-120Bの`ChatWorker`は`think=low`と`reasoning_effort=low`へ固定し、IdleChat等の直接provider呼び出しも同じpolicyを通す。`Worker`はwrapperの対象外とし、thinkingを禁止しない。
- 入力prompt、会話context、`n_ctx=8192`は性能達成のために削減しない。

## 検証結果

- 修正前の利用者経路: 34.59秒、stream表示41.2 delta/sec相当。生成後の旧Ollama要約待ちを含む。
- 要約経路修正後: 6.20秒。旧Ollama要約timeoutなし。
- backend throughput伝搬後・TTS並走あり: 5,609 prompt token、39.2 token/sec。
- Gateway実SSE: `completion_tokens=7`、`predicted_per_second=51.1808`。
- 最終live E2E（Persona全文、context上限8192、TTSなしのCMD text chat）:
  - 5,283 prompt token: 46.1 token/sec、5.62秒
  - 5,413 prompt token: 45.8 token/sec、2.18秒（llama.cpp prefix cache hit）
  - 5,543 prompt token: 45.8 token/sec、5.52秒
  - 5,624 prompt token: 46.1 token/sec、5.89秒
  - 5,694 prompt token: 44.9 token/sec、2.41秒（prefix cache hit）
  - 最終再起動後、5,261 prompt token: 45.6 token/sec、5.68秒
  - 最小44.9、中央値45.8 token/sec。全6回で40超。
- classifier単体の実backend比較:
  - 修正前: 100 completion tokenがすべて`reasoning_content`に使われ、`content` は空、約2.47秒。
  - 思考無効・8 token上限: `CHAT`を2 completion tokenで返し、約0.47秒。同一prefix cache hit時は約0.16秒。
- 同じ速度計測文のlive E2E:
  - classifier修正前: route decision 2,854〜4,378ms、response complete 5,877〜7,988ms。
  - classifier修正後: `classifier:matched:CHAT`、route decision 858.656ms、first token 1,912.695ms、response complete 2,104.968ms、50.23 token/sec。
  - 高確度CHAT短絡後: `rule_dictionary:matched:CHAT`、route decision 0.583ms、first token 834.707ms、response complete 1,013.196ms、49.78 token/sec。
  - 最終slotは`n_ctx=8192`、`n_prompt_tokens=5719`。入力とcontextの削減は行っていない。
- 各CMD jobで`operation_source="RenCrow_CMD"`を確認し、同じjobの`TTS chunk_ready`は0件。
- TDD Red:
  - 通常本文が`[DONE]`まで届かない。
  - 明白な挨拶がrule dictionaryにmatchしない。
  - Gateway有効時も要約providerが`ollama_summary`になる。
  - streaming最終chunkのusage/timingsを取得しない。
  - Mioとorchestratorがbackend metricsを伝搬しない。
  - CMDがbackend rateよりdelta数を優先する。
  - CMD由来でもTTS sessionが作られる。
  - 12メッセージ境界で`Store()`がLLM要約完了を待つ。
  - background要約timeout後に期限切れcontextを使い、簡易要約のL2保存も失敗する。
  - classifierが100 token上限で思考生成を無効にしない。
  - 直接回答だけで完結する高確度の会話もclassifierに送る。
  - CHATのthinking設定が未指定な場合にfalseを送らない。直接trueを渡せる。input_audio経路も未指定時にfalseを送らない。
- Green後にCORE/CMDの全Go test、vet、build、thread rolloverの`go test -race`、live E2Eを実行し、CORE serviceを再ビルド・再起動した。

## 教訓

token throughput、TTFT、総応答時間、会話終了時maintenanceは別々に測る。CLI側のchunk数や文字数をbackend token数として表示せず、providerが返すusage/timingをrequest ID付きで運ぶ。optional memory処理の停止endpointをproduction会話の同期経路へ残さない。性能目標をprompt/context削減で達成したことにせず、同一入力条件で不要な並走処理を外して検証する。
