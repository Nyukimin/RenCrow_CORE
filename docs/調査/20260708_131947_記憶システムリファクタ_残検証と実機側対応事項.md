## 実機側で対応すべき事項

### A. 記憶有効でのフル起動 end-to-end 確認

#### 事実

macOS ローカルの Docker daemon が I/O エラーで破損し、Qdrant コンテナのストレージが壊れている。エラーは `os error 5` であり、`conversation.enabled:true` のフル起動は最後まで通っていない。

DuckDB darwin 対応により、以前の `duckdb archive is not supported on this platform` fatal は解消済みである。`.duckdb` 生成も確認済みである。

現在残っている停止点は Qdrant 側の I/O error のみである。

#### 仮説

コードは健全な Qdrant 相手に統合テストで PASS 済みであるため、健全なインフラがあれば `conversation.enabled:true` のフル起動は通る見込みである。

#### 未確認 / 要実施

Linux 実機で以下を実施する。

1. `git pull` で `feature/RenCrow_Start` の最新を取得する。
2. Redis + Qdrant を起動する。

```bash
docker compose -f docker-compose.infra.yml up -d
```

3. `conversation.enabled:true` で `picoclaw` を起動する。
4. Viewer の memory layers API が 200 + JSON を返すことを確認する。

```bash
curl -i http://127.0.0.1:18790/viewer/memory/layers
```

期待:

- HTTP status が `200` である。
- response body が JSON である。
- Qdrant I/O error で起動停止しない。
