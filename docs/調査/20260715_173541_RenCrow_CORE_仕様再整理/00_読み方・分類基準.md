# 読み方・分類基準

## 概要

本一式は、2026-07-15 時点の `RenCrow_CORE` を、正本仕様、派生仕様、現行コード、テスト、Git 履歴から再判定した監査スナップショットである。

既存の `docs/02_正本仕様/` には同時進行の再編差分があるため、本一式は正本を直接上書きせず、正本更新候補として `docs/調査/` に分離した。現在仕様を確認するときは、単独の古い章やチェックリストではなく、次の4文書を組み合わせて読む。

1. [最新の全体仕様](01_最新の全体仕様.md)
2. [実装済み仕様](02_実装済み仕様.md)
3. [未実装仕様](03_未実装仕様.md)
4. [古い仕様・不採用・要再確認](04_古い仕様・不採用・要再確認.md)

所見の反証確認は [検証ログ](20260715_173541_検証_RenCrow_CORE仕様分類.md) に残した。

## 判定の優先順位

矛盾がある場合は、次の順で現在状態を判定した。

1. 現行コードの production wiring と、そこを通るテスト
2. `docs/02_正本仕様/00_正本仕様Tree.md` が定める正本階層
3. 最新 To-Be の要求と acceptance criteria
4. `12_CORE_機能台帳.md` の状態
5. `docs/04_構築指標/` の派生実装資料
6. 日付付き実装スナップショット、旧 `pkg/*` path、アーカイブ記述

コードが存在しても起動時に接続されないものは「実装済み」としない。逆に、`internal/features/*` への移管前でも legacy-body が production wiring されている機能は、「機能は実装済み、所有権移行は未完了」と判定する。

## 状態分類

| 状態 | 意味 | 未実装一覧に含めるか |
| --- | --- | --- |
| `current_contract` | 現在採用されている全体要求・責務境界 | 要求そのものなので実装状態は別判定 |
| `implemented_code_test` | production code に接続され、対象テストが通る | 含めない |
| `implemented_opt_in` | 実装・接続済みだが config や人間操作で有効化する | 含めない |
| `implemented_legacy_body` | 挙動は稼働するが feature ownership の移管前 | 挙動は含めず、移管だけ含める |
| `unimplemented_confirmed` | 最新の採用要件があり、production 経路がない | 含める |
| `stale_snapshot` | 現行実装より古い状態記述、旧 path、履歴 | 含めない |
| `readoption_required` | 現行責務モデルと競合し、実装前に再採用判断が必要 | 未実装ではなく要再確認 |
| `external_boundary` | 別 RenCrow module が実装本体を所有する | CORE の未実装に含めない |
| `runtime_unverified` | コードはあるが今回ライブ環境を再検証していない | 未実装と断定しない |

## 今回の検証範囲

- 対象 repo: `/home/nyukimi/RenCrow/RenCrow_CORE`
- 正本・派生仕様、機能台帳、現行 Go/Viewer 実装、対象テストを確認した。
- Advisor、AgentProfile、Knowledge Relation、Economic Objective、Heartbeat、Viewer、client の対象 Go テストは通過した。
- 外部の RenCrow_LLM / TTS / STT / Vision / GAMES のライブ起動、実ブラウザ表示、音声再生は今回再検証していない。
- Git の commit、push、service restart は行っていない。

## 重要な読み違い防止

- `facade_only` や空の feature registrar は「機能本体がない」という意味ではない。legacy-body に本体が残る非破壊移行状態である。
- 派生資料の「後続」「未接続」は、資料作成後に実装された場合がある。2026-07-14 の派生資料より、2026-07-15 の実装が新しい。
- full backfill、複数 adapter 安定前の `modules/advisor` 新設など、明示的な非目標は未実装 backlog に数えない。
- 外部 runtime の利用不能やライブ未確認は、CORE の機能未実装とは別である。
- 品質上の不足と、採用仕様の未実装は分けて扱う。
