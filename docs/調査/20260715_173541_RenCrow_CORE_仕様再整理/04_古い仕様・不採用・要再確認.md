# 古い仕様・不採用・要再確認

## 概要

古い文書に「未実装」と書かれていても、現在の backlog とは限らない。本章は、現行実装より古い状態記述、旧 package path、現行責務モデルと競合する仕様、単なる計画履歴を分離する。

## 1. 現行実装より古い派生資料

次の派生資料は 2026-07-14 の状態を基にし、2026-07-15 の実装より古い。

```text
docs/04_構築指標/03_Advisor_AgentProfile接続実装仕様.md
docs/04_構築指標/04_KnowledgeRelation接続実装仕様.md
docs/04_構築指標/05_EconomicObjective接続実装仕様.md
docs/04_構築指標/06_ToBe_Ops表示実装仕様.md
```

古くなった主な状態記述:

- Advisor persistence / policy / Viewer 接続が後続
- Knowledge relation importer / batch / 1-2 hop expansion が後続
- Economic scheduler / Viewer 接続が後続
- To-Be Ops 表示が後続

これらは現行コードで実装済みであり、未実装一覧から除外する。派生資料は目的・安全条件の参照には使えるが、現在状態は更新が必要である。

## 2. 旧 `pkg/*` architecture snapshot

`docs/02_正本仕様/02_実装仕様/01_エージェント・責務境界.md` は冒頭で、Worker routing と `pkg/*` path を旧 architecture と明記する。一方、同じ active 文書の後半には次のような旧 path が残る。

```text
pkg/agent/loop.go
pkg/agent/context.go
pkg/agent/router.go
pkg/session/manager.go
pkg/providers/http_provider.go
```

`docs/02_正本仕様/02_実装仕様/06_参照・実装状況.md` も、15章以降を日付付き snapshot と明記している。そこにある未チェック項目や「次フェーズ」は現行 backlog へ自動転記しない。

現行の route owner は Mio、Worker は決定済み route の実行 owner である。現行コード、routing tests、機能台帳を優先する。

## 3. Heartbeat・合議制は再採用判断が必要

`docs/02_正本仕様/02_実装仕様/05_Heartbeat・合議制.md` は active/canonical child を名乗るが、内容は2026-03のモデルを保持している。

- `order3` を含む agent heartbeat を Worker が集約する。
- `/deliberate` で Order1/2/3 を並列実行する。
- Worker が route 集約し、Chat が最終決定する旧責務を前提にする。

現行 Heartbeat は scheduled work、status、Economic draft discovery を中心に実装され、旧 agent heartbeat/deliberation protocol の production 実装は確認できない。さらに現行の Mio/Shiro/Coder/Advisor 境界とそのままでは整合しない。

したがってこれは「採用済みの単純な未実装」ではなく `readoption_required` とする。必要なら先に次を決める。

1. agent liveness と scheduled-work heartbeat を同名機能にするか分離するか。
2. 合議の request owner、proposal owner、decision owner、side-effect owner。
3. Advisor と Coder proposal の違い。
4. cost、timeout、approval、trace、fallback。
5. `/deliberate` を現行 API として再採用するか。

## 4. To-Be phase 記述は計画履歴を含む

`11_RenCrow_ToBe_統合実装仕様.md` の Phase 2-7 や「後続実装仕様」は、実装順を示す計画文でもある。現在は Advisor/AgentProfile、Knowledge Relation、Economic MVP、Viewer、Opportunity discovery の大部分が接続済みである。

残件は phase 全体ではなく、次の小さい acceptance gap として扱う。

- Advisor adoption/outcome writer と日次 score 集計
- Economic chain の Artifact 接続
- feature ownership migration

## 6. 不採用または非目標として扱うもの

- 外部検索、Advisor 出力、Revenue 候補の無審査 memory promotion
- approval なしの公開、送信、請求、契約、価格決定
- 通常起動時の全 Knowledge data 即時 backfill
- 外部 module の実装本体を CORE に戻すこと
- 安定前の contract を理由なく `modules/*` へ公開すること
- 空 registrar を埋めるためだけの大規模一括移動
- unavailable runtime の暗黙成功や Mio への無言 fallback

## 正本へ反映するときの更新候補

1. `12_CORE_機能台帳.md` の「後続」注記を現在コードへ合わせる。
2. `docs/04_構築指標/03`〜`06` に implementation status と完了日を追記する。
3. `02_実装仕様/01` と `06` から現行節に混じる旧 `pkg/*` path を履歴 appendix へ移す。
4. `05_Heartbeat・合議制.md` を `historical` または `re-adoption-required` に変更する。
5. Go version と最新 codebase-map link を更新する。
6. [未実装仕様](03_未実装仕様.md) の3 gap だけを現在 backlog として追跡する。
