# RenCrow_CORE Ver0.80 現状モジュール検証

作成日時: 2026-07-01 17:09:23 JST
対象: `RenCrow_CORE` / branch `feature/RenCrow_Start`
目的: Chat / IdleChat / Viewer / Agent などを機能単位で閉じ込める Ver0.80 化の前に、現行実装の境界・仕様ズレ・テスト状態を検証する。

## 結論

現行コードは Go の全体テストと `go vet` を通過しており、コンパイル可能性と既存テスト上の健全性は保たれている。

ただし、Ver0.80 で「機能ごとのフォルダに実体を閉じ込める」には、まだ未整合が大きい。特に以下は先に対処すべきである。

1. `AGENTS.md` / `CLAUDE.md` の正本参照先が、整理後の `docs/02_正本仕様/` ではなく旧 `docs/01_正本仕様/` / `docs/10_新仕様/` を指している。
2. Viewer 通常チャット仕様は `to=mio|shiro|kuro|midori` だけを送る前提だが、実装とテストはまだ `model_alias` / `route_prefix` による旧 route 変換を正としている。
3. `cmd/rencrow` が composition root として自然な配線を超えて、Viewer route、IdleChat 起動、モジュール連携、各 feature handler を広く抱えている。
4. `modules/` は契約化が始まっているが、`browseractor` / `voicechat` / `webgather` は README と依存方向テストの対象から漏れている。
5. IdleChat は `internal/application/idlechat` と一部 `modules/chat` に分かれており、機能フォルダとして閉じている状態ではない。

Ver0.80 の第一段階は、大規模移動ではなく「仕様入口の修正」「境界テストの対象拡張」「Viewer 旧 alias 経路の扱い決定」「cmd 配線の feature registrar 化」から入るのが安全である。

### P0: Viewer 通常チャットの実装が正本仕様と逆方向

正本仕様は Viewer が `to` だけを送ると定義している。しかし `internal/adapter/viewer/handler_send.go` は以下を持つ。

- request field: `model_alias`, `base_url`, `model`, `route_prefix`
- hardcoded alias: `Worker -> http://127.0.0.1:8082 /ops`
- hardcoded alias: `Coder -> http://127.0.0.1:8082 /code2`
- hardcoded alias: `Heavy -> http://127.0.0.1:8083 /analyze`
- hardcoded alias: `Wild -> http://127.0.0.1:8084 /wild`
- explicit route がない場合、`route_prefix + " " + message` に変換する。

該当箇所:

- `internal/adapter/viewer/handler_send.go:33-39`
- `internal/adapter/viewer/handler_send.go:48-73`
- `internal/adapter/viewer/handler_send.go:122-128`
- `internal/adapter/viewer/handler_send.go:162-178`

さらに既存テストはこの旧挙動を期待している。

- `TestHandleSendAppliesViewerLLMAlias` は `model_alias=Worker` を `/ops ...` に変換することを期待。
- `TestHandleSendUsesRuntimeAliasFields` は `model_alias=Heavy` と `route_prefix=/heavy` を受け入れることを期待。

該当箇所:

- `internal/adapter/viewer/handler_send_test.go:18-61`
- `internal/adapter/viewer/handler_send_test.go:91-133`

影響:

- `go test ./...` が通っても、Viewer 通常チャット仕様と一致している証拠にはならない。
- Viewer が backend / route 解決を抱えており、Viewer module を疎にできない。
- `Runtime_Config` の「Viewer/rencrow は backend を直接扱わない」方針と衝突する。

推奨:

1. 旧 `model_alias` / `route_prefix` 経路を「legacy explicit route support」として隔離するか、通常チャットから除外する。
2. `SendRequest` に `To string` を追加し、Viewer 通常チャットは `to` だけを渡すテストを先に作る。
3. `/ops` / `/analyze` / `/wild` はユーザー本文の明示 command としてだけ扱う。
4. 旧 alias テストは削除ではなく、legacy API として残すか廃止判断を明記してから更新する。

### P1: cmd/rencrow が Ver0.80 の機能境界を超えて肥大化している

`cmd/rencrow/runtime_dependencies.go` の `Dependencies` は 178 field を持つ。composition root として配線を持つこと自体は問題ではないが、Viewer、Memory、Sandbox、Skill、Revenue、Persona、BrowserTrace、Complexity、SuperAgent、AIWorkflow、Scheduler、KnowledgeMemory などの handler / store / controller が 1 struct に集約されている。

該当箇所:

- `cmd/rencrow/runtime_dependencies.go:60-220`

`cmd/rencrow/routes.go` も Viewer base route、LLM ops、STT/audio、dynamic route を横断登録している。

該当箇所:

- `cmd/rencrow/routes.go:28-95`
- `cmd/rencrow/routes.go:97-138`
- `cmd/rencrow/routes.go:140-260`

影響:

- Chat / IdleChat / Viewer / Agent の機能単位に切り出しても、起動配線が `cmd/rencrow` に残り続ける。
- feature を追加するたびに `Dependencies` と route 登録が膨らむ。
- module ではなく巨大 app singleton を中心に実装が伸びる。

推奨:

1. `cmd/rencrow` は main と最小 composition root だけに寄せる。
2. 各 feature に `RegisterRoutes(mux, deps)` または `BuildRuntime(cfg, deps)` の小さい registrar を持たせる。
3. `Dependencies` を feature ごとの dependency bundle に分割する。
4. 初回は移動ではなく wrapper / registrar 抽出に留め、behavior diff を出さない。

### P1: IdleChat が機能フォルダとして閉じていない

`cmd/rencrow/runtime_idlechat.go` は IdleChat runtime の構築で以下を直接行う。

- `idlechat.NewIdleChatOrchestrator`
- speaker provider map の構築
- topic generation config の変換
- forecast provider の選定
- persona recorder の接続
- topic store path の構築
- TTS prefetch と Viewer event 変換
- `idleChatOrch.Start()`

該当箇所:

- `cmd/rencrow/runtime_idlechat.go:20-139`

また IdleChat runtime は `internal/application/idlechat` にあり、forecast policy などの一部だけ `modules/chat` を使う。

影響:

- IdleChat が Chat module の一部なのか、独立 feature なのかが曖昧。
- Viewer event / TTS / Persona / LLM provider 配線が runtime 起動関数に集まり、IdleChat の交換・停止・テスト単位が大きい。

推奨:

1. `features/idlechat` または `modules/idlechat` 相当の facade を設ける。
2. `cmd/rencrow` からは `idlechatfeature.BuildRuntime(cfg, ports)` のような 1 呼び出しにする。
3. provider 選定、topic store、TTS event bridge、Viewer event 変換を IdleChat feature 内の port として整理する。
4. `internal/application/idlechat` の中核ロジックはすぐ移動せず、facade で境界を固定してから段階移動する。

### P1: module boundary test の対象が実体と一致していない

`modules/dependency_rules_test.go` は `core` / `llm` / `tts` / `stt` / `worker` / `chat` だけを対象にしている。

実体として存在する `browseractor` / `voicechat` / `webgather` は対象外で、README もない。

影響:

- 新 module が増えても依存方向ルールが自動で効かない。
- Ver0.80 で module を疎にしたつもりでも、後から逆依存が混入する。

推奨:

1. 3 module の README を追加し、責務・禁止 import・外部 port を明記する。
2. `allowedModuleImports` に追加する。
3. `TestModulePackagesAreDocumented` が全 `modules/*` を自動列挙するように変える。
4. `modules/DEPENDENCY_RULES.md` の layout と forbidden dependencies を実体に合わせて更新する。

### P2: Viewer adapter が feature 集積場になっている

`internal/adapter/viewer` は 122 Go files あり、Chat UI / TTS / STT / Memory / Domain Graph / Sandbox / Skill / Revenue / Persona / BrowserTrace / Complexity / SuperAgent / AIWorkflow など、多数の feature handler を同一 package に持つ。

影響:

- Viewer shell と feature-specific viewer API が分離されていない。
- 機能単位で閉じ込める場合、Viewer 側だけが横断 package のまま残る。

推奨:

1. Viewer core は page / asset / SSE / shell state に絞る。
2. feature-specific handler は `features/<name>/viewer` または `internal/adapter/viewer/<name>` 相当に分ける。
3. 最初は package 分割より route registrar 分割を優先する。

### P2: module/chat はまだ独立 Chat 実装ではなく legacy orchestrator adapter

`cmd/rencrow/module_chat_route.go` は `/viewer/modules/chat/route` を提供するが、これは generic route decision API であり、Viewer 通常チャットの `to=mio|shiro|kuro|midori` とは別の contract である。

`internal/adapter/modulebridge/chat.go` の `Respond` は `chat.RuntimePorts` を引数として受けるが使用せず、legacy orchestrator の `ProcessMessage` に委譲する。

該当箇所:

- `cmd/rencrow/module_chat_route.go:12-32`
- `internal/adapter/modulebridge/chat.go:65-75`
- `internal/adapter/modulebridge/chat.go:75-95`

影響:

- `modules/chat` があることは、Chat 機能が module 内に閉じたことを意味しない。
- Ver0.80 で Chat を独立 feature にする場合、`to` 解決、history、response event、LLM recipient selection を Chat module 側へ寄せる必要がある。

推奨:

1. `modules/chat` に Viewer recipient contract を追加する。
2. `modulebridge` は legacy adapter として明記する。
3. Chat の新 contract を先にテストし、orchestrator から段階的に呼び出す。

## 推奨移行順

1. **仕様入口修正**
   - `AGENTS.md` / `CLAUDE.md` を現行 docs 実体へ合わせる。
   - 正本は `docs/02_正本仕様/`、履歴は `docs/refs/` と明記する。

2. **module boundary test 更新**
   - `browseractor` / `voicechat` / `webgather` を依存方向テストへ追加する。
   - README 必須チェックを実体列挙型にする。

3. **Viewer 通常チャット contract 修正**
   - `to=mio|shiro|kuro|midori` の request / test を追加する。
   - `model_alias` / `route_prefix` の扱いを legacy として隔離する。
   - 明示 `/ops` などの command は本文 route として残す。

4. **cmd/rencrow の registrar 分割**
   - Viewer base / Viewer feature APIs / STT / TTS / LLM Ops / modules / IdleChat を registrar に分ける。
   - `Dependencies` を feature dependency bundle に分ける。

## 現時点の判定

RenCrow_CORE Ver0.80 化の土台はある。

- `modules/` 契約化は始まっている。
- `modulebridge` により legacy 実装との互換層もある。
- 依存方向テストも一部存在する。
- 全体テストと vet は通っている。

しかし、機能単位に「閉じ込めた」と呼ぶにはまだ早い。

- Viewer は旧 alias 変換を持つ。
- Chat module は legacy orchestrator adapter であり、相手別 Chat contract ではない。
- IdleChat は runtime 起動が `cmd/rencrow` に強く残る。
- 新 module の依存テストと README が不足している。
- docs 入口が実体とずれている。

したがって、次の作業は refactor 本体ではなく、Ver0.80 の境界を守るための「仕様入口修正 + 依存テスト拡張 + Viewer Chat contract の失敗テスト追加」から始めるのが妥当である。
