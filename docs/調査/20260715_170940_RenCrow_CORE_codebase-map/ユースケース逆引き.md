# RenCrow_CORE ユースケース逆引き

## 概要

利用者・運用者から見た8つの主要操作を、入口、判断点、状態、実装ファイルへ逆引きする。図は正常系の骨格であり、実際のprovider可用性や設定によりdegraded pathへ分岐する。

## UC-01 Runtimeを起動する

### トリガー

`rencrow run`、または既定commandでprocessを開始する。

### 処理フロー（シーケンス図）

~~~mermaid
sequenceDiagram
  actor Operator
  participant Main
  participant Config
  participant DI as Dependencies
  participant HTTP
  Operator->>Main: rencrow run
  Main->>Config: load and validate
  Main->>DI: buildDependencies
  DI-->>Main: services and stores
  Main->>HTTP: register routes and listen
~~~

### 処理フロー（テキスト）

1. CLI commandを解決する。
2. Configを読み、必須値とprovider設定を検証する。
3. store、LLM、agent、tool、Viewer serviceをcomposition rootで構築する。
4. routeとbounded background jobを登録し、HTTP serverを開始する。
5. signal受信時は`Dependencies.Shutdown`で保持資源を閉じる。

### 判断ポイント（フローチャート）

~~~mermaid
flowchart TD
  A[Start] --> B{Config valid}
  B -->|no| C[Fail fast]
  B -->|yes| D[Build dependencies]
  D --> E{Required dependency ready}
  E -->|no| C
  E -->|yes| F[Listen]
~~~

### 状態遷移

`created -> configuring -> ready -> serving -> stopping -> stopped`。現実装ではsignal時のHTTP drainは明示されず、資源close後にprocess終了する。

~~~mermaid
stateDiagram-v2
  [*] --> Configuring
  Configuring --> Ready
  Ready --> Serving
  Serving --> Stopping
  Stopping --> Stopped
~~~

### よくある誤解

- module単体のhealth成功だけではCore全体のreadyを意味しない。
- `Dependencies.Shutdown`と`http.Server.Shutdown`は別の責務である。

### 関連モジュール

cmd_runtime、infrastructure_persistence、feature_facades。

### 関連ソースファイル

- `cmd/rencrow/main.go`
- `cmd/rencrow/runtime_dependencies.go`
- `cmd/rencrow/routes.go`
- `internal/application/runtimeconfig`

## UC-02 Viewerから会話する

### トリガー

Viewerが`/viewer/*`会話APIへrecipient、text、任意のcontextを送信する。

### 処理フロー（シーケンス図）

~~~mermaid
sequenceDiagram
  actor User
  participant Viewer
  participant Handler
  participant Orchestrator
  participant Agent
  participant Hub
  User->>Viewer: message
  Viewer->>Handler: POST send
  Handler->>Orchestrator: ProcessMessage
  Orchestrator->>Agent: route with RecallPack
  Agent-->>Orchestrator: response and events
  Orchestrator->>Hub: publish
  Hub-->>Viewer: SSE projection
~~~

### 処理フロー（テキスト）

1. Viewer handlerが入力を検証し、sender/recipient/threadを正規化する。
2. `MessageOrchestrator`が明示recipient、command、sourceなどからrouteを決める。
3. 会話履歴とRecallPackを組み、Mio、Shiro、Coder、Workerの対象へ渡す。
4. tool/adviceを必要に応じて実行し、応答とtrace eventを保存・配信する。

### 判断ポイント（フローチャート）

~~~mermaid
flowchart TD
  A[Input] --> B{Explicit recipient}
  B -->|yes| C[Use explicit route]
  B -->|no| D{Command or policy match}
  D -->|yes| E[Use policy route]
  D -->|no| F[Default chat route]
  C --> G[Recall and execute]
  E --> G
  F --> G
~~~

### 状態遷移

`accepted -> routed -> recalling -> generating -> tool/advice optional -> completed|failed`。

~~~mermaid
stateDiagram-v2
  [*] --> Accepted
  Accepted --> Routed
  Routed --> Recalling
  Recalling --> Generating
  Generating --> Completed
  Generating --> Failed
~~~

### よくある誤解

- Viewerはagent本体ではなくHTTP/SSE projectionである。
- Coderは提案責務、Workerは実行責務であり、副作用権限は同一ではない。

### 関連モジュール

adapter_viewer_channels、domain_application、infrastructure_persistence。

### 関連ソースファイル

- `internal/adapter/viewer/handler_send.go`
- `internal/adapter/viewer/hub.go`
- `internal/application/orchestrator/message_orchestrator.go`
- `internal/infrastructure/tools/runner.go`

## UC-03 音声会話する

### トリガー

Viewerで音声入力を開始し、音声chunkまたは確定音声を送る。

### 処理フロー（シーケンス図）

~~~mermaid
sequenceDiagram
  actor User
  participant Viewer
  participant STT
  participant Chat
  participant TTS
  User->>Viewer: voice
  Viewer->>STT: audio
  STT-->>Viewer: transcript
  Viewer->>Chat: finalized text
  Chat-->>Viewer: response text
  Viewer->>TTS: synthesize
  TTS-->>Viewer: audio reference
~~~

### 処理フロー（テキスト）

1. audioをSTT contractへ渡し、partial/final transcriptを受ける。
2. final transcriptだけを会話入力として確定する。
3. Chatの応答textをTTSへ渡す。
4. Viewerは表示成功、音声取得成功、browser playback成功を別状態として扱う。

### 判断ポイント（フローチャート）

~~~mermaid
flowchart TD
  A[Audio] --> B{Final transcript}
  B -->|no| C[Update partial UI]
  B -->|yes| D[Send chat]
  D --> E{TTS ready}
  E -->|no| F[Text-only response]
  E -->|yes| G[Fetch and play audio]
~~~

### 状態遷移

`capturing -> transcribing -> finalized -> responding -> synthesized -> fetched -> played`。各層で個別に失敗し得る。

~~~mermaid
stateDiagram-v2
  [*] --> Capturing
  Capturing --> Transcribing
  Transcribing --> Finalized
  Finalized --> Responding
  Responding --> Synthesized
  Synthesized --> Fetched
  Fetched --> Played
~~~

### よくある誤解

- synthesis成功はbrowser playback成功ではない。
- transcript表示はChat送信完了ではない。

### 関連モジュール

~~~mermaid
sequenceDiagram
  participant Shiro
  participant Policy
  participant Advisor
  participant Codex
  participant Audit
  Shiro->>Policy: request advice
  Policy-->>Shiro: allow deny or approval-required
  Shiro->>Advisor: bounded brief
~~~

### 処理フロー（テキスト）

1. request目的と対象範囲を分類する。
2. approval policyとpermission境界を確認する。

~~~mermaid
flowchart TD

  B -->|no| C[Reject]

  B -->|yes| E[Run advisor]
  D --> E
  E --> F[Return proposal]
~~~

### 状態遷移

## UC-08 Repair Planeで修復jobを扱う

### トリガー

ViewerのRepair Planeから診断・修復候補の作成または許可済みjobを要求する。

### 処理フロー（シーケンス図）

~~~mermaid
sequenceDiagram
  actor Operator
  participant Viewer
  participant Repair
  participant Policy
  participant Worker
  participant Monitor
  Operator->>Viewer: request repair
  Viewer->>Repair: create job
  Repair->>Policy: classify risk and approval
  Policy-->>Repair: allowed scope
  Repair->>Worker: execute bounded action
  Worker->>Monitor: logs and result
  Monitor-->>Viewer: progress
~~~

### 処理フロー（テキスト）

1. 対象、症状、要求actionをjobとして受け付ける。
2. risk、approval、許可tool、timeoutを確定する。
3. Workerが許可範囲だけを実行する。
4. progress、evidence、resultをMonitorStoreとViewerへ反映する。

### 判断ポイント（フローチャート）

~~~mermaid
flowchart TD
  A[Repair request] --> B{Safe and in scope}
  B -->|no| C[Reject or request approval]
  B -->|yes| D[Create bounded job]
  D --> E{Execution success}
  E -->|yes| F[Verify and close]
  E -->|no| G[Record failure and evidence]
~~~

### 状態遷移

`requested -> classified -> approved -> queued -> running -> verifying -> completed|failed|cancelled`。

~~~mermaid
stateDiagram-v2

  Requested --> Classified
  Classified --> Approved
  Approved --> Queued
  Queued --> Running
  Running --> Verifying
  Verifying --> Completed
  Running --> Failed
  Running --> Cancelled
~~~

### よくある誤解
