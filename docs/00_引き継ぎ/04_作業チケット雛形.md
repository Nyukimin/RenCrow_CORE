# 作業チケット雛形

ドキュメント整理の作業は、なるべくこの形式でチケット化してください。

```markdown
# [DOC] タイトル

## 背景
なぜこの作業が必要か。

## 対象ファイル
- `path/to/file.md`

## 作業内容
- [ ] 何を確認するか
- [ ] 何を更新するか
- [ ] 何を更新しないか

## 正本への影響
- 正本変更あり / なし
- 変更する場合: `02_正本仕様/...`

## 派生文書への影響
- `01_理解/...`
- `04_構築指標/...`
- `05_運用/...`

## 元 docs.zip から戻す資料
- なし / あり
- 戻す場合の理由:

## 完了条件
- [ ] 正本と派生文書が矛盾しない
- [ ] 戻した資料の理由が記録されている
- [ ] 読む順番が壊れていない
- [ ] 旧仕様・作業ログが入口になっていない

## リスク
- 仕様変更と単なる説明改善が混ざる可能性
- 旧仕様を復活させてしまう可能性
- 詳細資料を戻しすぎて最小セットが肥大化する可能性
```

## すぐ作るべきチケット例

### [DOC] 正本と派生文書の整合性確認

対象:
- `02_正本仕様/01_仕様.md`
- `02_正本仕様/02_実装仕様.md`
- `01_理解/01_システム概要.md`
- `04_構築指標/01_構築ロードマップ.md`

完了条件:
- 正本にある重要な制約が派生文書に反映されている。
- 派生文書が正本と矛盾していない。

### [DOC] 未解決仕様競合の扱いを整理

対象:
- `05_運用/02_Codebase_Run_Summary.md`
- `02_正本仕様/02_実装仕様.md`

完了条件:
- 承認フロー存廃、risk=high 停止、サニタイズ責務の扱いをチケット化する。
- 仕様確定までは正本を書き換えない。

### [DOC] 除外ファイルの復帰候補レビュー

対象:
- `99_整理/除外ファイル一覧.csv`

完了条件:
- すぐ戻す資料、一時参照する資料、戻さない資料を分類する。
- 戻す場合は `restore_when` と理由が一致している。

### [DOC] 引き継ぎ導線の検証

対象:
- `README_引き継ぎ版.md`
- `00_引き継ぎREADME.md`
- `00_読む順番.md`

完了条件:
- 初見の担当者が読む順番で迷わない。
- 「次に何を作ればいいか」が分かる。
