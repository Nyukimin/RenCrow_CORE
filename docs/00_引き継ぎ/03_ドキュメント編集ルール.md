# ドキュメント編集ルール

## 文書の種類

| 種類 | 場所 | 役割 | 更新条件 |
|---|---|---|---|
| 正本 | `02_正本仕様/` | 仕様・実装契約・Runtime境界の本体 | 挙動や契約を変える時 |
| 理解用 | `01_理解/` | 初見の人向けの統合説明 | 正本更新後、読みやすさ改善時 |
| 構築指標 | `04_構築指標/` | 実装順・受入基準 | 正本更新後、構築判断が変わる時 |
| 運用補助 | `05_運用/` | セットアップ・解析サマリ | 実行環境や手順が変わる時 |
| 整理根拠 | `99_整理/` | 残す/外す判断、構成サマリ | ファイルを戻す/外す/再分類する時 |
| 引き継ぎ | `00_引き継ぎ/` | 次担当者向け作業導線 | 引き継ぎ方法が変わる時 |

## 更新順序

仕様の意味が変わる場合は、必ずこの順番で更新します。

1. `02_正本仕様/` を更新する。
2. 影響する `01_理解/` を更新する。
3. 影響する `04_構築指標/` を更新する。
4. 必要なら `05_運用/` を更新する。
5. 戻した資料や外した資料があれば `99_整理/` を更新する。

派生文書だけを直して、正本と矛盾させてはいけません。

## 新規文書を追加する条件

新しい文書は、次のいずれかを満たす時だけ追加します。

- 正本に入れるほど重要な仕様変更である。
- 初見の人が理解するために、既存文書では明らかに不足している。
- 構築・受入・運用の判断に直接必要である。
- 元 docs.zip から一時復帰する詳細資料で、必要期間と戻す理由が明確である。

## 新規文書の最小ヘッダ

新規文書には、先頭に次の情報を置くことを推奨します。

```markdown
# タイトル

- status: draft | active | source | generated | archive | temporary
- owner: 未定 / 担当者名
- source: 参照元ファイルまたは作成理由
- last_reviewed: YYYY-MM-DD
- scope: この文書が扱う範囲
```

## やってはいけない整理

- 「念のため」で詳細資料を大量に戻す。
- 旧仕様を正本より上に置く。
- 日付付きログを現在仕様の入口にする。
- `README.md` に全情報を詰め込む。
- 同じ説明を複数ファイルにコピペして管理する。
- `generated` な統合ガイドを正本の代わりにする。

## 戻した資料の扱い

元 docs.zip から資料を戻す場合は、必ず次を記録します。

- 戻したファイル名。
- 戻した理由。
- どの正本と関係するか。
- いつまで必要か。
- 最小セットに恒久追加するのか、一時参照なのか。

記録先は `99_整理/最小化判定.md` または作業チケットです。
