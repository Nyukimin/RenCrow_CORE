## ルーティング決定ルール（優先順位）

優先順位は固定：

1. 明示コマンド
2. ルール辞書（強い証拠）
3. 分類器（推論LLM）を1回だけ呼ぶ
4. 信頼度不足、error、空routeなら`CHAT`へ安全側fallbackする

### 明示コマンド

* `/chat` `/plan` `/analyze` `/heavy` `/ops` `/research` `/wild` `/code` `/code1` `/code2` `/code3` `/code4` をサポート
* 明示コマンドは **プロンプト注入より強い**（入力本文に「/codeって書いて」とあっても、実際に先頭コマンドでなければ無効）
* command直後が文字列末尾または空白の場合だけ有効とし、`/codebase`等のprefix衝突を除外する

### ルール辞書（初期セットは「強い証拠」だけ）

* Codex work path、file参照を伴う編集要求、dictionary配列のkeywordをこの順で評価する
* dictionary ruleにpriority fieldはなく、競合時は配列の定義順を使う
* match時はroute、confidence、evidenceを残す
* それ以外は分類器へ

  * 分類器呼び出し自体が失敗した場合（後述）はCHATへ倒す

## 分類器（推論LLM）の仕様

* 目的：辞書で確定できない入力に対し、route候補を提案する
* 呼び出し回数：1ユーザー入力につき最大1回（必要時のみ）
  （この上限は分類器に対する制約。再ルート時にワーカーを1回追加実行することはありうる）
* 公開契約は`routing.Decision`の`route`、`confidence`、`reason`、`evidence`である
* 現行`LLMClassifier`はprovider応答に含まれる有効route tokenを抽出する。JSON固定schemaではない

### Router側の採用ゲート

* routeが空、または`confidence < 0.7`なら分類器結果を採用しない
* 現行codeにCODE専用`0.8`gateはない。変更時はcodeとtestを先に同期する

### 分類器の故障時仕様（重要）

* provider error、classifier未設定、空route、低confidenceの場合、分類器結果を採用せず`CHAT`へfallbackする
* 列挙外tokenは`LLMClassifier`で`CHAT`候補へ落とし、Mio側の採用gateとevidenceで追跡する

## ワーカー出力契約

全経路共通の戻り値は設けず、agentの`(string, error)`、moduleの`worker.Result`、orchestratorの`ProcessMessageResponse`を境界ごとに使う。`failed`、`denied`、`canceled`を成功textへ埋め込んで握りつぶさない。

### ワーカーの故障時仕様

* errorまたは失敗statusを当該境界の失敗として扱い、別の成功結果へ暗黙変換しない
* 入力、route、job / session identity、失敗理由を安全なlogへ残す

## ループ制御（次の処理判断）

通常message routeは1回のDecisionに基づきroute別executorへ渡す。全worker共通の`needs_next_loop`契約や、共通JSONを使う自動再routeは現行code contractではない。自律実行など個別use caseの反復は、そのuse case固有の上限、停止、承認、結果型で管理する。

### ユーザー確認（キャッチ）が発生する条件（例）

* `risk == high`
* `confidence` が低く、分岐が重大
* 追加情報がないと正しく進めない（環境・前提・選択肢の確定が必要）

### 自動ループの上限

* 無限ループ防止のため、システムで最大回数/最大時間を持つ（値は実装仕様で決定）
* 上限到達時は強制停止し、会話LLMが「ここまでの成果＋次に必要な入力」を短く報告する
