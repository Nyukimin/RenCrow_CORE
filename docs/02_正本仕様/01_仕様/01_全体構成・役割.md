## 目的と前提

* ユーザー（れん）が直接会話する相手は **会話LLM 1つ**（人格・文体・窓口は常に1つ）
  * 例外: Viewer の通常チャット UI では、会話相手を明示選択できる。これはローカル運用・開発確認用の拡張であり、LINE/Slack の既定窓口を変更しない。
* 本仕様の初期適用対象チャネルは **LINE と Slack**
  （RenCrow本体が持つ他チャネル機能は本仕様の対象外。既存挙動を壊さない）
* ローカルLLMは **RenCrow_LLM の OpenAI 互換 endpoint** を使用する。Ollama 直結は legacy fallback とする。
* LLM / TTS / STT のサーバ所在は実運用 Config の `rencrow` ブロックを正本とし、Repo 内 Config はそのスナップショットとして扱う。
* **常駐 endpoint は最低2つ**（Chat＋Worker）。必要に応じて Heavy / Wild も使う。
* **コードが必要な場合のみ**クラウドAPIを **1本だけ**利用する（それ以外はローカル）
* 通常messageのroute決定は **Mioの`DecideAction`** が行い、route別executorへ委譲する
* 役割名は **Chat / Worker / Coder** で固定し、愛称と実体LLMは設定で差し替える

  * Viewerの`to`は会話相手であり、Routerのexecution routeとは別に扱う

## 用語

* Chat：ユーザー応答の最終文を生成する唯一の窓口役割（ユーザーに見える発話は必ずここ）
* Worker：分類器・データ蓄積・データ分析・検証などを担う非コーディング役割
* Coder：コード生成/修正に使うコーディング役割（必要時のみクラウド）
* Router：入力をカテゴリに分岐させ、ワーカー呼び出しを制御するシステム部
* LoopController：ループ回数/停止条件/安全制約を管理し、次処理を実際に回す最終決定者（Routerに含めてもよい）
* ワーカー：推論LLM／コードLLM／（必要なら）リサーチ処理など、裏方の実行ユニット
* ループ：ワーカー処理→追加処理の必要性判断→（必要なら）次ワーカー処理…の反復

## 全体構成

* 入口（LINE/Slack）
  → RenCrow本体（受信/送信・セッション管理）
  → Router/LoopController（分岐・制約・回数管理）
  → ワーカー（推論/コード/リサーチ等）
  → 会話LLM（最終整形・宣言・返信）
  → 入口へ返信

### 重要な制約（権限境界）

* 最終回答文は必ず会話LLMが生成する（ワーカー出力は“素材”）
* 明示command、RuleDictionary、classifier、safe fallbackの順でMioがrouteを決める
* Viewer の会話相手選択は明示操作として扱い、前段Routerの自動振り分けとは区別する
* 「次のループが必要かも」という判断はワーカーが出せるが、
  **次を回す/止める最終決定は LoopController が行う**

  * 会話LLMは「続行したい/止めたい」の意見（提案）を返せるが、最終権限はシステム

## ルーティングカテゴリ（拡張可能な最小セット）

* CHAT：説明/相談/雑談/要約/レビュー
* PLAN：段取り、仕様化、タスク分解、方針決定
* ANALYZE：抽出・構造化・タグ付け・集計・傾向分析（データ蓄積を含む）
* OPS：運用手順、障害対応、設定確認、コマンド案内（実行自動化は別途）
* RESEARCH：調査、出典確認、比較、最新情報（必要時のみ）
* CODE / CODE1 / CODE2 / CODE3：コード生成/修正/diff作成/実装
  - CODE1=仕様設計向け（DeepSeek等）
  - CODE2=実装向け（OpenAI等）
  - CODE3=高品質コーディング/推論（Claude API専用、2026-02-24追加）
  - CODE はタスク内容で自動振り分け

## Viewer 通常チャットの会話相手選択（2026-06-29追加）

### 目的

Viewer では、ユーザーが Mio 以外の Chat 相手とも直接会話できるようにする。

これは通常チャットの操作性を広げるための拡張であり、裏方ワーカーの自動委譲とは別の明示的な会話相手選択である。
Viewer 新設と Chat 経路新設を混ぜない。Viewer は選択された相手を `to` として渡すだけで、route/model alias 解決は Chat/Orchestrator 側で行う。

### 対象ロール

| 会話相手 | Viewer `to` | 利用モデル | 主用途 | 副作用 |
|---|---|---|---|---|
| `Mio` | `mio` | `Chat` | 通常会話、軽い相談、最終応答 | なし |
| `Shiro` | `shiro` | `ChatWorker` | 作業相談、手順整理、状態説明、実行前の見積もり | 実行しない |
| `Kuro` | `kuro` | `Heavy` | 深い分析、設計検討、原因調査、長文整理 | 実行しない |
| `Midori` | `midori` | `Wild` | 創作、画像生成プロンプト、ComfyUI 関連、自由発想 | 画像生成は明示要求時のみ |

### ルーティング原則

* 既定ターゲットは `Mio` とする。
* ユーザーが会話相手を選択した場合、Viewer は送信 payload に `to` を含める。
* Viewer は `to` を `/ops` `/wild` `/heavy` などの明示 route command、`model_alias`、`route_prefix` へ変換しない。
* Chat/Orchestrator は `to` を見て、Chat 系の相手別経路へ解決する。
* 入力本文が `/chat` `/ops` `/analyze` `/heavy` `/wild` `/code*` `/plan` `/research` などの明示コマンドで始まる場合、本文側の明示コマンドをターゲット選択より優先する。
* `Shiro` は `ChatWorker` を使うが、通常の `OPS` 実行とは分ける。通常会話は、相談・説明・計画補助までとし、コマンド実行やファイル変更を行わない。
* `Kuro` は `Heavy` を使い、深い分析を返すが、直接実行・直接修正は行わない。
* `Midori` は `Wild` を使い、創作・画像関連応答を返す。ComfyUI 等の画像生成は、ユーザーが明示的に生成を求めた場合に限る。

### 表示とログ

* 選択ロールの応答は、Viewer Timeline 上ではユーザー向け会話として表示する。
* イベントは `agent.response` を使い、`from` は応答ロール、`to` は `user` とする。
* ユーザー発話イベントは `message.received` を使い、`from=user`、`to=mio|shiro|kuro|midori` とする。
* 会話ログは全 Chat 相手で共通とする。`Mio` / `Shiro` / `Kuro` / `Midori` の誰に話しても、過去ログは共有される。
* 共通ログでは各発言の `from` / `to` / 発言者名を必ず保持し、誰の発話かを曖昧にしない。
* 内部委譲・中間報告は、従来通り coordination trace として通常会話本文とは分ける。
* セッションログには、実際に応答した route と job_id を保存する。

### 起動・失敗時の扱い

* `Shiro` / `Kuro` / `Midori` の利用モデルである `ChatWorker` / `Heavy` / `Wild` が未起動の場合、Chat/Orchestrator 側は既存の LLM Ops 経路で対象 runtime の起動を試みる。
* 対象 runtime が使えない場合、Mio へ黙ってフォールバックしない。ユーザーには対象ロールが利用できない理由を短く表示する。
* Runtime request は必ず `rencrow -> RenCrow_LLM -> RenCrow_XXX` の経路を通し、Viewer または rencrow server からモデル backend を直接呼ばない。

### Chat 相手切替と RenCrow_LLM 経路のあるべき姿（2026-06-30追加）

Viewer の Chat 相手切替は、表示だけの変更ではない。選択された相手に対応する RenCrow_LLM runtime を起動または選択し、利用可能になってから表示を確定する。

#### 経路の分離

推論本体と起動/切替管理は別責務とする。

| 種別 | rencrow 側設定 | 呼び先 | 主な API | 責務 |
|---|---|---|---|---|
| 推論 | `local_llm` | RenCrow_LLM OpenAI 互換 endpoint | `/v1/chat/completions`, `/v1/models` | 応答生成、公開 alias の確認 |
| 起動/切替管理 | `llm_ops` | RenCrow_LLM management endpoint | `/health`, `/v1/status`, `/v1/control/start`, `/v1/control/stop` | モデル起動、停止、選択状態確認 |

`local_llm` に endpoint が設定されていても、`llm_ops` が未設定なら Chat 相手の自動切替は完了できない。

#### 相手別 runtime

| Chat 相手 | 利用 runtime selection | 推論 alias | 既定推論 endpoint |
|---|---|---|---|
| `Mio` | `Chat` | `Chat` | `local_llm.chat_base_url` |
| `Shiro` | `ChatWorker` | `ChatWorker` | `local_llm.worker_base_url` |
| `Kuro` | `Heavy` | `Heavy` | `local_llm.heavy_base_url` |
| `Midori` | `Wild` | `Wild` | `local_llm.wild_base_url` |

`selection` は RenCrow_LLM の起動/切替管理用の論理名であり、推論 request の `model` と同一概念ではない。
`Shiro` の `selection=ChatWorker` は RenCrow_LLM 側で起動対象 role として `Worker` に正規化されるが、推論 request では `model=ChatWorker` を維持する。
これにより、起動対象は Worker endpoint でありながら、ChatWorker 固有の max_tokens / reasoning / context budget / prompt が適用される。

#### 切替シーケンス

1. Viewer で `Shiro` / `Kuro` / `Midori` が選択される。
2. Viewer は即座に選択表示を確定しない。現在の相手を保持したまま入力欄を停止する。
3. 切替前の相手が「次の相手に代わるので少し待って」という趣旨の発話を Timeline に追加する。
4. rencrow は server-side で `llm_ops` に `POST /v1/control/start` を送り、対象 selection を起動/選択する。
5. rencrow は `llm_ops` の `GET /v1/status` を短周期で確認し、対象 selection が ready になるまで待つ。
6. ready 確認後、Viewer は選択表示を新しい相手へ確定する。
7. 新しい相手が「おまたせ」相当の発話を Timeline に追加する。
8. 入力欄を再開し、以後の通常送信 payload は `to=<selected>` とする。

#### 起動直後と degraded 状態

起動直後は、RenCrow_LLM の `/v1/models` や `/v1/status` がまだ返らないことがある。この間はモデル名・現在 selection を確定情報として扱わない。

* モデル名未取得: UI はグレーダウンし、選択確定不可とする。
* `/v1/models` は返るが `/v1/status` が未取得: endpoint は存在するが、Chat 相手切替完了とはみなさない。
* `/v1/status` で対象 selection が ready: 切替完了として扱う。
* 接続拒否、認証失敗、管理 API 未設定: 切替失敗として前の相手へ戻す。

#### 送信前ガード

ユーザー送信時は、表示中の Chat 相手と実際に ready な runtime selection が一致していることを確認する。

一致していない場合は送信しない。入力欄を停止し、切替処理を完了させてから送信可能にする。Mio への暗黙フォールバックは禁止する。
