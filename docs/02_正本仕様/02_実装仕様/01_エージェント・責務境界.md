### 2026-06-30: 実運用 Config 正本と Repo 内スナップショット
- 実運用 Config の正本はユーザー環境側の `.rencrow/config.yaml` とする。
- Windows では `C:\Users\nyuki\.rencrow\config.yaml`、macOS / Linux では `$HOME/.rencrow/config.yaml` を正本 Config として扱う。
- `run/local-windows-runtime.yaml` は Repo 内に置く実運用 Config のスナップショットであり、起動時の自動 fallback 先ではない。
- 起動時は `RENCROW_CONFIG` が指定されていればその path を使う。未指定の場合はユーザー環境側の `.rencrow/config.yaml` を使う。
- `.rencrow/config.yaml` が存在しない、読めない、または validation に失敗する場合は起動失敗とし、Repo 内 Config へ黙って切り替えない。
- `idle_chat.enabled=true` でも、起動直後に `IdleChatOrchestrator.Start()` は呼ばない。`idle_chat.startup_delay_sec` が未指定または `0` の場合は `1800` 秒を補完し、起動時は `scheduleIdleChatStartup(30m, startIdleChatRuntime)` のみ行う。
- LLM / TTS / STT のサーバ所在は `rencrow.llm` / `rencrow.tts` / `rencrow.stt` を正本とし、既存の `local_llm` / `llm_ops` / `tts` / `stt` は互換・移行用として扱う。
- STT が LLM の音声入力機能を使う構成は `rencrow.stt.engine=llm_audio` で表す。rencrow / Viewer は RenCrow_STT を呼び、RenCrow_STT が RenCrow_LLM の audio endpoint を利用する。この構成では独立した STT server を起動しないため、`rencrow.stt.base_url` は必須にしない。
- Repo 内スナップショットを使う検証では、必ず `RENCROW_CONFIG=run/local-windows-runtime.yaml` のように明示する。

### 2026-06-30: Viewer Chat 相手切替と RenCrow_LLM 管理経路
- Viewer Chat 相手切替は、表示操作だけではなく RenCrow_LLM runtime selection の起動/切替を伴う。
- 推論本体の `local_llm` と、起動/切替管理の `llm_ops` を分離する。
- `local_llm` は OpenAI 互換推論 endpoint (`/v1/chat/completions`) を呼ぶ。モデル起動・停止・selection 切替は行わない。
- `llm_ops` は RenCrow_LLM management API への server-side proxy とし、`/v1/control/start`、`/v1/status`、`/v1/control/stop` を扱う。
- `llm_ops` が未設定の場合、Viewer は Chat 相手切替を完了扱いにしない。表示だけ成功させて送信可能にしてはいけない。
- 起動直後のモデル名未取得は failure ではなく `pending/degraded` として扱い、モデル選択 UI をグレーダウンする。
- 対象 selection が ready になった後に選択表示を確定し、新しい相手から「おまたせ」相当の発話を追加して入力を再開する。

## 1.0 旧エージェントアーキテクチャ記録（2026-03-01追加）

### 1.0.1 基本設計原則

**5つのエージェントは同じコアを共有し、機能モジュールで個性を付ける**

```
共通エージェントコア
├── ID, Alias, Provider, Model
├── LLM クライアント
└── 機能モジュール（プラグイン）
    ├── Chat 用モジュール
    ├── Worker 用モジュール
    └── Coder 用モジュール
```

### 1.0.2 エージェント構成（5体）

| ID | 愛称 | Provider | Model | 主要モジュール |
|----|------|----------|-------|---------------|
| **chat** | Mio | local_openai | Chat | LightweightReception, FinalDecision, ApprovalUI |
| **worker** | Shiro | local_openai | Worker / ChatWorker | Routing, LoopControl, Heartbeat, Aggregation, Execution |
| **order1** | Aka | deepseek | deepseek-chat | ProposalGeneration, CodeAnalysis |
| **order2** | Ao | openai | gpt-4 | ProposalGeneration, CodeAnalysis |
| **order3** | Gin | anthropic | claude-sonnet-4-5 | ProposalGeneration, ApprovalFlow, CodeAnalysis |

**注**: Coder1/Coder2/Coder3 という旧名称も設定で使用可能（後方互換）

### 1.0.3 モジュール分類

#### Chat 用モジュール
- **LightweightReceptionModule**: 軽量な入力受付、JobID 付与
- **FinalDecisionModule**: Worker からの報告を受けて最終決定
- **ApprovalUIModule**: 承認 UI（LINE/Slack へメッセージ送信）

#### Worker 用モジュール
- **RoutingModule**: ルーティング決定（Router/Classifier を内包）
- **LoopControlModule**: ループ制御
- **HeartbeatCollectorModule**: 全エージェントの Heartbeat を集約
- **AggregationModule**: 各エージェント結果の集約
- **ExecutionModule**: 実行・道具係（ファイル操作、コマンド実行）

#### Coder 用モジュール（Order1/2/3 共通）
- **ProposalGenerationModule**: 提案生成（plan/patch）
- **ApprovalFlowModule**: 承認フロー管理（Order3 専用）
- **CodeAnalysisModule**: コード分析
- **PatchApplicationModule**: パッチ適用（Worker へ委譲）

#### 共通モジュール
- **LoggingModule**: ログ出力
- **SessionModule**: セッション管理
- **MemoryModule**: メモリ管理

### 1.0.4 動作フロー（新設計）

```
1. LINE入力
   ↓
2. Chat (Mio) - LightweightReceptionModule
   - JobID 付与
   - Worker へタスク委譲
   ↓
3. Worker (Shiro)
   - RoutingModule: ルーティング決定（CHAT/CODE1/CODE2/CODE3）
   - LoopControlModule: ループ制御
   - 必要に応じて Order1/Order2/Order3 へ委譲
   ↓
4. Order1/Order2/Order3 (Aka/Ao/Gin)
   - ProposalGenerationModule: plan/patch 生成
   - (Order3 のみ) ApprovalFlowModule: 承認要求
   ↓
5. Worker (Shiro)
   - AggregationModule: 結果集約
   - HeartbeatCollectorModule: 各エージェント状態確認
   - Chat へ報告
   ↓
6. Chat (Mio) - FinalDecisionModule
   - 最終決定
   - ApprovalUIModule: 承認が必要なら LINE/Slack へ通知
   - ユーザーへ応答
```

## 1. スコープ・責務境界
- 対象チャネル: LINE / Slack（他チャネルは対象外、既存挙動を壊さない）
- 最終応答は Chat のみが生成する（軽量・最終決定に特化）
- 旧設計案ではWorker RoutingModuleを想定した。現行のルーティング決定はMioの`DecideAction`が担当する
- LLMは提案者であり決定権を持たない

### 1.1 Viewer 通常チャットの会話相手拡張

#### 1.1.1 適用範囲

本節は Viewer の通常チャット UI にのみ適用する。LINE / Slack / Telegram / Discord 等の外部チャネルには自動適用しない。

外部チャネルでは、従来通り `CHAT(Mio)` を入口にし、Mio から必要に応じて Worker / Coder / Wild / Heavy へ委譲する。

#### 1.1.2 Chat 相手と利用モデル

| Chat 相手 | Viewer payload | 利用モデル | 応答イベント | 主用途 |
|---|---|---|---|---|
| `Mio` | `to=mio` | `Chat` | `agent.response mio -> user` | 通常会話、軽い相談、最終応答 |
| `Shiro` | `to=shiro` | `ChatWorker` | `agent.response shiro -> user` | 作業相談、手順整理、実行前の見積もり |
| `Kuro` | `to=kuro` | `Heavy` | `agent.response kuro -> user` | 深い分析、設計検討、原因調査 |
| `Midori` | `to=midori` | `Wild` | `agent.response midori -> user` | 創作、画像プロンプト、ComfyUI 関連 |

`to` は会話相手 ID であり、route ではない。
Viewer は `to` を `/ops`、`/analyze`、`/wild`、`model_alias`、`route_prefix` などへ変換しない。
Chat/Orchestrator 側で `to` を Chat 系の相手別経路へ解決する。
既存の `OPS` は Shiro / Worker の実行系 route であり、`to=shiro` の通常会話と混同しない。

#### 1.1.3 入力解決順

Viewer 送信時の入力解決順は以下とする。

1. 本文が明示 route command で始まる場合、その route command を採用する。
2. 本文に明示 route command がない場合、Viewer は UI の Chat 相手を `to` として送る。
3. Chat/Orchestrator は `to` を Chat 相手へ解決する。
4. `to` が未指定または不明な場合、`Mio` / `to=mio` として扱う。

明示 route command は少なくとも以下を含む。

`/chat`, `/ops`, `/analyze`, `/heavy`, `/wild`, `/code`, `/code1`, `/code2`, `/code3`, `/code4`, `/plan`, `/research`

#### 1.1.4 Shiro / ChatWorker の制約

`Shiro` の通常 Chat は `ChatWorker` を使う。これは会話用 alias であり、以下を禁止する。

- ファイル作成・更新・削除
- shell / PowerShell / cmd / git 等の直接実行
- 外部 API への副作用を伴う操作
- Coder patch の適用

`Shiro` が実行の必要性を判断した場合は、実行せず、次に使うべき route (`OPS`, `CODE*` など) と理由を短く返す。

#### 1.1.5 Kuro / Midori の制約

`Kuro` は `Heavy` を使い、分析・診断・設計検討に限定し、直接実行しない。

`Midori` は `Wild` を使い、創作・画像関連を担当する。ComfyUI などの画像生成は、ユーザーが明示的に生成を求めた場合のみ実行する。

#### 1.1.6 Event / Timeline 契約

選択 Target の最終応答は、Timeline に通常会話として表示する。

- event type: `agent.response`
- `from`: `mio` / `shiro` / `kuro` / `midori`
- `to`: `user`
- `route`: Chat 系の応答 route。実装上 route を分ける場合でも、Viewer の `to` と混同しない。
- `job_id`: 必須

ユーザー発話は `message.received` として記録する。

- `from`: `user`
- `to`: `mio` / `shiro` / `kuro` / `midori`
- `session_id`: Viewer の通常チャットでは共通セッションを使う。

会話ログは全 Chat 相手で共通とする。各発言に `from` / `to` / 発言者名を保存し、Mio / Shiro / Kuro / Midori の誰が発言したかを明示する。

内部委譲、Worker 報告、Coder 途中出力は従来通り coordination trace に分類し、通常会話の本文として重複表示しない。

#### 1.1.7 Runtime 起動と失敗時

Chat/Orchestrator は `to=shiro|kuro|midori` の場合、利用モデル `ChatWorker` / `Heavy` / `Wild` の health / status を確認し、対象 runtime が停止していれば既存の LLM Ops start API で起動を試みる。

起動失敗、health 不良、queue timeout、generation timeout の場合は、Mio へ暗黙フォールバックしない。エラーは対象 Chat 相手名を含めてユーザーに表示する。

#### 1.1.8 Chat 相手切替の runtime ready 契約

Chat 相手切替は、Viewer UI の選択状態と RenCrow_LLM の実 runtime selection を一致させるための同期処理である。

##### 責務境界

- Viewer は、ユーザー操作、入力停止、切替中/完了/失敗の表示だけを担当する。
- Viewer は RenCrow_LLM management API や backend LLM を直接呼ばない。
- rencrow Viewer adapter は、Viewer からの切替要求を server-side で受け、`llm_ops` proxy を通して RenCrow_LLM management API へ渡す。
- Chat/Orchestrator は、通常送信 `to=mio|shiro|kuro|midori` を Chat 相手別 route へ解決する。
- RenCrow_LLM は、公開 alias (`Chat`, `ChatWorker`, `Heavy`, `Wild`) と backend 物理モデル、ロード状態、selection 状態の正本を持つ。

##### 推論経路

通常応答生成は `local_llm` を使う。

| Chat 相手 | Route | Provider alias | Base URL source | HTTP |
|---|---|---|---|---|
| `Mio` | `CHAT` | `Chat` | `local_llm.chat_base_url` | `POST /v1/chat/completions` |
| `Shiro` | `WORKER_CHAT` | `ChatWorker` | `local_llm.worker_base_url` | `POST /v1/chat/completions` |
| `Kuro` | `ANALYZE` | `Heavy` | `local_llm.heavy_base_url` | `POST /v1/chat/completions` |
| `Midori` | `WILD` | `Wild` | `local_llm.wild_base_url` | `POST /v1/chat/completions` |

##### 起動/切替管理経路

Chat 相手切替時は `llm_ops` を使う。

| rencrow route | RenCrow_LLM management API | 用途 |
|---|---|---|
| `GET /viewer/llm-ops/health` | `GET /health` | 管理 API 到達確認 |
| `GET /viewer/llm-ops/status` | `GET /v1/status` | role/selection の ready 判定 |
| `POST /viewer/llm-ops/start` | `POST /v1/control/start` | selection 起動/切替 |
| `POST /viewer/llm-ops/stop` | `POST /v1/control/stop` | selection 停止 |

`/viewer/llm-ops/start` の body は、Chat 相手に応じて以下にする。

| Chat 相手 | body |
|---|---|
| `Shiro` | `{"selection":"ChatWorker"}` |
| `Kuro` | `{"selection":"Heavy"}` |
| `Midori` | `{"selection":"Wild"}` |

`selection` は起動/切替管理用であり、推論 alias とは分離する。
`ChatWorker` selection は RenCrow_LLM 側で `Worker` role 起動に正規化されるが、通常応答生成では `local_llm.worker_base_url` に `model=ChatWorker` を送る。
rencrow は `ChatWorker` を `Worker` に置き換えて送信してはいけない。

##### 切替シーケンス

1. Viewer が現在の選択相手 `previous` と次の相手 `next` を保持する。
2. Viewer は入力欄、送信ボタン、添付ボタンを disabled にする。
3. Viewer はまだ `next` を確定表示しない。
4. `previous` から `next` へ代わる旨の発話を Timeline に追加する。
5. Viewer adapter は `llm_ops` configured/token を確認する。
6. 未設定または token 不足なら、切替失敗として `previous` 表示へ戻し、入力を再開する。
7. configured の場合、`POST /viewer/llm-ops/start` で `next` の selection を起動する。
8. `GET /viewer/llm-ops/status` を 500 ms 間隔で poll する。
9. timeout は通常 35 秒以上とし、初回モデルロードが遅い場合を考慮する。
10. 対象 selection が ready になったら、Viewer は `next` を確定表示する。
11. `next` から「おまたせ」相当の発話を Timeline に追加する。
12. 入力欄、送信ボタン、添付ボタンを enabled に戻す。

##### ready / degraded / failed 判定

| 状態 | 判定 | UI |
|---|---|---|
| `pending` | 起動直後で `/v1/models` / `/v1/status` 未取得 | モデル名をグレーダウン、選択確定不可 |
| `degraded` | `/v1/models` は返るが `/v1/status` が未取得または timeout | endpoint 存在として表示するが、切替完了にはしない |
| `ready` | `/v1/status` で対象 selection が health OK かつ halted でない | 選択確定、入力再開 |
| `failed` | 接続拒否、管理 API 未設定、認証失敗、start 失敗、timeout | 前の相手へ戻し、理由を表示 |

`/v1/models` の `backend_model` や model id は、モデル名表示の補助情報として使える。ただし Chat 相手切替の完了条件にはしない。

##### 送信前ガード

通常送信前に、Viewer が表示している Chat 相手と ready 済み selection が一致していることを確認する。

一致しない場合は `/viewer/send` を送らず、切替処理を先に完了させる。`Mio` への暗黙フォールバックは禁止する。

##### 失敗時表示

失敗メッセージは以下を満たす。

- 対象 Chat 相手名を含む。
- `llm_ops` 未設定、token 不足、接続拒否、timeout の区別を落とさない。
- backend の詳細パスや秘密情報は表示しない。
- Timeline には通常会話と同じ `agent.response` として短く表示する。

#### 1.1.9 実装対象

- Viewer UI: Chat 時のみ `Shiro` / `Kuro` / `Midori` の選択 UI を表示する。選択中の相手を上段に表示し、未選択の相手だけを下段候補に出す。
- Viewer send payload: `to=mio|shiro|kuro|midori` のみをサーバへ渡す。`model_alias` / `route_prefix` はこの Chat 相手選択では使わない。
- Viewer Timeline: `agent.response` の `from` が `mio` / `shiro` / `kuro` / `midori` で `to=user` の場合、通常会話として描画する。
- Orchestrator: `to` を受け取り、Chat 相手ごとの利用モデルへ解決する。`Shiro=ChatWorker`、`Kuro=Heavy`、`Midori=Wild`。
- Chat history: Viewer 通常チャットの会話履歴は共通化し、各 message に `from` / `to` / 発言者名を保存する。
- Agent runtime: `Shiro` / `Kuro` / `Midori` の Chat 応答 adapter を追加し、既存の実行系 route と分離する。
- Tests: send payload `to`、明示 route 優先、Timeline 表示、相手別モデル解決、共通履歴、runtime unavailable 表示を検証する。

### 1.1 役割命名（固定）と実体（可変）
- **エージェント ID は固定**: `chat / worker / order1 / order2 / order3`
  - 旧名称 `coder1 / coder2 / coder3` も設定で使用可能（後方互換）
- **愛称と実体LLMは可変**: 設定ファイル `routing.llm.*` で指定
  - `chat_alias/chat_provider/chat_model`（例: Mio / local_openai / Chat）
  - `worker_alias/worker_provider/worker_model`（例: Shiro / local_openai / Worker）
  - `order1_alias/order1_provider/order1_model`（例: Aka / deepseek / deepseek-chat）
    - 旧: `coder_alias/coder_provider/coder_model`
  - `order2_alias/order2_provider/order2_model`（例: Ao / openai / gpt-4）
    - 旧: `coder2_alias/coder2_provider/coder2_model`
  - `order3_alias/order3_provider/order3_model`（例: Gin / anthropic / claude-sonnet-4-5）
    - 旧: `coder3_alias/coder3_provider/coder3_model`

### 1.2 エージェント責務（2026-03-01 更新）

#### Chat（Mio）
- **軽量な入力受付**: LINE/Slack からの入力を受け取り、JobID を付与
- **Worker へタスク委譲**: ルーティング・実行は Worker に任せる
- **最終決定**: Worker からの報告を受けて最終的な応答を決定
- **承認 UI**: 承認が必要な場合、LINE/Slack へ通知

#### Worker（Shiro）
- **ルーティング決定**: Router/Classifier を使用してタスクを分類
- **ループ制御**: 最大ループ数・時間制限の管理
- **Order への委譲**: 必要に応じて Order1/2/3 へタスクを振り分け
- **結果集約**: 各 Order からの提案を集約
- **Heartbeat 管理**: 全エージェントの状態を収集・監視
- **実行・道具係**: ファイル操作、コマンド実行、パッチ適用

#### Order1（Aka / DeepSeek）
- **低コスト・大量処理向け**: 比較的単純な作業を担当
- **提案生成**: plan/patch を生成（実行はしない）

#### Order2（Ao / OpenAI）
- **コーディング・実装の中核**: 実装タスクを担当
- **提案生成**: plan/patch を生成（実行はしない）

#### Order3（Gin / Anthropic Claude）
- **高品質推論・複雑なタスク**: 失敗コストが高いタスクを担当
- **提案生成 + 承認フロー**: plan/patch を生成し、承認を要求
- **将来の拡張**: SSHブリッジ経由でリモート環境の LLM を利用可能

**重要原則**:
- すべての作業・応答に **JobID を必須** とする
- Order（旧Coder）は原則として破壊的操作を**直接実行せず**、`plan` と `patch` を生成
- 実行は Worker が承認後に担当
