## 2. ルーティング決定仕様

現行のルーティング決定ownerは`internal/domain/agent.MioAgent.DecideAction`である。`MessageOrchestrator`と`DistributedOrchestrator`はMioへ判断を委譲し、決定後にroute別executorへ処理を渡す。

2026-03-01資料にある「Worker（Shiro）のRoutingModuleが決定する」は移行案の履歴であり、現行code contractとして使わない。

### 2.1 優先順位（固定）
1. 明示コマンド
2. ルール辞書（強証拠）
3. 分類器（1入力につき最大1回）
4. 安全側フォールバック

### 2.2 明示コマンド
- `/chat /plan /analyze /heavy /ops /research /wild /code /code1 /code2 /code3 /code4`
- 前後空白を除いた後の行頭で、command直後が文字列末尾または空白の場合だけ有効
- `/codebase`や`/code3extra`のようなprefix衝突は明示commandとして扱わない

### 2.3 ルール辞書
- 初期辞書は強証拠のみ
- 現行`RuleDictionary`にpriority fieldはなく、評価順は次の固定順とする
  1. Codex work path検出
  2. file参照を伴うedit request検出
  3. dictionary配列の定義順でkeyword部分一致
- match時はroute、confidence、matchedを返し、`DecisionEvidence`へsource、route、confidence、reasonを記録する
- miss時は分類器へ進み、分類器が利用不能、error、空route、低confidenceの場合は`CHAT`へ安全側fallbackする
- keywordまたは順序を変える場合は、先に`rule_dictionary_test.go`と`mio_test.go`へ競合・miss・fallbackのtestを追加する

### 2.4 分類器
- `Classifier` interfaceは`routing.Decision`を返す。現行`LLMClassifier`はprovider応答中の有効route tokenを長い名称から順に抽出する
- Mioが採用するのはrouteが空でなく、`confidence >= 0.7`のDecisionである
- provider error、空route、低confidence、classifier未設定は`CHAT`へ安全側fallbackし、各段階のevidenceを残す
- 現行実装にJSON固定schemaやCODE専用`0.8`gateはない。導入する場合はparser、validation、回帰testを先に追加する

### 2.5 分類器プロンプト管理
- 分類器system promptは1枚固定
- 更新はカテゴリ追加時のみ

### 2.6 channel入口

- Viewer、LINE等の通常messageはorchestratorが`task.Task`を生成し、Mioの同じ4段階判断へ渡す
- 登録済みslash commandや明示DCIは、通常route判断より前に専用handlerが処理する場合がある
- channel名だけを理由に`CHAT`へ固定しない
- Viewerの`to`は会話相手の選択であり、execution routeと混同しない

## 3. ループ制御と再ルート

### 3.1 停止条件
- `max_loops` 到達
- `max_millis` 到達
- worker失敗
- ユーザー確認が必要

### 3.2 ユーザー確認停止
- 停止理由を `need_user_confirmation` に統一
- 発火条件:
  - `risk=high`
  - 低confidenceかつ重大分岐
  - 追加情報不足

### 3.3 再ルート（最大1回）
- `fit=false` かつ `suggested_route` 有効
- または会話LLM提案IFを採択
- 同一入力で最大1回

### 3.4 会話LLM提案IF
```json
{"propose_next_loop":false,"route":"CHAT|PLAN|ANALYZE|OPS|RESEARCH|CODE|CODE1|CODE2|CODE3","reason":"short text","confidence":0.0}
```
