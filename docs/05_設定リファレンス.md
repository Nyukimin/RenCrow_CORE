# 設定リファレンス

## 読込方法

`rencrow` は `RENCROW_CONFIG` が設定されていればそのパスを、なければ作業ディレクトリの `./config.yaml` を読み込みます。公開テンプレートは `config/config.yaml.example` です。

```bash
cp config/config.yaml.example config.yaml
RENCROW_CONFIG=./config.yaml ./build/rencrow
```

YAML 内の `${NAME}` は同名の環境変数で展開されます。secret は YAML に直書きせず、環境変数を使用してください。

外部向け`view`／`live`／`lab`画面を`RenCrow_PORTAL`へ誘導する場合は、PORTALの公開URLを環境変数で指定します。

```bash
RENCROW_PORTAL_URL=http://127.0.0.1:18791 ./build/rencrow
```

未設定時は、CORE内の従来`mode=live|lab`画面を互換経路として維持します。通常のDebug Viewerはこの設定に関係なくCOREが提供します。

## 最小の考え方

`server.port` は 1-65535、`session.storage_dir` は必須です。LLM provider を有効にする場合は endpoint、必要な model、timeout、concurrency を設定します。任意機能は `enabled: true` にしたときだけ、その機能固有の path や endpoint が必須になります。

endpoint は `http://<host>:<port>` のように環境ごとに指定してください。repository 内の例や test にある host、port、path を production 値としてそのまま利用しないでください。

`local_llm`は既存開発環境の互換経路です。productionでは無効にし、`llm_gateway`を使います。互換経路の`chat_timeout_sec`と`chat_model_context`はChat/Mioだけの上書きで、`0`はそれぞれ`timeout_sec`、`model_context`を使います。

`mio.generation` は Mio の通常会話と input_audio 直結経路で共有する生成設定です。`stream`、`max_tokens`、`temperature`、`top_p`、`top_k`、`min_p`、`seed` を指定できます。`chat_template_kwargs.enable_thinking`は互換性のため読み込みますが、`Chat`契約で常に`false`です。省略時もCOREが`think=false`と併せて送信し、`true`は設定validation errorになります。GPT-OSS-120Bの`ChatWorker`は別のprovider policyでreasoning effortを`low`へ固定します。`Worker`には固定を適用せず、Ollama互換経路でも未指定ならbackend既定、明示指定ならその値を維持します。`stream: true` は両方の経路で OpenAI 互換 streaming response を使います。WAV は `input_audio` part として送り、`mio.input_audio.prompt` を text part として同じ user message に付与します。

### RenCrow_LLM Gateway

productionでは`llm_gateway.enabled: true`にし、COREからRenCrow_LLMの単一URLへ接続します。COREが送るmodelは`mio`、`shiro`、`midori`、`kuro`などの論理Agent IDです。物理Model名、Backend URL、起動port、Codex commandはRenCrow_LLM側で管理します。Gateway障害時にCOREから物理targetへfallbackしません。

Kuroの通常会話をCodexで実行する設定はRenCrow_LLMの`agents.kuro` mappingで行います。COREの`codex.enabled`は`codex.run` Tool／Advisor用であり、通常会話経路とは独立しています。

```yaml
llm_gateway:
  enabled: true
  base_url: "http://127.0.0.1:8090"
  api_key_env: ""
  timeout_sec: 600
```

### IdleChat日次情報収集

COREはIdleChat用の外部シードをJSTの毎日04:00に強制更新します。CORE起動時は当日cacheがなければ初期収集し、同日cacheがあれば再利用します。04:00の定期更新は同日cacheの有無にかかわらず実行します。このcacheはIdleChatのお題候補専用であり、一般ニュースDBや利用者別PUSHの正本ではありません。

arXivは`cs.AI`、`cs.LG`、`cs.CL`、`cs.CV`、`cs.RO`を対象にし、基盤モデルだけでなく、機械学習、自然言語、画像・マルチモーダル、ロボティクスまで収集します。RSS／Atom全体は最大64件、Reddit／Xとの統合後は最大80件です。

利用toolと処理境界は次の通りです。

| tool／方式 | 用途 |
| --- | --- |
| Go `time.Timer` | JST 04:00の次回実行時刻を計算し、CORE停止contextで終了する |
| Go `net/http` | 10秒timeoutとRenCrow User-AgentでHTTP取得する |
| Go `encoding/xml` | RSS 2.0とAtomのtitle、link、description／summaryを解析する |
| Go `encoding/json` | Wikipedia MediaWiki Random APIとX APIの応答を解析する |
| Wikipedia Random API | Newsとは別の外部刺激シードを最大10件取得する |
| Reddit public Atom | communityの話題候補を取得する |
| X Recent Search API | 明示的に有効化した場合だけSNS話題候補を取得する |
| `web_gather.fetch` | 収集済みの特定URLと、不明語検索で選んだ確認先URLの本文を直接取得・抽出する |
| `web_search` | 元記事本文だけでは意味を説明できない用語の確認先URL発見に限って使う。記事内容の代替取得には使わない |
| Worker LLM | 取得済み原文の日本語訳を作り、用語補足を確定した後にサマリとShiroの見解を日本語で作成する。HTTP取得や検索自体には使用しない。ChatWorkerはチャット画面専用 |

`idle_chat.news_sources.reddit`は既定で有効で、`communities`を1つの公開Atom feedへまとめてhot投稿を取得します。`x`は既定で無効です。有効化する場合は`bearer_token_env`でtokenを保持する環境変数名を指定し、`queries`へ公式Recent Searchのquery、category、10から100件のlimitを設定します。token本体をYAMLへ保存しません。個別sourceの失敗、429、認証エラーは当該sourceだけをdegradedとして記録し、他のRSS／Atom、Wikipedia、SNS sourceの成功結果を破棄しません。

毎日04:00 JSTの収集成功後、Skill `core.build-daily-source-brief`を`Worker`で1件ずつ実行します。各記事は「特定URL本文の直接取得 → 原文翻訳 → 用語抽出 → 本文で説明できない用語だけ検索 → 検索先URL本文の直接取得 → 用語補足確定 → サマリ → Shiroの見解」を完了してcacheへ反映してから次の記事へ進みます。ユーザー会話または明示Worker実行が始まった場合は日次LLM requestをcancelし、前景処理終了後に同じ記事を入力・contextを短縮せず再試行します。この中断は失敗回数へ含めず、完了済み記事を保持します。利用者への出力順は「原文翻訳 → サマリ → Shiroの見解 → 用語補足」で、本文はすべて日本語とします。原文入力は最大12000文字、翻訳出力は最大16384 token、記事単位timeoutは10分です。元記事本文を取得できない場合は見出しやフィードsummaryで補完しません。LLM未設定、timeout、形式不正、一部記事失敗では`enrichment_status`を`fallback`または`partial`にし、推測せず確認不能を明示します。

現在のプロセス内cacheと解決済み取得先はDebug Viewerの`情報収集`タブ、または`GET /viewer/idlechat/collection`で確認できます。応答には使用Skill、取得項目のtitle、category、source、source type、元URL、本文取得状態、原文翻訳、サマリ、見解、用語補足と、credentialを除いたsource設定、利用toolを含みます。このGETは観測専用であり、収集jobを起動しません。

`conversation`の永続化設定は次を使用します。

| key | 用途 |
| --- | --- |
| `redis_url` | active threadを保持するRedis接続先 |
| `l1_sqlite_path` | L1 hot storeのSQLiteファイル |
| `archive_sqlite_path` | L2 archiveのSQLiteファイル |
| `vectordb_url` | 意味検索を行うQdrant接続先 |
| `profile_promotion_enabled` | L1 Rawからの非同期UserMemory候補抽出。既定`true` |
| `profile_promotion_idle_grace_seconds` | 全LLM利用数が0になってから処理開始までの継続Idle時間。既定`10` |
| `profile_promotion_timeout_seconds` | ProfileExtractor 1回の上限時間。既定`45` |
| `profile_promotion_batch_messages` | 1回に読む未処理ユーザー発話数。既定`24`。超過分を破棄せず次回batchへ送る |
| `profile_promotion_max_attempts` | LLM、parse、L1保存失敗の最大試行回数。既定`5`。busy/cancelは数えない |

会話要約と非同期ProfileExtractorのtext providerは、productionで組み立て済みのprimary Workerがある場合、`local_llm.enabled`に関係なくRenCrow_LLM GatewayのWorker経路を使用します。旧`ollama`設定はprimary Workerがないlegacy／開発構成の互換経路に限り、production会話処理から直接参照しません。`conversation.embed_model`が空の場合はvector embeddingと意味検索だけを無効化し、Redisのactive thread、L1 SQLiteのraw message、L2 SQLite archiveは維持します。

active threadが12メッセージへ到達した場合、次のactive thread作成と最新メッセージ保存を同期完了してから、旧threadのLLM要約を45秒上限のbackground処理でL2へarchiveします。応答完了はWorker要約を待ちません。Workerが時間内に応答しない場合は、期限切れcontextを保存処理へ再利用せず、新しい5秒contextで決定論的な簡易要約をL2へ保存します。

非同期ProfileExtractorは`llm_busy`の`active_count=0`が`profile_promotion_idle_grace_seconds`継続した場合だけ低優先度で開始します。この判定にはIdleChatを含む全provider sourceを含めます。処理開始後に通常LLM要求が来た場合はcancelし、busy/cancelを抽出失敗として数えず次回Idleへ延期します。`profile_promotion_batch_messages`は1回の処理量だけを制限し、入力Rawの削除や要約による代替を許可しません。

旧`duckdb_path`は既存Configを読み取るためだけの非推奨キーです。値が`.duckdb`の場合、元ファイルを変更せず、同じディレクトリの`*_archive.db`を新しいSQLite archiveとして使用します。新規Configには`archive_sqlite_path`を記載してください。

### TTS発音日次チェック

`tts.pronunciation_check`はCORE Schedulerが所有する運用タスクです。`schedule`はUTCの5フィールドcronで、`cron 30 19 * * *`はJST 4:30です。実行前に`gpu_match`で選んだGPUを`idle_samples`回確認し、空きVRAMが`min_free_mb`以上、利用率が`max_utilization_percent`以下のときだけRenCrow_TTSのTool APIを呼びます。条件を満たさない場合は失敗にせず、`retry_interval_seconds`後へ延期します。

## Secret とローカルデータ

- API key、token、private key は環境変数または外部 secret store に置く。
- `config.yaml`、runtime DB、log、cache、generated artifact は commit しない。
- 外部 bind を行う場合は TLS、認証、network allowlist を別途設定する。
- `LLM_OPS_TOKEN` など管理 API 用 token を inference 用 credential と混用しない。
