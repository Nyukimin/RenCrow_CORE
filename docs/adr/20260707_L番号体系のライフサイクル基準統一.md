# L番号体系のライフサイクル基準統一

**日付**: 2026-07-07  
**ステータス**: 採用  
**対象**: Memory lifecycle / Recall Context / 記憶システム関連ドキュメント

## 決定

記憶システムの L0〜L4 は、保存媒体ではなく lifecycle 上の位置で定義する。

正本定義は `docs/refs/01_正本仕様/18_Memory_Lifecycle_Recall_Context.md` とし、物理ストアおよびコード上の歴史的命名との対応表も同文書に集約する。

## 根拠

- `18_Memory_Lifecycle_Recall_Context.md` は 2026-06-19 時点の最新正本であり、「保存媒体ではなく lifecycle 上の位置により L0〜L4 を定義する」と明示している。
- 現行コードには `L1SQLiteStore`、`MemoryLayerL1`、`L1StagingItem` など物理ストア由来の命名が残るが、同じ SQLite store が hot memory、candidate / staging、confirmed metadata、pinned metadata を扱うため、保存媒体だけでは lifecycle 上の位置を表現できない。
- DuckDB と VectorDB も、thread summary、archive、validated knowledge、canonical knowledge index など複数 lifecycle の保存先になり得る。
- RecallPack は prompt 注入単位として L0 / L1 / L2 / L3 / L4 ラベルを使うため、保存媒体ではなく prompt 注入・昇格・減衰・retention と整合する定義が必要である。

## 却下した代替案

物理ストア基準の L番号体系を正本にする案を却下した。

旧文書には Redis=L0、SQLite=L1、DuckDB=L2、Qdrant=L3 とする記述がある。しかし、現行実装では SQLite が L1 以外の lifecycle 状態も保持し、DuckDB / VectorDB も複数 lifecycle の保存先となる。そのため、物理ストア基準は実装実態と今後の promotion / decay / forgetting の設計に合わない。

## 影響範囲

- L番号の意味を説明する文書は、`18_Memory_Lifecycle_Recall_Context.md` の定義と対応表を参照する。
- `L1SQLiteStore` などのコード上の命名は互換名として扱い、今回の Phase 0 では変更しない。
- 旧文書や履歴文書の物理ストア基準 L番号は、旧定義または実装命名として扱う。
- docs/archive 配下は今回の変更対象外とする。
