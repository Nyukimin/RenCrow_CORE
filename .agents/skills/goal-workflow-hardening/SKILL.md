# Goal Workflow Hardening

## 目的

ユーザー提案から作った仕様 / 実装仕様 / 作業手順書を、Goal に設定しても長時間の差分混在や実装同士の潰し合いが起きにくい形へ整える。

この skill は実装そのものではなく、**作業手順書を Goal 実行向けに強化する**ために使う。

## 入力として読むもの

必要な範囲だけ読む。

1. ユーザーの提案または元プロンプト
2. その提案から作られた仕様書
3. 仕様から作られた実装仕様書
4. Goal に設定する予定の作業手順書
5. 関連する repo ルール:
   - `AGENTS.md`
   - `docs/01_正本仕様/実装仕様.md`
   - 必要に応じて `rules/common/rules_testing.md`
   - 必要に応じて `rules/common/rules_observation_verification.md`

## 必ず入れる要素

作業手順書には、次を明示する。

- 作業は小さな検証済み commit 単位で進める。
- 各単位で、対象・変更範囲・検証コマンド・完了条件を定義してから実装する。
- 実装後に該当テストと必要な runtime / Viewer 確認を行う。
- 確認済みの関連ファイルだけを選択的に stage する。
- 日本語 commit message で commit する。
- commit 後は push する。
- push できたら、不要なユーザー確認を待たず次の未完了項目へ進む。
- blocked / skipped / fail を成功扱いしない。
- live E2E 生成物や証跡ファイルを明示指示なく commit しない。

## 停止条件

次の条件に当たる場合だけ、作業を止めて報告または質問する。

- テスト失敗や runtime 確認失敗が、現在の作業範囲内で短時間に解消できない。
- 変更が複数領域へ広がり、1 commit の責務が曖昧になった。
- 外部 secret / 実機 / 外部 API / 複数マシンなど、作業者側で準備できない依存が必要。
- destructive operation、依存追加、CI/deploy 設定変更、ファイル削除、セーフガード変更が必要。
- push 対象に未確認差分、別責務の差分、または live E2E 生成物が混ざりそうになった。
- Goal の達成条件が、元のユーザー提案または正本仕様と矛盾している。

## 禁止事項

- worktree 全体を一括 stage / commit しない。
- 落ちているテストを無視して push しない。
- blocked 項目を pass にしない。
- Viewer / domain / persistence / runtime / docs を、責務が曖昧なまま 1 commit に混ぜない。
- docs だけを更新して、実装や runtime の未確認を完了扱いしない。
- 「代表例」だけで複数の観測済み問題を完了扱いしない。

## 作業手順書へ挿入する標準プロンプト

必要なら、作業手順書の「作業ルール」章へ次を追加する。

```text
この作業手順書に沿って、未完了項目を小さな検証済み commit 単位で順に処理してください。

各単位で、対象・変更範囲・検証コマンド・完了条件を自分で定義してから実装してください。
実装後、該当テストと必要な runtime / Viewer 確認を行ってください。
確認済みの関連ファイルだけを選択的に stage し、日本語 commit message で commit してください。
commit 後は push してください。
push できたら、ユーザー確認を待たずに次の未完了項目へ進んでください。

ただし、次の場合だけ停止して報告してください。
- テスト失敗や runtime 確認失敗が、現在の作業範囲内で短時間に解消できない。
- 変更が複数領域に広がり、1 commit の責務が曖昧になった。
- 外部 secret / 実機 / 外部 API / 複数マシンなど、こちらで準備できない依存が必要。
- destructive operation、依存追加、CI/deploy 設定変更、ファイル削除、セーフガード変更が必要。
- push 対象に未確認差分や live E2E 生成物が混ざりそうになった。

禁止:
- worktree 全体を一括 stage / commit しない。
- `vault/` や live E2E 生成物を明示指示なく commit しない。
- 失敗したテストを無視して push しない。
- blocked 項目を成功扱いしない。
- Viewer / domain / persistence / runtime / docs を、責務が曖昧なまま 1 commit に混ぜない。

報告:
- 各 push 後に、commit hash、検証コマンド、次に進む対象を短く報告してください。
- ユーザー確認は不要です。止まる条件に当たった場合だけ質問または報告して停止してください。
```

## 変換チェックリスト

作業手順書を編集したら、以下を確認する。

- Goal 用プロンプトが、不要な確認待ちではなく自律継続を前提にしている。
- 停止条件が具体的で、外部依存・危険操作・テスト失敗・差分混在を含む。
- push 可能単位が「検証済み commit」で定義されている。
- blocked / skipped / fail の扱いが成功と分離されている。
- live E2E 証跡、`vault/`、一時生成物の commit 禁止が明記されている。
- repo の TDD、観測検証、commit message 日本語ルールと矛盾していない。

## 出力

編集後は、次だけを簡潔に報告する。

- 更新した手順書の path
- 追加した Goal 用ルールの要点
- Skill を追加または更新した場合は、その path
- 手順書が Goal に設定されたときの自律進行条件と停止条件
