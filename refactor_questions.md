## カテゴリ3: 部分実装の完成優先度

### Q4: 部分実装項目の優先度

**背景**:
`docs/01_正本仕様/12_実装状況_20260505.md` §3 に列挙された「部分実装」項目が多数ある：

1. 記憶システムの store 責務整理（Redis/DuckDB/VectorDB の3層構成）
2. Knowledge DB の高度化（GitHub/Hugging Face/MediaWiki API）
3. News の Recall trace 利用履歴
4. 6エージェントのキャラ会話ランタイム
5. SQLite L1 hot store の高度化
6. その他

**質問**:
1. このリファクタリングで完成させるべき部分実装項目はどれか？（優先度順）
　　　⇒実装仕様に引き上げる
2. どの項目は別タスク（次のマイルストーン）に回すか？
　　　⇒日常会話の記憶システムは今回終わらせて、他は別途実装仕様を作成
3. 各項目の「完成」の定義は？（どこまで実装すれば完成とするか）
　　　⇒E2Eテスト完了

**判断基準**:
- ユーザー影響が大きい項目 → 優先度高
- 他の機能に依存されている項目 → 優先度高
- 独立した機能で、影響範囲が限定的 → 別タスクに回してもよい

**提案**（実装担当モデルからの提案）:
以下の優先順位を提案します（人間の承認が必要）：

**Phase 3 で実施する項目**（優先度高）:
1. 記憶システムの store 責務整理 → 多くの機能が依存
2. Knowledge DB の高度化 → Source Registry との統合に必要

**別タスクに回す項目**（優先度中）:
3. News の Recall trace 利用履歴 → UI 改善、機能的には動作中
4. SQLite L1 hot store の高度化 → パフォーマンス改善、既存機能は動作中

**このリファクタリングでは触れない項目**（優先度低または大規模変更）:
5. 6エージェントのキャラ会話ランタイム → アーキテクチャ変更が必要、別プロジェクト

## カテゴリ5: データ互換性とマイグレーション

### Q7: 既存ユーザーの設定ファイル・データベースへの影響

**背景**:
- Deprecated 設定フィールドの削除、Config ファイルの分割を予定
- SQLite/DuckDB スキーマの変更が発生する可能性
- 既存ユーザーのデータが壊れないようにする必要がある

**質問**:
1. 既存ユーザーの設定ファイル（`config.yaml`）を自動マイグレーションするか？
　　⇒意味が分かりません
2. データベーススキーマを変更する場合、マイグレーションスクリプトを提供するか？
3. 後方互換性をどこまで維持するか？（1バージョン前まで / 2バージョン前まで）
　　⇒後方互換性は不要です。代わりに、過去のデータを変換してください

**判断基準**:
- 既存ユーザー数が多い → 自動マイグレーション必須
- 既存ユーザー数が少ない（開発者のみ） → ドキュメントによる手動マイグレーションでも可

**提案**（実装担当モデルからの提案）:

- 次のメジャーバージョン（v4.0）で削除予定とマーク
- `docs/migration/v3_config_migration.md` に migration guide を記載
