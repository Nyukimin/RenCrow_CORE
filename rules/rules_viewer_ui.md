# rules_viewer_ui.md - RenCrow Viewer UI 追加ルール

**目的**: RenCrow の新 UI / 新 Viewer タブを、日常的に使える読みやすい作業机として設計するためのルール
**対象**: Viewer の HTML / CSS / JavaScript、常用タブ、既存タブの新 UI 化
**位置づけ**: `rules/common/rules_frontend.md` を RenCrow Viewer 向けに具体化するプロジェクト固有ルール

関連文書:

- 視覚方針の正本: `DESIGN.md`
- Viewer 仕様: `docs/09_Viewer/Viewer仕様.md`
- 新 UI タブ生成プロンプト: `docs/09_Viewer/Viewer新UIタブ生成プロンプト.md`
- Viewer 文書入口: `docs/09_Viewer/README.md`

---

## 1. 基本方針

RenCrow の新 UI および新 Viewer タブは、監視ダッシュボード風の情報過多な画面ではなく、「日常的に使える、読みやすい、AI キャラクターが横にいる作業机」として設計する。

見た目、色、余白、密度、カード、ボタン、レスポンシブ、Do/Don't の判断は `DESIGN.md` を視覚方針の正本として扱う。このファイルは Viewer 実務ルール、`DESIGN.md` はデザイン言語という分担にする。

特に、れんさんが常用する次の画面では、実用性と読みやすさを優先する。

- Home / Daily Desk
- Chat
- Develop
- Instructions
- Reports

既存の Ops / Overview / System / Jobs のような監視・詳細確認タブは残してよい。ただし、常用 UI に内部監視情報をそのまま持ち込まない。

Ops / System / Jobs など監視系タブ自体を変更する場合でも、初期表示は 3 から 5 個程度の要約ブロックに絞る。監査ログ、生テーブル、詳細ステータス、全文エラーは初期表示せず、`details`、accordion、drawer、別タブなどへ分離する。初期表示で「まず何を見るべきか」が分からない画面は不合格とする。

---

## 2. 適用対象

このルールは、次の作成・変更時に必ず適用する。

- 新しい Viewer タブを追加するとき
- 既存 Viewer タブを新 UI へ置き換えるとき
- Viewer の HTML / CSS / JavaScript で新画面を作るとき
- Home / Chat / Develop / Instructions / Reports 系の画面を設計するとき
- モックではなく、実際に使う HTML を生成するとき

---

## 3. 最重要ルール

### 3.1 大きなキャラクター表示

新 UI では、キャラクターを小さなアイコンではなく、存在感のある表示にする。

必須:

- 画面の左側または左上に大きく表示する
- キャラクター名と役割を明示する
- 状態が分かるようにする
  - 待機中
  - 作業中
  - 確認待ち
  - エラー
- 装飾ではなく「誰が対話相手か」が分かる UI にする

避けること:

- 小さな丸アイコンだけで済ませる
- キャラクターを完全に隠す
- 画面の主役がログ表だけになる

### 3.2 大きな会話カード / 情報カード

会話、説明、報告、作業状態は、小さなログ行ではなく、読みやすい大きめのカードで表示する。

必須:

- 1 発話または 1 情報単位ごとに余白のあるカード表示にする
- 文字サイズを十分大きくする
- 視線移動が少なく読める構成にする
- 会話と説明文を「ログ一覧」ではなく「読む UI」として扱う

推奨:

- assistant / user のカードを明確に分ける
- 会話カードの横に小さなアバターを置く
- 箇条書きや短い要点が見やすいことを優先する

避けること:

- 1 画面に大量の小さい吹き出しを並べる
- 会話を表のように圧縮する
- 文字が小さく読みにくいカードにする

### 3.3 大きな入力欄または明確な次アクション領域

入力を主目的にする画面では、入力欄を補助 UI ではなく主役の 1 つとして扱う。

常用入力画面で必須:

- 入力欄は横幅・高さともに十分確保する
- Chat の最下行入力欄は常時画面に表示する
- Chat の最下行入力欄は縦に大きくしすぎない
- 音声入力ボタン、送信ボタンは押しやすく大きくする
- 入力欄のラベルや placeholder を明確にする
- 「ここから何を始めるか」が直感的に分かる

Reports のような閲覧主目的画面では、入力欄の代わりに次アクション領域を明確にしてよい。

- Copy Summary
- Markdown Export
- Create Follow-up Instruction
- Chat で相談
- Instructions へ送る

推奨:

- 複数行入力しやすい高さにする
- 送信先の切替を入力欄の近くに置く
- 添付やモード切替は、主入力動線を邪魔しない位置に置く

避けること:

- 細い 1 行入力欄だけにする
- チャット入力欄を画面外スクロールしないと触れない位置に置く
- チャット入力欄を縦に広げすぎて、会話カードの表示領域を圧迫する
- ボタンが小さく押しにくい
- 入力欄または次アクションの存在感が弱い

### 3.4 少ない情報量

新 UI では、情報を「見せられるだけ全部見せる」のではなく、その画面で本当に必要な情報だけを出す。

必須:

- 1 画面 1 目的を守る
- 不要なサイドパネルや小カードを常時表示しない
- 専用タブがある情報は、その専用タブへ逃がし、常用画面の右サイドへ重複表示しない
- 詳細情報は折りたたみ、別タブ、詳細画面へ分離する
- まず見るべき情報が一目で分かる構成にする

推奨:

- 主要情報は 3 から 5 ブロック程度に絞る
- 二次情報は `details`、accordion、modal、side drawer に分離する
- ログ全文より要約を優先する

避けること:

- 開発者向けの内部情報を常時表示する
- 小さなカードを何枚も並べる
- 右にも左にも下にも情報を詰め込む
- Memory / Source Registry / Recall Trace など専用タブがある情報を、常用画面のサイド棚として常時表示する

### 3.5 読みやすい文字

文字サイズと可読性を優先する。

必須:

- 本文は十分な大きさにする
- 文字色と背景色のコントラストを確保する
- 見出し、本文、補足の階層を明確にする
- 長文は行間をしっかり取る

推奨サイズ:

- 画面タイトル: 20px から 22px
- セクション見出し: 15px から 18px
- 本文: 12px から 14px
- 補足: 10px から 12px
- 小さなメタ情報: 10px を下限とする
- 入力欄: 13px から 14px

避けること:

- 9px 以下の小さな文字を多用する
- 細すぎるフォントを多用する
- 文字量を詰め込んで圧縮する

### 3.6 夜の作業机の雰囲気

新 UI の雰囲気は、「夜の作業机」「AI と一緒に作業する部屋」に寄せる。

必須:

- ダークテーマベース
- 深い青、紺、黒寄りの背景
- ほんのり光るアクセント
- 落ち着いた、長時間見られる画面にする

推奨:

- ガラス風または半透明パネル
- 柔らかいグロー
- 暗めの背景と明るすぎないアクセント
- キャラクター背景にも夜景・作業机感を持たせる

避けること:

- 明るすぎる白背景
- ビジネスダッシュボードのような無機質な表画面
- ネオン過剰で読みにくいデザイン
- SF 計器盤のように情報を詰め込むこと

## 4. 画面別の情報境界

### 4.1 Home / Daily Desk

出してよいもの:

- 今日の状態
- 最後の会話
- 進行中の作業
- 未読レポート
- 未完了指示
- 呼び出せるキャラクター
- 大きな入力欄

出しすぎてはいけないもの:

- 詳細ログ全文
- 開発者向けイベント一覧
- 技術監視の細かな内部状態

### 4.2 Develop

出してよいもの:

- 現在の開発タスク
- 指示内容
- 担当 Agent
- 進捗フェーズ
- 作業ログの要約
- 差分・成果物
- 確認待ち
- エラーと再試行
- 次に押すボタン

出しすぎてはいけないもの:

- 生のイベントログ全文
- 関係ないセッション一覧
- 監視用の細かすぎるメトリクス

### 4.3 Instructions

出してよいもの:

- 未完了指示
- 進行中指示
- 保留中指示
- 完了済み指示
- 優先度
- 対象 Agent
- 期限またはタイミング
- 再指示ボタン
- キャンセル
- 指示から生成された Job

出しすぎてはいけないもの:

- 指示と無関係な長文ログ
- 技術詳細の羅列

### 4.4 Reports

出してよいもの:

- 作業完了報告
- 要約
- 何を変更したか
- 何を確認したか
- 検証結果
- 失敗したこと
- 未確認のこと
- Evidence
- 成果物リンク
- 次の判断
- Copy Summary
- Markdown Export

出しすぎてはいけないもの:

- 生ログ全文
- 人が読みにくい JSON の羅列を初期表示すること

## 5. 実装ルール

### 5.1 HTML 構造

新タブの HTML は、次の思想で組む。

- 視覚的主役を明確にする
- header / main / aside / footer の役割を整理する
- 主要コンテンツから補助情報の順に並べる
- 詳細は折りたたみ可能にする

推奨構造:

- page shell
- header
- character panel
- main content area
- summary cards
- input area または next action area
- optional detail drawer

### 5.2 CSS

必須:

- 大きめの角丸
- 余白多め
- 透明感のある暗色パネル
- 文字サイズ大きめ
- hover / active / focus が見やすい
- スマートだが派手すぎない配色
- 固定ヘッダー、固定フッター、固定入力バー、固定ツールバーに隠れない余白を確保する
- スクロール領域の末尾には、固定 UI の高さ以上の `padding-bottom` または同等の逃げを持たせる
- 縦タブ、横タブ、カード一覧、詳細ペインなどのスクロール可能領域は、最後の要素までクリック可能にする

避けること:

- 小さな文字前提の詰め込みレイアウト
- カラムを増やしすぎること
- 罫線だらけの画面
- `overflow:auto` を指定しただけで到達性を確認した扱いにすること
- スクロールできるが、最後の要素が固定 UI に重なってクリックできない状態
- `z-index` の都合で見えている要素が別要素にクリックを奪われる状態

### 5.3 JavaScript

必須:

- まず静的に読みやすい画面を作る
- その後、必要なデータだけ動的差し込みする
- 要約を優先し、ログ全文は折りたたみで表示する
- 画面ごとの責務を分ける

避けること:

- 1 画面に全 API レスポンスをそのまま注入する
- UI 上で未加工ログを大量表示する

### 5.4 操作可能性と重なり防止

新 UI では、見えているだけでは合格にしない。すべての主要操作対象が実際にクリック、入力、選択、コピー、スクロールで到達できることを確認する。

RenCrow Viewer では、ブラウザ実機確認環境を必須とする。
Playwright Chromium、通常の Chrome / Chromium、または同等の実ブラウザ環境で、描画、console、network、実クリックを確認できない場合は、Viewer UI の変更を完了扱いしない。

必須:

- すべてのタブ、主要ボタン、入力欄、select、リンク、カード操作を実クリックで確認する
- 最上部と最下部の操作要素が、固定ヘッダー、固定フッター、固定入力バー、toast、drawer に隠れないことを確認する
- スクロール後の最後の要素が、固定 UI の下に潜らずクリックできることを確認する
- desktop と narrow viewport の両方で到達性を確認する
- 1920x1080、1366x768 に加えて、可能なら 390x844 前後の narrow / mobile 幅でも確認する
- narrow / mobile 幅では、主要情報が縦積みになり、長文エラーや URL がカードを押し広げないことを確認する
- PC の縦タブと mobile panel select の両方で、追加した画面へ移動できることを確認する
- `elementFromPoint`、Playwright の実クリック、または同等の確認で、クリック位置の最前面要素が意図した対象であることを確認する
- modal、drawer、details、accordion を開閉した後も、背後や下部の重要操作が永久に塞がれないことを確認する
- live-mode、lipsync、固定入力バー、toast、overlay などクリック干渉しやすい UI は、見た目だけでなく computed style を確認する
- computed style では特に `pointer-events`、`z-index`、`position`、`background`、`border`、`box-shadow`、`backdrop-filter` を確認し、会話、入力、主要操作対象を妨げないことを検証する

避けること:

- DOM に存在するだけで操作可能と判断すること
- 目視で見えているだけでクリック可能と判断すること
- スクロールバーがあるだけで到達可能と判断すること
- 主要導線を `force click` 前提にすること
- hover しないと重要操作が出ないのに、touch / mobile の代替導線がない状態

確認観点:

- 新規タブをすべてクリックできる
- 既存タブもすべてクリックできる
- 画面内の最下部ボタンをクリックできる
- 入力欄にフォーカスでき、入力できる
- select / menu を開ける
- copy / export / send などの主要操作が実行できる
- fixed / sticky 要素がクリックを奪っていない

## 6. 禁止事項

新 UI では次を禁止する。

- 情報量過多な監視ダッシュボードをそのまま常用 UI にすること
- 小さなカードを大量に並べること
- 12px 級の小さな文字を多用すること
- 会話、作業、レポート、監視ログを 1 画面に全部入れること
- キャラクターを飾りにしてしまうこと
- 入力欄または次アクション領域を小さく扱うこと
- 必要な情報より内部ログを目立たせること
- 固定 UI によってタブ、ボタン、入力欄、カード操作が隠れること
- 見えているのにクリックできない要素を残すこと
- スクロール末尾の要素が固定入力バーや footer の下に潜ること
- `z-index` と `position: fixed` / `sticky` の影響を確認せずに完了扱いすること
- 専用タブがある情報を、右側の常時表示パネルとして重複させること

## 7. 受け入れ基準

新タブは、以下を満たしたときに合格とする。

- 大きなキャラクター表示、または画面の対話相手が分かる強い視覚要素がある
- 大きな会話カード、または大きな情報カードがある
- 入力主目的の画面では大きな入力欄がある
- 閲覧主目的の画面では明確な次アクション領域がある
- 情報量が絞られている
- 文字が読みやすい
- 夜の作業机の雰囲気がある
- その画面に必要な情報だけが出ている
- 初見で「何をする画面か」が分かる
- すべての主要操作対象が実クリックできる
- スクロール可能領域の先頭から末尾まで、固定 UI に隠れず到達できる
- desktop と narrow viewport の両方で主要導線が成立する
- 既存タブを含むナビゲーションが固定入力バーや footer に遮られない
- DOM 存在チェックだけでなく、Playwright または同等の操作確認を通している
- ブラウザ実機確認環境が利用可能で、対象画面を実ブラウザで確認している

---

## 8. 実装時の優先順位

新タブを作るときの優先順位は以下。

1. 読みやすさ
2. 情報の少なさ
3. 主目的の明確さ
4. キャラクターの存在感
5. 入力しやすさ、または次アクションの分かりやすさ
6. 雰囲気の統一
7. 詳細情報の後回し

「機能を全部見せる」より、「迷わず使える」ことを優先する。

---

## 9. 生成指示

RenCrow が新しい HTML タブを生成するときは、必ず次を守る。

- このルールを満たす見た目で HTML / CSS / JavaScript を作る
- 既存機能を無理に全部載せない
- まずは必要最小限で成立する画面を作る
- 必要なら詳細画面、折りたたみ、別タブへ分離する
- 実装対象画面が Home / Chat / Develop / Instructions / Reports のどれかを明示する
- コード生成時は「どの要素を残し、どの要素を隠すか」を説明できるようにする
- 生成後は、追加タブだけでなく既存タブも実クリックして、固定 UI による遮蔽がないことを確認する
- スクロール末尾のボタン、カード、リンク、入力欄がクリックできることを確認する

## 10. 画像参照ありの新 UI 作成プロンプト方針

添付画像を参照して Viewer 新 UI を作る場合、画像は雰囲気と構成の参照として扱う。画像をそのまま模写しない。

目的は、れんさんが日常的に使うための「大きく、読みやすく、迷わない、AI キャラクターが横にいる作業机」を作ることである。

### 10.1 画像参照で再現するもの

- ダークネイビー系の背景
- 透明感のあるガラス風パネル
- 青から紫の控えめなグロー
- 大きな角丸カード
- 左側の大きなキャラクター領域
- 中央の大きな会話カードまたは情報カード
- 下部の大きな入力欄、または閲覧画面では明確な次アクション領域
- 余白の多い構成
- 読みやすい大きな文字

### 10.2 画像参照で再現しないもの

- 画像そのものの完全コピー
- 使わない装飾の追加
- 細かい監視パネルの追加
- ログ、DB、GPU、イベント、内部ステータスの常時表示
- 白背景や業務ダッシュボード風の見た目
- 情報量を増やして画像の余白感を壊すこと

### 10.3 画面構成の基本

1920x1080 を基準にしつつ、1366x768 でも破綻しない構成にする。

基本構造:

```text
┌──────────────────────────────────────────────┐
│ Header: RenCrow / tabs                         │
├──────────────┬───────────────────────────────┤
│ Character    │ Main Content                    │
│ Panel        │ 大きなカード表示                 │
│              │                                 │
│              │                                 │
├──────────────┴───────────────────────────────┤
│ Input Dock または Next Action Dock             │
└──────────────────────────────────────────────┘
```

左側 Character Panel:

- 幅 500px から 560px 程度を目安にする
- RenCrow ロゴ
- 大きなキャラクター画像領域
- キャラクター名
- 役割
- 現在状態
- 呼び出せるキャラクターカード

中央 Main Content:

- そのタブの主目的だけを表示する
- Chat なら大きな会話カードを主役にする
- Home なら今日の状態、最後の会話、進行中作業、未読レポート、未完了指示に絞る
- Develop なら現在の開発タスク、担当 Agent、進捗、確認待ち、次操作に絞る
- Instructions なら指示キューと状態に絞る
- Reports なら作業完了報告、確認内容、失敗、未確認、次の判断に絞る

下部 Dock:

- 入力画面では大きな入力欄を置く
- Chat では最下行の入力欄を常時表示する
- Chat の入力欄は 1 から 2 行程度の高さを基本にし、無駄に縦へ伸ばさない
- 閲覧画面では Copy / Export / Follow-up / Chat 相談などの次アクションを置く
- 固定配置にする場合は、主要スクロール領域の末尾が Dock に隠れない余白を必ず確保する

### 10.4 CSS トークン

画像参照ありの新 UI では、次のトークンを基準にしてよい。

```css
:root {
  --rc-bg: #07101f;
  --rc-bg-2: #0b1324;
  --rc-panel: rgba(18, 28, 48, 0.86);
  --rc-panel-strong: rgba(30, 48, 86, 0.82);
  --rc-card: rgba(22, 33, 56, 0.88);
  --rc-border: rgba(130, 160, 230, 0.24);
  --rc-border-strong: rgba(135, 165, 255, 0.48);
```

### 10.5 文字サイズ

必ず次を守る。

- 画面タイトル: 20px から 22px
- セクション見出し: 15px から 18px
- 本文: 12px から 14px
- 補足: 10px から 12px
- 小さなメタ情報: 10px を下限
- 入力欄: 13px から 14px
- タブ、ピル、短いメタ表示: 10px 以上

### 10.6 レイアウト制約

- 主要領域に十分な余白を取る
- 左キャラクターパネルは固定幅を基本にする
- メイン領域は広く取る
- 1 画面に詰め込まない
- 詳細は折りたたみ、drawer、別画面、別タブに逃がす
- 画面全体をスクロール前提にしすぎない
- スクロール領域を作る場合は、末尾要素まで実クリックできることを確認する
- 1920x1080 と 1366x768 の両方で、主要導線が隠れないことを確認する

### 10.7 追加禁止事項

画像参照ありの新 UI では、特に次を禁止する。

- 右側に監視パネルを追加する
- 小さい表を大量に並べる
- 12px 級の小さい文字を使う
- 生ログをそのまま並べる
- DB ステータス、GPU 使用率、イベントログを常時表示する
- コードエディタを常時表示する
- 1 画面に Chat / Develop / Reports / Debug を全部入れる
- 見た目だけ派手にして可読性を落とす
- 「便利そうな情報」を勝手に追加する

## 11. Viewer 再構築で使う Skill / ツール

Viewer の再構築、既存タブの新 UI 化、大きなレイアウト変更では、作業段階ごとに次の Skill / ツールを優先して使う。

### 11.1 調査・棚卸し

目的: 既存 Viewer の構造、状態管理、API、CSS、タブ責務、表示契約を壊さないための事前確認。

優先:

- `analyze-codebase`: Viewer 周辺コード、仕様、関連 docs / rules の棚卸し
- `debug-investigate`: 表示崩れ、状態不整合、クリック不能、イベント更新不良の原因調査
- `rg` / `git diff` / `go test`: 既存仕様、差分、近接テストの確認

確認すること:

- `DESIGN.md`、`docs/01_正本仕様/実装仕様.md`、`docs/09_Viewer/Viewer仕様.md`、このファイルを読んでいる
- `internal/adapter/viewer/viewer.html`、`assets/css/`、`assets/js/`、対象タブの責務を把握している
- 既存 API、表示 state、イベント、WebSocket、localStorage / sessionStorage の扱いを壊さない

### 11.2 UI 案・デザイン検討

目的: いきなり本体へ大きな差分を入れず、Viewer として使える画面密度、情報量、導線を確認する。

優先:

- `aidesigner-frontend`: Viewer 再設計、画面案、UI 構成の生成・改善
- `ui-design`: 1 ファイル HTML の検討用モック作成
- Figma MCP / `figma-generate-design` / `figma-use`: Figma 上で画面、コンポーネント、デザインシステムを作る場合
- `imagegen`: キャラクター、背景、補助ビジュアルなど、bitmap 素材が必要な場合

注意:

- `DESIGN.md` を視覚方針の正本にする
- 検討用モックを本体へそのまま貼り付けない
- Figma や画像生成は補助であり、最終判断は Viewer の実装制約、表示契約、実ブラウザ検証に戻す

### 11.3 実装・組み込み

目的: Viewer 本体へ、責務を崩さず段階的に組み込む。

優先:

- `apply_patch`: 小さい差分で HTML / CSS / JavaScript を編集する
- `shadcn`: shadcn/ui または同等のコンポーネント方針を導入・参照する必要がある場合
- 既存の `assets/css/`、`assets/js/tabs/`、Viewer API helper を優先する

注意:

- 既存タブ、既存 API、Viewer 表示契約を壊さない
- 1 画面ずつ実装し、差分を追える単位にする
- 新規依存、外部 UI ライブラリ、ビルド方式変更は高リスク操作として扱う

### 11.4 実ブラウザ検証

目的: 見た目だけでなく、実際に操作できる Viewer として成立していることを確認する。

優先:

- `playwright`: Viewer の実ブラウザ操作、DOM、console、network、スクリーンショット検証
- `playwright-interactive`: 継続ブラウザセッションで見ながら調整する場合
- `chrome-devtools`: performance trace、console / network、ページスクリーンショット確認
- `screenshot`: OS レベルのスクリーンショットが必要な場合
- RenCrow Browser Actor: RenCrow 内の browser 操作機能として検証・将来連携する場合

必須確認:

- desktop と narrow / mobile 幅で表示が破綻していない
- 主要タブ、ボタン、入力欄、select、copy / export / send が実クリックできる
- `pointer-events`、`z-index`、`position`、`background`、`border`、`box-shadow`、`backdrop-filter` が主要操作を妨げていない
- console error、network error、WebSocket 更新不良がない、または残リスクとして明示されている

### 11.5 完了条件

Viewer 再構築を完了扱いにするには、少なくとも次を満たす。

- 仕様、デザイン方針、Viewer 実務ルールを読んだ上で実装している
- 変更対象の責務と既存表示契約を説明できる
- desktop と narrow / mobile の実ブラウザ確認をしている
- スクリーンショットまたは検証ログで、主要導線が操作可能であることを確認している
- 未検証箇所、壊したくない既存機能、残リスクを報告している

## 12. 新 UI 作成プロンプト雛形

新しい Viewer UI を Codex や別 LLM に作らせるときは、`docs/09_Viewer/Viewer新UIタブ生成プロンプト.md` を使う。

この章の雛形は概要確認用である。実際の作業指示には、同ファイルの最新版を優先する。

````md
# TASK: RenCrow Viewer 新 UI タブの HTML を作成する

## 目的

RenCrow Viewer に、新しい常用 UI タブを追加する。

今回作りたい画面は、添付画像のような雰囲気の「夜の作業机」風 UI である。

画像はデザイン参照であり、完全コピーしない。実際に使える HTML / CSS / JavaScript として、読みやすく、迷わず使える画面にする。

## 最重要方針

新 UI は、従来の監視ダッシュボードではない。

目的は、れんが日常的に使うための「大きく、読みやすく、迷わない、AI キャラクターが横にいる作業机」を作ることである。

重要: このタスクでは「便利そうな情報を追加する」ことを禁止する。まず画像のような、少ない情報量で読みやすい画面を完成させること。

## 絶対に守る UI ルール

- 大きなキャラクター表示を左側に置く
- 大きな会話カード、または大きな情報カードを中央に置く
- 入力画面では大きな入力欄を下部に置く
- Chat の最下行入力欄は常時画面内に表示し、1 から 2 行程度の高さに抑える
- 閲覧画面では明確な次アクション領域を下部または主導線上に置く
- 情報量を少なくする
- 画面タイトルは 20px から 22px を基準にする
- 本文文字は 12px から 14px を基準にする
- 入力欄は 13px から 14px を基準にする
- タブ、ピル、短いメタ表示は 10px 未満にしない
- 夜の作業机の雰囲気にする
- 必要な情報だけを表示する
- 詳細ログや内部状態は初期表示しない
- 監視ダッシュボード風にしない
- 小さいログ一覧にしない
- 右側に細かいカードを大量に並べない

## デザイン参照

添付画像 `image.png` はデザイン参照である。

再現すべき要素:

- ダークネイビー系の背景
- 透明感のあるガラス風パネル
- 青から紫の控えめなグロー
- 大きな角丸カード
- 左側の大きなキャラクター領域
- 中央の大きな会話カードまたは情報カード
- 下部の大きな入力欄または次アクション領域
- 余白の多い構成
- 読みやすい大きな文字

再現しないこと:

- 画像そのものを完全コピーする必要はない
- キャラクター画像は仮でよい
- 細かい装飾を増やしすぎない
- ログ、DB、ステータス、監視パネルを追加しない

## 実装対象

まずは以下のどれか 1 画面を作る。

```text
Home / Daily Desk
Chat
Develop
Instructions
Reports
```

ユーザー指定がある場合はそれに従う。指定がない場合は `Chat` を作る。

## 成果物

UI 検討用モックの場合:

- 1 ファイル完結の HTML を作成する
- HTML / CSS / JavaScript を 1 ファイルにまとめる
- 外部ライブラリは使わない
- 画像がなくても崩れない
- 1920x1080 で見やすいこと
- 1366x768 でも破綻しないこと

RenCrow Viewer へ組み込む場合:

- 既存の `internal/adapter/viewer/viewer.html`、`assets/css/`、`assets/js/tabs/` の責務分割に従う
- 1 ファイル HTML モックをそのまま本体へ貼り付けない
- 既存タブ、既存 API、Viewer 表示契約を壊さない

## 推奨 DOM 構造

```html
<body>
  <main class="rc-shell">
    <aside class="rc-character-pane">
      <header class="rc-brand"></header>
      <section class="rc-character-portrait"></section>
      <section class="rc-status-card"></section>
      <section class="rc-character-grid"></section>
    </aside>

    <section class="rc-main-pane">
      <nav class="rc-tabs"></nav>
      <section class="rc-content-area"></section>
      <footer class="rc-input-dock"></footer>
    </section>
  </main>
</body>
```

## 受け入れ条件

- 左に大きなキャラクター領域がある
- 中央のカードが大きく読みやすい
- 入力画面では入力欄が大きい
- Chat の最下行入力欄が常時見えている
- Chat の最下行入力欄が縦に大きすぎず、会話表示領域を圧迫していない
- 閲覧画面では次アクションが分かりやすい
- 情報量が少ない
- 文字が読みやすい
- 夜の作業机の雰囲気がある
- 右側に監視ダッシュボードが増えていない
- 小さいログ一覧が主役になっていない
- 初見で「何をする画面か」が分かる
- 1920x1080 と 1366x768 で主要操作が隠れない
- 固定 UI によってタブ、ボタン、入力欄、カード操作が塞がれていない

## 追加確認

Playwright または同等の実ブラウザ環境で、1920x1080 と 1366x768 のスクリーンショットを撮り、上記の受け入れ条件を確認する。
可能なら 390x844 前後の narrow / mobile 幅でもスクリーンショットまたはレイアウト計測を行い、長文、URL、固定入力バー、overlay による崩れや遮蔽がないことを確認する。
ブラウザ確認環境が使えない場合は、環境不備として扱い、Viewer UI 変更を完了扱いしない。

また、主要タブ、主要ボタン、入力欄、select、copy / export / send などを実クリックし、固定 UI や `z-index` によって操作が遮られていないことを確認する。

修正が必要な場合は、以下の優先順位で直す。

1. 文字を大きくする
2. 情報量を減らす
3. 入力欄または次アクション領域を必要十分な大きさにする
4. キャラクター表示を大きくする
5. 余白を増やす
6. 右側の余計な情報を消す
7. 夜の作業机の雰囲気に寄せる
8. 固定 UI に隠れる要素をなくす
````
