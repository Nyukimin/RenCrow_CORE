# rules_observation_verification.md - 観測・検証・不具合調査ルール

**作成日**: 2026-05-09
**目的**: ユーザー観測、実機確認、E2E確認、調査記録の扱いを定義する
**適用範囲**: 不具合調査、UI/Viewer確認、外部サービス連携、実機運用を伴うすべての作業

---

## 1. 観測優先

- ユーザーの実機観測は、AI の推測・局所テスト・ログ解釈より優先する
- 観測と分析が矛盾した場合は、分析側を疑って調査をやり直す
- 「こちらのテストでは通る」は、ユーザー観測を否定する根拠にならない
- ユーザーが「さっきまでできていた」「前に決めた」「仕様にあるはず」と言った場合は、仕様存在または回帰の強い証拠として扱う
- 「仕様がない」と報告する前に、正本仕様、関連 docs、rules、直近テスト、直近差分を検索する
- 検索し切れていない場合は「未確認」と報告し、「仕様なし」と断言してはいけない

---

## 2. 完了判断の禁止事項

以下だけで「直った」「正常」と判断してはいけない。

- ユニットテストが通った
- Node/Go の局所テストが通った
- 画面が開いた
- DOM 要素が存在した
- health endpoint が ok を返した
- エラーがログに出ていない

これらは必要条件であって、実機動作の十分条件ではない。

---

## 3. UI / Viewer の確認基準

RenCrow の UI / Viewer 変更では、ブラウザ実機確認環境を必須とする。
Playwright Chromium、通常の Chrome / Chromium、または同等に実ブラウザ描画・console・network・クリック操作を確認できる環境がない状態では、Viewer 変更を完了扱いしてはいけない。
環境が壊れている場合は、先にブラウザ確認環境を復旧するか、未検証として明示する。

UI 表示不具合では、対象ユーザーフローを最低 1 セッション分追う。

確認すること。

- 開始イベント
- 主要な中間イベント
- 表示された本文
- 境界イベント
- 終了イベント
- 表示ログまたは DOM 上の実表示
- イベントログとの対応

Viewer のようにストリーム表示を行う UI では、単に要素を見るだけでは足りない。
「混ざらない」「欠けない」「順序が正しい」「読める」を確認する。

---

## 4. 表示ログの原則

目視だけで確認できない場合は、表示された文字列をログ化する。

- イベント受信ログ
- 描画ログ
- DOM 上の最終表示
- 必要ならスクリーンショット

ログは「後から再現できる形」で残す。
調査記録の保存先と形式は `rules/common/rules_logging.md` に従う。

---

## 5. 網羅性の扱い

不具合や違和感を列挙したら、優先度で消してはいけない。

- すべての事象を番号付きで保持する
- 優先度は作業順のためだけに使う
- 品質判断では、各事象の状態を `未調査` / `原因特定` / `修正済み` / `検証済み` のように追跡する
- 「代表例」だけで完了扱いしない

バグは細部に宿る。小さく見える事象も、根本原因の入口として扱う。

---

## 6. 再試行の上限

同じ仮説や同じ手法の微修正は 2 回までにする。
2 回で解決しなければ、別のパッチを重ねる前に情報収集へ戻る。

確認する情報源の例。

- ユーザー観測
- サービスログ
- ブラウザ console
- ネットワーク / WebSocket 状態
- 設定ファイル
- 実行中プロセス
- 実機でのイベント順

---

## 7. 報告形式

不具合調査の報告では、最低限これを分けて書く。

- 観測された事実
- こちらで再現できた事実
- まだ推測のもの
- 修正したもの
- 検証したもの
- 未検証の残リスク

推測を事実のように報告してはいけない。
