# rules_regression_prevention.md - 再発防止・教訓網羅ルール

**作成日**: 2026-05-09
**目的**: 過去の失敗を個別事象として散逸させず、同種の過ちを再発させないための網羅ルールを定義する
**適用範囲**: 不具合調査、UI/Viewer、STT/TTS、状態管理、実機運用、ルール更新、完了報告を伴うすべての作業

---

## 1. このルールの位置づけ

このファイルは、特定の 1 バグではなく、過去に発生した失敗群を横断して再発防止するためのルールである。

ここで扱う失敗群:

- ユーザー観測を Ground Truth として扱えなかった
- 実機 / Viewer の確認が不足した
- 仕様理解より先に実装へ進んだ
- 仕様が存在するのに、確認不足で「仕様なし」と誤報告した
- ID、cache、queue、pending 状態を増やして状態の真実を複数にした
- TTS / STT / 表示 / 口パク / ログの責務を混同した
- 再起動、ルール探索、報告の運用が雑だった
- バグ一覧を優先度で圧縮し、細部を取りこぼす危険を作った

このルールは `AGENTS.md` の絶対ルールを補強し、詳細は以下と連携する。

- `rules/common/rules_observation_verification.md`
- `rules/common/rules_state_management.md`
- `rules/common/rules_logging.md`
- `rules/common/rules_testing.md`
- `rules/common/rules_architecture.md`

## 2. 絶対原則

### 2.1 観測を否定しない

ユーザーの実機観測は、AI の推測、単体テスト、局所ログ、health check より優先する。

禁止:

- 「こちらでは通ったので直った」と判断する
- ユーザー観測と矛盾する結果を、説明なしに正常扱いする
- 実機で見えている症状を、再現できないという理由だけで除外する

必要な行動:

- 観測内容を番号付きで保持する
- 観測と自分の検証結果が矛盾したら、検証方法を疑う
- 実機相当の確認を追加する

### 2.2 完了条件を局所テストにしない

局所テストは必要条件であり、完了条件ではない。

UI、Viewer、音声、ストリーム、外部サービス連携では、対象フローを最低 1 セッション通して確認する。

最低確認項目:

- 入力または開始操作
- 中間イベント
- 表示本文
- 境界イベント
- 終了状態
- イベントログとの照合
- 表示ログまたはスクリーンショット

### 2.3 網羅リストを維持する

不具合や教訓を列挙したら、優先度で消してはいけない。

- 優先度は作業順のためだけに使う
- 品質判断では全項目を追跡する
- 「代表例」「トップ 5」「重要なものだけ」で完了扱いしない
- 小さく見える事象も、根本原因の入口として残す

## 3. 仕様理解と調査順序

### 3.1 実装より先に仕様を固定する

仕様が曖昧なまま、実装都合で解釈してはいけない。

「仕様がない」「未定義」「以前からそうだっただけ」と報告する前に、必ず以下を確認する。

- `docs/README.md` と対象領域の現行正本
- `rules/` 配下の関連ルール
- 直近で追加・変更されたテスト
- 直近差分と、仕様に反する期待値へ変わっていないか

ユーザーが「さっきまでできていた」「前に決めた」「仕様にあるはず」と指摘した場合、それは仕様存在または回帰の強い証拠である。
この場合は、実装・テスト・自分の検索範囲を先に疑い、確認できていない状態で「仕様なし」と断言してはいけない。

特に以下は、先に意味を確認する。

- 既存 ID の意味
- session / request / response / utterance / chunk の単位
- cache の目的
- 表示完了と発話完了の違い
- ユーザー操作とシステム状態の関係
- normal chat と Idle / Viewer / STT / TTS の共通化境界

### 3.2 調査順序を守る

不具合調査では、以下の順序を基本にする。

1. ルールと仕様を読む
2. ユーザー観測を番号付きで整理する
3. 関連コードと周辺コードを読む
4. ログ、設定、プロセス、ネットワーク、イベントを集める
5. 仮説を立てる
6. 最小の再現または検証を行う
7. 必要最小限の変更を行う
8. 実機または E2E で確認する
9. 未解決項目と残リスクを報告する

同じ手法で 2 回失敗したら、次のパッチを重ねる前に情報収集へ戻る。

## 4. ID・状態・キャッシュ

### 4.1 ID を増やす前の必須確認

新しい ID を追加する前に、以下を確認する。

- 既存の `session_id` などで表現できないか
- その ID が表す単位は何か
- 既存 ID と寿命が違うか
- どこで生成し、どこで破棄するか
- 欠落時に合成 ID を作るなら衝突しないか
- 表示、再生、重複排除、ログ、キャッシュのどれに使うか

単位が曖昧な ID を表示制御や重複排除に使ってはいけない。

### 4.2 状態の真実を一つにする

event、DOM、pending queue、TTS queue、rendered log、cache、derived key が同じ意味を持ってはいけない。

複数状態が必要な場合は、以下を明記する。

- 主たる真実
- 補助状態の目的
- 所有モジュール
- 破棄タイミング
- セッション境界
- 不整合時の復旧方法

### 4.3 cache / queue は整合性の代替にしない

cache、stock、queue、pending は便利な逃げ道ではない。

禁止:

- 空値や不正値を cache / stock / queue に入れる
- topic や session の境界を越えて古い pending を残す
- queue の中身を表示済みの真実として扱う
- cache の存在を理由に元イベントの整合性検証を省く

## 5. 表示・TTS・STT・口パクの分離

### 5.1 表示は TTS chunk に依存させない

本文表示の主たる入力は表示イベントまたは表示用 state とする。製品契約は `docs/02_機能仕様.md` と `docs/06_Public_API仕様.md` を参照する。

TTS chunk は以下のためのもの。

- 音声再生
- 口パクのきっかけ
- 必要に応じた再生同期

TTS chunk を本文表示の唯一の根拠にしてはいけない。
TTS chunk の `text` / `display_text` で本文を埋める、置換する、再構成することも禁止する。
`message_id` / `turn_index` の一致は対応付けの証拠であり、TTS chunk へ本文表示権限を与える根拠ではない。
対応する表示イベントまたは表示用 state が無い場合は、本文を補完せず、診断表示またはログへ倒す。
音声再生の reset / clear / cleanup で、表示正本や ID 対応の状態を消してはいけない。

IdleChat 通常会話では、TTS chunk の `display_text` と `speech_text`（現行 `text`）を別々に分割して同じ `chunk_index` で対応させてはいけない。
chunk は `message_id` に従属し、表示補助情報と音声ファイルは同じ chunk 計画から作る。

### 5.2 発話完了と表示完了を混同しない

発話完了は音声再生制御の状態である。
本文表示完了ではない。

発話完了を待ってから次の本文を表示する設計は、仕様として明示されていない限り禁止する。
逆に、仕様で発話・TTS chunk 同期が明示されている場合は、実装やテストをその仕様に合わせる。
実装の現状に引っ張られて、仕様側を「存在しない」扱いにしてはいけない。

### 5.3 TTS 先行エンコード

TTS に時間がかかる場合、発話完了を待って次 chunk をエンコードしてはいけない。

原則:

- 文章を文単位などの自然な chunk に分ける
- 次以降の chunk を先行エンコードする
- 音声 cache は再生待ち時間の短縮に使う
- 表示本文は音声 cache の状態に依存させない

### 5.4 STT 録音と final 投入先を分離する

STT の録音状態と、認識 final をどの入力欄へ入れるかは別責務である。

例:

- 通常 chat 以外では final text を投入しない
- しかし録音セッション自体を止める必要があるとは限らない

UI タブ遷移や Viewer 確認操作が、録音を意図せず止めないことを確認する。

## 6. Topic / prompt / LLM 出力

### 6.1 空 topic を下流へ渡さない

空 topic、placeholder topic、不正 topic を LLM 会話生成へ渡してはいけない。

防御点:

- 生成直後
- cache / stock push 前
- cache / stock pop 後
- prompt 組み立て前
- Viewer / TTS 表示前

表示用 placeholder と LLM に渡す本 topic は分離する。

### 6.2 LLM 内部形式を表示・発話へ流さない

内部推論、Harmony 断片、prompt 断片、箇条書き分析、自己説明は topic / response として使わない。

検出した場合:

- visible answer を抽出する
- 抽出不能なら破棄または retry する
- そのまま Viewer / TTS / memory へ流さない

### 6.3 途中切れを正常扱いしない

finish reason、末尾、構文から途中切れを疑う。

途中切れを検出したら、表示や発話へ流す前に retry / fallback / stop のいずれかを選ぶ。

## 7. 実機運用と再起動

再起動は変更確認の一部ではなく、運用操作である。

RenCrow / RenCrow では以下を守る。

1. `systemctl --user stop rencrow.service`
2. 残存プロセス停止
3. `:18790` の listen 消失確認
4. `/health` が応答しないことを確認
5. build
6. service start
7. `/health` が ok になることを確認

途中を省略して `restart` だけで済ませてはいけない。

---

## 8. ルール探索と報告

### 8.1 ルールファイルの存在確認

ルールや入口ファイルの存在を確認するときは、検索条件だけに依存しない。

最低限:

- `find`
- `rg --files`
- 既知パスの直接 `ls`

存在しないと報告する前に、複数手段で確認する。

### 8.2 報告では事実と推測を分ける

報告には以下を分けて書く。

- 観測された事実
- 自分で再現した事実
- 推測
- 修正済み
- 検証済み
- 未検証の残リスク

確認していないことを確認済みのように言ってはいけない。
