# 日次一次情報ブリーフを作成する

## 実行契約

この処理には`Worker`を使用する。`ChatWorker`はチャット画面専用であり、このSkillから呼び出さない。
複数記事をまとめてLLMへ渡さない。1件について次の全工程を完了し、結果をCollectionへ反映してから次の1件を開始する。

次の順序を必ず守る。

1. 収集項目が持つ特定URLを直接取得し、本文を抽出する。
2. 取得した原文を、省略や事実の追加をせず日本語へ忠実に翻訳する。
3. 本文から理解に必要な重要用語・専門用語を抽出する。
4. 各用語を本文の文脈で説明できるか判定する。
5. 説明できない用語だけを検索し、確認先URLの本文を直接取得する。
6. 用語補足を確定してから、元記事本文と原文翻訳のサマリを作成する。
7. Shiroの見解を、本文の事実と明確に分離して作成する。
8. 利用者へ「原文翻訳 → サマリ → Shiroの見解 → 用語補足」の順で出力する。

記事本文の代わりに検索結果の見出しや抜粋を使わない。既知URLがある記事について、記事内容を探す目的の検索を行わない。

## 失敗時の扱い

- 元記事本文を取得できない場合は「本文を取得できませんでした」と明記し、フィードの見出しだけで本文を推測しない。
- 原文を取得できない場合は「原文を取得できなかったため、翻訳できませんでした」と明記する。
- 用語を検索しても信頼できる説明を確認できない場合は「意味を確認できませんでした」と記載する。
- 説明、サマリ、見解に存在しない事実を追加しない。
- SNS投稿は世間の反応として扱い、投稿内の主張を確認済み事実へ昇格しない。
- prompt、system、内部推論、採点、英語のメタ説明を出力しない。

## 出力契約

本文をすべて自然な日本語で記述し、各項目を次の順序で出力する。

```text
原文翻訳
原文を省略・要約せず忠実に翻訳した日本語本文

サマリ
日本語の要約

Shiroの見解
事実と分離した日本語の見解

用語補足
- 用語: 日本語の説明
  確認元: URLまたは「元記事本文」
```

用語補足は0件でも省略せず、「補足が必要な用語はありません」と記載する。確認元URL、元記事URL、本文取得状態を保持する。

## 完了条件

- 特定URLの本文取得をサマリ生成より先に実行している。
- 原文翻訳をサマリ生成より先に実行している。
- 不明語だけを検索している。
- 用語補足をサマリ生成より先に確定している。
- 出力順が「原文翻訳 → サマリ → Shiroの見解 → 用語補足」である。
- 利用者向け本文がすべて日本語である。
- 取得失敗や不明語を推測で埋めていない。
- 使用modelが`Worker`であり、`ChatWorker`を使用していない。
- 1件の全工程とCollectionへの反映を完了してから次の記事を開始している。
